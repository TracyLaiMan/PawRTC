ISP 3A 算法 — 直观理解

自动曝光 AE · 自动白平衡 AWB · 自动对焦 AF — 不依赖数学理论, 用直觉和类比理解每步的"为什么"

AE 自动曝光

让画面不太亮也不太暗

核心问题: 在给定光照下, 选多大的积分时间 (快门) + 增益 (ISO) 才能让画面达到目标亮度? 类比: 眼睛的瞳孔自动调节。

AWB 自动白平衡

让"白色"在任何光下都显示为白色

核心问题: 光源色温不同导致偏色, 如何估计光源颜色然后反向补偿? 类比: 大脑自动适应白炽灯的暖黄色。

AF 自动对焦

让拍摄主体清晰锐利

核心问题: 如何找到镜头的最佳位置使目标区域反差最大/相位差为零? 类比: 手动拧对焦环直到画面最清晰的那一刻。

1. 3A 在 ISP Pipeline 中的位置

3A 不是独立模块 — 它们与 ISP 流水线深度集成, 形成"采集→统计→决策→控制"的闭环

ISP Pipeline 中的 3A 闭环

Sensor → RAW → ISP → 统计 → 3A 决策 → 控制 → Sensor

Step 1 — 采集统计: ISP 从当前帧提取统计信息: 亮度直方图 (AE)、RGB 均值 (AWB)、对比度/相位 (AF)。这些统计是硬件加速的 — 不需要 CPU 逐像素计算, ISP 内部有专用的统计引擎 (HIST/IIR/AF stats)。
Step 2 — 3A 决策: CPU/固件读取统计数据 → 运行 3A 算法 → 计算下一步的控制参数。
Step 3 — 控制执行: 写 Sensor 寄存器 (曝光时间/增益) · 写镜头 VCM (对焦位置) · 写 ISP 寄存器 (白平衡增益/digital gain)。
反馈周期: 通常每帧或每 N 帧 (2~4 帧) 运行一次, 因为有帧延迟 (当前帧的统计反映的是上一帧的参数效果)。

3A 之间的互相影响 — 为什么不是独立调?

AE 改变亮度 → AWB 统计变化 → AF 对比度变化

AE → AWB: 曝光改变 → RGB 值整体缩放 → AWB 的灰色统计需要重新计算 (不能直接用上一帧的增益)。
AE → AF: 曝光不足 → 画面暗 → 噪声大 → 对比度信息不可靠 → AF 可能误判。
AF → AE: 对焦过程中画面模糊 → 邻像素混合 → 亮度统计可能偏离对焦后状态。
AWB → AE: 白平衡增益改变 RGB 通道的相对亮度 → 影响 AE 的亮度计算 (Y = 0.299R+0.587G+0.114B)。
收敛顺序: 实践中通常 AE 先收敛 → AWB 次之 → AF 最后 (因为 AE 影响最大, 曝光不对则一切统计不可靠)。

3A 统计的硬件卸载 — 为什么不能 CPU 逐像素算?
ISP 芯片内部有专用的统计引擎 (Statistics Engine), 在 Bayer 域或 YUV 域直接做硬件累加/直方图/IIR 滤波。对于 1080p@30fps (~62M 像素/秒), 逐像素 CPU 计算 VS 硬件统计: 硬件 1ms 完成 vs CPU 20ms+。这就是为什么嵌入式 IPC 的 3A 调优本质上是配置 ISP 的统计窗口 + 设置 3A 固件参数, 而非写底层像素算法。

2. AE (自动曝光) — 让画面"刚刚好"的亮度

核心: 测量当前画面亮度 → 如果太暗就增加曝光、太亮就减少曝光。难点在于"测哪里"和"怎么调"

测光: 测量"哪里"的亮度?

全局平均 vs 中心加权 vs 矩阵测光 vs 点测光

全局平均: 所有像素亮度取平均 → 最简单, 但逆光人脸会黑。
中心加权: 画面中心权重高, 边缘低 → 经典相机测光。
矩阵/多区测光: 分 N×N 格子, 每格独立统计 → 排除异常值 (天空过亮, 阴影过暗) → 取可信区的平均。
点测光: 只测画面 1~5% 区域 (一般是中心) → 用于逆光人像。ISP 中通过配置统计窗口 (AE Window)来实现。

目标亮度: 什么是"刚刚好"?

18% 灰 — 摄影学的"中间灰"

摄影学标准: 18% 反射率的"中灰色" 是人眼感知的"正常亮度"。AE 的目标就是让画面平均亮度落在这个值附近。
在 ISP 中, 目标亮度定义为亮度直方图的目标百分位: 例如 "让 50% 的像素的亮度值落在目标区间"。
实践: 目标亮度不是固定的 — 白天户外可以偏暗, 夜景可以偏亮 (人眼夜景更敏感), 通过曝光补偿表 (EV Table)来调整。

调节手段: 积分时间 + 增益

快门优先 · 增益优先 · 防闪烁

积分时间 (快门): 传感器像素收集光子的时长。越长越亮, 但运动模糊越严重。首选调节手段 (因为增加信号不增加噪声)。
增益 (ISO): 模拟/数字放大信号。会同时放大信号和噪声 → 暗光下才使用。
光圈: 手机上固定光圈 (如 f/1.8) → 不可调。相机的可变光圈由机械马达控制 → AE 需配合光圈步进。
防闪烁 (Anti-Flicker): 电网 50Hz/60Hz 导致灯光周期性亮暗。解决: 积分时间锁定为电网周期的整数倍 (50Hz→10ms, 60Hz→8.33ms)。

AE 收敛策略 — 为什么不是"一步到位"?

快慢两段调节 — 避免亮度震荡

大步快调 (离目标远) → 小步微调 (接近目标) → 死区锁定

Phase 1 — 大步快调 (Coarse): 当亮度与目标差 > 阈值 (如 30%), 用大步长 (如改变 2 档曝光) 快速逼近。这一阶段画面会明显变亮/变暗。
Phase 2 — 小步微调 (Fine): 差值 < 阈值, 步长减半 (如 0.5 档) → 避免 overshoot 导致的亮度来回摆动 (hunting)。
Phase 3 — 死区 (Deadband): 差值 < 最小阈值 (如 5%) → 不做任何调整 → 避免因噪声波动导致的无意义微调。
阻尼 (Damping): 连续 N 帧都朝同一方向时才调整, 防止单帧统计噪声导致误调。

防闪烁 (Anti-Flicker / Anti-Banding)

50Hz 灯光 = 10ms 周期 → 积分时间必须是 10ms 整数倍

交流电灯光 (LED/荧光灯) 亮度以 2× 电网频率闪烁: 50Hz 电网 → 100Hz 闪烁 (周期 10ms), 60Hz → 120Hz (周期 8.33ms)。
如果积分时间 不是 闪烁周期的整数倍 → 不同行的像素捕获到不同相位的光 → 画面出现明暗交替的条纹 (Banding)。
解决方案: AE 算法检测到 banding → 强制积分时间 = 10ms/8.33ms 的整数倍 (50Hz 用 10ms/20ms/30ms…)。代价: 无法使用任意积分时间 → 在某些亮度下转向调增益。
自动检测: 对比连续帧的亮度波动 → 若波动频率 = 100Hz/120Hz → 确认有 flicker → 启用防闪烁策略。

⚠️ AE 最常见的坑 — 画面亮度来回摆动 (Hunting/Oscillation)
根因: 调节步长太大 + 帧延迟 (当前帧的统计反映上一帧的参数) + 没有死区 → 曝光参数在目标值两侧来回振荡。
诊断: 打印每帧的 Y_mean → 若呈正弦波状波动 → 减小步长 + 增加阻尼帧数 + 增大死区范围。
极端场景: 场景中出现闪烁光源 (警灯/霓虹灯/电视屏幕) → 亮度本身在快速变化 → AE 无法区分"场景变化"还是"我调错了" → 需要延长收敛时间或切换到手动曝光。

3. AWB (自动白平衡) — 让"白纸"在任何光下都是白色的

核心: 估计当前光源的颜色 → 反向补偿。一张白纸在阳光下是白色, 在白炽灯下是暖黄色 — AWB 的任务就是"把白炽灯下的白纸变回白色"

灰度世界假设 (Gray World)

"大自然平均来说是灰色的"

核心思想: 在足够丰富的场景中, 红、绿、蓝三个通道的平均值应该相等。如果不等 → 说明有偏色 → 计算使三者相等的增益。
算法: 算 R/G/B 各通道均值 → 取三均值的中位作为"理想灰" → 每通道增益 = 灰/通道均值。例如: R=200,G=100,B=80 → 灰=(200+100+80)/3=127 → R_gain=127/200=0.64, G_gain=1.27, B_gain=1.58。
失效场景: 大草坪 (全是绿色) → 算法会认为"绿色太多了, 需要减绿" → 草坪变成紫色。大蓝天同理。

完美反射假设 (White Patch)

"画面中最亮的点就是白色"

核心思想: 场景中最亮的物体通常是白色/镜面反射 → 它的颜色直接反映光源颜色。
算法: 找亮度 top 1~5% 的像素 → 假设它们是"白色" → 直接算使它们变纯白的增益: R_gain=255/max_R, G_gain=255/max_G, B_gain=255/max_B。
失效场景: 场景中有红色霓虹灯 (最亮但非白) → 白平衡会把画面调成偏青。
改进: 只在接近灰轴 (Cr/Cb ≈ 0) 的候选像素中做完美反射 → 排除彩色光源的干扰。

色温估计法 (最鲁棒)

统计像素在色度空间的分布 → 匹配到最近的标准光源

核心思想: 不同色温的光源在 R/G vs B/G 色度空间中落在一条黑体辐射曲线 (Planckian Locus) 上。统计场景中"可能是灰色"的像素 → 它们的色度集中在哪个色温点 → 就假设光源是这个色温。
灰色像素的判定: 在 YCbCr 空间中, 白色/灰色像素的 Cr/Cb ≈ 0 (没有色度)。选出 |Cr|<阈值 且 |Cb|<阈值 的像素作为候选灰点。
鲁棒性: 草坪/蓝天不会被选为灰点 (它们的 Cr/Cb 不接近零) → 比灰度世界鲁棒得多。这是当前 ISP 最常用的 AWB 算法。

AWB 的工程实践 — 三个最重要的参数
① 灰点判定阈值 (Cr/Cb 窗口): 太小 → 找不到足够灰点 → 统计不可靠。太大 → 彩色物体混入 → 偏色。通常 Cr/Cb 在 ±0.1 范围内。
② 色温范围 (CCT Range): 限制白平衡的校正范围 (如 2500K~7500K) → 低于 2500K 的场景 (如日落) 保留暖色调, 不过度校正。
③ 切换平滑 (Temporal Filter): 场景切换时 (室内→室外), 白平衡增益不能突变 → 用 IIR 滤波平滑过渡: gain[n]=α·gain[n-1]+(1-α)·gain_calculated[n], α=0.9~0.95。

4. AF (自动对焦) — 让主体清晰锐利

核心: 找到镜头位置使得目标区域的细节最多 (反差最大 / 相位差为零)。两种主流技术: 反差检测 (CDAF) 和相位检测 (PDAF)

反差检测 AF (CDAF) — 爬山算法

"画面越清晰 → 高频分量越多 → 对焦越好"

原理: 对焦准确时, 物体的边缘最锐利 → 相邻像素的亮度差最大 → 高频分量最强。AF 统计引擎计算 ROI 区域的高通滤波响应值 (FV, Focus Value)。
爬山法: ① 粗扫 (大步长) 找到 FV 峰值大致位置 → ② 细扫 (小步长) 在峰值附近精确找顶 → ③ 如果没有比上一步的 FV 大 → 说明已经越过了峰值 → 回退一步。
缺点: ① 需要越过峰值才知道"已经过了" → 来回摆动 → 速度慢 ② 暗场景噪声大 → FV 值不可靠 → 可能对到错误位置。

相位检测 AF (PDAF) — 一次算出方向和距离

"左眼和右眼看到的同一物体, 水平错位了多少?"

原理: Sensor 上嵌入了遮罩像素对 (Shielded Pixel Pairs) — 每对像素, 一个只能看到镜头左侧的光, 另一个只能看到右侧的光。对焦准确时 → 左右看到的图像完全相同 (无错位)。离焦时 → 左右图像有水平错位 (Phase Difference) → 错位量正比于离焦量 → 一步就可算出需要移动多少。
优势: 不需要爬山 → 速度极快 (几十 ms)。
缺点: 需要专用 PDAF sensor (成本更高) + 暗光下信噪比低 → 相位差检测不准 → 退化到 CDAF。

激光对焦 (Laser AF / ToF) — 辅助手段

ToF (飞行时间) 测距

发射红外激光脉冲 → 测量反射回来的时间 → 距离 = c×t/2

在近距离 (< 2~5 米) 场景, ToF 传感器可以直接给出物体距离。AF 据此设镜头到对应对焦位置 → 几乎零延迟。
典型应用: 手机近摄 (美食/文档) — 激光对焦 + CDAF 微调 → 又快又准。
局限: 远距离 (>5m) 激光能量衰减 → 不准。黑色/透明物体吸收/透射激光 → 无法测距。

AF 窗口 (ROI) 配置

不是全屏对焦 — 只对 ROI 区域计算 FV

多窗口 AF: ISP 在画面上设置多个 AF 窗口 (如 9 宫格), 每个窗口独立计算 FV → AF 算法选择最近距离的物体 (或人脸检测到的窗口) 作为对焦目标。
人脸优先 AF: 人脸检测 (FD) 引擎输出人脸区域 → AF 自动将窗口对准人脸 → 保证人脸清晰。
触控对焦: 用户点击画面某处 → AF 窗口移到该位置 → 重新对焦。

⚠️ 3A 联动调优 — 最常见的踩坑场景
场景 1: 低光 + AWB 偏色 + AF 失败: 暗场景→AE 加大增益→噪声放大→灰点检测不准→AWB 漂移→同时对比度信息不可靠→AF 无法对焦。解决: 低光照时优先锁定 AE 和 AWB, 仅让 AF 操作 (可能需要补光灯)。
场景 2: 色温突变 (室内→室外): AWB 平滑因子 α=0.95→过渡需要 ~2 秒→用户看到明显色偏→投诉。解决: 检测到色温突变 (>500K 变化) → 临时降低 α 到 0.7 → 快速收敛。
场景 3: 强逆光人脸: AE 全局平均→天空过亮→缩减曝光→人脸全黑。解决: 用人脸检测区域做局部加权 AE (人脸区权重 ×3~5) → 保证人脸正常曝光。
