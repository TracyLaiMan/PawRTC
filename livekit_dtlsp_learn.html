自签名证书：为什么不用 CA？
HTTPS 需要 CA 证书链来验证"这个证书属于这个域名"。
WebRTC 不用 CA——它用 SDP 指纹绑定。两种信任模型有何区别？
WebRTC 信任模型
自签名证书 + SDP 指纹
签发者：自己 (O=KVS-WebRTC-Client)
有效期：365 天自动续签
密钥：ECDSA P-256 (临时生成)
签名算法：SHA-1 (应改为 SHA-256)
验证方式：信令通道传指纹 → DTLS 握手后比对
信任锚：受 TLS 保护的信令通道 (WSS)
代码：Dtls_mbedtls.c:809
HTTPS 信任模型
CA 证书链
签发者：受信任的 CA (如 Let's Encrypt)
有效期：90 天
密钥：RSA 2048+ 或 ECDSA P-256
签名算法：SHA-256
验证方式：浏览器内置 CA 根证书 → 逐级验证
信任锚：浏览器/OS 预装的 CA 根证书
优势：任何人都能验证，无需预先共享秘密
核心区别：HTTPS 需要让不特定的浏览器验证服务器身份，所以需要 CA 这个公共信任锚。WebRTC 的证书只需要让已经通过信令通道连接的对端验证，指纹通过 TLS 保护的信令通道传递就够了——自签名 + 指纹绑定 = 等效 CA 验证，且更简单。

为什么 WebRTC 不用 CA？因为 WebRTC 是 P2P：双方可能都没有域名，也不需要向全世界证明身份——只需要向对方证明"我是你通过信令通道连接的那个人"。

SDP 指纹绑定：防中间人的关键
DTLS 握手时，别人可以替换证书。怎么识别？
先通过 TLS 信令通道传证书的哈希指纹，握手后比对即可发现篡改。
↓
VERIFY_OPTIONAL：板端的致命漏洞
板端因为 CA 证书不完整，跳过了 TLS 证书验证。
看起来只是少了一步检查，但攻击者可以借此摧毁整条安全链。
✅ 正确做法
VERIFY_REQUIRED
mbedtls_ssl_conf_authmode(&cfg, MBEDTLS_SSL_VERIFY_REQUIRED); mbedtls_ssl_set_hostname(&ssl, "your-server.com"); // 板端验证服务器证书 → CA 链合法 → 连接建立 // 攻击者伪造证书 → CA 验证失败 → 连接拒绝
板端严格验证服务器证书的 CA 签名链。攻击者无法伪造受 CA 信任的证书，中间人攻击被阻止。
❌ 当前做法
VERIFY_OPTIONAL
mbedtls_ssl_conf_authmode(&cfg, MBEDTLS_SSL_VERIFY_OPTIONAL); // 板端跳过服务器证书验证 → 任何证书都接受 // 攻击者伪造 LiveKit Cloud 证书 → 板端不报错 // → 中间人成功 → 篡改 SDP 指纹 → DTLS 中间人
板端跳过验证，任何证书都被接受。攻击者伪造证书即可建立中间人位置，看到并篡改所有信令。
攻击路径：
1. 攻击者与板端在同一网络，伪造 LiveKit Cloud 的 TLS 证书
2. 板端 VERIFY_OPTIONAL → 不验证 → 握手成功 → 攻击者获得信令中间人位置
3. 篡改 SDP 中的 a=fingerprint:sha-256 为自己的 DTLS 证书指纹
4. 板端收到篡改后的 SDP → DTLS 握手时指纹匹配（因为指纹已经是攻击者的了）
5. 板端与攻击者协商 SRTP 密钥 → 攻击者解密全部视频/音频

修复方案：推送完整 ca-certificates.crt 到 /etc/ssl/certs/，然后改为 VERIFY_REQUIRED。

密钥导出：从 DTLS 到 SRTP
DTLS 握手完成后，双方都掌握了相同的 master_secret。
通过 TLS PRF 从 master_secret 导出 SRTP 密钥，然后加密通信正式开始。
