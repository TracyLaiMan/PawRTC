1
视频源初始化
Bento4 解析 MP4 容器，提取 H.264 AVC bitstream
log [Mp4Source] AVC: nalu_length_size=4, SPS/PPS prefix=41 bytes
[Mp4Source] Opened: /data/test_video.mp4 (1920x1080 @ 30fps, 1800 samples, timescale=15360)
📦 技术细节
AVCC → Annex-B 转换：MP4 中 NAL 使用 4 字节 length-prefix 格式（AVCC），推流前需转为 00 00 00 01 start code 格式
SPS/PPS 参数集：提前提取 41 字节参数集，每帧 IDR 前自动插入 SPS+PPS，确保浏览器能独立解码
时间基准转换：MP4 timescale=15360，15360/30fps = 512 ticks/帧，RTP 需统一到 90000Hz 时钟

▶ 为什么必须强制 ALPN=http/1.1？
LiveKit Cloud 前面有 Cloudflare CDN，CDN 默认协商 HTTP/2。但 HTTP/2 不支持 Upgrade 机制（RFC 8441 的 extended CONNECT 不在范围内），WebSocket 升级只能在 HTTP/1.1 上完成。

如果不设 ALPN，CDN 会协商 H2，WS upgrade 头会被吞掉，返回 HTTP 200 而非 101，连接看似成功但 WebSocket 未建立。
▶ 交叉编译 ALPN 的坑
LWS 内部用 CHECK_C_SOURCE_COMPILES 检测 mbedTLS 是否支持 ALPN，但交叉编译时 try_compile 在 x86 主机执行，链接不到 mipsel 的 mbedTLS，所以所有 feature detection 都失败。

必须手动注入宏：
cmake -DLWS_HAVE_mbedtls_ssl_conf_alpn_protocols=1 -DLWS_HAVE_mbedtls_ssl_get_alpn_protocol=1 -DLWS_HAVE_mbedtls_ssl_conf_sni=1

▶ TCP 候选被拒绝的原因
KVS ICE Agent 在 IceAgent.c:438 硬编码拒绝了所有 TCP 候选：

C CHK(STRNCMPI(ICE_TRANSPORT_TYPE_TCP, curr, tokenLen) != 0, STATUS_ICE_CANDIDATE_STRING_IS_TCP);

这是 KVS 的已知限制——没有实现 TCP 传输层。LiveKit Cloud 通过 CDN 返回的大部分候选是 TCP（tcptype passive），全部被丢弃。
Trickle ICE（RFC 8838）：候选不再等全部收集完才发送，而是边收集边通过信令通道逐条发送，加速连接建立。

DTLS 握手
ECDHE + ECDSA P-256 密钥协商
log [LkTransport] DTLS role from answer: server
[LkTransport] DTLS handshake started as server
[PROFILE] dtlsSessionChangeState(): [DTLS initialization completion] Time taken: 635.41 ms
DTLS 握手流程
SFU→ ClientHello (ECDSA P-256, ECDHE)
→板端 ServerHello + Certificate (自签名, SHA-256 指纹)
→板端 ServerKeyExchange (ECDHE 参数签名)
→板端 CertificateRequest + ServerHelloDone
SFU→ ClientKeyExchange (ECDHE 公钥)
SFU→ ChangeCipherSpec + Finished (encrypted)
→板端 ChangeCipherSpec + Finished (encrypted)
✓ DTLS 握手完成 (~635ms) — SRTP 密钥已派生
mbedTLS 3.6.3 实现 DTLS 1.2，密钥交换 ECDHE + ECDSA P-256
自签名证书 2048-bit，有效期 365 天，运行时生成 — 可用 lk_transport_pregenerate_dtls_cert() 预生成节省时间
SFU 验证：从 SDP a=fingerprint:sha-256 匹配板端证书指纹
⚠635ms 在 mipsel CPU 上较慢，预生成证书可降至 ~1.0s 首帧时间

安全技术详解
6 层安全机制 — 从信令到媒体的逐层保护
1
JWT 认证 — 信令通道身份校验
HMAC-SHA256 对称签名，API Secret 作为共享密钥
▶
属性实现风险
完整性HMAC-SHA256 签名任何人无法篡改 payload 而不被发现
时效性exp/nbf 字段过期 token 被 SFU 拒绝
最小权限canPublish/canSubscribe 独立控制设备 token 可只授权推流
⚠JWT 通过 URL query string 传递，可能泄露到 CDN 日志
⚠HMAC 对称签名，API Secret 泄露 = 任意权限伪造
2
WSS (TLS 1.2) — 信令通道加密
mbedTLS 3.6.3 + LWS 4.3.5，ECDHE 前向保密
▶
密码套件密钥交换加密
TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256ECDHE+ECDSAAES-128-GCM
TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256ECDHE+RSAAES-128-GCM
前向保密 (PFS)：ECDHE 密钥交换保证即使服务器私钥泄露，历史会话密钥也无法推导
⚠板端 VERIFY_OPTIONAL 不完全验证服务器证书 — 中间人攻击风险
3
ICE 认证 — STUN/TURN 消息完整性
短期凭证(STUN) + 长期凭证(TURN)，HMAC-SHA1
▶
STUN 短期凭证
USERNAME: "KXfxYlfOdnPeKSWc:4aHMxe4V" ← remote:local ufrag MESSAGE-INTEGRITY: HMAC-SHA1(key=ice_pwd, message) ← 20 字节 FINGERPRINT: CRC32(message) ← 检测中间盒篡改
TURN 长期凭证
TURN 服务器需要独立用户名/密码认证，由 LiveKit Cloud 在 JoinResponse 中动态下发，每个会话不同。
使用挑战-响应模式（401 → 重发带 MESSAGE-INTEGRITY 的请求），REALM + NONCE 防止重放攻击。
4
DTLS 握手 — 媒体通道密钥协商
ECDHE + ECDSA P-256 自签名证书，X.509 v3
▶
项值评估
密钥类型ECDSA P-256等效 RSA-3072，足够安全
证书签名SHA-1弱项 — 应改为 SHA-256
自签名Issuer=SubjectWebRTC 用 SDP 指纹绑定，不依赖 CA
DTLS Cookie禁用中风险 — 易受放大攻击
5
SRTP 加密 — 媒体数据机密性与完整性
AES-128-CTR 加密 + HMAC-SHA1-80 认证 + 滑动窗口重放防护
▶
overhead 原始 RTP: [Header 12B][Header Ext ~8B][Payload] SRTP: [Header 12B][Header Ext ~8B][Encrypted Payload][Auth Tag 10B] SRTCP: [Header 8B][Encrypted Payload][Index 4B][Auth Tag 10B]
重放攻击防护：滑动窗口（默认 128 packes），seq < 左边界丢弃，窗口内重复丢弃
6
SDP 指纹 — 端到端证书绑定
信令通道传递 SHA-256 指纹，DTLS 握手时独立验证
▶
WebRTC 不依赖 PKI/CA 信任链，而是通过信令通道（已受 TLS 保护）传递证书指纹：

1. 信令通道受 TLS 保护，指纹在传输中不可被篡改
2. DTLS 握手时，对端发送证书
3. 本地计算对端证书的 SHA-256 摘要
4. 与 SDP 中的 a=fingerprint 比对，不匹配则拒绝

安全属性：攻击者即使能拦截 DTLS 并替换证书，也无法匹配 SDP 中绑定的指纹。SFU 只做媒体转发，无法篡改端到端指纹。
📋 安全链路端到端分析
✓信令窃听 — WSS (TLS 1.2 + ECDHE) 防护
✓信令篡改 — TLS + JWT HMAC 防护
✓伪造设备身份 — JWT sub + API Secret 防护
✓媒体窃听 — SRTP (AES-128-CTR) 防护
✓媒体篡改 — SRTP Auth (HMAC-SHA1-80) 防护
✓DTLS 中间人 — SDP 指纹绑定防护
⚠DTLS 放大攻击 — Cookie 未启用
⚠板端证书验证 — VERIFY_OPTIONAL
⚠JWT URL 泄露 — query string 传递
优先级建议工作量

P0启用 DTLS Cookie 防止放大攻击1 天
