回声的物理成因 — 从声学到信号

IPC 对讲时，App 端说话→IPC 的 Speaker 播放→声音在空间中传播→被同一个 IPC 的 Mic 采集回去→App 用户听到自己的回声。AEC 要做的就是在 Mic 信号中减去 Speaker 信号的延迟副本。

回声路径 — 一个线性系统

Speaker → 空气 → 墙壁反射 → Mic 的传递函数

从 Speaker 到 Mic 之间存在一个物理传递函数 h(n)——它将 Speaker 播放的信号 x(n) 转换为 Mic 收到的回声信号 d(n)：d(n) = x(n) ∗ h(n)（∗=卷积）。h(n) 包含了：直达声（Speaker→Mic 直线路径，延迟最短 ~1ms）、早期反射（墙壁/桌面一次反射，延迟 5-30ms）、混响尾（多次反射叠加，持续 100-500ms）。
AEC 的目标：估计 h(n) 的近似值 ĥ(n)→从 Mic 信号 y(n) 中减去 x(n)∗ĥ(n) → 得到纯净的近端语音。

回声路径的长度 (Tail Length)

滤波器阶数 = 采样率 × 回声持续时间

AEC 的自适应滤波器需要覆盖整个回声持续时间——如果滤波器太短，只能消除直达声和早期反射，混响尾仍然存在。
滤波器长度计算：采样率 16kHz × 回声持续 200ms = 3200 阶。每阶一个系数（通常是 16-bit 定点或 float32）→ 3200 × 4B = 12.8KB 滤波器状态——内存开销可接受。
IPC 场景：Speaker 和 Mic 距离很近（同在一个机身内）→直达声占主导→回声路径短（≤100ms）→ 1600 阶足够。手机外放→Mic 距离远+反射多→需要 3200-6400 阶。

非线性成分 — AEC 的天敌

Speaker 失真 + 外壳振动 → 线性模型不成立

AEC 基于线性系统假设——回声 = x(n) ∗ h(n)。但现实中 Speaker 的非线性失真（喇叭在大音量时的谐波失真——导致 AEC 无法消除的信号分量）和 外壳/结构振动（声音引起 IPC 外壳共振→机械振动传导到 Mic——不是通过空气传播的声波→不是 Speaker 信号的线性函数）产生非线性回声。
解决方案：AEC 之后的 残余回声抑制 (RES) 用频谱减法进一步压制非线性残留。但 RES 会损伤近端语音——过度使用会导致声音失真。最好的方案是硬件设计：Speaker 和 Mic 之间加物理隔离（橡胶减振垫、声学腔体隔离）。

自适应滤波 — NLMS 算法

AEC 的核心数学引擎——持续跟踪回声路径的变化（人走动、设备位置变化、温度变化都改变 h(n)），用梯度下降逐步逼近真实回声路径。

LMS — 最小均方算法

梯度下降 · 每采样点更新 · O(N) 复杂度

核心公式（每个采样点迭代）：
① 滤波输出：ŷ(n) = wᵀ(n)·x(n)——用当前滤波器系数 w 和远端参考信号 x 计算估计回声
② 误差：e(n) = y(n) − ŷ(n)——真实 Mic 信号减去估计回声 = 残差（理想情况下=近端语音）
③ 系数更新：w(n+1) = w(n) + μ·e(n)·x(n)——沿梯度方向微调每个系数
μ (步长) 是关键参数：μ 太大→收敛快但不稳定（可能发散）；μ 太小→收敛慢（回声消除不干净）。μ 的取值受 x(n) 的信号功率影响——大声时 μ 需要调小，小声时 μ 需要调大。

NLMS — 归一化 LMS

用信号功率归一化步长 · 解决 LMS 的 μ 依赖问题

NLMS 改进：
w(n+1) = w(n) + μ / (||x(n)||² + ε) · e(n)·x(n)
分母 ||x(n)||² 是远端信号当前帧的能量——信号强时自动减小步长、信号弱时自动增大步长。ε（如 1e-6）是防止除零的微小常数。
μ 的取值范围：0 < μ < 2。（理论：μ=1 最速收敛，μ>2 发散）。实践中 μ 取 0.1~0.5——更保守但更稳定。
NLMS 的收敛速度：约 200-500ms 适应一个新的回声路径（如手机从桌面拿到耳边）——取决于 μ 和滤波器长度。收敛期间回声消除不干净——用户能听到短暂的回声。

// NLMS 自适应滤波器的逐帧实现 (16kHz, 10ms 帧 = 160 samples) void nlms_process_frame(float *far_ref, float *near_mic, float *output, float *w, int N, int frame_len, float mu) { for (int n = 0; n < frame_len; n++) { // 1. 构建远端信号缓冲区 (长度为 N 的滑动窗口) memmove(x_buf, x_buf + 1, (N - 1) * sizeof(float)); x_buf[N - 1] = far_ref[n]; // 2. 滤波: 估计回声 = w · x (点积) float echo_est = 0.0f; for (int k = 0; k < N; k++) echo_est += w[k] * x_buf[k]; // 3. 残差 = Mic 信号 - 估计回声 output[n] = near_mic[n] - echo_est; // 4. 计算远端信号能量 ||x||^2 float x_power = 1e-6f; for (int k = 0; k < N; k++) x_power += x_buf[k] * x_buf[k]; // 5. NLMS 系数更新: w = w + (mu / x_power) * e * x float step = mu / x_power * output[n]; for (int k = 0; k < N; k++) w[k] += step * x_buf[k]; } }

双讲检测 — 防止滤波器发散

AEC 最大的工程挑战不是收敛——是在近端说话时停止更新滤波器系数。如果不停止，NLMS 会试图把近端语音也"消除"掉→滤波器发散→回声反而变大。

什么是双讲 (Double-Talk)？

远端和近端同时说话 → NLMS 的 e(n) 包含近端语音 → 不能用于更新

正常模式（仅远端说话）：Mic 信号 = 回声 + 背景噪声。e(n) = Mic − 估计回声 ≈ 背景噪声——没有近端语音分量→可以安全更新系数。
双讲模式（双方同时说话）：Mic 信号 = 回声 + 背景噪声 + 近端语音。e(n) = Mic − 估计回声 ≈ 近端语音 + 噪声。NLMS 的更新项 μ·e(n)·x(n) 中包含了近端语音→滤波器系数被近端语音错误地推离正确值→发散→回声消除失效→甚至回声比不开 AEC 更严重。
DTD 的任务：实时检测是正常模式还是双讲模式→双讲时冻结滤波器系数（μ=0）→双讲结束恢复更新。

DTD 算法 — Geigel 和相关法

能量比 + 相关性 + 频谱差异 三指标投票

方法 1: Geigel 检测器（经典，简单）：比较 Mic 信号峰值和远端信号峰值。如果 Mic >> 远端 → 近端可能在说话。但远端信号通过回声路径后可能衰减——衰减后的远端信号可能小于 Mic 信号 → 误判。
方法 2: 相关性检测：计算 e(n)（残差）与 x(n)（远端）的互相关系数。仅远端说话时：残差小→相关性强。双讲时：残差包含近端语音→与远端相关性减弱。
方法 3: 频谱差异：比较 Mic 频谱和远端频谱。如果 Mic 频谱中有频段能量远大于远端（且不是回声路径能解释的）→近端说话。
工程实践：三种方法组合投票——各算各的，至少两种判定双讲才冻结。单纯的 Geigel 误判率太高。

工程实现 — 频域 AEC + 非线性处理

时域 NLMS 简单但计算量大（1600 阶 × 16000Hz = 25.6M 乘加/秒——对小 CPU 有压力）。频域 AEC 大幅降低计算量并更容易集成残余回声抑制。

频域分块 LMS (FDAF)

FFT → 频域滤波 → IFFT · 计算量 O(N log N)

将时域卷积（O(N²)）转换为频域逐元素乘法（O(N log N)）：远端信号 x(n) → FFT → X(f)；滤波器系数 h(n) → FFT → H(f)；估计回声 = IFFT(X(f) · H(f))。
Partitioned Block FDAF：将长滤波器分成多个短块（如 64 阶 × 50 块 = 3200 阶）——每块独立做 FFT 和更新。计算量从 25.6M 降到 ~3M 乘加/秒（~8x 降低）。
WebRTC AEC3 使用此架构——支持最长 500ms 回声尾（16kHz 下 8000 阶）。

残余回声抑制 (RES)

频域 Wiener 滤波 · 消除 AEC 之后的残留回声

AEC 输出 e(n) 中仍然有残余回声（线性部分因为滤波器未完全收敛，非线性部分因为 Speaker 失真）。RES 对 e(n) 做短时傅里叶变换 →每个频点计算增益 G(f)：
G(f) = max(0, (|E(f)|² − α·|Ŷ(f)|²) / |E(f)|²)
其中 |Ŷ(f)|² 是估计回声的功率谱，α 是过减因子 (1.0~2.0)。G(f) 接近 0→该频点主要是回声→抑制。G(f) 接近 1→该频点主要是近端语音→保留。

舒适噪声生成 (CNG)

RES 过度抑制导致"死寂"→填充自然噪声

当 AEC+RES 过度抑制时→输出信号几乎静音→听者感觉"断了"（dead silence）。舒适噪声生成在静音段填充模拟的背景噪声（功率与原始背景噪声匹配）——让听者感知到连接仍然活跃。
实现：在无近端语音时测量 Mic 的背景噪声功率 σ²_noise；RES 处理后如果输出功率过低→注入高斯白噪声（功率=σ²_noise）。

⚠ AEC 的延迟要求 — 为什么必须 < 10ms
AEC 的远端参考信号 x(n) 和 Mic 信号 y(n) 必须严格时间对齐——如果远端信号比 Mic 中的回声晚到 AEC（比如远端参考经过了额外的音频处理延迟 20ms）→ AEC 看到的 x(n) 与实际回声的 x(n−20ms) 不匹配→NLMS 试图用"未来的 x 消除过去的回声"→完全无法收敛。
经验法则：远端参考路径和 Mic 路径的处理延迟差必须 < 1ms。IPC 的音频链路（Speaker 播放 + Mic 采集）总延迟必须 < 10ms——超出此限制→AEC 性能急剧下降→需要额外的延迟估计和补偿机制。

WebRTC AEC3 — 工业级实现

WebRTC 内置的 AEC3 (AEC version 3) 是目前最广泛部署的开源 AEC 实现。IPC 对讲功能直接用它——比自己从头实现可靠 100 倍。

AEC3 的核心特性

Partitioned Block FDAF · 多频带 · 延迟自适应

① 频域分块：滤波器 64 阶/块，最多 50 块 = 3200 阶 @16kHz = 200ms 回声尾覆盖 ② 多频带处理：将频谱分为多个子带（0-4kHz, 4-8kHz）——各子带独立配置滤波器长度（低频回声尾长→更多块；高频回声尾短→更少块）③ 自动延迟估计：通过互相关检测远端参考和 Mic 信号之间的初始延迟——自动补偿音频链路的固定延迟（如蓝牙 Speaker 的 ~30ms 延迟）④ 内置 RES + CNG：不需额外模块。

AEC3 的配置参数

采样率 · 滤波器长度 · 延迟 · 抑制等级

采样率：8k/16k/32k/48kHz 自适应——宽带(16k)语音质量好，超宽带(32k)更自然。
回声尾长度：默认 200ms (3200阶@16k)——可通过 Config::filter.refined.length_blocks 调整。
初始延迟：config.filter.default_delay = 0（无额外延迟时设 0；有蓝牙 Speaker 时设 ~30ms）。
抑制等级：config.filter.config_change_duration_blocks——控制 AEC 和 RES 之间的平衡。保守（更多 AEC,更少 RES）=声音更自然但可能残留回声；激进（更多 RES）=回声更彻底但可能损伤语音。

AEC3 的 API 调用

每帧 10ms (160 samples @16k) 调用一次

WebRTC 的 AEC3 封装在 AudioProcessing 模块中——配置后逐帧送入音频数据即可：
① AudioProcessing::Config.echo_canceller.enabled = true
② 每帧调用 ProcessStream(&near_mic_frame)——输入近端 Mic 信号→AEC 处理后输出
③ 但需要提前告诉 AEC 远端参考：ProcessReverseStream(&far_ref_frame)——送入即将播放的远端音频——AEC 在内部对齐两个流并执行回声消除。
顺序重要：先调 ProcessReverseStream（送入远端参考），再调 ProcessStream（处理 Mic 信号）——确保 AEC 内部有对齐的参考。

IPC 实战 — 对讲 AEC 的完整音频链路

IPC 相机的双向对讲需要AEC + ANS（降噪）+ AGC（自动增益）三者协同——AEC 只是其中一环。

IPC 对讲的完整音频处理链 (近端)

Mic → ANS → AEC → AGC → 编码 → 网络发送

处理顺序有讲究：
① ANS (降噪) 必须放在 AEC 之前：AEC 的自适应滤波器对平稳噪声不敏感（噪声不可预测→不会被滤波器"学习"）。但噪声会干扰 DTD 的判断（噪声大时 Geigel 容易误判）→先降噪再 AEC→DTD 更准确。
② AEC：消除远端回声。远端参考 = 从网络收到的即将播放的音频。
③ AGC (自动增益)：AEC 之后的信号可能有音量波动（近端人距离 Mic 远近不同）→AGC 将音量归一化到目标电平。
④ 编码：AAC-LC / Opus 编码→RTP 发送。
关键：远端参考必须从网络接收后、Speaker 播放前的节点取——不能在 Speaker 播放后取（那时已经被 DA 转换和功放失真污染了）。

IPC AEC 的硬件约束与调优

CPU 预算 · 延迟预算 · 声学设计

CPU 预算：AEC3 @16kHz 宽带的典型 CPU 占用 ~5-10 MIPS（ARM Cortex-A）。频域分块模式大幅降低计算量——IPC SoC (如 Rockchip RV1106 单核 A7) 可以运行。如果 CPU 不够→降采样到 8kHz 窄带→计算量减半但语音质量下降。
延迟预算：10ms 音频帧 + AEC 处理 < 5ms + 编码 < 10ms → 总延迟 < 25ms——可接受。如果总延迟 > 50ms → 对讲体验差。
声学设计比算法更重要：Speaker 和 Mic 之间用橡胶减振垫隔离（阻断结构传导）+ 密封腔体（减少内部声短路）+ Speaker 背面填充吸音棉（减少外壳共振）→ 将回声路径的非线性分量降低 10-15dB→AEC 的残余回声大幅减少→RES 可以更轻→语音更自然。
