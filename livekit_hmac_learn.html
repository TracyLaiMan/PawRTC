HMAC：为什么哈希不够？
哈希只能验证"数据是否被改动"，但任何拿到数据的人都能算一样的哈希。
HMAC 加了一层密钥——只有持有密钥的人才能生成正确的认证码。

HMAC vs 数字签名：有什么区别？
都能防篡改、验来源。但 HMAC 是对称的（同一密钥签和验），数字签名是非对称的（私钥签、公钥验）。
项目里 JWT 和 ICE 用 HMAC，DTLS 用数字签名——为什么？
HMAC
对称密钥认证
  • 密钥：同一密钥，签发方和验证方都知道
  • 速度：快（只需哈希计算，微秒级）
  • 方向：1对1 或 1对多（一方签，多方用同一密钥验）
  • 前提：双方必须有安全渠道共享密钥
  • 本项目：JWT（后端和SFU共享API Secret）、ICE（双方共享ice-pwd）、SRTP（双方共享DTLS导出的密钥）
ECDSA 数字签名
非对称密钥签名
  • 密钥：私钥签名，公钥验证——公钥可以公开
  • 速度：慢（椭圆曲线运算，毫秒级，比HMAC慢1000倍+）
  • 方向：1对任意（任何人都能用公钥验证）
  • 前提：需要一个可信渠道分发公钥（CA证书链或SDP指纹）
  • 本项目：DTLS证书（ECDSA P-256自签名），签名用于握手完整性，公钥通过SDP指纹验证
核心区别：HMAC 适合"双方已经共享了密钥"的场景（JWT、ICE、SRPT 都是如此），因为它更快更简单。数字签名适合"需要让不特定的人验证"的场景（DTLS 证书需要被任何一个 WebRTC 客户端验证）。

为什么 JWT 不用 ECDSA？因为 JWT 只需要让 SFU 验证——后端和 SFU 都知道 API Secret，没必要引入公钥体系增加复杂度。
为什么 DTLS 不用 HMAC？因为 DTLS 握手时双方还没有共享密钥——握手的目的正是协商密钥。必须用非对称签名来保证握手完整性。

HMAC 在 LiveKit IPC 中的三处应用
同一个密码学原语，在不同的安全层扮演不同的角色。
Layer 1 · JWT 签名
HMAC-SHA256 — 身份证明的防伪标记
后端用 API Secret 作为密钥签发 JWT。SFU 用同一个密钥验证签名。

角色：身份认证——确认"你是谁"、"你能做什么"。
密钥来源：后端和LiveKit Cloud共享的 API Secret，通过环境变量/配置文件注入，从不通过网络传输。
验证方：SFU（只有一方需要验证）。
有效期：JWT 的 exp 字段控制，通常几小时。
// LiveKitSignaling.cpp:386 JWT.sign({iss: "API_KEY", sub: "device_001", exp: now + 3600, video: {canPublish: true, canSubscribe: false}}, "API_SECRET") // ← HMAC-SHA256 签名
Layer 3 · ICE MESSAGE-INTEGRITY
HMAC-SHA1 — 连通性检查的哨兵口令
STUN Binding Request 用 ice-pwd 计算 MESSAGE-INTEGRITY。接收方用 SDP 中的 ice-pwd 重新计算并比对。

角色：消息完整性认证——确认"这条STUN消息确实来自持有ice-pwd的对端"。
密钥来源：SDR 中交换的 ice-pwd，每次 ICE 会话随机生成（16-32字节），通过 TLS 信令通道传输。
验证方：双方互相验证（对称的）。
有效期：一个 ICE 会话（几分钟到几小时）。
注意：用 SHA1 而非 SHA256 是因为 RFC 5389 规定 STUN 的 MESSAGE-INTEGRITY 必须用 HMAC-SHA1。
// Stun.c:70 — 序列化 STUN 包时 STATUS serializeStunPacket(pPacket, password, passwordLen, TRUE, // generateMessageIntegrity (HMAC-SHA1) TRUE, // generateFingerprint (CRC32) ...)
Layer 6 · SRTP Auth Tag
HMAC-SHA1-80 — 视频封条的防拆封印
每个 RTP 包附加 10 字节 auth tag（= HMAC-SHA1 截断到 80 bit）。接收方先验证 auth tag，不符则丢弃。

角色：数据完整性+来源认证——确认"这个视频包没被篡改，且来自持有SRTP密钥的对端"。
密钥来源：DTLS 握手通过 TLS PRF 从 master_secret 导出，每方向 16 字节 key + 14 字节 salt = 30 字节。
验证方：双方互相验证（对称的）。
有效期：一个 DTLS 会话（直到重新握手）。
截断到80bit：完整 HMAC-SHA1 是 160 bit，SRTP 只取前 80 bit 以节省带宽（每包10字节 vs 20字节）。80 bit 的伪造概率为 2^-80 ≈ 10^-24，在实际中足够安全。
// SrtpSession.c — 加密 + 认证 srtp_protect(ctx, rtp_buf, &len); // 1. IV = (SSRC || ROC || SeqNum || 0x000000) XOR salt // 2. AES-128-CTR 加密 payload // 3. HMAC-SHA1 over 整个 RTP 包，取前 10 字节作 auth tag

密钥生命周期：从生成到销毁
HMAC 的安全性完全取决于密钥的安全性。这三处HMAC的密钥从哪里来？怎么用？何时过期？如何销毁？
关键风险点：
1. JWT 的 API Secret 是长期密钥——如果泄露，攻击者可以签发任意设备的 token。建议：定期轮换，使用短有效期 exp，监控异常 token。
2. ICE 的 ice-pwd 是短期临时密钥——每次 ICE 重启都换新的，泄露后影响仅限单个会话。
3. SRTP 密钥是会话级临时密钥——来自 ECDHE 协商，会话结束后密钥从内存清除（前向保密）。即使录制的流量被保存，以后也无法解密。

板端特有风险：/dev/urandom 启动早期熵不足 → 生成的 ice-pwd 或 DTLS 临时密钥可能可预测。量产前必须在启动各阶段验证熵质量。
