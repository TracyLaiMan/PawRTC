抖动 (Jitter) 的本质

ΔT到达 ≠ ΔT发送

网络不是等时的。每包走不同的路由、不同的排队深度，导致包间延迟抖动。JitterBuffer 用额外的缓冲延迟换取均匀的播放间隔。

核心矛盾

延迟 vs 丢包 vs 卡顿

缓冲太小 → 网络抖动时缓冲区干涸 (underrun) → 卡顿。缓冲太大 → 端到端延迟增加 → 实时性差。自适应调节是这个不可能三角的求解器。

WebRTC 双轨实现

NetEQ (音频) + VideoJB (视频)

音频用 NetEQ (GIPS 遗产，2007 年收购): 含 DSP 级别的 PLC 和时间拉伸。视频用独立 VideoReceiveStream + RtpVideoStreamReceiver 内置的帧级缓冲。

1. JitterBuffer 本质 — 为什么要"没事找事"地延迟？

包到达时间的随机性 → 需要缓冲区吸收抖动 → 引入额外延迟 → 需要自适应的缓冲长度

没有 JitterBuffer

收到就播 → 严重卡顿

包 0 在 t=20ms 到达 → 立即解码播放。包 1 晚到了 10ms (t=50ms 才到) → 解码器缺数据 → 静音/卡帧 → 用户感知到"卡"了。
PLC (Packet Loss Concealment) 可以掩盖一两个缺包，但频繁的 underrun 会让音视频不可接受。
结果: 零缓冲延迟 → 零抗抖动能力 → 网络稍微波动就卡。

有 JitterBuffer

预先缓冲 N 毫秒 → 平滑播放

包 0 到达 → 不立即播放，先放入缓冲区等 40ms。
包 1 晚了 10ms → 没关系，缓冲区内还有足够数据 → 播放线程以均匀 20ms 间隔取包 → 解码 → 用户感知流畅。
代价: 端到端延迟 = 网络延迟 + 缓冲延迟 (40ms)。
核心问题: 缓冲设多长？太长延迟高，太短扛不住抖动。

2. 抖动模型 — 网路如何让包"乱来"

包间延迟 · 网络排队 · WiFi 重传 · 蜂窝切换 — 抖动的物理来源

固定间隔发送 → 随机间隔到达

Sender: ΔT = 20ms · Receiver: ΔT = 15~45ms

发送端用匀速的编码节奏 (如 50fps → 20ms/帧) 发送 RTP 包。每个包经过：WiFi 竞争窗口等待 (0~10ms) → 路由器排队 (0~50ms) → ISP 骨干网路由变化 (±5ms) → 蜂窝基站切换 (±100ms) → 到达时间间隔不再是常数。

Jitter 的统计定义

RFC 3550: interarrival jitter
// RFC 3550 Section 6.4.1 J(i) = J(i-1) + (|D(i-1,i)| − J(i-1)) / 16 // D(i,j) = 到达时间差 − 发送时间差 D(i-1,i) = (R_i − R_{i-1}) − (S_i − S_{i-1}) = (R_i − S_i) − (R_{i-1} − S_{i-1}) // 这是一个指数加权移动平均 (EWMA) // 平滑因子 α = 1/16 → 对新样本较迟钝 // 反应的是抖动的"长期趋势"，非瞬时值

不只是迟到 — 四种异常

JitterBuffer 需要同时处理

1. 抖动 (Jitter): 包间隔不均匀 → 缓冲吸收
2. 乱序 (Reordering): 包 5 比包 4 先到 → 按 seq 重排
3. 丢包 (Loss): 包永远不到 → PLC 或 NACK/FEC 恢复
4. 突发 (Burst): 连续丢 N 包 → 缓冲区快速干涸
一个好的 JitterBuffer 必须同时处理这四种情况，且在它们之间自适应切换策略。

3. 音频 JitterBuffer — NetEQ 深度剖析

WebRTC 音频抖动缓冲的"大脑" — MCU 控制 + DSP 信号处理 + 自适应决策

3.1 NetEQ 架构 — 三个缓冲区 + 一个决策器

Packet Buffer

RTP 包仓库 — 按 seq 排序的优先队列

PacketBuffer: 存储未解码的 RTP 包 (含 payload + timestamp + seq)。
插入: 按 seq 号二分查找 → 乱序插入自动排序。
查询: GetNextPacket() 返回最早未播放的连续包。
Flush 策略: seq 号间隔过大 (gap > threshold) 或 timestamp 过期 → 丢弃旧包，避免播放延迟无限增长。
大小: 动态，由 MCU 模块根据网络条件增减。

Decoded Buffer

已解码 PCM — 播放的直接来源

DecodedDatabase: 存储已解码的 PCM 音频帧。
当 Packet Buffer 积累足够的包后，解码线程取出并解码 → 放入 Decoded Buffer。
播放线程以 10ms 为单位 从这里取数据 → 交给 AudioDevice 播放。
关键: 已解码 buffer 不能太大（内存）也不能太小（抗抖动差）。NetEQ 根据网络条件决定何时解码、解码多少。

Algorithm Buffer

DSP 算法层 — 扩展/压缩/生成

当 Decoded Buffer 缺数据 (underrun) 时，Algorithm Buffer 介入：
PLC (丢包隐藏): 用上一帧的 LPC 系数 + 基音周期来外推合成缺失的音频 → 用户听不出丢包。
时间拉伸/压缩: WSOLA 或基音同步重叠相加 (PSOLA) → 在 buffer 太长时加速播放 (听不出音调变化)，太短时减速。
CNG (舒适噪声): 长期无包时生成背景噪声代替静音。

操作模式触发条件DSP 动作用户感知
NORMALbuffer 在目标范围内正常解码播放正常
EXPANDbuffer 干涸，无包可用PLC 外推合成 (基于 LPC + pitch)听不到丢包 (短时)
ACCELERATEbuffer 过长 (积累太多)PSOLA 时间压缩 (加速 ~3%)几乎无感知
PREEMPTIVE_EXPANDbuffer 变短 (即将干涸)PSOLA 时间拉伸 (减速 ~3%)几乎无感知
MERGEEXPAND 后有新包到达在 PLC 合成和新解码间平滑过渡无缝衔接

3.2 MCU (Micro Controller Unit) — 决策大脑

目标延迟 (Target Delay)

Buffer 想维持在多少 ms —— 核心控制变量

MCU 维护一个 target_level_ms — 期望的缓冲水平。这是 JitterBuffer 唯一需要调节的控制变量。
过大: 积累过多 → ACCELERATE → 压缩播放
过小: 面临干涸 → PREEMPTIVE_EXPAND → 拉伸播放
刚好: NORMAL → 正常解码
Target 的设定基于：网络抖动估计 + 丢包率 + 历史 underrun 频率。
初始值: 通常在 30~100ms，根据首个包的到达延迟设定基线。

延迟估计 — 如何知道 target 该设多少？

基于百分位 (percentile) 的抖动估计

简单方法: target = J(i) × k + base (J 是 RFC 3550 估计，k 是安全系数 2~4)。问题：J 是 EWMA，反应迟钝。
NetEQ 方法: 维护最近 N 个包的 IAT (Inter-Arrival Time) 直方图 → 取 95th percentile → target = P95(IAT) × α + base。
Percentile 的好处: 自动过滤极端异常值 (99.9th 可能是 WiFi 扫描造成的 200ms 突发，不应据此把 target 设到 200ms)。
自适应收敛: 如果过去 10s 内出现 2 次 underrun → target += 10ms。如果过去 30s 无 underrun → target -= 5ms。

时间伸缩 — 加减速的 DSP 原理

PSOLA: Pitch-Synchronous Overlap and Add

加速 (ACCELERATE): 在音频信号中找到重复的基音周期 → 把相邻两个周期重叠相加 → 信号缩短 ~1 个周期但音调不变。
减速 (PREEMPTIVE_EXPAND): 在基音边界复制插入一个周期 → 信号延长但音调不变。
极限: 单次加速/减速不超过 ~30% (3ms 在 10ms 帧上)，否则人耳可感知失真。多次累积操作后，总伸缩量可逐步达到 ~100ms。
为什么不做简单重采样: 重采样会改变音调 → 人耳极其敏感 → 绝对不可接受。

PLC — 丢包隐藏的数学

Linear Predictive Coding + Pitch Estimation

步骤 1: 对上一帧 PCM 做 LPC (10~16 阶) → 得到声道模型的滤波器系数。
步骤 2: 估计基音周期 (pitch lag) — 计算自相关函数，找除 0 外的最大峰值 → pitch = 50~400 Hz。
步骤 3: 用 LPC 合成滤波器 + 基音周期脉冲激励 → 外推生成缺失帧的 PCM。
步骤 4: 缺失超过 1 帧时，逐步衰减增益 → 60ms 后静音。
限制: 适用于语音 (LPC 假设人声道是线性时不变滤波器) — 对音乐/背景噪声效果差很多。这也是为什么 Opus 内置 PLC 效果比 NetEQ 的通用 PLC 好。

4. 视频 JitterBuffer — 帧级缓冲与参考帧管理

VideoReceiveStream · FrameBuffer · RtpFrameReferenceFinder · 帧完整性检查

4.1 视频与音频的根本区别

4.2 视频 JitterBuffer 处理流水线

RTP 包 → 完整帧的组装

RtpFrameReferenceFinder · PacketBuffer

视频帧通常超过 MTU → 一个帧被分割为多个 RTP 包 (FU-A / FU-B 分片)。
1. 收包: RTP 包到达 → 按 SSRC + seq 存入 PacketBuffer。
2. 拼帧: RtpFrameReferenceFinder 检查: seq 连续 + Marker bit (最后一包) + Start bit (第一包) → 拼成完整帧。
3. 参考链: 检查该帧依赖的参考帧 (通过 RTP generic descriptor / FrameDependency 扩展头) → 确认参考帧已解码。
4. 可解码帧: 自身完整 + 参考帧就绪 → 放入 Decodable Frame Queue。

帧级延迟管理 — 视频独有的"等还是跳"

min_playout_delay / max_playout_delay

每一帧有一个 Render Time (来自 RTP timestamp)，JitterBuffer 据此决定：
帧提前到达: Render Time 还未到 → 等待。
帧准时到达: Render Time 到了且可解码 → 取出解码。
帧迟到: Render Time 过了还没完整 → 决策点: 继续等 (增加延迟) 还是跳过 (丢帧 + 请求 I 帧/keyframe)?
min_playout_delay: 最小缓冲延迟 (即使网络完美也保持此缓冲) → 防止瞬时抖动。
max_playout_delay: 最多等多久 → 超过则丢帧。两者均来自远端 SDP/RTCP 协商或本地自适应设定。

Keyframe Request — 视频的最后防线

PLI / FIR — 请求发送端生成 I 帧

当参考帧丢失且重传也失败时，解码器完全无法解码后续任何 P/B 帧 → 画面冻结。
PLI (Picture Loss Indication): RTCP Feedback 消息 → 告诉发送端"我的解码器卡住了，寄个 I 帧过来"。
FIR (Full Intra Request): 更强力的 PLI 变体 — 要求发送端丢弃当前编码状态，从 I 帧重新开始。
触发时机: 连续丢帧 > 阈值 (通常 3~5 帧) → JitterBuffer 发出 PLI → 等待 I 帧到达 → 丢弃中间所有不可解码的帧 → 从 I 帧恢复播放。

视频 JitterBuffer 的收敛算法

TargetDelay 的 AIMD (加性增/乘性减)

视频使用类似 TCP 的 AIMD 策略调节目标延迟：
增窗 (Additive Increase): 每 1 秒无丢帧 → target += 5ms (缓慢增加缓冲以应对潜在抖动)
减窗 (Multiplicative Decrease): 发生丢帧 → target = target × 0.9 (快速降低延迟)
实际实现更复杂:
inter_frame_delay_variation 估计 → 计算 P95 百分位 → target_delay = max(min_playout_delay, P95 × factor)。
配合 Google Congestion Control (GCC) 的延迟估计 → 当网络变差时 Buffer 反而要缩小 (因为延迟已经很高了，再缓冲更糟)。

5. 延迟估计 — 如何"感知"网络状态

KALMAN 滤波 · 百分位估计 · EWMA · 丢包率 · 过冲/干涸计数 — 多信号融合决策

Kalman Filter — 时延估计的数学最优解

WebRTC 音频 jitter 估计的默认算法

状态模型: 往返时延 (RTT) + 时钟偏置 (offset)。
观测模型: 每个 RTP 包的 (发送时间, 到达时间) 对 → 观测到的单向延迟。
Kalman 优势: 自动分离真实延迟变化和时钟漂移（收发两端的时钟频率可能有 ppm 级别差异，导致观测延迟线性漂移）。
滤波输出: 当前网络的最小延迟 (基线) + 抖动方差 (σ²) → target = base_delay + k × σ。
参数 k: 决定激进程度。k=2 → 约 95% 包能赶上播放 (假设正态分布)。k=3 → 约 99.7%。

Min-Max 滤波器 — 简单有效的工业替代

WebRTC 视频 jitter 估计使用的方法

滑动窗口: 维护最近 N 个帧 (或 N 秒) 的到达延迟。
最小延迟: 滑动窗口内的 min(arrival_delay) — 认为是网络的"物理极限延迟" (无排队时延)。
抖动估计: current_jitter = P95(arrival_delay) - min_delay。
target = min_delay + P95_jitter × safety_margin
为什么不用 mean：均值掩盖了分布的尾部。用户感知差来自尾部的大延迟，不是均值。
P95 vs P99: P99 更保守 (抗抖更强) 但延迟更高。WebRTC 默认倾向 P95。

6. 自适应缓冲算法 — 求解"不可能三角"

Buffer 长度如何实时调整 — 延迟 · 丢包率 · underrun 频率三重约束下的最优解

AIMD — 音频 JitterBuffer 的经典算法

Additive Increase / Multiplicative Decrease

增窗 (AI): 每个正常播放周期 → target += kI × packet_len
(典型 kI = 1/500 → 每秒增加 0.1ms 级别)
减窗 (MD): 每次 underrun → target = target × kD
(典型 kD = 2.0 → target 翻倍)
问题: 减窗太快 (翻倍) → 延迟爆增 → 之后需要很长时间的 AI 才能降回来。
改进: 区分软 underrun (丢 ≤ 2 包) 和硬 underrun (丢 > 2 包) → 软 underrun 时 target += 5ms (不翻倍)，硬 underrun 时翻倍。

决策树方法 — 视频 JitterBuffer 的多信号融合

根据多种状态组合做出决策

视频 JitterBuffer 不使用简单的 AIMD，而是信号融合决策树:
输入: underrun 频率、丢包率、解码延迟、播放延迟、buffer 占用率、RTT。
规则:
• 无 underrun + buffer_occupancy > 80% → target -= 10ms (缓冲太多)
• underrun + RTT 低 → target += 20ms (网络突发，增加缓冲)
• underrun + RTT 高 → target += 5ms (网络已经拥堵，慎加延迟)
• 连续 underrun 3 次 → target *= 1.5 (紧急扩张)
• buffer_occupancy < 20% + 无丢包 → target -= 2ms (轻微回收)
