WSS 信令通道

在两台设备建立媒体连接之前，必须先通过信令通道交换身份、连接参数和密钥材料。
这个通道就是 WSS — WebSocket over TLS，它是整条信任链的地基。

DNS 解析
pawrtc-qbbhiyec.livekit.cloud → CDN IP
浏览器/板端通过 DNS 获取 LiveKit Cloud CDN（如 Cloudflare）的 IP 地址。DNS 本身可被 DNS 劫持，但后续 TLS 会验证服务器身份。
TCP 连接
三次握手建立可靠通道
经典 TCP 三次握手：SYN → SYN-ACK → ACK。为后续的 TLS 握手提供可靠传输基础。DTLS 不需要这步（基于 UDP）。
TLS 1.2 握手
ECDHE + AES-128-GCM 建立加密通道
密码套件：TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256
• ECDHE → 前向保密 (PFS)
• RSA → 服务器证书签名算法（LiveKit Cloud 用的是 CA 签发证书，不是自签名）
• AES-128-GCM → 加密 + 认证一体化
• ALPN = http/1.1 → 强制不走 H2
WebSocket 升级
HTTP/1.1 101 Switching Protocols
在已加密的 TLS 通道内，发送 HTTP 升级请求：
GET /rtc?access_token=eyJ...&protocol=17
Upgrade: websocket
Sec-WebSocket-Protocol: livekit-protocol

服务器返回 101 Switching Protocols，此后连接变为双向 WebSocket 帧。
应用层信令
Join/Offer/Answer/Trickle/Leave
板端发送 JoinRequest (含 JWT)，服务器回复 JoinResponse (含 TURN 凭证)。随后交换 SDP offer/answer 和 ICE 候选。所有消息都在 TLS 加密通道内传输。

ALPN 陷阱：为何必须强制 HTTP/1.1

板端连接 LiveKit Cloud 时遇到 HTTP 200 而非 101——根因是 CDN 协商了 H2，而 WebSocket 只能走 HTTP/1.1。

Bug 现象
CDN 协商 H2 → WS 升级失败
1. 板端 TLS 握手时未指定 ALPN
2. Cloudflare CDN 默认协商 h2
3. 板端发送 WS 升级请求
4. H2 不支持 WS 升级（RFC 8441 未普及）
5. CDN 返回 HTTP 200（当普通请求处理）
6. 板端期待 101，连接无效

致命陷阱：LWS 不报错，只是静默收到 HTTP 200 然后终止。日志显示 ESTABLISHED_CLIENT_HTTP: status=200 而非 CLIENT_ESTABLISHED。
修复方案
显式声明 ALPN = http/1.1
在 LWS 上下文和连接信息中同时设置：

ctxInfo.alpn = "http/1.1";
ccInfo.alpn = "http/1.1";

这强制 TLS 握手时 ALPN 只协商 http/1.1，CDN 不再选择 H2。

为什么两处都要设？ LWS 文档说 ctxInfo 是全局默认，ccInfo 是连接级覆盖。实测只有 ccInfo 设置才能生效——ctxInfo 的 alpn 在某些 LWS 版本被忽略。
// LiveKitSignaling.cpp — ALPN 修复（三重 bug） // Bug 1: LWS 交叉编译模式下 LWS_WITH_ALPN 宏可能未定义 // 解决：CMakeLists.txt 强制 -DLWS_WITH_ALPN=1 // Bug 2: ctxInfo.alpn 在某些 LWS 版本不生效 // 解决：ccInfo.alpn 也要显式设置 // Bug 3: ctxInfo.client_ssl_ca_filepath 指向不存在的路径 // → LWS 静默跳过证书验证 → 等同于 VERIFY_OPTIONAL // 最终正确配置： ctxInfo.options = LWS_SERVER_OPTION_DO_SSL_GLOBAL_INIT; ctxInfo.client_ssl_ca_filepath = "/etc/ssl/certs/ca-certificates.crt"; ctxInfo.alpn = "http/1.1"; ccInfo.protocol = "livekit-protocol"; ccInfo.alpn = "http/1.1"; ccInfo.origin = "https://" + hostname; // Cloud CDN 要求

VERIFY_OPTIONAL：信令通道的致命漏洞

跳过 TLS 证书验证 = 允许中间人 → 篡改信令 → 全链路崩溃。

1
板端 TLS 设为 VERIFY_OPTIONAL（因为 CA 证书不完整）。
后果：板端不验证 Cloud 的证书是否合法。
2
中间人在网络层拦截 TLS 握手，伪造服务器证书。板端不拒绝 → TLS 连接与攻击者建立，而非 Cloud。
3
攻击者解密并检查所有信令消息——包括 a=fingerprint:sha-256（DTLS 证书指纹）和 a=ice-pwd（ICE 认证密码）。
攻击者用自己的证书指纹替换 SDP 中的指纹，再转发给真正的 Cloud。
4
Cloud 返回 SDP answer，攻击者同样篡改指纹后转发给板端。双方都以为在和对方建立 DTLS，实际都与攻击者握手。
5
攻击者持有双方 DTLS 协商的密钥 → 解密所有 SRTP 媒体数据。设备摄像头画面、麦克风音频全部泄露。
起因：仅仅是跳过了 TLS 证书验证。
修复路径：
1. 推送完整 ca-certificates.crt 到板端 /etc/ssl/certs/
2. 修改 Tls_mbedtls.c：无条件使用 MBEDTLS_SSL_VERIFY_REQUIRED
3. 同时调用 mbedtls_ssl_set_hostname() 验证证书 CN/SAN
4. 确保 client_ssl_ca_filepath 路径存在且包含 Cloud 的 CA 链

IPC 设备信任链：断裂与修复

浏览器端信任链完整，但板端从起点就断裂了——CA 门槛没过，后续分层皆不可信。

浏览器端 — 信任链完整
CA 根证书 → → → SRTP 安全
✓CA 根证书（OS/浏览器预装）
✓验证 WSS 服务器证书 → TLS 加密通道
✓SDP 指纹安全传递（信令通道保护）
✓DTLS 握手 + 指纹比对
✓SRTP 媒体加密 ✓
板端 — 信任链断裂
VERIFY_OPTIONAL → 全链失效
✗无 CA 根证书库 / VERIFY_OPTIONAL 跳过验证
✗WSS 服务器身份未确认 → 中间人可伪造
✗SDP 指纹在伪造通道中可被篡改
✗DTLS 指纹比对无意义（比对值本身就是假的）
✗SRTP 密钥泄露 → 媒体裸奔
关键洞察：自签名证书不是"凭空可信"的——它的信任间接锚定在 CA 体系上。SDP 指纹经由 TLS 保护的信令通道传递，而 TLS 通道的安全性依赖 CA 验证。一旦 CA 验证被跳过，整条信任链从根基处倒塌。
当前状态：client_ssl_ca_filepath 指向不存在的路径 → LWS 静默跳过证书加载 → 等同于 VERIFY_OPTIONAL。修复：推送完整 ca-certificates.crt 到 /etc/ssl/certs/，并修改 Tls_mbedtls.c 为 MBEDTLS_SSL_VERIFY_REQUIRED。

双向认证（mTLS）与证书角色

单向 TLS 只验证服务器身份；mTLS 让服务器也能验证设备——JWT 可被盗走，私钥偷不走。

🔍
我信任谁 — 验证工具
ca-bundle.crt
内容：多个 CA 的公钥（公开信息，任何人都能下载）
用途：验证对方发来的证书是否由这些 CA 签发
类比：你手持一张"哪些公安局可信"的名单

⚠️ 把这份名单交给对方，对方只能说"嗯，这些 CA 我也认识"，但完全无法证明你的身份
🪪
我是谁 — 身份凭证
device.crt + device.key
内容：设备公钥 + 设备身份信息 + 私有 CA 签名

// IPC 设备证书存储布局 /etc/ssl/certs/ ├── ca-bundle.crt # 公共 CA 根证书（验证 LiveKit 服务器） ├── linkric-private-ca.crt # 私有 CA 根证书（P2P 场景验证其他设备） └── device.crt # 本设备证书（私有 CA 签发，含 SN/MAC） /etc/ssl/private/ └── device.key # 本设备私钥（永远不离开设备，不通过网络传输） // LWS 代码配置 ctxInfo.options = LWS_SERVER_OPTION_DO_SSL_GLOBAL_INIT; ctxInfo.client_ssl_ca_filepath = "/etc/ssl/certs/ca-bundle.crt"; // 验证服务器 // mTLS 额外配置 SSL_CTX_use_certificate_file(ctx, "/etc/ssl/certs/device.crt", SSL_FILETYPE_PEM); SSL_CTX_use_PrivateKey_file(ctx, "/etc/ssl/private/device.key", SSL_FILETYPE_PEM); // 嵌入式设备三大现实挑战 // 1. 无时钟 → TLS 证书有效期验证失败 // 修复：启动时先 NTP 同步，同步前拒绝 WSS // 2. 无 CA 根证书库 → Buildroot 最小系统不带 ca-certificates // 修复：编译时打包 ISRG Root X1 + 私有 CA // 3. 设备证书部署 → 出厂时如何预置不同的设备证书 // 修复：工厂流水线签发 + 写入固件分区；或首次启动注册协议
mTLS vs JWT 不是替代关系：
mTLS 解决 "这个连接来自合法设备吗"（传输层，连接建立时验证）
JWT 解决 "这个设备有权加入这个房间吗"（应用层，连接建立后验证）
两者互补：mTLS 防伪造设备，JWT 防未授权访问。当前项目只有 JWT，缺少传输层的设备身份验证。

项目中的应用

LWS 交叉编译、nanopb 协议、三重 TLS bug 修复——信令层是项目最复杂的部分之一。

LWS 交叉编译
libwebsockets for MIPS
LWS 是板端唯一的 WebSocket 客户端库。交叉编译需要：
• cmake -DLWS_WITH_SSL=1 -DLWS_WITH_ALPN=1 -DLWS_WITHOUT_SERVER=1
• 链接 mbedTLS（而非 OpenSSL，减少二进制体积）
• 关键：LWS_WITH_ALPN=1 默认可能未开启→导致 alpn 字段被忽略→H2 bug

二进制大小：~200KB (strip 后)，远小于 WebSocket++ + Boost.Asio 的方案
nanopb 协议
Protobuf 信令消息
LiveKit 使用 Protobuf 定义信令消息，板端用 nanopb（轻量 Protobuf C 库）编解码：
• livekit_signaling.proto → livekit_signaling.pb.c/h
• 消息帧格式：[1B msgType][varint msgSize][protobuf payload]
• msgType 映射：1=Join, 2=Answer, 3=Offer, 4=Trickle, 5=Leave...

代码路径：LiveKitSignaling.cpp → handleSignalResponse()
// handleSignalResponse — 解码消息类型 auto msgType = static_cast<SignalRequest>(buf[0]); if (msgType == SignalRequest::JOIN) { JoinResponse jr; pb_decode(&stream, JoinResponse_fields, &jr); // jr.ice_servers — TURN 凭证在此下发 // jr.subscriber_primary — 决定谁是 Publisher/Subscriber }
