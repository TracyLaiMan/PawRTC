ECDHE 密钥交换

两个陌生人，在不安全信道上，协商出只有彼此知道的共享密钥。
窃听者看到所有通信内容，却算不出密钥——这就是 Diffie-Hellman 的魔力。

板端 (Client)
窃听者只能看到 A 和 B
服务器 (Server)

正向容易，逆向极难

ECDHE 的安全性基于椭圆曲线离散对数问题 (ECDLP)：给定 G 和 G*a，求 a 极其困难。
就像知道 2 和 8，算 2^3=8 很容易，但只知道 2 和 8 反推指数 3 就难多了。

正向 — 毫秒级
标量乘法
已知私钥 a 和基点 G：
A = G * a

椭圆曲线上做标量乘法：把 G 加自身 a 次。
即使 a 是 256-bit 整数，Double-and-Add 算法只需 256 次倍点 + ~128 次加点。
逆向 — 宇宙级时间
离散对数
已知公钥 A 和基点 G：
求 a = log_G(A)

P-256 曲线上最佳算法约需 2^128 次运算。
全球最强超算每秒 10^18 次，需要 10^21 年——
比宇宙年龄 (1.4×10^10 年) 还长 10^11 倍。
↓

ECDHE 五步握手

双方各生成临时密钥对 → 交换公钥 → 各自算出相同的共享密钥。
信道上传输的只有 A 和 B，没有 a 或 b。

↓

前向保密 (PFS)

ECDHE 中的 "E" = Ephemeral（临时的）。每次会话用新密钥对，用完丢弃。
即使事后长期私钥泄露，历史会话也无法解密。

会话建立
板端生成临时私钥 a₁，服务器生成 b₁，双方算出 S₁ = G*a₁*b₁。
SRTP 用 S₁ 导出的密钥加密媒体。
会话结束
双方丢弃 a₁ 和 b₁。S₁ 不再存在于任何地方。
录制的密文流量安然保存在攻击者硬盘上。
服务器被攻破
攻击者获取了服务器的 ECDSA 长期私钥。
能伪造新会话的证书，但拿不到 a₁ 或 b₁。
尝试解密历史流量
攻击者只有 A₁=G*a₁ 和 B₁=G*b₁。
没有 a₁ 或 b₁ → 无法计算 S₁ → 无法推导 SRTP 密钥 → 密文不可解密。
结论：ECDHE 保证了前向保密——历史会话的安全性不受未来密钥泄露的影响。这就是 WebRTC 强制使用 ECDHE 的原因。
对比：如果用 RSA 密钥交换（非 Ephemeral），预主密钥用服务器公钥加密。服务器私钥泄露后，攻击者可以解密所有 histórico 会话——没有前向保密。

项目中的应用

DTS 握手中使用 ECDHE 协商 SRTP 密钥，板端使用 P-256 曲线。

DTLS 握手 — ECDHE
ECDHE 在 DTLS 握手中的位置
DTLS 握手的 ServerKeyExchange 和 ClientKeyExchange 消息承载 ECDHE 公钥。

密码套件 TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256 的含义：
• ECDHE — 密钥交换方式（临时椭圆曲线 Diffie-Hellman）
• ECDSA — 签名算法（用 ECDSA 证书签名的 ServerKeyExchange）
• AES_128_GCM — 加密算法（仅 TLS 信令通道，SRTP 用 CTR）
• SHA256 — PRF 和 MAC 算法

板端代码位置：LkTransport.c — createDtlsSession()
// LkTransport.c — 密码套件配置 static const UINT8 ciphersuites[] = { 0xC0, 0x2B // TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256 }; // curve: mbedtls_ecp_group_id = MBEDTLS_ECP_DP_SECP256R1 (P-256) // 每次连接重新生成 ECDHE 临时密钥对 → PFS ✓
密钥导出链路
ECDHE → master_secret → SRTP 密钥
ECDHE 共享密钥 S = G*a*b 经过 TLS PRF 派生为 master_secret：
master_secret = PRF(S, "master secret", client_random + server_random)

然后 master_secret 再通过 PRF 导出 SRTP 密钥材料：
key_material = PRF(master_secret, "EXTRACTOR-dtls_srtp", randoms)

最终切成 4 段：client_key(16B) → server_key(16B) → client_salt(14B) → server_salt(14B)

关键点：server_random 通过 TLS 保护的信令通道传输（WSS），攻击者无法篡改 randoms。

DHE vs ECDHE 对比

都是 Diffie-Hellman 密钥交换，但基于不同的数学难题。ECDHE 用更短的密钥达到同等安全强度。

总结：ECDHE 是现代 WebRTC 的唯一选择。更短的密钥、更快的握手、更低功耗——对嵌入式设备尤其重要。板端 P-256 的 ECDHE 运算在 MIPS 上约需 50-100ms，完全可接受。项目不应考虑回退到 DHE。

ECDHE + ECDSA 缺一不可

ECDHE 协商密钥解决"怎么保密"，但无法证明"你是谁"。
ECDSA 签名验证解决"你是谁"，但无法生成共享密钥。
两者在 DTLS 握手中配合才能建立端到端安全通道。

ECDHE — 管保密 双方各生成临时密钥对，交换公钥后协商出共享密钥 S。
每次连接重新生成 → 前向保密。
但没有签名能力，无法防中间人替换公钥。
ECDSA — 管信任 服务器用长期私钥对握手消息签名，客户端用公钥验签。
确认对方身份 → 防中间人。
但不能协商对称密钥，无法加密后续媒体流。
