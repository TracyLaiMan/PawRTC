LiveKit IPC 方案详细设计文档

基于 LiveKit SFU 的 IPC 产品详细设计方案，覆盖实时视频推流、本地录像远程回看、弱网优化等核心场景。

LiveKit WebRTC IPC mipsel JY1C
版本: v1.0 日期: 2026-05-20 读者: 后台/App/嵌入式开发 状态: 设计阶段

1. 文档概述

1.1 目的

本文档为基于 LiveKit 的 IPC（IP Camera）产品制定详细设计方案，覆盖实时视频推流、本地录像远程回看、弱网优化等核心场景，使后台和 App 开发人员能明确量产落地路线。

1.2 范围

  • 板端：JY1C（Ingenic T41, mipsel）嵌入式 Linux 设备
  • 传输：LiveKit SFU（Cloud 或自建）
  • 客户端：App（iOS/Android）/ 浏览器
  • 场景：实时视频预览、双向语音对讲、本地录像远程回看

1.3 术语定义

术语说明
SFUSelective Forwarding Unit，选择性转发服务器，只转发不编解码
PCPeerConnection，WebRTC 连接单元
Publisher PC设备端推流连接（设备 offer → 服务器 answer）
Subscriber PC设备端收流连接（服务器 offer → 设备 answer）

2. 系统架构

2.1 端到端架构

媒体 SRTP
信令 WebSocket
控制 DataChannel
TURN 中继
REST Token

所有媒体流经 SFU 转发，非端到端 P2P。SFU 只做选择性转发，不编解码，延迟低。LiveKit 采用 subscriber_primary 模式：信令连接后先处理 Subscriber PC（收 offer、发 answer），然后创建 Publisher PC。

2.3 SFU 转发机制

  • 设备发送一路 H.264 码流，SFU 转发给所有订阅者
  • 每个订阅者可独立请求不同质量层（Simulcast 层级）
  • 当前方案设备只发单层（无 Simulcast），SFU 直转

视频管线各节点详解

CMOS 传感器 → ISP

CMOS 传感器采集原始光信号，输出 Bayer 格式数据。ISP（Image Signal Processor）执行 Demosaic（去马赛克）、降噪、白平衡、锐化等处理，输出 YUV420P 格式帧。

FrameSource → H.264 硬编

IMP FrameSource (FS_CHN=0) 从 ISP 获取 YUV 帧，送入硬件编码器 (ENC_CHN=0)。编码器输出 H.264 Baseline NAL 单元，码率 1500kbps CBR，每 30 帧一个 IDR 关键帧。

RTP 打包 → SRTP → UDP

H.264 NAL 按 RFC 6184 打包：SPS+PPS 用 STAP-A 聚合，超过 MTU 的 NAL 用 FU-A 分片。打包后经 SRTP 加密（AES-128-CTR + HMAC-SHA1，密钥由 DTLS 握手导出），通过 UDP socket 发送到 SFU。支持 TURN relay 中继。

音频管线各节点详解

I2S 数字麦克风 → ALSA

I2S 数字麦克风硬件采集，16kHz/16bit/单声道。ALSA 驱动层将 I2S 数据读取为 PCM 帧，通过 snd_pcm_readi() 提供给应用层。采集缓冲区大小通常为 20ms 的一帧（320 samples × 2 bytes = 640 bytes）。

Opus 编码

将 16kHz PCM 输入送入 Opus 编码器，内部重采样到 48kHz 输出。码率 32kbps CBR，帧长 20ms（960 samples @48kHz）。启用 useinbandfec=1（带内前向纠错），丢包时可利用前一帧冗余信息恢复。编码输出每帧约 80 字节。

RTP 打包 → SRTP → UDP

Opus 帧按 RTP 打包：Payload Type 111，时间戳间隔 960（48kHz × 20ms），序列号递增。SRTP 加密复用视频的 DTLS 会话密钥（BUNDLE 机制，同一 5-tuple）。UDP 发送与视频共用同一 socket（rtcp-mux + BUNDLE 端口复用）。

音频与视频的 BUNDLE 复用

SDP 中通过 a=group:BUNDLE 0 1 将视频 (mid=0) 和音频 (mid=1) 绑定到同一传输。所有 RTP/RTCP 包走同一 UDP 端口，同一 DTLS 会话，同一套 SRTP 密钥。优点：减少 NAT 状态数、简化 ICE 候选收集。注意：需要通过 RTP 头的 SSRC 区分音视频包。

5. 本地回看实现方案

5.1 回看架构

核心思路：回看 = 切换视频源 + 远程控制 + 弱网自适应，复用实时推流的全部通道。

优势与局限

优势

  • 零额外存储成本（视频在设备本地）
  • 复用 LiveKit 信令 + SFU 通道，无需 HLS/DASH
  • App 端无需额外播放器，同一渲染组件

局限

  • 设备必须在线
  • 不能精确到帧拖拽（WebRTC 1-3s 缓冲）
  • 同时只能推一个源（回看/实时互斥）

seek 实现 
bool Mp4FileSource::seek(uint64_t targetMs) {
    AP4_UI32 targetSampleIndex = 0;
    AP4_Cardinal sampleCount = videoTrack_->GetSampleCount();
    AP4_UI32 timescale = videoTrack_->GetMediaTimeScale();
    AP4_Timestamp targetTs = targetMs * timescale / 1000;

    for (AP4_Ordinal i = 0; i < sampleCount; i++) {
        AP4_Sample sample;
        videoTrack_->ReadSample(i, sample);
        if (sample.IsSync() && sample.GetCts() >= targetTs) {
            targetSampleIndex = i;
            break;
        }
    }
    m_sampleIndex = targetSampleIndex;
    requestNextIdr = true;
    frameBuffer.clear();
    refillBuffer();
    return true;
}

6.3 RTP 协议

H.264 RTP 打包（RFC 6184）

  • STAP-A：将 SPS + PPS + 小 NAL 打包到单个 RTP 包
  • FU-A：大 NAL 分片，每个 RTP 包 ≤ MTU（默认 1200，弱网建议 1000）
  • marker bit：帧的最后一个 RTP 包 marker=1，SFU 依赖此检测帧边界
