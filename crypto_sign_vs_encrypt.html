问题引入 双向运算 攻击推演 ECDSA 差异 ECDHE·G 的奥秘 角色分配 总结

为什么不能"签名用对方公钥"？

加密 = 公钥加密，私钥解密  |  签名 = 私钥签名，公钥验签
为什么角色不能互换？如果有人提出"用对方公钥来签名"会怎样？

▽

RSA：数学上确实可以"双向"

RSA 的 e 和 d 在数学上完全对称 —— 幂运算不分先后。但这不是问题的全部。

📐 RSA 核心数学：e 和 d 互为逆运算
正向：加密 → 解密（公 → 私）
C = Me mod n  →  M = Cd mod n
公钥 (n, e) 加密 → 私钥 (n, d) 解密
经典用法：任何人用公钥加密，只有私钥持有者能解密
反向：签名 → 验签（私 → 公）
S = Md mod n  →  M = Se mod n
私钥 (n, d) 签名 → 公钥 (n, e) 验签
签名用法：只有私钥持有者能签，任何人用公钥验证
⚠ 关键发现：两个方向的数学运算是 完全对称 的！
e 和 d 没有"方向性"—— 只是 人为规定了哪个公开、哪个保密。
如果强行把上面两个公式的 e/d 角色互换，数学上完全成立 —— 但安全模型彻底崩塌。
▽

如果"签名用对方公钥"—— 安全模型崩塌

下面推演：如果把签名定义为"用对方的公钥对消息做运算，对方用私钥验证"，会发生什么。

⚠ 攻击场景推演

Bob 想给 Alice 发一条签名消息："请转账 $100 给 Bob"

👨‍💻
Bob
持有：Bob 私钥
→
✍️
签名 = 用 Alice 公钥
← 这是"错误定义"
→
👩‍💻
Alice
验证 = 用 Alice 私钥
但是…… Alice 的公钥 是 公开的！
💀
攻击者 Mallory
也有 Alice 的公钥
→
✍️
伪造签名
同样用 Alice 公钥
→
📨
发送："转账 $9999 给 Mallory"
签名"有效"！
🚫 结论：Alice 用私钥验证 → 验证通过！
Alice 以为消息来自 Bob，实际上来自 Mallory。
因为 任何人都有 Alice 的公钥，所以 任何人"签名"出来的结果都一样——
签名彻底失去了"证明发送者身份"的意义。

ECDSA：根本不存在"加密"方向

上面说的是 RSA。但在 WebRTC/DTLS 实际使用的 ECDSA 中，
签名根本不是"反向加密"——它是一种完全不同的数学运算。

🔄
RSA
数学结构：大整数模幂
M^e mod n

加密和签名是同一个数学运算的正反方向
→ 可以"双向"用
⛓️
ECDSA
数学结构：椭圆曲线点乘
k × G = (x, y)

签名输出 (r, s) 是"零知识证明"
→ 根本不存在加密功能
📌
WebRTC 分工
ECDSA：只做签名验签
ECDHE：只做密钥协商
AES-GCM：只做数据加密

三项各司其职，不可互换

ECDHE 握手与 G 的奥秘

G 是什么？为什么 k × G 正着算是"点乘"，反着算是"无解"？
ECDHE 5 步握手中，双方如何在窃听者眼皮底下协商出只有他们知道的密钥。

📍
G = Generator（基点）
椭圆曲线上一个公开固定的点。
secp256r1 的 G 坐标写在标准文档里，全世界所有人用同一个 G。
就像数轴上的"1"——它是所有运算的起点。
🔃
单向陷门：k × G
k × G = P  →  正向计算 极快（微秒级）
从 P 反推 k  →  计算上不可行
这是 椭圆曲线离散对数难题（ECDLP）
ECDHE 的全部安全性都建立在这个假设之上。
🔄
循环群：自我闭合
G → 2G → 3G → ... → nG → G
所有的 k × G 都在同一个曲线上，
永远走不出去，也永远不会重复（直到 n）。
这保证了密钥空间的 封闭性和均匀分布。
1
Server 生成私钥
选取随机大整数 a（保密）
a ∈ [1, n-1]
2
Client 生成私钥
选取随机大整数 b（保密）
b ∈ [1, n-1]
3
交换公钥
A = a×G B = b×G
在公开信道传输
窃听者可看到 A、B
4
各自计算共享密钥
Server: S = a×B
Client: S = b×A
都到达 ab×G 这个点
5
密钥派生
S 的 x 坐标 → HKDF
→ AES 对称密钥
窃听者：有 A、B 但无法算出 ab×G
G 的角色总结：
G 是椭圆曲线上的公共原点，所有人都从同一个 G 出发。
a × G = A（私钥 × G = 公钥）：正向秒算，反向破解需要宇宙年龄。
双方各走半程（a×G 和 b×G），交换后再走对方的半程（a×(b×G) 和 b×(a×G)），
最终在 同一个点 ab×G 会合 —— 窃听者只能看到中间点 A 和 B，永远到不了终点。
这就是 ECDHE 能在公开信道上协商密钥的根本原因。

正确的角色分配：私钥做的事 vs 公钥做的事

核心原则：私钥做的事必须是"只有这个身份能做"的事。

🔑 加密 / 解密
加密用 对方公钥
→ 任何人都能给你发加密消息
→ 门槛低 = 好（反正只有你能解密）

解密用 自己私钥
→ 只有你能看到内容
→ 门槛高 = 必要（机密性的根基）

保护目标：机密性 — 数据不给别人看
✍️ 签名 / 验签
签名用 自己私钥
→ 只有你能产生这个签名
→ 门槛高 = 必要（签名的全部意义）

验签用 签名者公钥
→ 任何人都能验证是不是你签的
→ 门槛低 = 好（任何人都能确认真伪）

保护目标：认证性 + 完整性 — 证明你是谁、内容没被改
⚠️ 反面教材：如果反过来
签名用 对方公钥
→ 公钥是公开的 → 任何人都能"签名"
→ 签名失去唯一性，形同虚设

验证用 对方私钥
→ 只有对方能验证 → 我竟然无法自己验证
→ Bob 发的消息，只有 Alice 能"确认"

结论：完全不可用
💡 一句话心法
私钥做的事 = 只有我能做的
→ 解密：只有我能看
→ 签名：只有我能签

公钥做的事 = 任何人都能做的
→ 加密：谁都能给我发
→ 验签：谁都能验证我的身份

本质：私钥 = 门槛，公钥 = 无门槛

完整心智模型

加密与签名的区别不是"用哪把钥匙"，而是保护什么和信任谁。

加密 保护的是 内容（confidentiality），信任的是 对方的接收能力。
签名 保护的是 身份（authenticity），信任的是 自己的唯一性。

私钥 = 只有我拥有 → 签名用私钥（证明我是我）、解密用私钥（只有我能看）
公钥 = 所有人都拥有 → 验签用公钥（任何人都能验证）、加密用公钥（任何人都能发送）
