理解证书、签名与信任
六层安全模型建立在三个密码学原语之上 —— 不懂它们，就无法理解"为什么自签名证书也能可信"

非对称密钥对

一把钥匙两把锁 —— 公钥公开，私钥保密
每人生成一对密钥：私钥只自己持有，公钥可以任意分发。
• 加密：别人用你的公钥加密 → 只有你的私钥能解密
• 签名：你用私钥签名 → 所有人用你的公钥验证
常见算法：ECDSA P-256（WebRTC 选用）、RSA-2048（HTTPS 常用）

数字签名

亲笔签名 —— 不可伪造，不可抵赖
发送方对消息做 哈希 → 用 私钥加密哈希 → 得到签名。
接收方：用公钥解密签名 → 自己算哈希 → 比对是否一致。
一致 = 消息确实来自私钥持有者，且未被篡改。
LiveKit 项目中的签名：ECDSA 签 DTLS 证书、HMAC 签 JWT / STUN

X.509 数字证书

派出所开的身份证 —— 证明"公钥属于谁"
证书 = 公钥 + 身份信息 + 签发者签名 三合一。
有了证书，你不需要亲自认识对方，只需要信任签发者。
证书内容示例：Subject=*.livekit.cloud、Issuer=Let's Encrypt、公钥、有效期、签名

CA 证书颁发机构

公安局 —— 你信它，它签的证你就信
CA 是被广泛信任的第三方，用自己的 私钥 给服务器证书签名。
浏览器 / 操作系统预装了 根证书（CA 的公钥），形成信任链：
根证书 → 中间 CA → 服务器证书
HTTPS 用 CA：浏览器自动验证。WebRTC 用 自签名：没 CA，靠 SDP 指纹。

自签名证书

自己给自己开证明 —— 没有第三方背书
自己生成密钥对 → 自己签发证书 → 没有 CA 参与。
任何人都能造一个自签名证书，所以 单独看证书无法判断真伪。
WebRTC 的做法：通过 TLS 信令通道 传递证书的 SHA-256 指纹，握手后比对。
⚠️ 板端 SSL_VERIFY_OPTIONAL 跳过了指纹比对 → 中间人风险

证书指纹

身份证的防伪码 —— 一个指纹对应一张证书
对整个证书内容做 SHA-256 哈希，得到一串十六进制摘要。
任何一位改动 → 指纹完全不同（雪崩效应）。
SDP 中的指纹格式：sha-256 79:5A:3B:...
两条路径比较：① TLS 信令传的指纹 ② DTLS 握手收到的证书算出的指纹

六大密码学基石
六层安全模型依赖这些底层原语 —— 理解它们才能理解安全设计
🔐
HMAC
防伪封印 —— 任何篡改都会被察觉
用在 L1 JWT 签名、L3 STUN 完整性、L6 SRTP 认证
📦
AES 对称加密
铁盒子 —— 只有持钥匙的人能打开
用在 L6 SRTP 媒体加密（AES-CTR）、L2 TLS 批量加密（AES-GCM）
🗝️
非对称加密
两把钥匙 —— 公钥公开上锁，私钥私密开锁
用在 L4 DTLS 证书传递公钥、L5 SDP 指纹绑定公钥身份
🤝
ECDHE 密钥交换
暗号协商 —— 公开讨论却只有彼此能算出密钥
用在 L4 DTLS 握手、L2 TLS 握手 —— 提供前向保密
🪪
ECDSA + 证书
身份证 —— 证明"我就是我"
用在 L4 DTLS 证书签名、L5 SDP 指纹绑定 —— 通过 SHA-256 指纹防中间人
🔍
SHA-256 哈希
数字指纹 —— 任意输入压缩为固定摘要，不可逆、雪崩效应
用在 L5 SDP 指纹计算、ECDSA 签名前哈希、HMAC 底层引擎
