为什么需要 AGC？

麦克风距离 · 说话音量 · 网络会议

同一场景不同人说话音量可能差 30dB。AGC 自动将输入电平调整到目标范围，保证下游编码器 (Opus/SILK) 接收到最优电平时的信号——不会因太轻而量化噪声大，也不会因太响而削波。

AGC ≠ 压缩器

时间常数决定本质区别

AGC 的 Attack/Release 以数百毫秒到秒计 — 目标是让"整句话"听起来均匀。压缩器 (Compressor) 的 Attack/Release 以微秒到毫秒计 — 目标是控制单个瞬态峰值。WebRTC 同时有 AGC 和 Limiter。

三大核心指标

Target Level · Attack · Release

Target Level: 输出目标电平 (dBFS)。Attack Time: 信号变响后多快降低增益。Release Time: 信号变轻后多快提升增益。三者共同决定了 AGC 的"手感"——太快则呼吸效应，太慢则响应迟钝。

1. 信号基础 — 电平、分贝、响度的三角关系

理解 dBFS · RMS · Peak · 人耳响度感知 (等响曲线) — AGC 处理对象的物理定义

分贝 (dB) — 对数世界的标尺

人耳感知的是比值，不是绝对值

人耳可以感知 20µPa ~ 20Pa 的声压范围——跨度 1,000,000 倍。线性标尺不可用 → 必须用对数。
dBFS (dB Full Scale): 数字音频的绝对电平。0 dBFS = 满量程 (不能超过，超过即削波)。所有正常信号都是负值。
dBov: 相对于 overload point — 通话标准常用。
dB SPL: 声压级 (物理世界) — 0dB SPL = 人耳听阈，120dB SPL = 痛阈。
感知法则: +10dB ≈ 主观"响了一倍"；+3dB ≈ "刚能听出差异"。

三种电平度量 — Peak vs RMS vs Envelope

不同度量方式决定了 AGC 的行为
// Peak (峰值) — 10ms 帧内绝对值的最大值 Peak = max(|x[n]|), n = 0..N-1 Peak_dB = 20 × log₁₀(Peak) // RMS (均方根) — 反映信号"平均功率" RMS = √( 1/N × Σ x[n]² ) RMS_dB = 20 × log₁₀(RMS) // Crest Factor (峰值因子) = Peak/RMS // 语音: 12~18dB | 正弦波: 3dB | 白噪声: ~10dB // Envelope (包络) — 攻击快、释放慢的峰值跟踪 // 更接近人耳对响度的感知
🔑 Peak vs RMS — AGC 该用哪个？
Peak 检测: 防止削波——但语音 crest factor 高达 18dB，纯 peak 会导致 AGC 过度压低 → 大部分信号低于目标 10~15dB → 听感偏轻。
RMS 检测: 与响度感知更相关——但无法及时捕捉瞬态峰值 → 有削波风险。
最佳实践: RMS 做主控 + Peak 做安全限幅。RMS 决定增益值，Peak 仅用于检测是否需要硬性限幅 (limiter)。这正是 WebRTC AGC 的做法。

2. AGC 信号处理流水线 — 五个阶段详解

电平估计 → 增益计算 → 时间平滑 → 噪声门限 → 增益应用 — 一步步拆解

2.1 Stage ① — 电平估计 (Level Estimation)

滑动窗口 RMS

最常用的数字 AGC 电平估计方法

将输入 PCM 分成 N 个采样点的帧 (典型 10ms = 480 samples @ 48kHz)，计算每帧 RMS：
rms = sqrt( Σx[n]² / N )
优点: 简单、稳定、与响度相关性好。
缺点: 帧内时间分辨率低 (10ms → 无法捕获 < 10ms 的瞬态)。
优化: 用 重叠窗口 (50% overlap) → 每 5ms 就能更新一次电平估计，同时保持 10ms 的积分窗口长度。

指数平滑包络 (RC 滤波器)

模拟硬件的攻击/释放时间常数

不用固定帧，而是用两个不同时间常数的单极点 IIR 滤波器跟踪信号包络：
Attack (快): 当 |x[n]| > envelope[n-1] → α = α_attack (小，快速跟踪)
Release (慢): 当 |x[n]| < envelope[n-1] → α = α_release (大，缓慢衰减)
envelope[n] = α × |x[n]| + (1-α) × envelope[n-1]
这直接对应模拟 VU 表的指针行为——快升慢降。

K-weighted 滤波 — 模拟人耳

ITU-R BS.1770 响度计量标准

在做 RMS 之前，先通过K-weighting 滤波器:
高通: 38Hz 二阶高通 (滤除次声波/风噪/空调嗡声)
高频 shelf: +4dB @ 1.5kHz 以上 (补偿人耳对中高频更敏感)
这样做出来的电平估计更接近人耳实际感知到的响度，而不是物理声压。ITU BS.1770 的 LUFS 响度计量就是基于 K-weighted RMS。
WebRTC AGC 的数字模式使用了类似思路的预加重。

2.2 Stage ② — 增益计算 (Gain Computation)

// 核心公式 — 看似简单，细节都在曲线形状里 所需增益: G_dB = Target_dB − Input_dB // 但直接应用这个 G_dB 会有问题: // 1. 太轻的信号 → G_dB 过大 → 噪声也被放大 → 需要 Noise Gate // 2. 太响的信号 → G_dB 为负 → 需限制最大衰减量 → 需要 Limiter // 3. G_dB 剧烈变化 → 人耳感知到"呼吸声" → 需要平滑

压缩曲线 (Compression Curve)

增益不是线性的 — 它是一条分段曲线

Knee 下方 (低电平): 增益随输入升高而降低——小声时多放大，大声时少放大 → 压缩动态范围。
Knee 上方 (高电平): 增益固定为 0dB (1:1 直通) — 已经够响了，不需要再处理。
Noise Floor 以下: 增益设为 −∞ dB (静音) — 纯噪声，不放大。
Compression Ratio: 比如 2:1 — 输入增加 2dB 只让输出增加 1dB。
典型语音 AGC ratio: 3:1 ~ 10:1 (激进压缩动态范围，让音量更均匀)。

查表法 (LUT) — 避免实时计算对数

WebRTC 的做法: dB → gain 预计算
// 预计算增益表 (0~255 量化) for (int i = 0; i < 256; i++) { // i = 输入电平的量化表示 float input_dB = i_to_dB(i); float target_dB = compression_curve(input_dB); gain_table[i] = pow(10, (target_dB - input_dB) / 20); } // 运行时 O(1) 查表: gain_linear = gain_table[level_index]; // 好处: 无 log/pow 浮点运算 // 曲线形状可任意设计 (分段/多项式)

3. 时间平滑 — Attack · Hold · Release 三段式状态机

这 5 个参数决定了 AGC 的"手感"——太大则呼吸效应，太小则响应迟钝

Attack Time (起音时间)

输入突然变响 → 增益快速降低

信号从轻声变到大声时，AGC 多快降低增益。
典型值: 1~10ms (非常快)。
为什么这么快？因为大声信号可能立刻削波——必须在几个采样周期内把增益拉下来。
太快的问题: 增益变化跟不上波形 — 引入可听失真。需要微小平滑 (1~3ms 过渡)。

Hold Time (保持时间)

增益降低后保持多长时间不变

Attack 把增益拉低后，不要立即 Release — 先保持一段时间。
典型值: 50~200ms。
作用: 防止语音在音节间隙（如爆破音 p/t/k 之后的短暂停顿）时增益回升 → 下一个音节又被放大 → 增益再次降低 → "泵浦感"。
语音的音节间隔约 100~200ms → hold > 200ms 可跨过大部分音节间隙。

Release Time (释放时间)

输入变轻 → 增益缓慢提升

信号从大声变到轻声时，AGC 多快提升增益。
典型值: 200~1000ms (慢)。
为什么这么慢: 语音有自然的强弱起伏 — 如果增益回升太快，正常说话的句末降调会被放大 → 听感不自然。
太慢的问题: 说话者突然变小声 → 需要等很久才能恢复到正常音量。
自适应释放: 根据信号统计 (历史峰值 vs 当前) 动态调整释放速率。

⚠️ 呼吸效应 (Breathing / Pumping) — AGC 最常见的音质问题
现象: 背景噪声随语音的强弱而起伏——说话时噪声被压下去，停顿时噪声又升上来 → 感觉"有人在耳机里呼吸"。
根因: Release time 太短 + Noise Gate 阈值设定不当 → AGC 在语音间隙放大背景噪声。
解决方案: ① 增加 Hold Time → 跨过音节间隙 ② 降低 Noise Gate 阈值 → 噪声以下信号不放大 ③ 使用独立噪声估计器 (WebRTC AGC 的 noise estimation 子模块) → 增益计算时自动扣除噪声分量。

4. 噪声门限 (Noise Gate) — 别放大"不该放大"的东西

噪声估计 · 自适应阈值 · 软拐点 — 区分"轻声说话"和"背景噪音"的科学

固定阈值 vs 自适应阈值

噪声是会变的 — 阈值必须跟着变

固定阈值: 若 RMS < −50 dBFS → 静音。问题：安静房间里 −50dBFS 是正确的，但嘈杂咖啡厅里 −50dBFS 已经是有效语音了。
自适应阈值: 持续估计噪声底噪 (Noise Floor) → 阈值 = NoiseFloor + margin (典型 margin = 12~15dB)。
噪声估计方法:
• 最小值统计: 过去 N 秒内 RMS 的最小值 → 近似噪声底噪。
• 语音活动检测 (VAD): VAD=0 的帧 → 更新噪声估计。
• 分位数跟踪: 取 P10 (10th percentile) → 它就是噪声底噪 (统计上语音不太可能出现在 P10)。

软拐点 (Soft Knee) — 消除开关感

不要"啪"一下关掉，要平滑过渡
// 硬门限 (Hard Gate) — 有开关声 if (level < threshold) gain = 0; // 软门限 (Soft Gate) — 过渡区线性渐变 if (level < low_knee) gain = 0; else if (level < high_knee) gain = (level - low_knee) / (high_knee - low_knee); else gain = 1; // 典型: low_knee = NoiseFloor+6dB // high_knee = NoiseFloor+15dB // 在 9dB 过渡区内, 增益从 0 平滑上升到 1

5. 全数字 AGC — 定点实现与量化

对数域运算 · 定点 Q 格式 · dB → linear 转换 · SIMD 加速

为什么在对数域做 AGC？

乘法变加法 · 人耳对数感知 · 动态范围压缩

线性域: y[n] = x[n] × gain_linear
对数域: y_dB = x_dB + gain_dB
对数域运算把乘法变加法 → 在定点 DSP 上效率高得多。而且增益的平滑、压缩曲线、阈值判断都在对数域更自然。
格式: 常用 Q8.8 或 Q4.12 定点格式表示 dB 值，0.1dB 精度足够。
dB → linear 转换: 只在最后增益应用时做一次 pow10(dB/20) → 可用多项式近似 或查表 避免真正的 pow 计算。

定点 AGC 完整流程

Q15 PCM → Q8.8 dB → 查表 → Q15 PCM 输出
// 1. PCM → dB (对数域) — 用近似 log2 int16_t rms_q15 = calc_rms(pcm_in); int16_t level_db = log2_to_db(approx_log2(rms_q15)); // 2. 查压缩曲线表 → 目标增益 (dB) int16_t target_gain_db = compression_lut[level_db_idx]; // 3. Attack/Hold/Release 平滑 gain_db = smooth_gain(gain_db, target_gain_db, state); // 4. dB → linear (查表) int16_t gain_q15 = db_to_linear_lut[gain_db_idx]; // 5. 应用增益 pcm_out[i] = (int16_t)(((int32_t)pcm_in[i] * gain_q15) >> 15);

7. AGC 常见音质问题与解决方案

泵浦感 · 削波 · 噪声放大 · 过慢响应 — 根因分析与调优

问题听感描述根因解决方案
泵浦感 (Pumping)背景噪声随说话起伏，"耳机里有呼吸声"Release 太快 · 噪声门限太低 · Hold 太短↑ Hold 200→400ms · ↑ Release 500→800ms · 启用自适应噪声门限
削波 (Clipping)声音突然炸裂、刺耳的"咔哒"声Attack 太慢 · Limiter 未启用 · targetLevel 太靠近 0dBFS↓ Attack 5→1ms · 启用 Limiter · targetLevel ≤ −6dBFS
噪声放大安静时听到明显的"嘶嘶"底噪Noise Gate 阈值太低 · 噪声估计不准确↑ Noise Gate margin 10→15dB · 降低 compressionGaindB 上限
响应迟钝说话者音量变了但 AGC 很久才跟上Attack/Release 过长 · 电平估计窗口太大↓ Release 800→300ms · 用 5ms 重叠窗口替代 10ms 帧
过冲 (Overshoot)刚开口时前几个字特别响，然后才降下来Attack 不够快 · 初始增益太高↓ Attack 10→3ms · 初始化增益为保守值 0dB
音量不均匀不同人/不同距离音量差别仍然大Compression Ratio 太小 · targetLevel 设置不当↑ Ratio 3:1→8:1 · 确保 targetLevel 在编码器推荐范围内
编解码器兼容性编码器输入电平过高导致内部削波AGC 输出 > 编码器期望输入Opus: targetLevel ≤ −6dBFS · SILK: targetLevel ≤ −3dBFS
✅ AGC 调优黄金法则
1. 先测后调 — 录一段典型场景的原始音频，分析 RMS 分布，确定需要的增益范围。
2. 从保守开始 — targetLevel = −6dBFS、compressionGaindB = 6dB、Noise Gate = 15dB margin → 听效果 → 逐步调高。
3. 用 VAD 辅助 — 始终只在语音段做 AGC (VAD=1)，否则噪声估计会偏 → 底噪被放大。
4. 两个场景分别测试: 近距离 (10cm) 大声 + 远距离 (1m) 小声 — 看 AGC 能否把两者调到相近的输出电平。
5. AGC 之后接 Limiter — AGC 防止长期电平不均，Limiter 防止瞬时削波。两者不是替代关系。
6. 别在 AGC 之后再来一个 AGC — 如果编码器自带 AGC (如 Opus 的 OPUS_SET_GAIN)，确保只用一个。
