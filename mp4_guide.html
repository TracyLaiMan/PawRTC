MP4 容器格式深度解析

从 ISO BMFF 规范到 stbl 六表运作
从 FastStart 到 fMP4 流媒体片段

容器 ≠ 编码 —— 三层分离

MP4 是容器——不决定视频怎么压缩，只决定压缩后的数据怎么组织和索引

像素层 — 原始图像

YUV / RGB 采样数据

1080p@30fps，YUV420 每像素 12bit
一帧 ≈ 3.1 MB 原始数据
这是编码器的输入

编码层 — 压缩码流

H.264 / H.265 / AAC / Opus

将原始图像压缩为码流
一帧 H.264 ≈ 10-100 KB
压缩比 50-500×

容器层 — 组织索引

MP4 / MKV / TS / FLV

打包编码数据 + 建立索引
"第 5 秒的帧在文件第 98304 字节"
定位、同步、元数据

编码器只管压缩，容器管"怎么找到数据"
H.264 编码器输出一串 NAL Unit——没有时间戳、没有"哪段是音频哪段是视频"的标记。容器做的事：给这些数据加目录、加书签、加标签——让播放器能快速 seek 到任意秒数、区分音视频轨、知道何时播放哪帧。

第一层：MP4 文件的物理结构 —— 三块拼图

任何 MP4 文件都由三块构成：ftyp（格式声明）、moov（元数据索引）、mdat（编码数据）

ftyp — 文件类型标识

File Type Box · 始终在文件最开头

包含 major_brand（如 "mp42"）和兼容品牌列表（"isom""avc1"等）。播放器第一个就读取 ftyp——如果自己的解码器不支持该 brand，直接拒绝播放。大小通常 20-32 字节。

moov — 元数据容器

Movie Box · 文件的大脑

包含所有轨道的描述（编码格式、分辨率、帧率、时长）和完整的样本索引表（每个帧在 mdat 中的位置和大小）。播放器 seek 时只读 moov——不碰 mdat。通常 < 1MB。

mdat — 编码数据仓库

Media Data Box · 文件的正文

存放实际的编码数据——H.264 NAL Unit 和 AAC 帧按 Chunk 交错排列（V₁ A₁ A₂ V₂ …）。mdat 内部没有 Box 结构——它就是一个巨型 blob，完全依赖 moov 中的 stbl 表来索引数据位置。

关键认知：moov 是目录，mdat 是正文
就像一本书——moov = 目录（告诉你第 3 章在第 150 页），mdat = 正文（第 150 页的实际文字）。想跳到第 3 章？先查目录，再翻到对应页——不需要把整本书读一遍。

第二层：Box 的基本结构 —— 一切皆为 Box

MP4 文件 = 一棵嵌套的 Box 树。每个 Box 都有完全相同的 8 字节头部格式

Box 头部 = 8 字节

统一格式，极简设计
字节     含义
00-03    size     Box 总字节数（含自身这 4 字节）
04-07    type     四字符码（"moov""mdat""ftyp"…）
08-..    payload 子 Box（Container 型）或具体数据（Leaf 型）

如果 size=1 → 后 8 字节为 64 位扩展大小（>4GB 的 mdat）。size=0 → Box 延伸到文件末尾。

两种 Box 类型

Container（容器）vs Leaf（叶子）

Container Box：moov、trak、mdia、minf、stbl、edts——payload 全部是子 Box，自身不存业务数据。

Leaf Box：mvhd、tkhd、stsd、stts、stsz、stco——payload 是具体的元数据数值。

特殊 — mdat：payload 是原始编码数据（NAL Unit / AAC 帧），没有 Box 包装。由 moov 中的 stbl 表来索引每个 Chunk 的位置和大小。

Box 树的解析不需要状态机——这是 MP4 设计最优雅的地方
解析器只需要一个循环：① 读 4 字节 size ② 读 4 字节 type ③ 若 type 是已知的 Container → 递归解析子 Box；若 type 是已知的 Leaf → 按规范解释 payload ④ 跳过 size - 8 字节到下一个 Box。整个过程每个 Box 自描述。

第三层：moov 内部 —— 轨道体系

moov 内部按轨道（trak）组织——每个 trak 对应一个音频流或视频流

moov 的直接子 Box

moov ├── mvhd 影片全局信息 — 创建时间、时长、首选速率、时间标尺 ├── trak 视频轨道（至少一个） ├── trak 音频轨道（至少一个） └── udta 用户自定义数据（封面、版权、作者标签）

mvhd 中的 timescale 是整个影片的时间基准——所有时间戳都以 timescale 为单位。例如 timescale=1000 → 所有时间以毫秒计。

单个 trak（轨道）的内部层次

trak ├── tkhd 轨道属性 — 轨道ID、宽度、高度、音量 ├── edts 编辑列表 — 可选，用于裁剪或延迟 └── mdia 媒体信息容器 ├── mdhd 媒体时间标尺 + 时长 + 语言代码 ├── hdlr 处理器类型 — "vide"（视频）/ "soun"（音频） └── minf 媒体信息容器 ├── vmhd 视频媒体头（仅视频轨） ├── smhd 音频媒体头（仅音频轨） ├── dinf 数据信息 — 数据引用（dref） └── stbl ★ 样本表 — MP4 的心脏
为什么分两层时间标尺？—— mvhd.timescale vs mdhd.timescale
mvhd.timescale 是影片级时间（用于章节、字幕同步），通常设为 1000（毫秒）。
mdhd.timescale 是媒体级时间（用于精确帧定位），通常设为帧率的整数倍——如 30fps → timescale=30000 → 每帧=1000 单位 → 支持 1/30000 秒精度。

第四层：stbl — MP4 的心脏（六张索引表）

stbl 回答一个终极问题："要播第 3.5 秒的画面，数据在文件的哪个位置？"

stsd — 编码器配置

Sample Description · "用什么解码？"

stbl 的第一张表。告诉播放器：这是 H.264 还是 H.265？SPS/PPS 参数在哪？对 H.264 → 内含 avcC Box 存储 SPS 和 PPS（解码器初始化必需）。一个轨通常只有一种编码→stsd 通常只有 1 个 entry。

stts — 解码时间表

Time-to-Sample · "第 N 帧的 DTS 是？"

将每个样本（帧）映射到解码时间戳 (DTS)。采用压缩表格式存储："连续 50 个样本，每个持续 512 个时间单位"。恒定帧率时 stts 只有 1 条记录。

ctts — 合成时间偏移

Composition Offset · "B 帧的显示时间差多少？"

仅含 B 帧时才存在。B 帧的显示序 ≠ 解码序（PTS ≠ DTS）→ ctts 记录每个样本的 PTS−DTS 偏移量。无 B 帧时此表不存在。同样采用压缩表格式。

stsc — 样本→Chunk 映射

Sample-to-Chunk · "第 N 帧在哪个 Chunk？"

Chunk = 文件中连续存放的一组样本。stsc 格式："从 Chunk X 起，每 Chunk 含 Y 个样本，持续 Z 个 Chunk"。大多数文件每 Chunk=1 帧→stsc 只有 1 条记录。

stsz — 样本大小表

Sample Size · "第 N 帧数据多大？"

每个样本的字节数。CBR 音频用恒定大小模式（只需表头一个值）。视频帧大小剧烈变化（I 帧 80KB vs P 帧 10KB）→ 必须逐帧存储。

stco + stss — 偏移 + 关键帧

Chunk Offset + Sync Sample

stco：每个 Chunk 在文件中的绝对偏移量。>4GB 文件用 co64（64 位）。stss：哪些样本是IDR / I 帧（随机访问点）。seek 时先查 stss → 找最近 I 帧 → 从那里开始解码。

查表链完整演示：时间 3.5s → 文件偏移

六步查询

① stts：3.5s × 30000 = 105000 → 样本 #105
② ctts：样本 #105 偏移 = 0（P 帧无偏移）
③ stsc：#105 ÷ 1 帧/Chunk → Chunk #105
④ stco：Chunk #105 → 文件偏移 983040
⑤ stsz：样本 #105 → 24567 字节
⑥ stss：最近 I 帧 = #100 → 从 #100 开始解码

为什么不是"每个样本直接存偏移"？

1080p 2h 电影 ≈ 21.6 万帧。若每帧存 4 字节偏移 → 864KB 索引。

引入 Chunk 聚合：一个 Chunk 含 30 帧（1 GOP）→ 7200 个 Chunk → 索引仅 29KB（省 30×）。

这是空间换 I/O 的折中——索引越小，moov 读取得越快，首帧显示越快。

mdat — 编码数据的仓库 + FastStart

moov 是目录，mdat 是正文。编码数据按 Chunk交错存放

内部无 Box 结构

原始编码数据直接存放

H.264：AVCC 格式——每个 NAL Unit 前有 4 字节长度前缀（不用起始码，解析更快）。AAC：连续 raw AAC 帧。mdat 完全依赖 stbl 表来定位每个 Chunk。

音视频交错排列

V₁ A₁ A₂ V₂ A₃ A₄ V₃ …

顺序读取即可同时获取音视频——减少磁盘寻道。网络流式播放时至关重要（HTTP 请求连续范围）。非交错存储只适合本地文件。

Annex B vs AVCC — 两种 NAL 分隔方式

同样的 H.264 数据，两种"包装"格式

Annex B：00 00 00 01 起始码分隔 → 适合流式传输，丢包后可重新同步。
AVCC：4 字节长度前缀 → 已知长度直接跳转，适合容器随机访问。
MP4 用 AVCC，原始 .h264 文件和 TS 流用 Annex B。

FastStart — moov 前置是流式播放的前提

默认：moov 在末尾 → 无法边下边播

编码器先写 mdat → 文件关闭前才写 moov（因为 stbl 需要所有帧的最终大小和偏移）。
文件结构：ftyp → mdat(巨大) → moov。
播放器必须下载完整文件才能读到 moov → 无法边下边播。几 GB 的电影 = 几分钟等待。

FastStart：moov 前置 → 即时播放

后处理将 moov 搬到 ftyp 之后：ftyp → moov → mdat。
播放器先下载 moov（< 1MB）→ 立刻知道所有帧位置 → 对所需 mdat 范围发起 HTTP Range 请求 → 秒级开始播放。

FFmpeg 加 -movflags +faststart 即可。

深入理解：为什么 Annex B 和 AVCC 不合并？—— 流式传输 vs 随机访问

Annex B — 为"不知道从哪开始的接收者"设计

起始码 00 00 00 01 → 可自同步，适合流式传输
NAL 1:  00 00 00 01 [H.264 数据...]
NAL 2:  00 00 00 01 [H.264 数据...]
NAL 3:  00 00 00 01 [H.264 数据...]

核心优势：自同步。接收者可以在任意时刻开始接收——扫描字节流找到下一个 00 00 00 01 就能定位 NAL 边界。

适用场景：① 广播/IPTV（TS 流）——接收者随时可能换台 ② RTP 实时传输——UDP 丢包后需重新同步 ③ 原始 .h264 文件——简单、无需额外元数据。

致命弱点：要定位第 47 个 NAL Unit → 必须从开头扫描，逐个计数起始码 → O(n) 复杂度，无法随机访问。

AVCC — 为"知道目标位置的容器"设计

4 字节长度前缀 → O(1) 跳转，适合容器存储
NAL 1:  00 00 01 A5 [421 字节的 H.264 数据...]
NAL 2:  00 00 00 42 [66 字节的 H.264 数据...]
NAL 3:  00 00 03 E8 [1000 字节的 H.264 数据...]

核心优势：O(1) 随机访问。已知目标 NAL 的偏移 → 读 4 字节长度 → 读 N 字节数据 → 下一个 NAL 偏移 = 当前偏移 + 4 + N。不需要扫描任何字节。

适用场景：① MP4/MKV 容器——stbl 表已索引每个 Chunk 的文件偏移，AVCC 让容器可以直接跳转读 NAL ② 需要快速 seek 的场景——跳帧时不需从 I 帧开始逐 NAL 扫描。

致命弱点：如果从中间开始接收（漏了前面的长度前缀）→ 完全无法恢复同步——4 字节长度值可能恰好等于 00 00 00 01，但没有上下文你无法区分它是"长度"还是"起始码"。

为什么不合并？—— 因为"流式传输"和"容器存储"是两种截然不同的使用模式
流式传输（Annex B）的核心需求是容错和自同步——接收者可能在任何时刻加入、UDP 可能丢包、广播没有回传通道。起始码让接收者能"盲目"恢复同步。

容器存储（AVCC）的核心需求是快速定位和解析效率——播放器已有完整的索引（stbl），知道每个 Chunk 的精确偏移和大小。长度前缀让解析器用一条 CPU 指令（读 4 字节整数）替代字节扫描循环。

MP4 选择 AVCC 还有一个精妙之处：SPS/PPS 不放在 mdat 的 NAL 流里，而是单独存在 moov→trak→mdia→minf→stbl→stsd→avcC Box 中。这样解码器初始化时不需要扫描 mdat 找 SPS/PPS——直接读 avcC 即可。这是 Annex B 做不到的。

关键追问：Annex B 流中，编码数据本身不会恰好等于 00 00 00 01 吗？

会。H.264 编码后的 bit 串完全可能凑出 00 00 00 01。但标准用一个巧妙的机制杜绝了这种歧义——竞争防止字节（Emulation Prevention Byte）。

问题：编码数据可能恰好出现 00 00 00 01

那样解码器就会错误地认为这里是一个新的 NAL 起始边界
编码器输出（未处理）： ... 45 A6 00 00 00 01 8C 2F ... ↑ 这只是像素数据！ 解码器会误判为 NAL 边界 → 解析错乱

H.264 码流是变长熵编码后的二进制——任何字节序列都可能出现。如果不加处理，起始码 00 00 00 01 无法和"恰好等于这个值的编码数据"区分。

解决：插入 0x03 竞争防止字节

编码器在遇到 00 00 后紧跟 00/01/02/03 时，插入一个 0x03
原始数据： ... 45 A6 00 00 00 8C 2F ... ↑ 像素数据恰好是 00 插入 0x03 后（写入码流）： ... 45 A6 00 00 03 00 8C 2F ... ↑ 编码器插入的"防竞争字节" 解码器读取时： 看到 00 00 03 → 丢掉 03 → 还原为 00 00 00

规则：编码器扫描 NAL payload，每遇到 00 00 且下一个字节 ∈ {00, 01, 02, 03}，在 00 00 之后插入一个 0x03。解码器做相反操作——读到 00 00 03 → 丢弃 03，后一个字节保留。

为什么连 00 00 02 和 00 00 03 也要防止？
H.264 定义了四种"禁止模式"：00 00 00、00 00 01（NAL 起始码）、00 00 02（slice 边界标记，某些 profile 使用）、00 00 03（保留）。全部都要防止——因为未来扩展可能用到。

这就是 Annex B 能在"盲同步"下工作的原因：解码器可以无条件信任——码流中出现的每一个 00 00 00 01 都一定是 NAL 起始码，绝不可能是编码数据。这个保证是靠编码器多花那 1 个字节（每个 00 00 0X 多花 0x03）换来的。在大部分视频中，这个开销不到总码率的 0.1%。

fMP4（Fragmented MP4）—— 直播和流媒体的标准

普通 MP4 = 一个 moov + 一个完整的 mdat（写完不能改）。fMP4 = 多个独立可解码的片段

结构对比

普通 MP4: ftyp → moov → mdat（一整块，写死不能改） fMP4: ftyp → moov(含mvex) → moof₁+mdat₁ → moof₂+mdat₂ → …（可持续生成）

fMP4 的 moov 只含轨道描述（编码格式、timescale），不含 stbl 索引。索引功能由每个 moof 里的 trun 表承担。

moof 内部（取代 stbl 的角色）

moof ├── mfhd 片段序列号（递增） └── traf 轨道片段（每轨一个） ├── tfhd 轨道 ID + 默认参数 ├── trun ★ 片段样本表——等同普通 MP4 的 stbl │ 每帧记录：大小、时长、偏移、关键帧标志 └── tfdt 本片段的起始 DTS
应用场景容器格式结构
本地存储普通 MP4一个 moov + 一个 mdat
HTTP 点播FastStart MP4moov 前置 + mdat
DASH / HLSfMP4 片段init段 + 媒体段 × N
LiveKit / WebRTCfMP4 流连续 moof+mdat 流式写入
RTMP 直播FLV（非 MP4）Header + Tag 流

实战：用命令行工具查看 MP4 内部

ffprobe — 查看 Box 树和流信息

# Box 树 + 流信息
ffprobe -show_format -show_streams file.mp4

# 每一帧的详细信息
ffprobe -show_frames -select_streams v:0 file.mp4

# JSON 格式（方便程序解析）
ffprobe -print_format json -show_format \
  -show_streams file.mp4

# 只看是否已 FastStart
ffprobe -show_format file.mp4 2>&1 \
  | grep faststart

hexdump — 肉眼识别 Box 头部

$ hexdump -C file.mp4 | head -8
00000000  00 00 00 20 66 74 79 70 69 73 6f 6d  │... ftypisom│
         ↑ size=32    ↑ type="ftyp"   ↑ payload="isom"
00000020  00 00 20 08 6d 6f 6f 76 00 00 00 6c  │.. .moov...l│
         ↑ size=8200  ↑ type="moov"

Box 头部永远是 8 字节：4 字节 size（大端序）+ 4 字节 ASCII type。可以肉眼直接读出任何 Box 类型。

⚠ 所有 Web 播放的 MP4 必须加 FastStart
FFmpeg 编码时加 -movflags +faststart。如果是已有的 MP4 文件，可用 ffmpeg -i input.mp4 -c copy -movflags +faststart output.mp4 无损重写（只移动 moov 位置，不重新编码）。
