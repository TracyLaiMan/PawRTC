信令层 — WebSocket + Protobuf

所有控制信令通过 WSS (WebSocket over TLS 1.2) 传输，消息体为 Protobuf 序列化。
Join → Answer → Trickle → TrackPublished，一条连接完成全部协商。

WSS 连接建立
从 DNS 到 WSS 的四层握手
1. DNS 解析 → 获得 LiveKit Cloud 服务器 IP
2. TCP 三次握手 → 建立可靠传输通道
3. TLS 1.2 握手 → 协商加密参数（AES-128-GCM），验证服务器证书
4. WebSocket 升级 → HTTP GET /rtc?protocol=v3 + Upgrade: websocket头，服务器回复 101 Switching Protocols

ALPN 陷阱：LWS 默认添加 h2 ALPN，部分 CDN/代理优先协商 HTTP/2 导致 WebSocket 升级被拒。项目修复：仅在 WSS 连接时发送 http/1.1 ALPN。
Protobuf 编码
为什么不用 JSON？
项目使用 nanopb 编译 LiveKit proto 定义：
  • 体积：Protobuf 二进制编码比 JSON 省 30-50%，对嵌入式板端带宽敏感
  • 解析：nanopb 零堆分配，适合 JY1C 板 128MB RAM
  • 兼容：字段号而非字段名，前后向兼容
关键文件：protos/nanopb/livekit_rtc.pb.h
信令与 ICE 并行
Trickle ICE 加速连通
传统 ICE：收集齐所有候选后一起发送 → 延迟高
Trickle ICE：候选一旦发现立刻通过 trickle 消息发送 → 并行检测

项目中板端只产生 host 候选（局域网），无需等完整集合；SFU 提供 srflx 和 relay 候选也通过 trickle 逐条下发。

协商层 — SDP 会话描述

SDP (RFC 4566) 承载了媒体能力、ICE 凭证、DTLS 指纹等所有协商参数。
点击每行查看详细解释。

Offer/Answer 流程
谁发 Offer？
LiveKit 中 SFU 先发 Offer（SignalResponse.offer），板端回复 Answer（SignalRequest.answer），与 Web 浏览器角色相反。

Offer 中包含 SFU 的 ICE 凭证和 DTLS 指纹；Answer 中板端填入自己的凭证。双方各自将自己的候选通过 trickle 补充。
BUNDLE 策略
多路复用一个端口
a=group:BUNDLE 0 1 将音频和视频绑定到同一传输通道：
  • 只需一组 ICE + DTLS 协商
  • RTP 分流靠 SSRC 和 PT 区分
  • 减少 NAT 映射数量（嵌入式关键）
项目代码：LkSdp.c 中 addBundleGroup() 添加 BUNDLE 声明

连通层 — ICE / STUN / TURN

ICE (RFC 8445) 编排 STUN 和 TURN 探测，找到两端可达的候选路径。
点击查看每个状态的触发条件和角色。

ICE 候选类型
四种候选优先级递减
  • host — 本地网卡地址，优先级最高，无需 NAT 穿透
  • srflx (Server Reflexive) — STUN 映射的公网地址
  • prflx (Peer Reflexive) — 对端 STUN 请求中发现的地址
  • relay — TURN 中继地址，保底选择，延迟最大
项目场景：板端在局域网内只有 host 候选；SFU 通常在同一机房，host 直连即可连通。跨网时依赖 SFU 的 relay 候选。
Controlling vs Controlled
谁决定用哪条路？
Controlling 侧负责提名最终候选对，通常是 SFU（因为 SFU 知道最优路径）。
Controlled 侧被动接受提名，板端通常扮演此角色。

提名机制：Controlling 在 STUN 请求中加 ICE-CONTROLLED 属性，收到成功响应后该候选对被提名。Controlling 随后发送带 ICE-CONTROLLING 属性的请求确认。
STUN Binding 流程
NAT 类型探测的基石
1. 客户端发送 Binding Request（含 USERNAME=ufrag，MESSAGE-INTEGRITY 用 ice-pwd 计算）
2. STUN 服务器回复 Binding Response（含 XOR-MAPPED-ADDRESS = 客户端公网 IP:Port）
3. 比较本地地址与映射地址：相同 → 非对称 NAT（直接连通）；不同 → 对称 NAT（需 TURN）

Magic Cookie 0x2112A442 固定在 5-8 字节，用于区分 STUN 报文与其他 UDP 流量。
TURN 中继生命周期
四步建立中继通道
1. Allocate — 请求分配中继地址，返回 401 Unauthorized（需先认证）后再发带 REALM/NONCE 的请求
2. CreatePermission — 允许特定对端 IP 通过中继发送数据
3. ChannelBind — 建立 2B 通道号与对端地址映射，后续数据封装为 ChannelData（头仅 4B）而非 Send Indication（头 ~36B）
4. Refresh — 定期续约 Allocation，默认 LIFETIME 600s

项目中 TURN 服务器由 LiveKit Cloud 自动配置，通过 JoinResponse.ice_servers 下发 URL 和凭证。

安全层 — DTLS 1.2 握手

ICE 连通后立即启动 DTLS 握手，协商 ECDHE-ECDSA-AES128-GCM-SHA256，
并通过 use_srtp 扩展导出 SRTP 密钥。手动步进 10 步握手。

密码套件拆解
每一项都有协议意义
ECDHE-ECDSA-AES128-GCM-SHA256
  • ECDHE — 密钥交换，每次握手生成临时密钥对 → 前向保密
  • ECDSA — 签名算法，P-256 曲线验证服务器身份
  • AES128-GCM — 认证加密一体化 (AEAD)，无需额外 HMAC
  • SHA256 — PRF 和 Finished 消息的哈希
为什么不用 AES-256？ WebRTC 标准仅定义了 AES-128 的 SRTP Profile。
use_srtp 扩展
DTLS 如何顺便导出 SRTP 密钥
ClientHello 携带 use_srtp 扩展，列出支持的 Profile：
SRTP_AES128_CM_HMAC_SHA1_80 ← 项目使用

ServerHello 确认选中的 Profile。握手完成后通过以下调用导出密钥材料：
mbedtls_ssl_conf_dtls_srtp_protection_profiles()
→ key_material 拆分为 SRTP enc_key + auth_key + salt
SRTP 密钥导出公式
从 master_secret 到加密钥
Step 1: master = PRF(pre_master, "master secret", CR+SR)
Step 2: key_block = PRF(master, "EXTRACTOR-dtls_srtp", CR+SR)
Step 3: 拆分 key_block → 每方向：
  16B enc_key + 14B salt
  HMAC-SHA1 auth_key 从后续字节导出

CR = client_random, SR = server_random
DTLS vs TLS
为什么 WebRTC 不用 TLS？
  • 传输层：DTLS 基于 UDP，TLS 基于 TCP → 实时媒体不能容忍重传延迟
  • 消息边界：DTLS 保留 UDP 数据报边界，每条记录独立解密
  • 防重放：DTLS 加 epoch+sequence_number 头（6B），接收方用滑动窗口检测
  • 超时重传：DTLS 自带简易重传定时器，不依赖 TCP 拥塞控制
握手消息比 TLS 多出 message_seq 和 fragment 头。

媒体层 — RTP / SRTP / RTCP

DTLS 完成后，所有媒体走 SRTP 加密通道。
逐步查看 RTP 包结构与 SRTP 加密过程。

RTP 头逐位解读
12 字节决定媒体路由
V(2b)=2 固定；P(1b) 填充标志；X(1b) 扩展标志
CC(4b) CSRC计数；M(1b) H.264帧末包置1
PT(7b) 负载类型，本项目 H.264=102（动态分配96-127）
SeqNum(16b) 逐包递增，丢包检测+排序
Timestamp(32b) 90kHz，同帧同值
SSRC(32b) 随机生成标识媒体流
SRTP IV 构造
每个包的初始向量如何生成
IV[16] = 0x00, 中间填 SSRC(4B), 后半填 ROC(2B)||SeqNum(2B)

ROC (Roll-Over Counter)：SeqNum 从 65535→0 回绕时+1，保证 48bit 唯一性

AES-CTR：不断递增 BlockCounter 产生 keystream，与 Plaintext 异或 → Ciphertext。Header + Extension 保持明文。
H.264 FU-A 分片
大帧跨越多个 RTP 包
MTU ≈ 1200B，超过此大小的 NAL 单元必须分片：
FU Indicator(1B)：F=0, NRI=原 NAL 的 NRI, Type=28 (FU-A)
FU Header(1B)：S(1b) 首片, E(1b) 末片, R(1b)=0, Type(5b)=原 NAL 类型

完整帧还原：去掉每个包的 Indicator+Header，拼接即为原始 NAL
末片的 RTP 包 M bit 置1，通知接收方帧结束。
RTCP 反馈闭环
四类反馈驱动自适应
  • SR (Sender Report) — 发送方报告：已发包数/字节数/时间戳
  • RR (Receiver Report) — 接收方报告：丢包率/抖动/往返延迟
  • REMB — 接收方估计可用带宽，告知发送方降码率
  • Transport-CC — 逐包到达反馈，SFU 精确估计带宽
项目板端仅发送 SR，接收 RR/REMB；SFU 负责带宽估计决策。

数据层 — DataChannel / SCTP

SCTP over DTLS 提供可靠/不可靠数据传输通道。
板端 SDP 支持 m=application，但暂未启用数据通道。

SCTP 核心特性
比 TCP 更适合实时数据
  • 多宿主 — 单个关联可有多个 IP 路径，自动故障切换
  • 多流 — 一个关联内多个独立流，避免队头阻塞
  • 四次握手 — INIT → INIT-ACK → COOKIE-ECHO → COOKIE-ACK（防 SYN flood）
  • 部分可靠 — PR-SCTP 支持限时/限次重传，适合实时控制
SCTP over DTLS = 加密通道 + 消息边界保留。
DataChannel API 映射
Web 层如何映射到 SCTP
new DataChannel(label, {ordered, maxRetransmits})
  • label → SCTP 流标识 (SID)
  • ordered=true → SCTP 有序交付
  • maxRetransmits=N → PR-SCTP 限次重传
  • maxPacketLifeTime=ms → PR-SCTP 限时重传
DataChannel 开启消息 DATA_CHANNEL_OPEN 在 SCTP 流 0 上发送，协商参数。
