ECDSA 签名与证书

数字世界的身份证——证明"我就是我"，且不可伪造。
在 WebRTC 中，自签名证书通过 SDP 指纹绑定信任链路。

一把钥匙，两种用途

同一对公私钥，操作方向相反，安全目标截然不同：
私钥签名 → 身份证明，公钥加密 → 机密通信。

方向一 · 私钥出场
签名 & 验证 (ECDSA)
签名：用私钥对消息哈希做数学运算，生成签名 (r,s)。
验证：用公钥 + 消息验证签名是否源自对应私钥。

核心原理：
知道私钥 d → 能算出签名 (r,s)。
不知道 d → 给定任意消息，计算上不可能找到能通过验证的 (r,s)。
这就是 不可伪造性 —— 签名 = 身份证明。

安全属性：不可伪造 · 不可抵赖 · 可公开验证
// 签名 — 只有私钥持有者能做 sign(privateKey d, message) h = SHA-256(message) k = random() // 临时随机数 r = x(G·k) mod n // 隐藏 k s = k⁻¹·(h + r·d) mod n // 混入 d → signature (r, s) // 验证 — 任何一个有公钥的人都能做 verify(publicKey Q=G·d, message, (r,s)) → true / false
方向二 · 公钥出场
加密 & 解密 (ECDHE)
加密：用公钥将明文编码，只有对应私钥能还原。
解密：用私钥从密文恢复明文。

核心原理：
知道公钥 Q → 能加密消息给 Q 的持有者。
不知道私钥 d → 给定密文，计算上不可能还原明文。
这就是 机密性 —— 只有收件人能读。

安全属性：机密性 · 但无法证明发送者身份
// 实际中 ECDHE 不直接加密消息 // 而是协商共享密钥： keyExchange() Board: a = random(); A = G·a → send A Server: b = random(); B = G·b → send B Board: S = B·a = G·a·b Server: S = A·b = G·b·a // 双方得到相同的 S → 导出加密密钥 // 窃听者只有 A, B → 算不出 S
为什么同一个密钥对能双向使用？
数学上，签名和加密都基于同一个 单向函数：
• 正向：d × G = Q（标量乘法，毫秒级）
• 逆向：Q → d（椭圆曲线离散对数，宇宙级时间）

签名利用了"逆向极难"→ 只有知道 d 的人才能构造满足方程的 (r,s)，验证者只需 Q 就能验证等式。
密钥交换利用了"逆向极难"→ 双方各出一份 G·a 和 G·b，各自乘自己的私钥得到 G·a·b，旁观者看到 G·a 和 G·b 却算不出 d。

一句话：ECDSA 问"谁能证明自己是 d 的持有者？"，ECDHE 问"谁能和我共享秘密？"。底层数学相同，安全目标不同。

为什么需要证书、签名与指纹？

三个概念层层递进——解决了"谁是谁"、"谁签的"、"证书有没有被换掉"三个问题。

01
问题一：你怎么知道消息是谁发的？
没有签名的世界
板端发送 "我是设备 A，请加入房间" → 但攻击者也能发同样的话。

接收方无法区分这条消息到底来自合法设备还是攻击者。

这就好比任何人都可以自称"张三"——没有身份证的社会。
有了签名
板端用私钥对 "我是设备 A" 签名 → 发送消息 + 签名。

验证者用板端的公钥验证 → 签名有效 = 确实持有对应私钥的人发的。

不可伪造性保证：没有私钥的攻击者无法冒充。
签名解决："这条消息确实声称者发的" —— 身份证明。
但还有问题：我从哪拿到板端的公钥？我拿到的是真的还是假的？
02
问题二：你怎么知道公钥是谁的？
裸公钥的问题
板端告诉你 "这是我的公钥 Q" → 但攻击者可以截获，替换成自己的公钥 Q'。

验证者拿到 Q' → 用 Q' 验签名 → 攻击者用自己私钥签的消息反而能通过验证！

这就是中间人替换公钥攻击。
证书绑定身份
证书 = 公钥 + 身份信息 + 签发者签名 三合一。

CA 体系：浏览器信任 CA → CA 签证 → 证书包含 "公钥 Q 属于 *.livekit.cloud"。
WebRTC 体系：自签名证书 + SDP 指纹绑定 → 证书包含 "公钥 Q，指纹通过 TLS 信令传递"。

证书的核心价值：把公钥和身份绑在一起，而不是孤零零的一个公钥。
证书解决："这个公钥确实属于这个身份" —— 身份与公钥绑定。
最后的问题：证书本身也可能被替换，怎么检测？
03
问题三：你怎么知道证书没被替换？
自签名证书的困境
攻击者生成自己的自签名证书 → 在 DTLS 握手中替换对端证书。

自签名证书没有 CA 验证 → 接收方无法分辨真假。

任何人都能造一张自签名的"身份证"——光看证书，全都是"合法"的。
SHA-256 指纹绑定
对端通过 WSS/TLS 信令通道预先传来证书的 SHA-256 指纹。

DTLS 握手后，本地从收到的证书也算一个指纹。
两个指纹一致 → 证书和信令传的一致 → 可信。
指纹不一致 → 证书被替换 → 攻击！

前提：TLS 信令通道本身必须是安全的（WSS 用 CA 签发证书，浏览器自动验证）。
指纹解决："这个证书就是信令传的那个" —— 完整性绑定。
⚠️ VERIFY_OPTIONAL 跳过了指纹比对 → 整条信任链从这一步断裂！

两种信任模型

CA 体系靠根证书预装，WebRTC 靠 TLS 信令传递指纹。
一旦指纹比对被跳过（VERIFY_OPTIONAL），信任链即断裂。

模型一
HTTPS：CA 信任链
浏览器/OS 预装了 根 CA 证书，形成自动验证链。
用户无需手动比对 —— 验证过程全自动。
根 CA → 中间 CA → 服务器证书 → ✓ 自动信任
安全性：依赖 CA 安全性。CA 被攻破 → 整条链失效（DigiNotar 事件）。
模型二
WebRTC：SDP 指纹绑定
无 CA 参与。自签名证书的 SHA-256 指纹 通过 WSS/TLS 信令安全传递。
对端的指纹 = 证书的哈希 → 一致则可信。
自签名证书 → SHA-256 → 指纹 → TLS 信令传递 → ✓ 比对
风险点：SSL_VERIFY_OPTIONAL 跳过指纹比对 → 信任链断裂 → 中间人攻击畅通无阻。

对证书做 SHA-256 哈希，得到唯一指纹。修改证书任意一字节 → 指纹完全改变。

项目中的应用

板端每次连接实时生成 ECDSA 自签名证书，通过 DTLS 握手传递，SDP 指纹通过 WSS 信令保护。

证书生成
LkTransport.c — 证书生成与配置
板端在每次建立 DTLS 会话时，动态生成 P-256 ECDSA 密钥对和自签名 X.509 证书。

• mbedtls_pk_setup：初始化 ECDSA 密钥对 MBEDTLS_ECP_DP_SECP256R1
• mbedtls_x509_crt_sign：用自身私钥签发证书（自签名）
• mbedtls_ssl_conf_own_cert：将证书配置到 DTLS session
• 证书有效期设为 86400s（1天），每次重启重新生成

关键：自签名证书的真实性不依赖证书本身，而依赖后续的 SDP 指纹比对。
// LkTransport.c — createDtlsSession() 证书生成 mbedtls_pk_context pk; mbedtls_pk_init(&pk); mbedtls_pk_setup(&pk, mbedtls_pk_info_from_type( MBEDTLS_PK_ECKEY)); mbedtls_ecp_gen_key(MBEDTLS_ECP_DP_SECP256R1, mbedtls_pk_ec(pk), ...); // 自签名证书 mbedtls_x509write_crt_set_subject_key(&crt, &pk); mbedtls_x509write_crt_set_issuer_key(&crt, &pk); mbedtls_x509write_crt_set_validity(&crt, "20250101000000", "20260101000000"); mbedtls_x509write_crt_sign(&crt, &pk, md_sha256);
VERIFY_OPTIONAL 风险
指纹比对被跳过 — 中间人攻击窗口
板端 DTLS 配置中使用了 mbedtls_ssl_conf_authmode(MBEDTLS_SSL_VERIFY_OPTIONAL)。

• VERIFY_OPTIONAL：握手继续，即使证书验证失败
• VERIFY_REQUIRED：证书验证失败则握手中止

WebRTC 规范要求：DTLS 握手后，应用层必须比对指纹。
但当前板端代码没有实现指纹比对逻辑，导致：
• 攻击者可以用自己的证书替换对端证书
• DTLS 握手仍然成功完成
• ECDHE 协商出的密钥与攻击者共享 → SRTP 密钥泄露
• 音视频媒体流完全暴露

DTLS 握手完成后，获取对端证书 const mbedtls_x509_crt *peer = mbedtls_ssl_get_peer_cert(&ssl); // 2. 计算 SHA-256 指纹 mbedtls_sha256(peer->raw.p, peer->raw.len, hash, 0); // 3. 与 SDP 中的 a=fingerprint 比对 if (memcmp(hash, sdp_fingerprint, 32) != 0) { // 指纹不匹配 → 中止连接！ mbedtls_ssl_close_notify(&ssl); }
完整信任链路
JWT → WSS → SDP 指纹 → DTLS 证书 → SRTP
板端安全的完整信任传递路径：

① L1 JWT：HMAC-SHA256 签名 → 服务器验证身份
② L2 WSS：TLS 1.2 加密 → SDP 指纹安全传输
③ L4 DTLS：自签名 ECDSA 证书 + ECDHE 密钥交换
④ L5 SDP 指纹：SHA-256 绑定证书 → 防中间人
⑤ L6 SRTP：DTLS 导出密钥 → AES-CTR 加密媒体

只要 L5 指纹比对正确执行，整条链路安全。
唯一断点：VERIFY_OPTIONAL 跳过了 L5 → 链路从 L5 开始断裂。

ECDSA vs RSA 签名对比

都是数字签名方案，但基于不同数学难题。ECDSA 用更短的密钥和签名达到同等安全强度。

总结：现代 WebRTC 强制使用 ECDSA。更短的密钥（64B vs 256B）、更小的签名（64B vs 256B）、更快的运算 —— 对 MIPS 嵌入式设备尤其重要。板端 P-256 的 ECDSA 签名约 20ms，RSA-2048 签名约 150ms。项目不应考虑回退到 RSA 证书。

ECDHE vs ECDSA：各司其职

两者都基于椭圆曲线密码学，但在 DTLS 握手中职责完全不同：
ECDHE 管保密（密钥协商），ECDSA 管信任（身份签名）。缺一不可。

密钥协商
ECDHE
Elliptic Curve Diffie-Hellman Ephemeral
解决问题：通信双方在不安全信道上协商出相同的对称密钥

核心机制：双方各自生成临时密钥对，交换公钥后各自算出相同的 shared secret

为什么用 Ephemeral：每次连接生成新密钥对，前向保密 (PFS) — 即使长期私钥泄露，过去的会话也无法解密

在项目中的位置：DTLS 握手的密钥交换阶段，协商出的 shared secret 用于推导 SRTP 加密密钥
身份签名
ECDSA
Elliptic Curve Digital Signature Algorithm
解决问题：验证对方确实是它声称的身份，防中间人攻击

核心机制：Server 用长期私钥对握手消息签名，Client 用公钥验签

为什么不用 ECDHE 做认证：ECDHE 没有签名能力 — 它只产生共享密钥，无法证明"我是谁"

在项目中的位置：DTLS 自签名证书用 ECDSA 签发，SDP a=fingerprint 绑定证书公钥

一句话总结：ECDSA 管信任，ECDHE 管保密，两者缺一不可。

• 没有 ECDSA 认证 → ECDHE 公钥交换被中间人替换 → 协商的密钥与攻击者共享 → 无认证的密钥协商 = 与攻击者保密
• 没有 ECDHE 密钥交换 → 只有身份认证，无法协商对称密钥 → 没有加密通道传输媒体
• 两者配合 → DTLS 握手既验证身份又协商密钥 → SRTP 用协商出的密钥加密音视频 → 端到端安全
