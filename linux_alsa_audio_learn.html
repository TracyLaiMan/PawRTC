ALSA 子系统架构

ALSA 分为内核核心层、PCM 中间层、硬件驱动层，以及用户空间的 alsa-lib 和插件层。逐层理解才能定位音频问题。

用户空间 USERSPACE
👤

应用层 Application

使用 snd_pcm_readi() / snd_pcm_writei() 采集/播放。GStreamer、PulseAudio、本项目 AlsaAudioCaptureWrapper 均在此层。
alsa-lib 内部：snd_pcm_open() 时根据 ~/.asoundrc 或 /etc/asound.conf 的 snd_config_t 配置树构建插件链。snd_pcm_t 维护 PCM 流状态机 (SETUP→PREPARED→RUNNING)。参数协商：snd_pcm_hw_params() 逐一尝试参数，与驱动取交集，失败返回 -EINVAL。声道选择、增益调节、pre_gain 自适应均在此层完成。
▼snd_pcm_readi() → ioctl()
📦

alsa-lib (C 库 + 插件链)

核心抽象
snd_pcm_t — PCM 流对象 (封装 fd、插件链、状态机)；snd_config_t — 配置树 (解析 asoundrc)；snd_ctl_t — 控制接口 (管理 mixer/kcontrol)。
ioctl 封装：snd_pcm_readi() → mmap 或 read 模式 → ioctl(fd, SNDRV_PCM_IOCTL_*, arg)。
错误码：-EPIPE(XRUN)、-ESTRPIPE(挂起)、-EBADFD、-EAGAIN、-ENODEV。snd_pcm_recover() 统一处理。
插件链 Plugin Chain
hw(直通硬件) → plughw(自动格式/速率/通道转换) → dmix(多路软件混音，shm+flock) → dsnoop(多路软件分路) → pulse(PulseAudio 代理)。
hw vs plughw：hw:X,Y 严格匹配驱动参数，不支持则 -EINVAL；plughw:X,Y 自动插入转换滤镜，对应用透明但增加 CPU 开销。IPC 场景下直接用 hw:X,Y 避免延迟。

⚙️

ALSA Core

设备注册体系 snd_card → snd_device → snd_minor，每个 minor 对应 /dev/snd/ 下的一个设备节点。控制层通过 snd_ctl_* API 管理，每个控制项是一个 snd_kcontrol_new 实例。内存管理用 snd_dma_buffer 分配连续物理内存供 DMA。
🔄

PCM Midlevel

Ring buffer 核心——内核维护 hw_ptr(DMA 写位置)，用户维护 appl_ptr(应用读位置)。hw_params 回调分配 DMA 缓冲；sw_params 配置 start_threshold/avail_min/stop_threshold。中断回调 snd_pcm_period_elapsed() 更新 hw_ptr 并唤醒等待进程。XRUN：hw_ptr 追上 appl_ptr 时触发 overrun。
▼snd_pcm_ops 驱动回调
🔧

ASoC Driver 硬件驱动层

ASoC 体系：DAI 驱动配置 I2S/PCM 格式和时钟 (BCLK/LRCK)；DAI Link 连接 CPU DAI 和 Codec DAI；Codec 驱动控制 ADC/DAC 增益、电源管理；Platform 驱动管理 DMA 引擎。四者通过 snd_soc_card 组合。
Ingenic T31：内置 ADC 直接挂 I2S 接口，无需外部 Codec 芯片。Mic 信号直连 SoC I2S 输入引脚，Codec DAI 是内部虚拟设备。双 Mic (2ch) 通过 I2S 同时采集。
设备注册流程
snd_card_new() 创建声卡 → snd_pcm_new() 注册 PCM 设备 → 驱动填充 snd_pcm_ops 回调 → snd_card_register() 创建 /dev/snd/ 节点。
命名约定：hw:X,Y 中 X=Card 编号，Y=Device 编号。C=Card，D=Device，c=capture/p=playback。本项目 Mic 用 hw:1,0 (对应 /dev/snd/pcmC1D0c)，Speaker 用 hw:0,0。

ALSA 架构深度

展开下方四个板块，深入理解 ALSA 内核数据结构、PCM 状态机、完整数据流路径和调试工具集。 这些是你在嵌入式 Linux 上定位音频问题的底层武器。

01 内核核心数据结构 内核

ALSA 内核用三层嵌套结构管理 PCM 设备：snd_card 是声卡根对象，snd_pcm 是 PCM 设备， snd_pcm_runtime 是运行时上下文。理解它们的包含关系是阅读内核音频代码的钥匙。

struct snd_card
// 声卡根对象 — 所有子设备的容器 struct snd_card { int number; // 声卡编号 → /dev/snd/Cx char id[16]; // 声卡 ID 字符串 char driver[16]; // 驱动名 char shortname[32]; // 短名 (proc 显示) char longname[80]; // 长名 struct list_head devices; // 子设备链表 struct list_head controls; // kcontrol 链表 struct proc_dir_entry *proc_root; void *private_data; // 驱动私有数据 };
← 通过 snd_card_new() 创建，snd_card_register() 注册后创建 /dev/snd/ 节点
struct snd_pcm
// PCM 设备 — 含 playback + capture struct snd_pcm { struct snd_card *card; int device; // Device 编号 struct snd_pcm_str streams[2]; // [0]=PLAYBACK, [1]=CAPTURE struct snd_info_entry *proc_entry; };
每个 stream 含 substream_count 个 snd_pcm_substream，substream 才是实际操作的 PCM 流
struct snd_pcm_runtime
// 运行时上下文 — ring buffer + 参数 struct snd_pcm_runtime { snd_pcm_state_t state; // 当前状态 snd_pcm_format_t format; unsigned int rate; unsigned int channels; snd_pcm_uframes_t period_size; snd_pcm_uframes_t buffer_size; snd_pcm_sframes_t hw_ptr; // DMA 写位置 snd_pcm_sframes_t appl_ptr; // 应用读位置 struct snd_dma_buffer dma_buffer; struct snd_pcm_ops *ops; // 驱动回调 };
→ hw_ptr - appl_ptr = 可读帧数 (capture)；ring buffer 由 dma_buffer 描述物理内存
包含关系： snd_card → snd_pcm → snd_pcm_str[0/1] → snd_pcm_substream → snd_pcm_runtime
关键路径： 用户 open("/dev/snd/pcmC1D0c") → 内核通过 snd_minor 找到 snd_pcm_substream → 应用 ioctl 操作 snd_pcm_runtime

03 完整数据流路径 数据流

这是从麦克风声波到用户空间 PCM 缓冲区的每一个步骤。理解每一步的延迟贡献，才能精确计算端到端音频延迟。

1
Mic 声电转换 — 驻极体/硅麦将声波转为模拟电信号（mV 级）
~0µs
2
ADC 采样量化 — 48kHz 采样率，每 1/48000≈20.8µs 采样一次，24bit 量化。I2S 格式输出数字码流
~1 sample
3
I2S 传输 — 采样数据通过 I2S 总线 (BCLK=3.072MHz, LRCK=48kHz) 传入 SoC
~1 sample
4
DMA 写入 Ring Buffer — DMA 引擎将 I2S 接收的采样数据搬运到 snd_dma_buffer，每填满一个 period 触发一次中断
1 period
5
中断回调 — snd_pcm_period_elapsed() 更新 hw_ptr，调用 wake_up() 唤醒在 poll 上等待的进程
~10-50µs
6
snd_pcm_readi() — 触发 ioctl(SNDRV_PCM_IOCTL_READI_FRAMES)，内核执行 copy_to_user() 从 ring buffer 拷贝一个 period 数据到用户空间
1 period copy
7
用户空间处理 — S24_LE→float 转换、降采样、AGC、编码等
应用决定
延迟计算 (本项目)： period_size=256, rate=48000 → 1 period = 5.33ms
硬件延迟 ≈ 1 period (DMA 填充) + 中断唤醒 ≈ 5.5ms
软件延迟 ≈ snd_pcm_readi 阻塞等待 + copy_to_user ≈ 5.5ms
缓冲延迟 ≈ buffer = 3 periods → 最坏情况 16ms (2 periods 已填充 + 当前 period)
总延迟 ≈ 11~16ms (不含编码和网络传输)

PCM 参数深度解析

ALSA PCM 参数决定了音频数据的格式和延迟特性。理解 frame / period / buffer 的关系，才能调出低延迟且稳定的采集参数。

ALSA Ring Buffer 结构
frame = 1 采样点 × 通道数 × 位深/8
本项目: 1 frame = 1 × 2(ch) × 4(S24_LE) = 8 bytes
period = period_size × frame_size = 256 × 8 = 2048 bytes
buffer = 3 × period = 6144 bytes ≈ 16ms 延迟

硬件参数 (hw_params)

运行时不可更改，决定数据格式

访问模式：SND_PCM_ACCESS_RW_INTERLEAVED — 通道数据交错存储（L R L R...），最常用。NONINTERLEAVED 则按通道分平面存储。

采样格式：SND_PCM_FORMAT_S24_LE — 24bit 小端有符号整数，3字节有效但4字节对齐。本项目 JY1C 使用此格式。S16_LE 更通用但精度低。

采样率：48kHz — IPC 音频主流。高采样率减少延迟但增加 CPU 和带宽开销。

通道数：2 — 双 Mic 阵列降噪场景必需。单通道录音用 _handle_type=E_HANDLE_TYPE_LEFT。

软件参数 (sw_params)

控制流启动/停止行为

start_threshold：缓冲填充到此帧数后自动启动 DMA。设为 1 = 即时启动；设为 buffer_size = 整个 buffer 填满才启动（增加延迟）。

avail_min：snd_pcm_sw_params_set_avail_min() — 可用帧数达到此值时 snd_pcm_wait() / poll 返回可读。通常设为 period_size。

stop_threshold：当可写/可读帧数低于此值时自动停止。通常设为 buffer_size，即 XRUN 时不自动停止。

silence_threshold：静音填充阈值。Capture 场景不常用。

S24_LE 的坑
SND_PCM_FORMAT_S24_LE 在内存中占 4 字节（3 字节有效 + 1 字节 padding），不是 3 字节！采样值范围 -8388608 ~ +8388607。读取时需手动解包：低3字节组合后用符号位扩展到 32bit。

ALSA 插件 plughw 会自动做 S24_LE → S16_LE 转换，但 hw 模式下必须自己处理。本项目 _alsa_capture_work 中的转换循环就是这个原因。

ALSA 采集实战

从 snd_pcm_open 到 snd_pcm_readi 循环，完整的最小采集流程。理解错误处理路径是生产级代码的关键。

#include <alsa/asoundlib.h> int main() { snd_pcm_t *pcm; int ret; // ═══ 1. 打开采集设备 (阻塞模式) ═══ ret = snd_pcm_open(&pcm, "hw:1,0", SND_PCM_STREAM_CAPTURE, 0); if (ret < 0) return -1; // ═══ 2. 配置硬件参数 ═══ snd_pcm_hw_params_t *hw; snd_pcm_hw_params_alloca(&hw); snd_pcm_hw_params_any(pcm, hw); // 2a. 访问方式 & 采样格式 snd_pcm_hw_params_set_access(pcm, hw, SND_PCM_ACCESS_RW_INTERLEAVED); snd_pcm_hw_params_set_format(pcm, hw, SND_PCM_FORMAT_S24_LE); // 2b. 采样率 & 声道数 unsigned int rate = 48000; snd_pcm_hw_params_set_rate_near(pcm, hw, &rate, 0); snd_pcm_hw_params_set_channels(pcm, hw, 2); // 2c. 周期 & 缓冲区大小 snd_pcm_uframes_t period = 256; snd_pcm_hw_params_set_period_size_near(pcm, hw, &period, 0); snd_pcm_uframes_t buffer = period * 3; snd_pcm_hw_params_set_buffer_size_near(pcm, hw, &buffer); snd_pcm_hw_params(pcm, hw); // ═══ 3. 准备并启动 ═══ snd_pcm_prepare(pcm); // ═══ 4. 采集循环 ═══ uint8_t buf[2048]; // period_size × channels × sample_bytes while (1) { snd_pcm_sframes_t frames = snd_pcm_readi(pcm, buf, period); // Overrun: 应用读得太慢，缓冲数据被覆盖 if (frames == -EPIPE) { snd_pcm_prepare(pcm); continue; } // 系统挂起恢复 if (frames == -ESTRPIPE) { while (snd_pcm_resume(pcm) == -EAGAIN) sleep(1); snd_pcm_prepare(pcm); continue; } // 未知错误，尝试恢复 if (frames < 0) { snd_pcm_recover(pcm, frames, 0); continue; } // ✓ 成功读取 frames 帧数据，进行处理... } snd_pcm_close(pcm); return 0; }

XRUN 处理

Overrun / Underrun — 音频最常见错误

Overrun (Capture)：应用读得太慢，DMA 新数据覆盖了未读数据。内核计数 overrun 次数，snd_pcm_readi 返回 -EPIPE。

恢复：snd_pcm_prepare(pcm) 清空缓冲区重新启动。注意：prepare 会丢失当前 period 数据。

预防：提高采集线程优先级(SCHED_FIFO)、增大 buffer_periods、减少处理耗时。本项目采集线程用 SCHED_FIFO + 最高优先级。

阻塞 vs 非阻塞

三种 IO 模式各有取舍

阻塞模式 (0)：snd_pcm_readi 阻塞直到一个 period 数据就绪。代码最简单，本项目使用此模式。

非阻塞 (SND_PCM_NONBLOCK)：无数据时立即返回 -EAGAIN，需配合 poll/select 使用。适合事件驱动架构。

异步 (SND_PCM_ASYNC)：每填满一个 period 发 SIGIO 信号。信号处理函数中做 IO 操作不安全，生产环境极少使用。

S24_LE → float32 转换

24bit 有符号整数 → 浮点归一化

ALSA 采集的 S24_LE 数据每样本占 4 字节（3有效+1padding）。拆包过程：拼接低3字节，符号扩展到 32bit。

本项目处理：先将 24bit 值存为 int32_t，再直接赋值给 float（保留整数精度）。后续处理用 pre_gain / 8388608.f 归一化到 [-1, 1]。

关键代码：val = (conv_p[0])|(conv_p[1]<<8)|((conv_p[2]&0x7F)<<16)，然后处理符号位 0x80 & conv_p[2]。

48kHz → 16kHz 降采样

3:1 降采样 + 抗混叠低通滤波

简单 3:1 抽取（每隔3个样本取1个）会导致频谱混叠。必须先低通滤波截止到 8kHz (Nyquist = 16k/2)，再 3:1 抽取。

本项目使用 MLP (Multi-rate Low-pass) 滤波器：mlp_create(31, 16000, 8000)，31阶 FIR，通带8kHz，阻带起始于8kHz。

实际代码更简洁：直接每隔3个样本取1个 lp_inbuf[i] = lynr_outbuf[i*3]，因为 lynr 降噪输出已经是带限信号（≈16kHz以内），滤波主要留给 MLP。

MLP 低通滤波

抗混叠 FIR 滤波器

mlp_process(_lp, lp_inbuf, lp_inbuf, 480) — 原地滤波，输入输出同一缓冲区。

31阶 FIR 滤波器：输入480个样本（16kHz下30ms），输出480个滤波后样本。截止频率 8kHz，保证 3:1 抽取后的频谱纯净。

FIR 滤波延迟 = (N-1)/2 / fs = 30/2 / 16000 ≈ 0.94ms，可忽略。

AGC 自动增益控制

WebRtcAgc — 语音响度归一化

使用 WebRTC AGC 引擎：WebRtcAgc_Init(_agc, 0, 255, kAgcModeAdaptiveDigital, 16000)。

处理粒度：每次 160 个样本 (10ms @16kHz)，480 个样本分 3 次处理。

目标电平：targetLevelDbfs=2，压缩增益：compressionGaindB=18。效果：说话声音小的时候自动放大，声音大的时候压低，保持输出音量稳定。

此外还有 pre_gain 机制：初始化为 8.0，检测到溢出(-32760~+32760)时自动减小 0.5，连续 28800 帧(≈180s) 无溢出则缓慢回升 0.01。这是一种慢速自适应增益保护。

为什么降到 16kHz？
Opus 编码语音模式最高支持 16kHz 宽带语音（超宽带/全频带需更高码率）。16kHz 采样率覆盖 0-8kHz 频谱，包含语音的全部关键频段。48kHz→16kHz 的降采样将数据量减少 2/3，同时保留语音清晰度。

音频编码格式全景

从 PSTN 遗留的 G.711 到 WebRTC 标配的 Opus，音频编码的选择取决于带宽、延迟和音质要求的三角权衡。

G.711 A-law / μ-law

PCM → 对数压缩，零编码延迟

将 14bit 线性 PCM 非线性量化为 8bit。A-law 用于欧洲/中国(E1)，μ-law 用于北美/日本(T1)。听起来像电话音质。

编码/解码仅需一次查表操作，延迟几乎为零。但 64kbps 对带宽的要求甚至高于 Opus 低码率模式，性价比低。

SDP 协商：a=rtpmap:0 PCMU/8000 或 a=rtpmap:8 PCMA/8000。

AAC-LC

感知编码，高压缩比，高延迟

利用心理声学模型遮蔽效应，去除人耳不可闻频率分量。48kHz 采样支持全频带音频。典型 64-128kbps 码率下接近透明音质。

编码延迟 ~20ms (1024样本帧 @48kHz)，对通话场景太高。主要用于录像存储、流媒体分发（HLS/DASH）。

本项目中 ACS SDK 的录像存储使用 AAC-LC 编码，实时通话使用 Opus。

Opus

WebRTC 标配，SILK+CELT 混合编码

SILK 引擎处理语音（线性预测编码，6-40kbps，适合低码率），CELT 引擎处理音乐/混合内容（MDCT 变换，40-510kbps）。编码器根据信号特征自动切换。

帧长灵活：2.5/5/10/20/40/60ms。10ms 帧总延迟 ≈ 20ms (编码+抖动缓冲)，满足交互实时性要求。

内置 In-band FEC：每帧嵌入前一帧低码率副本，丢包可恢复。DTX 模式：静音时只发舒适噪声帧，大幅节省带宽。

SDP 协商：a=rtpmap:111 opus/48000/2，a=fmtp:111 useinbandfec=1; maxaveragebitrate=24000。

项目编码选择

Agora RTC / LiveKit 均使用 Opus

RemoteLiveView (Agora)：内置 Agora RTC SDK，自动使用 Opus 编解码。开发者只需提供 16kHz PCM 数据，SDK 内部完成 Opus 编码和网络传输。

LiveKit Demo：使用 LiveKit C SDK，Opus 是唯一音频编码格式。客户端通过 livekit_webrtc_publish_audio_frame() 提交 16bit PCM 数据。

完整通路：Mic → ALSA S24_LE 48kHz → float转换 → 声道提取 → 48k→16k降采样 → MLP滤波 → AGC增益 → int16 16kHz PCM → Opus编码 → RTP → SFU → 浏览器。
