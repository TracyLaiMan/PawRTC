弱网的四维画像

理解弱网，先要量化衡量。四个维度刻画网络质量，每个维度都有针对性的对抗手段。

核心维度
📉
带宽受限 ▸ 点击展开
可用带宽低于视频编码码率，导致排队延迟→丢包→卡顿。
典型场景：4G 弱信号、WiFi 拥塞共享

根因：物理链路容量有限（如 4G 信号差仅 500kbps），或共享瓶颈链路被其他流占用（WiFi 多人同时看视频）

衡量方式：通过 GCC 等拥塞控制算法估计可用带宽（A_hat），对比当前编码码率判断是否受限

级联效应：带宽不足 → 路由器排队增长 → 延迟梯度上升 → 丢包发生 → GCC 降码率 → 画质下降

对抗手段：GCC 带宽估计 + 编码器自适应码率 + Simulcast/SVC 分层 + SFU 选流

板端现实：JY1C WiFi 带宽约 5-10Mbps，编码 1080p@2Mbps 通常够用，但 4G 回传场景带宽仅 500kbps-1Mbps

核心维度
📦
随机丢包 ▸ 点击展开
数据包在传输途中被路由器尾部丢弃或因链路错误丢失。
2%~20% 丢包都会严重影响质量

根因：①路由器缓冲满 →tail drop（最常见）②无线链路干扰导致 CRC 错误③中间设备 MTU 不匹配导致大包被丢

衡量方式：发送端通过 RTP 序列号缺口检测丢包率；RTCP Receiver Report 中的丢包统计字段

影响分级：<2% 基本无感；2-5% 轻微马赛克；5-10% 明显卡顿；10-20% 严重花屏；>20% 无法正常通话

丢包模式：均匀丢包（FEC 更有效）vs 突发丢包（连续丢多个包，FEC 行列编码更强）

对抗手段：NACK 重传（低 RTT）→ FlexFEC 前向纠错（高 RTT/高丢包）→ PLI 关键帧刷新（严重丢包）

核心维度
🔄
延迟与 RTT ▸ 点击展开
信号传播+排队+处理延迟。RTT 直接决定 NACK 重传时效上限。
100ms+ 则重传价值递减

组成部分：传播延迟（光速/距离，如上海→旧金山 ≈ 60ms）+ 排队延迟（路由器缓冲，0-100ms+ 不等）+ 处理延迟（设备转发，通常 <1ms）

衡量方式：STUN Binding Request/Response 的时间差；WebRTC 内部通过 TWCC 反馈计算

对重传的限制：播放缓冲 200ms 时，RTT < 100ms 可重传 1 次；RTT < 50ms 可重传 2 次；RTT > 200ms 重传无用

对交互的影响：<150ms 无感；150-300ms 感知迟钝；300-500ms 明显延迟；>500ms 对话困难

对抗手段：高 RTT 场景放弃 NACK，改用 FEC 前向纠修；SFU 部署靠近用户降低 RTT

核心维度
〰️
突发抖动 ▸ 点击展开
包间延迟变化导致包到达时间不均匀。轻则音画不同步，重则缓冲欠载卡顿。

根因：①路由器排队不均匀（交叉流量突发占用）②无线链路竞争（WiFi CSMA/CA 退避）③运营商流量整形（token bucket 突发）

衡量方式：计算连续包到达间隔的标准差（RFC 3550 Jitter 定义）；WebRTC 统计中的 jitterBufferDelay / jitterBufferEmittedCount

抖动模式：稳态低抖动（局域网 <1ms）→ 周期性抖动（WiFi beacon 干扰，100ms 周期）→ 突发大抖动（跨运营商，峰值 200ms+）

核心矛盾：缓冲越大→播放越平滑→延迟越高；缓冲越小→延迟越低→容易欠载卡顿

对抗手段：自适应 Jitter Buffer（NetEq）动态调整缓冲深度 + WSOLA 加速/减速播放 + PLC 丢包掩盖

GCC：Google 拥塞控制

WebRTC 默认的带宽估计算法，由 Google 设计，基于延迟梯度和丢包率双信号驱动。
核心思想：网络拥塞时排队延迟先升高（信号更早），丢包后发生（信号滞后），因此延迟梯度是更领先的拥塞指示器。

1

到达时间滤波器 (Arrival-Time Filter)

计算连续包组的到达间隔 Δta 与发送间隔 Δts 的差值。
延迟梯度 m(i) = Δta - Δts 反映了排队延迟变化。

2

过载检测器 (Overuse Detector)

将延迟梯度送入 Trendline Filter（线性回归或 Kalman Filter），
判断当前网络状态：Underuse → Normal → Overuse。

3

速率控制器 (Rate Controller)

根据过载检测结果执行 AIMD 策略：
Overuse → 乘性下降 (×0.85)
Normal → 加性上升 (+Δ)
Underuse → 保持当前估值

4

丢失率修正 (Loss-Based Modifier)

延迟梯度估值作为基准，再用丢包率做修正：
丢包率 > 2% → Arecv × (1 - 0.5 × loss_rate) 降码率
丢包率 < 2% → Arecv × 1.05 试探升码率

延迟梯度计算详解

// 包组定义：同一帧的 RTP 包组成一个包组 // i-th 包组的到达时间: t_a(i), 发送时间: t_s(i) Δt_a(i) = t_a(i) - t_a(i-1) // 到达间隔 Δt_s(i) = t_s(i) - t_s(i-1) // 发送间隔 // 延迟梯度（排队延迟变化量） m(i) = Δt_a(i) - Δt_s(i) // Trendline Filter: 对最近 w 个梯度做线性回归 trendline_slope = slope(m(i-w), ..., m(i)) // 过载阈值 γ = adaptive_threshold // 自适应调整的阈值

过载检测状态机

// 三个状态: Underuse / Normal / Overuse if trendline_slope > γ then // 梯度超过上阈值 → 排队在增长 overuse_counter++ if overuse_counter >= overuse_threshold then state = OVERUSE elif trendline_slope < -γ then // 梯度低于下阈值 → 排队在消退 state = UNDERUSE else state = NORMAL // 自适应阈值 γ: 防止在噪声大的网络中误判 γ(i) = γ(i-1) + k_d * (|m(i)| - γ(i-1)) + Δγ

AIMD 速率调节

// A_recv: 当前带宽估计值 // A_recv_last: 上一次过载时的带宽估计 switch(state): case OVERUSE: // 乘性下降：快速退让 A_recv = A_recv_last × β // β ≈ 0.85 case NORMAL: // 加性增长：缓慢试探 A_recv += Δ // Δ 按时间计算 // 同时不超过延迟梯度的安全上限 A_recv = min(A_recv, 1.5×A_recv_last) case UNDERUSE: // 保持当前估值不变 // 等待网络恢复确认 A_recv = A_recv

丢包率修正公式

// loss_rate: 近期丢包率 // A_recv: 延迟梯度估计值 if loss_rate > 0.10: // 高丢包：强力降码率 A_final = A_recv × (1 - 0.5 × loss_rate) elif loss_rate > 0.02: // 中等丢包：温和降码率 A_final = A_recv × (1 - loss_rate) else: // 低丢包：试探升码率 A_final = A_recv × 1.05 // 最终带宽取延迟估计与丢包修正的较小值 target_bitrate = min(A_delay, A_final)

Transport-CC 反馈机制

GCC 的信号来源。接收端逐包记录到达时间，通过 RTCP Transport-CC Feedback 报文发送给发送端，
发送端据此计算延迟梯度。比旧方案 REMB（接收端估计）更精确，已成为 WebRTC 主流。

Transport-CC Feedback 每 ~100ms 发送一次，包含自上次反馈以来的所有 RTP 包到达时间戳。
发送端通过比较 到达间隔 Δta 与 发送间隔 Δts，精确计算排队延迟变化。
这比旧方案 REMB（接收端估计后在 RTCP 中上报码率数值）更精确，因发送端掌握全局发送节奏。
Transport-CC vs REMB
REMB (Receiver Estimated Maximum Bitrate)：接收端估算带宽后通过 RTCP REMB 消息告知发送端。
问题：接收端缺少发送节奏信息，对突发流量的梯度计算不准确。
Transport-CC：接收端只报告到达时间，计算逻辑移到发送端，与 GCC 配合更紧密。
Chrome 75+ 已默认启用 Transport-CC，REMB 逐步弃用。

KVS-WebRTC 的现状

抗丢包四维方案

丢包是弱网最直接的症状。NACK 重传、FEC 前向纠错、关键帧请求和音频专用方案各有适用场景，WebRTC 实战中联合使用。

① NACK + RTX 重传
检测丢包 → 请求重发 → 受 RTT 时效约束

原理：接收端发现 RTP 序列号缺口后，发送 RTCP NACK 消息请求发送端重传对应包。NACK 消息格式：FCI = (PID, BLP)，PID 指明首个丢失包序号，BLP 用 bitmap 指示后续连续丢包，一条 NACK 最多报告 17 个丢包。

RTX 机制：使用独立 SSRC 重传，避免 RTP 序列号冲突。发送端维护 RtpRollingBuffer 缓存最近发送的包供重传取用。原始包 seq=1000，重传包使用 RTX SSRC 不同的 seq 空间。

时效约束：重传包必须在播放 deadline 前到达。播放缓冲 200ms 时：RTT < 100ms 可重传 1 次；RTT < 50ms 可重传 2 次；RTT > 200ms 重传无意义。WebRTC 设定每个包最多 NACK 2-3 次后放弃。

适用场景：低丢包 (1-5%) + 低 RTT (<100ms)，消耗带宽极小（仅丢包量的 1-2 倍）。

局限性：高 RTT 时完全失效；高丢包时大量 NACK 报文本身占带宽；无法抗突发性连续丢包（整个 GOP 丢失时重传来不及）。

NACK 时序流程 (RTT=100ms)
t=0ms TX 发送 RTP seq=100,101,103（102 在路由器 tail drop）
t=50ms RX 收到 101 后发现缺口 seq=102 → 发送 RTCP NACK PID=102 BLP=0
t=100ms TX 收到 NACK → 查 RtpRollingBuffer → 从 RTX SSRC 重传 seq=102
t=150ms RX 重传包到达 → 送入 Jitter Buffer 排序 → 解码恢复 ✓（deadline=200ms）
② FEC 前向纠错
冗余编码 → 无需等待 → 零额外延迟恢复

原理：发送端额外发送携带冗余信息的 FEC 包，接收端利用冗余信息重建丢失的媒体包，无需反向请求。恢复延迟 = 0（等 FEC 包组收齐即可解码），这是与 NACK 的关键差异。

FlexFEC (RFC 8627)：当前 WebRTC 主推方案。支持行异或、列异或、行列异或三种编码，灵活选择保护粒度。行列编码可抗突发丢包（同一行/列丢 2+ 个包仍可恢复）。

UlpFEC (RFC 5109)：早期方案，整包级别异或保护，保护粒度粗，已被 FlexFEC 取代。Chrome 已弃用。

冗余率自适应：WebRTC 根据实时丢包率动态调节 FEC 冗余比例。丢包 5% 时加 ~10% FEC 冗余，丢包 15% 时加 ~30% 冗余。总发送码率 = 媒体码率 + FEC 开销，需在 GCC 带宽估计上限内分配。

适用场景：中高丢包 (5-20%) 或 高 RTT (>100ms) NACK 重传失效时。是以带宽换可靠性的核心手段。

局限性：消耗额外带宽（10-50% 额外开销）；行列编码增加发送延迟（需等整行/列包齐才发 FEC 包）；丢包超过 FEC 恢复能力时仍需 PLI 兜底。

FlexFEC 行列编码详解

4行×4列 媒体包 + 行FEC + 列FEC 编码：

  媒体包:  M1  M2  M3  M4  → 行FEC: R1 = M1⊕M2⊕M3⊕M4
           M5  M6  M7  M8  → 行FEC: R2 = M5⊕M6⊕M7⊕M8
           M9  M10 M11 M12 → 行FEC: R3 = M9⊕M10⊕M11⊕M12
           M13 M14 M15 M16 → 行FEC: R4 = M13⊕M14⊕M15⊕M16
           ↓    ↓    ↓    ↓
  列FEC:   C1   C2   C3   C4
           (各列异或)

  丢失 M7: 用 R2⊕M5⊕M6⊕M8 恢复，或用 C3⊕M3⊕M11⊕M15 恢复
  丢失 M7+M8: 仅 R2 就能同时恢复两个（行冗余包覆盖两个丢包）
  丢失 M7+M11: R2 恢复 M7，C3 恢复 M11（行列联合恢复）
  丢失 M7+M8+M11: R2 恢复 M7+M8，C3 恢复 M11（仍可恢复！）

关键优势：同一行丢任何 ≤1 个包，或行列交叉丢包，均可恢复。这是 FlexFEC 行列编码比纯行编码更强的地方。

③ 关键帧请求 (PLI / FIR)
严重丢包 → 放弃修补 → 全帧刷新重启解码

PLI (Picture Loss Indication)：接收端检测到参考帧（I 帧或关键 P 帧）丢失，后续所有 P 帧无法解码时发送。发送端收到后立即编码一个新 I 帧。这是最常用的关键帧请求方式。

FIR (Full Intra Request)：比 PLI 更强力，强制请求生成 I 帧，用于 SIMULCAST 层切换、MCU 合流布局变化、新订阅者加入等场景。FIR 带有命令序列号，防止乱序重复处理。

触发时机：① NACK 重传超时（同一包 NACK 2-3 次）；② 解码器报告参考帧缺失；③ SFU 切换 Simulcast 层时发送 FIR；④ 新订阅者加入房间时。

代价与风险：I 帧大小是 P 帧的 10-50 倍（1080p I 帧可达 100-200KB），会瞬间占用大量带宽，可能加剧拥塞形成恶性循环。必须配合 GCC 码率估计控制 I 帧大小。

防抖策略：WebRTC 对 PLI/FIR 请求做限流，最小间隔 T_minPLI = 500ms（可配置），避免接收端反复请求 I 帧导致发送端码率爆炸。

④ 音频抗丢包方案
Opus In-band FEC + RED + PLC — 音频比视频更怕丢包

为什么音频更需要保护：人耳对音频断续极其敏感（50ms 静音即可感知），且音频码率极低（6-32kbps），FEC 开销占比小、收益大。视频丢一个 P 帧最多花屏，音频丢一个包就是明显卡顿。

Opus In-band FEC
编码器每帧嵌入前一帧的低码率副本。丢包时用副本替代，音质略降但不断续。
开销：约 20% 额外码率
SDP：useinbandfec=1
限制：只能恢复单包丢失
RED (RFC 2198)
在 RTP 包头附加前 1-4 帧的完整编码数据。可恢复连续多包丢失。
开销：比 In-band FEC 更大（完整帧冗余）
状态：Chrome 已弃用 RED，改用 In-band FEC
优势：可抗连续 2-4 个包丢失
PLC (Packet Loss Concealment)
解码器基于前一帧信号特征做波形插值，掩盖 1-2 帧丢失。无需额外带宽。
开销：零带宽开销
内置：Opus 自带 PLC
限制：仅能掩盖 20-40ms 丢包

发送端编码:
  Frame N:   [当前帧 20ms 音频 @24kbps] + [Frame N-1 低质量备份 @6kbps]
  Frame N+1: [当前帧 20ms 音频 @24kbps] + [Frame N   低质量备份 @6kbps]

接收端丢包恢复:
  正常:  收到 N, N+1 → 正常解码播放
  丢包:  丢失 Frame N → 从 Frame N+1 中提取低质量备份解码 → 音质下降但不断续
  连丢:  丢失 N, N+1 → PLC 插值 40ms → FEC 无法恢复 → 短暂静音
联合策略 — 弱网下的分层防护

自适应码率 (ABR)

带宽预测给出了"天花板"，如何利用这个天花板？编码器自适应调节、Simulcast、SVC 三层架构协同工作。

🎬

编码器码率自适应

WebRTC BitrateAllocator 将 GCC 估计的目标码率分配给各路流。
编码器通过调整 QP (量化参数) 和 帧率 来匹配目标码率。
关键：编码器必须在一帧内响应码率变化，否则迭代回路断裂。
VideoStreamEncoder::OnBitrateUpdated() → 设置编码器目标码率。

📐

Simulcast 同时多流

发送端同时编码多个空间/质量层（如 320p/640p/1280p），
SFU 根据每个订阅者的带宽选择转发对应的层。
LiveKit 默认启用 Simulcast（3层），这是房间视频的核心抗弱网手段。
代价：发送端 CPU 和带宽开销 ×层数。

🧬

SVC 可伸缩视频编码

单流内嵌多层：空间层（分辨率）+ 时间层（帧率）。比 Simulcast 节省带宽。
VP9/AV1 支持 SVC，H.264 不支持。
SFU 可按层过滤（strip 高层），实现带宽自适应转发。
LiveKit 对 VP9 SVC 有实验性支持。

Simulcast 三层流选择示意

低流 320p ~150kbps
中流 640p ~400kbps
高流 1280p ~800kbps
板端 Simulcast 的现实约束
JY1C (mipsel) CPU 性能有限，多路实时编码对 CPU 和内存压力大。
当前 lk_demo 只发送单流 H.264，未启用 Simulcast。
若需弱网自适应，建议依赖 SFU 端转码降分辨率（LiveKit 已支持），而非板端多流编码。
↓

Jitter Buffer 与延迟对抗

网络抖动导致包到达时间不均匀。Jitter Buffer 通过短暂缓冲来平滑播放，但也引入额外延迟。
核心矛盾：缓冲越大越平滑，延迟越高。自适应 Jitter Buffer 动态寻找平衡点。

自适应缓冲深度
监测包到达间隔方差，动态调整缓冲目标延迟。
音频：20ms~500ms
视频：0~3000ms
NetEq 是 WebRTC 音频 Jitter Buffer 的核心组件
加速/减速播放
缓冲欠载时：正常播放（PLC 掩盖）
缓冲过深时：WSOLA 算法加速播放（变调不明显）
利用加速/减速逐步将缓冲深度调向目标
音视频同步
RTP 时间戳 + NTP 时钟映射
对齐音频和视频的渲染时刻
同步误差 < 50ms 人眼不可感知

KVS-WebRTC 弱网能力现状

对照完整 WebRTC 技术栈，分析 KVS-WebRTC（LiveKit 板端采用）的弱网能力缺口，为后续开发指明方向。

NACK 重传

已实现

基础 NACK + RTX 重传可用。支持 RTCP NACK 解析和 RTP 重传包发送。

  • JitterBuffer 负责丢包检测和 NACK 生成
  • RtpRollingBuffer 缓存发送包用于重传

GCC 拥塞控制

未实现

最大的短板。无延迟梯度和 AIMD 带宽估计，发送速率完全依赖应用层设定。

  • 无 Transport-CC Feedback 处理
  • 无 Trendline Filter / Kalman Filter
  • 编码器码率无法自适应调节

FlexFEC 前向纠错

未实现

无 FEC 编解码能力，中高丢包场景完全依赖 NACK 重传（高 RTT 时失效）。

  • FlexFEC / UlpFEC 均未集成
  • 音频 Opus In-band FEC 也未启用

Jitter Buffer

部分实现

有基础包重排和缓冲，但缺乏自适应深度调节和 NetEq 式的加速/减速播放。

  • 固定缓冲深度，非自适应
  • 无 WSOLA 加速算法
  • 音频/视频同步未实现

Simulcast / SVC

未实现

仅支持单流发送，无法提供多分辨率选择。SFU 端需做转码降分辨率。

  • 无多空间层编码
  • 无 SVC 时间/空间层封装

PLI / FIR

部分实现

支持 RTCP PLI 消息的发送和接收，但关键帧编码请求的响应依赖编码器集成。

  • PLI 消息收发可用
  • 编码器侧 OnPictureLoss 未完整对接

板端弱网增强路线图

基于 KVS-WebRTC 现状和 LiveKit 架构，分阶段增强板端弱网能力。

1

Opus In-band FEC + PLI 完善

最低成本最高收益。启用 Opus FEC 保护音频；完善 PLI→编码器 I 帧请求链路。预计 1-2 天。

2

自适应 Jitter Buffer

基于到达间隔方差动态调整缓冲深度。实现简版 NetEq 加速/减速逻辑。预计 3-5 天。

3

FlexFEC 集成

集成开源 FEC 库（如 OpenFEC / librpbfec），支持行列编码。按丢包率动态调节冗余率。预计 5-7 天。

4

简化版 GCC 发送端

实现 Transport-CC Feedback 解析 + Trendline Filter + AIMD 速率控制。与 LiveKit SFU 的 Transport-CC 配合。预计 7-10 天。

5

SFU 联动优化

利用 LiveKit SFU 的订阅降级、Simulcast 转码、带宽估计下推等能力，配合板端形成端到端闭环。持续迭代。
