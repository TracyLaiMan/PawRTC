1. 时钟硬件基础 — 晶振 · PLL · 分频器

在理解 Linux 的软件时钟框架之前，必须先理解物理上时钟是怎么产生和变换的——软件框架只是硬件行为的抽象。

晶振 (OSC) — 时钟的"心脏"

石英晶体 + 皮尔斯振荡电路 = 精确的频率源

嵌入式系统通常有一个主晶振 24MHz 或 26MHz——这是整个 SoC 所有时钟的源头。石英晶体的频率精度取决于负载电容匹配——标称负载电容（如 12pF）必须与 PCB 上实际的总电容（晶体引脚对地电容 + PCB 走线寄生电容）匹配。不匹配 → 频率偏差 → 所有派生时钟全偏。
频率偏差的来源：初始精度（±10-20ppm @25°C）、温度漂移（±20-50ppm across -20~60°C——石英的 AT 切角决定温漂曲线形状）、老化（±3-5ppm/年——第一年最大，之后递减）。IPC 户外 -20~60°C → 总偏差可达 ±70ppm → 24MHz 时钟偏差 ±1.68kHz。

PLL — 倍频器

Phase-Locked Loop — 从低频晶振生成 GHz 级时钟

原理：PLL 是一个负反馈控制系统——相位比较器 (PFD) 比较参考时钟 (REF, 来自晶振 24MHz) 和反馈时钟 (FB, VCO 输出 ÷ M) → 输出误差电压 → 低通滤波器 → 控制压控振荡器 (VCO) 的频率 → VCO 输出 ÷ N 得到输出频率、÷ M 得到反馈频率。
输出频率 = REF × N / (M × R)。如 REF=24MHz, R=1, M=24, N=1200 → VCO=1200MHz, 输出=1200/24=50MHz。
PLL 锁定的条件：VCO 必须在它的工作频率范围内（如 600-1600MHz），M/N 必须在寄存器可编程范围内。PLL 从启动到锁定的典型时间 100-500μs（取决于环路滤波器带宽）。锁定期间输出频率不稳定→外设必须等 PLL 锁定后才能用。

分频器 + 门控

Divider · Gate · Mux — 三种基本时钟操作

Divider (分频器)：输出频率 = 输入频率 / N。N 通常是 2 的幂（1/2/4/8...），也可为任意整数。整数分频器输出占空比可能不是 50%（N=3 → 33%/67% 占空比——对某些外设有影响）。Fractional Divider（分数分频）：N = 整数 + 分数（如 2.5）→ 通过 sigma-delta 调制平均实现——音频 PLL 需要（如 24MHz → 12.288MHz 需要 N=1.953125——必须用分数分频器）。
Gate (门控)：开关——enable/disable 时钟到某个外设的路径。关掉不用的外设时钟→省电（SoC 的动态功耗与时钟频率成正比）。
Mux (选择器)：从多个父时钟中选择一个作为当前时钟源。如 I2S 音频时钟可以从 PLL_AUDIO(12.288MHz) 或外部晶振(12MHz)中选择。

2. CCF 架构 — Common Clock Framework

Linux 3.x 引入的 CCF 统一了 ARM 平台碎片化的时钟管理——此前每个 SoC 厂商有自己的时钟 API（Samsung/Qualcomm/TI 各不同）。CCF 用Provider-Consumer 模型和设备树统一了这一切。

Provider (时钟提供者) — 驱动侧

drivers/clk/ — 描述硬件时钟树的结构和行为

SoC 的时钟驱动（通常位于 drivers/clk/ 下）负责注册硬件上存在的每个时钟——每个 PLL、每个分频器、每个门控——都是 CCF 中的一个 struct clk_hw。驱动实现 clk_ops（recalc_rate / round_rate / set_rate / enable / disable / set_parent）——CCF Core 通过这些 ops 操作硬件寄存器。
注册时钟的流程：clk_hw_register_fixed_rate()（固定频率——晶振） / clk_hw_register_pll() / clk_hw_register_divider() / clk_hw_register_gate() / clk_hw_register_mux() → 每个注册的时钟自动构建时钟树（parent/child 关系）。of_clk_add_hw_provider() 将整个 Provider 的设备树节点与这些 clk_hw 关联——Consumer 通过设备树找到 Provider 并获取时钟。

Consumer (时钟消费者) — 外设驱动侧

clk_get → clk_prepare_enable → 使用 → clk_disable_unprepare

外设驱动通过设备树中的 clocks 属性声明自己需要哪些时钟。驱动通过 CCF 的标准 Consumer API 获取和控制时钟——无需知道 Provider 的实现细节。
标准调用序列：
① clk = devm_clk_get(dev, "mipi") → 从设备树的 clock-names 中找 "mipi" → 解析对应的 clocks phandle → 获取 struct clk
② clk_set_rate(clk, 500000000) → 设置时钟频率为 500MHz（CCF 自动选择/配置父 PLL 和分频器来实现目标频率——如果硬件支持）
③ clk_prepare_enable(clk) → 使能时钟（先 prepare→可能睡眠，确保 PLL 锁定；再 enable→原子操作，打开门控）
④ 外设工作...
⑤ clk_disable_unprepare(clk) → 关闭时钟。

// CCF Provider 注册的完整调用链 (以 Rockchip RK3399 为例) // drivers/clk/rockchip/clk-rk3399.c — 初始化阶段 rk3399_clk_init() ├── clk_register_fixed_rate(NULL, "xin24m", NULL, 0, 24000000) │ // 固定 24MHz 晶振 — 所有时钟的根 │ ├── clk_register_pll("ppll", "xin24m", &pll_ops, params) │ // PLL: 24MHz → 200~2200MHz 可编程 VCO 输出 │ ├── clk_register_divider("clk_uart0_div", "ppll", CLK_SET_RATE_PARENT, reg, shift, width, 0, &lock) │ // 分频器: ppll/N → UART 需要的频率 │ ├── clk_register_gate("clk_uart0", "clk_uart0_div", CLK_SET_RATE_PARENT, reg, bit, 0, &lock) │ // 门控: 开/关 UART0 模块时钟 │ └── of_clk_add_hw_provider(np, of_clk_hw_onecell_get, &data) // 导出: Consumer 可通过 DT phandle + index 获取任意时钟

3. 设备树时钟绑定 — clocks / assigned-clocks

CCF 的 Consumer 和 Provider 之间的桥梁就是设备树的 clocks 属性——外设驱动不需要硬编码时钟名称或 phandle。

Consumer 侧 — 外设节点声明时钟需求

clocks + clock-names = 声明"我需要哪些时钟"

每个外设节点通过 clocks 属性引用 Provider 的时钟（通过 phandle + index），用 clock-names 给每个时钟起语义名。驱动通过 clk_get(dev, "spi") 按名称获取——不需要知道 Provider 内部 index。
示例：SPI 控制器需要 1 个功能时钟：
clocks = <&cru CLK_SPI0>;
clock-names = "spi";
示例：MIPI DSI 需要 3 个时钟：
clocks = <&cru PCLK_DSI>, <&cru SCLK_DSI>, <&cru MIPI_CLK>;
clock-names = "pclk", "sclk", "mipi_clk";

assigned-clocks — 启动时设置时钟频率/父时钟

不需要驱动代码——设备树直接设置默认频率

当外设需要特定的默认时钟频率而驱动代码又没有调用 clk_set_rate 时——通过 assigned-clocks + assigned-clock-rates 在Provider 注册完成后自动设置。内核在 of_clk_set_defaults() 中遍历并应用。
示例：I2S 音频需要 12.288MHz：
assigned-clocks = <&cru SCLK_I2S>;
assigned-clock-rates = <12288000>;
设置父时钟：
assigned-clock-parents = <&cru PLL_AUDIO>;
同时设置多个时钟：
assigned-clocks = <&cru SCLK_I2S>, <&cru MCLK_I2S>;
assigned-clock-rates = <12288000>, <12288000>;
提示：顺序必须一一对应。

⚠ assigned-clocks 的常见陷阱
1. 设置频率但不检查返回值——CCF 的 set_rate 可能不能精确达到目标频率（受限于 PLL 配置和分频器精度）。实际频率可能 12287999Hz（差 1Hz）——某些外设接受、某些不接受。
2. 顺序依赖——assigned-clocks 列表中的时钟按顺序处理。如果第 1 个时钟的父时钟需要第 2 个时钟先配置 → 处理失败。列表顺序 = 从根到叶。
3. 冲突——两个外设通过 assigned-clock-rates 设置了冲突的频率 → 后启动的外设覆盖前一个 → 前一个外设工作异常。为共享时钟指定 assigned-clock-rates 必须所有 Consumer 一致。

4. Consumer API — 驱动的正确使用姿势

这 6 个 API 覆盖了 95% 的时钟 Consumer 需求——理解每个函数的语义和调用上下文是关键。

API可睡眠典型延迟作用备注
clk_get(dev, name)✓<1μs从设备树获取时钟句柄推荐 devm_clk_get——自动释放
clk_prepare(clk)✓100-500μs准备时钟（PLL 锁定、电压稳定）可能睡眠——不能在中断上下文
clk_enable(clk)✗<1μs打开门控——时钟到达外设原子操作——可在中断上下文
clk_prepare_enable(clk)✓100-500μsprepare + enable 合二为一probe 中最常用的组合调用
clk_set_rate(clk, hz)✓~100μs设置时钟频率（含 PLL 重配置）可能改变父时钟——需 CLK_SET_RATE_PARENT
clk_get_rate(clk)✗<1μs查询当前频率返回缓存值——不读硬件寄存器
clk_round_rate(clk, hz)✓<1μs查询"能实现的最接近频率"不修改硬件——纯查询
⚠ prepare vs enable 为什么要分开？
prepare 可能睡眠——因为 PLL 锁定需要等待 100-500μs（休眠等待 vs CPU 忙等）。enable 是原子操作——只操作一个寄存器 bit（门控开关），不能在中断上下文中睡眠。
典型使用模式：probe 中 clk_prepare_enable（一次性准备+使能）→ 运行时 pm_runtime 只调 clk_enable/clk_disable（原子门控——快速进入/退出低功耗）→ remove 中 clk_disable_unprepare（完全关闭）。分开 prepare 和 enable 让 runtime PM 的延迟从 500μs 降到 1μs。

当外设工作不正常时——第一件事就是查时钟树。Linux 的 debugfs 提供了无与伦比的时钟调试能力。

6. IPC 相机时钟实战 — 从 Sensor 到 SD 卡的全景

IPC 相机涉及10+ 种不同的外设时钟——每种有独特的频率精度和抖动要求。理解 IPC 的完整时钟树是调试一切"外设偶尔异常"问题的起点。

Sensor MCLK24/27MHz 外部晶振级MIPI Sensor 的主时钟——必须由 SoC 的专用 CCI (Camera Control Interface) 时钟输出引脚提供。频率由 sensor 数据手册决定 (常见 24MHz)。频率偏差 > ±100ppm → 图像出现横纹 (PCLK 与 MCLK 的锁相环失锁)
MIPI D-PHY Clock200-800MHz 高速MIPI CSI-2 的 D-PHY HS 模式时钟——通常从 PLL_CPLL 或 PLL_VPLL 分频得到。频率 = Lane数 × 数据速率。D-PHY 对时钟的抖动 (jitter < 10ps RMS) 极其敏感——PLL 的环路滤波器带宽不足会导致数据眼图闭合→CRC 错误
ISP 核心时钟300-600MHzISP 硬件模块的处理时钟——直接影响 ISP 管线吞吐量。时钟过低→帧率下降；时钟过高→功耗增加但画质不提升（ISP 内部有固定延迟的流水线 stage——时钟太快只是更多空闲周期）
视频编码器时钟200-400MHzH.264/H.265 硬件编码器的核心时钟——编码 1080p@30fps 需要 ≥ 200MHz；4K@30fps 需要 ≥ 400MHz。时钟不足→编码帧输出延迟→环形 buffer 溢出→丢帧
I2S 音频时钟12.288/24.576MHz 精确音频 CODEC 的 MCLK (主时钟)——必须精确匹配采样率的 256× 或 512× 倍 (48kHz×256=12.288MHz)。频率偏差 > 100ppm → 音调偏移半音——人耳极度敏感。需要分数分频 PLL
SD/MMC 时钟400kHz 初始化 / 50MHz HSSD 卡的两个阶段用两种频率——初始化 ≤400kHz (开漏模式)，数据传输 ≤50MHz (推挽模式)。内核自动在 ACMD41 完成后调用 clk_set_rate 提升频率

关键设计原则

时钟源选择 + 频率等级 + 独立门控

① PLL 分配策略：一个 PLL 可以驱动多个子时钟——但总有一个 PLL 频率是所有子时钟的公倍数约束。如 PLL=1200MHz 可以分频出 600/400/300/200/150/100MHz——灵活分配各外设。
② 关键时钟独立 PLL：I2S 音频时钟必须独占一个 PLL（PLL_AUDIO）——因为音频频率（12.288/24.576MHz）与视频/系统频率（24/50/200MHz）没有简单的整数倍关系——共享 PLL 时一方频率调整会拖累另一方。
③ 门控省电：未使用的外设时钟必须关闭（clk_disable）——IPC 系统中 30-50% 的动态功耗来自不必要的外设时钟。

时钟调试 Checklist

按顺序排查——时钟问题是"外设不工作"的头号嫌疑

① cat /sys/kernel/debug/clk/clk_summary → 检查目标外设的 enable_cnt > 0？（=0→驱动没调 clk_enable）
② rate 是否符合预期？（!=→检查 assigned-clock-rates 是否被覆盖、或 clk_set_rate 是否失败）
③ 该外设的父时钟的 enable_cnt > 0？（父时钟没开→子时钟也无法工作——prepare/enable 不会自动向上传播，除非设了 CLK_SET_RATE_PARENT 标志）
④ dmesg | grep clk 查 CCF 错误——PLL 锁定失败、set_rate 返回 -EINVAL
⑤ 示波器测外设的 CLK 引脚——确认晶振/PLL 的输出频率、占空比、抖动
