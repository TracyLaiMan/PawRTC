1. 系统负载 = 什么？本质定义

系统负载（Load Average）是 Linux 内核统计的一个指数衰减移动平均值，表示一段时间内处于 可运行状态（R） 和 不可中断睡眠状态（D） 的平均进程数。它不直接等于 CPU 使用率——理解这个区别是正确诊断系统性能的第一步。

🔵 R 状态 — TASK_RUNNING

Running + Runnable

进程要么正在 CPU 上执行，要么在运行队列（runqueue）中排队等待 CPU。两种都算 R 状态。CPU 密集型负载主要推高 R 态任务数。ps aux | grep ' R' 可查看。

🟣 D 状态 — TASK_UNINTERRUPTIBLE

Uninterruptible Sleep

进程正在等待 I/O 完成（磁盘读写、网络请求、NFS 等待），在此期间不响应任何信号（即使用 kill -9 也无法杀死）。此设计的初衷是防止数据丢失——进程必须在 I/O 完成后才能被中断。

🟢 S 状态 — TASK_INTERRUPTIBLE

不计入负载

可中断睡眠是进程在等待事件（如网络数据、锁、定时器），但可以接收信号唤醒。这种等待不贡献负载。绝大多数"空闲"进程（如等待请求的服务端程序）处于 S 状态，不会推高 Load Average。

💡 关键洞察：为什么 Load Average 包含 D 状态？
D 状态进程虽然不占 CPU，但它们是系统吞吐量的瓶颈指示器。如果磁盘 I/O 是瓶颈，大量的 D 状态进程会导致系统"感觉"繁忙——CPU 可能空闲，但 I/O 子系统饱和，新的 I/O 请求必须排队。负载平均反映的是"系统资源争用"，而不仅仅是 CPU 资源争用。

2. 三个数字的含义：1分钟 · 5分钟 · 15分钟

每次看到 load average: 2.10, 1.85, 1.62 三个数字时，它们分别代表过去 1 分钟、5 分钟、15 分钟的指数衰减移动平均值。这三个数字在一起，可以判断负载是正在上升、下降还是保持稳定。

1 分钟
2.10 → 短期：高
5 分钟
1.85 → 中期：中
15 分钟
1.62 → 长期：偏低
趋势判断法则

1min < 5min < 15min
→ 负载在下降，高峰已过 ✓

1min > 5min > 15min
→ 负载在上升，需要关注 ⚡

1min ≈ 5min ≈ 15min
→ 负载稳定，稳态运行

📊 指数衰减原理

内核每 5 秒（LOAD_FREQ）采样一次 R + D 状态的进程数，然后用指数加权移动平均（EWMA）更新三个值。衰减因子：

1min 衰减: exp(-5/60) ≈ 0.9200
5min 衰减: exp(-5/300) ≈ 0.9834
15min衰减:exp(-5/900) ≈ 0.9944

衰减因子越接近 1，历史值的影响越大、变化越慢。15分钟值比 1 分钟值平滑得多。

⚠️ 1 分钟值的"欺骗性"
1 分钟平均值使用 12 个采样周期（12 × 5s = 60s），实际上63% 的权重集中在最近的 60 秒内，但残余影响可以延伸到过去数分钟。这意味着一个短促的 CPU 尖峰（如编译了几秒）会在 1 分钟值上留下几分钟的"尾巴"。不要仅凭 1 分钟值就断定系统过载。

3. 内核如何计算：从源码到 /proc/loadavg

Load Average 的计算发生在内核的 sched/loadavg.c 中，通过定时器周期性触发。理解这个过程可以让你明白为什么负载值"滞后但稳定"。

采样触发：sched_timer

kernel/sched/loadavg.c → do_timer()

每次时钟中断（tick）触发 do_timer()，累加 calc_load_tasks（当前 R + D 状态的任务数）。当累积达到 LOAD_FREQ（5s）时，调用 calc_global_load() 计算新的负载平均值。

核心公式：EWMA

指数加权移动平均
// loadnew = loadold * exp(-Δt/τ) + n * (1 - exp(-Δt/τ)) #define EXP_1 1884 // exp(-5/60) * 2^11 #define EXP_5 2014 // exp(-5/300) * 2^11 #define EXP_15 2037 // exp(-5/900) * 2^11 av1 = (av1 * EXP_1 + n * (2048 - EXP_1) + 1024) >> 11; av5 = (av5 * EXP_5 + n * (2048 - EXP_5) + 1024) >> 11; av15 = (av15 * EXP_15 + n * (2048 - EXP_15) + 1024) >> 11; // 全部使用定点整数运算，避免浮点开销

其中 n 是当前采样到的活跃任务数，τ 是时间常数（60s / 300s / 900s）。所有计算使用定点数（2^11 = 2048 缩放因子），高效且无浮点误差。

// ─── /proc/loadavg 的内容 ─── $ cat /proc/loadavg 2.10 1.85 1.62 2/1024 30456 │ │ │ │ │ │ │ │ │ │ │ └── 最近创建的 PID │ │ │ │ └────── 系统总线程数 │ │ │ └────────── 当前运行中的线程数（R 状态） │ │ └─────────────── 15 分钟负载平均 │ └──────────────────── 5 分钟负载平均 └───────────────────────── 1 分钟负载平均
🧠 为什么用指数平均而不是简单平均？
简单滑动窗口平均存在"悬崖效应"——当高负载时段刚好移出窗口时，平均值会突然下降。指数衰减平均的权重平滑过渡，近期的采样权重更大，历史的采样逐渐"淡出"，没有突兀的边界。同时内存开销极小——只需存 3 个数，无需缓存完整历史数据。

5. 查看负载的常用命令

Linux 提供了多种查看系统负载的方式——从最简单的 uptime 到功能丰富的 top/htop。

uptime

最简洁——一行知晓全局
# 系统运行时间 + 三值负载 $ uptime 14:32:01 up 45 days, 3:12, \ 2 users, \ load average: 0.52, 0.38, 0.29

适合快速健康检查的第一条命令。看到数字后立即除以 CPU 核数判断健康状况。

top / htop

动态查看 + 进程级详情
top - 14:32:15 up 45 days Tasks: 178 total, 1 running %Cpu(s): 2.3 us, 0.7 sy, 0.0 ni, 96.8 id, 0.1 wa Load Avg: 0.52, 0.38, 0.29

top 同时显示 CPU 使用率、I/O 等待和负载平均值。对比三个指标可以判断：高负载 + 低 CPU 使用率 = I/O 瓶颈。

/proc/loadavg

procfs——不需要命令
$ cat /proc/loadavg 0.52 0.38 0.29 1/178 24561

程序的直接数据源。许多监控工具（Prometheus node_exporter、Zabbix、Datadog Agent）都从这里读取负载数据。额外显示运行中/总线程数和最近 PID。

命令适用场景额外信息

6. CPU 密集型 vs I/O 密集型负载

Load Average 包含 R 和 D 两种状态，这两种不同的来源意味着相同的负载值可能有完全不同的原因和对策。区分它们是高负载诊断的核心任务。

CPU 密集型

高 R 状态 + 高 CPU 使用率

典型场景：数学计算、视频编码、编译、加密/解密、机器学习训练。

# 现象：CPU 使用率高 (id≈0%)，I/O 等待低 (wa≈0%) $ top %Cpu(s): 98.3 us, 1.7 sy, 0.0 ni, 0.0 id, 0.0 wa $ vmstat 1 procs: r=12 b=0 ← 大量就绪任务排队 cpu: us=98 id=0 wa=0

对策：优化算法 / 减少并发度 / 增加 CPU 资源 / 使用 nice 调整优先级 / cgroup CPU 限制

I/O 密集型

高 D 状态 + 低 CPU 使用率

典型场景：大量磁盘读写、数据库查询、NFS 挂载响应慢、日志刷盘、swap 抖动。

# 现象：CPU 空闲 (id 高)，但 I/O 等待 (wa) 显著 $ top %Cpu(s): 5.2 us, 2.1 sy, 0.0 ni, 68.5 id, 24.2 wa $ vmstat 1 procs: r=1 b=18 ← 大量 D 状态阻塞在 I/O cpu: us=7 id=68 wa=24

对策：升级存储（HDD → SSD / NVMe）/ 增加文件系统缓存 / 优化 SQL / 排查 NFS / 检查 swap 使用

# 列出系统中所有 D 状态进程——高负载排查利器 $ ps aux | awk '$8 ~ /D/' $ ps -eo pid,stat,wchan:32,comm | grep ' D' # wchan 列显示进程在内核中阻塞在哪个函数（如 blkdev_issue_flush、rpc_wait_bit_killable） # 查看每块磁盘的繁忙程度和等待队列 $ iostat -x 1 Device r/s w/s rkB/s wkB/s await %util sda 45 120 520 3840 8.5 72.3 # await：平均 I/O 响应时间(ms)；%util：磁盘繁忙百分比

7. 高负载诊断流程

当监控告警响起——负载飙升到异常水平时，按照下面的系统性流程快速定位根因。

1

确认负载水平

uptime; nproc; echo "负载/核比：" $(awk '{print $1}' /proc/loadavg) / $(nproc)

负载 / CPU 核数 的比值确定严重程度。比值 > 核数意味着大量任务在排队。

2

区分 CPU 瓶颈 vs I/O 瓶颈

vmstat 1 5 # 观察 r 列（R状态）和 b 列（D状态） top # 观察 %Cpu 行的 us/sy/id/wa 分布

高 r 值 → CPU 瓶颈；高 b 值 + 高 wa% → I/O 瓶颈。两种都高 → 混合型，分别处理。

3

定位罪魁祸首进程

# CPU 瓶颈场景 ps -eo pid,ppid,stat,%cpu,%mem,time,comm --sort=-%cpu | head -20 top -o %CPU -bn1 | head -20 # I/O 瓶颈场景——找到 D 状态进程和它们的 wchan ps -eo pid,stat,wchan:32,comm | grep ' D' iotop -o # 实时 I/O 排序（需 root）
4

深入根因

# CPU 高：perf 火焰图 / strace 系统调用分析 perf top -p <PID> strace -c -p <PID> # 统计系统调用耗时分布 # I/O 高：块层追踪 + NFS 诊断 iostat -x 1 # 磁盘级 I/O 统计 iotop -oP # 进程级 I/O 排序 cat /proc/<PID>/stack # 查看 D 状态进程的内核调用栈（需 root）
5

解决问题

CPU 瓶颈：调整进程优先级（nice/renice）、限制并发度（如 make -j 参数）、cgroup CPU 配额、代码优化
I/O 瓶颈：升级存储（NVMe）、增加内存做缓存、调整内核 I/O 调度器（mq-deadline → kyber）、排查 NFS 服务器、禁用 swap
通用/紧急：kill -STOP 暂停非关键进程、重启服务、迁移负载到其他节点

8. 常见误解与澄清

关于 Load Average 的误区非常普遍，即便经验丰富的工程师也常犯错。下面澄清最常见的几个。

❌ 误区：Load 高 = CPU 使用率高 这是最常见也是危害最大的误解。一个系统可以在 CPU 使用率不到 10% 的情况下 Load Average 达到 50+——只需大量的 D 状态进程（如 NFS 挂载无响应、磁盘 I/O 饱和）。负载 ≠ CPU 使用率。
❌ 误区：Load < 1.0 就安全 在 32 核机器上，Load 0.8 表示只用不到 3% 的 CPU 能力，极端安全。但在单核机器上，Load 0.9 意味着 90% 的 CPU 被占用，接近饱和。永远除以 CPU 核数再判断。
❌ 误区：负载值变化应该很灵敏 指数衰减平均天然滞后。1 分钟平均值的 63% 权重分布在最近 60 秒，但仍有约 37% 的权重落在更早的时间。一个只持续 5 秒的 CPU 尖峰可能需要 2-3 分钟才能从 1 分钟值中完全消失。
❌ 误区：htop 的 CPU 条满了就没救了 htop 是瞬时采样，top 的 CPU 百分比也是瞬时值。而 Load Average 是时间平滑的。可以出现 htop CPU 条 100% 但 Load 只有 0.5（短暂尖峰），也可以 Load 很高但 CPU 条很低（大量 D 状态任务）。
⚠️ systemd-oomd 的 Load 阈值陷阱
现代 Linux 发行版（Fedora、Ubuntu 22.04+）默认启用 systemd-oomd，它会监控 Load Average 并在超过阈值时杀进程。默认阈值 Load > CPU 核数 × 5。如果一台 8 核机器 Load 达到 40，systemd-oomd 会静默杀掉它认为"最应该死"的进程——而这个阈值对于高负载设计的生产服务可能太低了。通过 oomctl 调整策略或适当提高阈值。

📖 关键常量速查
LOAD_FREQ = 5 · 采样间隔（秒）
FSHIFT = 11 · 定点小数位数（2048 = 2^11）
EXP_1 = 1884 · exp(-5/60) × 2048
EXP_5 = 2014 · exp(-5/300) × 2048
EXP_15 = 2037 · exp(-5/900) × 2048
LOAD_INT(x) = x >> 11 · 取出整数部分
LOAD_FRAC(x) = ((x & (FSHIFT-1)) * 100) >> 11 · 取出小数部分（%）

12. 总结：记住这几点就够了

前面的所有内容浓缩成几个核心要点——理解这些，日常运维和调试就够用了。

① 负载 = R + D 的任务数

不是 CPU 使用率

Load Average 统计的是正在运行和等待 I/O 不可中断的进程数。一台 CPU 空闲的机器可以因磁盘或 NFS 问题而负载爆表。

② 除以核数再判断

1.0 在 1 核和 64 核机器上完全不同

Load 是未归一化的绝对值。load / 核数 < 0.7 安全，≈ 1.0 满载，> 1.0 有排队。永远先执行 nproc。

③ 三个数字看趋势

1min · 5min · 15min

1min > 5min > 15min = 负载在上升（关注）；反之在下降（高峰已过）；三者接近 = 稳态运行。结合趋势比看单一值更有意义。

④ 区分 CPU vs I/O

vmstat 的 r 列和 b 列

vmstat 1，r 高 = CPU 瓶颈，b 高 = I/O 瓶颈。ps aux | grep ' D' 找到 I/O 阻塞的进程，wchan 揭示具体阻塞位置。

⑤ 指数平均有惯性

不会立即反映瞬时尖峰

1 分钟平均需要约 5 分钟才能完全反映长期变化（约需 4-5 个时间常数）。不要因短时尖峰就判定系统过载——看 5min 和 15min 确认。

⑥ D 状态不可杀

kill -9 对 D 状态无效

TASK_UNINTERRUPTIBLE 是内核设计的保护机制——防止在 I/O 中途被杀死导致数据损坏。大量 D 状态进程堆积只能从根源解决（如恢复 I/O 通路）。
