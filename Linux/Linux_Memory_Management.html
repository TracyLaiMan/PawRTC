① kmalloc 是物理内存吗？

答：是的，物理连续

kmalloc 分配的内存在内核直接映射区——这片区域在系统启动时就建立了所有物理内存的 1:1 页表映射。kmalloc 返回的虚拟地址减去 PAGE_OFFSET 直接等于物理地址。不需要缺页中断，不创建 VMA。分配出来就是物理存在的、连续的页。

② vmalloc 是物理内存吗？

答：是的，但物理不连续

vmalloc 确实分配了物理内存——每次调用 alloc_page() 从 Buddy 获取一个真实物理页。但多个物理页不是连续的。vmalloc 通过修改内核页表，把这些分散的物理页映射到一个连续的虚拟地址范围。同样不创建 VMA。VMA 是用户空间的概念。

③ VMA 到底管什么？

答：只管用户空间的虚拟地址区域

VMA (vm_area_struct) 只用于用户进程。每个进程有自己的 mm_struct → 红黑树 VMA → 描述 text/heap/mmap/stack 等区域。内核自身的内存管理（kmalloc/vmalloc/slab）完全不经过 VMA。/proc/PID/maps 列出的是用户空间的 VMA，看不到内核分配。

💡 一句话总结
kmalloc = 内核用的，直接映射区，物理连续，速度快（VA→PA 减法即可），不走 VMA
vmalloc = 内核用的，vmalloc 区，物理不连续但虚拟连续（页表拼凑），开销大，不走 VMA
malloc = 用户程序用的，用户地址空间，延迟分配（首次访问缺页才分配物理页），必须经过 VMA + 缺页中断
三者最终都从 Buddy 获取物理页——区别在于虚拟地址空间的位置和是否需要 VMA。

1.7 深度追问 — 为什么返回虚拟地址而非物理地址？

既然 kmalloc/vmalloc 分配的都是物理内存，为什么返回的是虚拟地址？VMA 管理的用户空间也是虚拟地址——那整个系统中虚拟地址和物理地址到底是什么关系？

◈ 根本原因：MMU 一旦使能，CPU 只认识虚拟地址

CPU 只发出虚拟地址

从 bootloader 跳转到内核那一刻起

Linux 启动早期（head.S 汇编阶段）MMU 就必须使能。MMU 使能之后，CPU 核心里每一个取指、每一条 load/store 指令，地址总线上发出的都是虚拟地址。CPU 硬件自动把虚拟地址送给 MMU → 查 TLB → 命中的话翻译成物理地址 → 送到内存控制器。CPU 核心永远不知道也看不到物理地址。

物理地址只有 MMU 和 DMA 在用

C 指针 = 虚拟地址 = 永远

在内核 C 代码中，任何指针都是虚拟地址。即使 kmalloc 返回的那个 "va = pa + PAGE_OFFSET" 值——它仍然是虚拟地址，只不过碰巧和物理地址差一个常数偏移。MMU 仍然要翻译它（通过大页映射实现高效翻译）。只有 DMA 引擎、IOMMU、页表硬件（PTE 中存储的地址）才使用物理地址。

🎯 回答你的问题
Q: 为什么 kmalloc/vmalloc 返回虚拟地址而不是物理地址？
A: 因为 MMU 使能后 CPU 只能通过虚拟地址访问内存。C 代码里的任何指针都是虚拟地址。即使内核直接映射区的 "va = pa + 偏移"，它仍然是经过 MMU 翻译的虚拟地址——只是页表条目恰好在启动时按 1:1 偏移建立好了而已。物理地址只在两个地方使用：页表条目（PTE）中存储的值，以及 DMA 引擎/IOMMU 的总线地址。

Q: VMA 不就是虚拟地址吗？那 kmalloc 的虚拟地址也归 VMA 管？
A: VMA 只管理用户空间的虚拟地址。内核空间的虚拟地址（无论直接映射区还是 vmalloc 区）通过 init_mm（内核的 mm_struct）和 swapper_pg_dir（内核页表）管理，完全不用 VMA。VMA 是用户进程地址空间的 "登记簿"——分配用户虚拟地址空间时先登记 VMA，缺页时再查 VMA 分配物理页。内核没有这个 "先登记后缺页" 的过程——直接映射区启动时就建好了，vmalloc 区调用时当场建页表。

2. Buddy Allocator — 物理页管理器

Linux 物理内存分配的核心 — 以 2 的幂次块管理空闲页 · 分裂与合并 · 迁移类型防碎片

Buddy 算法原理

2^n 页块的分裂与合并

分配 (分裂): 从 free_area[order] 链表取空闲块 → 若无则向上级 order+1 请求并分裂 → 一半分配，一半入对应链表。
释放 (合并): 归还块 → 检查 buddy (邻近同大小块) 是否也空闲 → 合并为 order+1 → 递归向上合并。
MAX_ORDER = 11（内核常量，include/linux/mmzone.h:24），最大可分配阶为 MAX_ORDER-1 = 10 → 2^10 = 1024 页 = 4MB。free_area 数组有 MAX_ORDER 个元素 (索引 0~10)。

迁移类型 (反碎片)

5 种迁移类型隔离页面

MIGRATE_UNMOVABLE — 内核分配 (slab, page table)
MIGRATE_MOVABLE — 用户空间页 (可迁移)
MIGRATE_RECLAIMABLE — 页缓存 (可回收)
MIGRATE_CMA — 连续内存区预留
MIGRATE_HIGHATOMIC — 原子上下文紧急分配
同类页面聚合 + 压缩 (compaction) = 生成高阶连续块。

/* ===== struct free_area — buddy 每个 order 的空闲链表 ===== */ struct free_area { struct list_head free_list[MIGRATE_TYPES]; // 每种迁移类型一个链表 unsigned long nr_free; // 此 order 的空闲块总数 }; struct zone { struct free_area free_area[MAX_ORDER]; // order 0~(MAX_ORDER-1) 的空闲链表数组, MAX_ORDER=11 → 11 个 unsigned long watermark[NR_WMARK]; // min, low, high 水位线 struct per_cpu_pages __percpu *per_cpu_pageset; // Per-CPU 单页缓存 atomic_long_t managed_pages; // 此 zone 管理的总页数 }; /* ===== 分配页的 GFP 标志 (Get Free Pages) ===== */ /* ═══ 实际 GFP 标志组合 (include/linux/gfp.h:237-248) ═══ */ GFP_KERNEL = __GFP_RECLAIM | __GFP_IO | __GFP_FS // 标准内核分配 — 可回收、可 I/O、可访问文件系统 GFP_ATOMIC = __GFP_HIGH | __GFP_ATOMIC | __GFP_KSWAPD_RECLAIM // 原子上下文 — 不可睡眠，可使用紧急预留，不可 direct reclaim GFP_USER = __GFP_RECLAIM | __GFP_IO | __GFP_FS | __GFP_HARDWALL // 用户空间分配 — 遵守 cpuset 硬墙限制 GFP_HIGHUSER_MOVABLE = GFP_HIGHUSER | __GFP_MOVABLE // 用户空间可移动页 — page fault 缺页分配的默认标志 __GFP_RECLAIM = __GFP_DIRECT_RECLAIM | __GFP_KSWAPD_RECLAIM // 允许 direct reclaim + kswapd reclaim __GFP_ZERO // 返回清零的页
Per-CPU Page Cache (PCP)
对于最常见的 order-0 单页分配，buddy 通过 Per-CPU 缓存避免锁争用。每个 CPU 预取一批页缓存本地 → 分配从 CPU 本地取 (无锁) → 缓存耗尽时批量从 buddy 补充。这是多核环境 page_alloc 高性能的关键。

2.5 迁移类型 (Migrate Types) — 反碎片的核心机制

如果所有类型的页面混在一起，运行一段时间后系统会碎片化——虽然有大量空闲页，但无法分配高阶连续块（如 2MB THP 大页）。迁移类型通过按可移动性分类存放来解决这个问题。

◈ 碎片化问题：为什么需要分类？

无迁移类型：灾难场景

所有页混在一起

物理内存中 不可移动的页（内核 slab） 和 可移动的页（用户空间匿名页） 随机交错分布。即使有 50% 空闲内存，一个不可移动的 4KB 内核页挡在中间 → 无法合并出连续 2MB 高阶块。这就是 "外部碎片"——有空闲但不可用的内存。
[slab][free][anon][free][slab][free][anon]... → 无法得到连续 2MB

有迁移类型：解决方案

同类相聚，按需迁移

将物理页按 "能不能被移动" 分到不同迁移类型的空闲链表中。
① UNMOVABLE 页面聚在一起 → 它们占用的区域"牺牲"给内核
② MOVABLE 页面聚在一起 → 需要大块时，把这些页迁移到其他物理地址，腾出连续空间
③ 内存压缩 (compaction) 只在 MOVABLE 区域工作 → 高效生成高阶块。

◈ 五种迁移类型详解

迁移类型可移动？谁来用典型分配者GFP 标志映射
MIGRATE_UNMOVABLE ✗ 不可移动 内核核心分配 slab (kmalloc) · 页表 (PTE) · 内核栈 · vmalloc 页表 · 内核代码/数据 GFP_KERNEL · GFP_ATOMIC 等大部分内核 GFP 标志
MIGRATE_MOVABLE ✓ 可移动 用户空间页 匿名页 (堆/栈/mmap ANON) · 进程页表用户部分 GFP_HIGHUSER_MOVABLE
MIGRATE_RECLAIMABLE △ 可回收 文件缓存 Page Cache · inode/dentry 缓存 · tmpfs 文件内容 __GFP_RECLAIM
MIGRATE_CMA ✓ 可移动 CMA 预留区域 设备驱动（相机/视频解码/GPU）需要连续大块 DMA 缓冲区时使用

MIGRATE_HIGHATOMIC ✗ 紧急预留 原子上下文 中断处理中的紧急分配 · 网络数据包处理 (softirq) GFP_ATOMIC（order>0 时优先从此取）

UNMOVABLE 为什么不可移动？

内核地址是硬编码的

内核代码中的指针指向具体的物理地址（通过直接映射区的 1:1 对应）。slab 对象、task_struct、内核栈等分配后，内核到处持有这些物理地址的引用。要"移动"它们意味着 扫描整个内核空间更新每一个指针——不现实。所以这些页面标记为不可移动。

MOVABLE 怎么移动？

用户页通过页表间接访问

用户空间的页通过页表 (PTE) 间接访问——进程看到的是虚拟地址。要移动一个物理页：
① 分配新物理页并拷贝数据
② 更新 PTE 指向新物理地址
③ 刷新 TLB
④ 释放旧页
进程完全无感知，不需要修改任何用户代码或指针。

RECLAIMABLE 怎么回收？

文件页有后备存储

Page Cache 中的文件页有磁盘上的原始副本（如果是干净页）或可以先回写到磁盘（如果是脏页）。回收时：干净页 → 直接丢弃（需要时再从磁盘读）；脏页 → 先回写再丢弃。不需要拷贝到新位置。

◈ 后备回退规则 (Fallback Order)

当某迁移类型的空闲页耗尽时，Buddy 不会直接返回失败——它会按预设的优先级顺序从其他类型"借用"页面。这个顺序被精心设计以最小化碎片化。

/* ===== mm/page_alloc.c: fallbacks[][] — 每种迁移类型的后备列表 ===== */ static int fallbacks[MIGRATE_TYPES][3] = { // UNMOVABLE 耗尽时 → 先借 RECLAIMABLE → 再借 MOVABLE [MIGRATE_UNMOVABLE] = { MIGRATE_RECLAIMABLE, MIGRATE_MOVABLE }, // MOVABLE 耗尽时 → 先借 RECLAIMABLE → 再借 UNMOVABLE [MIGRATE_MOVABLE] = { MIGRATE_RECLAIMABLE, MIGRATE_UNMOVABLE }, // RECLAIMABLE 耗尽时 → 先借 UNMOVABLE → 再借 MOVABLE [MIGRATE_RECLAIMABLE] = { MIGRATE_UNMOVABLE, MIGRATE_MOVABLE }, };

回退规则的设计哲学

核心原则：尽量不污染可移动区域

① 可回退的优先级高于不可回退的——RECLAIMABLE 和 MOVABLE 可以通过回收/迁移腾出空间，UNMOVABLE 不行。所以 fallback 优先选择 "可以抢救" 的页面避免累积碎片。
② 借用 UNMOVABLE 页面风险最大——如果 MOVABLE 分配借了 UNMOVABLE 的页，这块物理区域从此被"钉死"，再也无法用于生成大块连续内存。所以 MOVABLE 的回退链将 UNMOVABLE 放在最后。
③ CMA 被隔离对待——CMA 地块专门留给 DMA 大块需求，普通分配默认不进 CMA 区域（除非显式使用 __GFP_CMA）。

回退过量时的后果

碎片化的累积

当系统长期运行且内存压力大时，回退频繁发生 → UNMOVABLE 页面逐渐渗入 MOVABLE 区域 → 可移动区域被污染 → 即使有足够空闲页，compaction 也无法在 MOVABLE 区域生成连续大块 → THP 分配失败率上升 → 性能下降。
监控指标：/sys/kernel/debug/extfrag/unusable_index 反映各 order 的碎片化程度。

◈ GFP 标志如何决定迁移类型

调用者传递的 GFP 标志会自动映射到合适的迁移类型——调用者通常不需要关心迁移类型，内核帮你做正确的选择。

/* ===== include/linux/gfp.h: GFP 标志 → 迁移类型映射 ===== */ static inline int gfp_migratetype(const gfp_t gfp_flags) { // __GFP_MOVABLE 显式指定 → MOVABLE if (unlikely(gfp_flags & __GFP_MOVABLE)) return MIGRATE_MOVABLE; // __GFP_RECLAIMABLE → RECLAIMABLE if (unlikely(gfp_flags & __GFP_RECLAIMABLE)) return MIGRATE_RECLAIMABLE; // 默认 → UNMOVABLE（大多数内核分配走这里） return MIGRATE_UNMOVABLE; } /* ===== 实例：常见分配对应的迁移类型 ===== */ kmalloc(size, GFP_KERNEL) // → MIGRATE_UNMOVABLE alloc_pages(GFP_HIGHUSER_MOVABLE) // → MIGRATE_MOVABLE (用户空间页) alloc_pages(__GFP_RECLAIMABLE) // → MIGRATE_RECLAIMABLE (page cache) alloc_pages(GFP_ATOMIC, order=2) // → MIGRATE_HIGHATOMIC (原子+高阶, 紧急预留)

◈ 内存压缩 (Memory Compaction) — 把碎片整理成连续块

Compaction 的工作原理

双游标扫描 — 从两端向中间推进

左边游标 (migrate scanner): 从区域起始向后扫描，找到已分配的 MOVABLE 页。
右边游标 (free scanner): 从区域末尾向前扫描，找到空闲页。

过程: 左游标找到的 MOVABLE 页 → 拷贝到右游标找到的空闲页 → 更新页表 → 释放原位置 → 原位置变为空闲页 → 和一墙之隔的空闲页合并 → 逐步聚拢出连续大块。

UNMOVABLE 页无法迁移 → 左游标遇到 UNMOVABLE 跳过 → 这些页成为"钉子"，阻止该位置产生连续块。

Compaction 的触发时机

自动 (kcompactd) + 手动 (分配路径中)

① kcompactd[N] — 后台内核线程，每个 NUMA 节点一个。监控碎片指数，在问题恶化之前主动压缩。使用 sysctl vm.compact_memory 控制。
② 直接压缩 (direct compaction) — 当 alloc_pages() 分配高阶块（如 THP 2MB）失败时，在分配者进程上下文中同步执行压缩。阻塞分配者，类似直接回收。
③ 手动触发: echo 1 > /proc/sys/vm/compact_memory

UNMOVABLE — 内核的"钉子"

不可移动 · 占住就不走

kmalloc/slab/页表/内核栈都属于此类。
默认 GFP 标志（GFP_KERNEL 等）全部落入 UNMOVABLE。
这些页面越多 → 可生成连续大块的区域越小。

MOVABLE — 可迁移的"活页"

用户空间按需腾挪

匿名页/用户栈/堆。
compaction 的目标就是这类页。
通过页表重映射迁移 → 进程无感。
GFP_HIGHUSER_MOVABLE 显式请求。

RECLAIMABLE — 可丢弃的"缓存"

有后备存储的页

Page Cache / inode 缓存。
干净页直接丢弃 → 脏页先回写再丢弃。
回收比迁移更轻量（不需拷贝数据）。
__GFP_RECLAIMABLE 显式请求。

3. SLUB 分配器 — 内核对象缓存

Buddy 之上的一层 · 管理固定大小的内核对象 · kmalloc 的底层引擎 · slab 家族现代实现

slab / slub / slob

三代内核对象分配器

slab (Solaris 起源) — 复杂但功能全，多队列管理开销大
slub (2007, 当前默认) — 简化设计，无队列，低 CPU 开销
slob (已移除) — 极简，嵌入式专用
SLUB 通过 struct kmem_cache 管理每种大小的对象缓存，底层以页为单位从 buddy 获取，按对象大小切分为 slot。

kmem_cache 机制

每类对象一个专用缓存

kmem_cache_create(name, size, ...) — 创建对象缓存
kmem_cache_alloc(cachep, flags) — 分配对象
kmem_cache_free(cachep, objp) — 释放对象
内核预建常见大小的通用缓存 (8B→8KB)，kmalloc() 直接查表使用这些通用 cache，避免逐次创建。

vmalloc — 虚拟连续

非物理连续的大块分配

vmalloc(size) 分配虚拟地址连续但物理页可能分散的内存。内部逐个调用 alloc_page() 获取单个物理页，然后修改页表建立连续映射。
缺点: 需修改页表 + TLB 刷新 → 开销高。仅用于大块分配 (≥ 几 MB) 或需要虚拟连续但物理不连续的场合。

4. 虚拟内存 — VMA · 页表 · MMU

进程视角的地址空间 · vm_area_struct 管理区域 · 页表转换虚拟到物理 · 缺页中断

vm_area_struct (VMA)

进程虚拟地址空间的基础单元

每个进程的地址空间由多个 VMA 组成，用红黑树 + 链表管理。VMA 描述一段连续的虚拟地址范围 (vm_start → vm_end) 及其属性：权限、映射文件、匿名标志。
cat /proc/<PID>/maps — 查看进程的 VMA 列表。
VMA 不存储数据 — 数据在 page 中，VMA 只描述区域属性。

缺页中断 (Page Fault)

虚拟内存的核心机制

CPU 访问虚拟地址 → MMU 查 TLB → miss → 遍历页表 → PTE 为空 → 触发缺页异常。
Minor fault: 页已在内存 (映射到页缓存)，只需更新页表。
Major fault: 页不在内存，需从磁盘读取 (swap in 或读文件)。
COW fault: fork 后写时复制。

/* ===== VMA 关键字段 ===== */ struct vm_area_struct { unsigned long vm_start; // 起始虚拟地址 unsigned long vm_end; // 结束虚拟地址 struct vm_area_struct *vm_next; // 链表 (线性遍历) struct mm_struct *vm_mm; // 所属进程的 mm pgprot_t vm_page_prot; // 页保护位 (r/w/x) unsigned long vm_flags; // VM_READ | VM_WRITE | VM_EXEC | VM_MAYSHARE... struct file *vm_file; // 映射的文件 (文件映射) unsigned long vm_pgoff; // 文件内偏移 (页单位) const struct vm_operations_struct *vm_ops; // 操作回调 (open/close/fault) }; /* ===== 页表层级 (ARM64 4 级页表) ===== */ 虚拟地址 (48-bit): | PGD[9] | PUD[9] | PMD[9] | PTE[9] | 页内偏移[12] | └── L0 ──┘── L1 ──┘── L2 ──┘── L3 ──┘── offset ──┘
TLB — 页表查找的硬件缓存
Translation Lookaside Buffer 是 MMU 内部的高速缓存，存储最近使用的虚拟→物理地址映射。TLB 命中 → 零周期延迟；TLB miss → 需遍历 4 级页表 (数十周期)。进程切换时 ASID (Address Space ID) 机制避免刷新整个 TLB，提升上下文切换性能。

5. 用户空间内存管理 — malloc · mmap · 进程布局

从用户空间看内存 — 进程地址空间布局 · brk vs mmap · ptmalloc2 分配器 · arena/bin/chunk 体系

进程地址空间布局 (64-bit)

从低地址到高地址

Text (代码): 只读可执行指令
Data / BSS: 全局/静态变量
Heap: 动态分配，brk() 向上增长
mmap 区域: 共享库 + 匿名映射，向下增长
Stack: 局部变量，向下增长
内核空间: 高地址，用户不可访问
64-bit 地址空间巨大 (48-bit = 256TB)，heap 和 mmap 之间几乎无限。

brk vs mmap — 两种分配路径

内核向用户空间提供的原始接口

brk/sbrk: 调整 program break，使堆连续向上增长。简单但只能从顶部释放 → 碎片化后内存无法归还 OS。glibc 仅用于小块分配。
mmap/munmap: 在地址空间任意位置映射页。灵活、即时归还。大块分配 (≥128KB) 直接走 mmap。线程 arena 也通过 mmap 创建。

延迟分配 — 虚而不实

虚拟 ≠ 物理

malloc 和 mmap 只分配虚拟地址空间，不分配物理页。首次访问时触发缺页中断 → 内核分配物理页并建立映射。这就是为什么 malloc 几乎不会失败，但 OOM 随时可能在后来的写入中爆发。Linux 的过度承诺 (overcommit) 策略允许虚拟内存远超物理内存。

glibc ptmalloc2 — 总览

基于 dlmalloc 的多线程版本

glibc 的 malloc 实现称为 ptmalloc2。它在应用和内核之间建立缓存层，减少系统调用频率。核心策略：
① 小块 (<128KB) → 从 heap (brk) 分配，free 后缓存到 bin
② 大块 (≥128KB) → 直接 mmap，free 时立刻 munmap
③ 多线程 → 创建多个 arena (每 arena 独立锁)，arena 数 ≤ 8×CPU

Arena — 分配区

解决多线程锁竞争的关键

main_arena: 主线程使用，通过 brk 扩展 (唯一用 brk 的 arena)。
thread_arena: 当线程争用 main_arena 锁时，glibc 通过 mmap 创建新 arena。每个 arena 独立管理自己的 bin 链表和 top chunk，线程间减少锁竞争。但 arena 过多会导致内存碎片化。

Chunk — 内存块单元

malloc_chunk — 元数据 + 用户数据

每次 malloc/free 操作的对象是 chunk。chunk 有 16 字节对齐 (64-bit)，最小 32 字节。
已分配 chunk: [prev_size(复用)] [size|flags] [用户数据…]
空闲 chunk: [prev_size] [size|flags] [fd] [bk] [fd_nextsize] [bk_nextsize]
size 的低 3 位：P (前一个 chunk 在用)、M (mmap 分配)、A (非主 arena)

Bins — 空闲链表体系

5 类 bins = 128 个链表

Fast bins (10 个): ≤128B，单向 LIFO，不合并 — 极速分配
Small bins (62 个): 32B~1008B，双向 FIFO，精确大小，会合并
Large bins (63 个): ≥1KB，双向 + 跳表，best-fit 降序
Unsorted bin (1 个): 最近释放的缓存池，分配时扫描并按大小归类
tcache (76 个, glibc 2.26+): 每线程的快速缓存，每链最多 7 个 chunk，无锁

Top Chunk & 扩展

堆顶的万能备胎

所有 bin 都无合适 chunk → 从 top chunk (arena 最高地址的空闲区) 分割。
top chunk 不够 → main_arena 调用 sbrk() 扩展；thread_arena 调用 mmap() 创建新 sub-heap。
top chunk 超过 128KB 时空闲部分调用 sbrk(-size) 或 madvise(DONTNEED) 归还 OS。

5.5 malloc 的内核路径 — glibc 到 Buddy 的完整调用链

前面 Ch1.5 的框图中画了 malloc → ptmalloc2 → brk/mmap → page fault → alloc_pages 的路径。这里把每一步具体的内核代码串起来。

◈ 路径 1：小块分配 < 128KB — brk() → 缺页中断

/* ═══ Step 0: glibc 决定走 brk 路径 ═══ */ malloc(100) → tcache miss → bins miss → top chunk 不够 → sysmalloc() → sbrk(4096) // 请求扩展堆 4KB /* ═══ Step 1: 内核 brk() 系统调用 — mm/mmap.c ═══ */ SYSCALL_DEFINE1(brk, unsigned long, brk) { // 1. 找到当前进程的 mm_struct struct mm_struct *mm = current->mm; // 2. 新 brk 必须 > 代码段结束地址 & 不能和 mmap 区域冲突 if (brk < min_brk || brk > mm->mmap_base) goto out; // 3. 对齐到页边界 newbrk = PAGE_ALIGN(brk); oldbrk = PAGE_ALIGN(mm->brk); // 4. 扩展 heap VMA: [heap] 区域的 vm_end 增大 if (newbrk > oldbrk) { do_brk(oldbrk, newbrk - oldbrk); // ★ 关键：do_brk() 只修改 VMA 边界，不分配物理页！ // VMA: vm_start=0x1000000, vm_end 从 0x1001000 → 0x1002000 } mm->brk = brk; // 更新 program break } // 返回后：heap VMA 扩大了，但新扩展的虚拟地址页表全为空 // → 首次访问这些地址时触发缺页中断 → 才真正分配物理页
/* ═══ Step 2: 缺页中断 — 首次访问 brk 扩展的地址 ═══ */ /* ═══ arch/arm64/mm/fault.c 或 arch/x86/mm/fault.c → do_page_fault() ═══ */ static int __kprobes do_page_fault(struct pt_regs *regs, unsigned long addr) { struct vm_area_struct *vma; struct mm_struct *mm = current->mm; down_read(&mm->mmap_sem); // 1. 查找 addr 属于哪个 VMA vma = find_vma(mm, addr); if (!vma) goto bad_area; // 段错误！地址不属于任何 VMA if (vma->vm_start <= addr && addr < vma->vm_end) { // 2. 调用 handle_mm_fault → __handle_mm_fault → handle_pte_fault int ret = handle_mm_fault(mm, vma, addr, flags); if (ret & VM_FAULT_OOM) goto out_of_memory; // 连缺页分配都失败了 → 触发 OOM } up_read(&mm->mmap_sem); } /* ═══ mm/memory.c — handle_pte_fault() ═══ */ static int handle_pte_fault(struct mm_struct *mm, struct vm_area_struct *vma, ...) { if (!pte_present(entry)) { if (pte_none(entry)) return do_anonymous_page(mm, vma, address, ...); // ↑ 匿名页 (heap/stack/mmap ANON) 走这里 else return do_swap_page(mm, vma, address, ...); // ↑ 换出的页走这里 (swap in) } } /* ═══ mm/memory.c — do_anonymous_page() ═══ */ static int do_anonymous_page(struct mm_struct *mm, struct vm_area_struct *vma, unsigned long address, ...) { // 1. 从 Buddy 分配一个物理页 (order-0, MOVABLE) page = alloc_zeroed_user_highpage_movable(vma, address); // └→ alloc_pages(GFP_HIGHUSER_MOVABLE, 0) // └→ __alloc_pages_nodemask() // ├→ get_page_from_freelist() (fast path) // └→ __alloc_pages_slowpath() (slow path, 可能触发回收/OOM) // 2. 建立 PTE: 把虚拟地址 → 物理页的映射写入进程页表 set_pte_at(mm, address, page_table, entry); // 3. 清零页 → 返回用户态 → cpu 重试访问指令 → 成功！ // 此后进程中对该地址的访问通过 TLB 直接命中，不再触发缺页 }

◈ 路径 2：大块分配 ≥ 128KB — mmap(MAP_ANONYMOUS)

/* ═══ Step 0: glibc 决定走 mmap 路径 ═══ */ malloc(200*1024) // 200KB ≥ 128KB (MMAP_THRESHOLD) → sysmalloc() → mmap(NULL, 200*1024, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANONYMOUS, -1, 0) /* ═══ Step 1: mm/mmap.c — do_mmap() ═══ */ unsigned long do_mmap(struct file *file, unsigned long addr, unsigned long len, unsigned long prot, unsigned long flags, ...) { // 1. 在进程地址空间中找到 len 大小的空闲虚拟地址范围 addr = get_unmapped_area(file, addr, len, ...); // 2. 创建新的 VMA: vm_area_struct vma = kmem_cache_zalloc(vm_area_cachep, GFP_KERNEL); vma->vm_start = addr; vma->vm_end = addr + len; vma->vm_flags = VM_READ | VM_WRITE | ...; vma->vm_page_prot = vm_get_page_prot(vma->vm_flags); // 3. 将 VMA 插入进程的 mm_struct 红黑树 vma_link(mm, vma, prev, rb_link, rb_parent); } // ★ 关键：mmap 也只创建 VMA，不分配物理页！ // 后续首次访问 → 缺页 → do_anonymous_page → alloc_pages // 和 brk 路径的缺页中断完全一样

◈ free() 的内核路径 — 物理页如何被归还

/* ═══ 场景 A: 大块 — mmap 分配 → munmap 归还 ═══ */ free(mmap_ptr) // ptr 来自 mmap (chunk size 的 IS_MMAPPED 标志位=1) → munmap_chunk(p) → munmap(ptr, size) // 系统调用 → mm/mmap.c do_munmap() → unmap_region(mm, vma, start, end) → unmap_page_range() // 遍历页表，清除 PTE → free_pages_and_swap_cache() // ★ 释放物理页回 Buddy → free_one_page() → __free_one_page() → Buddy 分裂/合并 // 物理页回到 pageblock 当前类型的 freelist → remove_vma(vma) // 从红黑树中删除 VMA → 物理页立即归还 Buddy！ /* ═══ 场景 B: 小块 — brk 分配 → free → chunk 合并 → trim 归还 ═══ */ free(brk_ptr) // ptr 来自 brk (heap) → 放入 glibc bin (tcache/fastbin/unsorted) → glibc 不立即归还给内核——缓存起来给后续 malloc 用 → 只有当: ① malloc_trim() 被显式调用，或 ② top chunk 超过 128KB → glibc 调用 sbrk(-size) 收缩 heap → 内核 do_brk() 反向收缩 VMA → 释放物理页回 Buddy → 但 heap 碎片化 → top chunk 下面有已分配 chunk → 无法收缩 → ★ 这就是 brk 路径"碎片不归还 OS"的根因
📐 malloc/free 内核路径总结
· malloc: glibc bins → sysmalloc → brk/mmap (仅扩展 VMA，0 个物理页) → 首次访问 → 缺页中断 → alloc_pages (才分配物理页)
· free (mmap 大块): munmap → 清页表 → free_one_page → 物理页立即归还 Buddy
· free (brk 小块): 放入 glibc bin → 物理页暂不归还 → 等 trim 或 top chunk 收缩才归还 → 碎片可能导致永久不归还
· 缺页路径: do_page_fault → find_vma (验证地址合法) → handle_mm_fault → do_anonymous_page → alloc_page → set_pte

6. Page Cache — 文件数据的缓存

所有文件 I/O 的必经之路 · address_space 管理 · folio 统一单页和大页 · 写回的决策引擎

address_space

每个 inode 一个

管理一个文件在内存中的所有缓存页。i_pages (xarray) 将文件偏移映射到 struct folio。a_ops 函数表定义 readpage/writepage/write_begin/write_end 的实现。读取时先查 xarray → hit 零拷贝 → miss 分配+读磁盘。

folio — 现代页抽象

取代 struct page 的复合页

struct folio (5.16+) 统一了单页和大页的接口。每个 folio 含 1 或 2^n 个 page (透明大页为 512 页 = 2MB)。统一语义：一个 folio = 页缓存的一个连续内存单元，消除了 compound page 管理的混乱。

脏页与回写

SetPageDirty → flusher

write() 只修改页缓存并标记 SetPageDirty，不直接写磁盘。后台 flusher 线程 ([writeback]) 在三种条件下写回：① 脏页超 dirty_background_ratio ② 脏页老化超 dirty_expire_interval ③ 显式 sync/fsync。

/* ===== address_space — 页缓存的核心管理对象 ===== */ struct address_space { struct inode *host; // 所属 inode struct xarray i_pages; // ⭐ folio 索引: 文件偏移 → folio pgoff_t writeback_index; const struct address_space_operations *a_ops; unsigned long nrpages; // 缓存页数 unsigned long nrexceptional; }; /* ===== a_ops — 页缓存与文件系统的桥接 ===== */ struct address_space_operations { int (*read_folio)(struct file *, struct folio *); int (*writepages)(struct address_space *, struct writeback_control *); int (*write_begin)(struct file *, struct address_space *, loff_t, unsigned, struct folio **, void **); int (*write_end)(struct file *, struct address_space *, loff_t, unsigned, unsigned, struct folio *, void *); };

7. 内存回收 — kswapd · LRU · 三级递进响应

水位线触发链: kswapd 后台回收 → 直接回收 (阻塞分配进程) → OOM Killer (杀进程)

水位线 (Watermark)

三个阈值触发的递进回收

pages_high (150% min): kswapd 停止回收的目标
pages_low (125% min): kswapd 唤醒的触发线
pages_min (100%): 紧急 — 仅内核可用，触发直接回收
基准值 vm.min_free_kbytes → 自动按内存比例计算。

LRU 四链表

Active/Inactive × Anon/File

Active(anon): 活跃匿名页 → swap
Inactive(anon): 非活跃匿名页 → swap
Active(file): 活跃文件页 → 回写/丢弃
Inactive(file): 非活跃文件页 → 回收
访问时 inactive→active 提升；未访问时 aging 降级；回收从 inactive 开始。
MGLRU (Linux 6.1+): 代际 LRU，按出生代淘汰，大系统回收更精准。

直接回收 (Direct Reclaim)

分配者自己被阻塞做回收

当 fast path 失败 + kswapd 来不及回收时，__alloc_pages_slowpath() 在分配进程上下文中同步执行 try_to_free_pages() → 扫描 LRU → 回收页 → 收缩 slab。进程直接承担回收成本——这是为什么"内存不足时系统变慢"的根因。

8. Swap — 匿名页的二级存储

当 RAM 不足时将匿名页换出到磁盘 · swap in/out · SSD swap 的性能影响

Swap 生命周期

匿名页的换出与换入

① 换出 (swap out): kswapd/直接回收选中 inactive(anon) 页 → 写入 swap 设备 → 释放物理页 → PTE 标记为 swap entry
② 换入 (swap in): 进程访问被换出的地址 → 缺页异常 → 识别 PTE 中的 swap entry → 从 swap 设备读回 → 重新放入物理内存。
swap 设备可以是磁盘分区或文件 (swapon /swapfile)。

交换缓存 (Swap Cache)

避免重复换入/换出

被换出的页在 swap 设备和 page cache 之间多了一个swap cache。如果进程再次访问 → swap cache hit → 无需从磁盘读取。如果另一个进程同时换入同一页 → swap cache 确保只有一次 I/O。
/proc/meminfo 中的 SwapCached 即为此。

/* ===== Swap 操作 ===== */ # 创建 swap 文件 dd if=/dev/zero of=/swapfile bs=1G count=4 mkswap /swapfile swapon /swapfile // 启用 swap swapoff /swapfile // 关闭 swap # 查看 swap 使用情况 cat /proc/swaps // 所有 swap 设备 free -h // Swap: total/used/free cat /proc/meminfo | grep 'Swap\|Shmem' /* ===== 内核 swap 路径 ===== */ Swap Out: kswapd → shrink_lruvec → try_to_unmap() → pageout() → swap_writepage() Swap In: page fault → do_swap_page() → lookup_swap_cache() → swap_readpage()
⚠️ swap 的正确认知
swap 不是内存不够时的"慢速应急方案"——它是正常的内存管理策略。即使有足够 RAM，swap 也有价值：① 换出极少使用的守护进程页面，腾出物理内存给 page cache（提升文件 I/O 性能）② 给 OOM Killer 争取决策时间。完全禁用 swap (swappiness=0) 会让回收器没有匿名页可以回收，在内存紧张时直接触发 OOM。

9. OOM Killer — 最后手段

当所有回收手段都失败时 · 按 badness 评分选受害者 · SIGKILL 结束进程

受害进程选择算法

badness = f(RSS, oom_score_adj, ...)

select_bad_process()（line 300）遍历所有进程调用 oom_badness()，选 points 最高且可杀的进程 → oom_kill_process() → send_sig(SIGKILL)。被杀的进程获得 TIF_MEMDIE 标志，可以使用预留内存完成退出。如果找不到任何可杀进程（所有进程都被豁免）→ panic("Out of memory and no killable processes")。

oom_reaper — 异步收割

防止被杀进程阻塞 OOM 恢复

被杀进程可能持有锁而无法快速退出。oom_reaper 内核线程异步收割被杀进程的匿名页和页表，立即释放大部分物理内存，让系统尽快恢复。不需要等待进程完全退出。
cat /proc/<PID>/oom_score — 查看当前 badness 评分 (越高越危险)。
echo -1000 > /proc/<PID>/oom_score_adj — 保护关键进程永不被杀。

◈ 三个关键规则——解释了 OOM 看似"不公平"的行为

规则 1: 高阶分配不触发 OOM

order > 3 ( > 32KB) → 直接返回 NULL

源码 line 2773-2774: if (order > PAGE_ALLOC_COSTLY_ORDER) goto out;
含义：如果需要 32KB 以上连续物理页但无法满足 → 不杀进程，直接返回 NULL 给调用者。因为杀一个进程也可能释放不出这么大连续块。
影响：驱动 dma_alloc_coherent(128KB) 失败 → 返回 NULL → 驱动不检查 NULL → kernel panic。这不走 OOM，而是直接 crash。

规则 2: GFP_ATOMIC 不触发回收

中断上下文 → 不能睡眠 → 不能回收 → 不能 OOM

源码 line 3063: wake_all_kswapds 只在有 __GFP_KSWAPD_RECLAIM 时触发。
GFP_ATOMIC 分配：不能唤醒 kswapd → 不能 direct reclaim → 不能触发 OOM → 只能用紧急预留 (ALLOC_WMARK_MIN)。紧急预留用尽 → 直接返回 NULL。
影响：中断 handler 中的内存分配最容易静默失败 → 未检查 NULL → panic。

规则 3: 递归/已 OOM 进程不再次 OOM

避免 OOM 自己也需要分配内存 → 无限递归

源码 line 3118: if (current->flags & PF_MEMALLOC) goto nopage;
源码 line 3122: if (test_thread_flag(TIF_MEMDIE) && !__GFP_NOFAIL) goto nopage;
含义：① 正在回收内存的进程（kswapd/直接回收的分配者）自身需要内存时不再触发 OOM ② 已被 OOM 选中正在死去的进程需要内存时不再触发 OOM。
影响：被 OOM 的进程可能因需要内存而卡住 → oom_reaper 异步收割解决。

🎯 OOM 触发条件总结
OOM Killer 会被触发：① 分配 order ≤ 3 (≤ 32KB) ② 允许回收 (__GFP_FS 且非 GFP_ATOMIC) ③ 非 NUMA 绑定 (无需 __GFP_THISNODE) ④ 五级递进全部失败: kswapd 异步回收 → async compaction → direct reclaim → 重试多次 → 最后手段 OOM

OOM Killer 不会被触发：① order > 3 (> 32KB，直接返回 NULL) ② GFP_ATOMIC (不能睡眠，不能回收) ③ __GFP_THISNODE (NUMA 单节点限制) ④ PF_MEMALLOC (正在回收的进程)

10. 大页内存 — THP · hugetlbfs

通过增大页尺寸减少 TLB miss — 两种机制：透明大页 (THP) 自动、hugetlbfs 显式预留

为什么需要大页？

TLB miss 的代价

4KB 页 × 大工作集 → 大量 TLB 条目 → 高 miss 率 → 性能下降。2MB 大页将覆盖范围扩大 512 倍 → TLB 条目需求减为 1/512。数据库、JVM、DPDK 等大内存应用受益显著。
ARM64 支持 4KB / 16KB / 64KB 页 + 2MB / 512MB 大页。

THP — 透明大页

内核自动管理，对应用透明

khugepaged 线程后台扫描，将连续的 4KB 页合并为 2MB 大页。应用无需修改代码。
缺点：分配延迟 (需要连续 2MB 物理块)、内存碎片化时可能失败。
/sys/kernel/mm/transparent_hugepage/enabled — always/madvise/never

hugetlbfs — 显式大页

应用主动请求的预留大页

预留在 /proc/sys/vm/nr_hugepages 中配置，挂载在 /dev/hugepages。应用通过 mmap(MAP_HUGETLB) 使用。优势：保证可获得 (启动时预留)、无碎片问题。缺点：需应用主动修改代码。
