1. 编译参数 — 调试的基石

正确的编译选项是有效调试的前提 — Debug 符号 · 优化级别 · Sanitizer 注入 · 帧指针 · 地址无关

Debug 符号

-g · DWARF 调试信息

-g: 生成 DWARF 调试信息 (函数名/行号/变量/类型)。
-g3: 额外包含宏定义 信息。
-ggdb: 针对 GDB 优化格式。
-gdwarf-4/5: 指定 DWARF 版本。
Debug 符号可后续用 strip 移除或 objcopy --only-keep-debug 分离。

优化级别对调试的影响

-O0 → -O2 → -Os

-O0: 不优化 — 变量值精确，但性能差。
-Og: ★ 调试推荐 — 开启不影响调试的优化。
-O2: 常见优化 — 变量常被优化掉 (<optimized out>)，但可配合 -fvar-tracking 缓解。
-Os: 体积优化 — 嵌入式常用。

Sanitizer 与帧指针

运行时检查 + perf 回溯

-fsanitize=address (ASan) — 堆/栈越界、use-after-free
-fsanitize=undefined (UBSan) — 整数溢出、空指针
-fsanitize=thread (TSan) — 数据竞争
-fno-omit-frame-pointer — perf 回溯必需 (x86_64 上 -O2 默认省略 rbp)

/* ===== 推荐的调试编译组合 ===== */ # 开发调试构建 CFLAGS="-g3 -Og -fno-omit-frame-pointer -Wall -Wextra" # 运行时内存检查构建 CFLAGS="-g3 -O0 -fsanitize=address -fsanitize=undefined" export ASAN_OPTIONS=detect_leaks=1:halt_on_error=0:log_path=asan.log # 线程竞争检查构建 CFLAGS="-g3 -O1 -fsanitize=thread" # perf 性能分析构建 CFLAGS="-g -O2 -fno-omit-frame-pointer" /* ===== 分离 Debug 符号 (生产环境) ===== */ objcopy --only-keep-debug myapp myapp.debug strip --strip-debug myapp objcopy --add-gnu-debuglink=myapp.debug myapp # GDB 会自动查找 .debug 文件
⚠️ 常见误区
-O2 调试: 变量显示 <optimized out> 是正常的 — 编译器已将变量放入寄存器或消除。使用 -Og 或 -O0 获得更好的调试体验。
Release build 无符号: 即使不带 -g，仍可调试汇编级别。配合 objdump -d 和 addr2line 定位崩溃地址。

2. Core Dump — 内核崩溃转储机制

致命信号 → do_coredump() → ELF 格式 core 文件 · core_pattern 管道 · coredump_filter 过滤 · systemd-coredump

触发流程 — 内核侧

fs/coredump.c — do_coredump()

① 进程收到致命信号 (SIGSEGV/SIGABRT/SIGBUS/SIGFPE/SIGILL/SIGQUIT...)
② 检查条件: get_dumpable() (suid_dumpable) · RLIMIT_CORE (`ulimit -c`) · 文件系统可写
③ 格式化路径: format_corename() 根据 /proc/sys/kernel/core_pattern 生成文件名
④ 输出格式: binfmt->core_dump() (ELF: fs/binfmt_elf.c) → 写内存段 + Notes 元数据
⑤ 管道模式: 若 core_pattern 以 | 开头 → 数据 pipe 到用户空间 handler

ELF Core 文件内容

Program Headers + Notes + 内存段

ELF Header: 架构 (x86_64/ARM64)、类型 ET_CORE
PT_NOTE Notes: NT_PRSTATUS — ★ 每线程寄存器状态
NT_PRPSINFO — PID/PPID/信号号
NT_FPREGSET — 浮点寄存器
NT_AUXV — 辅助向量
NT_FILE — 映射文件路径列表
NT_SIGINFO — siginfo_t (含 si_addr 故障地址!)
PT_LOAD 段: 每个被转储 VMA 的内存页

/* ===== Core Dump 配置 ===== */ # 查看当前 core 大小限制 (0 = 禁用, unlimited = 不限制) ulimit -c ulimit -c unlimited // ★ 必须设置! 否则段错误只输出 "Segmentation fault" # 查看/设置 core 文件名模式 cat /proc/sys/kernel/core_pattern # 默认: core (仅 core 文件名, 保存在当前目录) # 推荐: 含 PID + 程序名 + 时间戳 echo "/var/coredumps/core.%e.%p.%t" | sudo tee /proc/sys/kernel/core_pattern # systemd 管道模式 (systemd-coredump) # 查看: cat /proc/sys/kernel/core_pattern # 输出: |/usr/lib/systemd/systemd-coredump %P %u %g %s %t %c %h # core_pattern 格式符: # %e=执行文件名 %p=PID %t=时间戳 %s=信号号 %h=主机名 # %P=PID(全局) %u=UID %g=GID %c=rlimit大小 %E=执行文件路径 /* ===== coredump_filter — 控制转储哪些 VMA ===== */ cat /proc/$$/coredump_filter // 查看当前值 (默认 0x33) // bit 0 (0x01): 匿名私有内存 // bit 1 (0x02): 匿名共享内存 // bit 2 (0x04): 文件映射私有内存 // bit 3 (0x08): 文件映射共享内存 // bit 4 (0x10): ELF 头部 // bit 5 (0x20): HugeTLB 私有 echo 0x3f > /proc/$$/coredump_filter // 转储所有映射 (包括 .text/.rodata)
systemd-coredump — 现代 Linux 的 core 管理
若 core_pattern 是 systemd 管道，core 被压缩存储在 /var/lib/systemd/coredump/。使用 coredumpctl 交互式管理:
coredumpctl list — 列出所有 core · coredumpctl info — 最后崩溃详情
coredumpctl gdb — ★ 直接启动 GDB 分析最近崩溃 (自动找到二进制和 core!)

/* ===== GDB Core 分析 — 标准工作流 ===== */ # 1. 加载 core dump gdb /path/to/myapp /path/to/core.myapp.12345 # 2. 查看崩溃概况 (gdb) bt // 崩溃线程的回溯 (通常 bt 足够定位大部分崩溃) (gdb) bt full // 加上全部局部变量值 # 3. 查看所有线程 (gdb) info threads // 列出所有线程 — 看是否有其它线程也异常 (gdb) thread apply all bt // ★ 所有线程的回溯 — 多线程死锁/竞争诊断必需 # 4. 深入崩溃帧 (gdb) frame 0 // 进入崩溃帧 (#0) (gdb) info locals // 局部变量 — 检查指针是否为 NULL/野指针 (gdb) info args // 函数参数 (gdb) info registers // 寄存器快照 (gdb) p $_siginfo // ★ 信号信息 — si_addr = 故障地址 # 5. 反汇编崩溃位置 (gdb) disassemble /m // 反汇编当前帧 (带源码/行号对照) (gdb) x/10i $rip - 20 // RIP 前 20 条指令 — 看是什么操作导致崩溃 /* ===== 一键批处理 — 自动化 Core 分析 ===== */ gdb -batch -ex "thread apply all bt full" -ex "quit" \ /path/to/myapp /path/to/core > crash_analysis.log 2>&1
⚠️ Core 分析的前置条件
① 二进制与 core 必须匹配 — 使用生成 core 的同一二进制文件。编译后的二进制不可重新编译 (地址/偏移变化)。
② 共享库路径 — 若目标环境 lib 路径不同，用 set sysroot /path/to/target/rootfs 指定。
③ .debug 分离符号 — 若符号被 strip 分离，需 objcopy --add-gnu-debuglink 关联。
④ 交叉调试: aarch64-linux-gnu-gdb 配合 set sysroot 可分析 ARM64 目标 core。

4. 内存错误诊断工具 — Valgrind · ASan · TSan

Valgrind 无侵入运行时检查 · AddressSanitizer 编译期注入 · ThreadSanitizer 数据竞争检测

Valgrind (Memcheck)

无侵入 — LD_PRELOAD 等不需要重编译

检测: 内存泄漏、非法访问、未初始化值、double-free、错误 free。
代价: ~10-20× 慢 — 适合测试环境。
valgrind --leak-check=full --show-reachable=yes ./myapp
--track-origins=yes — 追踪未初始化值来源。
--suppressions=ignored.supp — 过滤已知误报。
Helgrind (--tool=helgrind) — 线程竞争检测。

ASan (AddressSanitizer)

编译期注入 — 需重新编译

检测: 堆/栈/全局缓冲区溢出、use-after-free、double-free。
代价: ~2× 慢 — 可在准生产环境使用。
gcc -fsanitize=address -g -O0 myapp.c
ASAN_OPTIONS=detect_leaks=1 ./myapp
原理: 在每次内存访问前后插入影子内存 检查指令，在分配周围创建红区 (Redzone) 检测越界。

TSan + UBSan

数据竞争 + 未定义行为

TSan (ThreadSanitizer): -fsanitize=thread
检测数据竞争 (data race)、锁顺序反转。
UBSan (UndefinedBehavior): -fsanitize=undefined
检测整数溢出、除零、空指针解引用、对齐错误、类型转换越界。
MSan (MemorySanitizer): 检测未初始化内存读取 (需全部代码用 MSan 编译)。

/* ===== ASan 实战 — 快速定位越界/释放后使用 ===== */ # 编译 gcc -fsanitize=address -g -O0 myapp.c -o myapp # 运行 (出错时自动打印详细报告) $ ./myapp # ================================================================= # ==12345==ERROR: AddressSanitizer: heap-buffer-overflow on address 0x60300000fff4 # READ of size 4 at 0x60300000fff4 thread T0 # #0 0x402a3b in process_data myapp.c:42 ← 直接指向源代码行! # #1 0x4031c5 in main myapp.c:87 # 0x60300000fff4 is located 0 bytes after 100-byte region [0x60300000ff90,0x60300000fff4) # allocated by thread T0 here: # #0 0x7f9a in malloc # #1 0x402a10 in process_data myapp.c:40 ← 分配点
CI 集成 — 自动化内存检查
ASAN_OPTIONS=detect_leaks=1:halt_on_error=0:log_path=/tmp/asan.log ./tests
检查退出码 (ASan 检测到错误 → exit code = 1) + 收集 log 文件用于事后分析。
Valgrind 使用 --error-exitcode=1 确保 CI 检测到问题。

4+. 应用层内存分配器 — jemalloc · mimalloc 调试实战

替换 glibc malloc — LD_PRELOAD 无侵入接入 · 内置 Heap Profiling · 内存统计 · 泄漏检测 · 碎片分析

💡 为什么需要关注应用层内存分配器？
ASan/Valgrind 需要重新编译或在测试环境运行 (10-20× 慢)；而 jemalloc/mimalloc 可通过 LD_PRELOAD 在生产环境中无侵入接入，自带 heap profiling、内存统计和泄漏检测能力，性能反而优于 glibc 默认分配器。

4+.1 jemalloc — Facebook 的通用内存分配器

jemalloc 调试特性

Heap Profiling · 统计 · 泄漏检测 · 碎片控制

Heap Profiling: --enable-prof 编译后，通过 MALLOC_CONF=prof:true 环境变量开启。周期性 dump heap 快照到 jeprof.*.heap 文件。
jeprof 分析工具:
• jeprof --text myapp jeprof.123.heap — 文本总览
• jeprof --pdf myapp jeprof.123.heap > prof.pdf — 调用图可视化
• jeprof --gif myapp jeprof.123.heap > prof.gif — 火焰图
内存统计 API: malloc_stats_print() — 运行时打印详细分配统计（arena、bin、huge 等各层指标）。

生产环境接入 — 零修改

LD_PRELOAD · 环境变量控制
# 编译 jemalloc (带 profiling 支持) git clone https://github.com/jemalloc/jemalloc.git cd jemalloc ./autogen.sh ./configure \ --enable-prof \ # Heap profiling --enable-stats \ # 统计信息 --enable-debug \ # 断言 + 额外检查 --prefix=/usr/local make -j$(nproc) && sudo make install # LD_PRELOAD 接入 — 无需重新编译应用 export LD_PRELOAD=/usr/local/lib/libjemalloc.so ./myapp # 开启所有调试特性 export MALLOC_CONF="prof:true,prof_prefix:jeprof.out,\ lg_prof_interval:30,prof_leak:true,\ stats_print:true" ./myapp # 强制全面检查 (类似 valgrind 但快得多) export MALLOC_CONF="junk:true,quarantine:1024,\ redzone:true,abort:true"
MALLOC_CONF 选项作用适用场景
prof:true启用 heap profiling，周期性 dump 内存分配快照定位内存泄漏、分析内存占用增长趋势
prof_leak:true进程退出时 dump 尚未释放的分配 — 内置泄漏检测CI 集成自动化泄漏检查
lg_prof_interval:30每当分配量达到 230 (1GB) 时 dump 一次长期运行服务的增量分析
prof_active:false启动时不激活 profiling，运行时通过 API 动态开启按需诊断，减少常态开销
junk:true分配时填充 0xa5，释放时填充 0x5a检测 use-after-free、未初始化内存使用
quarantine:1024释放后延迟 1024 字节才真正回收——加长 use-after-free 检测窗口类似 ASan 的 quarantine zone
redzone:true在每个分配块前后加保护页 (红区)，检测堆缓冲区溢出类似 ASan 的 redzone
stats_print:true进程退出时打印完整内存统计了解内存使用模式、碎片率
abort:true检测到内存错误时立即 abort() — 获取 core dumpCI 中强制 Early Abort
confirm_conf:true启动时打印实际生效的配置确认配置是否成功应用

4+.2 mimalloc — Microsoft 的紧凑型安全分配器

mimalloc 调试特性

内置安全检测 · 极低碎片 · 快速

安全模式: 编译时 -DMI_SECURE=ON，运行时编码 (encode) 所有指针、在 free 块填充随机 canary 值、对所有 free 操作验证指针有效性。
Guard Pages: 每个 page 后跟保护页——越界访问立即 SIGSEGV。
Heap 一致性检查: mi_heap_check_all() — 运行时验证所有内部数据结构的完整性。
Verbose 日志: MIMALLOC_VERBOSE=1 — 打印每个 alloc/free 操作 + 调用栈。
统计 API: mi_stats_print(NULL) / mi_stats_print_out(mi_main_heap(), stderr)。

快速接入 + 安全加固

LD_PRELOAD · 环境变量 · API 替换
# 编译 mimalloc (安全模式) git clone https://github.com/microsoft/mimalloc.git cd mimalloc && mkdir build && cd build cmake .. -DMI_SECURE=ON -DCMAKE_BUILD_TYPE=Debug make -j$(nproc) && sudo make install # LD_PRELOAD 零修改接入 export LD_PRELOAD=/usr/local/lib/libmimalloc.so ./myapp # 开启 verbose 详细日志 export MIMALLOC_VERBOSE=1 export MIMALLOC_SHOW_STATS=1 ./myapp # 开启安全检测 + 错误时 abort export MIMALLOC_SHOW_ERRORS=1 export MIMALLOC_ABORT_ON_ERROR=1 export MIMALLOC_PAGE_RESET=1 # 释放后立即归还 OS

4+.4 实战：用 jemalloc 定位生产环境内存泄漏

# === 场景: 长期运行的服务进程内存持续增长 (RSS 上涨) === # 1. 接入 jemalloc (带 profiling) export LD_PRELOAD=/usr/local/lib/libjemalloc.so export MALLOC_CONF="prof:true,prof_prefix:jeprof.out,\ lg_prof_interval:26,prof_leak:true,\ stats_print:true" ./my_service # 2. 运行一段时间后，触发一次 heap dump (发 SIGUSR1) kill -USR1 $(pidof my_service) # 或者通过代码调用: mallctl("prof.dump", NULL, NULL, NULL, 0); # 3. 用 jeprof 分析增量 jeprof --text --inuse_space my_service jeprof.out.*.heap \ | head -30 # 输出示例: # Total: 512.3 MB # 245.1 MB (47.8%): FrameProcessor::process # 120.7 MB (23.6%): VideoEncoder::encode # 68.3 MB (13.3%): AudioCapture::read # 4. 对比两次 dump 的差异 (定位增量泄漏) jeprof --text --base=jeprof.out.001.heap my_service jeprof.out.002.heap # 5. 生成调用图 PDF jeprof --pdf --base=jeprof.out.001.heap my_service jeprof.out.002.heap \ > leak_graph.pdf
✅ jemalloc/mimalloc 调试选型建议
jemalloc 更适合: 需要 heap profiling + 内存泄漏追溯的生产环境，尤其是多线程服务 (jemalloc 的 arena 模型减少了锁竞争)；jeprof 工具成熟，与 pprof 生态兼容。
mimalloc 更适合: 需要 安全加固 的场景 (Guard Pages 防溢出、指针编码防 corruption)；对碎片率和分配延迟要求极致的实时系统。
组合使用: 开发/测试阶段用 ASan + Valgrind (全覆盖)，预发布/生产环境加 jemalloc LD_PRELOAD + prof_leak (性能友好 + 最后防线)。
注意: jemalloc/mimalloc不能替代 ASan/Valgrind——它们检测不了栈溢出、全局变量越界、未初始化栈变量。它们是互补而非替代关系。

5. strace · ltrace · perf — 系统调用追踪与性能分析

strace 追踪系统调用 · ltrace 追踪库函数 · perf 硬件性能计数器 + 火焰图

strace — 系统调用追踪

进程与内核交互的完整记录

strace -f -tt -o trace.log ./myapp — 完整追踪 (+fork +时间戳 +输出到文件)
strace -e trace=open,openat,read,write — 仅文件 I/O
strace -e trace=network — 仅网络相关
strace -e trace=signal — 信号处理
strace -c ./myapp — ★ 按系统调用统计耗时 (快速定位性能瓶颈)
strace -p <PID> — 附加到运行中进程

perf — 硬件性能分析

CPU 计数器 · 调用图 · 火焰图

perf record -g -e cpu-clock ./myapp — 以 CPU 时钟采样 + 调用图
perf record -g -e cache-misses ./myapp — 缓存未命中事件

6. 崩溃分析实战方法论

从信号类型判断崩溃性质 · SEGV_MAPERR vs SEGV_ACCERR · SIGABRT 双重释放 · SIGBUS 对齐 · SIGFPE 除零

信号典型原因GDB 中的关键线索首选工具
SIGSEGV (11)
SEGV_MAPERR		NULL 指针解引用 / 访问未映射地址$_siginfo.si_addr = 0x0 或极小值GDB + ASan
SIGSEGV (11)
SEGV_ACCERR		访问权限错误 (写只读内存/执行数据段)si_addr 指向合法映射但权限不对GDB + /proc/PID/maps
SIGABRT (6)assert() 失败 / double-free / 堆损坏

SIGBUS (7)未对齐访问 / mmap 文件被截断

方法论: 6 步系统化分析

从拿到 core 到定位根因

① 确认信号: p $_siginfo.si_signo → 判断崩溃类型
② 查看崩溃现场: bt full → 定位崩溃栈帧
③ 检查故障地址: p $_siginfo.si_addr → 判断 NULL/野指针/已释放
④ 回溯分配历史: 如果有 ASan/Valgrind 日志 → 找到 "allocated by" 行 → 对比 "freed by"
⑤ 全线程诊断: thread apply all bt → 排除多线程死锁/竞争
⑥ 复现验证: 用 ASan/TSan 构建重现 → 确认修复
