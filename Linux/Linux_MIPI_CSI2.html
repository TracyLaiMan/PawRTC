MIPI CSI-2 协议

Camera Serial Interface 2

MIPI 联盟定义的图像传感器与主机处理器 之间的串行接口标准。五层架构 (PHY→Lane→LLP→Pixel→App)，支持 RAW/RGB/YUV 多种像素格式，虚拟通道 多数据流复用。

D-PHY / C-PHY

两种物理层实现

D-PHY: 差分信号 + 独立时钟，1-4 lane，最高 9 Gbps/lane。
C-PHY: 三线编码 + 嵌入时钟，~2.28 bits/symbol，更低 EMI。CSI-2 v4.1 支持 D-PHY v3.5 嵌入式时钟。

Linux V4L2 框架

Video4Linux2 + Media Controller

v4l2_subdev 管理 Sensor/CSI/ISP 子设备。Media Controller 通过 entity→link→pad 描述数据流拓扑。videobuf2 提供 DMA 缓冲区管理，支持 mmap/DMABUF/USERPTR。

2. D-PHY vs C-PHY — 物理层深度对比

D-PHY: 差分信号 + 独立时钟 · C-PHY: 三线三元编码 + 嵌入时钟 · 2024 D-PHY v3.5 新增嵌入式时钟

D-PHY — 差分信号 + 源同步时钟

1 Clock lane + 1~8 Data lane

HS (高速) 模式: 低压差分信号 (LVDS-like), 200mV 摆幅, 80Mbps~9Gbps/lane。DDR 双沿采样。
LP (低功耗) 模式: 单端 LVCMOS, 1.2V 摆幅, ≤10Mbps。用于控制命令和空闲。
模式切换序列: LP-11 → LP-01 → LP-00 → HS-Enable
2024 v3.5: 新增 ECM 嵌入式时钟 (128b/132b 编码) — 无需独立时钟 lane! + ALP 替代低功耗模式 + 非线性 DFE 均衡。v4.0 目标 16Gbps/lane。

C-PHY — 三线三元编码 + 嵌入时钟

1~3 Trio (每 Trio 3 线, 无独立时钟)

工作原理: 三线电压差 (A-B, B-C, C-A) 产生 6 种状态 (+x,-x,+y,-y,+z,-z)。每符号 ~2.28 bits (7 符号→16 bits)。
优势: 带宽效率 ~2.28× D-PHY · 每 lane 少 1 根线 · EMI 低 ~10dB (谐波能量分散)。
速率: 6 Gsps trio → ~13.7Gbps 净数据率。CDR 时钟数据恢复。模式切换延迟低于 D-PHY。

3. CSI-2 数据包 — 短包 · 长包 · 像素格式 · 虚拟通道

短包传输帧/行同步 · 长包传输像素数据 · VC 实现多流复用 · ECC/CRC 保证可靠性

短包 (Short Packet) — 4 字节

Data ID + Word Count + ECC

FS (0x00): 帧起始 · FE (0x01): 帧结束
LS (0x02): 行起始 · LE (0x03): 行结束
每个短包携带 16-bit 帧号或行号。Data ID 的 bit[7:6] 编码 虚拟通道 (VC 0~3)，bit[5:0] 为数据类型。

长包 (Long Packet) — 像素数据

Header (4B) + Payload (0~65535B) + Footer (2B CRC)

Header: Data ID + Word Count (16-bit) + ECC (8-bit)
Payload: 像素数据 (Word Count 字节)
Footer: 16-bit CRC 校验和
RAW10/12/14 等非字节对齐格式通过字节打包传输 (如 5 个 RAW10 像素 = 50 bits → 7 字节)。

虚拟通道 (Virtual Channel) — 多数据流复用
D-PHY: 最多 16 个 VC (v2.0+) · C-PHY: 最多 32 个 VC。多个相机通过时分复用 (TDM) 共享一个 CSI PHY 接口，硬件聚合器按 VC ID 区分不同相机数据流。单 CSI 解码器可同时解码所有 VC 的数据包 → 每 VC 在用户空间对应独立的 /dev/videoX。

5. Linux V4L2 + Media Controller — MIPI 摄像头驱动架构

Sensor (I2C subdev) → CSI RX (Platform) → ISP → DMA → /dev/videoX — 完整驱动栈

v4l2_subdev — 子设备模型

Sensor / CSI / ISP 统一为 subdev

每种硬件组件 (Sensor, CSI RX, ISP, DMA) 注册为独立的 v4l2_subdev。每个 subdev 有一组 v4l2_subdev_ops (core/video/pad)。V4L2 Async 框架: Sensor (I2C) 和 CSI (Platform) driver 各自独立 probe，通过 v4l2_async_notifier 等待全部就绪后完成绑定。

Media Controller — 拓扑管理

media_entity → pad → link 描述 pipeline

entity: 每个 subdev 是图中的一个节点。
pad: 节点的输入/输出端口 (Sink/Source)。
link: 两个 pad 之间的有向数据流连接。
用户空间通过 /dev/mediaX 接口配置 pipeline (media-ctl 工具)。
拓扑示例: [Sensor]→source pad==[CSI]=sink pad==[ISP]==source pad→[Capture]

videobuf2 — DMA 缓冲区

mmap · DMABUF · USERPTR

CSI 控制器通过 DMA 将解析后的像素写入 vb2 缓冲区。一帧完成后 IRQ handler → vb2_buffer_done(VB2_BUF_STATE_DONE)。
用户空间通过 ioctl(VIDIOC_DQBUF) 获取填充好的帧数据。
三种内存模式: mmap (内核分配, 零拷贝) · DMABUF (跨设备共享) · USERPTR (用户分配)。

/* ═══ MIPI 摄像头 — 从 Sensor 到用户空间 ═══ */ Camera Sensor (CMOS → MIPI D-PHY 输出) │ I2C/I3C 控制: v4l2_subdev_ops.video.s_stream(1) │ 格式配置: v4l2_subdev_ops.pad.set_fmt() → 设置 RAW10, 1920×1080, 4-lane ▼ CSI-2 RX Controller (D-PHY 接收 → LLP 解析 → Pixel/Byte 解包) │ DMA 传输: 像素写入 vb2 缓冲区 ▼ DDR Buffer (videobuf2 queue — mmap 映射) │ 帧完成 IRQ → vb2_buffer_done() ▼ V4L2 Video Device (/dev/video0) │ ioctl(VIDIOC_DQBUF) → 获取帧数据 ▼ User Space (GStreamer / OpenCV / libcamera → 显示/编码/分析)
libcamera — 现代 Linux 相机框架
libcamera 是 Linux 基金会托管的新一代相机用户空间栈，替代传统的 V4L2 直连方式。它直接操作 Media Controller 拓扑，管理整个 pipeline 的配置、buffer 分配和流控制，屏蔽了不同 SoC (i.MX/Rockchip/RPi/Starfive) 的平台差异。2024 年已成为嵌入式 Linux 相机开发的首选。

6. Device Tree 绑定 — Sensor · CSI · Endpoint 互联

port/endpoint + remote-endpoint 描述硬件连接 · data-lanes/clock-lanes 指定物理通道 · link-frequencies 时钟协商
