核心概念

nodev — 不依赖块设备

所有内存文件系统都是 nodev 类型 — 不存在磁盘后备存储，数据完全驻留在内存中或实时生成。cat /proc/filesystems 中标记为 nodev 的条目即为此类。

两大分类

页缓存型 vs 动态生成型

页缓存型：ramfs / tmpfs — 数据存储在 page cache 中。
动态生成型：procfs / sysfs / debugfs — 每次读时实时生成内容，无持久数据存储。

Linux 中的角色

无处不在的内存文件系统

/dev → devtmpfs · /proc → procfs · /sys → sysfs
/dev/shm → tmpfs · /tmp → tmpfs · /run → tmpfs
/sys/kernel/debug → debugfs · initramfs → ramfs/tmpfs

2. ramfs — 最简页面缓存文件系统

tmpfs 的模板 · 无条件使用 Page Cache · 无大小限制 · 无 Swap · 内核内部 initramfs 的基础

设计原理

fs/ramfs/ — 内核内置，无独立 Kconfig

ramfs 是 Linux 最简内存文件系统。文件数据直接以 page 形式存储在 Page Cache 中，目录结构通过 dentry cache 和 inode cache 维护。没有 backing store，没有 writepage 操作 — page 永远不标记为脏页需要回写。它是 tmpfs 的代码模板。

关键限制

无大小限制 = OOM 定时炸弹

没有容量上限 — 文件可以增长到耗尽全部可用内存，触发 OOM Killer。内存不足时不能 swap 出 — 数据永久钉在 RAM 中。因此 ramfs 不适合用户空间直接使用。它的正确用途是内核内部场景：initramfs（rootfs 在 pivot_root 之前的临时根文件系统）。

/* ===== ramfs 关键实现 ===== */ /* 注册文件系统类型 */ static struct file_system_type ramfs_fs_type = { .name = "ramfs", .mount = ramfs_mount, .kill_sb = ramfs_kill_sb, .fs_flags = FS_USERNS_MOUNT, }; /* 超级块操作 — 几乎全部使用通用实现 */ static const struct super_operations ramfs_ops = { .statfs = simple_statfs, /* 报告文件系统统计 (始终返回 0 可用空间) */ .drop_inode = generic_delete_inode, /* 引用计数为 0 时立即释放 inode 和页面 */ }; /* ⚠️ ramfs 没有 writepage → 页面永不 swap → 数据永久驻留 RAM */
⚠️ ramfs vs tmpfs — 关键区别
ramfs 没有 address_space_operations.writepage 实现 → 内核无法将页面写入 swap → 数据永久驻留 RAM。
tmpfs (通过 shmem) 实现了 writepage → 内存压力下可以 swap out → 适合用户空间长期使用。
结论：永远不要手动 mount ramfs，使用 tmpfs 代替。ramfs 的存在价值是作为 initramfs 的内核内部实现和 tmpfs 的代码参考。

3. tmpfs — 可交换的内存文件系统

基于 mm/shmem.c · 支持 Swap · 大小限制 · POSIX 共享内存 · Linux 最广泛使用的 nodev 文件系统

shmem 核心引擎

mm/shmem.c — 整个 tmpfs 的后端

tmpfs 不是独立实现 — 它的读/写/分配/回收全部委托给 shmem (shared memory) 子系统。shmem_get_folio() 处理页面获取（先查 page cache → miss 则分配新页），shmem_writepage() 将页面标记为 dirty 并交给 swap 子系统。内核内部维护一个隐藏的 tmpfs 实例用于匿名共享映射和 SysV shmem。

Swap 生命周期

页缓存 ↔ Swap 区的双向迁移

① 写入文件 → 页在 page cache 中，标记 dirty
② 内存压力 → LRU 回收选中该页 → shmem_writepage → swap_writepage 写入 swap 区 → 页释放回 buddy
③ 再次读取 → page cache miss → shmem_swapin_page 从 swap 读回 → 重新加入 page cache
④ 长期不使用 → swap 中的页面最终被丢弃

内存统计

关键指标获取

cat /proc/meminfo：
Shmem — 所有 tmpfs + 共享内存占用的页面
SwapTotal / SwapFree — swap 总/空闲量
df -t tmpfs -h — 每个 tmpfs 挂载点使用量
free -h — shared 列即 tmpfs 数据

挂载选项示例说明
sizesize=2G · size=50%最大容量 (默认: 物理 RAM 的 50%)。支持 k/m/g 后缀和百分比
nr_blocksnr_blocks=524288最大块数 (PAGE_SIZE 为单位, x86 上 4K)
nr_inodesnr_inodes=100000最大 inode 数量 (文件+目录总数)
modemode=1777初始根目录权限 (如 1777 = sticky + rwx 全部)

inode64 / inode32—inode 编号宽度 (64-bit 默认无限制 / 32-bit 兼容)

隐藏的 tmpfs 实例 — /proc/meminfo 中的 "Shmem"
即使没有挂载任何 tmpfs，内核内部也存在一个不可见的 tmpfs 实例，用于：① 匿名 mmap (MAP_SHARED|MAP_ANONYMOUS) ② SysV 共享内存 (shmget) ③ memfd_create。这些页面在 /proc/meminfo 中显示为 "Shmem"，但不出现在 df 中。这就是 free -h 中 "shared" 列的实际含义。

4. devtmpfs — 内核自动设备节点文件系统

CONFIG_DEVTMPFS · 在用户空间之前挂载 /dev · 驱动注册时内核自动创建设备节点

设计动机

消除对早期用户空间设备管理器的依赖

在 devtmpfs 出现之前，/dev 要么是静态填充的（mknod 手动创建，维护噩梦），要么依赖 udev 在早期用户空间启动后才能工作。devtmpfs 让内核本身在设备注册时就创建节点 — 不需要任何用户空间程序。内核在 boot 时自动挂载 devtmpfs 到 /dev，udev 运行后只需追加权限、符号链接等元数据。

工作流程

从 device_add() 到 /dev 节点

① device_add() (驱动注册设备)
→ ② devtmpfs_create_node(dev) (driver core 回调)
→ ③ 在 devtmpfs 中创建 /dev/sda1 等设备节点
④ 设备移除时：device_del() → devtmpfs_delete_node() 删除节点。
/dev 内容由内核 driver core 自动维护，无需手动 mknod。

/* ===== 内核配置 ===== */ CONFIG_DEVTMPFS=y /* 启用 devtmpfs 支持 */ CONFIG_DEVTMPFS_MOUNT=y /* 内核自动挂载到 /dev (在 rootfs 后就绪) */ /* ===== 启动序列 ===== */ 1. 内核初始化 devtmpfs 内部实例 2. rootfs 挂载后 → 内核自动 mount devtmpfs to /dev 3. 驱动通过 device_add() 注册设备 → /dev 节点自动出现 4. 用户空间启动 udev → 添加 permissions/symlinks/device aliases /* ===== mount 显示 ===== */ devtmpfs on /dev type devtmpfs (rw,nosuid,size=4096k,nr_inodes=1022382,mode=755)

5. procfs — 进程 & 内核信息伪文件系统

CONFIG_PROC_FS · 挂载于 /proc · 每次读实时生成内容 · seq_file API 的核心使用者

/proc 关键目录与文件

进程信息 + 系统运行时数据

/proc/<PID>/ — 进程信息 (maps, status, fd, cmdline, environ, cwd, exe...)
/proc/meminfo — 内存统计 · /proc/cpuinfo — CPU 详情
/proc/interrupts — IRQ 计数器 · /proc/iomem — MMIO 映射
/proc/filesystems — 已注册文件系统类型
/proc/modules — 已加载模块 (lsmod 的数据源)
/proc/sys/ — sysctl 参数 (kernel.hostname 等)

seq_file 接口

解决 procfs 缓冲区溢出的标准方案

早期 procfs 直接用 sprintf 往固定大小缓冲区写 → 输出超过 PAGE_SIZE 时数据截断。seq_file 提供迭代器模式：
start() → next() → show() 循环，每次 show 输出一条记录。内核自动管理页大小的缓冲区，自动分页。支持 lseek 随机访问。

/* ===== procfs 模块最小示例 — 使用 seq_file + single_open ===== */ #include <linux/proc_fs.h> #include <linux/seq_file.h> static int my_show(struct seq_file *m, void *v) { seq_printf(m, "Hello from kernel! jiffies=%lu\n", jiffies); return 0; } static int my_open(struct inode *inode, struct file *file) { return single_open(file, my_show, NULL); } static const struct file_operations my_fops = { .owner = THIS_MODULE, .open = my_open, .read = seq_read, /* seq_file 标准读 */ .llseek = seq_lseek, /* seq_file 标准定位 */ .release = single_release, /* single_open 配对释放 */ }; static int __init my_init(void) { proc_create("myproc", 0, NULL, &my_fops); // 创建 /proc/myproc return 0; } static void __exit my_exit(void) { remove_proc_entry("myproc", NULL); // 删除 } module_init(my_init); module_exit(my_exit);
seq_file API用途
single_open(file, show, private)简单场景 — 整个输出在一个 show 调用中完成
seq_open(file, &seq_ops)复杂场景 — 需要定义 start/next/stop/show 四件套迭代大列表
seq_printf(m, fmt, ...)格式化输出 (类 printf) — 内核自动管理缓冲区
seq_puts(m, str) / seq_putc(m, c)输出字符串 / 输出单个字符
seq_read() / seq_lseek()填充 file_operations.read / .llseek
single_release() / seq_release()填充 file_operations.release
single_open vs seq_open — 如何选择
single_open：输出数据量小，无需迭代 — 只需实现 show()。适用于简单的状态导出 (如模块参数、统计信息)。
seq_open：需要遍历大列表/数组 — 实现全套 start / next / stop / show。适用于 /proc/<PID>/maps、/proc/modules 等需要逐条输出的场景。
seq_file 保证 start→stop 之间不持有锁，因此可以在 seq_ops 回调中使用 spinlock/mutex。

6. sysfs — 内核对象模型文件系统

CONFIG_SYSFS · 挂载于 /sys · 基于 ramfs + kobject · 设备/驱动/总线/模块/类的结构化导出

/sys 顶级目录结构

每个子目录映射一个内核子系统

/sys/block/ — 各块设备的符号链接 → /sys/devices/
/sys/bus/ — pci, usb, platform, sdio... (devices + drivers)
/sys/class/ — net, tty, block, graphics, sound...
/sys/devices/ — 真实设备树根 (物理拓扑)
/sys/dev/ — block/ + char/ (major:minor 符号链接)
/sys/kernel/ — 内核运行时参数
/sys/module/ — 每个已加载模块一个目录
/sys/firmware/ — ACPI, EFI, DMI...
/sys/fs/ — 文件系统特定导出 (ext4, cgroup...)
/sys/power/ — 电源管理状态

kobject — sysfs 的基石

一个 kobject = 一个 sysfs 目录

kobject_add() → 在 sysfs 中创建目录
kobject_put() → 引用计数减 1，0 时删除目录
sysfs_create_file() — 创建属性文件 (show/store 回调)
sysfs_create_group() — 批量创建属性组
一个值一个文件：sysfs 设计哲学是每个文件只导出一个值，保证 machine-parsable。

/* ===== sysfs 属性操作 — show/store 模式 ===== */ static ssize_t my_attr_show(struct kobject *kobj, struct kobj_attribute *attr, char *buf) { return sysfs_emit(buf, "%d\n", some_value); // 推荐 sysfs_emit 而非 sprintf } static ssize_t my_attr_store(struct kobject *kobj, struct kobj_attribute *attr, const char *buf, size_t count) { int ret = kstrtoint(buf, 10, &some_value); // 解析用户输入 return ret ? ret : count; } static struct kobj_attribute my_attr = __ATTR_RW(my_attr); static struct attribute *my_attrs[] = { &my_attr.attr, NULL, /* 必须 NULL 结尾 */ }; ATTRIBUTE_GROUPS(my); /* 生成 my_groups[] */ /* 在 kobject 上应用 */ struct kobject *kobj = kobject_create_and_add("mykobj", kernel_kobj); sysfs_create_groups(kobj, my_groups);
sysfs_emit() — Linux 5.10+ 推荐的安全输出函数
传统 sprintf(buf, "%d\n", val) 可能产生缓冲区溢出。新推荐 sysfs_emit(buf, "%d\n", val) 自动检查边界，返回实际写入字节数。show 函数必须使用 sysfs_emit 而非 sprintf。同理，store 函数使用 kstrtoint/kstrtouint/kstrtobool 安全解析用户输入。

7. debugfs — 内核调试接口

CONFIG_DEBUG_FS · 挂载于 /sys/kernel/debug · 非稳定 ABI · 开发者专用

设计理念

为内核开发者而生 — 无稳定接口承诺

debugfs 是一个简单的伪文件系统，专门用于内核调试。与 sysfs (稳定的用户空间 ABI) 不同，debugfs 的接口可能在内核版本间变化。提供了极简的 API：创建目录、创建文件（带自定义 file_operations）、创建类型化节点 (u32/bool/regset...)。使用者：ftrace、DRM、无线监管、块层、文件系统调试。

便捷 API

一行创建类型化调试节点

debugfs_create_dir(name, parent) — 创建目录
debugfs_create_file(name, mode, parent, data, fops) — 创建文件
debugfs_create_u32(name, mode, parent, &u32_var) — 直接映射到 u32 变量 (自动 show/store)
debugfs_create_u8/u16/u64/x8/x16/x32/x64 — 各宽度无符号/十六进制
debugfs_create_bool(name, mode, parent, &bool_var) — 布尔值

/* ===== debugfs 快速上手 ===== */ #include <linux/debugfs.h> static struct dentry *my_dir; static u32 my_counter = 0; static bool my_enable = true; static int __init my_init(void) { my_dir = debugfs_create_dir("my_driver", NULL); // 创建目录 debugfs_create_u32("counter", 0644, my_dir, &my_counter); // 自动 show/store debugfs_create_bool("enable", 0644, my_dir, &my_enable); // 输出 Y/N return 0; } static void __exit my_exit(void) { debugfs_remove_recursive(my_dir); /* 递归删除整个目录树 */ } /* 使用后效果: */ # cat /sys/kernel/debug/my_driver/counter 42 # echo 100 > /sys/kernel/debug/my_driver/counter # cat /sys/kernel/debug/my_driver/enable Y
