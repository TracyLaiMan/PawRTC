1. 驱动分类与内核视角

Linux 将设备分为三大类——字符设备、块设备、网络设备。这是选择驱动框架的第一决策点。

字符设备 (cdev)

最常见的类型 — 字节流访问

以字节流方式顺序读写，不支持随机寻址。通过 /dev/xxx 节点访问（major+minor 号标识）。实现 struct file_operations（open/read/write/ioctl/llseek/mmap）。典型：串口(/dev/ttyS0)、GPIO 控制器、I2C 设备(/dev/i2c-N)、Input 设备(/dev/input/eventX)。绝大多数外设驱动都是字符设备。通过 cdev_init() + cdev_add() 注册。

块设备 (blkdev)

支持随机寻址 — 固定大小块

以固定大小的块（通常 512B/4KB）为单位随机访问，支持 seek。通过 /dev/sdX、/dev/mmcblkX 访问。实现 struct block_device_operations + 请求队列（blk_mq 多队列）。典型：eMMC/SD 卡、SATA/NVMe 磁盘、U 盘。内核提供完整的通用块层——驱动只需处理硬件 I/O 请求，调度/缓存/合并由 blk-mq 负责。

网络设备 (netdev)

完全不同的模型 — sk_buff 数据包

网络设备没有 /dev 节点——通过 struct net_device 注册，暴露为 ethX/wlanX。数据以 sk_buff（内核网络数据包结构）传递。实现 struct net_device_ops（ndo_open/ndo_stop/ndo_start_xmit）。典型：以太网 MAC 驱动、WiFi 驱动。网络子系统有自己的 NAPI（中断+轮询混合）接收模型，与字符/块设备完全不同。

2. 设备模型：bus · device · driver · class

Linux 2.6 引入的统一设备模型是所有驱动框架的基石——理解这四个核心结构体，任何驱动框架都能触类旁通。

struct bus_type — 总线

platform_bus_type · i2c_bus_type · spi_bus_type

总线是device 和 driver 的匹配桥梁。核心函数：match(dev, drv)——返回 true 则绑定。platform 总线通过 dev->name 与 drv->name 字符串比较匹配；I2C/SPI 通过设备树 compatible 或 I2C device ID table 匹配；USB/PCI 通过 vendorID/productID 匹配。注册了 bus_type 后，内核自动为每对该总线上的 device-driver 尝试 match→probe。

struct device_driver — 驱动

platform_driver · i2c_driver · spi_driver

每种总线有自己的 driver 封装，但都内嵌 struct device_driver。核心函数：probe(dev)——匹配成功后初始化硬件（ioremap、request_irq、注册 cdev）、remove(dev)——模块卸载时清理。所有现代驱动都不应手动创建设备节点——应由 device/driver 绑定后自动触发 uevent → udev 创建设备节点。

struct device — 设备

platform_device · i2c_client · spi_device

代表物理总线上真实存在的一个硬件设备。对于 platform 总线，设备由内核从设备树解析创建（of_platform_populate()）；对于可枚举总线（USB/PCI），由总线控制器在枚举时创建。device 存储硬件资源（IORESOURCE_MEM/IORESOURCE_IRQ）、设备树节点指针（of_node）、DMA 掩码、电源管理域。

struct class — 设备类

input_class · rtc_class · gpio_class

Class 将同一类功能的设备（而非同一总线）组织在一起。主要职责：① 在 /sys/class/xxx/ 创建 sysfs 目录 ② 向 udev 发送 uevent（触发 /dev 节点创建）。驱动调用 device_create() 时指定 class——内核自动创建 /dev 节点并设置权限。class 还提供 class_attribute 让驱动导出 class 级别的 sysfs 属性。

// 设备模型的核心绑定流程 1. 注册 driver driver_register(&drv->driver) → bus_add_driver() → bus_for_each_dev(bus, drv, __driver_attach) // 遍历 bus 上所有 device 2. 匹配 driver_match_device(drv, dev) → bus->match(dev, drv) // platform: strcmp(dev->name, drv->name) // i2c/spi: of_match_table 或 id_table 3. 绑定与 probe driver_probe_device(drv, dev) → really_probe(dev, drv) → dev->driver = drv → drv->probe(dev) // 驱动的核心入口函数 4. 创建用户态接口 probe 成功后 → device_create(dev, class, ...) → udev 收到 uevent → mknod /dev/xxx → 用户态可用

3. Platform 总线与设备树绑定

Platform 总线是嵌入式 Linux 最核心的总线模型——所有挂载在 SoC 内部总线（非 USB/PCI/I2C/SPI）上的外设都是 platform device。设备树是描述这些设备的现代方式。

设备树 → platform_device

内核启动时将 DT 节点转换为 platform_device

内核初始化时调用 of_platform_default_populate()，遍历设备树中 /soc 或根节点下有 compatible 属性的子节点，为每个节点创建 platform_device。设备树中的 reg 属性 → IORESOURCE_MEM 资源、interrupts → IORESOURCE_IRQ 资源、clocks → 时钟引用。驱动通过 platform_get_resource() 或更现代的 devm_platform_ioremap_resource() 获取资源。

of_match_table 匹配

struct of_device_id — .compatible 字符串匹配

platform 驱动通过 of_match_table 声明支持的设备树 compatible 字符串。内核将 device 的 compatible 与 driver 的 of_match_table 逐条比较——匹配成功 → probe 调用。同一个驱动可以匹配多个 compatible（如支持多代硬件）。

static const struct of_device_id my_match[] = { { .compatible = "vendor,ipc-camera-isp" }, { .compatible = "vendor,ipc-camera-isp-v2" }, { /* sentinel */ } }; MODULE_DEVICE_TABLE(of, my_match);
设备树 vs 板级文件 — 历史与现状
在设备树引入之前（ARM Linux 早期），platform_device 在板级文件（arch/arm/mach-xxx/board-xxx.c）中硬编码创建设备并注册。这导致每个板子都需要修改内核代码。设备树将硬件描述从内核代码中分离——同一份内核镜像适配多块板子，只需更换 .dtb。ARM 从 2011 年起全面迁移到设备树，板级文件已基本废弃。

4. 字符设备驱动完整模板

从 platform_driver 注册到 /dev 节点创建——一个完整的、可编译的嵌入式设备驱动骨架。

完整的 platform 字符设备驱动模板
#include <linux/module.h> #include <linux/platform_device.h> #include <linux/cdev.h> #include <linux/io.h> struct my_device { struct device *dev; // 指向 platform_device.dev void __iomem *regs; // ioremap 后的寄存器基址 struct cdev cdev; // 字符设备 dev_t devt; // 主次设备号 int irq; // 中断号 }; // ═══════ file_operations — 用户态接口 ═══════ static int my_open(struct inode *inode, struct file *filp) { struct my_device *dev = container_of(inode->i_cdev, struct my_device, cdev); filp->private_data = dev; // 将设备私有数据挂在 file 上 return 0; } static ssize_t my_read(struct file *filp, char __user *buf, size_t count, loff_t *ppos) { struct my_device *dev = filp->private_data; u32 val = readl(dev->regs + MY_REG_DATA); // 读硬件 FIFO if (copy_to_user(buf, &val, sizeof(val))) return -EFAULT; return sizeof(val); } static long my_ioctl(struct file *filp, unsigned int cmd, unsigned long arg) { struct my_device *dev = filp->private_data; switch (cmd) { case MY_IOC_RESET: writel(BIT(0), dev->regs + MY_REG_CTRL); break; case MY_IOC_GET_STATUS: return readl(dev->regs + MY_REG_STATUS); } return 0; } static const struct file_operations my_fops = { .owner = THIS_MODULE, .open = my_open, .read = my_read, .unlocked_ioctl = my_ioctl, }; // ═══════ IRQ Handler ═══════ static irqreturn_t my_isr(int irq, void *data) { struct my_device *dev = data; u32 status = readl(dev->regs + MY_REG_IRQ_STATUS); writel(status, dev->regs + MY_REG_IRQ_CLEAR); // W1C 清中断 return IRQ_WAKE_THREAD; // 或 IRQ_HANDLED / IRQ_NONE } // ═══════ platform_driver probe/remove ═══════ static int my_probe(struct platform_device *pdev) { struct my_device *dev; int ret; // 1. 分配私有数据结构 dev = devm_kzalloc(&pdev->dev, sizeof(*dev), GFP_KERNEL); // 2. ioremap 寄存器（设备树 reg = <0xXXXX 0xYYYY>） dev->regs = devm_platform_ioremap_resource(pdev, 0); if (IS_ERR(dev->regs)) return PTR_ERR(dev->regs); // 3. 获取并注册中断 dev->irq = platform_get_irq(pdev, 0); ret = devm_request_irq(&pdev->dev, dev->irq, my_isr, IRQF_SHARED, "my-device", dev); // 4. 分配设备号 + 注册 cdev ret = alloc_chrdev_region(&dev->devt, 0, 1, "mydevice"); cdev_init(&dev->cdev, &my_fops); cdev_add(&dev->cdev, dev->devt, 1); // 5. 创建 /dev 节点（自动触发 udev） dev->dev = device_create(my_class, &pdev->dev, dev->devt, dev, "mydevice%d", 0); platform_set_drvdata(pdev, dev); return 0; } static int my_remove(struct platform_device *pdev) { struct my_device *dev = platform_get_drvdata(pdev); device_destroy(my_class, dev->devt); cdev_del(&dev->cdev); unregister_chrdev_region(dev->devt, 1); return 0; } static const struct of_device_id my_of_match[] = { { .compatible = "vendor,my-device" }, { } }; MODULE_DEVICE_TABLE(of, my_of_match); static struct platform_driver my_driver = { .probe = my_probe, .remove = my_remove, .driver = { .name = "my-device", .of_match_table = my_of_match, }, }; module_platform_driver(my_driver); // 一行替代 init/exit 样板 MODULE_LICENSE("GPL"); MODULE_AUTHOR("Your Name");
⚠ devm_* 系列函数 — 内核的资源管理革命
devm_kzalloc / devm_ioremap_resource / devm_request_irq 等 devm (Managed Device Resource) 函数在内核中分配资源并自动绑定到 device 生命周期——probe 失败或 remove 时无需手动释放。强烈推荐所有新驱动使用 devm——大幅减少 remove 函数的复杂度和内存泄漏风险。唯一例外：cdev 和 device_create 没有 devm 版本，需在 remove 中手动清理。

5. I2C 与 SPI 驱动框架

外围传感器、编解码芯片、RTC、EEPROM 大多挂在 I2C 或 SPI 总线上——这两种框架是嵌入式驱动开发的日常。

I2C 驱动框架

i2c_driver · i2c_client · i2c_transfer

设备树绑定：I2C 设备作为子节点挂在 I2C 控制器节点下。内核自动从 reg 属性创建 i2c_client。
probe 参数：i2c_probe(struct i2c_client *client)——client 包含 addr、adapter、of_node。
数据传输：i2c_transfer(adapter, msgs, num) 发送 i2c_msg 数组——每个 msg 包含 addr/flags/buf/len。SMBus 快捷函数：i2c_smbus_read_byte_data/write_byte_data 封装了常见的单字节寄存器读写。I2C 设备地址 7bit（标准）或 10bit——设备树中写 7bit 地址（内核自动左移 1 位加 R/W 位）。

SPI 驱动框架

spi_driver · spi_device · spi_transfer

设备树绑定：SPI 从设备挂在 SPI 控制器节点下，reg 为 chip select 编号。
probe 参数：spi_probe(struct spi_device *spi)——包含 max_speed_hz、mode (CPOL/CPHA)、bits_per_word。
数据传输：spi_sync_transfer() 或异步 spi_async()。支持全双工（MOSI+MISO 同时收发）。spi_message 可包含多个 spi_transfer（如先发命令字节、再读数据字节——中间 CS 保持拉低）。SPI 比 I2C 快得多（10MHz+ vs 400KHz/3.4MHz），适合高频数据传输（ADC、DAC、Flash）。

I2C 设备 — 设备树示例

// Sensor 挂在 I2C-0 上，地址 0x36 &i2c0 { status = "okay"; clock-frequency = <400000>; // 400KHz Fast Mode light-sensor@36 { compatible = "vendor,light-sensor"; reg = <0x36>; // 7-bit I2C 地址 interrupt-parent = <&pio>; interrupts = <5 IRQ_TYPE_EDGE_FALLING>; }; };

SPI 设备 — 设备树示例

// SPI Flash 挂在 SPI-1 的 CS0 &spi1 { status = "okay"; flash@0 { compatible = "jedec,spi-nor"; reg = <0>; // CS0 spi-max-frequency = <50000000>; // 50 MHz spi-cpha; // CPHA=1 spi-cpol; // CPOL=1 → SPI Mode 3 }; };

6. 中断处理与下半部

中断是设备驱动的脉搏——理解上半部/下半部分离、threaded IRQ、工作队列的选择是写出高性能驱动的关键。

上半部 (Top Half)

硬中断上下文 — 不能睡眠 — 越快越好

中断触发 → CPU 跳转到中断向量 → do_IRQ() → 调用注册的 handler。上半部运行在硬中断上下文：① 不能调用可能睡眠的函数（mutex_lock、kmalloc(GFP_KERNEL)）② 不能被同 IRQ 线中断（IRQF_DISABLED）③必须尽可能快（μs 级）——长时间占用中断上下文会让系统响应迟钝。典型操作：清中断标志、读关键寄存器、禁用该中断、触发下半部。

Threaded IRQ

request_threaded_irq — 推荐方式

request_threaded_irq(irq, hard_handler, thread_fn, flags, name, dev)。hard_handler 在硬中断上下文中运行（可返回 IRQ_WAKE_THREAD 唤醒 thread_fn），thread_fn 在内核线程中运行——可以睡眠、可以使用 mutex、可以调用 I2C/SPI。大部分驱动只需要 thread_fn 而 hard_handler 设为 NULL（内核自动用默认 handler）。这是现代 Linux 推荐的 IRQ 处理方式。

下半部机制对比

softirq · tasklet · workqueue — 选择指南

softirq：最底层的下半部——ISR 中触发，在 do_softirq 中执行。固定数量（10 种），用于网络(TX/RX)、块设备等高频关键路径。普通驱动不应使用。
tasklet：基于 softirq 的轻量下半部——同一 tasklet 不会在多 CPU 上并发，实现简单。
workqueue：运行在内核线程中——可以睡眠，适合需要调用 I2C/SPI 或持锁较长的场景。alloc_ordered_workqueue() 保证顺序执行。
推荐：首选 threaded IRQ，其次 workqueue。避免在普通驱动中使用 tasklet/softirq。

中断共享 (IRQF_SHARED) — 注意事项
多个设备可以共享同一条中断线（IRQF_SHARED）。共享时：① 所有共享该 IRQ 的 handler 都会被调用 ② handler 必须检查"是不是我的设备产生的中断"——不是则返回 IRQ_NONE ③ request_irq 必须指定 IRQF_SHARED ④ 不能使用 IRQF_ONESHOT（threaded IRQ 用 IRQF_ONESHOT 保证同一 IRQ 线在 thread_fn 执行期间不会被重新触发——与共享不兼容时需权衡）。

7. DMA 与内存映射

高性能设备驱动离不开 DMA——理解 cache 一致性与 streaming/coherent DMA 的区别是避免数据损坏的关键。

Coherent DMA (一致性)

dma_alloc_coherent — CPU 和设备看到的永远一致

不需要手动 cache 操作——内核自动保证 CPU 写入后设备立即可见，反之亦然。分配的内存通常是non-cacheable或 write-through。代价：每次 CPU 读写都绕开 cache → 性能较低。适用场景：设备控制结构体（descriptor ring）、小数据量频繁交互的 buffer。dma_alloc_coherent(dev, size, &dma_handle, GFP_KERNEL) 返回 CPU 虚拟地址 + DMA 地址。

Streaming DMA (流式)

dma_map_single — 需要手动同步 cache

使用普通的 kmalloc 内存（cacheable），但在 DMA 传输前后显式调用同步操作。CPU 写完后：dma_map_single(DMA_TO_DEVICE) → cache flush 到 DDR → 设备可读。设备写完后：dma_unmap_single(DMA_FROM_DEVICE) → cache invalidate → CPU 可读。性能高（CPU 读写时用 cache），但必须正确同步否则有数据损坏。适合：视频帧 buffer、大块数据传输。必须检查 dma_mapping_error()。

DMA-BUF 零拷贝共享

跨设备、跨驱动、跨进程的 DMA 缓冲区共享

DMA-BUF 框架允许一个驱动导出 DMA buffer（exporter），另一个驱动导入（importer）——物理内存只存在一份。IPC 相机的典型链路：ISP 驱动导出 YUV 帧 DMA-BUF → 编码器驱动（或用户态 V4L2）导入同一 DMA-BUF → 硬件编码器直接从 DDR 读帧编码。全程零 memcpy。用户态通过 DMA_BUF_IOCTL_EXPORT 获取 fd 传递给另一进程。内核态用 dma_buf_attach() + dma_buf_map_attachment()。

ioremap 与寄存器访问

物理地址 → 虚拟地址 → readl/writel

CPU 无法直接访问物理地址——需要 ioremap(phys, size) 将设备寄存器的物理地址映射到内核虚拟地址空间（返回 void __iomem *）。访问映射后的寄存器必须使用专用函数：readl/writel（32bit LE）、ioread32/iowrite32（自动处理字节序）、readb/writeb（8bit）。绝对不能用普通指针解引用（在某些架构上会出错、且破坏了 sparse 静态检查）。现代驱动用 devm_ioremap_resource() 一步完成 request_mem_region + ioremap。

8. 内核模块机制

Linux 驱动以内核模块(.ko)形式编译和加载——理解模块的编译体系、加载过程和参数传递。

模块编译 (Kbuild)

obj-m := xxx.o → make -C /lib/modules/... → xxx.ko

① 编写 Kconfig（定义配置选项和依赖）→ make menuconfig 可见 ② 编写 Makefile：obj-$(CONFIG_MY_DRIVER) += my_driver.o ③ 模块签名：内核配置 CONFIG_MODULE_SIG_FORCE → 未签名模块无法加载 ④ modpost：编译后检查未导出符号引用、license 不匹配 ⑤ .ko 文件是可重定位 ELF，insmod 时内核链接器完成符号解析（MODULE_LICENSE 会影响能引用哪些符号——GPL 模块只能引用 EXPORT_SYMBOL_GPL 的符号）。

模块参数与依赖

module_param · MODULE_DESCRIPTION · depmod

模块参数：module_param(name, type, perm) — insmod 时传递（如 insmod my_drv.ko debug=1）。参数暴露在 /sys/module/xxx/parameters/ 可运行时读写（如果 perm 允许）。
模块依赖：depmod -a 扫描所有 .ko 生成 modules.dep 依赖图。modprobe 自动加载依赖模块。符号导出：EXPORT_SYMBOL(func) 使其他模块可引用。
模块版本：MODULE_VERSION("1.0") + CONFIG_MODVERSIONS 编译时生成 CRC 校验——内核版本不同时拒绝加载。

9. 驱动调试与验证

驱动 crash 可能导致内核 panic——掌握调试工具链是驱动开发者的必备技能。

printk 与动态调试

dev_dbg · pr_debug · dynamic_debug

内核日志级别：KERN_EMERG(0)~KERN_DEBUG(7)。dmesg -n 7 让所有级别打印到控制台。
dev_* 系列：dev_err/dev_warn/dev_info/dev_dbg(dev, fmt, ...)——自动带设备名前缀。
Dynamic Debug（CONFIG_DYNAMIC_DEBUG）：运行时通过 /sys/kernel/debug/dynamic_debug/control 启用/禁用特定文件/函数的 pr_debug/dev_dbg 输出——不需要重新编译。非常适合生产环境按需开启。

sysfs / debugfs

导出驱动内部状态给用户态

sysfs：device_create_file() 在 /sys/devices/... 为设备的属性文件创建属性文件（read/write 回调）。适合导出稳定 ABI。
debugfs：debugfs_create_file/u32_regset32 在 /sys/kernel/debug/ 创建调试文件——无 ABI 稳定性承诺，可以随时修改格式。适合导出寄存器 dump、ring buffer 内容、统计计数器等调试信息。嵌入式设备可以直接在 release build 中保留——不挂载 debugfs 就零开销。

KASAN · lockdep · sparse

静态分析 + 运行时检测

sparse：make C=2，检测 __user/__iomem/__rcu 等地址空间混用、endian 错误。
lockdep：运行时锁依赖分析——检测死锁可能性（锁序违反）、中断上下文不安全的锁使用。
KASAN（Kernel Address Sanitizer）：运行时内存错误检测（use-after-free、out-of-bounds、stack overflow）——类似用户态的 ASan。代价约 2x 慢。
Kmemleak：检测内核内存泄漏——周期性扫描未引用的分配内存块。

10. IPC 相机驱动全景：从硬件到用户态

以 IPC 相机为实例，展示完整的设备驱动栈——从 Sensor 驱动到 /dev/video0 的全链路。

应用层/dev/video0 · /dev/i2c-0V4L2 应用 / GStreamer / LinkricAI
V4L2 框架video_device · vb2_queue格式协商 · 缓冲管理 (DMA-BUF) · 流控制
Sensor 驱动v4l2_subdev · I2C寄存器配置 · 曝光/增益控制 · MIPI 输出配置
MIPI 控制器驱动platform_driver · DMAMIPI D-PHY 配置 · CSI 接收 · DMA 到 DDR
ISP 驱动platform_driver · V4L2Bayer→YUV 硬件管线 · 3A 统计 · DMA-BUF 输出
硬件层Sensor → MIPI → ISP → DDR物理信号链：光子→电荷→MIPI D-PHY→CSI-2→ISP→AXI Master→DDR
IPC 相机驱动关键设计决策
Sensor 驱动：v4l2_subdev，实现 get_fmt/set_fmt、s_ctrl（曝光/增益/AWB）、s_stream（开始/停止输出）。通过 I2C 写寄存器。设备树中声明为 I2C 子节点。
MIPI 控制器：platform_driver，配置 D-PHY lane 数/速率、CSI-2 virtual channel。通常不需要用户态直接访问——由 ISP 驱动或 Media Controller 管理。
ISP 驱动：platform_driver + video_device，注册为 /dev/videoX。内部管理 3A 统计中断、DMA 完成中断。输出通过 vb2_queue + DMA-BUF 供编码器零拷贝消费。
Media Controller：通过 /dev/mediaX 暴露上述子设备的拓扑（Sensor→MIPI→ISP），应用通过 pad/link 配置数据通路。
