1. DMA 的本质：为什么要 DMA？

在没有 DMA 的时代，所有外设与内存之间的数据传输都由 CPU 逐字节搬运——每一个字节都消耗 CPU 周期。DMA 将 CPU 从"搬运工"角色中解放出来。

PIO — 编程 I/O (无 DMA)

CPU 逐字搬运 — 100% CPU 占用

流程：外设 FIFO 有数据 → 触发中断 → CPU 进入 ISR → readl() 读外设寄存器 → writel() 写内存 → 重复直到传输完成。
代价：传输 1MB 数据 = CPU 执行 ~262,144 次 readl+writel 循环——每次 ~50ns → 总 CPU 时间 ~13ms。如果每秒需要传输 30MB（1080p YUV 帧）→ CPU 390ms/s 被占用 → 单个 CPU 核心 40% 时间在搬运数据。DMA 将这一切降到接近 0%。

DMA — 直接存储器访问

硬件自动搬运 — CPU 仅发起和收尾

流程：CPU 配置 DMA 控制器（源地址、目标地址、传输长度）→ 启动 DMA → CPU 去做其他事 → DMA 控制器接管总线，直接从外设读数据写入 DDR → 传输完成 → DMA 控制器发中断通知 CPU。
收益：同样的 1MB 传输，CPU 仅需 ~1μs 配置 DMA 寄存器 + ~1μs 处理完成中断。CPU 利用率从 40% 降到 ~0.01%。这就是为什么DMA 是一切高性能 I/O 的基石。

DMA 控制器 — 总线主设备

DMA Controller / DMAC — 第三方总线控制器

DMA 控制器是挂在系统总线（AXI/AHB）上的独立硬件模块——它可以像 CPU 一样发起总线读写事务（bus master）。CPU 通过 DMA 通道寄存器告诉 DMAC 三个核心参数：源地址（从哪里读）、目标地址（写到哪里）、传输长度（多少字节）。现代 SoC 的 DMAC 有多个物理通道（8/16/32 个），可同时进行多路 DMA 传输。

2. Cache 一致性 — DMA 编程的最大陷阱

这是 DMA 编程中最常见、最隐蔽、后果最严重的 bug 来源——CPU 看到的数据和 DMA 设备看到的数据不是同一份。

问题根源：CPU Cache 与 DDR 的不一致

CPU 写到了 Cache 里，DMA 从 DDR 上读——读到的是旧数据

场景 1：CPU 写 → DMA 读
CPU 填充数据 buffer → 数据暂存在 CPU L1/L2 Cache 中（尚未写回 DDR）→ CPU 启动 DMA 从该 buffer 读取 → DMA 直接从 DDR 读 → 读到了旧数据（甚至全零，如果该页从未被写过）。
解决：DMA 启动前，CPU 必须 Cache Flush/Clean——将 dirty cache line 强制写回 DDR。

场景 2：DMA 写 → CPU 读
DMA 将数据写入 DDR → CPU 读同一地址 → CPU 从 Cache 中命中（因为之前读取过该地址）→ 读到了 DMA 写入前的旧数据。
解决：DMA 完成后、CPU 读取前，CPU 必须 Cache Invalidate——使对应的 cache line 失效，下次访问强制从 DDR 读取。

硬件 Cache Coherency (IOMMU/SMMU)

硬件自动维护一致性 — 无需软件 flush/invalidate

现代 SoC 的 IOMMU/SMMU（ARM System MMU）或 ACP（Accelerator Coherency Port）可以在硬件层面维护 DMA 与 CPU Cache 的一致性。DMA 设备通过 IOMMU 访问内存时，IOMMU 自动查询 CPU Cache 控制器——如果数据在 Cache 中，直接从 Cache 取（无需 flush）；如果数据不在 Cache 中，从 DDR 取。
代价：硬件复杂度增加，AC Port 带宽有限。并非所有 SoC 都支持。
软件方案（Linux DMA API）在无硬件一致性支持时通过 dma_map_*()/dma_unmap_*() 自动做 flush/invalidate——对驱动开发者透明。

3. Linux DMA API：Coherent vs Streaming

Linux 内核提供两套 DMA 内存管理 API——Coherent（一致性）和 Streaming（流式）。选择哪一套取决于性能要求和 buffer 使用模式。

Coherent DMA — 永远一致

dma_alloc_coherent / dma_free_coherent

特性：分配的 buffer 自动保持 CPU 和设备之间的一致性——CPU 写入后设备立即可见，设备写入后 CPU 立即可见，无需手动 flush/invalidate。
实现：内核将这片内存映射为non-cacheable或write-through——CPU 每次读写都绕过/穿透 Cache 直达 DDR。
代价：CPU 访问速度极慢（~100ns vs cached ~5ns）——不适合 CPU 频繁读写的 buffer。
适用场景：① DMA 描述符环（CPU 偶尔写，DMA 频繁读）② 小尺寸的控制结构体（几十到几百字节）③ 设备固件加载区域。
void *cpu_addr = dma_alloc_coherent(dev, size, &dma_handle, GFP_KERNEL);——返回 CPU 虚拟地址 + DMA 总线地址。

Streaming DMA — 手动同步

dma_map_single / dma_unmap_single / dma_sync_single_*

特性：buffer 可以是任何 kmalloc/vmalloc 分配的普通内存（享受 CPU Cache 加速）。仅在 DMA 传输前后显式调用同步操作。
方向标记：
• DMA_TO_DEVICE：CPU 写 → 设备读 → map 时 flush cache
• DMA_FROM_DEVICE：设备写 → CPU 读 → unmap 时 invalidate cache
• DMA_BIDIRECTIONAL：双向——map flush + unmap invalidate
适用场景：① 视频帧 buffer（CPU 需要高带宽读写 YUV 数据）② 网络包 buffer（sk_buff 使用 streaming DMA）③ 大块应用数据（MB 级别）
性能：CPU 侧访问享受全速 Cache，DMA 侧通过 flush/invalidate 保证正确性——这是高性能 DMA 的标准选择。

⚠ Streaming DMA 的 4 条铁律（违反任何一条 = 数据损坏）
1. 在 dma_map 和 dma_unmap 之间，CPU 绝对不能访问 buffer。这段时间 buffer 属于设备。dma_sync_single_for_cpu 可以临时交还 CPU 访问权。
2. 永远检查 dma_mapping_error()。某些平台（如使用 swiotlb 的 32bit DMA 设备 + 64bit 系统）映射可能失败——忽略检查 = 使用 0x00000000 地址 DMA = 静默损坏数据。
3. dma_map 返回的是 DMA 地址（总线地址），不是物理地址。IOMMU 存在时 DMA 地址 ≠ 物理地址——绝对不能用 virt_to_phys 算 DMA 地址。
4. unmap 的方向必须与 map 一致。如果 map 时标记 TO_DEVICE 但实际设备也写入了数据 → unmap 时不会 invalidate → CPU 读到旧数据。

4. DMA Engine 框架 — 统一的 DMA 驱动模型

Linux 的 dmaengine 子系统抽象了不同厂商的 DMA 控制器硬件——提供统一的 API 让外设驱动请求和使用 DMA 通道，屏蔽底层 DMAC 差异。

核心概念

dma_device · dma_chan · dma_async_tx_descriptor

dma_device：一个 DMA 控制器实例（SoC 中的 DMAC IP 块）。通过 dma_async_device_register() 注册到 dmaengine 核心。提供 device_alloc_chan_resources、device_prep_dma_memcpy/slave_sg/cyclic 等回调。
dma_chan：DMA 控制器的一条物理/逻辑通道——外设驱动通过 dma_request_chan() 获取通道。
dma_async_tx_descriptor：一次 DMA 传输的描述符——包含源/目标地址、长度、完成回调。dmaengine_submit() → dma_async_issue_pending() 提交并启动。

传输类型

Mem-to-Mem · Slave-DMA · Cyclic · Interleaved

Mem-to-Mem（device_prep_dma_memcpy）：内存到内存拷贝——CPU 完全解放。用于加速 memcpy 操作（如 GPU 纹理上传）。
Slave-DMA（device_prep_slave_sg）：外设 ↔ 内存——最常见。用于 MMC/SD 读写、SPI Flash DMA、UART DMA、I2S 音频 DMA。
Cyclic（device_prep_dma_cyclic）：环形缓冲自动循环——I2S 音频播放/录音的核心模式。DMA 自动从环尾回到环首。
Interleaved：用于 2D/3D 图形——支持 stride（行间距）的非连续传输。

Scatter-Gather (SG)

多个不连续的内存片段 → 一次 DMA 传输

问题：应用程序的 buffer 可能由多个物理上不连续的页组成（通过 mmap 或 ION 分配）。CPU 看到连续的虚拟地址，但 DMA 需要物理上连续的地址。
SG 解决方案：构建一个 scatterlist——每个条目描述一段连续的物理内存（page + offset + length）。DMA 控制器自动遍历 SG 表——一段传完自动跳到下一段——对应用程序透明。
dma_map_sg() 遍历 scatterlist，为每段生成 DMA 地址，处理 Cache 一致性——是 dma_map_single 的 SG 版本。

// 外设驱动使用 DMA Engine 的典型流程（Slave-DMA 模式） // 1. 请求 DMA 通道 struct dma_chan *chan = dma_request_chan(dev, "tx"); // 设备树 dmas 属性解析 // 2. 准备 slave 配置（外设 FIFO 地址、突发大小、方向） struct dma_slave_config cfg = { .direction = DMA_MEM_TO_DEV, .src_addr = 0, // 内存地址由 scatterlist 提供 .dst_addr = fifo_phys_addr, // 外设 FIFO 寄存器物理地址 .src_addr_width = DMA_SLAVE_BUSWIDTH_4_BYTES, .dst_addr_width = DMA_SLAVE_BUSWIDTH_4_BYTES, .src_maxburst = 16, // 一次突发传输 16×4=64 字节 .dst_maxburst = 16, }; dmaengine_slave_config(chan, &cfg); // 3. 映射 scatterlist + 准备传输描述符 int nents = dma_map_sg(dev, sg, sg_len, DMA_TO_DEVICE); struct dma_async_tx_descriptor *tx = dmaengine_prep_slave_sg(chan, sg, nents, DMA_MEM_TO_DEV, DMA_PREP_INTERRUPT); // 4. 设置完成回调 + 提交传输 tx->callback = my_dma_callback; tx->callback_param = my_data; dmaengine_submit(tx); dma_async_issue_pending(chan); // 启动！ // 5. 传输完成 → 回调中 unmap + 通知上层 static void my_dma_callback(void *data) { dma_unmap_sg(dev, sg, nents, DMA_TO_DEVICE); complete(&done); // 唤醒等待的线程 }

5. DMA-BUF — 跨设备零拷贝共享

DMA-BUF 是 Linux 内核的跨设备、跨驱动、跨进程 buffer 共享框架——一块物理内存，ISP 写入、编码器读取、GPU 渲染、网络发送，全程零 memcpy。

Exporter-Importer 模型

Exporter 分配并导出 → Importer 导入并访问

Exporter（导出者）：拥有 buffer 的驱动。调用 dma_buf_export() 创建一个 struct dma_buf 对象，绑定一组 dma_buf_ops（attach/detach/map/unmap/mmap/vmap）。返回一个 fd（文件描述符）——通过 Unix Socket/ioctl 传递给其他进程。
Importer（导入者）：使用 buffer 的驱动。通过 dma_buf_get(fd) 从 fd 获取 dma_buf 对象 → dma_buf_attach() 注册自己的设备 → dma_buf_map_attachment() 获取自己设备可见的 DMA 地址（不同设备可能看到不同的总线地址——IOMMU 各自映射）。

IPC 相机 DMA-BUF 全链路

Sensor → ISP → DMA-BUF → Encoder → DMA-BUF → Network

① ISP 驱动 (Exporter)：分配 3 块 DMA buffer 做轮转缓冲 → ISP 硬件完成一帧后，DMA 写入 buffer → ISP 驱动调用 dma_buf_export() 导出 fd → 通过 V4L2 VIDIOC_EXPBUF ioctl 传给用户态。
② 用户态 LinkricAI 进程：拿到 fd → 通过 Unix Socket SCM_RIGHTS 传给编码进程。
③ 编码驱动 (Importer)：dma_buf_get(fd) → attach → map → 配置编码器硬件从该 DMA 地址读取 YUV 帧 → 硬件编码 → 编码输出写入另一个 DMA-BUF → 导出新 fd。
④ 网络进程：导入编码输出的 DMA-BUF fd → sendmsg 直接引用 DMA buffer → 网卡 DMA 从 buffer 读取发送。
全程零 CPU memcpy——所有数据搬移都在硬件 DMA 层面完成。

6. 设备树 — DMA 资源描述

DMA 通道和请求信号的分配通过设备树描述——内核驱动通过标准 API 解析，无需硬编码通道号。

DMA 控制器节点

dma-controller · #dma-cells · dma-channels

SoC 的 DMA 控制器在设备树中声明自己有多少通道以及 #dma-cells（客户端引用时需要多少个参数）。DMA 引擎驱动 probe 时解析这些属性，分配通道资源。常见 DMAC IP：ARM PL330（arm,pl330）、Synopsys DesignWare（snps,dma-spear1340）、Intel IDMA、Freescale eDMA。

客户端引用

dmas · dma-names — 声明需要的 DMA 通道

外设节点通过 dmas 属性引用 DMA 控制器 + 请求信号编号：
dmas = <&dmac0 12>, <&dmac0 13>;
dma-names = "tx", "rx";
驱动通过 dma_request_chan(dev, "tx") 获取——内核自动解析设备树找到对应的 DMA 控制器和请求信号号。
dma_request_slave_channel_compat() 兼容旧设备树写法（基于 channel index 而非 name）。现代驱动应使用 dma_request_chan。

7. DMA 调试与常见问题

DMA 问题的排查比普通驱动更难——错误通常表现为"数据偶尔不对"而非 crash，且难以复现。

问题 1: Cache 不一致

最常见 — 症状：数据偶尔错、部分字节对、部分字节旧

典型症状：视频帧上半部分正常、下半部分是上一帧的内容（Cache line 64 字节边界效应）。
排查：① 确认 dma_map/dma_unmap 调用正确且方向匹配 ② 检查 dma_map 和 unmap 之间 CPU 是否访问了 buffer ③ 尝试将 buffer 改为 dma_alloc_coherent（排除 Cache 问题）。
预防：驱动 review 时逐行检查 map→unmap 区间内所有 buffer 访问——这条是 DMA review 的最核心检查项。

问题 2: DMA 地址错误

dma_map 失败 · 地址越界 · 32bit DMA 碰 64bit 内存

典型症状：DMA 传输完成后目标 buffer 是空的/全0/随机数据。
根因：① dma_mapping_error() 未检查——映射失败返回 DMA_MAPPING_ERROR ② 在 IOMMU 存在时误用 virt_to_phys 作为 DMA 地址——必须用 dma_map 返回的地址 ③ 32bit DMA 设备 + 4GB 以上物理内存 → 需要 swiotlb 反弹缓冲（性能骤降但不报错）。
工具：cat /sys/kernel/debug/dma-api/dump 查看所有活跃的 DMA 映射。

问题 3: 传输未完成

DMA 中断丢失 · 描述符配置错误 · 传输静默停止

典型症状：驱动程序等待完成事件超时，或 buffer 部分有数据、部分没有。
排查：① /proc/interrupts 确认 DMA 通道中断计数是否增加 ② 检查描述符链是否正确——interrupt flag 是否设置 ③ 确认 DMA 控制器时钟是否被 pm_runtime 意外关闭 ④ 检查突发大小/对齐是否符合硬件限制（如 DMA 要求 4 字节对齐但 buf 是 2 字节对齐）。
CONFIG_DMA_API_DEBUG=y：内核编译该选项后，自动检测 DMA API 的违反使用规则（如 map 两次、unmap 未映射的地址、方向不匹配）。

从 Sensor MIPI 到网络发送——IPC 相机中每一跳数据搬移都应该通过 DMA 完成。以下是从硬件上电到网络发包的完整 DMA 链路。

IPC DMA 设计决策 1: ISP Buffer 数量

3 vs 4 vs 6 buffer — 延迟 vs 安全的权衡

最少 3 块：一块 ISP 正在写入、一块编码器正在读取、一块在排队。
推荐 4-6 块：编码速度不稳定时（I 帧比 P 帧慢 2-5×），多余的 buffer 吸收瞬时抖动——避免 ISP 因没有空闲 buffer 而丢帧。
DMA-BUF 方式分配：ISP 驱动通过 dma_alloc_coherent 或 CMA 分配物理连续内存（ISP 的 AXI Master 可能不支持 SG），导出 DMA-BUF fd 给编码器。比 ION 更标准、更通用。

IPC DMA 设计决策 2: 轮转策略

Fence 同步 vs 信号量 vs 无锁轮转

DMA Fence（dma_fence / sync_file）：内核 DMA 框架提供的同步原语——ISP 的 DMA 完成后 signal fence → 编码器 import fence → 编码器 DMA 等待 fence 才启动。硬件级精确同步，比用户态信号量可靠且低延迟。
简单轮转（适用于无 fence 支持的旧硬件）：Buffer 状态标记（FREE/FILLING/READY/READING）→ 生产者和消费者各自维护读/写指针 → 必须使用多 buffer（≥3）防止读写冲突。

IPC 相机 DMA 配置 Checklist
□ 内核配置：CONFIG_DMA_ENGINE=y · CONFIG_DMA_SHARED_BUFFER=y · CONFIG_DMA_CMA=y（Contiguous Memory Allocator——为 ISP 分配大块物理连续内存）
□ 设备树：MIPI CSI 控制器的 dmas/dma-names、ISP 节点的 dmas/dma-names、MMC 控制器的 dmas/dma-names
□ CMA 大小：内核命令行 cma=128M——为 ISP/编码器预留足够连续内存（1080p YUV420 单帧 ~3MB，6 块 buffer = 18MB，再加编码输出 buffer）
□ IOMMU：如果 SoC 支持 SMMU，为 ISP 和编码器配置 IOMMU domain——每个设备看到独立的 DMA 地址空间，即使物理内存不连续也能 DMA（IOMMU 做虚拟 DMA 地址→物理地址映射）
□ DMA-BUF 权限：确保 buffer fd 可以在进程间传递——/dev/dma_heap 或 ION 兼容接口
□ 验证方法：cat /sys/kernel/debug/dma-api/dump 确认每层的 DMA 映射都正确创建；对比 CPU 利用率的 memcpy 占比——理想情况下 <1%
