1. USB 总线基础：拓扑 · 描述符 · 端点 · 传输类型

理解 USB 协议核心概念是看懂整个驱动栈的前提——主机侧和设备侧共用这些基础。

总线拓扑

树形结构 — 一个 Host · 最多 127 设备 · 最多 7 层 Hub

USB 总线是严格的星型拓：一个 USB Host（根 Hub）通过 Hub 级联扩展。Host 是唯一的通信发起者——设备不能主动发送数据，只能响应 Host 的请求（IN token 或 OUT token）。即使用了中断端点，也是 Host 周期性轮询。这与网络（对等通信）有本质区别。USB OTG（On-The-Go）允许设备在 Host 和 Device 角色间切换。

设备描述符体系

设备→配置→接口→端点 — 四级描述符树

每个 USB 设备有一组层级化的描述符向 Host 宣告自己的能力：
设备描述符：VID/PID、设备类、制造商/产品字符串。
配置描述符：供电需求、接口数量——设备可有多个配置（如低功耗 vs 高性能模式）。
接口描述符：一组端点集合，代表一个逻辑功能（如 UVC 的 VideoControl + VideoStreaming 接口）。
端点描述符：端点号、方向(IN/OUT)、传输类型、最大包大小。

端点 (Endpoint)

通信的终点 — 由地址+方向唯一标识

端点是 USB 通信的基本单元。每个端点有：端点号（1-15，EP0 为默认控制端点保留）、方向（IN=设备→Host，OUT=Host→设备）、传输类型、最大包大小。端点 0 是唯一的双向控制端点，用于枚举和配置。端点存在于设备端——Host 端与之通信时使用管道 (Pipe)——管道是 Host 端对设备端点的逻辑关联。

四种传输类型

Control · Bulk · Interrupt · Isochronous

控制传输 (Control)：EP0 专用。可靠、有 CRC 校验、有重试。用于枚举（获取描述符、设置地址、选择配置）和类特定控制请求。最大包大小：FS 8/16/32/64B、HS 64B。
批量传输 (Bulk)：可靠、有 CRC 校验和重试、但不保证延迟——利用总线空闲带宽，低优先级。USB2.0 最大 512B/包，USB3.0 最大 1024B/包。适合大块数据：UVC 视频流、Mass Storage、ADB。Burst 模式在 USB3.0 中可连续多包。

（续）

中断传输 · 等时传输 — 定时 vs 可靠性

中断传输 (Interrupt)：Host 周期性轮询（间隔在端点描述符中定义，1-255ms 或 125μs-4s）。可靠——有 CRC 和重试。HID 设备（键盘/鼠标）的核心传输方式。最大包：LS 8B、FS 64B、HS 1024B。
等时传输 (Isochronous)：固定带宽保证（预约总线时间），但无 CRC 校验、无重试——丢了就丢了。适合实时流：USB 音频（耳机/麦克风）、UVC 摄像头的视频流端点几乎都是等时。HS 最大 1024B/包、每个微帧(125μs)可传输 3 个包 → 24MB/s。

XHCI (USB 3.0+)

eXtensible Host Controller Interface

USB 3.x 的唯一标准，也向下兼容 USB 2.0/1.1。驱动：xhci-hcd。支持 SuperSpeed (5Gbps) / SuperSpeed+ (10Gbps/20Gbps)。关键特性：① 原生 Stream 支持（UAS 协议必需）② 多端口并行处理 ③ 更高效的 TD (Transfer Descriptor) 链——减少 CPU 中断频率。现代 SoC 几乎全部用 XHCI。

EHCI (USB 2.0)

Enhanced Host Controller Interface

USB 2.0 High-Speed (480Mbps) 标准。驱动 ehci-hcd。通过同伴控制器 (Companion Controller) 机制将 FS/LS 设备路由到 OHCI/UHCI。周期性调度（Periodic Schedule）+ 异步调度（Asynchronous Schedule）双链表。在 XHCI 普及后逐渐被替代，但大量现有 IPC SoC 仍使用 EHCI+OHCI 组合。

URB (USB Request Block)

内核中 USB 通信的基本单元

USB 设备驱动不直接操作硬件寄存器——通过 URB（USB Request Block）向 USB Core 提交 I/O 请求。URB 包含：端点地址、传输类型、数据 buffer、完成回调函数。生命周期：usb_alloc_urb() → usb_fill_*_urb() → usb_submit_urb() → completion callback → usb_free_urb()。异步模型：submit 后立即返回→传输完成时回调通知。也支持同步等待（usb_control_msg() 等封装函数）。

UVC 驱动 (uvcvideo) — USB 视频类设备驱动实例
drivers/media/usb/uvc/ 是 Host 侧 UVC 驱动的完整实现——当 USB 摄像头插入 Host 时，该驱动通过 USB Core 枚举设备、解析 UVC 特定描述符（VC 接口 + VS 接口）、注册 V4L2 video_device 暴露为 /dev/videoX。支持 H.264/MJPEG/YUV 多种格式、通过 UVC 扩展单元（XU）控制 PTZ/OSD。视频数据通过等时或批量端点以 URB 方式接收。

3. Gadget 框架：让 Linux 设备变成 USB 外设

Gadget 是 Linux 的独特能力——让嵌入式设备在 USB 总线上扮演 Device 角色（U 盘模式、USB 摄像头、ADB 调试、虚拟网卡）。IPC 相机的核心竞争力。

Gadget 框架三层架构

UDC 驱动 → Gadget API → Function 驱动

UDC 层 (Controller Driver)：SoC 的 USB Device 控制器硬件驱动。实现 struct usb_gadget_ops（处理 EP0 控制请求、配置端点、启动/停止传输）。常见驱动：dwc3（DesignWare USB3.0）、musb（Mentor Graphics）、dwc2（DesignWare USB2.0 OTG）。
Gadget API 层：Core 层，提供 usb_gadget、usb_ep、usb_request 抽象——Function 驱动通过这套 API 与 UDC 交互。
Function 层：具体的 USB 功能实现（UVC/UAC/HID/Mass Storage/ADB/RNDIS/NCM）。每个 Function 实现一组端点+类特定请求处理。

ConfigFS — 用户态组合 Gadget

/sys/kernel/config/usb_gadget/ — 无需重编译内核

Linux 3.x 引入的 ConfigFS 是配置 Gadget 的现代标准方式——完全在用户态通过文件系统操作组合 Function，无需修改内核代码。
操作流程：mkdir /sys/kernel/config/usb_gadget/g1 → 写入 VID/PID/序列号等属性 → mkdir functions/uvc.0 创建 UVC 功能 → mkdir functions/ffs.adb 创建 ADB（FunctionFS 方式）→ ln -s 将 Function 链接到配置 → echo "UDC name" > UDC 激活。
优势：灵活组合（UVC+ADB+HID 三合一）、无需重编译、脚本化部署。

// ConfigFS 配置一个复合 USB 设备（UVC + ADB + HID） // 1. 创建 gadget 实例 cd /sys/kernel/config/usb_gadget/ mkdir g1 && cd g1 // 2. 设置设备级描述符 echo 0x1d6b > idVendor // Linux Foundation VID echo 0x0104 > idProduct echo 0x0100 > bcdDevice // 设备版本 1.0 echo 0x0200 > bcdUSB // USB 2.0 // 3. 设置字符串描述符 mkdir strings/0x409 // 英文 (en-US) echo "IPC Camera Inc." > strings/0x409/manufacturer echo "IPC-CAM-001" > strings/0x409/product echo "ABC123456" > strings/0x409/serialnumber // 4. 创建配置和 Functions mkdir configs/c.1 mkdir configs/c.1/strings/0x409 echo "UVC+ADB+HID" > configs/c.1/strings/0x409/configuration mkdir functions/uvc.0 // UVC 摄像头功能 mkdir functions/ffs.adb // ADB (通过 FunctionFS) mkdir functions/hid.usb0 // HID 按键（复位/抓图） echo 1 > functions/hid.usb0/protocol echo 8 > functions/hid.usb0/report_length // 5. 将 Functions 链接到配置 ln -s functions/uvc.0 configs/c.1/ ln -s functions/ffs.adb configs/c.1/ ln -s functions/hid.usb0 configs/c.1/ // 6. 激活：绑定 UDC 控制器 echo "38100000.dwc3" > UDC // 开始枚举！

4. UVC Gadget — USB 摄像头功能

将 IPC 相机变为标准 USB 摄像头——插入电脑直接识别为 Camera，无需驱动安装，无需网络配置。

UVC Gadget 的架构

内核 UVC Function + 用户态 V4L2 Source

UVC Gadget 由两部分组成：内核态：drivers/usb/gadget/function/f_uvc.c——处理 USB 枚举、UVC 类特定请求（VS_PROBE_COMMIT、VS_COMMIT_CONTROL）、管理视频流的等时/批量端点。通过 /dev/uvc-gadget 字符设备与用户态交互。
用户态：一个守护进程从 V4L2 设备（ISP 输出 /dev/videoX）读取视频帧，写入 /dev/uvc-gadget → 内核通过 USB 等时端点发送给 Host。这就是整个数据路径。

UVC 描述符结构

VC 接口 (控制) + VS 接口 (流) — 两个接口

VideoControl (VC) 接口：用于协商和控制——处理 VS_PROBE_CONTROL（Host 询问支持哪种格式/分辨率/帧率）和 VS_COMMIT_CONTROL（Host 确认选择）。端点 0 上的类特定请求。
VideoStreaming (VS) 接口：实际视频数据传输——一个等时或批量 IN 端点。每个微帧(125μs)传输多个包。支持多种格式：Uncompressed (YUYV/NV12/MJPEG) 和 Frame-Based (H.264/H.265)。后者允许传输编码后的帧——大幅降低 USB 带宽需求。

H.264 编码流 over UVC

Frame-Based Payload — 关键能力

UVC 1.1+ 支持 Frame-Based Payload——视频流端点传输的是编码后的 H.264/H.265 码流而非原始 YUV。Payload Header 标记每帧边界。这对 IPC 相机至关重要：1080p YUV @30fps 需要 ~1.5Gbps USB 带宽——USB2.0 HS (480Mbps) 无法承载。但 H.264 编码后 ~2-4Mbps → 轻松走 USB2.0 Bulk 端点。Host 侧的 UVC 驱动自动识别 Frame-Based 格式并通过 V4L2 的 V4L2_PIX_FMT_H264 输出给上层。

⚠ UVC Gadget 的关键配置参数
等时 vs 批量：等时端点固定带宽但无重试——适合没有缓存压力的实时场景；批量端点利用空闲带宽、有 CRC 重试——H.264 编码流推荐用批量（可靠性优先）。
dwMaxPayloadTransferSize：HS 等时最大 3072B（3×1024），USB3.0 最大 16KB。
dwMaxVideoFrameBufSize：VS_PROBE_CONTROL 返回的最大帧缓冲大小——必须大于实际编码帧的最大尺寸。
bEndpointAddress：等时 IN 端点使用 0x81 这种方向位为 1 的地址。

5. FunctionFS (FFS) 与 ADB Gadget

FunctionFS 将 USB Function 的全部协议逻辑移到用户态——内核只负责端点数据传输，用户态处理 USB 控制请求和协议。ADB 是 FFS 的最佳实践范例。

FunctionFS (FFS) 原理

/dev/usb-ffs/ — 用户态 USB Function 框架

FFS 允许用户态程序完整实现一个 USB Function而不写任何内核代码。用户态通过 /dev/usb-ffs/XXX/ep0 接收原始 USB 控制请求（setup packet + data stage），通过 /dev/usb-ffs/XXX/ep1（批量 IN）、/dev/usb-ffs/XXX/ep2（批量 OUT）收发端点数据。用户态程序可以是任何语言——Python、C、Go 均可。
工作流：① mkdir functions/ffs.adb → 创建 /dev/usb-ffs/adb/ 目录 ② 用户态程序打开 ep0 → 内核认为 Function 就绪 → 向 Host 宣告该 Interface ③ 用户态循环 read ep0（等待 Host 的 USB 控制请求）→ 处理 → write ep0 应答 ④ 同时 read/write 批量端点进行数据传输。

ADB (Android Debug Bridge)

adb daemon → FunctionFS → USB Gadget → Host adb client

ADB over USB 使用两个批量端点（IN+OUT）进行全双工通信。协议简单：AUTH（RSA 签名认证）→ CNXN（连接，交换版本+系统标识）→ OPEN（打开服务，如 shell:、sync:、framebuffer:）→ WRTE/OKAY/CLSE（数据传输和关闭）。
ADB 守护进程（adbd）打开 FFS ep0+两个批量端点，实现完整的 ADB 协议栈。Android 设备出厂就支持，嵌入式 Linux 同样可以——只需启动 adbd 并使用 ConfigFS 配置 ffs.adb 即可。对于 IPC 相机：ADB over USB = 工厂调试、固件烧录、logcat 抓日志的最便捷通道。

ADB 协议交互流程（简化版）
// Host(adb client) ←──USB BULK──→ Device(adbd via FFS) Phase 1: 认证 Host → AUTH(TOKEN, random_bytes) → Device Device → AUTH(SIGNATURE, rsa_sign(token)) → Host Host → AUTH(RSAPUBLICKEY, public_key) → Device // 首次连接时 Phase 2: 连接 Host → CNXN(0x01000000, 4096, "host::") → Device Device → CNXN(0x01000000, 4096, "device::") → Host Phase 3: 服务 (例如 adb shell) Host → OPEN(0, "shell:ls /data") → Device Device → OKAY(0, 1) → Host // local_id=0, remote_id=1 Device → WRTE(1, 0, "file1\nfile2\n") → Host // stdout 输出 Device → CLSE(1, 0) → Host // shell 命令执行完毕

HID Gadget — 自定义按键

Report Descriptor → /dev/hidg0 → write event → Host 识别

HID Function 通过HID Report Descriptor（一段二进制描述表）向 Host 宣告自己能发送什么类型的输入事件。例如：定义一个 Consumer Control 设备（音量+/-、静音、播放/暂停）或键盘设备（发送组合键）。用户态程序打开 /dev/hidg0，写入符合 Report Descriptor 格式的二进制数据 → Host 侧收到标准 HID 事件。
IPC 用途：PC 上的视频预览软件可以响应"抓图"按键 → 无需安装额外驱动。

NCM (Network Control Model)

USB Ethernet — 比 RNDIS 更标准、更高效

NCM（CDC-NCM）是USB-IF 标准的 USB 以太网协议（RNDIS 是 Microsoft 私有协议）。功能：NCM Function → 在 Host 侧创建 usbX 网络接口 → Device 侧创建 usbX 接口 → 通过批量端点传输以太网帧。支持多帧聚合（一个 USB 包中打包多个以太网帧——减少 USB 协议开销）。IPC 相机用途：USB 线直连 PC 实现 TCP/IP 通信——无需 WiFi/以太网，USB 线即网线。适合工厂产线配置、桌面调试。

7. UDC 驱动 — USB Device 控制器的硬件抽象

UDC (USB Device Controller) 驱动是 Gadget 框架的最底层——将 SoC 的 USB Device 硬件寄存器映射为 Gadget API

UDC 驱动实现的回调 (usb_gadget_ops)

SoC USB Device IP → usb_ep → usb_request

udc_start/stop：激活/停用控制器——连接/断开 D+ 上拉电阻。
udc_pullup：控制 D+ 上拉——通知 Host "设备已插入"（软件控制连接）。
ep_enable/disable：配置端点（地址、类型、最大包大小）→ 分配硬件 FIFO。
ep_queue/dequeue：提交/取消一个传输请求。
ep_set_halt/clear_halt：设置/清除端点 STALL 状态。
处理 EP0 SETUP 包：解析标准设备请求（GET_DESCRIPTOR/SET_ADDRESS/SET_CONFIGURATION）并响应。UDC 驱动负责自动处理这些标准请求——Gadget Function 只处理类特定请求。

常见 UDC 硬件 IP

DesignWare DWC3/DWC2 · MUSB · ChipIdea

DWC3（drivers/usb/dwc3/）：Synopsys USB3.0 DRD（Dual Role Device）控制器——嵌入式 SoC 的绝对主流（Rockchip、全志、高通、MTK、Intel 都在用）。支持 Host/Gadget/OTG 三种角色。
DWC2（drivers/usb/dwc2/）：USB2.0 HS OTG 控制器。常见于中低端 SoC。
MUSB（drivers/usb/musb/）：TI 设计的 Mentor USB OTG 控制器，TI/Allwinner 老型号使用。
ChipIdea（drivers/usb/chipidea/）：Freescale/i.MX 系列常用。
设备树中通过 dr_mode 属性指定角色："host"（仅 Host）、"peripheral"（仅 Gadget）、"otg"（可切换）。

设备树 — DWC3 控制器配置示例（支持 OTG 双角色）
usb@38100000 { compatible = "rockchip,rk3399-dwc3"; reg = <0x38100000 0x10000>; clocks = <&cru ACLK_USB3>; clock-names = "ref", "bus_early", "suspend"; usb@38100000 { compatible = "snps,dwc3"; reg = <0x38100000 0x10000>; interrupts = <GIC_SPI 20 IRQ_TYPE_LEVEL_HIGH>; dr_mode = "otg"; // ← 关键：支持 Host/Gadget 切换 maximum-speed = "high-speed"; // 或 "super-speed" phys = <&usb2_phy>, <&usb3_phy>; phy-names = "usb2-phy", "usb3-phy"; snps,dis_u2_susphy_quirk; snps,dis_u3_susphy_quirk; }; };

8. USB 调试与诊断

USB 问题排查的三板斧——dmesg、usbmon、sysfs + debugfs

dmesg — 枚举日志

内核 USB 子系统打印的所有事件

dmesg | grep -i usb：设备连接/断开、描述符解析、驱动绑定/解绑、URB 错误、传输超时。
开启详细日志：echo 1 > /sys/module/usbcore/parameters/usbfs_debug
开启 UVC 驱动调试：echo 0xFFFF > /sys/module/uvcvideo/parameters/trace（内核需编译 CONFIG_DYNAMIC_DEBUG）
关键信息：VID/PID、选择的配置、端点地址、dwMaxPacketSize、驱动 probe 是否成功。

usbmon — USB 总线嗅探

类似 tcpdump 的 USB 包捕获

Host 侧：mount -t debugfs none /sys/kernel/debug; cat /sys/kernel/debug/usb/usbmon/0u > trace.log → 捕获所有 USB bus 上的原始 URB（包括 Setup/Data/Status 阶段的完整内容）→ 用 Wireshark 打开 trace.log 分析。这是 USB 调试的终极大招——能看出描述符返回是否正确、端点数据是否到达、传输是否完成。
Gadget 侧：trace-cmd record -e dwc3 -e gadget 捕获 DWC3 控制器的 ftrace 事件（请求排队、完成、STALL、ep enable/disable）。

sysfs / debugfs

/sys/class/udc/ /sys/kernel/debug/usb/

/sys/class/udc/：列出可用的 UDC 控制器（如 38100000.dwc3）。
/sys/kernel/config/usb_gadget/：ConfigFS gadget 配置——查看当前激活的 Function、VID/PID、字符串。
/sys/kernel/debug/usb/：设备描述符树（devices 文件：T: Bus/Topology, D: Device, P: Port, I: Interface, E: Endpoint）、每个端点的传输统计。
/sys/kernel/debug/gadget：Gadget 侧的端点队列状态、传输统计。与 /sys/kernel/debug/usb/devices 对应——Host 侧看到的描述符和 Gadget 侧配置的一不一致。

调试场景命令说明
Gadget 未出现在 Host 上cat /sys/class/udc/*/state确认 UDC 状态——不是 "configured" 说明未完成枚举
Host 无法识别设备cat /sys/kernel/debug/usb/devices查看 Host 解析到的描述符——检查 VID/PID/接口是否正确
UVC 无视频流usbmon + Wireshark确认 VS_PROBE_COMMIT 是否成功→流端点是否有数据
USB 3.0 降级为 2.0dmesg | grep "SuperSpeed"PHY 配置 / VBUS / 线缆质量问题导致 SS 协商失败
大量传输错误 (-EPROTO)trace-cmd record -e dwc3DWC3 硬件 FIFO 溢出或 PHY 时钟不稳定
复合设备只显示一个 Function检查 ConfigFS 链接是否正确每个 Function 都必须 ln -s 到 configs/c.1/

9. IPC 相机 USB Gadget 实战方案

IPC 相机通过 USB 线连接 PC 的完整方案——从硬件连接到软件栈

方案 1: UVC + ADB 双模（推荐）

USB 摄像头预览 + 调试通道二合一

配置：ConfigFS 创建 UVC(uvc.0) + ADB(ffs.adb) 复合设备。
UVC 路径：IPC 的 ISP 输出 YUV → H.264 硬件编码 → Frame-Based Payload → UVC VS 批量端点 → Host 侧 uvcvideo 驱动 → PC 播放器（VLC/OBS/Chrome）直接显示。
ADB 路径：PC 端 adb shell → USB 批量端点 → 设备端 adbd → 命令行访问完整 Linux 系统。
优势：免驱预览（Windows/macOS/Linux 自带 UVC 驱动）、免驱调试（ADB 协议成熟稳定）、一根 USB 线同时做视频和调试。

方案 2: NCM + HTTP-RTSP

USB 虚拟网卡 → TCP/IP → 标准流媒体

配置：ConfigFS 创建 NCM(ncm.0) 复合设备。
网络路径：USB NCM → Host 侧 usbX 网卡（自动获取 169.254.x.x 链路本地地址）→ Device 和 Host 通过 TCP/IP 通信。
视频路径：IPC 运行 RTSP Server → Host 用 VLC 打开 rtsp://169.254.x.x:554/stream1。
优势：不受 USB 协议限制——可以传视频、文件、API、HTTP——完整的 TCP/IP 通信。适合自动化产测。

方案 3: Mass Storage（SD 卡导出）

IPC 的 SD 卡当 U 盘用

配置：ConfigFS 创建 mass_storage.0 → 指定 backing file（如 /dev/mmcblk0p1 或 loop 设备）。
注意：导出 SD 卡时必须确保 IPC 没有在写入（停止录像、卸载文件系统）→ 否则 Host 修改文件系统会导致数据损坏。解决方案：① 导出只读分区 ② 通过 UVC+HID 发送"停止录像"→"导出 SD"→"恢复录像"的状态机控制。

方案 4: 三合一（UVC + ADB + NCM）

USB3.0 SuperSpeed — 带宽充裕的组合

USB3.0 (5Gbps) 有足够的带宽同时承载：UVC 视频流（H.264 ~4Mbps）+ ADB 调试（突发 Bulk）+ NCM 网络（TCP/IP 复用）。一个 USB 线同时：视频预览、adb 调试、TCP/IP 固件升级。
USB2.0 HS (480Mbps 理论值，实际 ~240Mbps) 下同时跑 UVC+NCM 可能带宽不足——优先使用H.264 编码流降低 UVC 带宽占用，为 NCM 留出余量。

IPC 相机 USB Gadget 部署 Checklist
□ 内核编译 CONFIG_USB_GADGET=y + CONFIG_USB_CONFIGFS=y + CONFIG_USB_CONFIGFS_F_UVC=y + CONFIG_USB_CONFIGFS_FFS=y
□ 设备树 dr_mode = "peripheral" 或 "otg"（确保 USB 连接器方向正确——ID 引脚接地使能 Gadget 模式）
□ 启动时执行 ConfigFS 脚本创建复合 gadget（UVC+ADB 或其他组合）→ echo UDC name 激活
□ UVC 用户态守护进程：从 /dev/videoX 读帧 → 写入 /dev/uvc-gadget（确保帧率匹配 Host 请求的 dwFrameInterval）
□ ADB：启动 adbd 守护进程 → 打开 /dev/usb-ffs/adb/ep0 → 等待 Host 的 USB 控制请求
□ HID（可选）：配合 /dev/hidg0 发送按键事件（如 PC 端一键抓图）
□ 测试：dmesg | tail -50 确认枚举成功 → Host 侧 lsusb 看到复合设备 → UVC 用 VLC/Chrome 打开 → adb devices 看到设备 → NCM 网卡 ping 通
