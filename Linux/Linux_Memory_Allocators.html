ptmalloc2 (1996→)

glibc · Doug Lea → Wolfram Gloger

基于 dlmalloc 的多线程版本。arena 锁 + bin 体系。Linux 默认分配器，兼容性最好 但多线程锁竞争是瓶颈。所有 Linux 程序默认使用。

jemalloc (2005→)

FreeBSD · Jason Evans · Meta

强调并发性能与碎片控制。Per-CPU arena + 线程缓存 + 两阶段 decay 内存回收。Redis、Rust、FreeBSD 默认。独有 profiling/heap dump 内置支持。

mimalloc (2019→)

Microsoft · Daan Leijen

核心创新：Free List Sharding — 每个 page 维护独立空闲链表，将竞争分散到数千个独立链表。极致低延迟 (~7 指令 fast path)，v3.0 重构中。

2. ptmalloc2 — glibc 默认分配器

arena · bin · chunk · tcache — 最广泛使用的分配器，兼容性之王

核心架构

Arena → Bins → Chunks

Arena: 每个线程竞争 arena 锁。main_arena 使用 brk 扩展，thread_arena 通过 mmap 创建。arena 数 ≤ 8×CPU，超出后复用已有 arena。
Bins (5 类 128 链表): fastbins (≤128B, LIFO, 不合并) → smallbins (32B~1008B, FIFO) → largebins (≥1KB, best-fit+跳表) → unsorted bin (缓存池)。glibc 2.26 新增 tcache (每线程 76 链, 无锁)。
Top Chunk: 堆顶万能备胎，所有 bin 都空时从 top chunk 分割。

关键问题与瓶颈

多线程锁竞争 · 碎片 · 回收难

Arena 锁竞争: 多个线程映射到同一 arena → 互斥锁成为瓶颈。高并发下吞吐量急剧下降。
内存碎片: 无内存迁移/压缩机制。brk 只能从顶部释放 → 底部有存活对象时整段内存无法归还 OS。
回收被动: 依赖 malloc_trim (手动触发) 或 top chunk 自然收缩。无后台回收线程。
tcache 安全: 2.26 引入时无 double-free 检查 → 成为安全利用热点。2.32+ 增加 safe-linking XOR 保护。

3. jemalloc — Facebook 的并发内存引擎

arena · extent · slab · bin · 两阶段 decay — Redis / Rust / FreeBSD 的选择

分层架构

arena → extent → slab → region

arena: 独立内存管理域，默认每 CPU 4 个。arena 之间完全无锁竞争。
extent (5.0+): 取代旧 chunk，大小为 N×page 的虚拟内存块。通过 extent_hooks (9 个函数指针) 自定义生命周期。
slab: 用于 small class 的 extent，内部通过位图 追踪每个 region 的占用状态。
bin: 管理同一 size class 的所有 slab，有 slabcur (活跃) 和 slabs_nonfull 两个队列。

tcache — 高速分配路径

per-thread · 批量缓存 · 无锁

每个线程有独立 tcache (thread cache)。每个 bin 对应一个 cache_bin，预缓存多个 region。
分配: tcache→bin→slab，绝大部分请求在 tcache 命中 (无锁)。
tcache GC: 每 228 次分配/释放触发一次 GC。low_water>0 → 回收 ~3/4 缓存；low_water<0 → 增大缓存上限。
批量填充: tcache miss 时从 slab 批量拿一批 region (lg_fill_div 控制数量)。

两阶段 Decay 回收

dirty → muzzy → retained

Lazy Purge (dirty→muzzy): MADV_FREE — 内核在内存压力时才回收，此前内容仍有效。
Forced Purge (muzzy→retained): MADV_DONTNEED — 立即丢弃物理页，下次访问为全零。使用 smoothstep 平滑算法将 purge 分散到 200 个时间片中，避免 IO 尖峰。
dirty_decay_ms=10000 · muzzy_decay_ms=10000 (可调)
后台线程: opt-in 异步 purging，从用户关键路径剥离回收工作。

/* ═══ jemalloc 小对象分配路径 ═══ */ malloc(small_size) ├── tsd (Thread Specific Data) → tcache ├── tcache → bins_small[binind] — 缓存命中? → 弹出返回 (无锁!) ├── bin → slabcur — 当前 slabs 有空闲? ├── bin → slabs_nonfull — 扫描非满 slab 链表 ├── 从 extent heap 分配新 slab → 分配首个 region └── extent heap 也空 → 逐级查找: extents_dirty → muzzy → retained → OS

4. mimalloc — Free List Sharding 的极致设计

heap · segment · page · block — 微软研究院出品，~7 指令 fast path，代码量仅 8-10K

核心创新: Free List Sharding

每 page 独立空闲链表 → 数千个独立竞争域

传统分配器每个 size class 维护一个全局空闲链表 → 所有线程竞争同一把锁。mimalloc 每个 page 维护自己的空闲链表 → 竞争自然分散到成千上万个链表上。
三重空闲链表: 每个 page 维护三个链表 — free (主分配, 仅拥有线程访问)、local_free (本地释放)、xthread_free (跨线程释放, CAS 原子)。
局部性保证: "同时分配的靠近存储" — 同一 page 内的分配在内存上连续。

三层内存体系

Heap → Segment → Page → Block

mi_heap_t: 每线程一个 TLS 堆。包含 pages_direct[] 快速查找数组 (按 size/8 索引)。
mi_segment_t: 4MB 连续虚拟内存 (mmap)。内部划分为 slices → pages。通过 ptr & ~(4MB) 单指令定位。
mi_page_t: 固定大小 blocks 的集合。三种: small (64KB page, <8KB blocks)、large (4MB, <512KB)、huge (≥512KB, 动态)。
mi_block_t: 用户可见分配块，侵入式链表 (空闲 block 的前字节存 next 指针)。

/* ═══ mimalloc 小对象分配 — Fast Path (~7 条 x86 指令) ═══ */ malloc_small(n) ├── heap = TLS slot // 线程本地堆 ├── page = heap→pages_direct[(n+7)>>3] // 直接索引查 page ├── block = page→free // 弹出可用 block ├── if block == NULL → malloc_generic() // slow path — 回收/扩展 ├── page→free = block→next // 更新 free 头 ├── page→used++ // 计数 └── return block /* ═══ Slow Path (malloc_generic) — 也是"垃圾回收"时机 ═══ */ malloc_generic(heap, n) ├── 延迟释放 — 批量处理延迟释放的 blocks ├── Page 收集 — 遍历 page queue, 收集 local_free + xthread_free → free ├── 释放空 page — used - thread_freed == 0 → 归还 ├── Page 旋转 — 找到空闲 page 后移到队首 └── 分配新 page — 从 segment 获取或 mmap 新 segment

Full Page 优化

避免线性扫描满 page

满 page (free==NULL) 被移入独立的满页列表 (MI_BIN_FULL)，不在常规扫描路径中。当 page 中的 block 被释放时，通过延迟释放 + 指针低位编码 实现无锁通知 heap 该 page 不再满。同一 page 的多次释放合并为一次检查 (批量化)。

安全模式

~3-10% 性能损失

Guard Pages: 在 segment 末尾放置不可访问的保护页，检测缓冲区溢出。
Free List 随机化 & 加密: 空闲链表指针 XOR 混淆，防止攻击者预测。
Heap Corruption Detection: 释放时校验 block 合法性。
通过 MI_SECURE=1 编译选项启用。适合安全敏感的生产环境。

v3.0 重大变更 (alpha)
v3.0 正在移除 segment 概念。不再有 4MB 固定段，page 直接通过独立的 OS allocator 分配。这简化了所有权模型，改善了跨线程内存共享和归还 OS 的效率。当前 v3.1.5 beta (2025-06)。

5. 并发模型 — 线程安全设计的三种哲学

三种分配器对多线程并发的处理方式截然不同

ptmalloc2 — 粗粒度 Arena 锁

arena mutex — 互斥锁全局保护

多个线程共享少数 arena，每个 arena 一把互斥锁。锁竞争随线程数增长而急剧恶化 — 典型瓶颈。tcache (2.26+) 缓解了单页分配，但 tcache miss 仍需获取 arena 锁。
无锁退化: tcache → 加锁 arena → 扫描 bins。
适合: 低并发、兼容性优先场景。

jemalloc — Per-CPU Arena

arena × CPU — 细粒度隔离

每个 CPU 独占一组 arena，线程绑定到 CPU 的 arena。tcache 完全无锁。arena 内部使用多个细粒度 mutex (bin/extent 分别加锁)。
后台线程异步 purging，不受用户分配路径影响。
无锁退化: tcache → bin lock (细粒度) → extent lock。
适合: 高并发服务、NUMA 环境。

mimalloc — Free List Sharding

数千个独立竞争域 · 原子操作

每个 page 一个独立 free list → 竞争天然分散。全部使用原子 CAS 操作，无互斥锁。分配全在 TLS 中完成。跨线程 free 通过单次 CAS 原子推入 xthread_free → 拥有线程在 slow path 时批量合并。
无锁退化: fast path (TLS) → slow path (bulk collect)。
适合: 极致低延迟、无锁优先场景。

6. 内存回收 — 如何归还空闲内存给 OS

三种分配器对"何时及如何归还物理内存"有不同的策略 — 直接影响系统整体内存使用

ptmalloc2 — 被动回收

brk 收缩 + malloc_trim

brk 限制: 只能从堆顶收缩。底部有存活对象 → 整个堆段无法归还。
mmap 大块 (≥128KB): free 时立即 munmap，归还及时。
malloc_trim: 手动触发 top chunk 收缩，非自动。
问题: 长时间运行的服务 → 堆碎片化 → 内存越用越多 → 需要周期性重启。

jemalloc — 两阶段 Smooth Decay

dirty→muzzy→retained · 后台线程

Dirty Decay (默认 10s): MADV_FREE 标记 — 内核按需回收。
Muzzy Decay (默认 10s): MADV_DONTNEED 强制回收物理页。
Smoothstep 平滑: 200 个时间片逐步 purge，无 IO 尖峰。
后台线程: 异步完成 purging，零分配路径开销。
Retained: 未归还 OS 但可复用的虚存池。

mimalloc — Eager Page Reset

积极归还 + retire 防抖

Page 退休: page 变空后不立即释放 → 保留 retire_expire 时间 (防抖动)。超时后释放。
Eager Reset: 定期 MADV_DONTNEED 重置空闲 page。
Segment 释放: 整个 segment (4MB) 所有 page 都空闲 → munmap 归还 OS。
v3.0 改进: 移除 segment 后归还粒度更细。目前 v2.1.x 中 segment 级别的归还较 jemalloc 的 retained 池化稍弱。

8. 选型指南 & 实战切换

根据场景选择最合适的分配器 · 一行命令切换 · 典型配置示例

选择 ptmalloc2 当…

兼容性 > 性能

① 无法替换 libc 的嵌入式环境
② 低并发 (< 4 线程) 的应用
③ 需要 glibc 官方支持的生产环境
④ 调试工具链依赖 glibc malloc
⑤ 不愿引入额外依赖

选择 jemalloc 当…

并发 + 稳定 + 可观测

① 高并发网络服务 (Redis、Nginx)
② 需要 heap profiling / dump
③ 长时间运行的服务 (避免碎片)
④ NUMA 环境多核服务器
⑤ 内存敏感 + 需要细粒度控制

选择 mimalloc 当…

延迟 + 简洁 + 现代

① 延迟敏感 (游戏、HFT、GUI)
② 新项目、Rust 生态
③ 重视代码简洁性 (8K LOC)
④ 模块化部署 (单头文件可选)
⑤ 安全敏感 (secure mode)

/* ===== 切换分配器的三种方式 ===== */ # 方法 1: LD_PRELOAD — 运行时替换 (最常用) # jemalloc LD_PRELOAD=/usr/lib/libjemalloc.so ./myapp # mimalloc LD_PRELOAD=/usr/lib/libmimalloc.so ./myapp # 方法 2: 链接时替换 — 编译期静态选择 gcc myapp.c -o myapp -ljemalloc // 链接 jemalloc gcc myapp.c -o myapp -lmimalloc // 链接 mimalloc # 方法 3: Rust — Cargo.toml 配置 # jemalloc (Rust 默认 — 可通过 feature 切换) # mimalloc [dependencies] mimalloc = { version = "0.1", features = ["local_dynamic_tls"] } // 在 main.rs 顶部: use mimalloc::MiMalloc; #[global_allocator] static GLOBAL: MiMalloc = MiMalloc; /* ===== jemalloc 调优 — /etc/malloc.conf 或环境变量 ===== */ export MALLOC_CONF="background_thread:true,dirty_decay_ms:5000,muzzy_decay_ms:5000" /* ===== 确认当前使用的分配器 ===== */ ldd ./myapp | grep -E 'jemalloc|mimalloc|tcmalloc' pmap -x $$ | head -5 // 查看进程内存布局
⚠️ LD_PRELOAD 的注意事项
① LD_PRELOAD 只能替换动态链接的 malloc/free 调用 — 静态链接 glibc 或内联的 malloc 不受影响。
② 某些程序通过 dlsym(RTLD_NEXT, "malloc") 调用 — 可能与 preload 的分配器冲突。
③ 替换分配器后内存行为可能完全不同 — 必须重新做内存压力测试。
④ glibc 的 mallopt()、malloc_info()、malloc_trim() 接口在替换后不可用。
