1. Input 子系统架构总览

Linux Input 子系统是内核中最成熟、最统一的设备驱动框架之一——键盘、鼠标、触摸屏、游戏手柄、电源键、GPIO 按键，全都通过同一套接口向用户态报告事件

三层架构

Driver → Core → Handler

Driver 层：硬件驱动，分配 input_dev，设置支持的事件类型/编码，在中断/轮询中调用 input_report_*() 上报事件。
Core 层（drivers/input/input.c）：事件路由中枢——将驱动上报的事件分发给所有注册的 Handler。
Handler 层：将内核事件转换为用户态接口——evdev（通用，/dev/input/eventX）、joydev（游戏杆）、mousedev（鼠标）、leds（键盘 LED）。

核心数据结构

input_dev · input_handler · input_handle

input_dev：描述一个输入设备——支持的 event type、key/abs/rel bitmaps、设备名称/ID/phys 路径。
input_handler：事件消费者——evdev_handler、joydev_handler 等。
input_handle：连接 dev 和 handler 的桥梁——当 driver 注册 input_dev 时，Core 遍历所有 handler，为匹配的 handler 创建 handle 建立连接。

源码路径

2. 事件类型与编码体系

所有输入事件通过 type + code + value 三元组描述——这是 Input 子系统的通用语言

EV_KEY — 按键事件

value: 0=抬起 1=按下 2=长按(重复)

最常用的事件类型。涵盖 键盘所有按键（KEY_A~KEY_Z、KEY_0~KEY_9）、鼠标按钮（BTN_LEFT/MIDDLE/RIGHT）、功能键（KEY_POWER/KEY_VOLUMEUP/KEY_MUTE）、自定义按键（BTN_0~BTN_9, BTN_TRIGGER）。
代码定义在 input-event-codes.h，共 700+ 个 KEY_ 编码。
驱动中：input_report_key(dev, KEY_POWER, 1); input_sync(dev);

EV_REL — 相对位移

value: 相对偏移量 (正/负)

鼠标/轨迹球/滚轮的增量移动。常用轴：REL_X（水平）、REL_Y（垂直）、REL_WHEEL（滚轮）、REL_HWHEEL（水平滚轮）。值是相对偏移而非绝对坐标——每次上报的是"移动了多少"。

EV_ABS — 绝对坐标

value: 绝对位置 (需先设置 min/max/fuzz/flat)

触摸屏/触摸板/摇杆的绝对位置。常用轴：ABS_X/Y（主坐标）、ABS_MT_POSITION_X/Y（多点触控）、ABS_PRESSURE（压力）、ABS_DISTANCE（悬浮距离）。
必须先设置：input_set_abs_params(dev, ABS_X, 0, 4095, 0, 0);

其他事件类型速查

EV_SW · EV_LED · EV_SND · EV_REP · EV_MSC

EV_SW（Switch）：物理开关——SW_LID（笔记本盖子）、SW_HEADPHONE_INSERT（耳机插入检测）、SW_RFKILL_ALL（飞行模式）。
EV_LED：键盘 LED——LED_CAPSL/NUML/SCROLLL。
EV_SND（Sound）：蜂鸣器——SND_BELL/SND_TONE。
EV_REP：自动重复参数——REP_DELAY（首次延迟）/REP_PERIOD（重复间隔）。
EV_MSC（Misc）：杂项——MSC_SCAN（原始扫描码）、MSC_TIMESTAMP。

struct input_event

用户态读取的通用格式 — 24 字节

所有 Handler 最终都以 struct input_event 格式向用户态传递事件：
timeval time — 事件时间戳（秒+微秒）
__u16 type — 事件类型（EV_KEY/EV_REL/EV_ABS...）
__u16 code — 事件编码（具体按键/轴）
__s32 value — 事件值（按下/抬起/坐标值）
sizeof(struct input_event) == 24 字节。用户态 read(fd, &ev, sizeof(ev)) 循环读取。

EV_SYN — 事件分隔符（最容易忽视的重要事件）
SYN_REPORT（type=EV_SYN, code=SYN_REPORT, value=0）：标记一组事件的结束——相当于"帧分隔符"。一次完整的硬件事件（如触摸一下屏幕）可能产生多个 input_event（ABS_X + ABS_Y + ABS_PRESSURE + BTN_TOUCH），最后以 SYN_REPORT 收尾。用户态应以 SYN_REPORT 为边界处理事件组，而非收到一个事件就立刻响应。还有一个 SYN_MT_REPORT 用于多点触控的每个触点分隔。调用 input_sync(dev) 自动发送 SYN_REPORT。

3. 驱动开发：input_dev 的注册与事件上报

编写 Input 驱动是 Linux 驱动开发中最简单、最规范化的流程——核心只需四步

1   input_allocate_device()
分配 input_dev
2   设置 evbit/keybit/absbit
声明设备支持的事件
3   input_register_device()
注册到 Input 子系统
4   input_report_*()
在中断中上报事件
5   input_sync()
发送 SYN_REPORT 帧边界
6   input_unregister_device()
模块卸载时注销

GPIO 按键驱动 — 最简示例

不需要手写——内核已提供 gpio_keys.c 通用驱动

Linux 内核内置 drivers/input/keyboard/gpio_keys.c——只需在设备树中描述按键即可，无需写一行 C 代码。每个按键可指定：GPIO 编号、KEY 编码（如 KEY_POWER）、触发边沿、去抖时间、标签。Device Tree 示例：
gpio-keys { compatible = "gpio-keys"; power { label = "Power"; gpios = <&gpio0 12 GPIO_ACTIVE_LOW>; linux,code = ; debounce-interval = <30>; } };

自定义驱动模板 — 核心步骤

allocate → set capability → register → report

① input_dev *idev = devm_input_allocate_device(&pdev->dev);
② __set_bit(EV_KEY, idev->evbit); __set_bit(KEY_POWER, idev->keybit);
③ idev->name = "My Button"; idev->phys = "gpio-keys/input0"; input_register_device(idev);
④ 在中断处理函数中：input_report_key(idev, KEY_POWER, gpiod_get_value(desc)); input_sync(idev);
用 devm_* 系列函数可自动管理资源释放（无需手动 free/unregister）。

完整 GPIO 按键驱动骨架代码
#include <linux/input.h> #include <linux/gpio/consumer.h> #include <linux/interrupt.h> struct my_button { struct input_dev *input; struct gpio_desc *gpio; int irq; }; static irqreturn_t button_isr(int irq, void *dev_id) { struct my_button *btn = dev_id; int state = gpiod_get_value(btn->gpio); // 上报按键事件 (1=按下, 0=抬起) input_report_key(btn->input, KEY_POWER, state); input_sync(btn->input); // 发送 SYN_REPORT 帧边界 return IRQ_HANDLED; } static int my_button_probe(struct platform_device *pdev) { struct my_button *btn; int error; btn = devm_kzalloc(&pdev->dev, sizeof(*btn), GFP_KERNEL); // 1. 分配 input_dev btn->input = devm_input_allocate_device(&pdev->dev); // 2. 设置设备属性和支持的事件 btn->input->name = "IPC Reset Button"; btn->input->phys = "gpio-keys/input0"; __set_bit(EV_KEY, btn->input->evbit); __set_bit(KEY_POWER, btn->input->keybit); // 3. 获取 GPIO 和 IRQ btn->gpio = devm_gpiod_get(&pdev->dev, "reset", GPIOD_IN); btn->irq = gpiod_to_irq(btn->gpio); // 4. 注册中断（双边沿触发：按下+抬起都响应） devm_request_irq(&pdev->dev, btn->irq, button_isr, IRQF_TRIGGER_RISING | IRQF_TRIGGER_FALLING, "ipc-reset-btn", btn); // 5. 注册 input 设备 error = input_register_device(btn->input); if (error) return error; return 0; }

4. evdev — 用户态通用事件接口

evdev 是 Input 子系统最常用的 Handler——将内核事件以 /dev/input/eventX 字符设备暴露给用户态，支持阻塞 read + poll/select + ioctl

读取事件的三种方式

阻塞 read · poll/epoll · libinput

阻塞 read：最简单——read(fd, &ev, sizeof(ev))，无事件时阻塞。
poll/epoll：将 evdev fd 加入 event loop，与 socket/timer 统一调度——嵌入式主循环的标准做法。
libinput：Weston/Wayland 使用的用户态库，封装了 evdev 的细节（去抖、触摸屏校准、手势识别）。Weston 独占 evdev fd（设置 EVIOCGRAB），应用不应直接读。
evtest：命令行调试工具——evtest /dev/input/event0 实时打印所有事件。

关键 ioctl 命令

查询设备能力 · 独占抓取 · 获取信息

EVIOCGVERSION：获取 evdev 协议版本号。
EVIOCGID：获取 struct input_id（bustype/vendor/product/version）。
EVIOCGBIT(ev, len)：查询设备支持的 evbit/keybit/absbit 等——判断设备类型的标准方法。
EVIOCGABS(abs)：获取 ABS_* 轴的参数（min/max/fuzz/flat/resolution）。
EVIOCGRAB：独占抓取——阻止其他进程接收该设备的事件（Weston 使用）。
EVIOCSREP：设置自动重复参数（delay/period）。

识别设备类型的标准流程

EVIOCGBIT → 读取 capability bits → 判定角色

① ioctl(fd, EVIOCGBIT(0, EV_MAX), evbit) 获取支持的事件类型
② 如果 evbit 包含 EV_REL 且 包含 REL_X/REL_Y → 鼠标类设备
③ 如果 evbit 包含 EV_ABS 且 包含 ABS_X/ABS_Y 且 包含 BTN_TOUCH → 触摸屏
④ 如果 evbit 仅包含 EV_KEY → 键盘/按键类设备
⑤ 可以用 libevdev 库封装上述逻辑，避免手动解析 bitmask。

用户态读取示例

最小的 evdev 事件循环
struct input_event ev; int fd = open("/dev/input/event0", O_RDONLY); while (read(fd, &ev, sizeof(ev)) == sizeof(ev)) { if (ev.type == EV_KEY && ev.code == KEY_POWER) { if (ev.value == 1) printf("Power key pressed\n"); if (ev.value == 0) printf("Power key released\n"); if (ev.value == 2) printf("Power key auto-repeat\n"); } }

5. 多点触控协议 (Multi-Touch Protocol)

Linux 定义了两种多点触控协议——Type A（无状态，简单）和 Type B（有状态，追踪触点 ID）

Type A (无状态)

每次上报全部触点 — 驱动不需要跟踪触点 ID

每次上报所有当前活动的触点的完整坐标。用户态无法直接区分"哪个触点对应哪个手指"——需要上层根据上一帧的位置做最近邻匹配。
事件序列：ABS_MT_POSITION_X/Y[0] → SYN_MT_REPORT → ABS_MT_POSITION_X/Y[1] → SYN_MT_REPORT → SYN_REPORT。
优点：驱动极简单。缺点：上层需要复杂匹配算法，触点快速移动/交叉时易混淆。

Type B (有状态)

每个触点有唯一 Tracking ID — 驱动负责追踪

每个触点分配一个唯一的 Tracking ID（0~最大值）。通过 ABS_MT_TRACKING_ID 上报触点的生命周期：新触点出现 → 上报 ID+坐标 → SYN_REPORT → 每次移动更新坐标 → 触点抬起时上报 ABS_MT_TRACKING_ID=-1 表示触点消失。用户态可以精确追踪每个触点从按下到抬起的完整轨迹。
优点：上层处理简单精确。缺点：驱动需要维护触点状态、分配/回收 ID。

Type B 事件序列示例 — 两指触控
// 触点 0 按下 ABS_MT_TRACKING_ID 0 ABS_MT_POSITION_X 1200 ABS_MT_POSITION_Y 800 ABS_MT_PRESSURE 80 SYN_MT_REPORT // 触点 1 按下 ABS_MT_TRACKING_ID 1 ABS_MT_POSITION_X 2400 ABS_MT_POSITION_Y 600 SYN_MT_REPORT SYN_REPORT ← 这一帧有两个触点 // 下一帧：触点 0 移动 ABS_MT_TRACKING_ID 0 ABS_MT_POSITION_X 1250 向右滑了 50 ABS_MT_POSITION_Y 790 SYN_MT_REPORT // 触点 1 抬起 ABS_MT_TRACKING_ID 1 ABS_MT_TRACKING_ID -1 ← -1 = 触点消失 SYN_MT_REPORT SYN_REPORT ← 这一帧只有触点 0 活动

6. uinput — 用户态输入设备

uinput 允许用户态程序创建虚拟输入设备并向内核注入事件——自动化测试、虚拟键盘/鼠标、IR 遥控转发的核心技术

uinput 的工作原理

/dev/uinput → write input_event → 内核 → evdev → 用户态

用户态程序通过 /dev/uinput 字符设备创建虚拟 input_dev。设置支持的事件类型（与内核驱动一样设置 evbit/keybit），内核为其分配 eventX 节点。之后程序通过 write() 发送 struct input_event，内核 Core 层将其视为普通硬件事件分发给所有 Handler。GUI 程序无法区分事件来自真实硬件还是 uinput——这正是它的价值。

典型应用场景

IR 遥控 · 触摸屏模拟 · 自动化测试

IR 遥控转发：IR 接收驱动(LIRC)读红外码 → 查表转 KEY 事件 → uinput 注入 → Android/Linux GUI 响应。
自动化测试：注入触摸/点击事件模拟用户操作——不需要机械手/机械臂。
虚拟键盘：屏幕键盘（on-screen keyboard）通过 uinput 注入按键。
GPIO 扩展：用户态 GPIO 轮询（不适合内核中断的场景）→ uinput 上报按键。

uinput 创建虚拟按键设备的完整代码
#include <linux/uinput.h> int create_virtual_button() { int fd = open("/dev/uinput", O_WRONLY | O_NONBLOCK); // 1. 声明设备能力 ioctl(fd, UI_SET_EVBIT, EV_KEY); ioctl(fd, UI_SET_KEYBIT, KEY_POWER); ioctl(fd, UI_SET_KEYBIT, KEY_VOLUMEUP); // 2. 配置设备信息 struct uinput_setup usetup = { .name = "IPC Virtual Buttons", .id = { .bustype = BUS_VIRTUAL } }; ioctl(fd, UI_DEV_SETUP, &usetup); // 3. 创建虚拟设备 ioctl(fd, UI_DEV_CREATE); return fd; // 此后 write(fd, &ev, sizeof(ev)) 注入事件 } void inject_key(int uinput_fd, int key_code) { struct input_event ev; // 按下 memset(&ev, 0, sizeof(ev)); ev.type = EV_KEY; ev.code = key_code; ev.value = 1; write(uinput_fd, &ev, sizeof(ev)); // 同步 ev.type = EV_SYN; ev.code = SYN_REPORT; ev.value = 0; write(uinput_fd, &ev, sizeof(ev)); // 抬起 ev.type = EV_KEY; ev.code = key_code; ev.value = 0; write(uinput_fd, &ev, sizeof(ev)); // 同步 ev.type = EV_SYN; ev.code = SYN_REPORT; ev.value = 0; write(uinput_fd, &ev, sizeof(ev)); }

7. 设备树配置 — 声明即驱动的 Input 设备

大量 Input 设备无需编写驱动——内核已内置通用驱动（gpio_keys, input_leds, pwm-vibrator），只需在设备树中声明

gpio-keys — GPIO 按键

drivers/input/keyboard/gpio_keys.c
// 设备树：定义两个 GPIO 按键 gpio-keys { compatible = "gpio-keys"; autorepeat; // 支持长按自动重复 reset { label = "Reset Button"; gpios = <&pio 12 GPIO_ACTIVE_LOW>; linux,code = <KEY_RESTART>; debounce-interval = <30>; // 30ms 去抖 }; wps { label = "WPS Button"; gpios = <&pio 14 GPIO_ACTIVE_LOW>; linux,code = <KEY_WPS_BUTTON>; }; };

gpio-keys-polled — 轮询模式 GPIO 按键

用于不支持中断的 GPIO 或消抖需求复杂时

有些 GPIO 控制器不支持中断（或 GPIO 用于矩阵键盘、ADC 按键等无法直接用中断的场景）。gpio_keys_polled.c 用内核定时器周期轮询 GPIO 状态。每轮检测到变化→上报事件。设备树类似，但 compatible 为 "gpio-keys-polled"，额外的 poll-interval 指定轮询间隔（通常 20-50ms）。注意：轮询消耗 CPU，仅在中断不可用时作为 fallback。

8. 调试与诊断工具

Input 子系统调试非常方便——从确认设备被识别到追踪每一个输入事件

evtest — 事件实时监控

apt install evtest / 嵌入式: 交叉编译

evtest：列出所有 /dev/input/eventX 设备 → 选择 → 实时打印每个 input_event（type/code/value 全部解析为可读名称）。
evtest --grab /dev/input/event0：独占抓取模式（EVIOCGRAB），其他进程收不到事件——调试时避免事件被 GUI 消费。
最常用的调试命令：evtest /dev/input/event0 2>&1 | grep -v "SYN_REPORT"

getevent — Android 通用

Android 内置，无需安装

getevent -l：列出所有输入设备及详细能力（key/abs/rel bitmaps）。
getevent -lt /dev/input/event0：实时监控，-l 是文字模式（否则是十六进制原始数据）。
getevent -i：显示设备信息（vendor/product/version/bustype）。
输出格式：[time] /dev/input/event0: EV_KEY KEY_POWER DOWN

sysfs · proc · 其他

/sys/class/input/ · /proc/bus/input/

/proc/bus/input/devices：列出系统所有注册的 input 设备及属性（name/phys/sysfs/handlers/bits）。比 sysfs 更易读。
/sys/class/input/eventX/device/：设备属性（name、phys、id/vendor、capabilities/）。
input-utils 包：input-kbd（键盘映射）、lsinput（简洁设备列表）。
dmesg | grep input：查看设备注册/注销日志，每个 input device 注册/注销都会打印一条 kernel log。

9. IPC 实战：嵌入式设备的 Input 整合

IPC 相机中 Input 子系统的典型应用——复位按键、IR 遥控、GPIO 报警输入、触摸屏调试口

硬件复位按键

GPIO 按键 → KEY_RESTART → 用户态监听

IPC 外壳上的物理复位按钮：GPIO 连接到 SoC → gpio-keys 驱动 → /dev/input/eventX → 用户态守护进程 poll 监听 → 检测到长按 5 秒 → 触发恢复出厂设置或重启。比直接硬件复位（拉 RST 引脚）更好——可以先安全关闭录像文件、清理 SD 卡写入缓存、通知云端下线。
内核配置：CONFIG_KEYBOARD_GPIO=y + 设备树配置 gpio-keys 节点。

IR 遥控接收

IR Receiver → LIRC / gpio-ir-recv → uinput → KEY 事件

IPC 配套的 IR 遥控器：红外接收头（HS0038 等）输出解调后的方波 → GPIO 捕获 → Linux 内核有内置 IR 解码驱动（drivers/media/rc/gpio-ir-recv.c，设备树声明即可）→ 解码出 NEC/RC5/RC6 等协议 → 查表转换为 KEY_* 事件 → 通过 rc-core 子系统或 uinput 注入 evdev → 用户态 GUI 或守护进程响应。无需手写解码算法。

报警输入 (Alarm IN)

外部传感器 → GPIO 中断 → 事件上报

IPC 的报警输入端子（Alarm IN）：外部 PIR 传感器/门磁/烟感 → 干接点闭合 → GPIO 上拉/下拉变化 → 中断触发 → 守护进程检测到事件 → 触发录像/抓图/推送报警。实现方式：① gpio-keys 将报警 GPIO 映射为 KEY 事件 ② 用户态 epoll 等待事件 ③ 或直接使用 gpio 子系统 + libgpiod 进行边沿检测（更灵活但需要更多代码）。推荐 gpio-keys——零代码开销，Systemd 可以直接配置为触发条件。

IPC 相机 Input 设备整合 CheckList
□ GPIO 复位按键 → gpio-keys → KEY_RESTART → 守护进程监听长按 5s → 安全重启
□ IR 遥控 → gpio-ir-recv → rc-core/mapping → uinput → Android/Linux GUI 响应
□ Alarm IN → gpio-keys 或 libgpiod → 边沿触发 → 录像/推送通知
□ 调试串口 → 可选注册为 tty + input 复合设备（console + 按键输入）
□ 确保所有 gpio-keys 设置了 debounce-interval（硬件抖动 5-30ms）
□ 使用 devm_* 系列函数管理 input_dev 生命周期（驱动模块卸载时自动清理）
