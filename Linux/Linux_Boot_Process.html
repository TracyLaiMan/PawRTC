硬件启动阶段

Boot ROM → TPL/SPL

SoC 上电后 Boot ROM 从指定介质加载 SPL 到内部 SRAM。SPL 初始化 DDR、时钟树、引脚复用，加载 ATF 和 U-Boot 到 DRAM。

引导加载阶段

ATF → U-Boot → Kernel

ATF 设置 EL3 安全世界后，U-Boot 在 DRAM 中完整初始化设备驱动，从存储介质加载 内核 Image + DTB，设置启动参数后跳转到内核入口。

内核启动阶段

head.S → start_kernel → init

内核解压 → 架构初始化 → 挂载 rootfs → do_initcalls() 驱动初始化 → 解包 initramfs → 挂载真实根文件系统 → run_init_process("/sbin/init")。

2. Stage 1 — Boot ROM (BL1)

片内掩膜 ROM 中的第一行代码 · 不可修改 · 确定启动介质 · 加载 SPL 到内部 SRAM

硬件特性

SoC 厂商硬编码 — 用户不可修改

Boot ROM 是芯片制造时烧死在掩膜 ROM 中的代码。容量 32KB~128KB，上电复位向量直接指向 ROM 起始地址。因为不可修改，它是安全启动信任根 (Root of Trust) 的天然锚点。支持镜像签名验证的 SoC 在此阶段完成首次校验。

核心执行流程 — 5 步

从复位向量到加载 SPL

① 最小 CPU 初始化 — 设置栈指针、关闭看门狗
② 读取启动模式引脚 — SYSBOOT / BOOT_MODE 确定设备顺序
③ 遍历启动设备 — 按顺序查找有效镜像 (MMC→SPI→UART→USB)
④ 验证镜像头 — 魔数 + 校验和 + 镜像大小
⑤ 拷贝到 SRAM — DMA 加载 → 跳转到 SPL 入口地址

SoC 系列SRAM启动设备顺序特殊机制
TI AM335x (BeagleBone)128KBMMC0 → SPI0 → UART0 → USB0SPL 称为 MLO，需放在 FAT 分区首文件

3. Stage 2* — TPL (Tertiary Program Loader)

当 SRAM 太小时引入的极简阶段 · 通常只做一件事：初始化 DRAM · TPL = 可选 (*)

为什么需要 TPL？

SRAM 不足 — 必须拆分职责

SPL 体积 (驱动 + FAT/ext4 解析 + FIT 解析...) 超过 SRAM 容量时，分离出独立的 DDR 初始化阶段。TPL 仅做：配置 PLL → 校准 DDR PHY → 设置 DDR 控制器时序 → return 回 Boot ROM。ROM 随后从存储介质重新加载完整 SPL 到已就绪的 DRAM 中。

极限约束

无 global_data · 无 BSS · 无栈

TPL 运行在极简环境中：所有变量必须是局部或寄存器变量、不能使用 printf（依赖 GPIO+示波器调试）、典型体积7KB~30KB。完成后必须 return 回 Boot ROM，不能直接跳 SPL。需要 TPL 的平台: Rockchip RK3399、部分 NXP i.MX8 变体。不需要的: TI AM335x、i.MX6、Zynq。

4. Stage 3 — SPL (Secondary Program Loader)

最关键的中间引导 · 初始化时钟/引脚/存储 · 加载 FIT 镜像 (ATF+U-Boot+DTB) 到 DRAM

初始化序列

board_init_f → 条件重定位 → board_init_r

board_init_f(): 时钟树 → 引脚复用 (pinmux) → 串口 → DDR 初始化 (若 TPL 未做)。
board_init_r(): 存储设备 init (MMC/SPI/NAND) → 解析 FIT → 分别放置 ATF / U-Boot / DTB 到指定 DRAM 地址 → 跳转。

FIT Image 加载

Flattened Image Tree 统一打包

FIT 将多组件打包为一个文件: ATF (firmware) + OP-TEE (loadables) + U-Boot (standalone) + DTB (fdt)。每个组件有独立的 load 和 entry 地址。SPL 解析 FIT 结构后按地址分别放置到 DRAM。

SPL 大小约束

由 SRAM 容量决定上限

Allwinner H3: <32KB (仅 32KB SRAM)
TI AM335x: <128KB (MLO)
Rockchip: 可达 ~256KB (DRAM 已就绪)
减小体积: tiny-printf + 禁用非必要驱动 + -Os + LTO。

void board_init_r(struct spl_image_info *img, ulong dummy) { /* 1. 确定启动设备顺序 */ struct spl_boot_device bootdev = board_boot_order(); /* 2. 遍历启动设备加载 FIT */ for (int i = 0; bootdev[i] != BOOT_DEVICE_NONE; i++) spl_mmc_load_image(bootdev[i]); /* 3. 解析 FIT → 放置各组件到 DRAM */ spl_load_simple_fit(&img, &load_info, 0, header); // FIT 内部: firmware=ATF loadables=U-Boot fdt=DTB /* 4. 跳转到 ATF (BL31) — ATF 后将回跳到 U-Boot */ jump_to_image_no_args(&img); }
SPL 重定位
board_init_f() 在 SRAM 中运行。DDR 初始化后，SPL 可选择将自己重定位到 DRAM 高地址 (CONFIG_SPL_STACK_R) 获得更大的栈和堆。但多数平台 SPL 不重定位——SRAM 中直接执行完 board_init_r() 然后跳走。

5. Stage 4* — ARM Trusted Firmware (ATF / BL31)

ARMv8-A TrustZone 安全世界守门人 · 运行于 EL3 · OP-TEE (BL32) 可选 · ATF = 可选 (*)

五大核心职责

运行在 EL3 — 最高异常级别

① 异常级别配置: 初始化 EL3/EL2/EL1 的 CPU 上下文
② SMC 处理: 处理来自 Normal World 的安全监控调用
③ PSCI 电源管理: CPU hotplug、suspend/resume、系统关机重启
④ 安全世界初始化: 可选加载 BL32 (OP-TEE 可信执行环境)
⑤ 世界切换: ERET 退出到 Normal World U-Boot (EL2)

ATF 多阶段模型

与 U-Boot SPL 的集成映射

BL1 = Boot ROM (SoC 厂商)
BL2 = SPL (负责加载 BL31/32/33)
BL31 = ATF 运行时固件 (Secure Monitor)
BL32 = OP-TEE (可选 — 可信执行环境 OS)
BL33 = U-Boot / EDK2 (Normal World)
SPL 在 U-Boot 集成模型中充当了 ATF 的 BL2 角色。

/* ATF → U-Boot 交接: 通过 ERET 指令从 EL3 降至 EL2 */ void bl31_main(void) { cm_prepare_el3_exit(NON_SECURE); // 准备退出到 Normal World cm_el2_sysregs_context_save(); // 保存 EL2 系统寄存器上下文 cm_set_next_eret_context(NON_SECURE);// 设置 ELR_EL2=U-Boot入口, SPSR_EL2=AArch64 EL2 return; // 控制流到达异常处理框架 → ERET → 跳转到 U-Boot }
⚠️ 什么平台需要 ATF？
需要: 所有 ARMv8-A 64-bit (RK3399, i.MX8, BCM2711…) — TrustZone 安全启动必需。
不需要: ARMv7-A 32-bit (AM335x, i.MX6, Allwinner H3…) — SPL 直接跳 U-Boot。
OP-TEE: 仅 DRM / 安全支付 / 可信密钥存储等场景需要，多数嵌入式设备不配置。

7. Linux 内核启动 — 从 head.S 到 PID 1

汇编入口 → start_kernel → rest_init 三线程 → kernel_init PID1 → do_initcalls 七级驱动初始化

Phase A: 汇编入口

arch/arm64/kernel/head.S

① 保存 U-Boot 传来的 FDT 指针
② 检查 CPU 是否支持当前内核
③ 创建初始页表 (identity mapping + kernel mapping)
④ 启用 MMU → 切换到虚拟地址
⑤ 清零 BSS → 设栈 → 跳转到 start_kernel()

Phase B: start_kernel

init/main.c — 内核 C 代码入口

① setup_arch() — 解析 FDT + 架构初始化
② sched_init() — 调度器
③ vfs_caches_init() → mnt_init() → init_mount_tree() ★ 挂载初始 rootfs
④ mem_init() — buddy allocator 就绪
⑤ rest_init() — 创建 PID 1 & PID 2

Phase C: kernel_init (PID 1)

内核线程 → 用户进程转换点

① do_basic_setup() → do_initcalls() ★ 所有驱动初始化
② populate_rootfs() — 解包 initramfs
③ prepare_namespace() — 挂载真实根
④ init_post() → run_init_process("/sbin/init") ★ exec 到用户空间

8. 文件系统挂载 — 三次关键挂载

start_kernel 中的初始 rootfs → initramfs CPIO 解包 → prepare_namespace 的 pivot_root

① 初始 rootfs (极早期)

start_kernel → vfs_caches_init

在任何驱动初始化之前，mnt_init() → init_mount_tree() 挂载一个空的 ramfs 作为 "/"。确保 VFS 始终可用 — 所有后续操作都假设根已存在。

② initramfs 解包

kernel_init → populate_rootfs

将编译链接进内核的 CPIO 存档 (usr/initramfs_data.cpio)直接解包到已有的 rootfs。若存在 /init，内核直接执行它作为早期用户空间 init。

③ 真实根文件系统

kernel_init → prepare_namespace

根据 root=/dev/mmcblk0p2 挂载真实根 (ext4/SquashFS/NFS…)。成功后 sys_mount(MS_MOVE) 覆盖旧 rootfs → sys_chroot(".") → run_init_process("/sbin/init")。

/* ===== 挂载流程伪代码 ===== */ // 1. start_kernel() — 最早 rootfs vfs_caches_init() → mnt_init() ├── init_rootfs() // register_filesystem(&rootfs_fs_type) └── init_mount_tree() └── do_new_mount(rootfs, "/", "rootfs", ...) ★ "/" 诞生 // 2. kernel_init() — initramfs 解包 populate_rootfs() ├── if (内置 initramfs) → unpack_to_rootfs(__initramfs_start, size) └── if (initrd) → initrd_load() // 3. kernel_init() — 真实根挂载 if (init_eaccess(ramdisk_execute_command)) // /init 存在? run_init_process(ramdisk_execute_command) // → 早期用户空间 init prepare_namespace() ├── wait_for_device_probe() // 等存储驱动就绪 ├── mount_root() │ └── do_mount_root(root=, ...) // 根据 root= 参数挂载 └── pivot_root → sys_chroot(".") /* ===== 根文件系统相关内核参数 ===== */ root=/dev/mmcblk0p2 // eMMC/SD 分区 root=/dev/nfs // NFS 网络根 root=PARTUUID=xxxx // 按分区 UUID 查找 rootfstype=ext4 // 文件系统类型 (可省略，自动检测) rootwait // ⭐ 等待设备出现 — 最重要! rootdelay=3 // 延迟 N 秒后挂载
⚠️ rootwait — 最常见启动失败的根因
没有 rootwait 时，内核可能在存储驱动还未完成异步探测时尝试挂载根 → "VFS: Unable to mount root fs on unknown-block" panic。MMC 驱动通常是 device_initcall (级别 6)，而根挂载时机早于异步探测完成。rootwait 让内核无限等待直到设备节点出现。

9. 第一个用户空间进程 (PID 1)

kernel_init → run_init_process → systemd / BusyBox init — 完整用户空间系统的诞生

/sbin/init → systemd

① default.target 解析
② sysinit.target — 挂载 /proc /sys /dev
③ local-fs.target — /etc/fstab 文件系统

⑤ multi-user.target — sshd, crond…

BusyBox init (嵌入式)

/init → busybox init
