1. 进程本质：task_struct

Linux 内核不区分"进程"和"线程"——统一用 task_struct 表示所有可调度实体。这是理解 Linux 进程模型的起点。

task_struct 全景

include/linux/sched.h — 进程的"身份证"

task_struct 是内核中最大、最复杂的单体结构之一（~2KB），包含进程的所有元信息。通过 current 宏可零开销获取当前 CPU 上运行进程的指针（利用内核栈底部固定偏移）。每个 task 占据 THREAD_SIZE（通常 8KB/16KB）的内核栈空间 + 独立的 task_struct。

进程 vs 线程：clone() 视角

唯一的区别 = clone() 的 flag 参数

内核看到的所有执行实体都是 task_struct。线程通过 clone(CLONE_VM) 与父 task 共享 mm_struct（地址空间），进程则各自独立。pthread_create() 底层调用 clone，通过十几个 flag 实现 POSIX 线程语义：

// pthread_create 底层 ≈ clone 的参数 clone(CLONE_VM // 共享地址空间 (mm_struct) | CLONE_FS // 共享 cwd / umask | CLONE_FILES // 共享文件描述符表 | CLONE_SIGHAND // 共享信号处理表 | CLONE_THREAD // 放入同一线程组 (TGID 共享) | CLONE_SYSVSEM // 共享 SysV 信号量撤销 | CLONE_SETTLS // 设置线程局部存储 | CLONE_PARENT_SETTID | CLONE_CHILD_CLEARTID | SIGCHLD); // 退出时向父进程发送信号

PID 与命名空间

struct pid · upid · pid_namespace

一个进程有多个 PID——在它所属的每一层 pid_namespace 中各有一个。Docker 容器利用此实现隔离：容器内进程看自己 PID=1，宿主机看到的是不同 PID。内核用 struct pid（而非整数）标识进程，通过 radix tree 管理多层映射。alloc_pid() 分配新 PID，默认上限 32768。

内核线程 (kthread)

kthreadd(PID=2) 创建 · mm=NULL · 方括号命名

内核线程只在内核态运行，无用户空间地址（mm=NULL），借用前一个进程的页表访问内核空间。通过 kthread_create() + wake_up_process() 创建。常见实例：ksoftirqd/N（软中断）、kworker/N:M（工作队列）、writeback（回写脏页）、mmcqd/N（MMC 请求）、migration/N（NUMA 迁移）。ps 中以 [方括号] 标记。

进程组 · 会话 · 控制终端

PGID · SID · Controlling Terminal

进程组 (PGID)：一组进程的集合，用于 shell job control（Ctrl+Z 暂停整个 pipeline）。setpgid() 设置。
会话 (Session)：若干进程组的集合。setsid() 创建新会话并脱离控制终端——守护进程化的关键步骤。
前台进程组：会话中唯一能接收终端输入的进程组。Ctrl+C → 向前台进程组发 SIGINT。

2. 进程生命周期：fork → exec → exit → wait

Linux 通过 fork + exec 的两阶段模型创建新程序——fork 复制资源，exec 替换内容。这是 Unix 设计哲学的核心体现。

fork() — Copy-On-Write

kernel_clone() → copy_process() → dup_task_struct()

fork 的内核路径：① dup_task_struct() 分配新 task_struct + 内核栈 ② copy_files()/copy_fs()/copy_sighand() 复制或共享资源（取决于 clone flags）③ copy_mm() 复制 mm_struct，但将所有用户页表项标记为只读（COW）④ copy_thread() 在子进程内核栈设置返回值为 0。子进程首次写入时触发缺页异常 → do_wp_page() 分配物理页并复制原始内容。COW 是 fork 性能的基石——没有它，每次 fork 都需复制 GB 级内存。

execve() — ELF 加载

sys_execve() → do_execveat_common() → load_elf_binary()

exec 的内核路径：① 打开可执行文件，读取 ELF header 验证格式 ② flush_old_exec() 释放旧的地址空间（mm、VMAs、页表）③ load_elf_binary() 解析 ELF 的 Program Headers：LOAD 段映射到内存、INTERP 段加载动态链接器 (ld.so) ④ 如需要，先加载 ld.so 再让它加载实际程序 ⑤ setup_arg_pages() 在栈顶放置 argv/envp/auxv ⑥ START_THREAD() 设置新的指令指针 (ELF entry point 或 ld.so entry)。整个过程本质是用新的内存映像完全替换旧的。

clone() 的 flag 矩阵

线程 vs 进程 vs 容器的分界线

CLONE_VM：共享地址空间（线程标志）
CLONE_FS：共享 cwd/umask
CLONE_FILES：共享 fd 表
CLONE_SIGHAND：共享信号处理表
CLONE_THREAD：放入同一线程组（TGID 共享）
CLONE_NEWNS/NEWNET/NEWPID...：创建新命名空间（容器隔离）

退出：exit → zombie → wait

do_exit() → EXIT_ZOMBIE → release_task()

进程 exit() 时：① do_exit() 释放 mm、files、fs、sighand 等资源 ② 设置 state=EXIT_ZOMBIE，保留 task_struct（含退出码）③ 向父进程发 SIGCHLD ④ 父进程 waitpid()（或 waitid/wait3/wait4）读取退出码后，release_task() 彻底释放 PID。父进程不 wait → 僵尸累积到 PID 耗尽 → 系统瘫痪。父进程先死 → 子进程由 init(PID=1) 收养。

vfork() — 废弃的优化

阻塞父进程 + 共享地址空间（无 COW）

vfork 会阻塞父进程直到子进程 exec 或 exit，且子进程直接操作父进程的地址空间（无 COW 开销）。用于 fork+exec 的极端优化场景。但现代 Linux 用 COW 已消除 fork 的内存开销，vfork 的唯一剩余优势是避免页表复制——而 CLONE_VFORK 配合 CLONE_VM 在裸机场景仍有微小的延迟优势（<10μs）。几乎不再需要手动使用。

⚠ 不可中断睡眠 (D 状态) — 排查思路
进程陷入 D 状态通常意味着内核驱动或文件系统 bug（I/O 永不完成）。排查步骤：① cat /proc/PID/stack 看内核调用栈（定位卡在哪个函数）② dmesg 查是否有 I/O 错误 ③ 检查是否是 NFS 挂载超时（NFS 默认 hard mount 会让进程永久 D）④ 用 echo w > /proc/sysrq-trigger 打印所有 D 状态进程的栈。无法用 kill -9 强制结束 D 状态进程。

3. 虚拟地址空间：mm_struct · VMA · 页表

每个进程拥有独立的虚拟地址空间——不是平坦的内存块，而是由若干 vm_area_struct 描述的有序区域集合。

mm_struct

进程的"内存身份证"

mm_struct 聚合了进程地址空间的所有信息：mmap（VMA 红黑树+链表）、pgd（页全局目录根）、start_code/end_code（代码段边界）、start_data/end_data、start_brk/brk（堆）、start_stack、mm_users（引用计数——线程共享）、total_vm。线程的 mm 指向同一 mm_struct。

vm_area_struct (VMA)

描述一段连续的虚拟地址区域

每个 VMA 描述一段虚拟内存的属性：起始/结束地址、访问权限 (r/w/x)、映射类型（匿名/文件/设备）、vm_ops（缺页/访问/命名）。/proc/PID/maps 的每一行就是一个 VMA。/proc/PID/smaps 额外展示每段的 PSS/USS/Swap 占用。VMA 用红黑树（快速查找）和双向链表（遍历）双重索引。

缺页异常 (Page Fault)

虚拟内存按需分配的引擎

进程访问一个虚拟地址时：MMU 查 TLB → TLB miss → 遍历页表 → 页表项为空或权限不足 → 硬件触发缺页异常 → 内核 do_page_fault() 处理。三类缺页：Minor（页已在内存只需建映射——COW/共享库加载）、Major（需磁盘 I/O——mmap 文件首次访问 / swap-in）、Invalid（真正的非法访问 → SIGSEGV）。

4. CFS 完全公平调度器

Linux 2.6.23 引入，替代 O(1) 调度器。核心思想：跟踪每个进程的虚拟运行时间，始终选择"最应得 CPU"的进程。

CFS 核心机制

vruntime · 红黑树 · 动态时间片

vruntime = 实际运行时间 × (NICE_0_LOAD / weight)。权重高的进程 vruntime 增长更慢——它们"理应"获得更多 CPU。调度器用红黑树按 vruntime 排序，每次选择最左节点（vruntime 最小）。
动态时间片：target_latency = sched_latency (默认 6ms) 除以运行队列中的进程数。例如 3 个进程 → 各 2ms；12 个进程 → 各 0.75ms（受 min_granularity=0.75ms 限制不会无限缩小）。

nice 值与权重映射

nice -20=权重 88761, nice 0=1024, nice +19=15

nice 每降低 1 级 → CPU 份额约增加 10%。权重数组保证相邻 nice 值之间的 CPU 比例精确为 ~1.25×：nice=-20 进程的 CPU 时间是 nice=+19 的~5900 倍。getpriority()/setpriority() 或 nice() 系统调用修改。CFS 不直接存储 nice，而是存储 static_prio (100-139) 并查表转换为 weight。

EEVDF — Linux 6.6 新调度器

Earliest Eligible Virtual Deadline First

Linux 6.6 用 EEVDF 替代 CFS 作为默认调度器。核心变化：每个进程在获得 CPU 时分配一个虚拟截止时间 (virtual deadline)。调度器选择截止时间最早的进程，而非 vruntime 最小。这消除了 CFS 的滞后延迟问题——一个进程可能因 vruntime 偏大而被"饿"很久。EEVDF 保证：① 每个进程在延迟容限内获得 CPU ② 更可预测的唤醒延迟（对实时 IPC 友好）。

负载均衡与 CPU 拓扑

SMT → MC → DIE → NUMA 层级

调度域按 CPU 拓扑分层：SMT（超线程共享核）→ MC（同 socket 多核）→ DIE（同 socket 多 die）→ NUMA（多 socket）。每层有独立的负载均衡逻辑。idle CPU 拉取 (pull)：CPU 空闲时主动从最忙的 CPU 拉任务。busy CPU 推送 (push)：负载过重时推送任务。NUMA 层面的迁移代价最大（跨 socket 访存延迟 ~2×），调度器优先在同 NUMA node 内调度。

实时调度类

SCHED_FIFO / SCHED_RR — 优先级 1–99

SCHED_FIFO：无时间片，一直运行直到自愿让出或被更高优先级抢占。
SCHED_RR：同优先级 FIFO 进程间轮转（默认 100ms 时间片）。
实时优先级严格高于所有 CFS 普通进程——一个死循环的 FIFO 进程会让系统完全卡死（连 SSH 都无法响应）。永远给实时进程设 watchdog。

SCHED_DEADLINE

最早截止时间优先 (EDF) + CBS

Linux 3.14 引入的最硬实时调度类。每个任务声明：runtime（每周期需要的 CPU 时间）、deadline（必须在此期限内完成）、period（重复周期）。内核做准入控制——如果新任务会让总利用率超 100%，拒绝接受。比 SCHED_FIFO 更优：不需要手动分配优先级，内核自动保证所有 deadline 任务按时完成。

cgroup v2 CPU 控制器

cpu.max / cpu.weight — 容器级调度

cpu.max = "$MAX $PERIOD"：硬限制（如 "200000 1000000" = 每 1 秒最多用 200ms → 20% CPU）。
cpu.weight：软控制（1-10000），类似 nice 但在 cgroup 级别。
配合 cpuset.cpus 实现容器级 CPU 亲和性绑定。Docker --cpus=2.0 底层就是写 cpu.max。

⚠ 优先级反转 (Priority Inversion) — 实时经典陷阱
场景：低优先级 L 持 mutex → 高优先级 H 等 mutex → 中优先级 M 抢占 L → H 被 M 无限期阻塞。
解决：优先级继承 (PI)——H 等 L 的锁时，L 临时继承 H 的优先级，快速完成临界区。
Linux futex 原生支持 PI：pthread_mutexattr_setprotocol(&attr, PTHREAD_PRIO_INHERIT);
内核实现：futex_lock_pi() → rt_mutex 子系统 → 自动调整任务优先级。几乎所有 RT 应用都应启用 PI mutex。

5. 信号 (Signal) — 异步通知机制

信号是 Unix 最古老的 IPC 机制——用于通知进程"发生了某件事"。简单但不适合传递数据。

可靠信号 vs 不可靠信号

SIGRTMIN (34) ~ SIGRTMAX (64)

不可靠信号（1-31）：传统 Unix 信号。同一信号多次到达可能合并为一次（丢失）。不排队，不携带额外数据。
可靠信号（34-64）：POSIX 实时信号。排队传递，每次到达独立入队。可携带一个 int 或指针数据。sigqueue() 发送。

信号生命周期

生成 → 挂起 → 递送 → 处理

生成：kill() / tkill() / tgkill() / 硬件异常 / 终端驱动（Ctrl+C）
挂起：信号在 task_struct 的 pending 位图中标记
递送：进程返回用户态时检查 pending → 如不被阻塞，调用 handler
处理：默认动作 (ignore/terminate/core dump/stop) 或 用户注册的 sigaction handler

信号与线程

进程信号 vs 线程信号

进程信号（kill）：发送给线程组，任意一个不阻塞该信号的线程接收并处理。
线程信号（tkill/tgkill）：指定线程。
SIGSEGV/SIGFPE 等同步信号只递送给触发异常的线程。
多线程程序应阻塞所有信号在主线程外，由专用信号处理线程通过 signalfd 统一处理——避免 handler 中的重入问题。

signalfd — 将信号转为文件描述符（现代做法）
// 替代 signal handler：用 signalfd 将信号转为可 poll/select 的 fd sigset_t mask; sigemptyset(&mask); sigaddset(&mask, SIGTERM); sigaddset(&mask, SIGINT); sigprocmask(SIG_BLOCK, &mask, NULL); // 阻塞这些信号，不让默认 handler 处理 int sfd = signalfd(-1, &mask, SFD_NONBLOCK | SFD_CLOEXEC); // 现在可以在 event loop 中 poll(ssd) 等待信号到来 struct signalfd_siginfo fdsi; read(sfd, &fdsi, sizeof(fdsi)); // 读取信号信息 printf("收到信号 %d 来自 PID %d\n", fdsi.ssi_signo, fdsi.ssi_pid);

6. 进程间通信 (IPC) — 7 种机制

多进程协作的核心基础设施——不同机制在数据量、延迟、持久性、语义上差异巨大

IPC 机制数据量延迟亲缘关系要求跨进程持久性关键特性

共享内存 — 视频帧 IPC 最佳实践

shm_open → ftruncate → mmap → sem_t 同步

ISP 进程写入 → sem_post(sem_full) → 编码进程 sem_wait(sem_full) → 读帧编码。延迟 <10μs（仅信号量操作开销）。关键设计：三缓冲轮转——ISP 写 Buf1、编码读 Buf0、Buf2 在排队，避免读写冲突。对比 pipe：pipe 每帧 3MB memcpy → ~1ms，共享内存 → <10μs。零拷贝进阶：用 memfd_create() + seal 创建不可修改的共享内存区域。

Unix Socket — 传递 fd (SCM_RIGHTS)

内核内存传递 · 零协议栈 · 传递文件描述符

Unix Domain Socket 的独有能力：通过 SCM_RIGHTS 控制消息在进程间传递 fd。可以传递 DMA-BUF fd——编码器进程将编码后的帧 fd 传给网络进程，网络进程直接 sendmsg 发送，全程零拷贝。SO_PEERCRED 获取对端 PID/UID/GID 做权限检查。D-Bus、systemd、X11/Wayland 都基于 Unix Socket。性能：延迟约 5-15μs（vs TCP 20-50μs on localhost）。

eventfd — 轻量事件通知

比 pipe 更简单、更快的通知机制

eventfd(initval, EFD_NONBLOCK|EFD_SEMAPHORE) 创建一个 8 字节计数器的 fd。write 增加值，read 读取并清零（或减一，semaphore 模式）。可被 poll/select/epoll 监控。典型用途：用户态中断下半部通知、替代 pipe 的 event loop 通知。比 pipe 快：无需分配 ring buffer，单一原子计数器操作。

Android Binder — 专用 IPC 框架

Android 的跨进程通信骨干

Binder 是 Android 定制的高性能 IPC：① 身份传递——内核自动记录调用者的 UID/PID，被调用方无需额外验证 ② 对象引用计数——自动管理跨进程对象生命周期 ③ 零拷贝传输（大 buffer） ④ 死亡通知——一方 crash 时对方自动获知。普通 Linux 系统不需要 Binder，但理解其设计对了解跨进程对象管理有启发意义。

7. 同步原语：从用户态到内核态

并发编程的正确性取决于对同步原语的深刻理解——不仅是 API 用法，更是底层实现和性能模型

pthread_mutex_t → futex

用户态 fast path: ~25ns | 内核态 slow path: ~1-50μs

无竞争时：pthread_mutex_lock 只是对 futex word 做一次 CAS 原子操作——不进入内核（~25ns）。有竞争时：通过 futex(FUTEX_WAIT) 系统调用进入内核睡眠，被唤醒后通过 futex(FUTEX_WAKE) 出队。futex 是 Linux 同步的基石——pthread_mutex、sem_t、pthread_cond、pthread_rwlock 全都基于 futex 实现。

自旋锁 (spinlock)

不睡眠 · CPU 空转 · 临界区必须极短

用户态 pthread_spinlock_t，内核态 spinlock_t。持锁期间禁止睡眠/抢占（内核 spinlock 自动禁用内核抢占）。
用户态 spinlock 仅在临界区 <100ns 且不希望上下文切换开销时使用。内核 spinlock 是 SMP 并发控制的基础构件——几乎所有内核同步都基于 spinlock + 中断禁用组合。

RCU — 读端完全无锁

Read-Copy-Update — 内核的核心技术

RCU 允许读者在任何时候无锁访问受保护的数据结构——无需获取锁、无需原子操作、无需内存屏障（在 rcu_dereference 中隐式包含）。写者：拷贝 → 修改副本 → 替换指针 → 等待所有读者离开 (grace period) → 释放旧副本。适用场景：读极多写极少（路由表、模块列表、文件系统 dentry cache）。内核用 rcu_read_lock()/rcu_read_unlock() 标记临界区。

信号量 (sem_t)

计数同步 — mutex 是 sem=1 的特例

POSIX 信号量支持跨进程（命名信号量 sem_open）和跨线程（匿名信号量 sem_init）。生产者-消费者经典模式：sem_wait(sem_empty); 写数据 sem_post(sem_full); 消费端 sem_wait(sem_full); 读数据 sem_post(sem_empty);。比 condvar 更简洁——无需额外的 mutex 配合，不需要 while 循环检查条件。

条件变量 (pthread_cond_t)

wait + signal — 必须配 mutex

condvar 解决"等待某个条件变为真"的问题。两个关键陷阱：① 虚假唤醒——cond_wait 可能没有 signal 也返回 → 必须用 while 而非 if 检查条件 ② 丢失唤醒——条件在 cond_wait 之前已满足，随后 cond_wait 会永久阻塞 → 先检查条件再 wait。pthread_cond_signal() 唤醒一个等待者（通常用于生产者-消费者），pthread_cond_broadcast() 唤醒所有（通常用于状态变更通知）。

⚠ 死锁四要素 (Coffman Conditions) —— 破坏任意一个即可预防
① 互斥 — 资源不能共享 ② 持有并等待 — 持 A 等 B
③ 不可抢占 — 资源不能被强制释放 ④ 循环等待 — P1→P2→P3→P1
预防策略：统一锁获取顺序（破坏④）、pthread_mutex_trylock() + 释放已持锁 + 重试（破坏②）、pthread_mutex_timedlock() 超时放弃。

8. 进程监控与调试

嵌入式 IPC 产品的进程稳定性是核心质量指标——这些工具和方法是工程师的第二双眼睛

ps · top · htop

STAT · RSS · PRI · CPU% · 进程树

ps aux：全量进程快照。关注 STAT 列：R=运行、S=睡眠、D=不可中断、Z=僵尸。
top -H -p PID：显示该进程下所有线程的 CPU 使用。
ps -eLf：列出所有线程及其 LWP (轻量进程 ID)。
ps -eo pid,tid,class,rtprio,ni,comm：查看调度策略和优先级。
htop：交互式——F5 树状图、F6 排序、F9 信号、F2 自定义列。

strace — 系统调用追踪器

-p PID · -f 追踪子进程 · -c 统计 · -T 显示耗时

strace -p PID：实时显示每次 syscall。最快速的"卡在哪"诊断工具。
strace -c -p PID：统计各 syscall 调用次数和总耗时。
strace -e trace=futex -p PID：仅追踪 futex 调用（锁竞争分析）。
strace -f ./myapp：追踪所有子进程。
⚠ strace 会让被追踪进程变慢 10-100%（ptrace 开销），生产环境慎用。

perf — 性能剖析器

CPU 采样 + PMU 硬件计数器

perf top：类 top 的实时热点函数视图。
perf record -p PID -g -- sleep 10：录制 10 秒采样 + 调用栈。
perf report：生成火焰图数据（配合 FlameGraph 脚本）。
perf stat -p PID：硬件计数器（IPC/cache-miss/branch-miss）。
嵌入式板需交叉编译 perf + 内核开启 CONFIG_PERF_EVENTS。

/proc 文件系统诊断

内核提供的最丰富的进程数据源

/proc/PID/status：Name/State/VmRSS/VmSize/Threads/SigCgt（信号位图）。
/proc/PID/stack：内核调用栈——排查 D 状态进程的利器。
/proc/PID/maps：VMA 列表（每段地址范围+权限+映射源）。
/proc/PID/smaps：每段详细内存——RSS/PSS/USS/Swap。
/proc/PID/fd/：打开的文件描述符（ls -l 显示指向的目标）。
/proc/PID/oom_score：OOM killer 的评分（越高越先被杀）。
内存泄漏诊断：watch -n1 grep VmRSS /proc/PID/status。

coredump 分析

gdb + core dump = 事后复现 crash

启用：ulimit -c unlimited + /proc/sys/kernel/core_pattern 设置文件路径。
分析：gdb /path/to/binary core.dump → bt full 查看崩溃时的完整调用栈、所有局部变量、寄存器。thread apply all bt 查看所有线程栈。info registers 看寄存器状态。
嵌入式注意：受限存储空间可能无法保存完整 core dump（进程内存多大 core dump 多大）。用 coredump_filter 可以过滤掉匿名大页、共享库等区域减少体积。

9. 守护进程化与进程管理框架

从手动 daemonize 到 systemd service——理解进程生命周期管理的演进

传统 daemonize 六步骤

脱离终端 · 脱离会话 · 防止重关联

① fork() → 父进程 exit → 子进程成为孤儿由 init 收养
② setsid() → 子进程创建新会话，成为会话首领，脱离控制终端
③ fork() 第二次 → 孙进程不再是会话首领 → 永远无法重新关联终端
④ chdir("/") → 避免 daemon 占用某个目录导致无法 umount
⑤ umask(0) → 不使用父进程的文件创建掩码
⑥ close(0); close(1); close(2); → 重定向 stdin/stdout/stderr 到 /dev/null 或日志文件

systemd service — 现代进程管理

.service 文件 · cgroup 资源控制 · 自动重启

systemd 让手写 daemonize 逻辑成为历史：Type=simple（前台运行不 fork，推荐）、Type=forking（兼容传统 daemon）、Restart=on-failure（崩溃自动重启）、RestartSec=5s（重启间隔）、StartLimitBurst=3（连续失败 3 次→冷却）。
systemd 自动为每个 service 创建 cgroup，可以限制 CPU/Memory/IO——不需要应用自己处理资源限制。

systemd unit 文件示例 — LinkricAI 进程管理
# /etc/systemd/system/linkricai.service [Unit] Description=LinkricAI IPC Camera Service After=network.target [Service] Type=simple ExecStart=/system/usr/bin/LinkricAI Restart=on-failure RestartSec=5s StartLimitBurst=3 StartLimitIntervalSec=30s User=root Nice=-5 CPUSchedulingPolicy=fifo CPUSchedulingPriority=80 AllowedCPUs=1-3# 绑定到 CPU1-3, CPU0 留给系统 LimitNOFILE=65536 LimitRSS=256M# 硬限制物理内存 OOMScoreAdjust=-500# 降低被 OOM killer 选中的概率 WatchdogSec=10s# systemd 硬件看门狗: 进程需每 5s 调用 sd_notify(WATCHDOG=1) [Install] WantedBy=multi-user.target

10. IPC 相机实战：多进程架构设计

用进程隔离构建稳定、可维护的嵌入式产品——从传感器到云端的完整进程拓扑

为什么多进程？

崩溃隔离 · 内存隔离 · 独立 OTA

编码器 crash → 不影响录像。ISP 驱动 crash → 不影响网络。线程 crash → 整个进程死——这是嵌入式产品不可接受的风险。
代价：IPC 通信开销（共享内存 → 可忽略）、进程启动时间（fork exec 通常 <50ms）、跨进程调试比线程稍复杂。

推荐进程拓扑

从 Sensor 到 Cloud 的进程拆分

主控 (LinkricAI)：SCHED_OTHER，进程树根节点。
ISP 采集：SCHED_FIFO prio=80，绑定 CPU1 → 共享内存输出 YUV。
编码器：SCHED_FIFO prio=70 → 环形缓冲输出 H.264。
网络传输：SCHED_RR prio=60 → RTP/LiveKit 推流。
存储：SCHED_OTHER nice=-5 → SD 卡写 MP4。
AI 推理（可选）：SCHED_OTHER nice=+5 → 独立进程。

优先级分配原则

越靠近硬件实时时钟，优先级越高

Sensor 采集 > ISP > 编码 > 网络 > 存储 > AI。
CPU 不足时的优雅降级：先丢 AI 帧（降低模型推理频率），再降存储帧率（不影响实时预览），最后才丢网络帧。内核线程（IRQ、ksoftirqd、writeback）也必须留足 CPU。
⚠ FIFO 进程必须设软 watchdog——一个死循环的实时进程会让系统彻底无法响应。

// 多进程架构的 Supervisor 看门狗骨架 int main() { spawn_child("isp", isp_main, { .policy=SCHED_FIFO, .prio=80, .cpu=1 }); spawn_child("encoder", encoder_main, { .policy=SCHED_FIFO, .prio=70, .cpu=2 }); spawn_child("network", network_main, { .policy=SCHED_RR, .prio=60, .cpu=2 }); spawn_child("storage", storage_main, { .policy=SCHED_OTHER,.nice=-5 }); while (1) { int st; pid_t dead = waitpid(-1, &st, 0); if (restart_count(dead, 5) > 3) { // 5秒内重启超过3次 → 冷却 log("child %d crash-loop, cooling down 30s", dead); sleep(30); reset_count(dead); } restart_child(dead); } }
