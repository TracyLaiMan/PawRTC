Linux 内核的虚拟文件系统切换层，为所有文件系统提供统一的操作接口。无论 ext4、XFS、NFS 还是 procfs，上层应用看到的都是同一套 API。

super_block · inode · dentry · file

每个对象有对应的 操作函数表：super_operations、inode_operations、dentry_operations、file_operations。多态设计实现"统一接口、多种实现"。

1. VFS — 虚拟文件系统抽象层

Linux 文件系统的基石 — 面向对象的设计思想，用统一的"四大对象"模型屏蔽底层差异

什么是 VFS？

Virtual File System Switch

VFS 是内核中的一个抽象层，位于系统调用接口与具体文件系统实现之间。它定义了通用文件模型，要求每种文件系统将自己的概念映射到这个通用模型中。"统一接口、多种实现"——这正是面向对象设计思想在内核中的体现。

两组接口

上层统一 · 下层多态

对上：提供 open/read/write/close/mmap 等与文件系统无关的通用系统调用。
对下：定义抽象操作表（函数指针），如 file_operations.read()、inode_operations.create()，每种文件系统各自实现。

支持的分类

三大家族

磁盘文件系统：ext4、XFS、Btrfs、F2FS…
网络文件系统：NFS、CIFS、Ceph…
伪文件系统：procfs、sysfs、tmpfs、devtmpfs…
查看已注册类型：cat /proc/filesystems

VFS 在 I/O 栈中的位置

2. 四大核心对象

super_block · inode · dentry · file — 每个对象都有对应的操作函数表，形成多态架构

① super_block — 超级块

文件系统实例的全局元信息

每次 mount 创建一个 super_block。核心字段：s_type（文件系统类型）、s_op（→ super_operations）、s_root（根 dentry）、s_blocksize（块大小）、s_bdev（块设备）、s_fs_info（私有数据，如 ext4_sb_info）。
关键操作：alloc_inode、write_inode、put_super（卸载时释放）、sync_fs、statfs。

② inode — 索引节点

文件的"身份证" — 元数据但不含文件名

表示一个文件对象（普通文件、目录、设备、符号链接…）。核心字段：i_ino（编号）、i_mode（类型+权限）、i_size（大小）、i_op（→ inode_operations）、i_fop（→ file_operations）、i_sb、i_mapping（→ address_space 页缓存）。
inode 不存文件名——文件名由 dentry 管理，同一 inode 可被多个硬链接引用（i_nlink 计数）。

③ dentry — 目录项

文件名 → inode 的映射缓存

纯内存结构，不存储在磁盘上。通过 d_parent 构建目录树，通过 d_inode 关联 inode。被缓存在 dcache 中以加速路径查找。
四种状态：Used（使用中）、Unused（有效但未使用，可回收）、Negative（文件不存在，加速下次查找）、Free。
路径 /home/user/f.txt → 4 个 dentry：/、home、user、f.txt。

④ file — 打开文件描述

进程视角的打开文件 — 纯内存对象

每次 open() 创建新的 struct file。核心字段：f_path（dentry + vfsmount）、f_inode、f_op（→ file_operations）、f_pos（当前偏移）、f_flags（O_RDONLY/O_WRONLY…）。
dup() 可共享同一 file（偏移量同步），多进程打开同一文件各有独立 file（独立偏移）。

对象操作表关键字段存储位置生命周期
super_blocksuper_operationss_type, s_root, s_bdev, s_fs_info磁盘 + 内存mount → umount
inodeinode_operationsi_ino, i_mode, i_size, i_mapping磁盘 + 内存创建 → 删除 (i_nlink=0)
dentrydentry_operationsd_name, d_inode, d_parent纯内存路径查找时创建，LRU 回收
filefile_operationsf_path, f_pos, f_flags纯内存open → close
对象关系速记
进程 → fd→file → f_path→dentry → d_inode→inode → i_sb→super_block
每个 file 记录当前偏移 (f_pos)；dentry 记录文件名 (d_name)；inode 记录元数据 (i_mode/size/...)。三者各司其职，通过指针串联。

3. 操作函数表 — 多态的基石

四种 operations 结构体定义了各文件系统必须实现的接口 — 上层统一调用，下层各自填充

/* ===== inode_operations — 元数据操作 ===== */ struct inode_operations { struct dentry *(*lookup)(struct inode *, struct dentry *, unsigned int); int (*create)(struct inode *, struct dentry *, umode_t, bool); int (*link)(struct dentry *, struct inode *, struct dentry *); int (*unlink)(struct inode *, struct dentry *); int (*mkdir)(struct inode *, struct dentry *, umode_t); int (*rmdir)(struct inode *, struct dentry *); int (*rename)(struct inode *, struct dentry *, struct inode *, struct dentry *, unsigned int); int (*symlink)(struct inode *, struct dentry *, const char *); int (*setattr)(struct dentry *, struct iattr *); int (*getattr)(struct vfsmount *, struct dentry *, struct kstat *); int (*permission)(struct inode *, int); }; /* ===== file_operations — 数据 I/O 操作 ===== */ struct file_operations { ssize_t (*read)(struct file *, char __user *, size_t, loff_t *); ssize_t (*write)(struct file *, const char __user *, size_t, loff_t *); ssize_t (*read_iter)(struct kiocb *, struct iov_iter *); // 现代异步读 ssize_t (*write_iter)(struct kiocb *, struct iov_iter *); // 现代异步写 int (*mmap)(struct file *, struct vm_area_struct *); int (*open)(struct inode *, struct file *); int (*release)(struct inode *, struct file *); int (*fsync)(struct file *, loff_t, loff_t, int); loff_t (*llseek)(struct file *, loff_t, int); }; /* ===== address_space_operations — 页缓存操作 ===== */ struct address_space_operations { int (*readpage)(struct file *, struct page *); int (*writepage)(struct page *, struct writeback_control *); int (*writepages)(struct address_space *, struct writeback_control *); int (*write_begin)(struct file *, struct address_space *, loff_t, unsigned, unsigned, struct page **, void **); int (*write_end)(struct file *, struct address_space *, loff_t, unsigned, unsigned, struct page *, void *); int (*direct_IO)(struct kiocb *, struct iov_iter *); };
设计模式本质
VFS 使用的是经典的面向对象 C 语言实现：结构体嵌函数指针 = 虚函数表。每个具体文件系统填充自己的函数实现，上层调用时自动派发到对应的实现——这就是 C 语言实现的多态。

缓存命中 (Fast Path)

零磁盘 I/O

页已在 address_space.i_pages (xarray) 中 → 直接 copy_to_user()。这是绝大多数读请求的实际路径。页缓存命中率通常在 95%+，是 Linux 高性能文件 I/O 的核心保障。

缓存未命中 (Slow Path)

需要磁盘 I/O

① 分配新 page → ② 加入 address_space → ③ 调用 a_ops->readpage() → ④ ext4 计算磁盘块号（通过 extent tree） → ⑤ submit_bio() 提交 bio → ⑥ 块设备驱动执行 DMA → ⑦ 中断通知 I/O 完成 → ⑧ 标记 PageUptodate → ⑨ 拷贝数据给用户。

5. Page Cache 与 Writeback

address_space 管理文件缓存页 · folio 取代 page · 2024 年 writeback_iter() 重构

address_space

页缓存的核心管理者

每个 inode 有一个 address_space（通过 i_mapping 访问）。核心：i_pages (xarray) 索引文件偏移→页帧、a_ops (address_space_operations)、nrpages (缓存页数)。一个 address_space = 一个文件的页缓存全部状态。

folio — 多页操作单元

page → folio 演进 (5.16+)

传统以 struct page 为粒度在大页时代效率低。struct folio 将多个连续 page 编组管理，支持透明大页 (THP)。现代 I/O 路径（iomap、writeback_iter）已全面 folio 化。

writeback_iter (NEW)

2024 年回写路径重构

传统 write_cache_pages() 使用回调模式，代码臃肿。新 writeback_iter() 改为直接迭代器模式：while ((folio = writeback_iter(mapping, wbc, folio, &err))) {...}。更简洁、更高效、更易维护。

回写触发条件阈值 / 机制说明
内存压力dirty_background_ratio (默认 10%)脏页超此比例 → flusher 线程异步回写
写入节流dirty_ratio (默认 20%)超此比例 → 进程自行回写，阻塞写入
脏页老化dirty_expire_interval (3000ms)脏页超过此时间 → flusher 强制回写
周期回写dirty_writeback_interval (500ms)flusher 线程定时唤醒检查
显式同步sync() / fsync() / fdatasync()用户空间或内核显式调用
核心思想：Writeback 解耦
应用程序 write() 只将数据拷贝到页缓存并标记脏页 SetPageDirty(page)，不直接写磁盘。后台 flusher 线程 ([writeback]) 在合适的时机（内存压力、超时、显式 sync）将脏页批量回写。这就是 Linux 高性能写入的秘密——延迟分配 + 批量回写。

ext4 — 稳定之王

最广泛使用的 Linux 文件系统

优势：极端稳定，经二十年验证，fsck 可靠，默认几乎所有发行版。
弱点：缺 CoW 高级特性，无快照。
选择场景：通用桌面/服务器、启动分区、不想为文件系统操心的场景。

XFS — 性能之王

RHEL 默认，高性能并行 I/O

优势：分配组 (AG) 实现极致并发，大文件/高吞吐场景无出其右。在线扩容。
弱点：不能缩容，历史断电风险（已改善）。
选择场景：数据库服务器、大文件存储、高并发写入、HPC、媒体处理。

Btrfs — 特性之王

CoW 文件系统的 Linux 旗舰

优势：快照、子卷、压缩、数据校验和、内置 RAID、send/receive 备份。
弱点：CoW 带来性能开销，内核 <5.4 不稳定，RAID 5/6 仍有风险。
选择场景：NAS、需要快照回滚、数据完整性优先、openSUSE snapper 集成。

F2FS — 闪存专家

原生 NAND Flash 设计

优势：冷热数据分离、多头日志减少写放大、延长 SSD 寿命。安卓默认。
弱点：fsck 弱（突然断电有风险）、文件系统容量小。
选择场景：NVMe SSD / eMMC / SD 卡、安卓设备、编译密集型开发环境。

选型速决：一句话决策
不想操心 → ext4  |  追求性能 → XFS  |  需要快照/回滚 → Btrfs  |  纯闪存设备 → F2FS

8. ext4 内部设计

Extent Tree · JBD2 日志 · 延迟分配 · 块组布局 — 最经典文件系统的实现要点

/* ===== ext4 磁盘布局 ===== */ +-----------+------------+-------------+-----------------+ | Superblock| Group Desc | Inode Table | Data Blocks | | (Block 0)| Table | | (Extent-mapped) | +-----------+------------+-------------+-----------------+ /* ===== Extent — ext4 的核心块映射结构 ===== */ struct ext4_extent_header { // Extent 树头 (inode 或 extent block 中) __le16 eh_magic; // 魔数 0xF30A __le16 eh_entries; // 条目数 __le16 eh_max; // 最大条目数 __le16 eh_depth; // 树深度 (0=叶子) }; struct ext4_extent { // 叶子节点 — 一段连续物理块 __le32 ee_block; // 起始逻辑块号 __le16 ee_len; // 块数 (≤32768) __le16 ee_start_hi; // 物理块号高 16 位 __le32 ee_start_lo; // 物理块号低 32 位 → 支持 48-bit 块地址 }; /* Extent 示例: 逻辑块 0-99 → 物理块 5000-5099 (一个 Extent 描述 100 个连续块) */

Extent 树 vs 间接块

ext4 vs ext3 的关键差异

ext3 用间接块：inode → 间接块 → 双重间接块 → 三重间接块，大文件需要多次 I/O 才能找到数据块。
ext4 用 Extent 树：{逻辑起始, 块数, 物理起始} 直接描述连续区间。一个 extent 可映射 128MB (4K×32768)，大幅减少元数据开销和查找延迟。

JBD2 日志

Journal Block Device 2

ext4 使用 JBD2 实现写前日志 (Write-Ahead Logging)。三种模式：
data=journal — 数据+元数据全日志（最安全最慢）
data=ordered — 数据先写、元数据后日志（默认，推荐）
data=writeback — 仅元数据日志（快但有风险）

延迟分配 (Delayed Allocation)

减少碎片的关键技术

write() 时 不立即分配磁盘块，只缓存在页缓存中。等到 writeback 时才一次性分配。好处：写多个小块时，内核能看到全部数据量，可以分配连片的大 Extent，大幅减少文件碎片。

灵活块组 (Flex BG)

大容量优化的布局策略

传统 ext 文件系统每个块组独立管理 inode 位图和块位图。Flex BG 将多个块组绑定在一起，共享 inode 和块空间，减少位图元数据开销，对大容量 (TB 级) 文件系统特别有效。
