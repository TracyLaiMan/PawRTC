1. printk — 内核日志的唯一入口

printk() 是 Linux 内核中所有日志消息的唯一起点——从驱动报错到 Oops 崩溃信息，每一行 dmesg 输出都源于一次 printk 调用。

printk 的内部流程

格式化 → 加时间戳 → 写 ring buffer → 唤醒 console

① 格式化：类似 printf 的格式串处理（%s %d %p %px 等内核特有的格式说明符——%pK 对内核指针做哈希遮蔽保护安全）
② 加前缀：根据日志级别拼接 "<N>" 级别标记
③ 原子写入 ring buffer（log_buf 数组——自旋锁保护，可在任何上下文调用）
④ 唤醒 console 输出（如果当前级别 ≥ console_loglevel 阈值）
⑤ 最关键的保证：printk 可在中断上下文、持自旋锁期间、NMI 中安全调用——是内核中最可靠的日志方式。

日志等级 (Log Levels)

KERN_EMERG(0) → KERN_DEBUG(7)

0 · KERN_EMERG：系统不可用（即将关机/重启）
1 · KERN_ALERT：必须立即采取行动
2 · KERN_CRIT：临界条件（硬件严重错误）
3 · KERN_ERR：错误条件（驱动 probe 失败、I/O 错误）
4 · KERN_WARNING：警告（配置不推荐、降级运行）
5 · KERN_NOTICE：正常但值得注意
6 · KERN_INFO：信息（设备注册、初始化完成）
7 · KERN_DEBUG：调试信息

用法实例与格式说明符

printk(KERN_ERR "message") — 无逗号！

正确写法：printk(KERN_ERR "i2c timeout: addr=0x%02x\n", addr);——注意 KERN_ERR 和字符串之间没有逗号（预处理器拼接字符串）。错误写法：printk(KERN_ERR, "msg") ← 有逗号会编译失败。
推荐写法（现代内核）：pr_err("i2c timeout: addr=0x%02x\n", addr);——pr_* 系列宏自动加级别前缀，更简洁、防遗漏换行符。

printk 防遗忘一览使用方式自动加设备名适用场景
pr_emerg(fmt, ...)pr_emerg("System halted\n");✗系统级紧急
pr_alert(fmt, ...)pr_alert("Temperature critical\n");✗需要人工干预
pr_crit(fmt, ...)pr_crit("DDR ECC uncorrectable\n");✗硬件严重错误
pr_err(fmt, ...)pr_err("probe failed: %d\n", ret);✗驱动错误
pr_warn(fmt, ...)pr_warn("deprecated API used\n");✗警告
pr_notice(fmt, ...)pr_notice("version 2.0 loaded\n");✗值得注意
pr_info(fmt, ...)pr_info("device registered\n");✗信息输出
pr_debug(fmt, ...)pr_debug("reg=0x%08x\n", val);✗调试（需开启 dynamic_debug）

dev_dbg(dev, fmt, ...)dev_dbg(dev, "irq=%d\n", irq);✓设备调试
⚠ printk 的性能代价 — 生产环境必须控制
printk 虽然可在任何上下文调用，但不是免费的：格式化字符串 + 写 ring buffer + 唤醒 console 驱动（如果级别够高）——其中同步串口 console 输出是最耗时的（UART 115200bps ~11.5KB/s，打印一行 100 字节 ~9ms——在中断上下文中 9ms 是灾难）。
生产环境策略：① console_loglevel 设为 3（只显示 ERR 及以上）② 调试信息用 pr_debug/dev_dbg（编译时默认关闭）③ 不要在高频路径（中断、每帧调用）中使用 printk——改用 tracepoint 或统计计数器 ④ 启用 CONFIG_PRINTK_TIME 确认每条日志的时间戳可度量性能影响。

2. 环形缓冲区 (Ring Buffer) — 内核日志的存储核心

所有 printk 输出都写入一个固定大小的环形缓冲区——理解其结构和行为是解读/管理内核日志的基础。

物理结构

__log_buf[] — 启动时分配，永不释放

内核维护一个循环字符数组 __log_buf，默认大小由 CONFIG_LOG_BUF_SHIFT 决定（如 17 → 128KB，18 → 256KB，最大 21 → 2MB）。每条日志记录包含：前缀（级别 + 时间戳 + CPU ID + 进程名 PID）和消息体。通过启动参数 log_buf_len=2M 可增大。
循环覆盖：缓冲区满了以后，最老的日志被新日志覆盖——不会 OOM、不会阻塞。代价：长时间运行后最早的日志丢失。

读取路径

dmesg · /dev/kmsg · syslog syscall · kmsg dump

内核提供4 条读取路径访问 ring buffer：
① /dev/kmsg：字符设备——支持阻塞 read、poll、非破坏性 seek（读取后不清除——多消费者模式）。这是 systemd-journald 和 dmesg 的底层接口。
② sys_syslog 系统调用：syslog(SYSLOG_ACTION_READ, ...)——dmesg 命令的古老接口，读取后清除。不如 /dev/kmsg 功能全。
③ dmesg：用户态工具——通过 /dev/kmsg 或 syslog syscall 读取。
④ kmsg_dump：在内核 panic/Oops 时将整个 ring buffer dump 到 MTD 分区或 pstore，用于事后崩溃分析。

/dev/kmsg vs syslog — 为什么 /dev/kmsg 是未来的方向
/dev/kmsg 的优势：① 多消费者——多个进程同时打开/读取，各自独立 seek（syslog 只有一个全局 offset）② 非破坏性读取——读取后日志仍在 ring buffer 中 ③ 完整的元数据——每条消息包含 facility、序列号、时间戳、级别——不再是"一段文本"而是结构化记录 ④ 支持 poll/epoll——可集成到 event loop。内核3.5+引入。现代 Linux 发行版的 systemd-journald、dmesg 等都基于 /dev/kmsg。

3. Console 输出控制

printk 除了写 ring buffer，可选地将日志同步输出到注册的控制台（串口、屏幕、netconsole）——这个"可选"由日志级别阈值决定。

console_loglevel 阈值

/proc/sys/kernel/printk — 四个值

/proc/sys/kernel/printk 包含 4 个值，空格分隔：
① console_loglevel：当前控制台阈值——只有 <= 此值的日志才输出到控制台。默认 7（全输出）→ 嵌入式生产环境应设为 3（只输出 ERR 及以上）。
② default_message_loglevel：未指定级别的 printk 的默认级别。
③ minimum_console_loglevel：console_loglevel 可设置的最小值（硬底限）。
④ default_console_loglevel：启动时的默认值。
示例：echo "3 4 1 7" > /proc/sys/kernel/printk

控制台驱动类型

串口 · framebuffer · netconsole · earlycon

串口 console（8250/pl011/msm_serial）：最常用——UART 连接到调试 PC。内核配置 console=ttyS0,115200 earlyprintk。
framebuffer console：输出到显示屏（fbcon + tty）。
netconsole：日志通过 UDP 包发送到远程主机——无需物理串口。配置："netconsole=6666@10.0.0.1/eth0,6666@10.0.0.2/"
earlycon：内核极早期阶段（串口驱动加载之前）的 console——通过设备树 chosen/stdout-path 或命令行 earlycon=uart,mmio32,0x... 指定。

生产环境策略

降低 console 噪声 = 提升性能 + 清晰可读

① console_loglevel=3：只输出 ERR/CRIT/ALERT/EMERG——驱动 probe 成功、设备注册等信息不再刷屏。
② 关闭 CONFIG_PRINTK_TIME（如果不需要时间戳）。
③ quiet 启动参数：启动时静默模式——等价于 console_loglevel=4。
④ 使用 ignore_loglevel 恢复全量打印（调试时用）。
⑤ 嵌入式 IPC：保留串口 console 但级别设 3 + 关键信息用 pr_err 输出 + 详细调试走 netconsole 或事后 dmesg 查看（不刷串口）。

⚠ 串口 console 的性能陷阱 — 在 115200bps 下打印一行 100 字节 ≈ 9ms
在中断上下文中调用 printk(KERN_ERR ...) 且 console_loglevel ≥ 3 时，内核会同步等待串口 FIFO 有空位后将字符逐个发送——这 9ms 全部消耗在中断上下文中。如果这是一个 1ms 周期的定时器中断 → printk 导致中断执行时间从 10μs 膨胀到 9ms → 系统实时性彻底崩溃。
解决方案：① 将 console_loglevel 设低 ② 使用 printk_deferred()——将日志推迟到 workqueue 执行 ③ 使用 trace_printk 或 tracepoint 替代高频路径的 printk ④ 启用 CONFIG_PRINTK_NMI 将 NMI 上下文中的 printk 推迟到 safe context。

4. Dynamic Debug — 运行时的调试开关

pr_debug/dev_dbg 默认不输出任何内容——通过 Dynamic Debug 可以在运行时按文件/函数/模块/行号精确启用。

启用方式

CONFIG_DYNAMIC_DEBUG=y — 内核配置前提

内核启用 CONFIG_DYNAMIC_DEBUG 后，每个 pr_debug/dev_dbg 调用点在编译时注册到 __dyndbg 段（记录文件/函数/行号/模块名）。默认全部禁用（零开销）。运行时通过 /sys/kernel/debug/dynamic_debug/control 控制：
echo 'file my_driver.c +p' > control —— 启用 my_driver.c 中所有 pr_debug
echo 'module my_module +p' > control —— 启用指定模块
echo 'func my_isr +p' > control —— 启用指定函数
echo 'file * +p' > control —— 启用所有（谨慎——可能刷爆串口）

高级控制：标记与过滤

+p 启用 · -p 禁用 · +fl 行号 + 模块过滤

+p：启用 printk 输出。-p：禁用。
+fl：输出文件名 + 行号前缀（方便定位日志来源）。
精确到行号：echo 'file my_driver.c line 150 +p' > control
查看所有调用点：cat /sys/kernel/debug/dynamic_debug/control（列出每个 pr_debug 的module/func/file/line/enabled 状态）
启动时启用：内核命令行 dyndbg="file my_driver.c +p"
IPC 实践：用 dev_dbg 写调试日志 → 正常运行时零开销 → 现场出问题时远程 SSH/ADB 进去 echo 一行命令启用 → dmesg 看输出 → 排查完禁用。无需重编译、无需重启设备。

5. 用户态工具链：dmesg · journalctl · klogd

内核写日志，用户态读日志——从最原始的 dmesg 到结构化 journald 的完整工具栈。

dmesg — 最直接的读取

/dev/kmsg 或 syslog syscall

dmesg（无参数）：打印 ring buffer 中全部日志。
dmesg -w（--follow）：阻塞等待新日志——类 tail -f。
dmesg -n 3：等价于设置 console_loglevel=3。
dmesg -T：将时间戳转为可读时间（需要内核 CONFIG_PRINTK_TIME）。
dmesg --level=err,warn：只显示指定级别。
dmesg -c：读取后清空 ring buffer（危险——其他消费者将丢失日志）。

journald — systemd 的日志守护

systemd-journald → /dev/kmsg → 持久化存储

systemd-journald 打开 /dev/kmsg，持续读取内核日志 + 用户态服务的 stdout/stderr（通过 syslog socket），统一写入结构化的二进制日志文件（/var/log/journal/）。
journalctl -k：查看内核日志（等价于 dmesg）。
journalctl -k -p err：只看 ERROR 及以上。
journalctl -k --since "10 min ago"：时间范围过滤。
journalctl -k -b -1：上一次启动的日志（如果 /var/log/journal/ 是持久存储）。
嵌入式设备通常不安装 journald（体积大 + 需要持久存储）→ 直接用 dmesg + 串口 console。

klogd / syslogd

传统 syslog 方案 — 小型嵌入式常用

klogd 打开 /proc/kmsg（旧接口）或 /dev/kmsg，读取内核日志并转发给syslogd。syslogd 根据 /etc/syslog.conf 规则将日志写入文件（如 /var/log/messages）、转发到远程 syslog 服务器、或打印到终端。
轻量替代：busybox 的 syslogd + klogd——几 KB 的体积，适合嵌入式 IPC 产品。
日志轮转：logrotate 周期性压缩/归档/删除旧日志——防止 /var/log 撑满文件系统。

6. 内核编译配置 — 日志相关 Kconfig 全景

默认内核配置可能不适合嵌入式 IPC 场景——理解每个选项的影响才能做出正确选择。

内核配置默认含义IPC 建议
CONFIG_PRINTKy整个 printk 子系统——关闭后整个内核无日志输出必须开启
CONFIG_PRINTK_TIMEy每条日志带时间戳（[seconds.microseconds]）建议开启
CONFIG_LOG_BUF_SHIFT17 (128KB)ring buffer 大小 = 2^N 字节建议 18 (256KB) 或更大

CONFIG_PRINTK_NMIyNMI 上下文中的 printk 推迟到 safe context建议开启
CONFIG_CONSOLE_LOGLEVEL_DEFAULT7启动时默认的 console 阈值生产设为 3

7. ftrace & trace_printk — 比 printk 更适合高频调试

printk 在高频路径（每帧、每个中断、每秒几千次）中会严重影响性能——ftrace/tracepoint 是专业的替代方案。

ftrace — 内核内置追踪器

tracefs → /sys/kernel/tracing/

ftrace 是内核内置的无锁、每 CPU 环形缓冲追踪框架，开销远低于 printk（几 μs vs 几 ms）。
支持的事件源：系统调用、中断、调度事件、函数调用图、tracepoint、kprobe。每个事件源可以独立开关。
输出方式：trace-cmd（命令行工具）→ 生成 trace.dat → kernelshark（GUI 可视化）——比看 dmesg 强大几个数量级。
IPC 场景：用 ftrace 追踪 ISP 中断延迟、编码器调度延迟、Video Buffer 队列耗时——每帧一条 trace 记录，不会淹死在 printk 海洋中。

trace_printk — 极简追踪

printk 的 ftrace 等价物

trace_printk("buf %d filled, irq=%d\n", idx, irq);——用法和 printk 几乎一样，但输出到 ftrace 的 trace buffer 而非 console。适用于临时调试——写进代码、运行、看 trace、删除——完整的 printk 替代品。
优势：① 不经过 console 驱动——不会阻塞串口 ② per-CPU buffering——无全局锁竞争 ③ 输出包含自动的 task/CPU 信息。
限制：① 不能保证在所有上下文中安全 ② 不是 ABI 稳定的 ③ 编译时有警告提醒"这只是调试用，不应留在正式代码中"。

ftrace 快速入门 — 追踪一个函数的调用者
# 1. 挂载 tracefs mount -t tracefs none /sys/kernel/tracing # 2. 启用函数图追踪 + 过滤目标函数 echo function_graph > /sys/kernel/tracing/current_tracer echo my_isr > /sys/kernel/tracing/set_graph_function # 3. 开启追踪 echo 1 > /sys/kernel/tracing/tracing_on # 4. 读取结果（每个函数调用的耗时） cat /sys/kernel/tracing/trace # 0) | my_isr() { # 0) 0.834 us | readl(); // 读寄存器耗时 0.834μs # 0) 5.123 us | i2c_transfer(); // ← 这里是瓶颈！ # 0) 6.957 us | } # 5. 停止追踪 echo 0 > /sys/kernel/tracing/tracing_on echo nop > /sys/kernel/tracing/current_tracer

8. IPC 嵌入式日志实践

IPC 相机产品的日志策略——从开发调试到生产部署的完整方案。

开发阶段：全量日志 + 快速迭代

console_loglevel=7 · dynamic_debug · netconsole

• 串口 console 保持 loglevel=7——编译调试内核时不怕刷屏，性能不是瓶颈
• 驱动中使用丰富的 dev_dbg 输出寄存器读写、中断触发、状态变化
• 使用 dynamic_debug 按需开关调试日志——不用反复编译
• netconsole：如果板子没有引出串口，用 netconsole 通过 UDP 发日志到开发 PC
• earlycon：如果内核在 console 驱动加载前 crash，earlycon 可以看到 crash 前的最新打印
• Oops/Panic 后：通过 pstore + ramoops 读取上一次崩溃的日志（reboot 后 dmesg 发现 pstore 内容）

生产阶段：精简日志 + 按需调试

console_loglevel=3 · dmesg 存 SD · 远程诊断

• console_loglevel=3——只输出 ERR/CRIT/ALERT/EMERG，正常操作串口零噪音
• pr_err/deb_err 记录关键错误（驱动 probe 失败、I2C 超时、DDR ECC 错误）
• 定期将 dmesg 追加写入 SD 卡日志文件（如 /data/log/kmsg.log）——每次重启保留
• 远程诊断：用户在 App 点击"上传日志"→ IPC 通过 HTTP 上传 dmesg+应用日志到云端
• OOM/panic 诊断：pstore+ramoops 在重启后保留 panic 日志 → 下次启动时读 pstore 发送到云端
• 正常运行时 pr_debug 全部禁用——零开销。现场排查：远程 SSH/ADB 开启 dynamic_debug → 复现问题 → 导出日志 → 关闭 dynamic_debug

⚠ IPC 相机日志的三大禁忌
1. 禁止在 ISP/编码中断或高频回调中使用 printk。ISP 每帧完成触发一次中断（30fps = 每 33ms），在中断中 printk → 串口阻塞 9ms → 中断延迟从 2μs 膨胀到 9ms → 丢帧。替代：用 trace_printk 或 per-CPU 统计计数器，每 100 帧聚合 print 一次。
2. 禁止 SD 卡满时无限 write 日志。日志写满 SD 卡 → 录像无法写入 → SD 卡被写穿。日志文件必须设置大小上限（如 10MB 循环写入）。
3. 禁止将用户态日志和内核日志混淆。应用 crash 不是内核 panic——不要在内核日志中搜索应用错误。应用日志走 syslog/journald/自定义文件，内核日志走 dmesg。两者独立不能互相替代。
