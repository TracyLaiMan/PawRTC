1. 并发编程基础 — CPU 如何看内存

SMP 架构 · 缓存一致性 · 内存模型 · 并发的三重来源

并发的三重来源

为什么内核需要这么多同步机制？

1. SMP (多核并行): 两个 CPU 核心同时执行不同任务，可能同时访问同一变量。
2. 中断上下文: 硬中断/软中断可以在任何时刻打断进程上下文。中断处理函数与被打断的代码可能操作同一数据。
3. 内核抢占: CONFIG_PREEMPT=y 时，高优先级任务可以抢占当前在内核态运行的任务。

MESI 缓存一致性协议

硬件层面如何保证 cache line 一致性

每个 CPU 核心有自己的 L1/L2 cache，共享数据通过 MESI 协议保持一致性：
Modified — 本核独占，已修改 (dirty)
Exclusive — 本核独占，与内存一致 (clean)
Shared — 多核共享，只读
Invalid — 失效，需从其他核/内存重新获取
当 CPU0 写一个 Shared 行 → 需要先 Invalidate 其他核的副本 → cache line bouncing → 性能杀手。

🔬 理解乱序执行与编译器优化
CPU 乱序执行: 现代 CPU (x86 OoO、ARM) 会重排指令执行顺序以获得更高的 ILP (指令级并行)。例如 store 可能先于前面的 load 对其他核可见。
编译器优化: gcc -O2 可能会重排代码、将变量缓存在寄存器中、甚至优化掉看起来"无意义"的读操作。
后果: 你写的 C 代码执行顺序 ≠ CPU 实际执行顺序 ≠ 其他核看到的顺序。这就是为什么需要 内存屏障 和 READ_ONCE/WRITE_ONCE。

2. 原子操作 — 锁的基石

atomic_t · READ_ONCE/WRITE_ONCE · cmpxchg · 内存顺序语义

atomic_t — 不可分割的整数操作

<linux/atomic.h> — 架构无关的原子 API
// 声明与初始化 atomic_t v = ATOMIC_INIT(0); // 读/写/加减 (全原子，无锁) atomic_read(&v); // 原子读 atomic_set(&v, 1); // 原子写 atomic_inc(&v); // 原子自增 (LOCK incl on x86) atomic_dec(&v); // 原子自减 atomic_add(5, &v); // 原子加 // 原子读-改-写 (RMW — Read-Modify-Write) atomic_inc_return(&v); // 自增并返回新值 atomic_cmpxchg(&v, old, new); // CAS 原子比较交换 atomic_xchg(&v, new); // 原子交换

READ_ONCE / WRITE_ONCE

阻止编译器"自作聪明"地优化
// 不良示例 — 编译器可能优化掉循环读 while (flag == 0) // 被优化为 if (flag==0) loop_forever cpu_relax(); // 正确 — READ_ONCE 禁止编译器缓存该变量 while (READ_ONCE(flag) == 0) cpu_relax(); // WRITE_ONCE — 确保写入对其他 CPU 可见 WRITE_ONCE(shared_var, value); // 作用: 1. 禁止编译器优化掉该次读写 // 2. 禁止编译器将该读写与其他操作重排 // 3. 确保以单次指令完成 (非拆分访问)

2.1 atomic 的返回值语义 — 什么时候需要 memory ordering

API (非 RMW)内存序API (RMW + 返回)内存序
atomic_read()无屏障 (relaxed)atomic_inc_return()全屏障 (fully ordered)
atomic_set()无屏障 (relaxed)atomic_dec_return()全屏障
atomic_read_acquire()acquire 语义atomic_fetch_add()全屏障
atomic_set_release()release 语义atomic_cmpxchg()全屏障
⚠️ atomic_t 不是万能锁
atomic_t 只能保护单个变量的原子访问。它不保护多个变量之间的复合操作。例如：
if (atomic_read(&v) > 0) atomic_dec(&v); — 这不是原子的！Read 和 dec 之间另一核可能也读到 v>0 并 dec，导致 v 变成 -1。
解决方案: 用 atomic_dec_if_positive() 或改用 mutex/spinlock 保护复合操作。

3. 自旋锁 (Spinlock) — 忙等的艺术

raw_spinlock · spin_lock_irqsave · ticket spinlock · qspinlock · 使用规则

spinlock 数据结构

include/linux/spinlock_types.h
typedef struct spinlock { union { struct raw_spinlock rlock; // qspinlock 内部结构 (x86-64 / ARM64): // ┌──────┬─────────┬──────────┐ // │ tail │ pending │ locked │ // │ 16b │ 8b │ 8b │ // └──────┴─────────┴──────────┘ // locked: 当前持有者 // pending: 有一个等待者 (自旋) // tail: MCS 队列尾部索引 }; } spinlock_t;

核心 API — 记住这些变体

保护场景决定使用哪个变体
// 1. 纯进程上下文 (无中断竞争) spin_lock(&lock); spin_unlock(&lock); // 2. 共享数据会被中断处理函数访问 // → 必须关本地中断! 防止死锁 spin_lock_irqsave(&lock, flags); spin_unlock_irqrestore(&lock, flags); // 3. 共享数据会被 softirq/tasklet 访问 spin_lock_bh(&lock); // 关 bottom half spin_unlock_bh(&lock); // 4. 仅关中断 (不保存 flags，已知中断已关) spin_lock_irq(&lock); spin_unlock_irq(&lock);
⚠️ spinlock 三大铁律
1. 持锁期间不能睡眠 — 不能调用 copy_from_user()、kmalloc(GFP_KERNEL)、schedule() 等任何可能睡眠的函数。违反 → kernel panic.
2. 持锁时间越短越好 — spinlock 是忙等，长时间持锁会让其他 CPU 空转。临界区应只有几十条指令。
3. 不可递归 — spin_lock 不支持同一 CPU 递归获取。重复 lock 会死锁自己。
4. 中断上下文必须用 _irqsave — 如果中断处理函数也拿同一把锁，进程上下文不关中断就 lock → 中断到来 → 中断上下文再 lock → 本核死锁。

4. 互斥锁 (Mutex) & 信号量 (Semaphore)

睡眠锁 — 争用时让出 CPU · rt_mutex · PI 协议 · counting semaphore

struct mutex

include/linux/mutex.h

最常用的睡眠锁。争用时调用者进入 TASK_UNINTERRUPTIBLE 睡眠。
优化路径 (fastpath): 无竞争时只需一次原子 cmpxchg — 与 spinlock 一样快。
慢路径 (slowpath): 有竞争时走入 __mutex_lock_slowpath()，将任务加入等待队列，调用 schedule() 让出 CPU。
DEBUG_MUTEXES: 开启后跟踪持有者、检测死锁、检测释放非持有锁。

rt_mutex — 实时互斥锁

PREEMPT_RT 补丁的核心依赖

普通 mutex 有优先级反转问题。rt_mutex 实现 PI (Priority Inheritance，优先级继承)：
当高优先级任务等待锁时，临时提升锁持有者的优先级 → 持有者尽快执行完临界区 → 释放锁 → 高优先级任务立即获得锁。
futex(FUTEX_LOCK_PI) 用户态 PI mutex 的内核实现即基于 rt_mutex。
PREEMPT_RT 将所有 spinlock 替换为 rt_mutex，使内核完全可抢占。

5. 读写锁与 RCU — 读多写少的优化

rwlock · rwsem · RCU — 三种不同的"读优化"策略

rwlock_t (自旋读写锁)

允许多个读者并发，写者独占
read_lock(&rwlock); // 读者获取 (多读者可共存) read_unlock(&rwlock); write_lock(&rwlock); // 写者获取 (独占) write_unlock(&rwlock); // 同样有 _irqsave / _bh 变体 read_lock_irqsave(&rwlock, flags); write_lock_irqsave(&rwlock, flags); // 限制: 1. 同样是自旋锁，不能睡眠 // 2. 若读者持续到来，写者可能饿死 // 3. 开销比 spinlock 大 (需维护读者计数) // 适用: 读比例 > 90%，临界区极短

rwsem (读写信号量)

睡眠版读写锁 — 适用于长临界区
down_read(&rwsem); // 读者获取 (可睡眠) up_read(&rwsem); down_write(&rwsem); // 写者获取 (独占，可睡眠) up_write(&rwsem); // 也支持: down_read_interruptible() // down_write_killable() // downgrade_write() — 写锁降级为读锁 // 适用: 文件系统 (struct super_block->s_umount) // 内存管理 (mmap_lock，原 mmap_sem)

5.1 RCU — Read-Copy-Update · 完全无锁读

RCU 核心思想

读不加锁 · 写复制替换 · 宽限期

1. 读者: 通过 rcu_read_lock() / rcu_read_unlock() 标记临界区 — 但在大多数架构上 这两个是空操作！仅是禁止抢占的编译器屏障。
2. 写者: 不直接修改旧数据。先复制一份 → 修改副本 → 用副本替换指针 → 等待宽限期 → 释放旧数据。
3. 宽限期 (Grace Period): 等待所有 CPU 核都至少经历一次上下文切换（即确认没有任何读者还在引用旧数据）。
关键: 读者永远感知不到"半更新"的数据——指针替换是原子的。

RCU API 三件套

读者 · 写者 · 回收
// ── 读者 ── rcu_read_lock(); obj = rcu_dereference(global_ptr); if (obj) { val = obj->data; // 安全：保证 obj 不会被释放 } rcu_read_unlock(); // ── 写者 (更新) ── new_obj = kmalloc(...); *new_obj = *old_obj; // 复制 new_obj->data = new_value; // 修改副本 old = rcu_replace_pointer(&global_ptr, new_obj, lock_is_held(&my_lock)); // 或: rcu_assign_pointer(global_ptr, new_obj); // ── 回收旧数据 ── synchronize_rcu(); // 阻塞等宽限期 (可睡眠) kfree(old); // 现在安全了 // 或异步回收: call_rcu(&old->rcu_head, my_callback); // 非阻塞
✅ RCU 适用场景
完美适用: 读极多 (>99%)、写极少、读者不能阻塞、数据结构以指针链接。
内核典型用例: 路由表 (fib_lookup)、文件描述符表 (fdtable)、内核模块列表、audit 规则、网络过滤规则 (netfilter)。
不适合: 写频繁 (>10%) — synchronize_rcu 的开销不可忽略；读者需要阻塞 — rcu_read_lock 内不能睡眠。

6. 其他同步机制 — 工具箱补完

Completion · seqlock · wait_queue · Per-CPU · local_lock · memory barriers

Completion

一次性同步 — "我好了，你继续"

场景: 线程 A 等线程 B 完成某个初始化后再继续。
init_completion() / reinit_completion()
wait_for_completion() — 等待
complete() / complete_all() — 通知
异步 API: wait_for_completion_interruptible()、wait_for_completion_timeout()
典型应用: 内核模块初始化等待固件加载、设备探测完成信号。

seqlock (顺序锁)

写者优先 — 读者检测重试
// 读者: do { seq = read_seqbegin(&seqlock); // ... 读取数据 ... } while (read_seqretry(&seqlock, seq)); // 写者: write_seqlock(&seqlock); // ... 修改数据 ... write_sequnlock(&seqlock); // 读者无锁，但检测到写冲突时重试 // 典型: jiffies, timekeeping, 统计计数

Wait Queue

等待条件满足 — 睡眠/唤醒
DECLARE_WAIT_QUEUE_HEAD(wq); // 等待方: wait_event(wq, condition); wait_event_interruptible(wq, cond); wait_event_timeout(wq, cond, timeout); // 唤醒方: wake_up(&wq); wake_up_interruptible(&wq); // 唤醒所有 vs 唤醒一个: wake_up_all / wake_up_interruptible_all

Per-CPU 变量

消除 cache line 争用的终极武器

每个 CPU 一份独立副本 → 完全无锁访问。
DEFINE_PER_CPU(type, name)
get_cpu_var() / put_cpu_var() — 访问并关抢占
this_cpu_ptr() — 本核指针
per_cpu() — 跨核访问 (需要额外同步)
限制: 内存开销 × CPU 数、跨核汇总需要遍历所有核、不能用于需要全局一致视图的场景。

local_lock

PREEMPT_RT 时代的 per-CPU 锁

传统的 preempt_disable() + per-CPU 在 PREEMPT_RT 上不够 (spinlock 变成了可睡眠的 rt_mutex)。
local_lock() 显式声明"我在操作 per-CPU 数据，不要把我迁移到其他 CPU"。
local_lock_irqsave() / local_unlock_irqrestore()
内核 5.16+ 推荐使用 local_lock 替代手动 preempt_disable。

内存屏障 (Memory Barrier)

最底层的顺序保证
// 全屏障: 前后所有读写不跨越 smp_mb(); // StoreLoad barrier // 单向屏障: smp_rmb(); // 读不跨越 smp_wmb(); // 写不跨越 // 编译屏障 (仅约束编译器，不约束 CPU): barrier();

Workqueue (cmwq)

Concurrency-Managed Workqueue

最常用的异步机制 — 在进程上下文中延迟执行工作。
schedule_work() → system_wq (全局共享)
schedule_delayed_work() — 延迟执行
alloc_workqueue() — 创建专用队列
cmwq 亮点: 自动管理 kthread 池，根据负载动态增减线程数。避免线程爆炸。
特点: 可睡眠！适合需要调用 kmalloc(GFP_KERNEL)、mutex_lock() 的延迟工作。

Tasklet

轻量级软中断 · 已逐渐被 workqueue 取代

在软中断上下文执行，不可睡眠。
DECLARE_TASKLET() 声明
tasklet_schedule() 调度
特性: 同一 tasklet 不会同时在多核运行（简化同步），但不同类型 tasklet 可并行。
衰减: 内核 5.15+ 开始将 tasklet 迁移到 BH workqueue — tasklet 正在慢慢退出历史舞台。

Softirq

内核预定义的延迟处理类型

比 tasklet 更底层 — 静态编译，只有 10 种 (HI, TIMER, NET_TX, NET_RX, BLOCK, IRQ_POLL, TASKLET, SCHED, HRTIMER, RCU)。
同一 softirq 可在多核同时运行 → 需自备同步。
网络栈核心: NET_RX softirq (NAPI) 是 Linux 高性能网络的基础。
用户不能自定义 softirq — 用 tasklet 或 workqueue 替代。

8.2 io_uring — 新一代异步 I/O

io_uring 架构

共享内存环形队列 — 零拷贝提交/完成

Submission Queue (SQ): 用户态写提交请求 → 无需 syscall (用 IORING_SETUP_SQPOLL)
Completion Queue (CQ): 内核写完成事件 → 用户态轮询读取
共享内存: SQ + CQ 位于内核-用户态共享的 mmap 区域 — 零拷贝！
Fixed Files/Buffers: 预注册 fd 和 buffer，消除每次 I/O 的 fget() 和 buffer 映射开销
对比 AIO: io_uring 支持 所有 I/O 类型 (不仅 O_DIRECT)，延迟低至 ~2µs (AIO 约 10µs)
对比 SPDK: 不需要用户态驱动，通用性强。

io_uring 使用示例

liburing — 用户态封装库
struct io_uring ring; io_uring_queue_init(256, &ring, 0); // 获取一个 SQE struct io_uring_sqe *sqe; sqe = io_uring_get_sqe(&ring); io_uring_prep_read(sqe, fd, buf, nbytes, 0); io_uring_sqe_set_data(sqe, user_data); // 提交 (可批量提交多个 SQE) io_uring_submit(&ring); // 等待完成 struct io_uring_cqe *cqe; io_uring_wait_cqe(&ring, &cqe); // 处理 cqe->res (返回值) 和 cqe->user_data io_uring_cqe_seen(&ring, cqe);

9. 资源竞争 — 死锁、优先级反转与惊群

ABBA 死锁 · 优先级反转 · 惊群 · lock contention 分析

9.1 死锁的四要素与三种经典模式

死锁四要素 (Coffman)

四者缺一不可破

1. 互斥 (Mutual Exclusion): 资源不能共享
2. 持有并等待 (Hold & Wait): 持锁 A 等锁 B
3. 不可剥夺 (No Preemption): 锁不能被强制拿走
4. 循环等待 (Circular Wait): A→B→C→A 循环依赖
预防: 破坏任一条件即可。Linux 内核通过 lockdep 在运行时检测循环等待——不等死锁发生就报警。

三种经典内核死锁模式

代码正确 ≠ 不死锁

类型 1 — ABBA 死锁:
线程 A: lock(L1) → lock(L2) vs 线程 B: lock(L2) → lock(L1)
预防: 定义全局锁序，始终按同一顺序获取。

类型 2 — 本核递归死锁:
CPU0: spin_lock(L) → … → spin_lock(L) (不小心又拿同一把)
预防: 代码审查 + lockdep 检测。

类型 3 — 中断上下文死锁:
进程锁 L → 中断到达同核 → 中断处理函数尝试锁 L
预防: 永远用 spin_lock_irqsave() 而非 spin_lock()。

9.2 优先级反转 — RT 系统的头号杀手

9.3 惊群效应 (Thundering Herd)

什么是惊群

多个等待者同时被唤醒 → 只有一个能拿到资源

经典场景: 多个进程/线程 epoll_wait() 在同一个 listening socket 上。新连接到达时所有等待者都被唤醒，但只有一个能 accept() 成功——其余白白做了上下文切换。
accept() 惊群: Linux 4.5+ 通过 SO_REUSEPORT + EPOLLEXCLUSIVE 解决。
futex 惊群: FUTEX_WAKE 用 FUTEX_WAITERS 标志优化——如果只有一个等待者，走 fast-path 避免不必要的唤醒。

内核中的惊群场景与解决

wake_up vs wake_up_one — 选择正确的 API
// 不好的写法 — 唤醒所有等待者 wake_up_all(&wq); // 惊群! // 通常更好的写法 — 只唤醒一个 wake_up(&wq); // 唤醒一个 (exclusive) // 但需要确保所有等待者行为一致 // 若资源池有 N 个可用资源: wake_up_nr(&wq, N); // 精确唤醒 N 个 // 现代解决方案: rw_semaphore + handoff // 写者优先传递锁给下一个等待者，避免随机唤醒

10. 调试与诊断工具

lockdep · lockstat · ftrace · perf lock · KCSAN · 实战排错流程

lockdep

Lock Dependency Validator — 死锁预防之王

原理: 运行时记录所有锁的获取顺序，构建锁依赖图。检测图中是否存在环路 (ABBA 死锁)。
输出: 一旦检测到潜在死锁，立即打印详细的锁获取历史和调用栈。
CONFIG_PROVE_LOCKING=y 开启。
限制: 仅检测曾被"交叉获取"的锁对 → 未走过的代码路径可能漏报；有性能开销 (~10-20%)；不能检测中断上下文死锁的全模式。

lockstat / perf lock

锁竞争度量 — 谁在等谁
# lockstat — 内核锁统计 echo 1 > /proc/sys/kernel/lock_stat cat /proc/lock_stat # 输出: 锁名 / 等待次数 / 等待总时间 / 持有者 # perf lock — 基于 perf event 的锁分析 perf lock record <command> perf lock report # 总览 perf lock contention # 竞争热图

KCSAN

Kernel Concurrency Sanitizer

数据竞争检测器 — 类似用户态的 ThreadSanitizer (TSAN)。
CONFIG_KCSAN=y
原理: 在关键内存访问点插入watchpoint。检测到没有合适同步保护的并发访问时报告。
特点: 低开销 (仅 ~1-3% CPU 开销 vs KASAN 的 ~2-5x)，可在生产环境采样。
KCSAN_REPORT_RACE_SIGNAL=y 可通过信号实时通知。

10.1 实战: 死锁排错流程

1. 重现
开启 lockdep (CONFIG_PROVE_LOCKING=y) 运行触发场景
2. 读 lockdep 报告
lockdep 打印潜在死锁环路：谁持有什么、等什么、在哪行代码
3. 确认锁序
确定问题锁对的"正确获取顺序"，找出违反顺序的代码路径
4. 修复方案
方案A: 统一锁序 · 方案B: 减少锁粒度 · 方案C: 改用 trylock+回退
5. 验证
重新运行，确认 lockdep 不再报警
6. 性能回归
用 perf lock contention 检查修复是否引入新的锁竞争热点
⚠️ 常见锁错误排查清单
1. spin_lock 内调用了可能睡眠的函数? → CONFIG_DEBUG_ATOMIC_SLEEP 开启，会在 dmesg 打印 "scheduling while atomic"
2. 中断上下文用了 spin_lock 而非 spin_lock_irqsave? → lockdep 会报告 "possible irq lock inversion dependency"
3. 释放了未持有的锁? → CONFIG_DEBUG_SPINLOCK 开启，释放时检查 lock->owner == current
4. 持锁时间过长? → perf lock contention 查看每个锁的平均/最大等待时间
5. 内存顺序问题导致的"看不见"bug? → KCSAN + CONFIG_WEAK_MEMORY_ORDERING=y (ARM/PowerPC)
