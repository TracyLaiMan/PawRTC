基于 Socket

Netlink — 内核级网络协议

AF_NETLINK 是专门为内核通信设计的协议族。支持双向消息、多播组、异步通知。路由、Netfilter、udev、SELinux、audit 全部基于 netlink。

基于文件系统

sysfs · procfs · debugfs

利用 VFS 将内核数据映射为文件。用户空间通过标准 read/write 交互。sysfs (设备模型) · procfs (进程/内核信息) · debugfs (调试)。

基于系统调用

ioctl · read/write · mmap · signal

通过设备文件 + 系统调用传递数据。ioctl 传递结构化控制命令 · mmap 实现零拷贝共享内存 · signal 实现轻量异步通知。

2. Netlink — 内核级 Socket 通信

AF_NETLINK 协议族 · 双向消息 · 多播组 · 异步通知 — 最通用的内核↔用户通信方式

Netlink 核心特性

AF_NETLINK — 为内核设计的 Socket

双向通信: 内核可向用户发消息，用户可向内核发消息。
组播 (Multicast): 内核向一个多播组发消息 → 所有加入该组的用户进程同时收到。这是 udev 工作的基础。
异步通知: 内核主动发起，无需用户轮询。
序列号 + 端口号: 每条消息有 seq/pid 字段，请求-应答配对。
标准协议族: NETLINK_ROUTE (路由), NETLINK_KOBJECT_UEVENT (udev), NETLINK_AUDIT (审计), NETLINK_SELINUX...

消息格式 — TLV 编码

nlmsghdr + nla_attr 属性

Netlink 消息由 nlmsghdr 头 + 若干 TLV 属性 组成。TLV = Type-Length-Value: 每个属性有 16-bit 类型、16-bit 长度、可变长度数据。
对齐: 每个属性 4 字节对齐。
嵌套: 属性可以包含子属性 (nla_nest_start / nla_nest_end)。
内核端 API: nla_put_u32() / nla_put_string() / nla_get_u32() 等宏简化构造与解析。

/* ===== Netlink 用户空间 — libnl 标准 API ===== */ #include <linux/netlink.h> // 1. 创建 netlink socket int fd = socket(AF_NETLINK, SOCK_RAW, NETLINK_GENERIC); // 2. 绑定本地地址 (pid = 自己的 PID, groups = 订阅的多播组) struct sockaddr_nl addr = { .nl_family = AF_NETLINK, .nl_pid = getpid(), .nl_groups = 1 // 多播组 bitmask }; bind(fd, (struct sockaddr*)&addr, sizeof(addr)); // 3. 构造消息 (nlmsghdr + TLV 属性) struct msghdr msg = {...}; // sendmsg/recvmsg 发送/接收 sendmsg(fd, &msg, 0); recvmsg(fd, &msg, 0); // 4. 或使用 libnl 高级库 struct nl_sock *sk = nl_socket_alloc(); nl_connect(sk, NETLINK_GENERIC); nl_socket_add_membership(sk, GROUP_ID); // 加入多播
/* ═══ 内核端 Netlink — 创建套接字并发送广播 ═══ */ // 1. 创建内核 netlink socket netlink_kernel_create() ├── net = &init_net // 网络命名空间 ├── unit = NETLINK_MYPROTO // 协议编号 (< MAX_LINKS=32) └── cfg = { // netlink_kernel_cfg 配置 .groups = 2, // 多播组数量 .input = my_recv_cb, // ← 处理用户→内核消息的回调 } // 2. 内核→用户 — 单播发送 nlmsg_unicast(sk, skb, pid); // 发给特定 PID 的进程 // 3. 内核→用户 — 多播/广播发送 (⭐ 最常用) nlmsg_multicast(sk, skb, 0, group_id, GFP_KERNEL); // 所有订阅了 group_id 的用户进程同时收到 // 4. 构造消息 — TLV 属性封装 struct sk_buff *skb = nlmsg_new(size, GFP_KERNEL); void *hdr = genlmsg_put(skb, 0, 0, &family, 0, CMD_XXX); nla_put_u32(skb, ATTR_VALUE, val); // 添加 u32 属性 nla_put_string(skb, ATTR_NAME, "hello"); // 添加字符串属性 void *nest = nla_nest_start(skb, ATTR_NESTED);// 嵌套属性开始 nla_nest_end(skb, nest); // 嵌套属性结束 genlmsg_end(skb, hdr); // 完成消息封装
标准 Netlink 协议用途谁在用
NETLINK_ROUTE (0)路由表、网络接口、IP 地址、邻居表iproute2 (ip/link/addr/route 命令)
NETLINK_KOBJECT_UEVENT (15)内核事件通知 (热插拔)udev / systemd-udevd
NETLINK_AUDIT (9)安全审计日志auditd
NETLINK_SELINUX (7)SELinux 策略通知setroubleshootd
NETLINK_NFLOG (12)Netfilter 日志 (代替 syslog)ulogd / nflog 消费者
NETLINK_GENERIC (16)通用协议 — 动态注册子协议族内核模块自定义协议
NETLINK_CONNECTOR (11)进程 fork/exit 事件通知procmon / 进程监控
Generic Netlink — 动态注册子协议
NETLINK_GENERIC (协议 16) 允许内核模块动态注册子协议族，无需在内核头文件中预留全局唯一的协议编号。通过 genl_register_family() 注册，用户空间通过 family name 字符串查找。这是写内核模块 Netlink 通信的推荐方式—避免协议编号冲突。

3. Uevent — 热插拔与设备事件通知

kobject_uevent → Netlink 广播 → udev 接收 → 设备节点创建/规则匹配/固件加载

uevent 是什么？

kobject 的事件广播

uevent = user event。内核设备模型中每当 kobject 状态变化 (device_add / device_del / driver 绑定 / 属性变更) 时，通过 kobject_uevent() 发送一个 KEY=VALUE 文本格式的事件通知到用户空间。底层走 Netlink NETLINK_KOBJECT_UEVENT 协议 广播。

udev 如何处理 uevent

systemd-udevd 的完整流程

① udevd 启动时监听 netlink socket
② 收到 uevent → 解析 ACTION/DEVPATH/SUBSYSTEM...
③ 读取 sysfs: 根据 DEVPATH 找到 /sys/devices/.../ 下的设备属性文件
④ 匹配规则: 遍历 /etc/udev/rules.d/*.rules，匹配 ACTION/SUBSYSTEM/ATTR{...} 条件
⑤ 执行动作: 创建/dev 节点、设权限、创建 symlink、加载固件、重命名网卡

uevent 消息示例

KEY=VALUE 文本格式

ACTION=add
DEVPATH=/devices/pci0000:00/0000:00:14.0/usb1/1-1
SUBSYSTEM=usb
PRODUCT=1234/5678/100
TYPE=0/0/0
SEQNUM=2048
MAJOR=189
MINOR=3
DEVNAME=bus/usb/001/004
每条 uevent ≤ 2048 字节 (UINT_MAX of buffer)

/* ===== 内核端触发 uevent ===== */ // 最简单的 uevent (发送到用户空间) kobject_uevent(&dev->kobj, KOBJ_ADD); // 设备添加 kobject_uevent(&dev->kobj, KOBJ_REMOVE); // 设备移除 kobject_uevent(&dev->kobj, KOBJ_CHANGE); // 状态变更 // 带环境变量的 uevent char *envp[] = { "MY_KEY=my_value", NULL }; kobject_uevent_env(&dev->kobj, KOBJ_ONLINE, envp); // device_add() 内部自动触发 KOBJ_ADD // device_del() 内部自动触发 KOBJ_REMOVE /* ===== 用户空间监听 uevent (shell) ===== */ # udevadm — 最便捷的方式 udevadm monitor // 实时打印所有 uevent udevadm monitor --kernel --property // 显示完整环境变量 # 输出示例: # KERNEL[1234.567] add /devices/platform/leds/leds/my_led (platform) # ACTION=add # DEVPATH=/devices/platform/leds/leds/my_led # SUBSYSTEM=leds # 或直接用 netlink 工具监听 udevadm trigger // 重放所有设备冷插拔事件
⚠️ uevent vs sysfs — 互补关系
uevent 只携带最基本的设备标识信息 (ACTION/DEVPATH/SUBSYSTEM...)，不携带设备的详细属性。udev 收到 uevent 后需要再去 sysfs 读取设备属性文件来完成规则匹配。两者是协作关系而非替代关系：netlink uevent = 事件通知通道，sysfs = 属性数据通道。

4. sysfs · procfs · debugfs — 虚拟文件系统三剑客

将内核数据映射为文件 — 用户空间通过标准 read/write 操作与内核交互

sysfs — 设备模型

/sys · kobject 一对一映射

设计哲学: 一个文件一个值 → machine-parsable。
创建: DEVICE_ATTR_RO/RW/WO() 宏 + sysfs_create_group()
show/store: 读时 show 回调生成文本输出，写时 store 解析文本输入。
ABI 稳定: 用户空间工具 (udev/lsusb/lspci) 依赖 sysfs 格式。
位置: /sys/class/ /sys/bus/ /sys/devices/ /sys/kernel/

procfs — 进程与内核信息

/proc · seq_file 接口

关键 API: proc_create() + seq_file (解决 >PAGE_SIZE 输出问题)。
seq_file: start→next→stop→show 迭代器模式，内核自动管理页缓冲区。
/proc/sys/: sysctl 参数 (kernel.hostname 等)。
新驱动不推荐 procfs — 应优先选择 sysfs (设备相关) 或 debugfs (调试)。

debugfs — 调试接口

/sys/kernel/debug · 非稳定ABI

极简 API: debugfs_create_u32/bool/file/dir() 一行创建。
无格式限制: 可 dump 任意数据，不受"一个文件一个值"约束。
无 ABI 保证: 接口可随时变化 — 不可用于生产。
可完全禁用: CONFIG_DEBUG_FS=n 生产内核不包含 debugfs。

/* ===== sysfs 属性创建模板 ===== */ static ssize_t my_show(struct device *dev, struct device_attribute *attr, char *buf) { return sysfs_emit(buf, "%d\n", my_val); // ★ 使用 sysfs_emit 代替 sprintf } static ssize_t my_store(struct device *dev, struct device_attribute *attr, const char *buf, size_t count) { kstrtoint(buf, 10, &my_val); // ★ 安全解析 return count; } static DEVICE_ATTR_RW(my); // 生成 dev_attr_my // probe 中创建: sysfs_create_group() 或 device_create_file() /* ===== debugfs 快速创建 ===== */ static struct dentry *my_debug_dir; my_debug_dir = debugfs_create_dir("mydrv", NULL); // 目录 debugfs_create_u32("counter", 0644, my_debug_dir, &my_cnt); // 读写 u32 debugfs_create_bool("enable", 0644, my_debug_dir, &my_en); // 读写 bool // remove 中: debugfs_remove_recursive(my_debug_dir);

5. ioctl · mmap · Signal — 设备驱动的三件武器

二进制控制命令 · 零拷贝共享内存 · 轻量异步通知 — 设备驱动的基础通信方式

ioctl — 设备控制命令

通过设备文件 fd 传递结构体

命令编码: _IO/_IOR/_IOW/_IOWR 宏将方向/魔数/序号/大小编码为 32-bit cmd。
内核实现: file_operations.unlocked_ioctl 回调，用 copy_from_user/copy_to_user 安全传输。
优点: 单入口点处理大量命令；传递任意结构体。
缺点: 需编译 C 程序调用、非脚本友好、ABI 需维护兼容性。

mmap — 零拷贝共享内存

内核内存直接映射到用户地址空间

内核实现: file_operations.mmap → remap_pfn_range() 映射物理页到用户 VMA。
零拷贝: 用户直接读写内核缓冲区，无需 copy_from/to_user → 极致带宽。
同步: 需要额外的通知机制 (poll/signal/netlink) 告知用户数据就绪。
安全: 必须确认映射范围不泄露敏感内核内存。

Signal (FASYNC/SIGIO)

1 bit 异步通知

FASYNC 机制: 用户 fcntl(fd, F_SETFL, FASYNC) → 内核驱动调用 kill_fasync() → 用户进程收到 SIGIO → 然后主动 read。信号本身不携带数据，只告诉用户"有新数据可读"。
适用: 串口数据到达、输入事件等 I/O 通知。不推荐作为复杂通信的唯一手段。

/* ===== ioctl 内核端模板 ===== */ static long my_ioctl(struct file *filp, unsigned int cmd, unsigned long arg) { struct my_data data; switch (cmd) { case MY_IOC_GET: // _IOR(MY_MAGIC, 1, struct my_data) /* 填充 data 结构体 */ if (copy_to_user((void __user *)arg, &data, sizeof(data))) return -EFAULT; break; case MY_IOC_SET: // _IOW(MY_MAGIC, 2, struct my_data) if (copy_from_user(&data, (void __user *)arg, sizeof(data))) return -EFAULT; /* 使用 data */ break; } return 0; } /* ===== mmap — 内核缓冲区映射到用户空间 ===== */ static int my_mmap(struct file *filp, struct vm_area_struct *vma) { unsigned long pfn = virt_to_phys(my_buf) >> PAGE_SHIFT; return remap_pfn_range(vma, vma->vm_start, pfn, vma_size(vma), vma->vm_page_prot); } /* ===== FASYNC — 异步 I/O 通知 ===== */ static struct fasync_struct *my_fasync; // 保存通知队列 static int my_fasync_cb(int fd, struct file *filp, int mode) { return fasync_helper(fd, filp, mode, &my_fasync); } // 数据就绪时 — 中断上下文或内核线程中调用: kill_fasync(&my_fasync, SIGIO, POLL_IN); // → 用户进程收到 SIGIO

6. relayfs — 海量内核追踪的高速通道

Per-CPU 无锁环形缓冲区 · mmap 零拷贝读取 · 适用于 kprobes / LTTng / 内核追踪

relay 通道架构

Per-CPU 环形缓冲 + 子缓冲区

每个 CPU 一个独立的环形缓冲区，划分为多个子缓冲区。内核写入时无锁 (每 CPU 独立写)，消费者通过 mmap() 零拷贝读取。
两种模式: No-overwrite (满则丢弃 — 适合调试) · Overwrite (飞行记录仪 — 保留最新数据)。
通知: 子缓冲写满时通过 poll() 通知用户空间。

创建与使用

挂载在 debugfs 上的通道

内核端: relay_open("mychan", parent_dir, subbuf_size, n_subbufs, &cb, NULL) → relay_write(chan, data, len)
用户端: 打开 /sys/kernel/debug/mychan → mmap() 映射环形缓冲 → 直接读数据，或 poll() 等待子缓冲就绪通知。
适用: kprobes 追踪、LTTng、内核模块高性能日志输出。吞吐量可达 GB/s 级别。
