1. 低内存系统的困境全景

嵌入式 IPC 相机通常只有 64MB-512MB DDR——而其中 CMA 要占掉 64-128MB 给 ISP/编码器，实际留给内核和应用的内存极其紧张。

困境 1: 物理内存绝对不足

总内存 256MB → 内核+驱动 ~40MB → CMA 128MB → 应用只剩 ~88MB

IPC 相机的内存分配极不均衡：ISP 需要大块物理连续内存用于视频帧 buffer（通常通过 CMA 预留 64-128MB），编码器输出 buffer 也需要 DMA 可访问的连续内存。扣除内核自身（~20-40MB），留给用户态应用（LinkricAI、LiveKit SDK、AI 推理）的内存可能只有 50-80MB。任何模块的微小内存泄漏都会在数小时内触发 OOM。

困境 2: 外部碎片 — "内存够但分配失败"

空闲总量 50MB 但无法分配 4 个连续页(16KB)

这是低内存系统最反直觉的现象：free 显示空闲 50MB，但驱动调用 dma_alloc_coherent() 分配 16KB 连续物理内存时返回 -ENOMEM。根因：内存被碎片化为大量 4KB 小页，没有 contiguous order-2(16KB) 或 order-3(32KB) 的连续块。页面虽空但无法合并——这就是 外部碎片（External Fragmentation）。

困境 3: OOM 在内存"用完"之前就触发

空闲 30MB，但 OOM Killer 弹出来杀了进程

这是内核设计决定的——不是 bug。当 free 内存降到水位线(min)以下且 kswapd 回收速度跟不上分配速度时，内核进入 Direct Reclaim——分配者自己被阻塞去做内存回收。如果回收失败（无可回收页或时间过长），分配失败。当高阶分配(order>0)失败时，即使空闲内存总量还很多，OOM Killer 也会被触发——因为内核判定"无法满足当前分配请求"。

256MB DDR 典型布局CMA 128MBISP 视频帧 DMA buffer — 不可被内核/page cache 使用
Page Cache 30MB文件缓存 (可回收)——低内存时首先被 kswapd 回收
Slab 25MB内核数据结构 (dentry/inode/kmalloc-xxx)——部分可回收
Anon Pages 35MB用户态匿名页 (堆/栈)——可 swap 但嵌入式通常无 swap
内核代码/数据 20MB不可回收——内核自身的 text/data/bss + 模块
空闲 (Unused) 18MB"真正空闲"——但严重碎片化，大多为 order-0(4KB) 小页

2. 水位线 (Watermark) — 内核的"内存预警"机制

内核不等到内存耗尽才行动——通过三级水位线实现渐进式的内存压力响应。

三级水位线模型

min → low → high — 每级触发不同行为

high：高于此线 = 内存充足。kswapd 睡眠，不做任何回收。
low：低于此线 = 内存偏紧。唤醒 kswapd 开始后台异步回收（不影响分配者的延迟）。kswapd 持续回收直到水位回到 high 以上。
min：低于此线 = 内存危急。分配者自身进入 Direct Reclaim——同步阻塞回收。分配调用被阻塞数十毫秒到数秒，等待回收完成。这是导致"系统突然变慢"的常见原因。
min 以下：所有回收手段（包括 OOM Killer）都可能被触发。对于高阶分配(order>0)，即使空闲总量在 min 以上也可能触发——因为水位线是按 order 独立计算的。

水位线计算公式

由 min_free_kbytes 推导，/proc/sys/vm/min_free_kbytes

watermark[min] = min_free_kbytes（直接由用户设置或内核根据内存总量自动计算）。
watermark[low] = min × 1.25
watermark[high] = min × 1.5
内核默认值：min_free_kbytes = sqrt(16 × total_ram_kb)（近似）。128MB 系统 → ~14MB min。512MB 系统 → ~28MB min。
嵌入式问题：14MB 对于 128MB 系统占比高达 11%——这 14MB 是紧急储备不可用于正常分配（仅 GFP_ATOMIC 或 PF_MEMALLOC 标志的分配可以动用）。实际可用内存 = total - min_free。

// 水位线状态转换 — 从正常到 OOM 的完整路径 空闲页数量 (从高到低) [========HIGH========] high 水位线以上 --------------------------------- kswapd 睡眠 · 内存充足 · 正常分配 ~~~~~~~~~~~~~~~~~~~~~~~~~ --------------------------------- [=====LOW=====] low 水位线以上 --------------------------------- kswapd 被唤醒 · 后台异步回收页缓存 ~~~~~~~~~~~~~~~~~~~~~~~~~ --------------------------------- [===MIN===] min 水位线以上 --------------------------------- 分配者进入 Direct Reclaim · 同步阻塞 ~~~~~~~~~~~~~~~~~~~~~~~~~ --------------------------------- [=MIN=] min 水位线以下 OOM Killer 被触发 · 选择进程杀掉 系统响应链: free > high -> 无动作，休眠 free < low -> kswapd 异步回收 (不阻塞分配者) free < min -> Direct Reclaim (阻塞分配者，同步回收) Direct Reclaim 反复失败 -> OOM Killer 选择并杀死进程 杀进程后释放内存 -> 水位回升 -> 系统继续运行

2.5 物理内存"连续"到底是什么？— 底层原理

前面讨论的 buddy、pageblock、迁移类型、fallback、窃取——全部是为了保住物理连续页的分配能力。不先理解"连续"在物理层面意味着什么，就不知道为什么需要这么复杂的机制。

◈ 物理内存 = 从 0 开始的一维数组

[Page 32767] └────────────────── 128MB DDR ──────────────────────────┘ // "物理连续" = 页框号(PFN)连续 = 物理地址连续 = 在数组中是邻居 物理连续 4 页 = PFN {100, 101, 102, 103} = 地址 0x80000000+100*4K 起始的 16KB 物理不连续 4 页 = PFN {100, 500, 3, 8192} = 四个分散的 4KB，地址互不相邻

◈ Buddy 分配器如何保证"连续"

Buddy 块 = 一组物理连续的页

free_area[order] 中的每个块：2^order 个 PFN 连续

Buddy 分配器的核心数据结构 free_area[order]——每个链表节点指向一个起始 PFN 对齐到 2^order 边界的连续物理页块：
· order-0: 1 页 (4KB)——单页，天然连续
· order-1: 2 页 (8KB)——PFN 必须是偶数
· order-2: 4 页 (16KB)——PFN 必须是 4 的倍数
· order-3: 8 页 (32KB)——PFN 必须是 8 的倍数
关键：PFN 对齐规则保证分裂/合并时 buddy 块总是物理相邻的。

分裂与合并 = 连续性的维护机制

分裂保持连续 · 合并恢复连续

分裂：分配 order-2(16KB) 块但只有 order-3(32KB) 空闲 → 将 32KB 从中点劈成两个 16KB 块 → 一块分配出去，一块放入 order-2 freelist。劈开的两半各自内部仍然物理连续——因为它们在原始块中就是连续的。
合并：释放 order-2 块 → 查它的 buddy（伴随 PFN）是否也空闲 → 是 → 合并为 order-3 → 递归向上。合并的条件是两个 buddy 必须恰好相邻且在同一个对齐边界内。

/* ═══ Buddy 块的分裂与合并 — 物理连续性的可视化 ═══ */ 假设 order-3 块 = 8 页连续 (PFN 0~7, 32KB) order-3 块: [PFN0│PFN1│PFN2│PFN3│PFN4│PFN5│PFN6│PFN7] ← 1 个块，物理连续 │ │ 分配 order-2 (16KB) → 从中点分裂 ▼ order-2 块A: [PFN0│PFN1│PFN2│PFN3] ← 分配出去 (物理连续 16KB) order-2 块B: [PFN4│PFN5│PFN6│PFN7] ← 放入 order-2 freelist (物理连续 16KB) │ │ 块A 释放 → buddy (块B) 也空闲? → 是 → 合并 ▼ order-3 块: [PFN0│PFN1│PFN2│PFN3│PFN4│PFN5│PFN6│PFN7] ← 恢复为一个连续块 /* ═══ 什么是 buddy？═══ */ PFN 0 的 order-1 buddy 是 PFN 2 (0 XOR 2^1 = 2) PFN 4 的 order-2 buddy 是 PFN 0 (4 XOR 2^2 = 0) PFN 8 的 order-3 buddy 是 PFN 0 (8 XOR 2^3 = 0) // 公式: buddy_pfn = pfn ^ (1 << order) // 只有 XOR 计算出的那个特定 PFN 才是"buddy"——不是任意相邻页

◈ 碎片如何破坏连续性：被"钉子页"阻断的合并链

/* ═══ 碎片化场景 — 有空闲但无法合并 ═══ */ 初始状态: order-3 块 = PFN 0~7 全部空闲 [○0│○1│○2│○3│○4│○5│○6│○7] ○ = 空闲 Step 1: 分配 PFN 0 (order-0, 不可移动的内核 slab) [●0│○1│○2│○3│○4│○5│○6│○7] ● = 已分配 (不可移动) Step 2: 分配 PFN 1~7 → 使用后释放 PFN 2~7 [●0│●1│○2│○3│○4│○5│○6│○7] PFN 0,1 被内核永久占用 (不可移动) Step 3: 尝试合并: PFN 2,3 → order-1 buddy → 可以合并 (2^1=2, XOR: 2^2=0? ◈ 为什么必须"物理连续"而不能用"虚拟连续"凑合？

需要物理连续的场景

DMA 引擎看不到虚拟地址

DMA 控制器直接访问物理总线地址——它不经过 MMU，不认识虚拟地址。如果你给 DMA 一个"虚拟连续但物理分散"的 buffer：
· DMA 只看到第一页的物理地址 → 读完 4KB 后 不知道下一页在哪 → 写到错误的内存 → 数据损坏
· 解决方案：scatter-gather DMA（用一个描述符表告诉 DMA 每一段的物理地址和长度）——但不是所有硬件都支持，且编程复杂。
旧式/廉价 DMA 引擎只接受一个物理起始地址 + 长度——必须是物理连续的。

可以用虚拟连续替代的场景

CPU 访问时 MMU 可以"骗"CPU

vmalloc：物理页分散在各处，但内核修改页表让它们在虚拟地址空间看起来连续。CPU 在访问时：虚拟地址 → MMU → 查页表 → 翻译到各个分散的物理页。
用户态 malloc：同样的原理——进程看到的都是虚拟地址，页表背后可以映射到任意物理页。
但代价：TLB miss 增加（4KB 页表条目多）→ 性能下降。且 DMA 无法使用。

◈ 从"连续"角度重新理解 pageblock 和迁移类型

/* ═══ pageblock 隔离 = 保护"可合并区域"不被钉子穿透 ═══ */ 没有 pageblock 隔离时 (早期 Linux): 内核 slab (UNMOVABLE) 散布在物理内存各处: [●][○][○][●][○][○][●][○]...[○][●][○][○] ... └钉┘ └钉┘ └钉┘ └钉┘ 每个钉子 (UNMOVABLE 页) 阻断一条合并链 → 无法形成大的连续块 有 pageblock 隔离后: Pageblock#0 (UNMOVABLE): [●][●][●][●][●][●][●][●]... ← 钉子集中在这里 Pageblock#1 (MOVABLE): [○][○][○][○][○][○][○][○]... ← 干净，可合并 Pageblock#2 (MOVABLE): [○][○][○][○][○][○][○][○]... ← 干净，可合并 ... → UNMOVABLE 集中到少数 pageblock → 其余 pageblock 无钉子 → 可以合并出大块 → 这就是 pageblock 存在的全部意义：把钉子关进笼子，保护其余区域的连续性 窃取 (steal) 是钉子突破笼子的过程: Pageblock#0 (UNMOVABLE) 满了 → 钉子开始渗透到 Pageblock#1 (MOVABLE) → Pageblock#1 被窃取 → 变为 UNMOVABLE → 钉子又多了一个笼子 → 当足够多笼子被污染 → 干净的 MOVABLE 区域不够 → 无法满足高阶分配 → OOM
🎯 物理内存"连续"的本质
· 物理连续 = 页框号(PFN)连续 → buddy 块 = 一组 PFN 连续的页
· Buddy 的分裂/合并 = 基于 XOR 的相邻性检查——只有"buddy"才能合并
· 碎片化 = 一个 UNMOVABLE 页（钉子）卡在 buddy 链中间 → 阻断两侧的空闲页合并 → 无法形成高阶连续块
· pageblock 隔离 = 把钉子限制在少数 pageblock 中，保护其余区域的合并能力
· 窃取 = 钉子突破 pageblock 边界的过程——每窃取一个 pageblock，就多一个区域失去合并能力
· 物理连续的必要性 = DMA 引擎不经过 MMU，必须物理连续才能用单个地址+长度描述整个 buffer

3. 内存碎片 — 为什么空闲 30MB 却分配失败？

这是嵌入式系统内存问题中最容易被误判的场景——free 显示大量空闲，但应用或驱动分配失败。

外部碎片 — Buddy 系统的结构性缺陷

大量 order-0 小页散落各地 → 无法合并 → 无法满足高阶分配

Buddy 分配器将物理内存按2^order 个页组织（order-0=4KB, order-1=8KB, order-3=32KB, order-9=2MB）。分配时找最小能满足的 order，释放时尝试与相邻的"buddy"页合并升级到更高 order。
碎片化场景：100 次 order-0 分配散布在内存各处 → 释放其中 80 次 → 剩下 20 个孤立页（每个被夹在两个已分配页之间）→ 这些 20 个页的 buddy 都还处于"已分配"状态 → 无法合并 → 永远停留在 order-0 → 当有人请求 order-2(16KB, 4个连续页)时 → 有 20×4KB=80KB 空闲但无法满足。

诊断：/proc/buddyinfo — 碎片化的 X 光片

每个 order 的空闲块数量 — 一眼看出碎片化程度

cat /proc/buddyinfo 输出每行代表一个内存 zone，每列代表一个 order（0~10）。
健康状态：Node 0, zone Normal ... 120 85 40 20 8 4 2 1 1 0 0——各 order 都有空闲块，呈递减分布。
严重碎片化：... 5000 3 0 0 0 0 0 0 0 0 0——order-0 有 5000 个空闲页(20MB)，但order-2 及以上全是 0——无法满足任何 ≥16KB 的连续分配请求。这就是"free 20MB 但 -ENOMEM"的直接证据。

内存压缩 (Compaction) — 内核的碎片整理

/proc/sys/vm/compact_memory — 手动触发

内核的 Memory Compaction 机制将可移动页（Page Cache、匿名页）迁移到连续区域，腾出连续的大块物理内存。类似磁盘碎片整理——但操作对象是物理页。
触发方式：① kcompactd 守护线程在后台自动运行（当某个 order 的空闲块太少时）② echo 1 > /proc/sys/vm/compact_memory 手动强制全量压缩。
限制：不可移动页（内核代码、kmalloc 分配、DMA coherent buffer）无法被迁移——它们是碎片化的固定锚点。嵌入式系统大量使用 dma_alloc_coherent → 产生大量不可移动页 → Compaction 效果有限。

CMA — 为 DMA 预留的连续内存池

Contiguous Memory Allocator — IPC 相机的标配

CMA 在启动时预留一大块物理连续内存——在不需要 DMA 时，CMA 区域可以被可移动页（Page Cache、用户态匿名页）"借用"。当驱动需要 DMA 连续内存时 → CMA 将借用者迁移走 → 归还连续区域 → 分配给驱动。
CMA 是碎片化问题的终极方案——它为 DMA 需求提供了"无论如何都能满足"的最后保障。没有 CMA：碎片化后分配失败 → 驱动报错 → 录像中断。有 CMA：碎片化后从 CMA 区域分配 → 成功。
内核参数：cma=128M（预留 128MB）或设备树中 linux,cma 属性。

⚠ CMA 的代价 —— 了解后才知道权衡
CMA 区域虽然可以被 Page Cache/匿名页"借用"，但借用的页必须可迁移——任何不可移动的分配（如 kmalloc 的 slab 页）无法进入 CMA。如果系统做大量不可移动的小分配 → CMA 区域实际上大部分空闲但不可用 → 形成"虚假的空闲内存"——free 显示的 128MB CMA 实际上只有 Page Cache 能临时使用，DMA 之外的普通分配用不了。同时 CMA 减少了普通 Zone 的大小——加剧了碎片化问题在剩余 Zone 中的严重程度。

4. 为什么内存没"用完"也 OOM？

这是低内存系统开发中困惑最多的问题——free 还显示 20-30MB，但内核已经触发 OOM Killer 杀进程。

原因 1: 高阶分配失败

最隐蔽 — 总量够但不连续

内核驱动通过 __get_free_pages(GFP_KERNEL, order=3) 请求 32KB 连续物理内存。Buddy 查看 order-3 空闲块 = 0（碎片化导致）。尽管 order-0 总计有 5000 个空闲页(20MB)，但无法合并成 order-3 → 分配失败。在 GFP_KERNEL 标志下会触发 Direct Reclaim——回收失败 → OOM Killer 被唤醒。即使 free 显示 20MB。

原因 2: 水位线阻止了分配

free 在 min 水位线以下 → 即使是 order-0 也拒绝

正常 GFP_KERNEL 分配需要 free > watermark[low]。如果当前空闲刚好在 low 和 min 之间 → kswapd 已被唤醒但尚未回收足够内存 → 分配被阻塞在 Direct Reclaim。如果 Direct Reclaim 也回收不到足够内存（Page Cache 全被刷出、匿名页无 swap、slab 不可回收）→ OOM。
关键：OOM 触发时 free 可能恰好等于 min 水位线(比如 14MB)——不是 0MB。

原因 3: GFP_ATOMIC 的严格限制

中断上下文 — 不能回收不能睡眠 → 直接失败

中断 handler 中用 GFP_ATOMIC 分配——这种分配不允许 Direct Reclaim（不能睡眠）、不允许 OOM Killer（不能阻塞）。只能使用 free 在 watermark[min] 以下的紧急储备。如果紧急储备耗尽 → GFP_ATOMIC 分配静默失败返回 NULL → 驱动可能未检查 NULL → 空指针解引用 → kernel panic。这是低内存系统中最致命的 crash 类型之一。

5. OOM Killer 机制与调优

OOM Killer 是内核的最后防线——理解它是如何选择牺牲者的，才能保护关键进程不被误杀。

OOM Score 的计算公式

oom_score = (进程内存占比) × (root 惩罚/奖励) × oom_score_adj

OOM Killer 为每个进程计算一个 badness score（坏度分数）——分数越高越先被杀：
badness = (total_vm / total_system_ram) × 1000（基础：内存占比越大越该死）
+ 额外惩罚：fork 了大量子进程（消耗 PID 资源）、运行时间长（累积内存泄漏概率高）
- 减免：进程被标记为 oom_score_adj = -1000（完全豁免）、CAP_SYS_ADMIN 权限进程有微小减免
最终决策：选择 badness 最高且可以杀（不是 init、不是内核线程、不是被豁免）的进程。

实战：保护关键进程不被 OOM Kill

/proc/PID/oom_score_adj — 范围 -1000(完全豁免) ~ +1000(优先牺牲)

完全豁免：echo -1000 > /proc/$(pidof LinkricAI)/oom_score_adj——这个进程永远不会被 OOM Killer 选中。
调低优先级：echo -500 > /proc/PID/oom_score_adj——降低被杀概率。
主动牺牲：echo 1000 > /proc/PID/oom_score_adj——标记为"内存低时请优先杀我"（用于非关键模块如 AI 检测、日志系统）。
systemd 配置：OOMScoreAdjust=-1000 在 .service 文件中永久生效。
查看当前分数：cat /proc/PID/oom_score——值越低越安全。

⚠ OOM Killer 的三个常见误解
误解 1："OOM Killer 是 bug，内核不该杀进程。" 事实：OOM Killer 是理性的最后选择——不杀进程，系统会完全卡死（所有内存分配被阻塞→所有进程挂起）——这比杀一个进程糟糕得多。OOM Killer 是系统自保机制。
误解 2："关闭 OOM Killer（echo 2 > /proc/sys/vm/panic_on_oom）就不会有内存问题。" 事实：关闭 OOM Killer 后，内存耗尽时分配者被永久阻塞无法返回——内核不会自动恢复。系统表现为"假死"——无响应、无日志、无修复手段。这比 OOM Kill 更糟糕。
误解 3："增加 swap 就不会 OOM。" 事实：swap 增加虚拟内存总量，但不解决高阶分配失败的问题——因为 swap 只换出匿名页(4KB)，不能解决 order-3 连续物理内存的需求。交换出的页面回来时仍然散落在碎片化的物理地址空间中。

7. IPC 相机内存实战 — 从 panic 到稳定

以下案例来自真实 IPC 产品调试——从内存相关问题到解决方案的完整路径。

案例 1: 夜间 OOM 总是凌晨 3 点触发

根因: 定时任务触发 SD 卡文件扫描 → 大量 dentry cache → Page Cache 膨胀

用户反馈每天凌晨 3 点 IPC 自动重启。排查发现：cron 定时任务在凌晨 3 点执行 find /data/record/ -mtime +7 -delete（清理 7 天前录像）。find 遍历数千个 MP4 文件 → 每个文件在内核中创建 dentry + inode cache → slab 增长 20MB → free 内存从 35MB 骤降到 5MB 以下 → 低于 min 水位线 → 编码器下次 dma_alloc 触发 Direct Reclaim → 回收不够 → OOM Kill。解决：① 清理脚本优化为仅删除文件不递归 stat（find ... -delete 仍会 stat 每个文件）→ 改为按目录删除 ② vfs_cache_pressure 提高到 500→dentry 缓存被更快回收 ③ 清理完成后 echo 2 > /proc/sys/vm/drop_caches 释放 slab。

案例 2: 运行 48 小时后必然 kernel panic

根因: ISP 驱动的 GFP_ATOMIC 分配 + 碎片化 = 返回 NULL = 未检查 = panic

IPC 运行 48 小时后出现 Unable to handle kernel NULL pointer dereference panic。排查：① 分析 panic 日志——崩溃点在 ISP 驱动中断 handler 中 ② 代码审查——中断中调用 dma_alloc_coherent() 但未检查返回值 NULL ③ 为什么分配会失败：运行 48 小时内存严重碎片化——GFP_ATOMIC 无法做 Direct Reclaim → 紧急储备耗尽 → 返回 NULL。
解决：① 驱动改为启动时预分配 buffer，中断中只用预分配的 buffer——消除运行时分配 ② 添加 NULL 检查并记录错误日志 ③ CMA 增大到 128MB 确保 ISP buffer 永远有连续内存可用。

案例 3: 概率性 OOM — CONFIG_FORCE_MAX_ZONEORDER 从 16 降到 10

根因: 巨型 pageblock 被单个不可移动页"污染" → 整个块变为不可移动 → 可移动分配无内存可用

问题现象：系统概率性OOM——几小时到几天随机触发。dmesg 显示 free 还有 20-30MB 但 OOM Killer 被激活。
排查过程：① cat /proc/pagetypeinfo → Movable 类型的空闲页极少，但 Unmovable 和 Reclaimable 类型有大量空闲页 ② 进一步发现：大量 order-9/10 的 pageblock 被标记为 Unmovable——因为其中混入了少量不可移动页（如内核 slab 对象、DMA coherent buffer）③ 用户态 malloc 分配的是 Movable 类型 → Buddy 优先从 Movable 类型的空闲列表查找 → Movable 列表为空 → 触发 fallback：按优先级先借 RECLAIMABLE、再借 UNMOVABLE。但此时 UNMOVABLE pageblock 中的空闲页大部分已被内核 slab 和 DMA buffer 占满——即使能跨类型借，UNMOVABLE 池中的连续空闲块也不够应用的高阶需求 → Direct Reclaim 失败 → OOM。

Pageblock 迁移类型污染机制

单个不可移动页 → 污染整个 pageblock → 该 block 所有空闲页对 Movable 分配不可见

Linux 的 pageblock 规则：内核将物理内存按固定大小的 pageblock（通常是 MAX_ORDER-1 阶，即 2^(MAX_ORDER-1) 页）分组，每个 pageblock 有一个迁移类型（MIGRATE_UNMOVABLE / RECLAIMABLE / MOVABLE / CMA）。关键规则：如果一个 pageblock 被标记为 Unmovable，其内部所有空闲页都在 UNMOVABLE freelist 中。Buddy 优先从请求类型的 freelist 分配——Movable 请求先在 Movable freelist 中查找。但 Movable 并非"完全不能"用 Unmovable 的页：fallback 数组（mm/page_alloc.c:1482）允许 Movable 依次从 RECLAIMABLE → UNMOVABLE 借页。然而 UNMOVABLE 页借给 Movable 后——由于 can_steal_fallback() 对 Movable 类型的严格限制（仅 order ≥ pageblock_order/2 才触发窃取）——借来的页不会被永久转换为 Movable，释放后仍然回到 UNMOVABLE freelist。结果：Movable 可以"临时借用"但不能永久占有 UNMOVABLE 的空闲页——在高频分配场景下这个临时借用的吞吐量远不足以满足稳定需求。
order=16 的放大效应：pageblock 大小 = 2^15×4KB = 128MB。一个 4KB 的 kmalloc 分配落在某个 128MB pageblock 中 → 该 pageblock 被永久标记为 Unmovable → 即使用户态释放了 100MB 内存，如果这 100MB 都在这个被污染的 128MB pageblock 中 → 它们对新的 Movable 分配不可见 → Buddy 从 Movable 视角看是"内存耗尽" → OOM。

!

核心原理：为什么 order=10 能解决而 order=16 不能

pageblock = 2^(MAX_ORDER-1) 页。order=16 → pageblock=128MB。order=10 → pageblock=2MB。一个不可移动页的污染半径 = 一个 pageblock 的大小。

order=16: pageblock = 128MB

最大连续块 256MB · 污染半径 128MB · 256MB DDR 仅 2 个 pageblock

一个 4KB 的 kmalloc 不可移动页落在 Pageblock#0 中 → 整个 128MB 被标记为 Unmovable → 即使用户态释放了 100MB 内存在这个 pageblock 中 → 这些空闲页对 Movable 分配完全不可见。
仅剩 Pageblock#1 (~128MB) 可用于 Movable 分配 → 扣除 CMA 64MB + 内核 ~20MB → 实际留给用户态和 Page Cache 的 Movable 内存不到 40MB → 应用启动 + Page Cache 正常波动 → 随时触发 OOM。

order=10: pageblock = 2MB

最大连续块 4MB · 污染半径 2MB · 256MB DDR 有 128 个 pageblock

同样一个不可移动页只污染 2MB → 剩余的 127 个 pageblock (254MB) 保持 Movable 类型 → 用户态 malloc 和 Page Cache 有充足的 Movable 空闲页。
即使多个 pageblock 陆续被污染（内核运行中正常的 slab 分配散布各处），每个仅损失 2MB → 累计损失可控。内核的 fallback 机制也可在 Movable 紧张时临时从 Unmovable 列表借用——pageblock 越小，fallback 触发频率越低、compaction 迁移成功率越高。

⚠ CONFIG_FORCE_MAX_ZONEORDER 的真正影响 — pageblock 大小决定污染半径
核心公式：pageblock_size = 2^(MAX_ORDER - 1) × PAGE_SIZE。MAX_ORDER 越大 → pageblock 越大 → 单个不可移动页的"污染半径"越大 → 可移动内存的浪费越严重。
推荐值：嵌入式 IPC → 10 (pageblock=2MB, 最大连续分配=4MB)；桌面/服务器 → 11 (pageblock=4MB, 默认)；大内存服务器 → 13 (pageblock=16MB)。128MB DDR 用 order=16 → 只有 2 个 pageblock → 一个被污染就损失一半内存。
设置原则：2^(MAX_ORDER-1) × 4KB 应 ≪ 总内存 / 16（至少 16 个 pageblock）。256MB 系统 → pageblock 应 ≤ 16MB → MAX_ORDER ≤ 13。
验证方法：cat /proc/pagetypeinfo 查看各迁移类型的空闲页分布。Movable 类型的空闲页总量才是用户态真正可用的内存——比 free 更准确。

分支 A：大块直接窃取

current_order >= pageblock_order

当 fallback 找到的空闲块本身 ≥ pageblock 时（如 order-10 的 4MB 块在 pageblock=2MB 中），change_pageblock_range()（line 1574-1583）遍历覆盖的所有 pageblock，逐个调用 set_pageblock_migratetype()。立即永久改变。

分支 B + C：空闲过半才永久窃取

pages >= 2^(pageblock_order-1)

move_freepages_block()（line 1553-1572）以 pageblock 边界对齐后，将所有空闲页从原类型移到新类型。如果移动的页数 ≥ pageblock 的一半（如 2MB/512页 → ≥256页），调用 set_pageblock_migratetype() 永久改变pageblock 类型。

无分支：空闲不够一半 → 类型不变

pages < 2^(pageblock_order-1)

空闲页被移到新类型 freelist，但 pageblock 的类型标签不变。这就是 为什么污染不是瞬间的——在 pageblock 空闲率降到 50% 之前，窃取不会变成永久。

◈ 三阶段时间线：从启动到 OOM — 基于源码的完整推演

/* ═══ 128MB 系统 MAX_ORDER=10 (pageblock=2MB, 共 64 个 pageblock) ═══ */ ─── 阶段 0：启动时 — 所有 pageblock 初始为 MOVABLE (memmap_init_zone line 4566) ─── set_pageblock_migratetype(page, MIGRATE_MOVABLE); // 全部 64 个！ /* 设计意图 (源码注释 line 4550-4556): * "Mark the block movable so that blocks are reserved for movable at startup. * This will force kernel allocations to reserve their blocks rather than * leaking throughout the address space during boot" * — 全部初始为 MOVABLE，迫使内核分配时通过 fallback/steal "认领" pageblock */ ─── 阶段 1：内核启动分配 — 通过 fallback/steal 动态创建 UNMOVABLE pageblock ─── 内核初始化 (slab 创建、页表分配、驱动 probe): 每次 kmalloc → __rmqueue_smallest(UNMOVABLE) 失败 (无 UNMOVABLE pageblock) → __rmqueue_fallback() → fallbacks[UNMOVABLE] = {RECLAIMABLE, MOVABLE} → can_steal_fallback(UNMOVABLE) = true (始终) → steal_suitable_fallback() → move_freepages_block + set_pageblock_migratetype → pageblock#0 被窃取为 UNMOVABLE 依次窃取 ~8 个 MOVABLE pageblock → 变为 UNMOVABLE 内核自己通过 steal 机制"圈出"了 8 个 UNMOVABLE pageblock (16MB) ─── 阶段 2：运行时稳态 — 剩余 ~56 个 MOVABLE pageblock 服务用户态 ─── 用户态 malloc → MOVABLE freelist 命中 → 正常分配 内核 slab → UNMOVABLE freelist 命中 → 正常分配 MOVABLE ↔ UNMOVABLE 双向 fallback 均可工作（但窃取不对等） ─── 阶段 3：运行时退化 — UNMOVABLE 窃取更多 pageblock ─── 内核 slab 持续增长 → 8 个 UNMOVABLE pageblock 耗尽 → fallback 开始触及剩余的 MOVABLE pageblock → UNMOVABLE 的 can_steal 始终为 true → 逐页借用但不一定触发永久窃取 → 取决于目标 pageblock 的空闲率是否 >= 50% → 每个被永久窃取的 pageblock = 2MB MOVABLE 永久损失 经过数天/数周，~25 个 pageblock 被窃取 → MOVABLE 容量从 112MB 降到 62MB ─── 阶段 4：OOM — MOVABLE 池不足 ─── 应用 malloc 峰值 > MOVABLE 可用量 → MOVABLE freelist 空 → fallback 到 RECLAIMABLE → 也空 → fallback 到 UNMOVABLE → 能借到页但 can_steal=false (MOVABLE 请求者) → 借到的页释放后回到 UNMOVABLE (free_one_page 读 pageblock 类型) → 无法永久扩大 MOVABLE 池 → Direct Reclaim 也无法增加 MOVABLE 页 → OOM

🎯 基于源码的回答
Q: 一个 pageblock 只有一个迁移类型（你理解正确），为什么不是第一个 kmalloc 就让 128MB 全变 UNMOVABLE？
A: 对于多 pageblock 系统（MAX_ORDER=10，64 pageblocks）：
① 启动时 所有 pageblock 均为 MOVABLE（memmap_init_zone() line 4566）
② 内核初始化时通过 fallback/steal 动态认领若干 pageblock 变为 UNMOVABLE——仅接触需要的那几个
③ 剩余大量 MOVABLE pageblock 完全不受影响 → 存储用户态 MOVABLE 分配
④ steal_suitable_fallback() 的分支 C（空闲过半才永久窃取）进一步延缓了扩散

对于单 pageblock 系统（MAX_ORDER=16，1 pageblock）：
① 唯一的 pageblock 初始为 MOVABLE
② 第一个 kmalloc → fallback → steal → 整个 128MB 变为 UNMOVABLE
③ 此后用户态 MOVABLE 分配通过 fallback 从 UNMOVABLE 借用——但这只是"借用"，不触发窃取
④ 系统在 fallback 模式下苟活，存活时间取决于借用速率能否覆盖需求

Q: pageblock 多为什么"延寿"？
A: 多 pageblock = 空间隔离：内核只需从 64 个中偷 8 个用于 UNMOVABLE，其余 56 个保持 MOVABLE 毫发无伤。而 1 pageblock 时内核启动直接偷走唯一的一个 → 用户态从"拥有自己的领地"变成"只能向 UNMOVABLE 借地"。
📐 存活时间对比
多 pageblock 系统：内核启动 steal 少量 pageblock → 剩余 MOVABLE 直接服务用户态 → 后续退化缓慢
单 pageblock 系统：内核启动 steal 唯一的 pageblock → 全部变为 UNMOVABLE → 用户态从一开始就在 fallback 借用 → 每次借用不带窃取 → 释放后页回到 UNMOVABLE → MOVABLE freelist 始终接近空 → 一个 order > pageblock_order/2 的 MOVABLE 分配就会触发 OOM
本质区别：多 pageblock = MOVABLE 有自己的领地；单 pageblock = MOVABLE 从第一天起就是 UNMOVABLE 领地上的"二等公民"

7.6 实战根因 — MAX_ORDER=15 + GFP_ATOMIC 预留 = 致命组合

真实案例：128MB 系统 MAX_ORDER=15（pageblock=64MB，2 个 pageblock），无 CMA。系统空闲 UNMOVABLE 7-8MB，但 order-0 分配触发 OOM。降低 MAX_ORDER 后问题消失。

/* ═══ mm/page_alloc.c:1686-1712 — reserve_highatomic_pageblock() ═══ */ static void reserve_highatomic_pageblock(struct page *page, struct zone *zone, unsigned int alloc_order) { int mt; unsigned long max_managed, flags; // 上限 = 总内存的 1% + 一个 pageblock max_managed = (zone->managed_pages / 100) + pageblock_nr_pages; // MAX_ORDER=15: max_managed = 327 + 16384 ≈ 65MB // MAX_ORDER=10: max_managed = 327 + 512 ≈ 2.3MB if (zone->nr_reserved_highatomic >= max_managed) return; spin_lock_irqsave(&zone->lock, flags); if (zone->nr_reserved_highatomic >= max_managed) goto out_unlock; mt = get_pageblock_migratetype(page); if (mt != MIGRATE_HIGHATOMIC && !is_migrate_isolate(mt) && !is_migrate_cma(mt)) { zone->nr_reserved_highatomic += pageblock_nr_pages; // MAX_ORDER=15: += 16384 页 (64MB!) ← 一次吃掉半个系统 // MAX_ORDER=10: += 512 页 (2MB) ← 一次吃掉 2% set_pageblock_migratetype(page, MIGRATE_HIGHATOMIC); move_freepages_block(zone, page, MIGRATE_HIGHATOMIC); } } /* ═══ 触发条件 — mm/page_alloc.c:2644 ═══ */ // get_page_from_freelist() 中，分配成功后: if (unlikely(order && (alloc_flags & ALLOC_HARDER))) reserve_highatomic_pageblock(page, zone, order); // 条件 1: order > 0 — 分配 ≥ 8KB (不是 4KB 单页) // 条件 2: ALLOC_HARDER — 仅 GFP_ATOMIC 设置 // 两个条件同时满足 → 一次成功分配触发整个 pageblock 被预留

🎯 根因总结
不是"order-0 分配不到"——是你看到的 7-8MB UNMOVABLE 空闲在 __zone_watermark_ok 中被 nr_reserved_highatomic = 64MB 覆盖掉了。对于非 GFP_ATOMIC 分配者，这些页如同不存在。
不是"UNMOVABLE 不能分配"——是 unreserve_highatomic_pageblock 的"至少保留一个 pageblock"保护机制，在 pageblock=64MB 时等同于"永远不释放"。
不是"迁移类型污染"——是 nr_reserved_highatomic 的粒度等于 pageblock 大小。大 pageblock 让这个保护机制变成了毒药。
降低 MAX_ORDER 修复的根因：pageblock 变小 → 预留粒度变小 → unreserve 阈值变小 → watermark 扣除量变小 → 系统恢复正常。
📐 默认 min_free_kbytes 计算约 1.4MB (128M DDR)
公式：min_free_kbytes ≈ sqrt(16 * total_ram_kb) → 128MB 系统 ≈ 1448 KB ≈ 1.4MB
watermark[LOW] ≈ 1.8MB, watermark[HIGH] ≈ 2.1MB
7-8MB 空闲正常情况下远超水位线——只有 nr_reserved_highatomic=64MB 才能解释 OOM
