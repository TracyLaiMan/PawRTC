Pipe & Filter 框架 vs FFmpeg
工具集 / 库集合

GStreamer — 媒体框架

Elements · Pads · Caps · Bus

管道-过滤器架构。每个 Element 是一个处理单元，通过 Pad 连接形成 Pipeline。框架自动管理线程、时钟同步、QoS、背压。适合动态实时流媒体和交互式应用。

FFmpeg — 工具集

libavcodec · libavformat · libavfilter

7 个独立库组成的多媒体瑞士军刀。400+ 编解码器、所有主流容器格式。开发者手动编排数据流。适合批处理转码、CLI 脚本、嵌入式场景。

互补共存

gst-libav 桥接

GStreamer 通过 gst-libav 插件包装 FFmpeg 的全部编解码器，在 GStreamer Pipeline 中使用 FFmpeg 的编码/解码能力。实际项目中两者常组合使用：GStreamer 做实时流，FFmpeg 做录制/归档。

1. GStreamer — Pipeline-Filter 架构

Elements · Pads · Caps 协商 · Bins/Pipeline · Bus 异步消息 · Buffer 管理 · 状态机

Element — 处理单元

Source · Filter · Codec · Sink

GStreamer 的最小功能单元。Source (数据源: filesrc/v4l2src) → Filter (处理: videoconvert) → Encoder/Decoder (编解码: x264enc/avdec_h264) → Sink (输出: autovideosink/filesink)。
每个 Element 有 Pad (接头) — Src Pad 输出数据，Sink Pad 接收数据。

Caps 协商 — 格式协商

GstCaps = MIME + 属性键值对

连接两个 Element 时，通过 Caps Negotiation 协商双方都支持的格式:
① Pad Template 检查 → ② Caps Query (交集计算) → ③ 上游选定固定格式 → ④ CAPS Event 推向下游
Caps 字符串示例: video/x-raw,format=I420,width=1920,height=1080,framerate=30/1

State — 元素状态机

NULL → READY → PAUSED → PLAYING

NULL: 未初始化 · READY: 资源已分配
PAUSED: 就绪但不处理数据 (preroll 完成)
PLAYING: 数据流动中
状态切换由父 Bin 协调 — 上转: 父→子；下转: 子→父。
异步元素 (网络源) 返回 GST_STATE_CHANGE_ASYNC。

Bus — 异步消息通知

Pipeline → Application 的事件通道

ERROR: 致命错误 (gerror + debug 字符串)
EOS: 流结束 · STATE_CHANGED: 状态变化
BUFFERING: 缓冲百分比 (0~100)
STREAM_START: 新流开始
ASYNC_DONE: 异步状态切换完成
TAG: 元数据标签
QOS: 服务质量事件 (丢帧/降质通知)

Buffer — 数据传输单元

GstBuffer + GstMemory (零拷贝)

PTS/DTS: 显示/解码时间戳 · Duration: 持续时长
Flags: DISCONT/DELTA_UNIT/MARKER/CORRUPTED/DROPPABLE
零拷贝: GstMemory 引用计数 + gst_memory_share() 轻量共享
BufferPool: 预分配+回收 — 避免频繁 alloc/free
DMA-Buf: 跨硬件加速器零拷贝
写时复制: refcount>1 时不可变，gst_buffer_make_writable() 创建副本

/* ===== GStreamer: C API — 构建 Pipeline ===== */ #include <gst/gst.h> int main(int argc, char *argv[]) { gst_init(&argc, &argv); /* 1. 方法一: 字符串描述 Pipeline (最简单) */ GstElement *pipe = gst_parse_launch( "filesrc location=input.mp4 ! qtdemux ! h264parse ! " "avdec_h264 ! videoconvert ! autovideosink", NULL); /* 2. 方法二: 手动构建 Pipeline (精细控制) */ GstElement *src = gst_element_factory_make("filesrc", "src"); GstElement *demux = gst_element_factory_make("qtdemux", "demux"); GstElement *sink = gst_element_factory_make("autovideosink", "sink"); gst_bin_add_many(GST_BIN(pipe), src, demux, sink, NULL); gst_element_link(src, demux); /* 3. 监听 Bus 消息 */ GstBus *bus = gst_element_get_bus(pipe); GstMessage *msg = gst_bus_timed_pop_filtered(bus, GST_CLOCK_TIME_NONE, GST_MESSAGE_ERROR | GST_MESSAGE_EOS); if (GST_MESSAGE_TYPE(msg) == GST_MESSAGE_ERROR) { GError *err; gchar *dbg; gst_message_parse_error(msg, &err, &dbg); g_error("GStreamer error: %s (%s)", err->message, dbg); } /* 4. 控制 Pipeline 状态 */ gst_element_set_state(pipe, GST_STATE_PLAYING); // 开始播放 gst_element_set_state(pipe, GST_STATE_NULL); // 停止并回收资源 }
gst-launch — GStreamer 命令行工具
gst-launch-1.0 filesrc location=test.mp4 ! qtdemux ! h264parse ! avdec_h264 ! videoconvert ! autovideosink
gst-inspect-1.0 avdec_h264 — 查看 Element 的 Pad/Caps/属性
gst-discoverer-1.0 file.mp4 — 分析媒体文件信息

2. FFmpeg — 模块化工具集架构

libavcodec · libavformat · libavfilter · libswscale · libswresample · 线性处理流水线

libavcodec — 编解码库

400+ 编解码器 · Send/Receive 模型

FFmpeg 的心脏。包含所有编码器和解码器 (H.264/265, VP9, AV1, AAC, Opus...)。
Send/Receive API (FFmpeg 3.1+):
avcodec_send_packet() — 推送压缩数据
avcodec_receive_frame() — 获取解码帧
解耦的输入/输出允许解码器内部缓冲多包，支持硬件加速器流水线。

libavformat — 容器库

Demux · Mux · I/O Protocols

Demuxer: 解析容器 (MP4/MKV/TS/FLV...) → 提取 AVPacket 数据流
Muxer: 将编码后的 AVPacket → 写入容器 + 生成索引
I/O Protocols: 统一的 URL 抽象 — file:// rtmp:// udp:// tcp://
avformat_open_input() → av_read_frame() → av_interleaved_write_frame()

libavfilter — 滤镜图

200+ 滤镜 · 有向无环图

音频/视频滤镜的有向图引擎。滤镜通过 filtergraph 连接:
简单图: [in] → filter1 → filter2 → [out]
复杂图: 多输入多输出 — 拼接/画中画/混音
描述语法: [0:v]scale=1280:720,unsharp[out]
支持 AVFrame 零拷贝传递 — 引用计数共享 buffer。

/* ═══ FFmpeg 处理流水线 — 从输入文件到输出文件 ═══ */ [File/URL] │ avformat_open_input() // 打开容器，读取头信息 ▼ DEMUXER (libavformat) │ av_read_frame() // 逐包读取 AVPacket ▼ AVPacket (压缩数据包, 含 PTS/DTS) │ avcodec_send_packet() // 送入解码器 ▼ DECODER (libavcodec) │ avcodec_receive_frame() // 获取解码帧 ▼ AVFrame (原始像素/PCM 数据) │ av_buffersrc_add_frame() // 送入滤镜图 ▼ FILTER GRAPH (libavfilter) │ av_buffersink_get_frame() // 从滤镜图取出 ▼ AVFrame (处理后的原始帧) │ avcodec_send_frame() // 送入编码器 ▼ ENCODER (libavcodec) │ avcodec_receive_packet() // 获取编码包 ▼ AVPacket (编码后的压缩包) │ av_interleaved_write_frame() // 写入容器 ▼ MUXER (libavformat) │ av_write_trailer() // 写文件尾 + 索引 ▼ [Output File]

3. 核心差异深度对比

框架 vs 工具集 · 自动管理 vs 手动编排 · 动态 Pipeline vs 静态线性 · 同步 vs 异步

维度GStreamerFFmpeg
设计哲学框架 — 你组装积木，框架管理流程工具集 — 你手动编排每一步
Pipeline 模型有向无环图 — 可在运行时动态修改静态线性 — 启动时确定，运行时较难变更
线程管理自动 — 框架按 pipeline 阶段自动调度手动 — 开发者负责 pthread/线程池
时钟同步内置 — GstClock + NetClock + 音视频同步手动 — PTS 由开发者计算和对齐
QoS / 背压原生 — GstFlowReturn + QoS Event手动实现
内存模型框架管理 — BufferPool + 引用计数开发者管理 — 更低开销
编解码器数量自有 + gst-libav 包装 FFmpeg400+ — 业界最全
API 稳定性稳定 — 强向后兼容不稳定 — 版本间可能 Breaking
语言绑定C + GObject Introspection (Python/JS/Rust/Vala)C + 社区封装 (Python/Java)
启动时间快 — 按需加载插件慢 — 初始化全部编解码器
内存占用较高 — 插件隔离 + 内部缓冲较低 — 共享编解码器状态
硬加速支持VAAPI/V4L2/NVENC/DMA-BufCUDA/NVENC/VAAPI/Vulkan/Metal/AMF/DXVA2
学习曲线陡峭 — 概念多 (Pad/Caps/Bin/Bus)较平缓 — CLI 上手快，C API 中等
一句话区分
GStreamer: 你描述"要做什么" — 框架自动管理线程/同步/背压/状态。
FFmpeg: 你编写"怎么做" — 完全控制每一步，但需要自己处理所有工程细节。
类比: GStreamer = 自动驾驶，FFmpeg = 手动挡赛车。

4. 硬件加速支持

GPU 编解码 · DMA-Buf 零拷贝 · VAAPI/V4L2/NVENC/CUDA/Vulkan · 两种框架的实现方式

GStreamer 硬件加速

通过专用 Element + DMA-Buf 零拷贝

VAAPI: vaapih264dec / vaapih265enc (Intel GPU)
V4L2: v4l2h264dec / v4l2h264enc (嵌入式 SoC)
NVIDIA: nvv4l2decoder / nvv4l2h264enc (Jetson/DeepStream)
DMA-Buf: 跨 Element 共享 GPU 内存 — 零拷贝
特性: GStreamer 的 Buffer 体系天然支持 DMA-Buf 传递 — 解码→推理→编码全零拷贝。

FFmpeg 硬件加速

通过 hwaccel + hw_device_ctx 配置

VAAPI: -hwaccel vaapi -c:v h264_vaapi (Intel/AMD)
NVENC/NVDEC: -c:v h264_nvenc (NVIDIA GPU)
CUDA: -hwaccel cuda (NVIDIA 通用计算)
Vulkan: -hwaccel vulkan (跨平台新后端)
DXVA2/D3D11VA: Windows 平台
特性: 覆盖面更广 — 支持 Metal (Apple)、AMF (AMD)、QSV (Intel QuickSync)、MediaCodec (Android)。
