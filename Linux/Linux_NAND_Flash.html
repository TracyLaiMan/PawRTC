1. NAND Flash 硬件基础

四种存储单元 · 层次结构 (Package→Die→Plane→Block→Page) · ONFI 标准 · ECC 需求演进

SLC · MLC · TLC · QLC

每 Cell 存储 1/2/3/4 bits

SLC (1bit/cell): ~100K P/E 周期 — 工业/关键任务
MLC (2bit/cell): ~3K–10K P/E — 企业级
TLC (3bit/cell): ~300–3K P/E — 消费级 SSD/SD 卡
QLC (4bit/cell): ~500–1K P/E — 冷存/归档
bit 数 ↑ → 密度 ↑ 成本 ↓ → 寿命 ↓ 误码率 ↑
当前主流: 3D TLC 堆叠 128~232 层。

存储层次 — Page 与 Block

Package→Die→Plane→Block→Page

Page (4KB~16KB): 最小读写单元。含 Data Area + Spare Area (OOB — 存 ECC/坏块标记/元数据)。
Block (64~512 Pages): 最小擦除单元。擦除时整块归 1 (0xFF)。
Plane: 多个 Block 组成 — 允许并行操作。
Die/LUN: 独立芯片 — 可并行 I/O。
核心约束: 擦前写 · 页内顺序编程 · MLC/TLC 不允许部分页编程。

ECC — 纠错码演进

BCH → LDPC · 软判决

SLC (老制程): 1~4 bit/512B → Hamming
MLC (3x nm): 8~15 bit/1KB → BCH
MLC (2x nm): 15~24 bit/1KB → BCH
TLC / 3D-TLC: 40~84+ bit/1KB → LDPC + 软判决
QLC: 80+ bit/1KB → LDPC + RAID 恢复
OOB 区存储 ECC 校验字节: 每 8 bit 可纠错 ≈ 13 字节开销 (BCH)。

接口标准类型数据宽度最大速率主要厂商
ONFI (1.0→5.x)开放标准x8 / x16NVDDR-3: 最高 ~533 MT/sIntel/Micron, SK Hynix
Toggle NAND专有 (Samsung/Toshiba)x8 / x16Toggle 5.0: 最高 ~800 MT/sSamsung, Kioxia

2. Linux MTD 子系统 — 原始 Flash 抽象层

mtd_info · nand_chip · nand_controller · 分区 · OOB · ECC 引擎 · NAND 控制器驱动模型

MTD 核心数据结构

drivers/mtd/ · include/linux/mtd/

mtd_info: 每个 MTD 设备一个 — type/size/erasesize/writesize/oobsize。提供统一的 _read/_write/_erase/_read_oob 等回调。
nand_chip: NAND 芯片的私有数据结构 — 包含 nand_controller 指针、nand_ecc_ctrl (ECC 引擎)、options (芯片特性 BIT 掩码)、bbt (坏块表)。
nand_controller: 硬件 NAND 控制器抽象 — exec_op() 执行 NAND 命令序列。

NAND 驱动开发核心

nand_scan → 注册 mtd_device

① 分配 nand_chip — 填充 ops 回调
② 配置 ECC 引擎 (硬件 BCH / 软件 BCH / 无 ECC / on-die ECC)
③ 调用 nand_scan() — 自动探测 NAND 芯片参数 (ID→size/page/block/OOB)
④ 调用 mtd_device_register() — 注册 MTD 设备 + 创建分区
⑤ 驱动通过 nand_controller->exec_op() 发送 ONFI/Toggle 命令序列到硬件寄存器。

3. UBI — 非排序块镜像层

PEB/LEB 映射 · 磨损均衡 (WL) · 坏块管理 (BBM) · 数据擦洗 (Scrubbing) · Fastmap 加速启动

PEB ↔ LEB 映射

ubi->eba_table[vol_id][lnum] = pnum

PEB (Physical Erase Block): 物理擦除块，编号 pnum。每个 PEB 头含 EC (Erase Counter) + VID (Volume ID)。
LEB (Logical Erase Block): 卷内的逻辑块，编号 lnum。UBI 维护 EBA 表 将 (vol_id, lnum) → pnum。
写时复制: LEB 更新时分配新 PEB → 拷贝数据 → 更新 EBA → 旧 PEB 擦除回收。

磨损均衡 (Wear Leveling)

红黑树管理 PEB 按 EC 排序

free 树: 已擦除可用的 PEB (按 EC 升序)
used 树: 持有有效数据的 PEB (按 EC 升序)
scrub 树: 检测到 bit-flip 的 PEB — 需紧急迁移
err 树: 坏块 — 永久排除
WL 线程取 used 中 EC 最高 和 free 中 EC 最低 的 PEB 做数据迁移 → EC 差 < UBI_WL_THRESHOLD 时跳过。

Scrubbing — 数据擦洗

比磨损均衡更高优先级的数据抢救

MTD 读返回 -EUCLEAN (检测到可纠正 bit-flip) → UBI 将该 PEB 移入 scrub 树 → WL 线程优先迁移 scrub PEB 的数据到健康 PEB → 防止 bit-flip 累积为不可纠正错误。
主动 scrubbing: UBI 后台周期性读取 used PEB 检测早期退化。

/* ═══ UBI: 从 Attach 到正常运行 ═══ */ ubi_attach() (启动时 — 或 Fastmap 加速) ├── 扫描 所有 PEB — 读 EC header + VID header ├── 重建 EBA 表: (vol_id, lnum) → pnum ├── 重建红黑树: free / used / scrub / err ├── 恢复可能的中断操作 (power-cut recovery) └── 初始化 Fastmap (缓存 EBA 表到 anchor PEB) ubi_leb_write() (LEB 写入请求) ├── EBA 表查找 (vol_id, lnum) → 当前 pnum ├── 从 free 树取最低 EC 的 PEB → 新 pnum' ├── ubi_eba_copy_leb() — 复制数据到新 PEB ├── 原子更新 EBA 表: map[pnum'] = 新数据, map[pnum] = 无效 └── 旧 PEB → 擦除 → 重新加入 free 树 wear_leveling_worker() (后台 WL 线程) ├── used 树中取 EC 最高的 PEB → src ├── free 树中取 EC 最低的 PEB → dst ├── if EC(dst) - EC(src) < WL_THRESHOLD → 跳过 (不需 WL) ├── 迁移 src 数据到 dst → 原子更新 EBA └── 旧 src PEB → 擦除 → 重新加入 free 树
/* ===== UBI 配置与操作 ===== */ # 在 MTD 分区上创建 UBI 设备 ubiformat /dev/mtd2 // 格式化 (保留工厂坏块标记) ubiattach -m 2 // 挂载 MTD2 为 UBI 设备 # 创建 UBI 卷 ubimkvol /dev/ubi0 -N "rootfs" -s 100MiB // 静态大小卷 ubimkvol /dev/ubi0 -N "data" -m // 最大可用空间卷 # 查看 UBI 设备信息 ubinfo -a // 所有 UBI 设备 ubinfo /dev/ubi0 // 查看某设备详情 + 每个卷 /* ===== 内核模块参数 (UBI) ===== */ # Fastmap 池保留 PEB 数 (默认 256) modprobe ubi fm_reserved_pebs=128 # Fastmap 池大小 (默认 10% of total) modprobe ubi fm_pool_size=64

4. UBIFS — UBI 专用日志结构文件系统

写时复制 · 回滚日志 · 透明压缩 · 快速恢复 · 无需 FTL 层

UBIFS 核心设计

日志结构 + 回滚 + COW

日志 (Journal): 所有写入先进入日志 → 后台提交 (commit) 将日志合并到主数据区。
回滚 (Roll-back): 未提交的日志在断电后自动丢弃 → 无需 fsck。
写时复制 (COW): UBIFS → UBI → 新 PEB → 旧 PEB 擦除回收。数据永不被原地覆盖。
B+ 树索引: 用于快速文件查找和空间分配。
透明压缩: 支持 zlib/lzo/zstd — 在写入时压缩，读取时解压，对应用透明。

垃圾回收 (GC)

后台合并有效数据 → 释放 PEB

UBIFS 与 UBI 协同 GC: UBIFS 标记无效 LEB → UBI 擦除对应 PEB → PEB 回到 free 树。
B+ 树 GC: UBIFS 内部整理索引节点，回收被删除文件占用的空间。
后台线程: UBI WL 线程 + UBIFS GC 线程协作，保持足够的可写空间。

/* ===== UBIFS 使用命令 ===== */ # 在 UBI 卷上创建 UBIFS mkfs.ubifs -r /path/to/rootfs -m 2048 -e 126976 -c 2048 -o rootfs.ubi # -m: 最小 I/O 大小 (page size) # -e: 逻辑擦除块大小 (PEB size - UBI 开销) # -c: 最大 LEB 数量 # 挂载 UBIFS mount -t ubifs ubi0:rootfs /mnt/rootfs # 或: mount -t ubifs ubi0_0 /mnt/rootfs // vol_id=0 # 查看 UBIFS 统计信息 cat /sys/kernel/debug/ubifs/ubi0_0/stats /* ===== 设备树中标记 UBI 分区 ===== */ # 内核命令行: ubi.mtd=2 (MTD2 作为 UBI 设备, 3=vol_id) # 完整启动参数示例: console=ttyS0,115200 root=ubi0:rootfs rootfstype=ubifs ubi.mtd=2 rootwait
⚠️ 关键实践: 一个 UBI 设备 → 多个 UBI 卷
推荐: 一个 MTD 分区 → 一个 UBI 设备 → 多个 UBI 卷 (rootfs + data + config)
不推荐: 每个 UBI 卷一个独立 UBI 设备
原因: UBI 的磨损均衡跨所有卷共享 PEB 池。将多个卷放在同一 UBI 设备下 → 读写频繁的卷可以从只读卷的 PEB 池中受益 → 全局磨损更均衡。

5. SquashFS on NAND — 只读根文件系统方案

通过 ubiblock 将 UBI 卷导出为只读块设备 → SquashFS 挂载 · 与 UBIFS 互补 · 嵌入式固件最佳实践

SquashFS 是什么？

高压缩只读文件系统 — 嵌入式固件首选

SquashFS 是 Linux 内核原生的只读压缩文件系统。特点:
极高压缩比: 支持 zlib/lzo/xz/zstd/lz4，xz 压缩率最高 (~50% 原始大小)。
只读 = 天然安全: 不可篡改的根文件系统 — 断电/意外重启不会损坏。
块设备依赖: SquashFS 需要块设备作为底层存储。NAND 不是块设备 → 需要中间层桥接。

ubiblock — UBI 到 SquashFS 的桥梁

将 UBI 静态卷导出为只读块设备

ubiblock -c /dev/ubi0_0 将 UBI 静态卷 ubi0_0 导出为 /dev/ubiblock0_0 只读块设备。
原理: ubiblock 驱动将 UBI LEB 线性映射为连续块设备扇区，读取时由 UBI 透明处理坏块跳过和 ECC。
优势: ① UBI 层的坏块管理 + bit-flip 纠正 保护 SquashFS ② SquashFS 无需感知底层是 NAND ③ 压缩镜像体积小 → 节省 Flash 空间 + 更快的烧录速度。

SquashFS vs UBIFS — 如何选择？

嵌入式 NAND Flash 的两大文件系统对比

SquashFS 适用: ① 根文件系统 (rootfs) — 只读 ② 固件镜像 — 不可篡改 ③ OTA 时整卷替换 ④ 需要最高压缩比节省 Flash
UBIFS 适用: ① 需要可写持久存储 (配置、日志、用户数据) ② 需要磨损均衡保护的可写分区 ③ 需要原子更新的关键数据
经典组合: UBI vol0=SquashFS (rootfs, 只读) + UBI vol1=UBIFS (data, 可写) + UBI vol2=UBIFS (config, 可写)

UBI 静态卷 vs 动态卷

SquashFS 必须使用静态卷

静态卷 (Static Volume): -t static (ubimkvol 默认)。支持CRC 完整性校验，UBI 读取时自动检测数据损坏。内容不可变 — UBI 不对其做 WL (只读)。
动态卷 (Dynamic Volume): -t dynamic。无 CRC 保护 (由上层 UBIFS 负责)。支持 WL 和 COW 写更新。
SquashFS 必须用静态卷: 只读性质 + UBI 层 CRC 保护 + 避免不必要的 WL 开销。

/* ===== SquashFS on NAND: 完整制作与部署流程 ===== */ # 1. 制作 SquashFS 镜像 mksquashfs /path/to/rootfs rootfs.squashfs -comp xz -b 256K -noappend -always-use-fragments # -comp xz: 使用 xz 压缩 (最高压缩比, 嵌入式首选) # -comp zstd: 使用 zstd (更快解压, 较新内核) # -b 256K: 块大小 (更大 = 更高压缩率, 但内存占用大) # 2. 制作 UBI 镜像 (含 SquashFS rootfs 卷 + UBIFS data 卷) # a) 先创建 UBIFS 可写卷的镜像 mkfs.ubifs -r /path/to/data -m 2048 -e 126976 -c 400 -o data.ubifs # b) 用 ubinize 将 SquashFS + UBIFS 打包为完整 UBI 镜像 # ubinize.cfg: [rootfs] mode=ubi image=rootfs.squashfs // SquashFS 镜像作为卷内容 vol_id=0 vol_type=static // ★ 静态卷 — SquashFS 必需 vol_name="rootfs" vol_size=50MiB [data] mode=ubi image=data.ubifs vol_id=1 vol_type=dynamic // ★ 动态卷 — 可写数据 vol_name="data" vol_size=100MiB # c) 生成最终 UBI 镜像 ubinize -o ubi.img -m 2048 -p 128KiB ubinize.cfg # 3. 烧录 UBI 镜像到 NAND ubiformat /dev/mtd2 nandwrite /dev/mtd2 ubi.img # 4. 启动时挂载 (手动) ubiblock -c /dev/ubi0_0 // → /dev/ubiblock0_0 mount -t squashfs /dev/ubiblock0_0 /mnt/rootfs # 5. 完整的 kernel bootargs (自动挂载根文件系统) console=ttyS0,115200 root=/dev/ubiblock0_0 rootfstype=squashfs ubi.mtd=2 ubi.block=0,rootfs rootwait # ubi.block=0,rootfs → 内核自动: rootfs 卷 → /dev/ubiblock0_0
🔧 SquashFS + UBIFS 组合 — 嵌入式黄金搭档
经典布局: MTD2 → UBI0 → vol0=SquashFS(static,rootfs,50MB) + vol1=UBIFS(dynamic,data,100MB) + vol2=UBIFS(dynamic,config,10MB)
OTA 升级: 新固件写入 vol0(新 SquashFS 镜像) → 原子替换 → 重启后运行新版本。vol1(用户数据) 不受影响。
工厂恢复: vol0 是只读 SquashFS → 即使 vol1/vol2 损坏，系统仍可启动到恢复模式。
安全: rootfs 不可篡改 + dm-verity 可选 → 完整的启动链完整性保护。
