DDS 架构：从 ROS1 到 ROS2 的进化

ROS1 依赖中心化的 roscore（XML-RPC + TCP）协调节点通信——单点故障、延迟高、不适合实时系统。DDS 彻底解决了这些问题。

ROS1 的问题

中心化 — roscore 单点故障 · XML-RPC 慢 · TCP only

• roscore 必须最先启动——一旦 crash，整个系统通信中断
• XML-RPC 握手——每次建立话题连接需要数次 XML-RPC 往返（~10ms/次）
• 传输协议单一——仅 TCP，无 UDP 多播、无实时性保证
• 无 QoS——无法指定可靠性、持久性、截止时间等策略
• 无内置安全——需额外添加

ROS2 + DDS 的优势

去中心化 — 自动发现 · 共享内存 · UDP 多播 · QoS

• 无中心节点——节点直接对等发现（Simple Discovery Protocol），零依赖
• 自动发现——新节点加入，自动被其他节点感知，无需配置
• 共享内存传输——同主机节点通信走 SHM，延迟 <10μs
• 22 种 QoS 策略——可靠性、持久性、截止时间、存活检测……
• 内置安全——DDS-Security（认证、加密、访问控制）

DDS

核心概念：DCPS 模型 (Data-Centric Publish-Subscribe)

Domain → Participant → Publisher/Subscriber → DataWriter/DataReader → Topic → QoS

Domain（域）：一个逻辑隔离的通信空间——Domain ID 不同的 Participant 互相不可见。
Participant（参与者）：通信的入口——一个进程通常创建一个 Participant。
Publisher + DataWriter：发布端——DataWriter 绑定 Topic 并指定 QoS，Publisher 管理多个 DataWriter。
Subscriber + DataReader：订阅端——DataReader 指定回调函数接收数据。
Topic（话题）：数据的逻辑通道——由名称 + 类型唯一标识。
QoS（服务质量）：22 种可组合的策略控制通信的每一个方面。

RTPS — Real-Time Publish-Subscribe 线协议

RTPS (Real-Time Publish-Subscribe) 是 DDS 的标准线协议——定义了 Participant 之间如何通过 UDP 交换数据。这是 DDS 互操作性的基础。

RTPS 消息类型

Header + Submessage 列表

每条 RTPS 消息 = RTPS Header (4B 协议标识 + 版本 + Vendor ID + GUID 前缀) +一个或多个 Submessage。
每个 Submessage 有类型 + 标志位（Endianness）+ 长度 + 负载。Submessage 类型包括：DATA（用户数据）、HEARTBEAT（Writer 声明可用序列号范围）、ACKNACK（Reader 请求缺失序列号）、GAP（Writer 告知"这些序列号被跳过"）、INFO_TS（时间戳）、INFO_DST（目标 GUID）。

可靠通信 (Reliable)

Heartbeat + ACKNACK + 重传 = 类 TCP 保证

Writer 定期发送 HEARTBEAT（声明"我有 seq 1~100 的数据"）。Reader 检查本地缺失的序列号，回复 ACKNACK（"请重传 seq 3, 7, 42"）。Writer 收到 ACKNACK 后重传缺失序列号的数据。
与 TCP 的差异：① NACK-based——只通知缺失的，不需要每包都 ACK（比 TCP 更高效）② 应用层重传——可以给不同 seq 设不同的重传策略 ③ 支持多播可靠（TCP 做不到）。

Best-Effort 通信

零开销 · 无保证 · 低延迟

Writer 直接发送数据，无 Heartbeat、无 ACKNACK、无重传。Reader 收到什么算什么——丢了就丢了。
适合：高频传感器数据（1000Hz IMU、Camera 视频帧）——丢一帧不影响整体，关键是最低延迟。
与 Reliable 的延迟对比：Best-Effort ~50μs vs Reliable ~200μs（SHM 模式下）。

RTPS Writer-Reader 可靠通信的交互流程
Writer Reader │ │ │── DATA(seq=1) ─────────────────────────────→ │ 持续发送数据包 │── DATA(seq=2) ─────────────────────────────→ │ │── DATA(seq=3) ────X (丢包) │ │── DATA(seq=4) ─────────────────────────────→ │ │ │ │── HEARTBEAT(available: 1..4) ───────────────→│ 周期心跳: 声明已有 seq 范围 │ │ Reader 检查本地: 发现缺 seq 3 │←── ACKNACK(missing: seq=3) ─────────────────│ 请求重传缺失的 seq 3 │── DATA(seq=3) (repair) ────────────────────→│ Writer 从 HistoryCache 取出重传 │ │ │── HEARTBEAT(available: 1..100) ─────────────→│ 后续心跳: seq 1~100 已全部就绪 │←── ACKNACK(final: seq=100) ────────────────│ 全部确认收到，通信完成 ✓

QoS — 22 种服务质量策略

QoS 是 DDS 的灵魂——也是它与 ROS1、MQTT、ZeroMQ 等所有消息中间件最根本的区别。组合 QoS = 为每个通信通道定制行为。

QoS 策略可选值作用ROS2 默认
RELIABILITYRELIABLE / BEST_EFFORT是否需要保证送达 + 重传RELIABLE
DURABILITYVOLATILE / TRANSIENT_LOCAL / TRANSIENT / PERSISTENT后加入的 Subscriber 能否收到历史数据VOLATILE
HISTORYKEEP_LAST(n) / KEEP_ALLWriter 缓存多少条历史数据KEEP_LAST(1)
DEADLINE最大发布间隔（秒）超时 → 通知应用层 missed deadline无限
LIFESPAN数据有效时间（秒）超时的数据自动丢弃无限
LIVELINESSAUTOMATIC / MANUAL_BY_TOPIC / MANUAL_BY_PARTICIPANTWriter 存活检测——多久没心跳判定离线AUTOMATIC
OWNERSHIPSHARED / EXCLUSIVE多个 Writer 同 Topic 时——谁的优先SHARED
PARTITION字符串列表在同一 Topic 内进一步逻辑分区分组空（全局）
TRANSPORT_PRIORITYuint32多个 Transport 时的优先级0
RESOURCE_LIMITSmax_samples / max_instances限制 Reader/Writer 资源使用100 / 10

DURABILITY — 迟到订阅者能收到历史数据

ROS2 的 "latched topics" = TRANSIENT_LOCAL

VOLATILE：默认——Subscriber 只收到订阅之后发布的数据。迟到 = 错过。
TRANSIENT_LOCAL：Writer 在内存中保留最近 n 条（HISTORY 决定）——后加入的 Subscriber 自动收到历史数据。这是 ROS2 中"latched topic"的实现原理（如 /clock、/map、/robot_description）。
TRANSIENT：同上，但保留在持久存储（如磁盘）而非内存。
PERSISTENT：数据在 Writer 退出后仍然保留——永久存储。

OWNERSHIP + STRENGTH — 多 Writer 的高可用

主备 Writer 自动切换的秘诀

SHARED：默认——所有 Writer 的数据都被接受。
EXCLUSIVE：同一 Topic 同时只接受一个 Writer的数据——OWNERSHIP_STRENGTH 最高的 Writer 胜出。主 Writer 下线 → 自动切换到 STRENGTH 次高的备 Writer。
应用场景：冗余传感器（主传感器故障 → 自动切到备用传感器）、Leader 选举（STRENGTH 最高的节点成为 Leader，无需外部选举协议）。这是 DDS 实现零切换时间冗余的核心机制。

自动发现机制 — 去中心化的核心

没有 roscore、没有 DNS、没有配置——FastDDS 通过两阶段自动发现让节点互相感知。这是 ROS2 去中心化的技术基础。

阶段1: PDP (Participant Discovery)

SPDP — Simple Participant Discovery Protocol

每个 Participant 周期性（默认每 1 秒）通过UDP 多播（239.255.0.1:7400）或单播（Initial Peers 列表）发送 SPDP DATA 包——包含自己的 GUID、支持的 Topic 列表、QoS 配置、Locator（IP+Port）列表。
收到其他 Participant 的 SPDP 包 → 记录其 Locator → 建立对等连接。
多播适用于局域网（零配置自动发现），单播适用于跨子网或云部署（需指定 IP 列表）。
存活检测：如果 8 秒（leaseDuration）未收到 SPDP → 判定该 Participant 离线 → 自动清理。

阶段2: EDP (Endpoint Discovery)

SEDP — Simple Endpoint Discovery Protocol

PDP 建立 Participant 连接后，进入端点级发现——交换 DataWriter 和 DataReader 信息。
每个 Participant 通过内置的 3 个可靠 Topic（Writers → Readers → 端点信息 Topic）通告自己的端点：
① PublicationBuiltinTopic（我有哪些 Writer）。
② SubscriptionBuiltinTopic（我有哪些 Reader）。
③ TopicBuiltinTopic（我有哪些 Topic 类型）。
当 Reader 发现匹配的 Writer（Topic + type + QoS compatible）→ 自动建立通信通道——应用层无需任何手动 connect 调用。

发现参数 — 可调节的"心跳"频率
SPDP 周期（leaseAnnouncementPeriod）：默认 1 秒——影响发现速度。减小到 100ms 可加快发现，但增加多播流量。
租约时长（leaseDuration）：默认 8 秒——超时判定离线的时间。需 > leaseAnnouncementPeriod × 安全系数。
Initial Peers：静态 IP 列表——用于跨子网或禁止多播的环境。
ROS2 默认：多播发现 + localhost only（ROS_LOCALHOST_ONLY=1 时禁用多播，仅本地 IPC）。

FastDDS 在 ROS2 中的集成

ROS2 通过 rmw（ROS Middleware）抽象层接入 FastDDS——每一行 ros2 topic pub 调用最终映射到 FastDDS 的 DataWriter API

rmw 抽象层

ROS Middleware — 中间件可插拔

ROS2 的 rmw（ros middleware）层是 DDS 实现与 ROS2 Client Library（rcl）之间的抽象接口。通过实现 rmw_init/node_create/publisher_create/subscription_create... 等 API，可以接入任意 DDS 实现——FastDDS 只是 rmw_fastrtps，还有 rmw_cyclonedds、rmw_connextdds。
切换 DDS 实现无需改应用代码——只需设置 RMW_IMPLEMENTATION=rmw_fastrtps_cpp。

ROS2 概念 → DDS 映射

Node · Topic · QoS → Participant · DataWriter · DataReader

ROS2 Node → DDS Participant（一个进程多个 Node 共享一个 Participant）。
ROS2 Publisher → DDS DataWriter（每个 Topic 一个）。
ROS2 Subscription → DDS DataReader。
ROS2 Topic name → DDS Topic name（如 /cmd_vel）。
ROS2 QoS → DDS QoS 策略组合（reliability/durability/history/depth/deadline...）。
ROS2 Service → DDS 的 Request/Reply 模式（底层用两个 Topic 实现：Request Topic + Reply Topic）。

共享内存 (SHM) 加速

同主机节点通信 → 零拷贝

FastDDS 的 SHM Transport 检测到 Reader 和 Writer 在同一台机器 → 自动切换到共享内存传输（绕过 UDP 协议栈）。延迟从 UDP 的 ~80μs 降到 ~5-10μs，吞吐从 ~2Gbps 升到 ~10Gbps+（受限于内存带宽）。
完全透明——应用代码不需要任何修改，同一份 Topic 的 QoS 配置对 SHM 和 UDP 都有效。
底层实现：POSIX 共享内存段（/dev/shm/fastdds_*）+ 互斥锁 + 条件变量。

ROS2 的 Topic/Service/Action 在 DDS 层面的实现

配置与性能调优

FastDDS 通过 XML Profile 或 ROS2 QoS API 配置——默认参数适合通用场景但远非最优。

XML Profile 配置

完整的 DDS 配置通过 XML 文件注入
<participant> <rtps> <builtin> <!-- 发现配置 --> <discovery_config> <leaseDuration>10</leaseDuration> <leaseAnnouncement>2</leaseAnnouncement> </discovery_config> </builtin> <!-- SHM + UDP 双 Transport --> <transport_descriptors> <transport_id>udp</transport_id> <transport_id>shm</transport_id> </transport_descriptors> </rtps> </participant>

ROS2 QoS Profile 速查

4 种预定义 QoS Profile + 自定义

rmw_qos_profile_sensor_data：Best-Effort + Volatile + KeepLast(5)。适合传感器（高频、允许丢帧）。
rmw_qos_profile_services_default：Reliable + Volatile + KeepLast(1)。Service 默认。
rmw_qos_profile_parameters：Reliable + TransientLocal + KeepLast(1000)。参数服务器——后加入节点也能获取已设置的参数。
rmw_qos_profile_system_default：Reliable + Volatile + KeepLast(1)。Topic 默认。
自定义：rclcpp::QoS(10).reliable().transient_local() C++ API。

SHM Transport 调优

segment_size · port_queue_capacity · 零拷贝

• segment_size：共享内存段大小——默认 512MB。大数据话题（图像/点云）需调大。
• port_queue_capacity：每个端点的环形缓冲大小——默认 512。高频话题（>1KHz）需调大避免丢帧。
• 零拷贝模式（Zero-Copy）：Writer 直接将数据写入共享内存 → Reader 直接从共享内存读（无 memcpy）。需数据类型支持（固定大小 + 无指针）。吞吐可提升 2-3×。

网络 Transport 调优

UDP socket buffer · 异步发送 · 多播配置

• socket buffer：增大 send_buff_size/recv_buff_size——默认 65KB 太小。设置为 2-4MB 避免因应用来不及读而丢包。
• 异步模式（ASYNC）：Publisher 的 write() 非阻塞——数据进入内部队列，后台线程批量发送。适合高吞吐场景。
• 多播 TTL：默认 1（局域网）。跨路由器需增大 TTL。

关键性能指标

FastDDS 在典型场景的延迟/吞吐

• SHM, 256B payload：~6μs 延迟（Reliable）/~3μs（Best-Effort）
• UDP loopback, 256B：~40μs（Reliable）/~20μs（Best-Effort）
• SHM 吞吐：~12 GB/s（256KB payload）
• 对比 Cyclone DDS：SHM 延迟 Cyclone 略快（~4μs vs ~6μs），但 FastDDS 的零拷贝模式在大 payload 时有优势
• 实际数字取决于 CPU 架构、内核版本、负载。

为什么 ROS2 选 DDS（FastDDS）而不是其他 IPC？

关键决策因素：去中心化 + 自动发现 + 丰富的 QoS + 工业标准

① 去中心化：ROS1 的 roscore 单点故障在机器人上不可接受——物理机器人不能在 roscore crash 后失去所有通信。DDS 的 P2P 自动发现解决了这个根本问题。
② 丰富的 QoS：机器人系统中有完全不同的通信需求——传感器数据容忍丢帧（Best-Effort）、控制命令必须可靠送达（Reliable）、TF 变换需要后加入节点也能获取（Transient Local）——DDS 是唯一能在一个框架内覆盖所有这些需求的标准。
③ 工业标准：DDS 是 OMG 标准，在军工/航空/医疗/自动驾驶领域有 20+ 年验证历史——NASA、Airbus、Siemens、Bosch 都在用。
④ 可替换性：rmw 抽象层让 ROS2 不锁定任何 DDS 实现——FastDDS 是默认选择（开源、社区活跃），但可以根据性能/安全需求切换到 Cyclone DDS 或 RTI Connext。
