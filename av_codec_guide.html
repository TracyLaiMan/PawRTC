基础概念

理解音视频的三大维度：时间（采样率/帧率）、空间（分辨率/色深）、数据量（码率）

采样率 / 帧率

时间维度的精度

音频：44.1kHz CD、48kHz 视频/专业、16kHz 语音
视频：24fps 电影、30fps TV、60fps 游戏
奈奎斯特定理：采样率 ≥ 2×信号最高频率

分辨率 / 色深

空间维度的精细度

视频：1080p、4K
像素格式：YUV420 最常用、NV12 硬编、RGB24 显示
色深：8bit SDR、10bit HDR

码率

数据量控制的核心

CBR 恒定码率：网络友好，画质波动
VBR 可变码率：画质稳定，码率波动
CRF 恒定质量：x264/x265 首选，18-28 常用

YUV 为什么比 RGB 更适合视频编码？
YUV 将亮度（Y）与色度（UV）分离。人眼对亮度敏感但对色度不敏感，所以可以对 UV 分量下采样（4:2:0），在几乎不损失感知画质的情况下将数据量直接减半——这是视频压缩的第一道"免费午餐"。

GOP 与帧类型

I帧是锚点，P帧靠预测，B帧压得最小——三者的编排决定了压缩效率与随机访问能力的平衡

I 帧 — 帧内编码帧

关键帧 / IDR 帧

只使用本帧内像素预测，不依赖任何其他帧。压缩比最低（~7:1），但可作为随机访问点。IDR 帧是特殊的 I 帧，会清空参考帧缓冲。

P 帧 — 前向预测帧

Predicted Frame

参考前一个 I 或 P 帧进行帧间预测。压缩比中等（~20:1），实时编码常用。只有与参考帧不同的部分才需要编码。

B 帧 — 双向预测帧

Bi-predictive Frame

同时参考前向和后向帧预测。压缩比最高（~50:1），但引入编码延迟。不被其他帧参考，丢 B 帧不会扩散错误。

PTS vs DTS — 解码序与显示序的分离
PTS（Presentation Time Stamp）= 该帧何时显示；DTS（Decode Time Stamp）= 该帧何时解码。
含 B 帧时：解码序 I₀ P₃ B₁ B₂ → 显示序 I₀ B₁ B₂ P₃。只有无 B 帧时 PTS = DTS。
GOP 大小如何影响画质与延迟？
短 GOP（GOP=15~30）：I 帧频繁 → 容错性好、seek 快，但压缩效率低。适合实时通信。
长 GOP（GOP=60~300）：I 帧稀疏 → 压缩效率高，但 seek 慢、丢 I 帧则灾难。适合离线编码/点播。
全 I 帧（GOP=1）：编辑友好但码率极高。用于视频编辑中间格式（ProRes/DNxHD）。

H.264 编码器架构总览

编码器由前向编码通路（预测→变换→量化→熵编码）和反馈重建通路（反量化→反变换→去块滤波→参考帧缓冲）组成闭环。编码器内嵌解码器副本是理解视频压缩的关键。

VCL — 视频编码层

Video Coding Layer · 压缩的核心

负责实际的压缩工作：预测、变换、量化、熵编码。输出 Slice 数据，即编码后的宏块流。VCL 与网络无关，只关心如何高效压缩视频。

NAL — 网络抽象层

Network Abstraction Layer · 打包与传输

将 VCL 输出的数据封装成 NAL Unit，添加头部（nal_unit_type, nal_ref_idc）。适配不同传输场景：RTP 包化实时流、字节流文件、MP4容器。

编码器设计精髓：反馈环路中的解码器副本
编码器必须知道解码器能看到什么重建帧，才能保证后续预测的一致性。如果两侧参考帧不同，就会产生漂移（Drift），画质快速恶化。这就是为什么 x264 源码中同时存在 encode 和 decode 路径。

H.264 帧内预测

利用当前帧内已编码的相邻像素预测当前块——消除空间冗余。H.264 支持 4×4（9种方向模式）、8×8（9种）、16×16（4种）三种粒度

4×4 亮度块 · 9 种方向

最精细预测粒度

0:垂直 ↓, 1:水平 →, 2:DC, 3:左下对角, 4:右下对角, 5:右垂直偏, 6:下水平偏, 7:左垂直偏, 8:上水平偏。
每种模式从相邻已编码像素沿特定方向外推。

16×16 亮度块 · 4 种

平坦区域粗粒度

0:垂直, 1:水平, 2:DC, 3:Plane(平面)。
Plane 模式对渐变区域效果最好，在水平和垂直方向同时平滑过渡。大面积天空/墙壁常用。

8×8 色度块

与16×16亮度共用4模式

色度 UV 使用与 16×16 亮度相同的 4 种模式，块尺寸为 8×8。YUV420 下色度分辨率只有亮度 1/4，8×8 色度对应 16×16 亮度宏块。

预测类型块尺寸模式数量适用场景编码效率
I4MB (Intra 4×4)4×4 亮度9 种方向纹理复杂区域、细节丰富高（逐块最优）
I8MB (Intra 8×8)8×8 亮度9 种方向中等纹理、过渡区域中（High Profile）
I16MB (Intra 16×16)16×16 亮度4 种（V/H/DC/Plane）平坦区域、天空、墙壁低（粗粒度）
色度 Intra8×8 色度4 种（同 I16MB）色度分量（U/V）—
帧内预测的本质：空间邻居的智能拷贝
编码器遍历所有可用预测模式，选择SATD（Hadamard 变换后绝对差和）最小的模式作为最优模式。预测越准 → 残差越小 → 量化后越接近零 → 码率越低。

H.264 帧间预测

在参考帧中搜索与当前块最相似的区域——利用时间冗余。运动估计是编码器最耗时的模块（占 60-80% 编码时间）

树状运动补偿 · 可变块大小

16×16 → 4×4 灵活划分

一个 16×16 宏块可划分为：1×16×16 · 2×16×8 · 2×8×16 · 4×8×8。每个 8×8 子块还可分：1×8×8 · 2×8×4 · 2×4×8 · 4×4×4。平坦区域用大块，复杂运动用小块——码率与精度平衡。

亚像素运动估计 · 1/4 像素精度

亮度：6-tap Wiener 滤波器

1/2 像素：6 抽头滤波 (1, -5, 20, 20, -5, 1)/32
1/4 像素：相邻半像素点线性平均
色度：1/8 像素精度，双线性插值。
亚像素精度使压缩效率提升 15-20%。

帧类型参考方向典型压缩比编码延迟应用
I 帧无（帧内）~7:10（即时）随机访问点
P 帧前向 1 帧~20:10实时/低延迟
B 帧（参考）前向+后向~50:11-2 帧离线编码
B 帧（非参考）前向+后向~60:11-2 帧最大压缩
运动估计的搜索范围是双刃剑
更大搜索范围（如 ±64 像素）→ 更好匹配 → 更低码率，但搜索时间与范围平方成正比。
x264 默认 merange=16，H.265 可用 64。快速运动场景需更大范围。
加速技术：钻石搜索、六边形搜索、TZ Search——核心思想是先粗搜再精搜。

H.264 变换与量化

将空间域残差变换到频域，再通过量化丢弃人眼不敏感的高频分量——编码器唯一的有损环节。QP 每增加 6，量化步长翻倍，码率约减半

整数 DCT 变换

用加法+移位替代浮点乘法

H.264 使用整数 DCT（非浮点），矩阵元素都是整数，变换仅需加减和移位。
正向变换：Y = H·X·Hᵀ，H 为 4×4 整数变换矩阵。
DC 系数 = 均值，AC 系数 = 细节。
16×16 模式下 DC 系数会再做 Hadamard 变换以进一步去冗余。

量化 — QP 控制一切

QP 0-51，每+6则Qstep翻倍

量化公式：Z_ij = round(Y_ij / Qstep)，其中 Qstep 由 QP 查表决定。
H.264 支持自适应量化：不同宏块可用不同 QP（ΔQP=±2），在画质和码率间做局部平衡。

H.264 熵编码

将量化后的系数和语法元素压缩为最终二进制码流——利用统计冗余做无损压缩。H.264 支持 CAVLC（变长编码）和 CABAC（算术编码）两种方案

CAVLC · 变长编码

Baseline Profile 支持

基于 Huffman 编码的上下文自适应方案。用查表方式编码，速度快。
利用已编码相邻块的非零系数数量自适应选择 VLC 码表。
典型节省：比固定码表省 5-10%。

CABAC · 算术编码

Main/High Profile 支持

三阶段流水线：① 二值化→② 上下文建模（364 个概率模型）→③ 二进制算术编码。
编码后更新概率供下个 bin 使用。
比 CAVLC 节省 9-14% 码率，但计算复杂度高 ~40%。

H.264 环路滤波

对重建帧的块边界进行平滑——环路内的滤波，消除方块效应的同时提升后续预测的参考帧质量

方块效应的成因

分块编码的副作用

① 变换/量化是块级独立的 → 相邻块量化误差不同 → 边界不连续
② 运动补偿使用不同参考块 → 边界像素来源不同
③ 低码率下高频大量归零 → 差异更明显

边界强度 Bs · 0~4

决定滤波强度

滤波强度由边界强度 Bs（0~4）决定，同时受阈值 α（块间）和 β（块内）约束。
Bs 越大 → 滤波越强 → 边界越平滑，但过强会模糊真实细节。
完整判定逻辑见下方 Bs 对照表。

滤波决策与裁剪

只滤真边界，不过度平滑

先判断是否为伪边界（阈值 α/β 检查）。Bs=4 用最强 5 抽头滤波，Bs=1-3 用标准 4 抽头。
Clip 操作设上下限，防止过度滤波导致真实细节丢失。

Bs 值适用条件滤波强度抽头数说明
4帧内编码块边界最强5 抽头帧内块边界差异通常最大，需要最强滤波
3帧间 + 非零变换系数（大残差）强4 抽头残差大说明两个块差异大，边界明显
2帧间 + 有变换系数（有残差）中4 抽头有残差但较小，适度降低块效应
1帧间 + 无残差 + 不同参考帧/MV弱4 抽头虽无残差但参考来源不同，轻微滤波
0同参考帧 + 相同运动矢量跳过—两个块来源一致，无块效应，不滤波
环路滤波 ≠ 后处理滤波
关键区别：环路滤波作用于参考帧缓冲之前——滤波后的帧会作为后续帧的参考。更好的去块效果不仅改善当前帧画质，还能提升后续所有帧的预测精度。
后处理滤波（如 FFmpeg 的 pp 滤镜）只改善播放显示效果，不影响编码效率。

PCM · 脉冲编码调制

数字音频的基础

采样率 × 位深 × 声道 = 原始码率
CD: 44100×16×2=1411kbps
无压缩，每个采样点独立存储。
S16LE/S24LE/S32LE 最常见格式。

AAC · 高级音频编码

MP4 标配，128kbps≈CD

核心工具：心理声学模型（掩蔽阈值）、MDCT 变换、TNS（时域噪声整形）、SBR（频带复制）。
AAC-LC：128-192kbps 最常用
HE-AACv2：32-64kbps 低码率

Opus · WebRTC 首选

6-510kbps 全场景

SILK（语音 6-32kbps）+ CELT（音乐 32-510kbps）混合架构。
延迟 5-65ms 可配，自适应码率，丢包隐藏（PLC）。
WebRTC 强制要求支持 Opus。

MP4 · ISO BMFF

最通用的容器格式

基于 Box/Atom 树结构：ftyp → moov → mdat
moov 含所有索引信息（stbl六表），mdat 含实际音视频数据。

fMP4：分片 MP4，每个分片独立可解码。DASH/HLS/LiveKit 推流的基础。

FFmpeg 实战

命令行是理解编解码最快速的工具——从信息查询到码流分析，从格式转换到参数调优

# 1. 查看视频完整信息 ffprobe -show_streams -show_format input.mp4 # 2. 提取裸 YUV 流（用于分析单帧） ffmpeg -i input.mp4 -c:v rawvideo -pix_fmt yuv420p output.yuv # 3. 查看每一帧的类型 (I/P/B) ffprobe -show_frames -select_streams v input.mp4 | grep pict_type # 4. 分析 H.264 码流头部信息 ffmpeg -i input.mp4 -c:v copy -bsf:v trace_headers -f null - 2>&1 | head -100 # 5. 显示每一帧的编码信息（QP、帧大小等） ffprobe -show_frames -select_streams v -of compact input.mp4
场景命令
转码 H.264→H.265ffmpeg -i in.mp4 -c:v libx265 -crf 28 out.mp4
调整 GOP 大小ffmpeg -i in.mp4 -c:v libx264 -g 30 -keyint_min 30 out.mp4
CBR 码率控制ffmpeg -i in.mp4 -c:v libx264 -b:v 2M -maxrate 2M -bufsize 4M out.mp4
提取关键帧ffmpeg -i in.mp4 -vf "select=eq(pict_type\,I)" -vsync vfr frames_%03d.png
FastStart 优化ffmpeg -i in.mp4 -c copy -movflags +faststart out.mp4
无损裁剪ffmpeg -ss 10 -i in.mp4 -t 5 -c copy clip.mp4
FFmpeg 学习建议
先用 ffprobe 看信息，再用 ffmpeg -c copy 做无损操作（秒级完成），最后才加编码参数触发完整编解码（需数分钟）。
关键意识：-c copy 直接拷贝码流不编解码，改变 GOP/码率/编码格式才会触发重编码。

API 编程入门

FFmpeg C API 提供了解码器的完整控制权——理解 AVPacket（压缩包）和 AVFrame（解码帧）之间的异步流转是入门关键

// FFmpeg 解码最小流程（伪代码） // 1. 打开输入文件 avformat_open_input(&fmt_ctx, "input.mp4", NULL, NULL); avformat_find_stream_info(fmt_ctx, NULL); // 2. 找到视频流 & 打开解码器 int v_idx = av_find_best_stream(fmt_ctx, AVMEDIA_TYPE_VIDEO, -1, -1, &codec, 0); AVCodecContext *c_ctx = avcodec_alloc_context3(codec); avcodec_parameters_to_context(c_ctx, fmt_ctx->streams[v_idx]->codecpar); avcodec_open2(c_ctx, codec, NULL); // 3. 解码循环（send/receive 异步模型） AVPacket *pkt = av_packet_alloc(); AVFrame *frame = av_frame_alloc(); while (av_read_frame(fmt_ctx, pkt) >= 0) { avcodec_send_packet(c_ctx, pkt); // 送入压缩包 while (avcodec_receive_frame(c_ctx, frame) >= 0) { // frame->data[0/1/2] = Y/U/V 平面数据 sws_scale(sws, frame->data, frame->linesize, 0, c_ctx->height, rgb->data, rgb->linesize); // YUV→RGB 转换 // 渲染或保存 frame } av_packet_unref(pkt); }
send/receive 模型的本质
FFmpeg 的解码 API 采用异步推送模型：send_packet 和 receive_frame 是独立的——一个 packet 可能产生 0 或多个 frame（B 帧重排序），一个 frame 可能消耗多个 packet。
这反映了编解码器内部的缓冲和重排序机制，不是简单的"输入一个包 → 输出一个帧"。
