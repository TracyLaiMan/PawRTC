LiveKit 本地视频回看方案

复用实时推流通道，实现设备在线时的远程历史录像回看

DataChannel Mp4FileSource 弱网对抗 Pacing
JY1C / Ingenic T41 · LiveKit SFU

方案概述

利用现有 LiveKit 通道实现本地视频远程回看，不需要额外的 HLS/CDN 存储，设备在线即可回看历史录像。

核心思路

回看 = 切换视频源 + 远程控制 + 弱网自适应

复用实时推流的全部通道，无需额外服务。

优势

零额外存储成本视频在设备本地磁盘
复用 LiveKit 信令 + SFU无需 HLS/DASH 服务
浏览器端无需额外播放器同一个 <video> 标签

局限

设备必须在线离线时不可用（家用摄像头场景可接受）
不能精确到帧拖拽WebRTC 本身 1-3s 缓冲延迟
同一时间只能推一个源回看或实时，互斥

架构设计

浏览器点"回看" → LiveKit DataChannel → 设备收到指令 → Mp4FileSource 切文件 → 推流到 SFU → 浏览器订阅观看

指令说明

指令作用设备行为
playback:start开始回看指定文件关闭当前源，打开 MP4 文件，从起点开始推帧
playback:seek跳转到指定位置计算目标 sample index，seek 并请求 IDR
playback:pause暂停推帧停止从缓冲取帧，保持连接
playback:resume恢复推帧从暂停位置继续
playback:speed变速播放调整帧间隔（0.5x/1x/2x/4x）
playback:stop停止回看关闭文件，切回实时源（IMP）

Mp4FileSource 扩展

当前 Mp4FileSource 只支持顺序读帧 + EOF 循环，回看需要增加 seek/pause/resume/speed 能力。

新增接口

class Mp4FileSource : public VideoSource {
public:
    bool open(const std::string& path) override;
    bool readFrame(Frame& frame) override;
    void close() override;

    // 回看扩展
    bool seek(uint64_t targetMs);       // 跳转到指定时间点
    void pause();                        // 暂停推帧
    void resume();                       // 恢复推帧
    void setSpeed(float rate);           // 设置播放倍速（0.5/1/2/4）
    uint64_t getPositionMs() const;      // 当前播放位置
    uint64_t getDurationMs() const;      // 文件总时长

private:
    bool requestNextIdr = false;         // seek 后强制请求 IDR
    bool paused = false;
    float speed = 1.0f;
    std::deque<Frame> frameBuffer;       // 帧预读缓冲
};

seek 实现（Bento4 API）

bool Mp4FileSource::seek(uint64_t targetMs) {
    AP4_UI32 targetSampleIndex = 0;
    AP4_Cardinal sampleCount = videoTrack_->GetSampleCount();
    AP4_UI32 timescale = videoTrack_->GetMediaTimeScale();
    AP4_Timestamp targetTs = (AP4_Timestamp)(targetMs * timescale / 1000);

    // 二分查找最近的 sync sample
    for (AP4_Ordinal i = 0; i < sampleCount; i++) {
        AP4_Sample sample;
        videoTrack_->ReadSample(i, sample);
        if (sample.IsSync() && sample.GetCts() >= targetTs) {
            targetSampleIndex = i;
            break;
        }
    }

    m_sampleIndex = targetSampleIndex;
    requestNextIdr = true;  // 标记需要 IDR 前缀
    frameBuffer.clear();     // 清空预读缓冲
    refillBuffer();          // 重新填充
    return true;
}

void Mp4FileSource::setSpeed(float rate) {
    speed = rate;
    // 帧间隔调整：33ms @30fps × 1/rate
    // 2x → 16.5ms/帧，0.5x → 66ms/帧
}

// 推帧主循环中使用

usleep(frameIntervalUs);

弱网优化

回看场景比实时直播更有优化空间——拥有完整文件，可预读和精确控制节奏。

4.1 发送端优化

帧预读缓冲

提前读 N 帧到内存，网络抖动时不会卡在磁盘 IO。

std::deque<Frame> frameBuffer;  // 缓存 5-10 帧
int bufferLowWatermark = 3;     // 低于 3 帧触发预读

void refillBuffer() {
    while (frameBuffer.size() < 8) {
        Frame f;
        if (!readNextFrameFromFile(f)) break;
        frameBuffer.push_back(std::move(f));
    }
}

bool readFrame(Frame& out) {
    if (frameBuffer.empty()) return false;
    out = std::move(frameBuffer.front());
    frameBuffer.pop_front();
    if (frameBuffer.size() < bufferLowWatermark) refillBuffer();
    return true;
}

Pacing 平滑发送

一帧的 RTP 分片按间隔发出，避免突发拥塞。

void sendFramePaced(std::vector<RtpPacket>& packets, int frameDurationMs) {
    int intervalUs = (frameDurationMs * 1000) / packets.size();
    for (auto& pkt : packets) {
        sendRtpPacket(pkt);
        if (&pkt != &packets.back())
            usleep(intervalUs);
    }
}

RTCP 驱动的自适应帧率

struct PlaybackController {
    int targetFps = 30;
    int64_t frameIntervalMs = 33;

    void onRtcpReceiverReport(int lossRate, int rttMs) {
        if (lossRate > 10 || rttMs > 500) {
            targetFps = max(targetFps - 5, 10);   // 弱网降帧
        } else if (lossRate < 3 && rttMs < 200) {
            targetFps = min(targetFps + 2, 30);   // 恢复帧率
        }
        frameIntervalMs = 1000 / targetFps;
    }
};

IDR 间隔缩短

回看关键帧更密

实时直播: 每 30 帧 → IDR 每 ~1s
回看模式: 每 15 帧 → IDR 每 ~0.5s（编码参数调整或 seek 时强制插 IDR）

seek 后必须立即发送 IDR，浏览器才能解码渲染。

TURN TCP 启用

当前问题

TURNS 失败原因：板端 KVS_CA_CERT_PATH 指向 kvs-webrtc/certs/ca-bundle.crt（Amazon Root CA），不含 Cloudflare/Let's Encrypt 证书。

4.3 回看缓冲策略

用户点播放
  ↓
设备预读 2 秒帧到缓冲区，同时开始推流
  ↓
播放端浏览器积累 ≥1s 缓冲后开始渲染
  ↓
正常播放：设备按实际帧率推，缓冲保持稳定
  ↓
网络变差：设备降帧率/码率，缓冲消耗但不会立即耗尽
  ↓
网络恢复：设备恢复正常帧率，缓冲慢慢回充
  ↓
缓冲耗尽：画面停滞，缓冲重新积累后恢复
