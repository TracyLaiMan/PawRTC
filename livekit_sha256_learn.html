哈希 ≠ 加密
两者服务于完全不同的安全目标。混淆它们会导致致命的安全设计缺陷。
维度哈希 (SHA-256)加密 (AES / RSA)
方向单向双向（可加密可解密）
密钥无有（对称/非对称密钥）
目的完整性校验保密性
输出长度固定 32 字节≥ 输入长度
典型场景SDP指纹、签名前哈希SRTP加密、TLS加密
核心洞察：哈希帮你看穿篡改，加密帮你藏住内容 —— 两者配合才构成完整安全。

在 LiveKit IPC 中：DTLS 用 SHA-256 做证书指纹（看穿中间人），用 AES 做 SRTP 加密（藏住视频流）。两把锁，各管各的门。

LiveKit 三处关键应用
同一个哈希原语，在三处不同的安全层扮演不同角色。
Layer 5 · SDP 指纹计算
SHA-256 — 自签名证书的唯一指纹
对 DTLS 自签名证书的 DER 编码做 SHA-256，得到 sha-256 79:5A:3B:... 格式指纹。指纹通过 TLS 信令通道传递给对端，DTLS 握手完成后与收到的证书哈希比对——如果一致，说明证书未被替换。

风险点：项目代码中 VERIFY_OPTIONAL 跳过了比对，意味着中间人可以替换证书而不被发现。
// Dtls.c: AUTHENTICATION_CERTIFICATE_VERIFY SHA256(DER_certificate) → "79:5A:3B:CF:..." // SDP 中: a=fingerprint:sha-256 79:5A:3B:CF:... // 握手后应比对，但 VERIFY_OPTIONAL 跳过了
Layer 4 · ECDSA 签名前哈希
SHA-256 — 签名的"压缩器"
ECDSA 不直接对消息签名，而是先 SHA-256(message) → 对 32 字节哈希值签名。为什么？
1. ECC 运算量与输入长度无关——签名 1MB 文件和签名 32 字节哈希，计算量一样。
2. ECC 只接受固定大小的输入（P-256 曲线要求 256 bit）。
3. SHA-256 把任意长度映射到固定长度，充当"压缩器"角色。
// Ecdsa.c — 签名流程 digest = SHA256(message) // 任意长度 → 32字节 sig = ECDSA_sign(digest, private_key) // 验证端：SHA256(message) → ECDSA_verify(sig, public_key)
Layer 1/3/6 · HMAC 底层引擎
SHA-256 — HMAC 的内部引擎
HMAC-SHA256 = SHA256(opad ∥ SHA256(ipad ∥ message))。SHA-256 在 HMAC 内部执行两次哈希运算：
• 内层：SHA256(ipad ∥ message) → 32 字节中间值
• 外层：SHA256(opad ∥ 中间值) → 32 字节最终 HMAC

JWT 签名（L1）、STUN MESSAGE-INTEGRITY 用 HMAC-SHA1（L3）、SRTP auth tag 用 HMAC-SHA1-80（L6）——底层全是 SHA 系列。
// HMAC-SHA256 内部执行两次 SHA-256 inner = SHA256(key_xor_ipad ∥ msg) hmac = SHA256(key_xor_opad ∥ inner) // HMAC-SHA256 用于 JWT · HMAC-SHA1 用于 ICE/SRTP
