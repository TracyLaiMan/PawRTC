1. BLE 协议栈的三层分块与物理边界

理解 BLE 协议栈的第一件事不是学每层的协议——而是搞清楚哪层在芯片硬件里跑、哪层在主机 OS 里跑、哪层之间的通信你可以在 Wireshark 中看到。

Controller (控制器)

BLE 芯片硬件 — PHY + LL + HCI 下半层

物理位置：BLE 芯片内部的硬件状态机 + 嵌入式固件（如 nRF52 的 SoftDevice、TI CC2540 的 BLE Stack ROM）。
包含的层：PHY（射频收发、GFSK 调制、2.4GHz 信道跳频）→ Link Layer（状态机、PDU 构建/解析、CRC、ARQ 重传、加密/解密、跳频控制）。
特性：硬实时——LL 层的 150μs T_IFS 响应必须由硬件保证（CPU 调度延迟 >100μs 无法满足）。Controller 通常运行在裸金属或小型 RTOS 上——不跑 Linux。

Host (主机)

主处理器 OS — L2CAP + ATT + SM + GAP + GATT

物理位置：运行在主 CPU 的操作系统协议栈中（Linux BlueZ、Android Bluetooth Stack、Zephyr BLE Host）。
包含的层：L2CAP（逻辑链路复用）→ ATT（属性协议——GATT 的基础）→ SM（安全管理——配对/加密密钥管理）→ GAP（角色定义——Central/Peripheral/Broadcaster/Observer）→ GATT（属性数据库——Service/Char/Descriptor 的树形结构）。
特性：非实时——可以在 Linux 内核中运行（BlueZ）或在用户态运行。GATT Discovery 慢（几秒）完全可接受——不像 LL 层需要 150μs 响应。

HCI — 分割线

Host Controller Interface — 物理或虚拟的桥

物理 HCI：通过UART (H4/H5 协议) 或 USB 连接 Host CPU 和 BLE 芯片（如树莓派用 UART 连 nRF52840）。Android HCI Snoop Log 抓的就是这层。
虚拟 HCI：Host 和 Controller 在同一颗芯片上（如手机 SoC 内置 BLE）——通过共享内存通信，不走物理总线。但 HCI 协议层的格式完全相同——HCI Log 在物理和虚拟场景下完全一样。
HCI 的作用：标准化 Host↔Controller 的通信格式——任何符合 HCI 规范的 Host 可以和任何符合 HCI 规范的 Controller 对接——类似于 USB 的标准化。

2. PHY 物理层 — 无线电波的"字母表"

PHY 层回答的是最底层的问题：0 和 1 怎么通过无线电波表示？在哪个频率上发？用什么调制方式？

40 个物理信道

2.402~2.480 GHz, 2MHz 间隔, 3 广播 + 37 数据

BLE 将 ISM 频段分为 40 个信道，编号 0~39。信道 k 的中心频率 = 2402 + k×2 MHz。
CH37 (2402), CH38 (2426), CH39 (2480)——这三个故意选在 WiFi CH1/6/11 的间隙中——最小化 WiFi 干扰。
每个信道带宽 2MHz——BLE 使用 GFSK 调制（高斯频移键控）——每个符号表示 1 bit（1M PHY）或 2 bits（2M PHY 用 π/4-DQPSK）。

三种 PHY 模式

LE 1M · LE 2M · LE Coded — 速度 vs 距离

LE 1M (必选)：1 Msps 符号率 → ~1 Mbps 原始速率。所有 BLE 设备必须支持。
LE 2M (BLE 5.0)：2 Msps → ~2 Mbps。灵敏度 -3dB（同距离下信号更弱）。适合短距离高速传输（OTA、音频）。
LE Coded (BLE 5.0, 可选)：1 Msps 但每个 bit 编码为 2 或 8 个符号（FEC 前向纠错）→ 125 Kbps (S=8) 或 500 Kbps (S=2)。灵敏度 +6dB——通信距离翻倍（长距离模式，适合 IoT 传感器、资产跟踪）。

空中包格式 (PHY 视角)

Preamble + Access Address + PDU + CRC

Preamble (1-2 字节)：01010101... 交替序列——接收机用它做频率同步和 AGC 校准。1M PHY=1B, 2M PHY=2B。
Access Address (4 字节)：广播信道固定为 0x8E89BED6——所有 ADV_IND 都用这个值。连接建立后，每个连接由 Master 随机生成一个唯一的 Access Address（需满足随机性要求——不能全 0/全 1/连续 6 个相同 bit/与 0x8E89BED6 差距太小）。
PDU (2~257 字节)：实际承载的链路层数据。
CRC (3 字节)：CRC-24 校验——多项式 x^24+x^10+x^9+x^6+x^4+x^3+x+1。

3. Link Layer — 设备状态机与空中交互

LL 层是 BLE 协议栈的核心引擎——管理设备状态机、PDU 类型、ARQ 重传、加密、跳频。这是 BLE 协议规范中最复杂的部分。

LL 状态机 — 5 个状态 + 1 个瞬态

Standby → Advertising/Scanning/Initiating → Connection

Standby (待机)：默认状态。不发送、不接收。所有状态的起点和终点。
Advertising (广播)：Peripheral 在 3 个广播信道上周期性发送 ADV_IND。从 Standby 进入——设置 Adv Interval+Adv Data→开始广播。收到 CONNECT_IND→退出进入 Connection。
Scanning (扫描)：Central 在 3 个广播信道上监听 ADV_IND。被动扫描（只收不发）或主动扫描（发 SCAN_REQ→收 SCAN_RSP）。
Initiating (发起连接)：Central 收到目标 ADV_IND→在同一信道上发 CONNECT_IND→如果外设收到→双方进入 Connection。这是瞬时状态。
Connection (已连接)：双方按 Connection Interval 定期通信。此状态下可以发 LL Control PDU（参数更新/加密/Channel Map/PHY 更新/断连）和 LL Data PDU（承载 L2CAP 数据）。
Isochronous (BLE 5.2+, 可选)：LE Audio 的等时信道——固定的 ISO Interval——无重传（与 SCO 类似但走 BLE）。

LL PDU 类型 — 两大类

LL Data PDU (承载上层数据) + LL Control PDU (链路管理)

LL Data PDU：LLID=0x01 (L2CAP 分片续) 或 0x02 (L2CAP 起始或完整包)。承载 L2CAP 层的数据。Data PDU 受 ARQ 保护——接收端 CRC 错误→NACK→发送端重传（最多重传到 Supervision Timeout 超时）。
LL Control PDU：LLID=0x03。用于链路管理——包括：LL_CONNECTION_PARAM_REQ/RSP（参数更新）、LL_TERMINATE_IND（断连+原因码）、LL_ENC_REQ/RSP（加密握手）、LL_FEATURE_REQ/RSP（特性交换——声明支持哪些可选特性）、LL_PHY_REQ/RSP（PHY 切换——2M↔1M↔Coded）、LL_CHANNEL_MAP_IND（更新可用信道列表）。Control PDU 不重传——发一次，如果对方没收到→由上层处理超时。

空包 (Empty PDU)：LLID=0x00, Payload=0 字节。Connection Event 中无数据时发送——维持连接的"心跳"。每个 Connection Event 必须至少有一对空包交换（Master→Slave + Slave→Master）。如果连续丢失到 Supervision Timeout→断连。

// LL 层的包过滤与 CRC —— 一条空中包的生命周期 发送端 (Controller 内部): L2CAP 数据到达 → LL 封装为 Data PDU (LLID+SN+NESN+Payload) → 追加到发送队列 (每个 Connection Event 取出发送) → Baseband: 计算 CRC24 → 拼接 Preamble+AccessAddr+PDU+CRC → GFSK 调制 → RF 发射 接收端 (Controller 内部): RF 信号 → GFSK 解调 → Preamble 检测 → Access Address 匹配(不匹配→丢弃) → CRC24 校验 → CRC 错 → NAK → 等发送端重传 → CRC 对 → ACK → 提取 LLID: LLID=0x01/0x02 (Data) → 去 LL 解密(AES-CCM) → 重组 L2CAP SDU → 上传 Host LLID=0x03 (Control) → 解析 Opcode → 执行相应的链路管理动作 LLID=0x00 (Empty) → 无操作 (心跳确认) 为什么空中抓包中有时看到"重传"? 发送端发了 SN=3 → 接收端 CRC 错 → NAK → 发送端重传 SN=3 (同一个包, SN 不变) → 在 Wireshark 中看到连续两个 SN=3 的包 → 第一个 CRC 错(但 nRF Sniffer 看不到 CRC 错误!) → 只有 Ellisys 才能看到第一个包的 CRC Error 标注

4. HCI — Host 与 Controller 之间的标准语言

HCI 是理解"为什么我的代码行为与芯片行为看起来不一致"的关键——HCI Log 告诉你 Host 给 Controller 发了什么命令、Controller 回了什么事件。

HCI Command (Host → Controller)

OpCode(2B) + ParamLen(1B) + Params(N)

Host 通过 HCI Command 控制 Controller 的一切行为——没有 HCI Command，Controller 什么都不会做。
典型命令：LE Set Scan Enable (OpCode 0x200C——开启扫描)、LE Set Advertising Enable (0x200A——开启广播)、LE Create Connection (0x200D——发起连接)、LE Read Buffer Size (0x2002——查询 ACL 包大小限制)、Reset (0x0C03——复位 Controller)。

HCI Event (Controller → Host)

EventCode(1B) + ParamLen(1B) + Params(N)

Controller 通过 HCI Event 向 Host 报告事件。这是异步的——Controller 随时可能发送 Event。
典型事件：Command Complete (0x0E——命令执行完毕)、Command Status (0x0F——命令已接受但未执行完)、LE Meta Event (0x3E——所有 BLE 特定事件的容器)：LE Connection Complete（连接成功/失败+Status 错误码）、LE Advertising Report（扫描到设备+RSSI+AdvData）。

ACL Data (双向)

Handle(2B) + PB/BC Flag(2B) + DataLen(2B) + Data(N)

ACL Data 包承载实际的L2CAP + ATT 数据。双向传输——Host→Controller 发送给外设，Controller→Host 接收自外设。
PB Flag (Packet Boundary)：0b00 = L2CAP 起始或完整包；0b01 = L2CAP 分片续（L2CAP SDU 被拆分为多个 ACL 包）。
BC Flag：广播标志（LE 不使用——为 BR/EDR 保留）。
HCI Log 中最有价值的分析：ACL Data 包的时序——ATT 层的一次 Write Request → Write Response 往返 = ACL Data(WriteReq) → ACL Data(WriteRsp) 两个包的间隔 = 1 个 Connection Interval。

5. L2CAP — 多路复用与分片重组

L2CAP 在两个 BLE 设备的 LL 层之上提供多个逻辑通道（CID 标识），每个通道承载不同的上层协议（ATT、SM）。同时负责大 SDU 的分片重组。

CID (Channel Identifier)

16-bit 逻辑通道号 — 在一对设备间区分不同的上层协议

固定 CID：0x0004 = ATT（GATT 数据走这个通道）、0x0006 = SM（Security Manager——配对/加密密钥分发）、0x0005 = L2CAP LE Signaling（L2CAP 自己的信令——用于 Credit Based Flow Control 和 Connection Parameter Update）。
动态 CID (0x0040~0x007F)：LE Credit Based Flow Control 模式下分配的面向连接通道——用于 LE Audio 的等时数据传输。
关键认识：ATT 和 SM 是两个独立的 CID——配对过程中的 Pairing Request/Response 走 CID=0x0006(SM)，配对的密钥分发（LTK）也走 SM。GATT Discovery 和数据交互走 CID=0x0004(ATT)。两者在传输层上完全独立——一个通道阻塞不影响另一个。

SAR — Segmentation and Reassembly

L2CAP SDU 自动拆分为适配 LL 的报文——对 ATT 层完全透明

ATT 层可以发任意大小的 PDU（如 512 字节的 Read By Group Type Response）——L2CAP 将 512 字节的 SDU 封装后传给 LL 层。LL 层的 Data PDU 最大 Payload 受 DLE 限制（27~251 字节）。L2CAP 自动将 512 字节 SDU 分片为多个 LL Data PDU——每个 LL PDU 带相同的 CID 和 L2CAP 头（Length=512）。
接收端 L2CAP 自动重组——收集同一 CID 的 LL Data PDU，根据 L2CAP Length 判断是否收齐——收齐后上传 ATT 层。ATT 层完全不知道分片发生过。
LL 层的重传与 L2CAP 分片的交互：如果 LL PDU#2 在空口上丢包→LL 层 ARQ 重传→重传成功→L2CAP 继续收集→上传。LL 层的重传保证了分片链的可靠性。但如果重传超过 Supervision Timeout→整个 SDU 丢失→上层 ATT 超时。

6. ATT (属性协议) + GATT (通用属性配置文件)

这是 BLE 应用开发接触最多的两层——ATT 定义了"怎么读怎么写"的操作语义，GATT 定义了"数据在 Server 端如何组织成树形结构"。

ATT — 属性协议 (协议层)

Client-Server 模型 · 请求-响应 · 严格的串行执行

ATT 定义了6 种 PDU 类型的方法：Read Request/Response（读属性值）、Write Request/Response（写属性值——需确认）、Write Command（写属性值——无需确认）、Handle Value Notification（Server 主动推送——无需确认）、Handle Value Indication/Confirmation（Server 主动推送——需确认）、Read By Type Request/Response 和 Find Information Request/Response（用于 GATT Discovery——遍历属性数据库）。
核心约束：ATT 不允许多个请求同时进行——Client 发了 Read Request → 必须等 Read Response → 才能发下一个请求。Write Command 和 Handle Value Notification 不受此限制（无响应=不需要等待）。这就是为什么 GATT 操作要排队串行化。

GATT — 属性数据库 (数据层)

Service → Characteristic → Descriptor 三级树形结构

GATT Server 维护一个Attribute 表——每个 Attribute = [Handle(2B) + UUID(2/4/16B) + Permissions + Value]。Attribute 按 Handle 升序排列。
Service Declaration (UUID=0x2800)：Attribute 类型 = 0x2800，Value = [Service UUID]。标记一个 Service 的起始 Handle。
Characteristic Declaration (UUID=0x2803)：Attribute 类型 = 0x2803，Value = [Properties(1B) + Value Handle(2B) + UUID]。标记一个 Characteristic——Properties 声明了 Read/Write/Notify/Indicate 权限，Value Handle 指向实际存储数据的 Attribute。
Characteristic Value：实际的数据存储位置。App 读写的就是这个 Handle。
Descriptor：附加元数据。最关键的是 CCCD (UUID=0x2902, Client Characteristic Configuration Descriptor)——2 字节：0x0000=关闭 Notify/Indicate, 0x0001=启用 Notify, 0x0002=启用 Indicate。

7. SM (Security Manager) — 配对与加密

SM 负责设备身份认证和链路加密——通过独立的 L2CAP CID=0x0006 通信，与 GATT 数据通道完全分离。

SM 的三个阶段

特性交换 → 密钥生成 → 密钥分发

Phase 1：交换 Pairing Request/Response——IO Capabilities（有无键盘/显示器）、OOB 支持、AuthReq（Bonding/MITM/SC 标志位）、密钥长度、要分发的密钥类型。
Phase 2 (LE Legacy)：基于 TK (Temporary Key) 通过 AES-CMAC 生成 STK (Short Term Key)。Just Works 时 TK=0——安全脆弱。
Phase 2 (LE Secure Connections)：ECDH (P-256) 密钥交换 + AES-CCM 认证——即使窃听也无法破解。Just Works 仍然无 MITM 保护但防被动窃听。
Phase 3：通过加密链路分发 LTK (长期密钥, 128-bit, 用于后续连接加密)、IRK (身份解析密钥, 用于 RPA 地址)、CSRK (签名密钥, 用于数据签名)。

加密：链路层的 AES-CCM

SM 协商 LTK → LL 层用 AES-CCM 执行数据加密

SM 只负责生成和交换密钥——不负责实际数据加密。实际加密在 LL 层执行：LL_ENC_REQ/RSP 握手→双方确认使用相同的 LTK→LL 层在每个 Data PDU中执行 AES-CCM (Counter with CBC-MAC) 加密和认证。
加密后每个 Data PDU 尾部增加 4 字节 MIC (Message Integrity Check)——接收端验证 MIC → 错误→丢弃包→MIC Failure 可能导致 LL_TERMINATE_IND (Error=0x0E)。
加密切换的时刻：LL_START_ENC_REQ → LL_START_ENC_RSP → 从下一个 Connection Event开始，所有 Data PDU 加密。

Bonding — 持久化信任

配对一次 → 存储密钥 → 后续自动加密

Bonding 标志位设置后→配对完成后双方将 LTK/IRK/CSRK + 对端身份信息（BD_ADDR + 地址类型）持久化存储（手机侧存到 Secure Enclave/KeyStore，外设侧存到 Flash）。
下次连接→加密阶段时直接从存储中读取 LTK→加密握手（~30ms）→无需用户交互。
RPA (Resolvable Private Address)：Bonding 后外设可以使用 RPA 作为广播地址（每次上电生成新的随机地址）→ 手机通过存储的 IRK 运行哈希解析函数→确定这确实是之前 Bonding 过的设备→建立连接+自动加密。实现了防追踪+无缝自动连接。

HCI Layer (Host → Controller) 封装 HCI ACL Data 头: [Handle=0x0040][PB=0b10(起始)][BC=0b00][DataLen=13] 通过 UART(H4) 或 USB 发往 Controller 在 Android HCI Snoop Log 中会看到这条 ACL Data 5. Link Layer (Controller — 固件) 收到 ACL Data → L2CAP SDU(13B) + LL 头 → 排队等待下一个 Connection Event → Connection Event 时刻到 → 取出包 → 加密(AES-CCM, LTK) → 构造 LL Data PDU: [LL Header(2B): LLID=0x02, SN=3, NESN=5, MD=0] [Payload(13B+4B MIC)] → 交给 Baseband 6. Baseband + PHY (Controller — 硬件) 构造空中包: [Preamble(1B)][AccessAddr(4B)][LL_PDU(17B)][CRC24(3B)] 总大小 = 1+4+17+3 = 25 字节 → GFSK 调制 → 2.4GHz 射频发射 → 在空中占用约 200μs (2M PHY) 或 400μs (1M PHY) // 总计: 6 字节用户数据 → 25 字节空中包 → 约 76% 是协议开销

9. BR/EDR vs BLE 协议栈对照

两者共用 L2CAP 层以上完全相同，但 PHY/LL 层完全不同——这是理解 Dual-Mode 芯片设计的关键。
