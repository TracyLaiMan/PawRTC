1. 角色分配与物理信道

在一切开始之前，手机和外设必须先确定自己的角色——Central 还是 Peripheral。这个角色决定了谁发起连接、谁控制连接参数、以及物理信道如何使用。

Central (手机)

主动扫描 → 发起连接 → 控制连接参数

Central 是"主人"——负责扫描（监听广播信道）、发起连接（发送 CONNECT_IND）、管理连接参数（Connection Interval 等由 Central 最终决定）。典型设备：智能手机、平板、PC。
BLE 连接建立后，Central 在每个 Connection Event 的起始时刻首先发送包（master role）——Peripheral 必须在 150μs 内响应。

Peripheral (BLE 设备)

被动广播 → 接受连接 → 请求参数更新

Peripheral 是"仆人"——负责广播（定期在 3 个广播信道上发送 ADV_IND）、接受连接（收到 CONNECT_IND 后进入连接态）、可以请求但不能决定连接参数（Peripheral 发 LL_CONNECTION_PARAM_REQ，Central 决定是否接受）。典型设备：项圈、传感器、IPC 相机。
连接后 Peripheral 在每个 Connection Event 中等待 Central 先发，然后响应。

40 个物理信道

3 个广播信道 + 37 个数据信道

BLE 将 2.4GHz ISM 频段分为 40 个 2MHz 宽的信道（编号 0~39）。信道 37(2402MHz)、38(2426MHz)、39(2480MHz) 专门用于广播/扫描/连接建立——选择这三个频率是因为它们避开了 WiFi 最常用的信道 1/6/11 的干扰。
连接建立后，通信在 37 个数据信道（0~36）上跳频进行——每个 Connection Event 使用不同信道（由 Central 的 Channel Map 和 hopIncrement 决定）。

广播信道 vs 数据信道

建立连接前 vs 建立连接后

广播信道：Adv Interval 间隔周期性发送（20ms~10.24s）。三个信道轮流发（或同时发）同一包——提高被发现的概率。
数据信道：Connection Interval 频率周期性通信。自适应跳频 (AFH) ——检测到某个信道干扰严重 → 从 Channel Map 中移除 → 跳到干净信道。

空中包时序

每个 Connection Event 的精确时序

Central 在约定时刻（connInterval 的整数倍）发送第一个包 → Peripheral 在 150μs ± 2μs 内必须响应（T_IFS）→ Central 可继续发 → 直到双方都没有更多数据或 Event 结束。
每个 Event 持续至少 connSlaveLatency × connInterval（如果 Peripheral 无数据可发）。Central 在每个 Event 必须发送（至少空包轮询确认 Peripheral 存活）。

2. 阶段 1: Advertising — 外设广播自己的存在

这是整个连接生命周期的起点——Peripheral 在三个广播信道上周期性发送 ADV_IND 包，向世界宣告"我在这里，可以连接我"。

ADV_IND 空中包的内容 — 31 字节能装什么？

Address (6B) + AdvData (最多 31B) — BLE 4.x 限制

ADV_IND PDU 的 payload = AdvA (6 字节广播地址) + AdvData (0~31 字节广播数据)。31 字节的 AdvData 由多个 AD Structure 组成，每个 AD Structure = [Length:1B][AD Type:1B][AD Data: N 字节]。
典型广播数据：Flags (3B: 声明 BR/EDR 不支持 + LE General Discoverable)、Complete Local Name ("IPC-CAM-001", ~12B)、TX Power (2B: -4dBm)、Service UUIDs (2~16B: 0x180A Device Info, 0x180F Battery, 自定义 Service)。
31 字节非常紧张——名字长了 UUID 就放不下。BLE 5.0 引入 Extended Advertising (最大 1650B) 解决了这个问题。

Advertising Interval — 广播频率的选择

20ms ~ 10.24s — 发现速度 vs 功耗的博弈

20ms (快速广播)：手机开启扫描后 ~30ms 内就能发现。功耗极高（不休眠）。适合设备配网模式（用户按了配网键后 30 秒快速广播）。
100ms~1s (省电广播)：手机可能需要 1-3 秒发现。功耗低。适合已配网设备日常存活广播。
Apple 的诡异规则：iOS 后台扫描时只对特定 Adv Interval 敏感（30ms 倍数）。如果设备广播间隔不是 30ms 的倍数 → 概率性不被 iOS 发现。
广播包在三个信道上顺序轮流发送（不是同时发三个信道——芯片只有一套 RF 收发器）。

// ADV_IND 空中包完整结构 (链路层 PDU, BLE 4.x) // LL Header (2B) + Payload (6~37B) + CRC (3B) [LL Header: 2 字节] Byte 0: PDU Type = 0x00 (ADV_IND) // 4bit Type + 1bit RFU + 1bit ChSel + 2bit TxAdd/RxAdd Byte 1: Payload Length = 6 + AdvData_len // 6 (AdvA) + N (AD Structures) [Payload: 6 + N 字节] Byte 2-7: AdvA (6B) // 广播地址, 随机或 Public // 随机地址分两种: Static (上电不变) vs Private (定时更换防追踪) Byte 8-N: AdvData (最多 31B, AD Structures) // 由多个 Length-Type-Value 组成 // AD1: Flags Byte 8: Length = 0x02 Byte 9: AD Type = 0x01 (Flags) Byte 10: Flags = 0x06 (LE General Discoverable | BR/EDR Not Supported) // AD2: Complete Local Name Byte 11: Length = 0x0C (12) Byte 12: AD Type = 0x09 (Complete Local Name) Byte 13-24: "IPC-CAM-001" (UTF-8) // AD3: TX Power Level Byte 25: Length = 0x02 Byte 26: AD Type = 0x0A (TX Power) Byte 27: TX Power = 0xFC (-4 dBm, 有符号 int8) // AD4: Service UUID (16-bit, incomplete list) Byte 28: Length = 0x03 Byte 29: AD Type = 0x02 (Incomplete List of 16-bit Service UUIDs) Byte 30-31: UUID16 = 0xFFE0 (Custom Service) // Total AdvData: 2+2+12+2+3+3 = 24 字节 (在 31 字节限制内) [CRC: 3 字节] CRC24 覆盖 LL Header + Payload. 多项式: x^24 + x^10 + x^9 + x^6 + x^4 + x^3 + x + 1

3. 阶段 2: Scanning — 手机如何发现设备

手机打开 BLE 扫描 → 底层控制器在三个广播信道上轮询监听 → 收到 ADV_IND → 解析广播数据 → 过滤（按 Service UUID 或 Name）→ 上报给 App。

被动扫描 (Passive)

只监听不发送 — 纯接收模式

手机只接收广播包，不发送任何数据——外设完全不知道有人在扫描它。功耗最低。广播包中只能拿到 AdvA + AdvData (最多 31B)。如果 App 需要设备名称不在广播数据中（放在 Scan Response 里）→ 需要用主动扫描。

主动扫描 (Active)

收到 ADV_IND 后发送 SCAN_REQ → 外设回复 SCAN_RSP

手机收到 ADV_IND → 可以在同一广播信道上立即发送 SCAN_REQ（包含 ScanA 地址）→ 外设收到 SCAN_REQ → 回复 SCAN_RSP —— 额外的 31 字节数据（通常放完整设备名、TX Power、Service UUID 等 ADV_IND 中放不下的数据）。
ADV_IND 31B + SCAN_RSP 31B = 总共 62 字节广播数据——这是 BLE 4.x 在不建立连接的前提下能获取的全部设备信息。

扫描参数

Scan Window / Scan Interval · 占空比

Scan Interval：两次扫描之间的间隔（如 100ms）。Scan Window：每次扫描持续多长时间（如 50ms）。占空比 = Window / Interval（50% 意味着手机一半时间在扫、一半在休眠）。
影响：Window 越大 → 发现速度越快，但功耗越高。iOS 后台扫描强制低占空比（~10%）；Android 各厂商自定义。
App 收到 ADV_IND → 回调 onScanResult(device, rssi, scanRecord)——不是每个 ADV_IND 都上报（Android 默认 1-2 秒聚合一次）。

手机从"打开蓝牙"到"App 看到设备"的完整时间线
T+0ms App 调用 BluetoothLeScanner.startScan() T+5ms Bluetooth 协议栈向 Controller 发 LE Set Scan Enable (Enable=1, Filter_Duplicates=0) T+10ms Controller 开始在 CH37/38/39 上轮询监听 T+45ms 外设在 CH37 上发 ADV_IND (如果 Adv Interval=30ms, 每 30ms 在三个信道各发一次) T+46ms Controller 收到 ADV_IND → 解析 AdvA, AdvData, RSSI=-52dBm T+47ms Controller 发 SCAN_REQ (主动扫描) → 外设发 SCAN_RSP (额外 31B 数据) T+50ms Controller 通过 HCI (LE Advertising Report Event) 上报给 Host T+55ms Android BLE Stack 处理 → 匹配 App 注册的 ScanFilter T+60ms App.onScanResult(device, rssi, scanRecord) 被回调 → 总计: 从 startScan 到 onScanResult ~55ms (最快), 典型 100-500ms

4. 阶段 3: 建立连接 — CONNECT_IND 握手

手机决定连接某个设备 → 在同一广播信道上发送 CONNECT_IND → 外设收到后立即停止广播、进入连接态。这是整个过程中时序要求最严格的一步。

CONNECT_IND 里携带了什么？

6 个关键参数 — 定义了此后所有 Connection Event 的时序

InitA (6B)：发起者 (手机) 的地址。AdvA (6B)：目标外设的地址——外设看到自己的地址才知道"这是在叫我"。Access Address (4B)：随机生成的 32-bit 值——此后所有数据信道上的包都用这个 Access Address 标识这个连接（替代广播信道上的固定 Access Address 0x8E89BED6）。
CRCInit (3B)：CRC 计算的初始值——每个连接独立。
WinSize (1B)：传输窗口大小 = WinSize × 1.25ms——外设在这个窗口内监听第一个 Connection Event。
WinOffset (2B)：传输窗口偏移 = WinOffset × 1.25ms——外设在 CONNECT_IND 之后等待 WinOffset × 1.25ms 才开始监听。
Interval (2B)：Connection Interval = Interval × 1.25ms。
Latency (2B)：Slave Latency——外设可以跳过的 Event 数量。
Timeout (2B)：Supervision Timeout = Timeout × 10ms——超过此时间无通信 → 判定断连。
ChM (5B)：Channel Map——标记 37 个数据信道中哪些可用。
Hop (1B)：跳频增量——下一个 Connection Event 的信道 = (当前信道 + Hop) % 37。
SCA (3bit)：Sleep Clock Accuracy——外设休眠时钟精度 (ppm)，决定接收窗口宽度。

为什么 CONNECT_IND 一发外设就"消失"了？

外设立即停止广播 → 切换到数据信道 → 等待第一个 Connection Event

外设收到 CONNECT_IND → ① 立即停止所有广播活动（不再发 ADV_IND）② 根据 CONNECT_IND 中的参数计算第一个 Connection Event 的时刻 = CONNECT_IND 结束 + WinOffset × 1.25ms + transmitWindowDelay ③ 在计算的时刻打开接收窗口（WinSize × 1.25ms），监听 Access Address 匹配的包。
手机侧：发完 CONNECT_IND 后，在第一个 Connection Event 时刻发送一个空包（LL Data PDU with Length=0）——相当于"ping"外设。外设在 150μs 内回复空包 → 连接正式建立。双方记录对方的 Access Address + CRCInit → 此后所有数据信道上的包都用这些参数。

⚠ 连接失败的最常见原因 — 时序问题
1. WinOffset 太大：外设等了太久 → 在此期间手机以为连接失败 → App 收到 onConnectionStateChange(DISCONNECTED)。缩短 WinOffset 可加速。
2. 外设时钟精度不够：SCA (Sleep Clock Accuracy) 声明 50ppm 实际 500ppm → 外设醒来时间漂移 → 错过手机的发送窗口 → 连接失败。这是廉价 BLE 芯片的常见问题。
3. 干扰：第一个 Connection Event 的信道恰好被 WiFi 严重干扰 → 手机连发数包无 ACK → supervisionTimeout → 断连。解决：Channel Map 避开已知干扰信道。

5. 阶段 4: Service Discovery — GATT 数据库遍历

连接建立后，手机只知道外设的 MAC 地址和广播名——对外设能提供什么服务（Service/Characteristic/Descriptor）一无所知。GATT Discovery 就是手机的"探索"过程。

为什么需要 Discovery？

Handle 是动态分配的——每次连接可能不同

GATT 数据库中的每个 Attribute 由一个 16-bit Handle 标识。Handle不是固定的——每次连接、不同固件版本，Handle 都可能变化。手机必须通过 Discovery 过程动态查找想要访问的 Characteristic 的 Handle。
例外：如果 App 缓存了 Handle（上次连接时记录的），且外设未发生"Service Changed"（Handle 重排）→ 可以跳过 Discovery——直接使用缓存的 Handle 读写。

四层递进遍历

Services → Characteristics → Descriptors → 逐个 Handle

Step 1: Discover All Primary Services —— ATT Read By Group Type Request (UUID=0x2800)。外设返回每个 Service 的起止 Handle 范围 + UUID。
Step 2: Discover Characteristics —— ATT Read By Type Request (UUID=0x2803) 对每个 Service 范围内的 Handle。外设返回每个 Characteristic 的 Handle + Properties(Read/Write/Notify/Indicate) + Value Handle。
Step 3: Discover Descriptors —— ATT Find Information Request 遍历 Characteristic 的 Descriptor（CCCD 等）。
Step 4: 缓存 Handle —— App 记录下需要的 Handle → 后续直接读写。

时间代价

2~5 秒 (iOS) — 连接中最耗时的阶段

Discovery 需要多次 ATT 请求-响应往返——每次往返 = 至少 1 个 Connection Interval (30ms default)。一个典型的 GATT 数据库（5 个 Service × 3 个 Characteristic）→ 需要 ~20 次 ATT 请求 → 20 × 30ms = 600ms。加上 iOS 的 throttling → 实际 2-5 秒。
优化的关键：首次连接后缓存所有 Handle → 后续重连跳过 Discovery → 连接建立到可通信从 3 秒降到 100ms。仅在外设发送 Service Changed Indication (Handle 0x0001) 时重新 Discovery。

6. 阶段 5: 数据传输 — GATT 操作的完整流程

Discovery 完成后，手机拿到了所有 Handle → 可以开始真正有意义的数据交互。四种 GATT 操作各有不同的时序、确认机制和适用场景。

GATT Write Request (有响应)

1 Request → 必须等 1 Response → 才能发下一个

用途：命令下发（拍照、OTA 控制、参数设置）。
时序：手机发送 ATT_WRITE_REQ (Handle + Value) → 外设执行 → 发送 ATT_WRITE_RSP (或 ATT_ERROR_RSP) → 手机收到 RSP 之前不能发下一个 Write Request。
延迟：1 次往返 = 1 个 Connection Interval (30ms default)。
可靠性：ATT 层自动重传（如果 RSP 超时）。30s supervision timeout 内无响应 → 断连。
适用：需要确认执行结果的命令。不适合：高频数据（每 30ms 才完成一次 → 33 次/秒 → 吞吐极低）。

GATT Write Without Response (Write Command)

发送即忘 — 无 Response — 可连续发送

用途：OTA 数据块、音频流、高频传感器数据。
时序：手机发送 ATT_WRITE_CMD (Handle + Value) → 不等 Response → 立即发送下一个 Write Command。一个 Connection Event 内可连发 3-6 个（取决于 MTU/DLE/PHY）。
可靠性：无 ATT 层确认——包可能在控制器队列满时静默丢弃（应用层不知道！）。OTA/音频需要在应用层实现 Sequence Number + ACK。
适用：需要高吞吐的流式数据。不适合：需要确认每个操作是否成功的命令。

GATT Notify (外设主动推送, 无确认)

外设 → 手机, 不需要手机回复确认

用途：状态上报（电池、温度、传感器数据）、数据流输出。
前提：手机必须先写 CCCD (Handle 0x2902) 为 0x0001 启用 Notify。
时序：外设在 Connection Event 中主动发送 ATT_HANDLE_VALUE_NTF → 手机收到后不需要回复 → 外设可以继续发。
限制：Notify 只能由外设→手机（Peripheral → Central）。手机→外设必须用 Write。

GATT Indicate (外设主动推送, 需确认)

外设 → 手机 → 手机必须回复 ATT_HANDLE_VALUE_CFM

用途：关键状态变更通知（如 OTA 完成、错误告警——必须确保 App 收到）。
时序：外设发送 ATT_HANDLE_VALUE_IND → 手机必须回复 ATT_HANDLE_VALUE_CFM → 外设收到 CFM 才能发下一个 Indicate。
限制：和 Write Request 一样——必须等待确认 → 吞吐低。仅用于关键通知。

⚠ 同一时刻只能有一个未完成的 GATT 操作——这是 BLE 应用开发中最容易被忽视的约束
ATT 协议不支持 pipeline。如果 App 发了 Write Request → 必须等 Write Response → 才能发下一个 Write Request 或 Read Request。如果 App 在两个线程中同时发 → 第二个会失败（返回 Busy 或静默排队）。解决方案：应用层实现GATT 操作队列——所有 GATT 操作串行化，前一个完成（收到 callback）才发送下一个。Read/Write Request 之类的有响应操作共享同一个队列；Write Command 和 Notify 不受此限制（无响应 = 无需排队等待）。

7. 安全配对 — 加密与认证

安全配对是可选的——但如果你的设备需要传输敏感数据或防止未授权访问，则必须配对。BLE 4.2+ 推荐 LE Secure Connections。

配对三阶段

Phase 1: 能力交换 · Phase 2: 密钥生成 · Phase 3: 密钥分发

Phase 1 (Pairing Feature Exchange)：双方交换 IO Capabilities（有显示屏？有键盘？）、OOB 支持、AuthReq（是否需要 Bonding/MITM 保护）。决定使用哪种配对方法。
Phase 2 (Key Generation)：根据 Phase 1 协商的方法生成 STK (LE Legacy) 或 LTK (LE Secure Connections)。Just Works（无 MITM 保护——适合无 UI 的项圈设备）、Passkey Entry（手机显示 6 位数字，外设输入——但项圈没有键盘）、Numeric Comparison（双方显示 6 位数字，用户确认一致——最安全）。
Phase 3 (Key Distribution)：分发 LTK、IRK (Identity Resolving Key——用于 RPA 地址)、CSRK (Connection Signature Resolving Key)。

Just Works — 无 UI 设备的唯一选择

无 MITM 保护 — 可能被中间人攻击

项圈/传感器这类没有显示器和键盘的设备只能使用 Just Works。配对过程对用户完全透明——无需输入 PIN 码、无需确认数字。
LE Legacy (BLE 4.0/4.1) Just Works：STK 通过 TK=0 生成 → 可被被动窃听者暴力破解（窃听配对过程的全部空中包 → 离线计算 TK → 解密后续通信）。
LE Secure Connections (BLE 4.2+) Just Works：使用 ECDH (P-256 椭圆曲线) 密钥交换 → 即使窃听全部配对过程也无法破解——无 MITM 保护但防被动窃听。
强烈建议：任何新设备都用 LE Secure Connections + Just Works——向后兼容 BLE 4.2+ 手机（2015 年后几乎所有手机都支持）。

Bonding — 记住对方

配对一次 → 存储密钥 → 后续自动加密

配对时如果设置了 Bonding 标志 → 双方永久存储对方的 LTK 和 IRK。下次连接时：① 手机用存储的 LTK 直接启动加密（LL_ENC_REQ）——不需重新配对 ② 外设用存储的 LTK 解密验证 ③ 整个加密过程在 1-2 个 Connection Event 内完成（~30ms）——用户感知不到。
Bonding 后外设可以使用 RPA (Resolvable Private Address)——每次广播换一个随机地址，只有 Bonded 的手机能通过 IRK 解析出真实身份——防追踪、隐私保护。

8. 完整时间线 — 从开机到数据传输的每一毫秒

以下追踪一个典型的 IPC 相机开机 → 手机连接 → 读取电池电量 → 持续接收状态通知的完整过程。每个阶段标注了空中包类型、耗时和关键事件。

// IPC 相机 BLE 连接全生命周期 — 空中包级时间线 Phase 1: Advertising (相机开机自检完成, 开始广播) T+0ms 相机初始化 BLE Stack → 配置 Adv Interval=100ms, Adv Data 写入芯片 T+10ms 开始广播: CH37(2402)发 ADV_IND → CH38(2426) → CH39(2480) 每 100ms 重复一轮。AdvA=Random Static, Name="IPC-CAM-001" ... (持续广播, 等待手机连接) Phase 2: Scanning + Connection (用户打开 App, 扫描设备) T+5000ms App 调用 startScan() T+5050ms 手机收到 CH39 上的 ADV_IND (RSSI=-52) → 发 SCAN_REQ → 收 SCAN_RSP T+5100ms App.onScanResult("IPC-CAM-001", RSSI=-52) T+5200ms 用户点击设备 → App 调用 connectGatt() T+5210ms 手机在 CH39 上发 CONNECT_IND: Interval=30ms, Latency=0, Timeout=2000ms, WinOffset=0, WinSize=2 T+5210ms 相机收到 CONNECT_IND → 立即停止广播 → 切换到数据信道 T+5215ms 第一个 Connection Event: 手机发空包 → 相机回空包 → 连接建立! T+5230ms App.onConnectionStateChange(CONNECTED) Phase 3: Service Discovery T+5230ms App 启动 GATT Discovery (需要 ~15-20 次 ATT 请求) T+5260ms ATT Read By Group Type → 4 个 Service 返回 T+5320ms ATT Read By Type ×3 (遍历每个 Service 的 Characteristic) T+5600ms ATT Find Info (Descriptor) ×2 T+5800ms Discovery 完成 → App 缓存所有 Handle (下次重连跳过) Phase 4: 使能通知 + 第一次数据交互 T+5800ms App 写 Battery CCCD (Handle 0x000D ← 0x0001) → 相机回复 Write RSP T+5830ms 相机检测到 CCCD 写入 → 立即发 Notify: Battery=85% T+5840ms App.onCharacteristicChanged(Battery=85%) ← 第一次数据! Phase 5: 持续数据交互 (正常运行时) T+6000ms 用户点"拍照" → App 发 Write Request (CmdID=0x10) → 相机回复 Status=0x00 T+6100ms 相机周期性 Notify: Temperature=42°C, Activity=1200 steps T+6500ms 相机 RSSI 变化 → App 内部记录 (不涉及 GATT) Phase 6: 断开连接 T+60000ms App 进入后台 → 系统发 disconnect() T+60010ms 手机发 LL_TERMINATE_IND → 相机回复 ACK T+60020ms 相机恢复广播模式 (重新开始 Phase 1) T+60030ms App.onConnectionStateChange(DISCONNECTED)
连接优化的 4 个关键点
1. 快速广播配网模式：用户按配网键后 30 秒内用 20ms Adv Interval（而不是 100ms）——发现速度提升 5×。
2. 缓存 GATT Handle：首次 Discovery 后缓存所有 Handle → 重连跳过 Discovery (5s→100ms)——这是最有效的优化。
3. 连接参数更新：连接建立后立即请求 Connection Parameter Update——iOS 要求 Interval≥15ms, Latency≤2。不符合要求会被拒绝。
4. MTU 协商：连接建立后立即 Exchange MTU Request → 247。此后所有 GATT 操作可发更长的 value ——减少分片。
