mgmt Socket

BlueZ 5 核心控制通道

Management Interface — BlueZ 5 引入的内核-用户态控制接口。
socket(AF_BLUETOOTH, SOCK_RAW, BTPROTO_HCI) + bind to HCI_CHANNEL_CONTROL
替代旧版 ioctl 和 hci_sock，提供：adapter 上下电、扫描、配对、连接、设备白名单 等管理操作。
事件驱动：内核主动上报 MGMT_EV_* 事件（Device Found、Connected、Disconnected…）。

2.1 HCI Core — 设备注册与数据流

hci_register_dev()

驱动 → 内核：注册 HCI 设备

调用方: HCI 传输驱动 (hci_uart / btusb / …)
流程:
1. 分配 struct hci_dev (hci_alloc_dev())
2. 初始化工作队列 (cmd_work, tx_work, rx_work)
3. 设置 hdev->open/close/send 回调
4. hci_register_dev(hdev) → 分配 hciX 索引 → 创建 sysfs → 通知 mgmt
5. 结果: /sys/class/bluetooth/hci0/ 出现, bluetoothd 可检测到新 adapter。

数据收发路径

TX: 应用 → Controller | RX: Controller → 应用

TX 发送路径:
App → L2CAP socket → l2cap_chan_send() → hci_send_acl() → hdev->send() → HCI 驱动 → Controller
RX 接收路径:
Controller → HCI 驱动 → hci_recv_frame() → hci_event_packet() / hci_acldata_packet() → L2CAP → socket → App
关键: 所有 HCI 包通过 struct hci_dev 的工作队列 (workqueue) 异步处理，确保内核上下文安全。

2.2 L2CAP — 信道多路复用

L2CAP 信道类型

CID 分配 + LE Credit Based 流控

固定 CID: 0x0001 (Signaling), 0x0004 (ATT), 0x0005 (LE Signaling), 0x0006 (SMP)
动态 CID: 0x0040-0xFFFF — 面向连接信道
LE Credit Based Flow Control (BT 4.1+): 内核实现 l2cap_le_credits 跟踪信用额度。l2cap_le_flowctl 结构维护 credits 计数器，每收到 LE Flow Control Credit 信号增加额度。

关键函数

信道创建 → 连接 → 数据传输

l2cap_chan_create(): 分配 struct l2cap_chan
l2cap_chan_connect(): 发起连接请求
l2cap_connect_req(): 发送 L2CAP_LE_CONN_REQ
l2cap_chan_send(): 发送 SDU（自动分片为 LE-frames）
l2cap_recv_frame(): 接收帧处理（重组 SDU）
l2cap_le_flowctl_init(): LE Credit 流控初始化
信道状态机: BT_CONNECT → BT_CONNECT2 → BT_CONFIG → BT_CONNECTED

2.3 SMP — LE 安全管理

配对流程 (内核态)

smp.c — 密钥协商与加密

SMP 命令: Pairing Request/Response、Pairing Confirm/Random、Encryption Info、Identity Info 等。
LE Legacy: smp_cmd_pairing_req() → TK→STK → hci_le_start_enc()
LE Secure Connections: smp_cmd_public_key() → ECDH P-256 → DHKey Check → LTK
加密启动: hci_le_enable_encryption() → HCI_LE_Start_Encryption command → Controller 硬件 AES-CCM。
内核维护 struct smp_chan 记录配对会话状态。

关键数据结构

smp.h 中的常量与标志

struct smp_dev: 本地 SMP 配置 (IOCap, AuthReq, 密钥分发标志)
struct smp_chan: 活跃配对会话 (TK, STK/LTK, ECDH 密钥对)
SMP_IO_DISPLAY_ONLY / _YESNO / _KEYBOARD_ONLY / _NO_INPUT_OUTPUT
SMP_AUTH_BONDING / _MITM / _SC / _KEYPRESS
加密完成回调: hci_encrypt_change_evt() → 通知 L2CAP "链路已加密" → ATT 允许安全特征访问。

3. bluetoothd — 用户态守护进程

系统 D-Bus 服务 — 插件架构 — 协议实现 — 从 mgmt socket 到 D-Bus API

启动流程

main() → adapter_init() → plugin 加载

1. btd_init(): 初始化 D-Bus 连接、日志、存储后端

4.3 GATT Client 典型操作流程 (D-Bus)

1. StartDiscovery
调用 Adapter1.StartDiscovery，监听 InterfacesAdded 信号发现新设备
2. Connect
调用 Device1.Connect，等待 Connected: true + ServicesResolved: true
3. Discover GATT
ObjectManager GetManagedObjects() 枚举 /hci0/dev_XX/ 下的所有 Service/Char/Desc
4. Write CCCD
调用 GattDescriptor1.WriteValue([0x01,0x00]) 开启 Notify（Descriptor UUID 0x2902）
5. Read/Write
Char.ReadValue() 或 Char.WriteValue(bytes) 进行数据交互
6. Handle Notify
监听 D-Bus PropertiesChanged 信号，GattCharacteristic1.Value 属性变化时收到通知数据
💡 GATT Server 注册流程
GATT Client (手机/PC 作为 Central): 上述流程 — 调用 Remote 设备提供的 D-Bus 对象方法。
GATT Server (设备作为 Peripheral): 应用在 /org/bluez/app 注册 GattProfile1 对象 → 调用 GattManager1.RegisterApplication() → bluetoothd 将本地 GATT 数据库发布到 BLE 广播/连接中。

5. HCI 传输驱动 — Controller 通信链路

UART (H4/H5) · USB (btusb) · SPI · SDIO — 协议栈与硬件的最后一段

H4 (UART 4-Wire)

drivers/bluetooth/hci_h4.c · 最普遍的嵌入式传输

物理层: UART (TX/RX/CTS/RTS)，波特率 115200~4Mbps
协议: 包类型指示字节 (1B) + payload
0x01 HCI Command   0x02 ACL Data
0x03 SCO Data   0x04 HCI Event
优点: 简单、跨平台、几乎所有 BLE 模块支持。
缺点: 无纠错、无重传，依赖硬件流控。

H5 (3-Wire UART)

drivers/bluetooth/hci_h5.c · 可靠传输

物理层: UART (仅需 TX/RX，可选 RTS/CTS)
协议: 基于 SLIP (Serial Line IP) 的链路层协议。提供：
• 数据包分帧 (起始/结束标志 0xC0)
• 16-bit CRC 校验
• 链路建立 (Link Establishment) 协商
• 可靠重传 (基于序列号和 ACK)
状态机: Uninitialized → Initialized → Active
适用于不可靠物理链路（无硬件流控的 UART）。

btusb (USB Bluetooth Dongle)

drivers/bluetooth/btusb.c · 桌面/网关主流

USB Class: Wireless Controller (E0h), Subclass RF (01h), Protocol Bluetooth (01h)
端点: Bulk IN/OUT (ACL + Command) + Interrupt IN (Event) + Isochronous (SCO)
设备列表: btusb 驱动内置 ~400+ USB VID/PID 表 (Intel, Realtek, Broadcom, CSR, …)。
固件加载: 部分 dongle 需要启动时加载固件 (/lib/firmware/)。
带宽: USB 2.0 HS (480Mbps) 远超 BLE/Classic BT 需求。

设备树配置示例

嵌入式 Linux 中通过 DT 或 platform_data 注册
// Device Tree: nRF52840 via UART H4 &uart2 { bluetooth { compatible = "nordic,nrf52840-bluetooth"; reset-gpios = <&gpio0 12 1>; device-wake-gpios = <&gpio0 11 0>; host-wake-gpios = <&gpio0 13 0>; max-speed = 1000000; }; }; // 驱动 probe 时调用: hci_register_dev(hdev);
📦 hci_uart — UART 协议多路复用
drivers/bluetooth/hci_uart.c 是一个多协议 UART 驱动框架。它不直接处理 UART 硬件（由 Linux serial core 处理），而是在 UART 之上注册多个协议处理程序 (proto)：
h4 (HCI_UART_H4) · h5 (HCI_UART_3WIRE) · bcsp (HCI_UART_BCSP) · ath3k / intel / bcm / qca
通过 hci_uart_register_proto(&h4_proto) 注册协议。上层通过 HCIUARTSETPROTO (ioctl) 或 Bluetooth 配置选择协议。现代 BlueZ 中推荐使用 serdev (serial device bus) 替代传统 line discipline。

6.2 原生编译 (开发调试)

# 克隆源码 git clone git://git.kernel.org/pub/scm/bluetooth/bluez.git cd bluez # 安装依赖 (Ubuntu/Debian) sudo apt build-dep bluez # 或手动安装 sudo apt install libglib2.0-dev libdbus-1-dev libudev-dev libical-dev libreadline-dev # 配置与编译 ./bootstrap # 生成 configure (git 源码需要) ./configure \ --prefix=/usr \ --sysconfdir=/etc \ --localstatedir=/var \ --enable-library \ --enable-client \ --enable-tools \ --enable-experimental # 启用实验性 LE 特性 make -j$(nproc) sudo make install # 启动 sudo systemctl start bluetooth sudo systemctl status bluetooth
⚠️ 交叉编译注意事项
1. --enable-library: 必须开启，生成 libbluetooth.so 供 C/C++ 应用链接。
2. D-Bus 配置: 确保 /etc/dbus-1/system.d/bluetooth.conf 正确安装，否则 bluetoothd 无法注册 D-Bus 服务。
3. GLib 版本: BlueZ 5.60+ 要求 GLib ≥ 2.44。
4. 内核版本: LE 高级特性 (BT 5.0+ Coded PHY, Isochronous) 需要 Linux 5.x+。
5. firmware: 许多 BLE 芯片需要启动时加载固件到 Controller。固件需放入 /lib/firmware/ 或 /lib/firmware/brcm/。

9.2 常见问题速查

设备扫描不到

hcitool lescan 无结果

1. 检查 adapter 状态: hciconfig hci0 → 确认 UP RUNNING
2. 检查 firmware 是否加载: dmesg | grep -i bluetooth 或 dmesg | grep -i firmware
3. 检查射频 killswitch: rfkill list → 确认 bluetooth 未被 blocked
4. 检查 BLE 是否启用: btmgmt info → Supported settings 包含 "le"
5. 查看 mgmt 状态: btmgmt info → current settings 包含 "le" + "powered"

连接失败 / 频繁断连

Connection timeout / Disconnected

1. 检查连接参数: Supervision Timeout 是否过短 → btmon 抓包查看 CONNECT_IND 参数
2. 检查晶振精度: 32kHz RC 误差过大导致错过连接事件 → 校准或改用外部晶振
3. 检查 UART 流控: stty -F /dev/ttyS1 -a 确认 crtscts 已启用
4. 蓝牙芯片复位: 通过 GPIO 拉低复位脚，等待 100ms 后恢复
5. 供电噪声: 使用外部 DC-DC 时注意 switching noise 耦合到 RF 前端 → 加 LC 滤波

D-Bus 注册失败

GATT Application 无法注册

1. 确认 bluetoothd 运行: systemctl status bluetooth 或 ps aux | grep bluetoothd
2. 检查 D-Bus 策略配置: /etc/dbus-1/system.d/bluetooth.conf 是否允许当前用户
3. 验证对象路径: GATT Application 必须以 / 开头（如 /org/bluez/app）
4. 确认接口实现完整: GattService1 + GattCharacteristic1 所有必需属性和方法都实现
5. 使用 gdbus 检查: gdbus introspect --system --dest org.bluez --object-path /org/bluez/hci0

HCI 通信超时

Command Complete timeout / ACL lost

1. 检查 UART 波特率: 是否匹配 Controller 预期速率
2. 检查 CTS/RTS 连接: 若使用硬件流控，确保 CTS/RTS 引脚正确连接
3. 检查 H4/H5 协议匹配: kernel driver 协议设置与 Controller 固件一致
4. 查看内核错误: dmesg -w 实时监控 Bluetooth: hci0: command 0xXXXX tx timeout
5. 降低 ACL 负载: 减小 Connection Interval 或 Data Length 缓解 UART 拥塞

10. BlueZ 演进与未来

LE Audio · Isochronous Channels · EATT · Mesh · ELL 迁移 · Android 替代

LE Audio / Isochronous

Linux 5.x+ · BT 5.2+

内核 iso.c 实现 Isochronous Channels (CIS/BIS)。
bluetoothd 支持 org.bluez.MediaTransport1 等时传输。
Auracast (BIS) 广播音频仍在积极开发中。

EATT (Enhanced ATT)

多信道并行 ATT 操作

BlueZ 5.56+ 开始支持 EATT。
允许多个 L2CAP ATT 信道并行，解决传统 ATT 串行化瓶颈。
内核 l2cap_core.c 中 l2cap_eatt_chan 相关实现。

ELL (Embedded Linux Library)

Intel 出品 · 轻量化替代 GLib

ell 是 Intel 开发的嵌入式 Linux 库，提供类似 GLib 的 event loop / D-Bus / 数据结构。
BlueZ 正在逐步迁移部分组件到 ELL（目前主要用于 mgmt 和部分工具）。目标：减小嵌入式系统上的内存占用。

🎯 BlueZ 在嵌入式 IoT 中的最佳实践
1. 选择稳定的 BlueZ 版本 — 推荐 BlueZ 5.66+ (LE Audio 支持较完善)，或 5.55 (保守稳定版)
2. 使用 H4 协议 (UART 4-Wire) — 最简单可靠，硬件流控确保不丢包
3. 内核编译成模块 (m) — CONFIG_BT=m + 按需加载，节约内存 (模块化可节省 ~200KB)
4. D-Bus 策略最小化 — 仅允许必要用户访问，生产环境关闭实验性特性 (--disable-experimental)
5. 监控日志 — bluetoothd 日志重定向到 syslog，设置合理的 log level 避免日志刷屏
6. OTA 升级 — Controller 固件需支持 OTA (Device Firmware Update)，通过 HCI Vendor Specific 命令或 D-Bus 接口更新
