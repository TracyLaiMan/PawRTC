1. BR/EDR 与 BLE 的根本差异

传统蓝牙 (BR/EDR) 和低功耗蓝牙 (BLE) 共用同一个品牌名称但几乎是两套完全不同的协议栈。理解它们的差异是理解 BR/EDR 连接流程的前提。

BR/EDR — 经典蓝牙

79 信道 · 1MHz 间隔 · 1600 hop/s · ACL + SCO

物理层：79 个 1MHz 宽信道（2402~2480MHz），1600 次/秒跳频（每 625μs 一个 slot）。BR 用 GFSK 调制（1Mbps），EDR 用 π/4-DQPSK (2Mbps) 或 8DPSK (3Mbps)。
链路类型：ACL (异步无连接)——数据包传输，有重传，适合文件传输、音频流；SCO/eSCO (同步面向连接)——预留固定时隙，无重传，适合双向语音（HFP 通话）。
体系结构：L2CAP → RFCOMM → SDP。RFCOMM 模拟串口（蓝牙 SPP 的核心）。上层 Profile：A2DP(音乐)、HFP(免提通话)、SPP(串口透传)、HID(键盘鼠标)、OBEX(文件交换)。

BLE — 低功耗蓝牙

40 信道 · 2MHz 间隔 · 连接事件驱动 · 仅 ACL

物理层：40 个 2MHz 宽信道，连接后自适应跳频。1M/2M/Coded PHY。不是按固定时隙跳频，而是按 Connection Event 间隔醒来通信。
链路类型：仅 ACL——没有 SCO 信道（LE Audio 用 LC3 编码通过 ACL 传输——完全不同的机制）。
体系结构：L2CAP → ATT/GATT。服务以 Characteristic 树形结构暴露。没有 RFCOMM、没有 SDP（BLE 用 GATT Discovery 替代）、没有传统的 Profile 概念（用 GATT Service 替代）。

3. 阶段 1: Inquiry — 设备发现

手机打开蓝牙设置页面 → 开始搜索设备。底层执行的是 Inquiry 流程——这与 BLE 的 Advertising/Scanning 有本质区别。

Inquiry 的原理：跳频问询

手机在 32 个 Inquiry 信道上跳频发送 ID 包 → 外设在 32 个信道上轮询监听

BR/EDR 不像 BLE 那样有固定的广播信道——Inquiry 使用79 个信道中的 32 个（分为 A/B 两组，各 16 个）。
手机 (Inquiring Device)：在 32 个 Inquiry 信道上高速跳频——每 312.5μs（半个 slot）换一个信道——发送 ID 包（仅含 Inquiry Access Code，无数据 payload）。一个完整的 Inquiry 跳频序列覆盖全部 32 个信道需要 10ms。
外设 (Inquiry Scanning Device)：在自己的 32 个 Inquiry 信道上缓慢轮流监听——监听窗口 = 11.25ms，间隔 = 1.28s~2.56s（省电）。

FHS 包 — 外设的"身份证"回复

收到 ID 包 → 等待 625μs → 回复 FHS 包 (包含关键参数)

外设在监听信道上收到 ID 包（匹配自己的 Inquiry Access Code）→ 进入Inquiry Response 状态——等待 625μs (一个 slot) → 回复 FHS (Frequency Hop Synchronization) 包。FHS 包包含：
BD_ADDR (6B)：外设的蓝牙 MAC 地址——这是后续 Paging 的关键参数。
CLK (28bit)：外设的本地时钟值——手机需要此外设时钟来计算Paging 时的跳频序列。
CoD (3B)：Class of Device——告诉手机"我是什么设备"（0x200404=音频/免提, 0x080104=电脑/桌面, 0x000504=手机）。
Page Scan Mode：外设的 Paging Scan 模式（R0/R1/R2——影响后续 Paging 的速度）。

Inquiry 为什么比 BLE Advertising 慢得多？
BLE 广告在3 个固定信道上发——手机只监听 3 个信道，发现极快（~50ms）。BR/EDR 在32 个信道上跳频——手机要覆盖所有 32 个信道才能确保发现任意外设。一个完整的 Inquiry 扫描周期 = 1.28s~10.24s（取决于 Inquiry Length）。这就是为什么传统蓝牙设备在手机设置页面中"出来得慢"——比 BLE 设备慢 10-100 倍。

4. 阶段 2: Paging — 建立连接

用户点击设备名称 → 手机发起 Paging——这是 BR/EDR 连接建立的核心。Paging 的成功依赖于手机精确计算外设的跳频序列——这个信息来自 Inquiry 阶段获取的 BD_ADDR + CLK。

Paging 跳频序列的计算

BD_ADDR + CLK → 跳频算法 → 32 个 Paging 信道的序列

手机通过外设的 BD_ADDR 和 CLK（从 FHS 包获得），运行与基带规范相同的信道选择算法——计算出外设当前正在监听哪 32 个 Paging 信道（分为 A/B 两组，各 16 个）。
计算精度依赖于 CLK 的准确性：手机从收到 FHS 到发送 Page 之间有延迟 → 需要估计外设时钟的漂移量（CLK offset 补偿）→ 用补偿后的时钟值重新计算跳频序列。如果时钟估计偏差太大 → 手机在错误信道上发 Page → 外设收不到 → Paging 失败。

Paging 的握手过程：ID → ID → FHS → ID → POLL

Train A → Train B → 完整握手 — 标准流程约 2.56s

Train A (16 个信道 × 128 次 = 10ms)：手机在计算的 16 个 A-train 信道上轮流发送 ID 包（含外设的 Device Access Code）——每个信道重复 128 次（10ms）。
Train B (16 个信道 × 128 次 = 10ms)：如果 Train A 无响应 → 在 16 个 B-train 信道上重复。
外设收到 ID 包（匹配自己的 DAC）→ 625μs 后回复另一个 ID 包（同一信道上——确认收到）。
手机收到外设的 ID 回复 → 312.5μs 后发送 FHS 包（包含手机自己的 BD_ADDR + CLK——外设据此计算此后的跳频序列）。
外设回复 ID 包（确认收到 FHS）→ 外设切换到手机的时钟和跳频序列——连接建立！
手机发送 POLL 包（第一个 ACL 数据包，确认外设已同步）→ 外设回复 NULL 包 → 进入 Connection 状态。

5. 阶段 3: Connection 状态 — ACL/SCO 链路与 Piconet

连接建立后，手机成为 Master，外设成为 Slave。Master 控制整个 Piconet（微微网）的时钟和跳频序列——所有 Slave 都同步到 Master 的时钟。

Piconet — 微微网

1 Master + 最多 7 Active Slaves + 最多 255 Parked Slaves

一个 Piconet 由一个 Master 控制。所有 Slave 同步到 Master 的跳频序列和时钟。Master 在偶数 slot 发送，Slave 在奇数 slot 回复。每个 Slave 有一个 3-bit AM_ADDR (Active Member Address)——在 Piconet 内唯一标识自己（范围 1~7，0 留给广播）。
Scatternet (分散网)：一个设备可以同时属于多个 Piconet，在不同 Piconet 间分时切换（需同步跳频序列）。

ACL 链路

Asynchronous Connection-Less — 数据包交换

ACL 是尽力而为的分组交换链路——一个 Master 和 Slave 之间只有一条 ACL 链路。支持 DM1/DH1/DM3/DH3/DM5/DH5 六种包类型（DM=中速率带 FEC, DH=高速无 FEC, 数字=占用 slot 数）。
ARQ (Automatic Repeat reQuest)：每个 ACL 包都有 CRC，接收端检查 → 错误则在下一个 slot 中通过 ACK/NACK 告知发送端 → 发送端重传。这意味着 ACL 的可靠性由 基带层 ARQ 保证——上层 L2CAP/RFCOMM 不需要自己处理重传。

SCO/eSCO 链路

Synchronous Connection-Oriented — 实时语音

SCO 是电路交换链路——在固定 slot 上预留带宽。Master 和 Slave 之间最多 3 条 SCO 链路（对应 64Kbps PCM 语音）。
SCO 不重传——丢包就丢了（语音中 1-2 个包丢失几乎无感知）。延迟极低（~1.25ms）。
eSCO (Extended SCO)：支持重传窗口——在预留 slot 之后还有一段重传机会。适合需要一定可靠性但仍要保证延迟的音频场景（如 A2DP 音乐——使用 ACL 上的 AVDTP 而非 SCO）。

6. 阶段 4: SDP — 服务发现协议

连接建立后，手机只知道外设的 BD_ADDR 和 Class of Device——对外设能提供什么服务（Profile）一无所知。SDP 就是用来回答"你能做什么"的协议。

SDP 的查询模型：Service Record 树

每个 Service 由 UUID 唯一标识 · 属性 ID-Value 对描述细节

外设维护一个 SDP 数据库——每个支持的 Profile 对应一条 Service Record。每条 Service Record 由多个 Attribute (ID + Value) 组成：
ServiceRecordHandle (0x0000)：该 Record 的唯一 32-bit 标识。
ServiceClassIDList (0x0001)：Service 的 UUID 列表（如 SPP=0x1101, A2DP_Sink=0x110B, HFP=0x111E）。
ProtocolDescriptorList (0x0004)：使用哪些协议栈——如 SPP 的协议栈 = [L2CAP, RFCOMM]。
RFCOMM Channel (自定义属性)：RFCOMM 服务端的 Channel 号（如 Channel=1）——手机后续连接 RFCOMM 时需要。

SDP 请求-响应流程

Service Search → Attribute Request → 逐个 Record 获取

Step 1: SDP_ServiceSearchRequest——手机发送目标 UUID（如 0x1101=SPP）→ 外设返回匹配的 ServiceRecordHandle 列表。
Step 2: SDP_ServiceAttributeRequest——手机发送 Handle + 想要的 Attribute ID 列表（如 0x0004 ProtocolDescriptorList）→ 外设返回对应的 Attribute Value。
Step 3: 解析 RFCOMM Channel——手机从 ProtocolDescriptorList 中提取 RFCOMM 的 Server Channel 号（如 1）。
SDP 本身走 L2CAP——使用固定的 PSM (Protocol Service Multiplexer) = 0x0001。SDP 走 ACL 链路。

7. 阶段 5: L2CAP + RFCOMM — 数据通道建立

SDP 告诉了手机"RFCOMM Channel = 1"——现在手机要在此之上建立真正的数据传输通道。

L2CAP — 逻辑链路控制与适配

多路复用 · 分片重组 · QoS 协商

L2CAP 在 ACL 链路上提供多个逻辑通道——每个通道由 16-bit CID (Channel Identifier) 标识。CID 0x0001 = 信令通道（用于创建/配置数据通道），CID 0x0040~0xFFFF = 动态分配的数据通道。
手机连接 RFCOMM 的流程：① 发送 L2CAP_ConnectReq (PSM=0x0003 — RFCOMM 的协议标识, Source CID=0x0040) ② 外设回复 L2CAP_ConnectRsp (Dest CID=0x0041, Result=SUCCESS) ③ 双方互发 L2CAP_ConfigReq/Rsp 协商 MTU 等参数。
L2CAP 层支持分片重组 (SAR)——上层 RFCOMM 可以发送任意大小的帧，L2CAP 自动分片为适合基带包的 chunk 并在接收端重组。

RFCOMM — 串口仿真协议

基于 L2CAP 的 RS-232 串口仿真 · 多路复用 · 流控

RFCOMM 在 L2CAP 通道上模拟 RS-232 串口——提供 9 针串口的所有信号（TX/RX/RTS/CTS/DTR/DSR/DCD/RI）。每个 RFCOMM 会话由 DLCI (Data Link Connection Identifier, 6-bit) 标识——一个 L2CAP 通道上可以有多个 RFCOMM DLCI（多路串口）。
RFCOMM 连接建立：① SABM (Set Asynchronous Balanced Mode) 帧——手机请求建立 DLCI=0 (控制通道) ② 外设回复 UA (Unnumbered Acknowledgement) ③ 在 DLCI=0 上交换 PN (Parameter Negotiation) 帧——协商后续数据通道的 DLCI 号、帧大小、credit 流控参数 ④ 在协商好的 DLCI (如 2, 6, 10...) 上建立数据通道。
Credit-Based Flow Control：RFCOMM 不会让发送方淹没接收方——接收方给发送方一定数量的credits——每发一帧消耗一个 credit → credit 用完必须等接收方补充。

8. 安全配对 — PIN 码与 SSP 安全简单配对

BR/EDR 的配对经历了从 PIN 码到 SSP (Secure Simple Pairing) 的演进——BLE 的配对模型很大程度上借鉴了 BR/EDR 的 SSP。

Legacy Pairing (PIN 码)

蓝牙 2.0 及之前 — 双方输入相同的 PIN 码

用户在外设上输入 PIN 码（如 0000 或 1234）→ 手机上也输入相同的 PIN → 双方基于 PIN + BD_ADDR + 随机数通过 E21/E22 算法生成 Link Key → 互相认证。安全弱点：4 位 PIN 只有 10000 种可能——窃听配对过程后可暴力破解 Link Key。固定 PIN 码（0000/1234）的设备尤其脆弱。

SSP (Secure Simple Pairing)

蓝牙 2.1+ — 四种关联模型

Just Works（无 UI 设备）——无需用户交互。无 MITM 保护。
Numeric Comparison（双方都有显示器）——双方显示 6 位数字，用户确认一致。有 MITM 保护——最安全。
Passkey Entry（一方有键盘）——一方显示 6 位数字，另一方输入。
Out of Band (OOB)（NFC 辅助）——通过 NFC 交换密钥。
SSP 使用 ECDH (P-192/P-256) 密钥交换——即使窃听也无法破解（不同于 Legacy PIN 码的可暴力破解）。

Bonding — 记住对方

配对一次 → 存储 Link Key → 后续自动认证

配对完成后双方将Link Key 永久存储到 Flash——下次连接时：① 手机发送 LMP_au_rand (随机挑战) ② 外设用存储的 Link Key 计算 SRES (Signed Response) ③ 手机验证 SRES → 认证通过——整个过程在基带层完成 (~10ms)，用户完全无感知。
如果任何一方丢失了 Link Key → 需要重新配对（外设清除 Bonding 或手机"忘记设备"）。

T+5800ms Inquiry 完成 (持续 5.8 秒扫描所有设备) T+5900ms 手机显示设备列表 → 用户看到 "BT-Serial-Device" → Inquiry 总耗时: ~6 秒 (典型) Phase 2: Paging — 建立 ACL 连接 T+6200ms 用户点击设备 → 手机发起 Paging T+6200ms 手机用 BD_ADDR+CLK 计算 Page 跳频序列 T+6250ms Train A+B 完成 → 外设在 CH45 上收到 Page ID T+6251ms 外设回 ID → 手机发 FHS → 外设回 ID → 手机发 POLL T+6260ms ACL 连接建立! 手机成为 Master → Paging 总耗时: ~60ms (计算正确时极快) Phase 3: SDP — 服务发现 T+6280ms 手机发送 SDP_ServiceSearchRequest (UUID=0x1101 SPP) T+6320ms 外设回复 ServiceRecordHandle=0x00010000 T+6340ms 手机发送 SDP_ServiceAttributeRequest (Handle=0x00010000, Attr=0x0004) T+6380ms 外设回复 ProtocolDescriptorList: [L2CAP, RFCOMM Ch=1] → SDP 总耗时: ~100ms Phase 4: L2CAP + RFCOMM 通道建立 T+6400ms 手机发送 L2CAP_ConnectReq (PSM=0x0003) → 外设回 Rsp T+6440ms L2CAP ConfigReq/Rsp (MTU=672) T+6460ms RFCOMM SABM (DLCI=0) → UA → PN_Req → PN_Rsp → SABM(DLCI=2) → UA → RFCOMM 建立总耗时: ~100ms Phase 5: SPP 数据传输 (用户态的 AT 命令) T+6480ms 手机 App 调用 write(fd, "AT+BAUD=115200\r\n") T+6490ms RFCOMM UIH (DLCI=2, Data="AT+BAUD=115200\r\n") T+6500ms 外设回复 RFCOMM UIH (DLCI=2, Data="OK\r\n") T+6510ms App 收到 "OK" → SPP 连接完全就绪! → 从用户点击到首次数据传输: ~300ms
⚠ BR/EDR 连接的最常见失败原因
1. Inquiry 找不到设备——外设未开启 Inquiry Scan（如只在 Page Scan 模式）→ 外设只在 32 个 Page 信道上监听，不在 Inquiry 信道上监听 → 手机 Inquiry 收不到 FHS → 设备列表中不出现。
2. Paging 超时——Inquiry 和 Paging 之间的间隔太长 → 外设时钟漂移 → 手机计算的跳频序列与实际不符 → Page ID 全发在错误信道上 → Paging 超时（5.12s）。解决：Inquiry 完成后立即 Paging（时钟漂移 < 1ms）。
3. SDP 查询返回空——外设注册了 SPP 服务但未在 SDP 数据库中正确注册 Service Record → 手机 SDP search 返回 0 条 → App 报错"设备不支持 SPP"。
