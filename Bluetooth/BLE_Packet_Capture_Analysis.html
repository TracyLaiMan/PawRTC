2. nRF Sniffer — BLE 空中抓包实战

nRF52840 USB Dongle + nRF Sniffer 固件 + Wireshark 是目前性价比最高、最实用的 BLE 调试方案。100 元硬件看到 99% 的问题。

硬件准备与固件烧录

nRF52840 Dongle + nRF Sniffer for Bluetooth LE

① 购买 nRF52840 USB Dongle (PCA10059, ~100 元) ② 下载 nRF Connect for Desktop (Nordic 官方工具) → 安装 "Programmer" ③ 下载 nRF Sniffer for Bluetooth LE 固件 (Nordic 官网, 对应 dongle 的 hex 文件) ④ Programmer 烧录 hex → 设备变为 USB CDC-ACM 串口 (Wireshark 通过此串口接收捕获数据)。
注意：Sniffer 固件占用整个芯片——烧录后 Dongle 不能再做普通 BLE 应用。想恢复 → 烧录回原厂固件。

Wireshark 配置与捕获流程

nRF Sniffer 插件 + 串口选择 + 设备跟踪

① Wireshark 安装 nRF Sniffer 插件：帮助 → 关于 Wireshark → 文件夹 → Global ExtPlugins → 复制 nrf_sniffer.bat ② 插入 Dongle → Wireshark 顶部出现 nRF Sniffer 工具栏 ③ 选择 Dongle 对应的串口 (如 COM3 或 /dev/ttyACM0) ④ 选择要跟踪的设备：在设备列表中勾选目标的 MAC 地址 → Sniffer 开始在该设备的广播信道上监听 → 收到 CONNECT_IND 后自动提取 Access Address 和 CRCInit → 自动切换到数据信道跟踪跳频 ⑤ 开始捕获！

Sniffer 的限制与绕行

只抓 BLE · 需要紧跟连接 · 无法抓加密内容

只能抓 BLE：不支持 BR/EDR 抓包（BR/EDR 需要 Ellisys 或 Ubertooth）。
无法解密加密数据：除非提供 LTK。Wireshark 中可以配置 Edit → Preferences → Protocols → Bluetooth → Link Keys——粘贴 Bonding 的 LTK。或使用 nRF Connect 的 Sniffer + LTK 自动提取（从已 Bonding 的手机中导出 LTK）。
连接跟踪可能丢失：如果 Connection Interval 变化、或跳频算法未正确跟踪 → Wireshark 中出现大片空白。需要重新选取设备（nRF Sniffer 会自动重新捕获 Access Address）。

// nRF Sniffer + Wireshark 的过滤秘籍 — 常用 Display Filter === 按协议层级过滤 === btle // 只显示 BLE 空中包 btle.advertising_header // 只看广播包 (ADV_IND, SCAN_REQ, CONNECT_IND...) btle.data_header // 只看数据信道包 (连接后的所有包) btle.llid == 0x03 // 只看 LL Control PDU (LL_CONNECTION_PARAM_REQ, LL_TERMINATE_IND...) bthci_acl // 只看 ACL 数据包 (含 L2CAP + ATT 负载) === 按具体事件过滤 === btle.ll_control.opcode == 0x02 // LL_TERMINATE_IND — 查看断连原因码 btle.advertising_header.pdu_type == 0x00 // ADV_IND 包 btle.advertising_header.pdu_type == 0x05 // CONNECT_IND 包 btle.advertising_header.adv_addr == AA:BB:CC:11:22:33 // 指定设备 btatt.opcode == 0x12 // ATT Write Request btatt.opcode == 0x1B // ATT Handle Value Notification === 按错误/异常过滤 === btatt.opcode == 0x01 // ATT Error Response — GATT 操作失败! btle.crc.error // CRC 校验失败的包 (需要 Ellisys 才能检测) btle.ll_terminate.error_code // 断连原因码 (0x05=Auth Fail, 0x08=Conn Timeout, 0x13=Remote User Terminated...) === 组合过滤 — 实战中最常用 === btle.data_header && btle.llid == 0x03 // 连接后所有控制包 (参数更新/加密/断连) btatt && btat.handle == 0x000c // 只看 Handle 0x000C (如电池 Characteristic) btle && btle.access_address == 0x12345678 // 只看某个连接的 Access Address

3. 实战案例 1: 断连原因追踪 — 空中包还原真相

最常见的 BLE 问题：手机 App 显示"连接断开"——原因可能是外设主动断连、手机主动断连、超时断连、加密失败断连。空中包告诉你谁先发的 LL_TERMINATE_IND 以及为什么。

5. HCI 层分析 — Android HCI Snoop Log 与 Linux btmon

HCI (Host Controller Interface) 是 Host (协议栈) 和 Controller (射频芯片) 之间的标准接口。HCI Log 记录了两者之间所有的命令、事件和数据——是手机侧/设备侧最易获取的诊断数据。

Android HCI Snoop Log

开发者选项 → 启用蓝牙 HCI 日志 → /sdcard/btsnoop_hci.log

获取步骤：① 手机进入"设置 → 开发者选项" ② 勾选 "启用蓝牙 HCI 信息收集日志" ③ 复现问题（连接、配对、传输数据）④ 取消勾选 → 日志写入 /sdcard/btsnoop_hci.log ⑤ adb pull /sdcard/btsnoop_hci.log → 用 Wireshark 打开。
HCI Log 的典型内容：HCI_CMD: LE Set Scan Enable → HCI_EVT: Command Complete（开启扫描）、HCI_EVT: LE Advertising Report（发现设备——含 RSSI + AdvData 完整内容）、HCI_CMD: LE Create Connection → HCI_EVT: LE Connection Complete (Status=0x00 成功)（建立连接）、ACL Data: ATT_Write_Req → ATT_Write_Rsp（GATT 操作）。

Linux btmon

btmon --write hci.log — BlueZ 协议栈的官方调试工具

启动抓包：btmon -w hci.log（将日志写入文件，后台运行）。btmon 能看到比 Android HCI Log 更多的内容：mgmt 接口（BlueZ 管理接口——内核与 bluetoothd 之间的通信）、L2CAP 信令（Connection Request/Response、Configuration Request/Response——这些在 HCI 层被封装在 ACL 数据中但 btmon 会自动解析）、SDP PDU（Service Search Request/Response 的完整字段）、RFCOMM 帧（SABM/UA/PN/DISC 的控制帧 + UIH 数据帧）。
嵌入式设备的 btmon：如果使用 BlueZ 协议栈的嵌入式 Linux 设备（如 IPC 相机、树莓派）——btmon -w /tmp/bt.log & 持续记录所有蓝牙活动。这是设备侧最强大的诊断工具。

// btmon 输出的完整连接建立过程 (BLE Peripheral 侧) // btmon 输出格式: 时间戳 + 层级 + 方向 + 内容 = bluetoothd: (mgmt) // mgmt 接口 — bluetoothd ↔ 内核 @ MGMT Command: Add Advertising // bluetoothd 配置广播参数 @ MGMT Event: Command Complete (0x003E) // 内核确认 = btusb: (HCI) // HCI 层 — 内核 ↔ Controller < HCI Command: LE Set Advertising Enable (0x08|0x000A) // 开启广播 > HCI Event: Command Complete (0x0e) plen 4 > HCI Event: LE Meta Event (0x3e) plen 19 — LE Connection Complete Status: Success (0x00) Handle: 64 // Connection Handle = 64 Role: Slave (0x01) Peer address: AA:BB:CC:11:22:33 < ACL Data: Handle 64, ATT_Exchange_MTU_Req (MTU=247) // 外设发起 MTU 协商 > ACL Data: Handle 64, ATT_Exchange_MTU_Rsp (MTU=185) // 手机回复 // btmon 的价值: 同时看到 mgmt/HCI/L2CAP 三层——端到端追踪一个事件

6. 实战案例 3: Connection Interval 没有更新的排查

外设发 LL_CONNECTION_PARAM_REQ 请求更短的 Interval——手机要么接受、要么拒绝。不同的拒绝原因意味着不同的问题。

Slave → LL_CONNECTION_PARAM_REQ (重试, 最多 N 次) (继续等待...) // → LL_TERMINATE_IND (Error=0x22: LL Response Timeout) // 诊断: 手机 Controller 可能不支持 LL_CONNECTION_PARAM_REQ 这个 LL Control PDU // (极老的 Android 4.x 设备有此 bug)

配对失败的原因有数十种——每一种都在配对请求/响应的某个字段中留下了线索。

// 完整的 LE Secure Connections 配对过程 — 每个包的字段解读 // Step 1: 配对特性交换 Master → Pairing Request: IO Capability: 0x03 (NoInputNoOutput) // 双方都没有键盘/显示器 OOB Data Flag: 0x00 (Not Present) AuthReq: 0x05 // bit0=Bonding, bit1=MITM(no), bit2=SC(yes!) Max Enc Key Size: 16 Initiator Key Dist: 0x0D (LTK + IRK + CSRK) // 手机希望分发哪些密钥 Responder Key Dist: 0x0D (LTK + IRK + CSRK) Slave → Pairing Response: IO Capability: 0x03 (NoInputNoOutput) AuthReq: 0x01 // bit0=Bonding, MITM=no, SC=no(!) // ⚠ 外设不支持 LE Secure Connections! // 结果: 手机检测到自己的 SC=yes 但外设 SC=no → 手机依据策略 // Android 8-12: 降级到 LE Legacy → 配对可能继续 // iOS 13+: 直接发送 Pairing Failed (0x05=Pairing Not Supported) → 配对失败! // 如果双方 AuthReq 不冲突, 进入 Step 2: 密钥交换 (ECDH) Master → Pairing Public Key (64 bytes: P-256 X + Y) // 手机公钥 Slave → Pairing Public Key (64 bytes) // 外设公钥 // Step 3: 认证阶段 (双方各自计算确认值) Master → Pairing Confirm (16 bytes) // 手机确认值 Slave → Pairing Confirm (16 bytes) // 外设确认值 Master → Pairing Random (16 bytes) // 手机随机数 Slave → Pairing Random (16 bytes) // 外设随机数 // 双方各自验证: Confirm == AES-CMAC(K, Random) → 匹配→成功 // Step 4: 密钥分发 Master → LL_ENC_REQ → LL_ENC_RSP → LL_START_ENC_REQ → LL_START_ENC_RSP // 加密链路建立 → 后续通过加密通道分发 LTK/IRK/CSRK

8. Ellisys Explorer — 专业级全信道并行抓包

Ellisys Bluetooth Explorer 是蓝牙协议分析的金标准——全信道并行监听、微秒级时间戳、PHY 层射频分析、WiFi 共存分析。成本 4-8 万元——芯片原厂、认证实验室、以及被疑难杂症逼到绝路的开发团队的终极武器。

与 nRF Sniffer 的本质区别

全信道并行 vs 单信道跟踪

nRF Sniffer：单射频收发器——需要先知道 Access Address 和 CRCInit → 计算跳频序列 → 跟踪一个连接的跳频。如果跟踪丢了（错过了 CRC、信道切换时未同步）→大片空白。只能抓 BLE。
Ellisys：内置宽带射频前端 + FPGA——同时监听全部 40 个 BLE 信道（或全部 79 个 BR/EDR 信道）→ 不需要跟踪跳频——所有信道上的所有包尽收眼底。即使错过了某个连接的 Access Address，回溯数据中也能找到。同时抓 BLE + BR/EDR + WiFi。

硬件连接与设置

USB 连接 + Ellisys 软件 (非 Wireshark)

连接：Ellisys Explorer B300/B400 通过 USB 3.0 连接 PC。设备自带全向天线（4 根——2.4GHz + 5GHz WiFi）。
软件：Ellisys Bluetooth Analyzer（自有软件，非 Wireshark）——提供比 Wireshark 更专业的蓝牙协议视图：Instant Piconet View（实时显示 Piconet 拓扑）、Timeline View（所有包的时序瀑布图）、Packet Detail View（逐字段协议解码）、Audio Expert View（SCO/eSCO 音频质量分析——抖动/丢包/延迟）。也可导出为 Wireshark 兼容格式。
供电：USB 供电即可，无需外部电源。便携——可以带到客户现场抓包。

Ellisys 独有的分析能力

PHY 层错误 · 时序违规 · 音频质量 · WiFi 共存

① CRC 错误检测：nRF Sniffer 固件丢弃 CRC 错误的包——你看不到。Ellisys 捕获所有包（包括 CRC 错误的），在 UI 中标注 CRC Error → 可以统计误包率 → 判断 RF 干扰程度。
② T_IFS 时序测量：BLE 要求响应在 150μs±2μs 内发送。Ellisys 对每个包打125ns 精度时间戳——可以精确测量外设的响应时间是否超出 150μs 限制。
③ Audio Expert：SCO/eSCO 链路的音频质量分析——显示每个 slot 的音频数据是否到达、是否在正确的时间到达——量化通话断续/卡顿。
④ WiFi 共存分析：同时抓 2.4GHz WiFi 和蓝牙——可视化两者在频域上的冲突时刻——判断是否是 WiFi 20MHz 信道干扰了正在跳频的蓝牙。

Ellisys 实战: 定位"偶发性断连"的 RF 干扰

nRF Sniffer 看不到的 CRC 错误 — Ellisys 全信道捕获还原真相

场景：IPC 相机 BLE 连接每隔 2-3 小时断连一次，App 报 error=133。nRF Sniffer 抓包正常（看不到 CRC 错误，因为固件丢弃了）。
Ellisys 分析步骤：① 同时抓 BLE 40 信道 + WiFi 2.4GHz ② 在 Timeline View 中定位断连时刻的 LL_TERMINATE_IND ③ 回溯断连前 200ms 的所有包——发现大量 CRC 错误包（Ellisys 用红色标注）集中在 CH6(2426MHz) 和 CH12(2438MHz) ④ 切换到 WiFi 频谱视图——发现断连时刻恰好有WiFi AP 在 CH6(2437MHz) 上以最大功率发送 Beacon 帧（每 100ms 一发——持续时间 2ms）⑤ Beacon 的频谱泄漏覆盖了 BLE CH6~CH12（2436-2440MHz）→ BLE 在这些信道上的包几乎全被干扰 → ARQ 重试多次后 Supervision Timeout → 断连。
解决：外设的 Channel Map 中排除 CH6~CH12（通过 LL_CHANNEL_MAP_IND 更新）——断连问题消失。

Ellisys 实战: T_IFS 时序违规 — 外设响应太慢

125ns 精度时间戳揭示的微秒级延迟问题

场景：BLE 外设频繁被手机判定超时断连，但外设固件日志显示"正常响应"。
Ellisys 分析：在 Packet Detail View 中选中任意 Master→Slave 包 → 查看下一个 Slave→Master 包的时间间隔。T_IFS 标准 = 150μs ± 2μs。Ellisys 实测显示外设的 T_IFS = 162-180μs——每次都超出 10-30μs。虽然外设固件"响应了"——但响应晚了 10μs——手机的 Controller 硬件在 152μs 时关闭了接收窗口（认为超时）。
根因：外设 BLE 芯片的 HCI 传输层 (UART) 波特率设置太低（115200）→ HCI 事件从 Controller 传到 Host 需要 ~80μs → Host 处理 GATT 请求耗时 ~50μs → HCI Command 回传 Controller 又 80μs → 总延迟 210μs > 150μs。将 UART 升级到 921600 → T_IFS 降到 130-140μs → 问题解决。

Ellisys 实战: BR/EDR Inquiry 找不到设备的根因

全 79 信道捕获 → 精确看到 Inquiry Scan 的时序漂移

场景：传统蓝牙 SPP 设备在手机上有时能搜到、有时搜不到——概率性问题。
Ellisys 分析：全 79 信道捕获。手机在 32 个 Inquiry 信道上跳频发 ID 包——每个信道停留 312.5μs。外设在 32 个信道上轮询监听——监听窗口 11.25ms，间隔 1.28s。Ellisys Timeline View 显示：外设的监听窗口在逐渐漂移（晶体老化→时钟偏差 → 窗口相对手机的 Inquiry 序列在移动）→ 每 8 个 Inquiry 周期（~10s）窗口恰好落在手机不扫描的信道上 → 那次 Inquiry 就找不到设备。
解决：手机端增加 Inquiry Length（延长扫描时间覆盖外设窗口的漂移范围）或 外设增大 Inquiry Scan Window（从 11.25ms 增加到 22.5ms——2 倍覆盖）。

Ellisys 实战: Audio SCO 断续定位

Audio Expert View → 量化通话质量

场景：蓝牙耳机通话时对方听到"咔咔"断续声——不是完全断，是有规律的杂音。
Ellisys 分析：Audio Expert View 显示 SCO interval=6 slots (3.75ms)。每个 SCO slot 中有数据到达，但大约每 14 个 slot（~52ms）有一个 slot 的数据包延迟了 1 个 slot（本该在 slot N 到的包在 slot N+1 才到）。这 1 个 slot 的迟到导致音频解码器收到两个连续的零填充帧 →"咔"一声。
根因：Master 的 ACL 数据流量过大（同时在传文件 via SPP）→ ACL 包占用了 SCO 预留 slot 前后的保护间隔 → SCO 包被推后 1 slot。解决：在文件传输期间降低 ACL 优先级或限制 ACL 吞吐。

⚠ Ellisys 的局限性 — 什么时候它也无能为力
1. 加密数据不可见：Ellisys 抓到了加密的空口包——但如果不知道 LTK，Payload 是乱码。需要在软件中导入 Link Key（从 Bonding 的设备中导出）。nRF Sniffer 一样有此限制。
2. 距离限制：全信道并行需要宽带天线——灵敏度略低于单信道窄带接收机。在10 米以外的高衰减场景（如穿过墙壁）——信号可能弱到 Ellisys 也无法解码。RSSI < -80dBm 时错误率急剧上升。
3. 不会自动识别协议错误：Ellisys 是被动监听——不会告诉你"这个包违反了协议"。需要工程师自己理解协议规范并判断。不过 Ellisys 软件会自动标注一些明显的错误（如 CRC Error、长度不匹配、T_IFS 超时）。
4. 价格——B400 型号约 $6,000-10,000 USD。对于个人开发者——nRF Sniffer 覆盖 90% 的问题；团队可以共享一台 Ellisys（远程预约使用）。

9. 工具组合拳 — 多层面交叉验证

最强大的诊断不是靠单一工具——而是同时从多个层面抓取数据，交叉验证。

组合 1: nRF Sniffer + Android HCI Log

空中真实包 vs 手机 HCI 层看到的包 — 找差异

同时抓空中包（nRF Sniffer）和手机侧（HCI Log）→ 对比两者：空中包显示外设发了 Notify → HCI Log 中却没有对应的 ACL Data 包 → 说明手机 Controller 收到了但没有上报给 Host（Android 栈内部过滤或 bug）。空中包显示外设没有发 LL_TERMINATE_IND → HCI Log 中却有 Disconnection Complete Event → 说明手机 Controller自己决定断连（如 Supervision Timeout 到期时 Controller 不通知 Host 直接断）。

组合 2: btmon + 外设固件日志

软件日志 vs 空中真实行为 — 找代码 Bug

外设固件打印"已发送 Notify (电池=85%)" → btmon 中却没有看到对应的 ACL Data → 说明外设的 GATT 层代码有 bug——应用层以为发送了，但底层没有真正发出（可能 CCCD 未使能、或 ATT 队列满、或 GATT 连接未建立）。
btmon 显示手机发了 ATT_Write_Req → 外设固件日志却显示"收到未知 Handle=0xFFFF 的写请求"→ App 缓存了错误的 Handle（Service Changed 后没有重新 Discovery）。

BLE 调试的黄金法则 — 永远不要只相信日志
① 应用层日志会说谎：调用了 writeCharacteristic() ≠ 空中真的发出去了（MTU 不足被分片？ATT 队列满被丢弃？Controller 忙？）。
② HCI Log 比应用日志可靠：HCI 层记录的是 Host↔Controller 的真实交互——如果 HCI 中也没有，那肯定是 Host 以上的问题。
③ 空中抓包是真相的唯一来源：电波不会说谎——如果空中没有这个包，那就是根本没有发出来。
④ 三层交叉验证：应用日志 ∩ HCI Log ∩ 空中包 = 精确定位问题层级。
