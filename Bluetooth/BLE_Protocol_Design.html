BLE 应用传输协议设计

BLE 4.x/5.x 只定义了物理层和链路层——应用层传输协议需要自己设计。OTA 升级、小数据上报、音频流传输需要完全不同的帧格式、分片策略、确认机制和差错控制。

BLE 传输基础：MTU · 连接间隔 · GATT 约束

为什么 BLE 应用层必须自己设计传输协议——ATT/GATT 只提供"读写特征值"的原子操作，没有流控、没有分片重组、没有校验、没有会话管理。

MTU 约束

ATT_MTU - 3 = 应用层可用字节

默认 MTU=23 → 应用层仅 20 字节可用——一个完整的 JSON 状态包都放不下。BLE 4.2+ 支持协商到 MTU=247 → 应用层 244 字节。这是协议分片设计的第一个硬约束——任何超过 MTU-3 的数据必须拆包。

连接间隔

7.5ms ~ 4s — 吞吐与功耗的博弈

BLE 不是持续连接——两个设备以固定间隔在约定时刻醒来通信（Connection Event）。每 Event 可交换多包（受 DLE 和控制器队列深度限制）。7.5ms：1.4Mbps 但功耗高（不休眠）。100ms+：功耗极低但吞吐极低。

GATT 串行约束

同时只能有一个未完成的 GATT 事务

ATT 是严格请求-响应：Write Request → 必须等 Write Response → 才能发下一个。Write Without Response (Write Command) 可连续发但无确认——适合音频/OTA 流式数据——但需要应用层自己实现确认。

为什么 BLE 不需要帧边界而 TCP/UART 必须？

根本区别不在于"协议"——在于协议栈下层有没有帮你做消息边界的保留和还原。

TCP/UART：字节流的困境 — 逐字节追踪真实场景

应用层发了两条消息，传输层合并成一个数据块发出去了

发送端：应用调用 write(fd, "REBOOT", 6) → 内核 TCP 栈将这 6 字节放入发送缓冲区。100μs 后，应用再调用 write(fd, "STATUS", 6) → 又 6 字节追加到同一个发送缓冲区。TCP 为了效率，把这 12 字节合并成一个 TCP 段发出去了（Nagle 算法 + 缓冲积累）。
接收端：调用 read(fd, buf, 128) → 一次性读到 12 字节："REBOOTSTATUS"。接收端应用完全不知道这是两个 6 字节的消息——它只看到一个 12 字节的字节流。
更糟的情况：发送端写 100 字节，接收端第一次 read 只拿到 40 字节（TCP 段还没到齐），第二次 read 再拿 60 字节——一条消息被拆成两次接收。这就是字节流的核心问题：write 的边界 != read 的边界。应用层必须自己在数据中嵌入边界信息——长度前缀或分隔符——才能重新切分出原始消息。

BLE：消息保留 — 逐层追踪一帧数据怎么到达的

每一层协议都有明确的消息长度字段，下层自动帮上层还原

发送端，自顶向下：
① 应用层：gatt_write_char(handle, "REBOOT", 6)
② ATT 层：封装为 ATT PDU = [Opcode:1B][Handle:2B][Value:6B]。ATT PDU 头部的 Opcode 隐式指定了 PDU 格式——接收端据此知道 Value 有 6 字节。
③ L2CAP 层：封装为 L2CAP SDU = [Length:2B][CID:2B][ATT_PDU:9B]。Length = 11 (CID 2B + ATT 9B)。
④ LL 层：L2CAP SDU 共 13 字节(Length 2B + 11B)。假设 LL MTU=27，13B < 27B → 无需分片。封装为一个 LL Data PDU = [Header:2B][Payload:13B][MIC:4B]。

接收端，自底向上：
⑤ LL 层：收到 LL Data PDU → 校验 MIC → 提取 Payload(13B) → 上传 L2CAP。
⑥ L2CAP 层：读 [Length:2B] = 11 → 精确知道这个 SDU 有 11 字节有效载荷 → 提取 CID + ATT_PDU → 上传 ATT。
⑦ ATT 层：读 [Opcode:1B] → 查表得知这是 Write Command → 根据 Opcode 定义的格式解析 [Handle:2B][Value:6B] → 触发回调 on_write(handle, value, len=6)。
⑧ 应用层：收到 value="REBOOT", len=6——与发送端完全一致的 6 字节消息。即使 LL 层做了分片（L2CAP SDU > MTU 时），LL 层也会先重组再上传——L2CAP 层以上永远收不到半包。

一图看懂：同样的发送，TCP vs BLE 接收端看到什么

发送端连续发送两条消息: msg1="REBOOT"(6B) msg2="STATUS"(6B) TCP 接收端 — 字节流, 边界完全丢失: read() 可能返回: "REBOOT" ← 刚好等于第一条 (运气好) read() 可能返回: "REBOOTSTATUS" ← 两条粘在一起 (常见) read() 可能返回: "REBO" ← 第一条被截断 (TCP 分片) + "OTSTATUS" ← 后半段跟第二条粘在一起 → 应用层必须改造数据: 加 Length 前缀或分隔符, 自己拼回原始消息 BLE 接收端 — 消息边界保留, 每次回调拿到完整的一条: on_write(handle, "REBOOT", 6) ← 第一条, 完整 6 字节 on_write(handle, "STATUS", 6) ← 第二条, 完整 6 字节 → 应用层不需要任何帧边界处理, 消息边界由 BLE 协议栈保证

那 BLE 协议帧里到底需不需要什么？

两个维度：长度 和 类型 — 结论不同

Length 不需要：ATT 层回调的 len 参数已经精确告知本帧有多少字节。在帧内再编码一个 Length = 完全冗余——ATT 层替你做了。
Frame Type 取决于架构：不同帧类型走不同 GATT Characteristic（Handle#10=Data, Handle#11=Control）→ 不需要 Type 字节——Handle 已区分。同一 Handle 上复用多种帧（如音频流）→ 需要 首字节 Type。
CRC/Checksum 取决于场景：OTA 需要 CRC16（链路层 CRC24 只覆盖单个 LL PDU，应用层数据跨多个 LL PDU 后无法保证端到端完整性）；小数据不需要（单包 = 单个 LL PDU，CRC24 已覆盖）；音频流不需要（丢帧可接受，PLC 填补）。

回到本质：为什么 BLE 能做到 TCP 做不到的事？

BLE 有 L2CAP SAR 和 ATT 消息模型 — TCP 没有等价物

L2CAP SAR (Segmentation And Reassembly) 是 BLE 协议栈内置的分片重组机制——它保证：不管 LL 层分了多少片，重组后上传的一定是完整的 L2CAP SDU。TCP 没有"消息边界"的概念——TCP 只保证字节按序到达，不保证"一次 send = 一次 recv"。
ATT 的消息模型：ATT PDU 的 Opcode 定义了 PDU 结构——接收端 ATT 层根据 Opcode 查表就知道 Handle 从哪开始、Value 有多长。每个 Opcode 对应的 PDU 格式是固定的——这本质上是协议层级的类型-长度编码，只是做在了 ATT 协议内部而非应用层 payload 中。
总结：不是 BLE "不需要"帧边界——而是 BLE 把帧边界的工作下沉到了 ATT/L2CAP 层，对应用层透明。TCP/UART 没有这些中间层，帧边界的责任完全落在应用层。

本文档协议帧的设计原则
① 不在 Payload 中编码 Length——ATT 回调已提供 len。
② 帧类型优先用 GATT Handle 区分——OTA Data vs Control、Command vs Status 走不同 Handle → 零字节开销。
③ 仅在同一 Handle 上复用多种帧时用首字节 Type——如音频流（0x00/0x02/0xFF）。
④ 仅 OTA 需要 Payload CRC——因为是跨 LL PDU 组装的大块数据，链路层 CRC24 不覆盖端到端。

场景 1: OTA 固件升级

传输几百 KB 到几 MB 的固件镜像——一块数据错误就可能导致设备变砖。可靠性第一，速度第二。

GATT 路由: OTA Data Characteristic (Handle#0x0012) 承载 Block 帧 · OTA Control Characteristic (Handle#0x0013) 承载 ACK/Control 帧 · 两个不同的 Handle → 不需要帧内 Type 字节。每个帧仅含纯业务数据。

OTA Data Block 帧格式 (Handle#0x0012, Write Without Response, MTU=247 时 Payload 最大 236B)

Block Index: 从 0 递增的序号。接收端用于检测乱序（非连续 Index → 请求重传缺失 Block）、去重（收到重复 Index → 丢弃）、断点续传（重连时接收端上报 Highest Continuous Index → 发送端从此处继续）。
Data Length: 实际有效载荷长度。最后一个 Block 通常 < 236 字节。接收端据此知道 Payload 边界——不能依赖 MTU 推断长度。
CRC16: CCITT CRC16 (多项式 0x1021, 初始值 0xFFFF)。覆盖范围: Block Index (2B) + Data Length (2B) + Payload (N B)。每 Block 独立校验——接收端检测到 CRC 不匹配 → 将该 Block Index 标记为缺失 → 在 ACK 中请求重传该 Block，而非整个固件。

接收端 ACK 确认帧格式 (GATT Write Request, 需 Response 确认送达)

Highest Continuous Index: 已连续收到的最高 Block 序号（不含缺口）。发送端据此判断"哪段是完成的"——此序号之前的所有 Block 都不需要重传。 Missing Bitmap: 16-bit 位图, bit0 = Index (Highest+1), bit1 = (Highest+2), ... bit15 = (Highest+16)。1 = 缺失/CRC错误, 0 = 已收到。
示例: Highest=100, Bitmap=0x0012 (bit1+bit4) → Block 101 和 Block 104 缺失 → 发送端重传这两个 Block 后继续。

ACK 触发时机：不是每个 Block 都 ACK

每 16 个 Block 或 500ms 超时 → 发一次 ACK

4200 次 Block 全部逐包 ACK → 额外 4200 × 2 × 15ms = 126s 往返延迟——不可接受。改为累积 ACK + 缺失位图：每收到 16 个新 Block（或 500ms 无新 Block 超时）发一次 ACK——ACK 次数从 4200 降到 ~263 次——节省 93% 的 ACK 往返开销。
超时 ACK 的必要性：如果发送端刚好发完最后 5 个 Block 等待 ACK → 永远达不到 16 个阈值 → 500ms 超时触发 ACK → 接收端确认收完。

OTA 状态机：四阶段生命周期

IDLE → TRANSFER → VERIFY → APPLY

IDLE：App 发送 Start OTA Request（含固件版本、大小、CRC32 期望值）。设备检查版本号（禁止降级）、检查剩余 Flash 空间 → 回复 ACK 或 REJECT。
TRANSFER：Block-by-Block 传输。接收端边收边写入 Flash（双缓冲：一个 Page 收满 → 整页写入 → 切换另一 Page 继续收）。
VERIFY：所有 Block 收完。接收端计算整个固件的 CRC32 → 与 Start OTA 中的期望值比对 → 不匹配 → 返回错误码 → App 重传。
APPLY：CRC32 验证通过。设备设置 Boot Flag → 重启 → Bootloader 检测 Flag → 校验固件签名 → 拷贝到运行区 → 清除 Flag → 跳转新固件。

⚠ 每个 Block 独立 CRC 的必要性 — 链路层 CRC 不够
BLE 链路层已有 3 字节 CRC24 校验每个 LL PDU。但链路层 CRC 只保证单个 LL PDU 的完整性——当一个应用层 Block 跨越多个 LL PDU 时，每个 PDU 的 CRC 都正确不等于整个 Block 的端到端完整性（ATT/GATT 层不做校验聚合）。应用层 CRC16 覆盖整个 Block 的所有字节——是 OTA 可靠性的最后防线。

场景 2: 小数据传输 — 命令与状态

包 < 20B · 频率 0.1~100Hz · 延迟敏感 · 不需要分片。协议设计的核心是零开销编码、向后兼容的命令体系和快速确认。

GATT 路由: Command Characteristic (Handle#0x0010, Write) 承载命令帧 · Status Characteristic (Handle#0x0011, Notify) 承载 TLV 上报帧 · 不同 Handle → 帧类型由 Handle 隐式确定。命令是 Write Request（带 Response 确认），状态是 Notify（无确认）。

Command ID 分配规则: 0x00~0x7F = 通用命令 (重启/恢复出厂/查询状态/进入OTA)；0x80~0xBF = 功能命令 (拍照/LED/电机/蜂鸣)；0xC0~0xFE = 厂商扩展；0xFF = 保留。
Parameters 编码: 各命令自定义。拍照命令 → 0 字节 Param (仅 CmdID 足够)；设置灵敏度 → 1 字节 Param (0-100)；WiFi 配网信息 → N 字节Param (SSID+Password)。

Status Code 定义: 0x00 = SUCCESS (命令已执行)；0x01 = ERR_PARAM (参数无效)；0x02 = ERR_STATE (当前状态下不可执行, 如 OTA 中不能拍照)；0x03 = ERR_BUSY (设备忙, 稍后重试)；0x04 = ERR_TIMEOUT (执行超时)；0x05 = ERR_NOT_SUPPORTED (固件不支持此命令)；0x10~0xFF = 厂商自定义错误。
Response Data: 查询命令需携带返回数据。如 CmdID=查询电池 → Data=1B (0-100)；CmdID=拍照 → Data=0B (仅 Status 即可)；CmdID=查询固件版本 → Data=3B (主.次.修订号)。

Type 分配: 0x01=电池, 0x02=温度, 0x03=RSSI, 0x04=活动计数, 0x05=错误码, 0x10~0x1F=传感器数据, 0x80~=厂商扩展。

小数据场景为什么不需要分片、重传和 CRC？
不分片: 单包 20B (default MTU) 或 244B (negotiated) → 命令和状态都在单包内 → GATT 层的原子操作保证"要么整包到、要么整包不到"。
不应用层重传: Write Request 有 ATT 内置超时重传 (30s supervision timeout)。Write Response 回带 Status → 命令是否执行成功在 ATT 层原子确认。
不应用层 CRC: 单包数据 = 单个 L2CAP PDU = 单个 LL PDU → 链路层 CRC24 覆盖完整 → 端到端完整性在 LL 层已保证。

场景 3: 编码音频流传输

AAC/Opus 编码后每秒 2~32 KB/s。丢一帧音频 = 听感卡顿而非功能失败——允许丢帧，不能容忍延迟累积。

GATT 路由: Audio Stream Characteristic (Handle#0x0020, Write Without Response) 单一 Handle 承载所有音频帧——音频数据帧、编码配置帧、流结束帧复用同一 Handle。因此第一个字节作为 Frame Type 区分帧类型（0x00=音频, 0x02=配置, 0xFF=结束）。

音频流帧格式 (Handle#0x0020, Write Without Response, 连续发送)

偏移01234 .. 3+PayloadLen
字段 Frame Type
uint8		 Sequence
uint8		 Timestamp Delta
uint16 LE, ms		 Encoded Payload
PayloadLen 字节
Frame Type: 0x00 = 音频帧 (AAC/Opus), 0x01 = 静音帧 (SID, Comfort Noise), 0x02 = 配置帧 (Codec 参数变更), 0xFF = 流结束。
Sequence (0~255 回绕): Write Without Response 无 ACK → 链路层丢包不通知应用层。接收端检测 Seq 跳跃 (收到 5→8, 缺 6,7) → 统计丢帧率 → 解码端对丢失帧位置做 PLC 插值。检测到重复 Seq → 去重。
Timestamp Delta: 相对上一帧的时间增量 (ms)。Connection Event 的突发到达 (几帧同时到) → 接收端按累计时间戳分散排列播放时间——消除 BLE 传输带来的时域抖动。

环形发送队列 + 老化丢弃策略

编码快于发送 → 队列积压 → 丢弃最老帧而非阻塞

队列设计: 预分配 8-16 个音频帧 buffer（每帧 ~200B，总计 3.2KB）。编码器写入环形队列 → BLE 发送线程取出发送。发送线程检查每帧的累计时间戳——与当前时间差 > 200ms → 直接丢弃。跳过老帧后 Sequence 不连续 → 接收端通过 Seq 跳跃感知丢帧 → PLC 填补。
关键设计决策: 不重传、不等待、不阻塞编码器。音频延迟 > 200ms → 双向对讲不可用——宁可丢帧引起微卡，不可累积延迟。这是音频流与 OTA 协议设计最根本的区别。

为什么必须用 Write Command 而不是 Indicate？

Indicate 每秒最多 66 帧 < 编码 50fps → 不可控积压

GATT Indicate 有 ATT 层强制确认：发一帧 → 等 Confirm → 一个 Connection Event (15ms) 只能完成 1 次 Indicate 往返 → 理论最大 66 帧/秒 (1/0.015)。加上 ATT 处理延迟 + 重传 → 实际 ~40 帧/秒 → 低于 50fps 编码速率 → 队列持续积压 → 延迟无限增长。
Write Command 方案: 无确认 → 每 Event 可连续发 3-5 帧 → 实际有效 >150 帧/秒 → 远超编码速率 → 队列始终保持低水位 (0-2 帧) → 端到端延迟 < 40ms。

音频编码参数协商帧 (带内控制)

Frame Type=0x02 — 不另开 GATT 通道

不需要单独的 GATT Characteristic 来控制音频参数——编码参数变更通过带内控制帧在同一个 Write Command 流中传递。Payload 包含: 1B Codec (0=AAC-LC, 1=Opus, 2=LC3) + 1B Bitrate Index (0~15 对应档次) + 1B Sample Rate (0=8K,1=16K,2=24K,3=48K) + 1B Frame Duration (0=10ms,1=20ms)。
发送端在任何时候插入此帧 → 接收端收到后调整解码器 → 后续音频帧使用新参数。带内控制比独立 GATT 通道的优势：时序上与音频流天然同步——参数切换点精确落在两个音频帧之间。

PLC 丢包隐藏 — 不重传的底气

Packet Loss Concealment 在解码端填补丢失帧

接收端检测到 Seq 跳跃 → 不是请求重传 → 而是调用 PLC 算法：基于前后帧的基频和频谱插值生成丢失帧的近似波形。AAC 和 Opus 解码器内置 PLC（启用 AAC_PLC 或 OPUS_SET_PACKET_LOSS_PERC）。
PLC 对单帧丢失效果极好（几乎无感知），连续 2-3 帧丢失听感微卡，连续 >5 帧丢失 → 明显断续但不引入额外延迟。而重传 5 帧 → RTT × 5 = 75ms 额外延迟 + 等待 → 对讲体验完全不可用。

⚠ 音频流的 Sequence 不是可靠性机制——是诊断和统计工具
Sequence 不用于确认（无 ACK/NACK），不用于重传。它的唯一作用是：① 接收端统计丢帧率（丢帧率 > 5% → 触发自适应降码率）② 解码端确定需要 PLC 填补的具体位置（Seq=7 丢了 → PLC 在 6 和 8 之间插值）③ 发送端诊断——每次 Connection Event 后记录发出的 Seq 范围 → 控制器队列满导致丢帧 → 日志中可见 Seq 间隙 = 控制器队列深度不足 → 调整队列深度参数。

传输性能优化

默认 BLE 配置的吞吐远低于硬件能力——以下优化可以将有效吞吐从 ~50Kbps 提升到 ~800Kbps。

连接间隔设最小

15ms → 7.5ms: 吞吐接近翻倍

每个 Event 有 1-2ms 无线电开销（死时间）。间隔越短，Event 密度越高 → 吞吐越高。OTA/音频用 7.5-15ms。iOS 最低只允许 15ms（Apple 硬限制）。

批量 Write Without Response

每 Event 连续发 4-6 包不等 Response

Write Command 无 Response → 同一 Event 可连发直到控制器队列满。2M PHY + DLE + 15ms 间隔 → ~640Kbps 有效吞吐。

DLE 使能 (BLE 4.2+)

LL PDU 从 27B → 251B, 吞吐 2-3x

Data Length Extension 将链路层 payload 扩展到 251B。每 Event 可发更多 ATT 包。对上层协议透明（ATT/GATT 不感知），但效果巨大——OTA 传输时间缩短 75%。

PHY 协商 (BLE 5.0)

2M PHY 物理层码率翻倍

1Msps→2Msps 符号率。实际有效吞吐 700K→1.4Mbps。代价：接收灵敏度 -3dB（距离稍短）。近距离(<5m)用 2M、远距离自动回退 1M。

缓存 GATT Handle

跳过 Discovery → 首次通信 3s→100ms

首次连接后缓存所有 Characteristic Handle 到本地——后续重连跳过 GATT Discovery。仅在 Service Changed Indication 触发时重新 Discovery。iOS 做一次完整 Discovery 耗时 2-5 秒。

IPC 实战：蓝牙项圈三通道协议设计

LinkricAI IPC 通过 BLE 与 Petkit P3D 项圈通信——小数据状态 + 命令下发 + OTA 升级三通道混合场景。

通道 1: 状态上报 (Notify)

电量/温度/RSSI/活动量 · 0.1~1Hz · TLV 编码

项圈周期性推送状态——走 GATT Notify（无需确认）。TLV 编码保证向后兼容。单包 < 20B。

通道 2: 命令下发 (Write Req)

寻铃/灯光/灵敏度 · 1B CmdID + Status

App 下发控制命令。Write Request + Write Response 保证确认。超时 500ms → 重试 3 次 → 仍失败 → 通知用户。

通道 3: 项圈 OTA

~200KB 固件 · Block 分片 + CRC16 + ACK

Block Index + Data + CRC16 + 累积 ACK 位图。OTA 前切换高速参数 (15ms+2M+MTU247)，完成后恢复省电模式。

BLE 应用传输协议 Checklist
□ MTU 协商: 连接后立即 Exchange MTU → 目标 247。检查协商结果——某些设备限制 65/128
□ DLE 使能: BLE 4.2+ 默认开启，验证双方支持
□ 连接参数: 正常 100ms 省电；OTA/音频切 15ms+2M PHY；完成后恢复
□ Handle 缓存: 首次 Discovery 后缓存 → 重连跳过 (3s→100ms)
□ 分片策略: 单包 < MTU-3 直发；超限则 Block 编号 + CRC + 累积 ACK
□ 校验策略: OTA = CRC16/Block；小数据 = LL CRC 足够；音频 = 不校验
□ 重传策略: OTA = NACK+重传；命令 = ATT 超时重试；音频 = PLC 不重传
□ 兼容性: TLV 编码 (数据)、CmdID 范围预留 (命令)——保证新老固件兼容
□ 错误处理: GATT 错误 → 重试 3 次 → 断连重连 → 仍失败 → 通知用户
