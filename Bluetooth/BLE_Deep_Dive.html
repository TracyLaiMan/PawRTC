1. BLE 概述与演进

从 Bluetooth Classic 到 Bluetooth LE — 两条技术路线的分道扬镳

Bluetooth Classic (BR/EDR)

Basic Rate / Enhanced Data Rate

始于 1999 年 (BT 1.0)。目标：持续数据流 — 音频 (A2DP)、文件传输 (OBEX)、电话 (HFP)。
数据率 1~3 Mbps，79 个 1MHz 信道，连接延迟 ~100ms，功耗较高 (~40mA)。

Bluetooth Low Energy (LE)

2010 · BT 4.0 · 原名 Wibree (Nokia)

目标：低功耗、小数据、突发传输 — 传感器、可穿戴、IoT。
数据率 125kbps~2Mbps，40 个 2MHz 信道，连接延迟 ~3ms，峰值电流 ~15mA。

3. 物理层 (PHY) — 射频 & 信道

2.4GHz ISM — 40 信道 × 2MHz 间隔 — GFSK 调制 — 射频前端 — 链路预算

频段 & 信道

2402 MHz ~ 2480 MHz

中心频率公式：f = 2402 + 2k MHz，其中 k = 0..39
广播信道：ch37 (2402), ch38 (2426), ch39 (2480)
数据信道：ch0~ch36 (共 37 个)
信道间隔 2MHz（WiFi 避开 3 个广播信道，分散在 WiFi 信道间隙）。

调制方式

GFSK — Gaussian Frequency Shift Keying

1M PHY: BT=0.5 GFSK，调制指数 0.45~0.55，符号率 1 Msps → 1 Mbps
2M PHY (BT 5.0): BT=0.5 GFSK，调制指数 0.45~0.55，符号率 2 Msps → 2 Mbps
Coded PHY (BT 5.0): 前向纠错 FEC，S=2 (500kbps) / S=8 (125kbps)

💡 为什么广播信道选 37/38/39？
WiFi 最常用的三个 non-overlapping 信道是 1 (2412), 6 (2437), 11 (2462)。BLE 广播信道正好落在它们之间的间隙：2402 (ch37) 在 WiFi ch1 下方，2426 (ch38) 在 ch1 和 ch6 之间，2480 (ch39) 在 ch11 上方。这种精心设计的避开策略保证了广播的抗干扰能力。

3.1 GFSK 调制 — 数学原理与硬件实现

GFSK 调制原理

Gaussian Filtered Frequency Shift Keying

FSK (频移键控): 数字信号直接控制 VCO 频率，bit '1' → f_c + Δf，bit '0' → f_c − Δf
GFSK 在 FSK 的基础上插入一个高斯低通滤波器，将陡峭的矩形比特脉冲平滑化，使频率过渡更柔和，从而压缩信号带宽。
代价是引入码间干扰 (ISI)——BT 积越小，滤波越强，带宽越窄，但 ISI 越严重。

关键参数

BT 积 · 调制指数 h

BT 积 (Bandwidth-Time product): 高斯滤波器 −3dB 带宽 B 与 bit 周期 T 的乘积。BLE 使用 BT = 0.5。
调制指数 h: h = 2 × Δf / R，其中 Δf 是峰值频偏，R 是符号率。BLE 规定 h = 0.45~0.55（宽松于经典 FSK 的 h=0.5 硬要求，允许工艺偏差）。
对于 BLE 1M PHY: Δf ≥ 185 kHz（蓝牙规范最小频偏要求），Δf_avg = 225~275 kHz (典型)。
频率偏差容限：±150 kHz 载波频率偏移 + ±50 kHz 符号间频率漂移。

发射信号链 (TX Path)

基带 → 射频 — IQ 调制器架构

1. 数据白化 (Whitening): 比特流与 PRBS9 序列异或，消除长连 0/1 的 DC 偏置
2. 符号映射: 0→−1, 1→+1
3. 高斯滤波 (FIR): 数字域用查表法 (LUT) 实现的 FIR 低通滤波器，截断长度通常 3~5 符号周期
4. 相位累加器: 积分滤波后的符号得到瞬时相位 φ(t)
5. CORDIC / NCO: 相位 → I/Q 正交信号 (cos φ + j·sin φ)
6. IQ-DAC: 数字 I/Q → 模拟 I/Q 基带信号
7. IQ 调制器 (上变频): I/Q 信号混频至 2.4GHz 载波，PA 放大后经匹配网络到天线

接收信号链 (RX Path)

射频 → 基带 — 低中频 (Low-IF) 架构

1. 天线 + 匹配网络: 50Ω 阻抗匹配 →
2. BPF / SAW 滤波器 (可选): 带外抑制 (主要抑制 WiFi 和蜂窝干扰)
3. LNA: 低噪声放大，NF ~2~5dB，增益 15~25dB
4. IQ 解调器 (下变频): LNA 输出与 LO 混频，产生 I/Q 基带信号
5. 低通 / 带通滤波器: 信道选择，抑制邻道干扰
6. IF-ADC: 通常分辨率为 12~14 bit
7. 数字解调: 频偏估计 → 符号同步 → 判决 (多数表决 / Viterbi 译码) → 解白化

3.2 射频前端 (RF Frontend) — 核心模块详解

频率综合器 (PLL)

LO 生成 · Sigma-Delta 小数分频

基于 Sigma-Delta 小数 N 分频 PLL。
参考时钟 (32MHz 晶振) → PFD (鉴频鉴相器) → Charge Pump → Loop Filter → VCO。
锁相时间: ≤ 150µs (信道切换)。
相位噪声: −95 dBc/Hz @1MHz offset (关键指标，影响邻道选择性)。
VCO 调谐范围覆盖 2402~2480 MHz，KVCO 约 50~200 MHz/V。

LNA

低噪声放大器

接收链路第一级有源器件，决定了整机噪声系数 (NF) 的下限。
增益: 15~25 dB (典型)
NF: 2~5 dB
IIP3: −5~−15 dBm (线性度，影响抗阻塞能力)
常见结构: 共源共栅 (cascode) 电感退耦 LNA，源极电感匹配至 50Ω。

PA

功率放大器

BLE 发射功率可编程，典型 −20~+4 dBm (片内 PA)，加外部 PA/FEM 可达 +20 dBm。
片内 PA 效率: 20~35% (PAE)
谐波抑制: 二次谐波 < −40 dBm, 三次谐波 < −40 dBm
输出匹配: 差分转单端 balun + π/T 型匹配网络。

T/R Switch

收发切换 (半双工)

BLE 是 TDD (时分双工): 发射和接收在时间上互斥。
TX→RX 切换时间: ≤ 150µs
片内开关通常使用 MOSFET 开关 (串联-并联拓扑)，插入损耗 < 1dB，隔离度 > 30dB。

3.3 链路预算 (Link Budget) & 传播模型

链路预算公式

Friis 传输方程 (自由空间)
// 接收功率 (dBm) P_rx = P_tx + G_tx + G_rx − L_path − L_misc // 自由空间路径损耗 (Friis) L_path = 20 × log₁₀(d) + 20 × log₁₀(f) + 32.44 d: 距离 (km) f: 频率 (MHz) // 链路余量 Margin = P_rx − Sensitivity // BLE 1M PHY 典型计算 @ 10m P_tx = +0 dBm, G_tx = G_rx = 0 dBi (片上天线) f = 2440 MHz → L_path = 60.2 dB @ 10m P_rx = 0 − 60.2 = −60.2 dBm Sensitivity = −93 dBm → Margin = 32.8 dB ✅

室内传播损耗修正

ITU-R P.1238 室内模型
// ITU 室内路径损耗模型 L_total(d) = 20·log₁₀(f) + N·log₁₀(d) + L_f(n) − 28 // N: 距离功率损耗系数 // 住宅: 28, 办公室: 30, 商业: 22 // L_f(n): 楼层穿透损耗 // 住宅: 4n (n=楼层数), 办公室: 15+4(n−1) // 人体遮挡额外损耗: 3~10 dB

灵敏度决定因素

RSSI · SNR · PER

接收灵敏度: Psen = −174 + 10·log₁₀(BW) + NF + SNRmin
−174 dBm/Hz: 室温热噪声底 (kTB)
BW: 1 MHz (1M PHY) → 10·log₁₀(1e6) = 60 dB
NF: 接收机噪声系数 (典型 5~8 dB，含 LNA+Mixer+ADC)
SNRmin: 解调所需最小信噪比 (GFSK 约 12~15 dB)
计算: −174 + 60 + 6 + 15 = −93 dBm ✓
每提高 3dB 灵敏度，链路余量翻倍，覆盖距离增加约 40%。

RSSI & 距离估算

Received Signal Strength Indicator

BLE 芯片内部提供 RSSI 寄存器（通常 1dB 分辨率，±4dB 精度）。
基于 RSSI 的距离估算 (对数距离路径损耗模型):
d = 10^((TxPower − RSSI) / (10 × n))
其中 n (路径损耗指数): 自由空间 n≈2.0，办公室 n≈2.5~3.5，工厂 n≈3.0~5.0。
⚠️ RSSI 受多径衰落、天线极化、人体遮挡影响剧烈 — 精度通常 ±3~5m。

3.4 BLE SoC 硬件架构 — 从晶振到天线

典型 BLE 芯片内部框图 — 电源域、时钟树、数字基带、模拟/RF 子系统

SoC 内部架构全景

CPU 子系统

ARM Cortex-M0/M4/M33

典型的 BLE SoC 使用 ARM Cortex-M 系列（nRF52: M4, nRF53: M33, CC26xx: M4, DA145xx: M0）。
主频 16~128 MHz，集成 NVIC、SysTick、MPU (可选)。
部分芯片采用 双核 架构 (nRF5340: App Core M33 + Net Core M33 分离应用和协议栈)。

存储系统

Flash + SRAM + OTP

Flash: 128KB~1MB，存储协议栈 (SoftDevice) + 应用固件。支持 OTA/DFU。
SRAM: 16~512KB，协议栈 RAM + 应用 heap/stack。
OTP/UICR: 存储 MAC 地址、校准数据、安全密钥。
Flash 编程电压由内部 Charge Pump 产生 (1.8V→~12V)，擦写寿命 10k~100k 次。

外设接口

数字与模拟外设

数字: UART/SPI/I²C/QSPI/QEI/PDM/I²S/USB
模拟: ADC (12/14-bit SAR, 100ksps~1Msps)、比较器、温度传感器
定时器: RTC (低功耗 32.768kHz 时基)、Timer/Counter
安全: AES-128 HW 加速器、TRNG、CryptoCell (ARM TrustZone)
DMA: EasyDMA / μDMA 自动搬运数据

电源管理

多电源域 · 深度睡眠

供电范围: 1.7V~3.6V (典型 CR2032 硬币电池：3.0V)。
内部 LDO / DC-DC Buck 产生核心电压 (1.1~1.3V)。
System ON: CPU + 外设 + RF 可工作，功耗 0.5~10mA
System ON Idle: CPU 休眠，外设/定时器保持，~3µA
System OFF: 仅 GPIO 唤醒 / NFC 唤醒，~0.3µA
System OFF (no RAM retain): ~0.1µA
DC-DC 效率 ~85%，LDO 模式约高 30% 功耗但无开关噪声。

3.5 时钟系统 — 精度就是一切

主时钟 (HCLK)

32MHz 晶振 — RF 频率综合器参考源

频率: 32.000 MHz (±40 ppm 初始精度)
对 RF 的影响: PLL 将 32MHz 倍频到 2.4GHz → 频率误差 = 晶振误差 × (2400/32) = ×75
40 ppm 晶振误差 = 40 × 2400 / 1e6 = ±96 kHz 载波误差。BLE 规范要求 中心频率误差 ≤ ±150 kHz。
温度漂移: ±20 ppm over −40°C~+85°C → 额外的 ±48 kHz 误差。
常用解决方案: 外部 32MHz TCXO (±2 ppm，但增加功耗和成本) 或 软件频偏校准。

低功耗时钟 (LCLK)

32.768 kHz — 休眠计时基准

频率: 32.768 kHz
用途: 连接间隔计时 (connInterval)、广播间隔计时 (advInterval)、超时检测。
可能来源：
外部 32.768 kHz 晶振: ±20 ppm，功耗 ~0.2µA，精度最高
内部 RC 振荡器: ±500 ppm (需软件校准到 32MHz 主时钟)，功耗 ~1µA。常用 nRF52/CC26xx
合成自 32MHz: 分频产生，无需额外晶振
关键要求: 连接事件时间误差 < ±2µs，否则错过 Master 的发送窗口导致丢包。

时钟启动时序

冷启动 → 稳定 → RF 活跃

从 System OFF 唤醒到 RF 发射的完整时序：
1. 内部 LDO/DC-DC 建立: 50~100µs
2. 32MHz 晶振起振: 200~500µs (取决于晶振 Q 值和负载电容)
3. PLL 锁定: 50~150µs (Sigma-Delta PLL 锁定时间)
4. 基带准备: RAM restore + 协议栈状态恢复 ~200µs
总唤醒时间: ~500~1000µs。这是决定 BLE 最低功耗的关键 — 每次 RF 事件 (广播/连接) 都从睡眠唤醒再休眠。

晶振电路设计

PCB Layout 要点

负载电容 CL: 晶振规格书的推荐 CL (通常 6~12pF)。
CL1 = CL2 = 2 × (CL − Cstray)
Cstray ≈ 2~5 pF (PCB 走线 + 引脚寄生)。
Layout 规则:
• 晶振紧邻芯片 XTAL 引脚，走线 < 10mm
• 对称走线，等长，地线包围
• 下方禁止走数字信号线（防止耦合干扰）
• 负载电容接地回路直接连芯片 VSS
常见故障: CL 选错导致频率偏移超出 ±150 kHz → 设备无法被发现或频繁断连。

3.6 天线设计 & 阻抗匹配

PCB 天线

零成本 · 占用 PCB 面积

IFA (倒F天线): 最常用。尺寸约 15×8mm，单端，50Ω。
Meander (蛇形): 曲折线缩短尺寸，效率 40~60%，带宽 80~100MHz。
PIFA (平面倒F天线): 需要地平面，带宽较宽，常用于手表。
关键参数: 驻波比 VSWR < 2.0 (S11 < −10dB)，效率 > 50%，增益 > −3dBi。
需要 π 型匹配网络 (L-C-L 或 C-L-C) 调谐至 2440 MHz 中心频率。

陶瓷贴片天线

紧凑 · 标准化 · 成本低

尺寸: 3.2×1.6mm ~ 7.0×2.0mm
厂家: Johanson, Yageo, TDK, Fractus
效率: 40~70% (受 PCB 地平面影响大)
优点: 体积小，无需定制设计，出厂已调谐
缺点: 需要足够的净空区域 (Keep-out Zone: 天线本体周围 5~10mm 内禁止铺铜)，对 PCB 材质和厚度敏感。

外置天线

最佳性能 · IPEX/U.FL 连接器

鞭状天线: 增益 ~2dBi，效率 > 80%
偶极子天线: 理论增益 2.15dBi
PCB 棒状天线: 灵活布局，增益 1~3dBi
适用场景: 智能网关、车载设备、工业设备
注意: 外置天线需要 FCC/CE/SRRC 认证。

📡 阻抗匹配网络 — 史密斯圆图
参考阻抗: 50Ω (单端)。芯片侧差分输出 → Balun (平衡-不平衡转换器) → 50Ω 单端 → π/T 型匹配网络 → 天线。
π 型网络 (推荐): Shunt C → Series L → Shunt C。提供灵活的二阶匹配，可同时调谐阻抗和带宽。
调试流程: VNA (矢量网络分析仪) 测量 S11 → 在史密斯圆图上找到当前阻抗点 → 计算所需 L/C 值 → 焊接验证 → 复测 S11 确认 S11 < −10dB @ 2400~2480MHz。

3.7 射频共存 (Coexistence) & 电磁兼容

2.4GHz 频谱竞争者

WiFi · Zigbee · Thread · 微波炉

WiFi: 11b/g/n/ax 占 20/40MHz BW，功率 +15~+20dBm。BLE 信号在频谱上被 WiFi 完全覆盖。
Zigbee/Thread: O-QPSK, 250kbps, 2MHz BW, 5MHz 信道间隔 (16 个信道在 2.4GHz)。与 BLE 部分重叠。
微波炉: 2450MHz 中心频率，宽频带 (~50MHz BW) 脉冲干扰 (~60Hz 占空比)。
USB 3.0: 2.5GHz 数据率产生的谐波干扰 (795MHz × 3 ≈ 2.385GHz)。

BLE 抗干扰机制

AFH · 重传 · CRC · 信道选择

1. AFH (自适应跳频): Channel Map 动态标记"坏信道"并排除。坏信道判定标准: CRC 错误率 > 阈值。
2. 快速重传: 无 ACK 时在下一个连接事件 自动重传，重传信道可能不同（跳频）。
3. 24-bit CRC: 强度够高 (残留错误率 < 2e-7)，检测到错误触发重传。
4. 白化 (Whitening): 数据随机化降低窄带干扰的误码相关性。
5. CSA#2 (BT 5.0+): 改进的伪随机跳频算法，信道分布更均匀。

WiFi/BT 共存方案

PTA · 天线隔离

在手机/平板等同时有 WiFi 和 BLE 的设备中，两者共享 2.4GHz 频段。共存策略：
PTA (Packet Traffic Arbitration): WiFi 和 BT 芯片之间通过 3 线接口 (REQUEST/GRANT/PRIORITY) 实时仲裁。
TDM (时分复用): eSCO/SCO 链路分配专用时隙，WiFi 在此期间静默。
天线隔离: 两根天线空间距离 > λ/2 (~6cm @ 2.4GHz) 可获得 > 15dB 隔离。
频域分离: WiFi 使用 ch1/ch6/ch11，BT 使用其间隙信道。

EMC/EMI 设计要点

PCB Layout · 去耦 · 屏蔽

去耦电容: 芯片每路电源引脚的 100nF 旁路电容尽可能贴近引脚 (< 2mm)，高频段加 10pF 并联。
DC-DC 噪声: 开关频率 (1~2MHz) 和谐波可能耦合到 VCO/PA。DC-DC 电感远离 RF 区域，加 LC 滤波。
地层完整性: RF 区域下方完整地平面 (不可分割)，过孔围栏 (via fence) 围绕 RF 走线。
传输线: RF 走线用共面波导 (CPWG) 或微带线，严格控制 50Ω 特性阻抗。
屏蔽罩: 金属屏蔽罩覆盖 RF 区域，接地良好，可降低辐射发射 10~20dB。

4. 链路层 (Link Layer) — 状态机与空中包

定义设备角色、广播/扫描/连接状态、空中包格式、跳频算法

4.1 状态机

Standby

初始状态 · 不收发

设备上电后的默认状态。不发送、不接收任何包。可从任何状态返回。在此状态下决定下一步角色：Advertiser / Scanner / Initiator。

Advertising

广播 · 通告自己的存在

在 3 个广播信道上周期性发送 ADV_IND 包。广播间隔 (advInterval)：20ms~10.24s。
可附带最多 31 字节广播数据 (BT 4.x) 或 1650 字节 (BT 5.0 Extended Advertising)。

Scanning

扫描 · 监听广播

Passive Scan: 只监不回复。
Active Scan: 收到 ADV 后发送 SCAN_REQ 请求更多信息，广播方回复 SCAN_RSP。

Initiating

发起连接 · CONNECT_IND

Scanner 收到目标广播后，在同一信道上回复 CONNECT_IND 包，携带连接参数（间隔、延迟、超时、跳频图）。双方随即进入 Connection 状态。

Connection

已连接 · Master/Slave

Master 管理连接时序和跳频。
Slave 按 Master 调度回应。
connInterval = 7.5ms~4s，每次连接事件可交换多包数据。

Isochronous (BT 5.2+)

等时信道 · LE Audio

CIS (Connected Isochronous Stream) 有连接等时流。
BIS (Broadcast Isochronous Stream) 无连接广播等时流。
为 LE Audio / Auracast 提供定时可靠的音频传输。

4.2 空中包格式 (LE 1M / 2M PHY)

字段大小说明
Preamble1 字节 (1M) / 2 字节 (2M)交替 0/1 序列，用于接收机 AGC 和时钟同步
Access Address4 字节广播固定 0x8E89BED6；连接为随机 32-bit（不得为广播地址）
PDU Header2 字节PDU 类型 (4b) + RFU (1b) + ChSel (1b) + TxAdd (1b) + RxAdd (1b) + Length (8b)
PDU Payload0~255 字节 (BT 4.2+)实际数据载荷，包含 AdvData 或 L2CAP 负载
MIC4 字节仅加密链路层包有 Message Integrity Check
CRC3 字节24-bit CRC，多项式 = x²⁴ + x¹⁰ + x⁹ + x⁶ + x⁴ + x³ + x + 1

4.3 自适应跳频 (AFH)

连接建立后，Master 和 Slave 在 37 个数据信道上伪随机跳频。Channel Map 指示哪些信道可用/被屏蔽（例如与 WiFi 干扰严重的信道）。跳频算法：

// 连接事件 n 使用的信道 fn = (HopIncrement × n) mod 37 channel = remap(fn, ChannelMap) // remap(): 如果 fn 对应信道被屏蔽，映射到最近可用信道 // BT 5.0+ 可选 Channel Selection Algorithm #2 (CSA#2): 更均匀的分布

5. L2CAP — 逻辑链路控制与适配

上层多协议复用 · 大数据包分片重组 · 面向连接/无连接信道

固定信道 CID

Channel Identifier — 协议多路复用

0x0004: ATT (Attribute Protocol)
0x0005: LE L2CAP Signaling
0x0006: SMP (Security Manager)
0x0020-0x003E: 动态分配 (面向连接信道)
0x003F: 固定值

LE Credit Based Flow Control

BT 4.1+ · 面向连接信道

类似 TCP 的信用额度流控。LE Credit Based Connection Request 建立连接时协商初始信用额度 (MTU 和 MPS)。
每发一个 LE-frame，信用额度 -1；收到 LE Flow Control Credit 信号，额度 +N。

分片与重组 (SAR)

Segmentation And Reassembly

应用层 ATT/GATT 数据可超过 Link Layer PDU (251B)。L2CAP 将 SDU 分段为多个 LE-frames：
First (开始) → Continuation (连续) → ... → Continuation (最后)
接收端根据序列重组为完整 SDU。

EATT (Enhanced ATT)

BT 5.2+ · 多 ATT 信道并行

允许同时存在多个 L2CAP ATT 信道（不同的 CID），每个信道独立 flow control。
解决传统 ATT 串行化瓶颈 — 一个慢操作不再阻塞所有 ATT 通信。

📦 LE Data Length Extension (BT 4.2)
BT 4.0/4.1 的 Link Layer PDU 最大只有 27 字节有效载荷。BT 4.2 引入 LE Data Length Extension，将 PDU payload 扩展到最多 251 字节。单次连接事件可以连续发送多个包（最多约 6 个满载荷包），有效吞吐率从 ~0.3 Mbps 提升到 ~1.4 Mbps (1M PHY) / ~2.8 Mbps (2M PHY)。

6. ATT & GATT — 数据组织核心

ATT 是传输协议 · GATT 是数据模型 · Service / Characteristic / Descriptor 三级树形结构

6.1 ATT 协议 — 属性操作

Attribute 结构

基本数据单元
struct attribute { uint16_t handle; // 0x0001~0xFFFF 索引 uuid_t type; // 2/4/16 字节 UUID uint8_t perm; // 访问权限 (R/W/Enc/Auth) uint8_t* value; // 可变长数据 };

ATT PDU 类型 (Method)

6 种操作 × 多种响应

Read: Read Request → Read Response / Read Blob
Write: Write Request → Write Response / Write Command (无响应)
Notify: Handle Value Notification (Server→Client, 无确认)
Indicate: Handle Value Indication → Confirmation
Find: Find By Type Value Request → Response
Read Multiple: Read Multiple Request → Response

⚠️ ATT 串行化限制
传统 ATT 是严格 请求-响应 协议：一次只能有一个未完成的 ATT 事务。如果一个 Write Request 需要长时间处理，所有后续 ATT 操作都会被阻塞。BT 5.2 的 EATT 通过多信道并行解决了这个痛点。

6.2 GATT — 树形数据模型

Profile

应用层规范 · 角色定义

一个 Profile 定义一个应用场景（如 Heart Rate Profile、Battery Service Profile）。
包含一个或多个 Service，定义 GATT Client 和 GATT Server 的行为约定。

Service

UUID 标识 · 功能集合

Primary Service: 独立功能，如 0x180D (Heart Rate)
Secondary Service: 辅助功能，被其他 Service 引用
包含一组相关的 Characteristic。

Characteristic

数据载体 · Properties

包含一个 Value (实际数据) + 一组 Descriptor。Properties 位掩码：
Read | Write | Notify | Indicate | Broadcast | WriteWithoutResponse
如 Heart Rate Measurement (UUID 0x2A37)。

Descriptor

元数据 · 配置

0x2902 CCCD: Client Characteristic Configuration — 开关 Notify/Indicate
0x2901 CUD: 用户描述字符串
0x2904 CPF: 数据格式说明 (int/float/string)
0x2908 SCCD: Server 端配置

✅ GATT 发现流程 (Service Discovery)
Client 连接后遍历 Server 的属性表：
Discover All Primary Services → 获取 Service UUID 和 handle 范围 → Discover Characteristics → 获取 Characteristic handle + Properties → Discover Descriptors → 获取 CCCD 等 → Write CCCD (0x0001 开启 Notify) → 开始数据流。

7. GAP — 通用访问规范

定义设备角色、广播模式、发现流程、连接参数 — BLE 的"交通规则"

7.1 GAP 角色

Broadcaster

纯广播 · 不接收连接

周期性广播，不接受连接请求。典型应用：iBeacon、Eddystone、温度传感器信标。只发不收，功耗最低。

Observer

纯扫描 · 不发起连接

被动监听广播，不发起连接。典型应用：室内定位网关（抓取所有 Beacon 信号上报云端）。

Peripheral

Broadcaster + 可被连接

广播 + 接受连接请求。对应 Link Layer Slave 角色。GATT Server 在此角色上。
典型应用：心率带、温度传感器、智能手表 (外围模式)。

Central

Scanner + 发起连接

扫描 + 发起连接。对应 Link Layer Master 角色。GATT Client 在此角色上。
典型应用：手机、PC — 一个 Central 可同时连接多个 Peripheral。

7.2 广播类型

PDU 类型名称可连接可扫描定向说明
ADV_IND通用广播指示✓✓—最常用：可被扫描和连接
ADV_DIRECT_IND定向连接广播✓—✓指向特定设备地址，快速重连 (1.25ms 间隔)
ADV_NONCONN_IND不可连接广播———纯 Beacon：不接受连接和扫描请求
ADV_SCAN_IND可扫描广播—✓—可被 SCAN_REQ 获取更多数据，但不可连接

8. SM — 安全管理

配对 (Pairing) · 绑定 (Bonding) · 加密 (Encryption) · 签名 (Data Signing)

LE Legacy Pairing (BT 4.0/4.1)

STK 生成 · 短密钥 · 易被窃听

TK (Temporary Key): 6 位数字或固定 000000 (Just Works)
STK (Short Term Key): 128-bit AES-CMAC(TK, Srand, Mrand)
⚠️ 已知不安全：被动窃听可暴力破解 6 位 PIN，不推荐。

LE Secure Connections (BT 4.2+)

ECDH P-256 · FIPS 合规

使用 ECDH P-256 密钥交换生成 LTK (Long Term Key)。
4 种关联模型：Just Works (无 MITM 保护)、Numeric Comparison (显示+确认 6 位数)、Passkey Entry (输入 6 位 PIN)、OOB (NFC 等带外信道)。

加密层次

多层密钥派生
Identity Resolving Key (IRK) ↑ 派生自 Long Term Key (LTK) ↑ ECDH 或 STK Temporary Key (TK)

隐私与地址

Resolvable Private Address

Public Address: IEEE 分配，固定
Random Static: 上电时随机生成，不变
Resolvable Private: IRK + 随机数 → 哈希，15 分钟自动轮换，防止追踪。只有持有 IRK 的受信设备才能解析。

⚠️ 配对方法选择建议
Just Works: 无 UI 设备（传感器），无 MITM 保护，仅适用于低风险场景。
Passkey Entry: 一方显示 6 位 PIN，另一方输入。有 MITM 保护。
Numeric Comparison (推荐): 双方显示同一 6 位数，用户确认一致。最适合有显示屏的 IoT 设备。
OOB: 通过 NFC / QR Code 交换密钥，安全性最高。

9.2 BT 5.0 Extended Advertising

传统广播限制

BT 4.x · 31 字节

广播载荷仅 31 字节，减去 Flags (3B)、TX Power (3B) 等必要字段后，实际用于应用数据的空间极少。

扩展广播

BT 5.0+ · 最多 1650 字节

Periodic Advertising

确定性间隔 · 无连接数据流

广播方以固定间隔发送数据到多个 Scanner（1:N 无连接模式）。Scanner 可精确同步时间。
用于 AOA/AOD 定位、传感器数据广播、Auracast 音频广播。

PAwR (BT 5.4)

Periodic Advertising with Responses

在 Periodic Advertising 基础上增加响应时隙，Scanner 可在指定子事件中回复。
为 ESL (Electronic Shelf Label) 电子价签等双向无连接场景设计。

11. BLE Mesh — 多对多网络

BT 5.0+ · Flooding 管理洪泛 · 最多 32767 节点 · 适用于智能照明/楼宇自动化

架构分层

建立在 BLE 广播之上的 Mesh 协议

BLE Mesh 运行在 BLE 广播信道之上 — 不需要连接！利用 ADV_NONCONN_IND 和 ADV_SCAN_IND。
Bearer Layer: Advertising Bearer (广播承载) / GATT Bearer (配置承载)
Network Layer: 加密、认证、中继、TTL
Lower Transport: 分段重组
Upper Transport: 应用密钥加密
Access Layer: 应用数据格式
Foundation Models: Configuration / Health
Models: Generic / Sensor / Lighting 等

Managed Flooding

节点角色 · 消息转发

Relay: 转发消息（默认启用，电池节点可关闭）
Friend: 为低功耗节点缓存消息
Low Power Node: 大部分时间休眠，定期向 Friend 查询
Proxy: GATT 网关，允许手机等非 Mesh 设备接入
每消息带 TTL (默认 127) + 序列号 (SEQ)，收过即丢弃，防止无限洪泛。

安全模型

双层密钥体系

Network Key (NetKey): 网络层加密 — 所有节点共享同一 NetKey
Application Key (AppKey): 应用层加密 — 绑定到特定应用场景
Device Key (DevKey): 每节点独有，用于配置 (Provisioning)
三层 AES-CCM 加密：Network + Transport + Access。

典型应用

照明 · 传感 · 楼宇

智能照明: 群组控制、场景、调光调色
传感器网络: 温度、湿度、人体感应
楼宇自动化: 窗帘、空调、门锁
资产追踪: 标签定位（配合 AoA/AoD）

12. LE Audio — 下一代蓝牙音频

BT 5.2+ · LC3 Codec · 更低功耗 · 更高音质 · Auracast 广播音频

LC3 Codec

Low Complexity Communication Codec

取代 Classic Audio 的 SBC。在 一半码率 下提供更好的音质。
典型参数: 10ms 帧长，采样率 8/16/24/32/44.1/48 kHz，比特率 16~320 kbps。
延迟和功耗均显著优于 SBC。

Isochronous Channels

等时传输 — 保证时序

CIS: 单播等时流 (一对耳机左右声道)
BIS: 广播等时流 (一对多音频广播)
利用 Link Layer 的 Isochronous 状态，提供可靠的定时音频传输，丢包容忍而不重传。

Auracast

广播音频 · 一对无限设备

一个 Auracast 发射器可同时向无限多个接收器广播音频。应用场景：
• 公共场所电视静音广播
• 会议同声传译
• 多人共享音乐
• 助听器系统

🎯 关键设计哲学
BLE 之所以能实现极端低功耗，核心在于三个设计原则：
1. 短时通信 — 每次无线电开启仅 ~200µs 发送一个包，其余时间深度休眠
2. 非对称设计 — Central/Master 承担复杂度（时序管理、跳频决策），Peripheral/Slave 只需"按时醒来听一下"
3. 无连接优先 — 广播 (ADV) 不需要建立连接即可传递数据，极大降低协议开销
