1. iOS BLE 的诡异行为 — Apple 的特殊规则

iOS 的 CoreBluetooth 框架在标准 BLE 协议之上叠加了大量Apple 私有策略——这些策略不写在任何规范中，只能通过抓包和逆向发现。没有适配好这些规则是 BLE 设备"iOS 连不上/连上就断"的最常见原因。

问题 1: 连接参数被 iOS 强制覆盖

外设发 Connection Parameter Update → iOS 说 No

现象：连接成功后外设立即请求 Connection Parameter Update（希望 Interval=20ms 提高吞吐）。iOS 直接拒绝（返回 LL_REJECT_EXT_IND 或 LL_CONNECTION_PARAM_RSP with Reject）。然后 iOS 主动发起自己想要的参数更新：Interval≥15ms（iOS 11+ 放宽到 ≥15ms，曾是最低 30ms）、Latency≤2、Supervision Timeout≤6s。
Apple 的逻辑：不是任何参数都能通过——必须满足：Interval × (Latency+1) ≤ 2s，Timeout ≥ (1+Latency) × Interval × 4。不满足 → 拒绝。
解决：外设不要主动请求参数更新——等 iOS 来请求。iOS 在连接建立后 1-3 秒内会主动发 LL_CONNECTION_PARAM_REQ——外设接受即可。如果想加快，在 CONNECT_IND 中设置符合 iOS 规则的参数。

问题 2: 后台扫描不可见

App 进入后台 → 特定 Service UUID 才被扫描到

现象：App 在前台能扫描到设备，切到后台后（或锁屏后）设备"消失"了。
原因：iOS 后台扫描只匹配 Service UUID——不再上报所有广播设备。App 必须在 CBCentralManager.scanForPeripherals(withServices: [CBUUID(string: "FFF0")]) 中明确指定 Service UUID。如果外设的广播数据中没有包含该 UUID → 后台扫描不到。
解决：① 外设的 ADV_IND 中必须在 Service UUIDs AD Structure 中包含 App 要扫描的 UUID ② App 的 Info.plist 添加 bluetooth-central 后台模式 ③ iOS 后台扫描频率极低（几分钟一次）——使用 iBeacon 或 BLE 作为唤醒源触发连接。

问题 3: MTU 协商的隐式限制

iOS 只协商一次 MTU → 失败后不再尝试

现象：外设发起 Exchange MTU Request → iOS 回复 Exchange MTU Response (MTU=185)——外设以为 MTU 协商成功（185 > 默认 23）。但实际上 iOS 的最终 MTU 以 ATT_MTU 字段为准——如果 iOS 回复的值小于外设请求的值，iOS 端可能不更新 MTU。
Apple 的隐藏规则：iOS 10+ 默认 MTU=185（不是 247！——需要 BLE 4.2+ 且 DLE enabled 才支持到 247）。如果外设请求的 MTU 超过 iOS 支持的最大值 → iOS 回复自己的最大值 → 外设必须接受这个值。
解决：不要假设 MTU 能协商到 247——编码时动态读取实际协商结果（peripheral.maximumWriteValueLength，iOS 11+）。如果长度不满足需求 → 应用层分片。

⚠ iOS BLE 的底线规则总结
1. 连接参数：Interval 必须 ≥15ms，Latency ≤ 2。由 iOS 主导参数更新，外设被动接受。
2. 广播间隔：后台发现依赖广播包中包含 App 正在扫描的 Service UUID。建议保持在 100ms~1s 之间。
3. MTU：iOS 默认 185（非 247）。不要在假设 MTU=247 的前提下设计单包大小。
4. GATT 操作：iOS 严格串行化所有 GATT 请求——无需 App 自己排队（Android 需要）。但同一 Characteristic 上不要同时进行 Read 和 Write——iOS 会返回 Busy。
5. Bonding 存储：iOS 的 Bonding 信息存储在 Secure Enclave 中——跨 App 共享。如果 A App Bonded 了设备，B App 也自动认识。用户"忘记此设备"会清除所有 App 的 Bonding。

2. Android 碎片化 — 每个厂商的 BLE 栈都不一样

Android 的 BLE 栈碎片化是业界公认的噩梦——三星、小米、OPPO、Pixel 的 BLE 行为各不相同，同一个代码在不同手机上表现完全不同。

问题 1: GATT 操作并发行为不一致

三星 vs Pixel vs 小米 — 谁在排队谁在拒绝？

现象：App 连续调用 3 个 GATT Write Request——在 Pixel 上正常执行（系统内部排队串行化）；在三星上第三个返回 status=GATT_BUSY；在小米上第二个开始静默失败不回调。
原因：Android BLE Stack 的实现没有统一标准：Pixel (AOSP 原生栈)——内部有 GATT 操作队列，自动串行化；三星 (Samsung BLE Stack)——没有队列，前一个未完成时再发新请求 → 直接返回 Busy；小米/OPPO——有队列但深度有限（通常 3-5 个，超出则静默丢弃）。
解决：App 层自己实现 GATT 操作队列——不管底层行为如何，确保同一时刻只有一个进行中的 GATT 操作。前一个 callback 返回后才发下一个。这是唯一跨设备可靠的做法。

问题 2: MTU 协商的安卓坑

requestMtu(512) → 返回 mtu=23 — 协商失败但不知道

现象：App 调用 gatt.requestMtu(512) → callback onMtuChanged(gatt, mtu=23, status=SUCCESS)——MTU 协商返回"成功"但实际 mtu 还是 23。
原因：Android 的 onMtuChanged 回调的 status=SUCCESS 不代表协商到了请求的 MTU——只代表"协商过程完成了"。真正的协商结果在 mtu 参数中。许多开发者误以为 status=SUCCESS = MTU 已更新。
更深层的问题：Android 10+ 默认 MTU=517（不是 23！——Android 做了硬件检测，如果芯片支持 DLE 则预设高 MTU）。但某些芯片（如便宜的 BLE 4.0 芯片）在 Android 10+ 上会出现 MTU 协商失败（芯片不支持 DLE → Android 降级逻辑有 bug → 返回 23）。
解决：① 使用 onMtuChanged 返回的实际 mtu 值，不要假设 ② 发送数据前用 gatt.getMtu() 动态查询 ③ 长包发送前验证 data.length <= (mtu - 3)。

问题 3: 扫描的 5.0 蓝牙地址 Bug

Android 5.0~6.0: getAddress() 返回随机地址而非真实 MAC

现象：同一个设备在 Android 5.0 上扫描到的 MAC 地址每次都不一样——App 无法通过 MAC 识别已配对设备。
原因：Android 5.0 (Lollipop) 引入 BLE Privacy 功能——系统自动将外设的真实 MAC (Public Address) 映射为一个随机 RPA (Resolvable Private Address) 再传给 App。但 Android 5.0~6.0 的实现有 bug——映射表没有持久化 → 每次扫描看到的 RPA 不同 → App 拿到的是每次都在变的假地址。
解决：① 无法通过 App 代码修复（系统 bug）② 绕行方案：不要依赖 MAC 地址识别设备——改用广播数据中的自定义字段（如 Manufacturer Specific Data 中嵌入设备序列号）③ Android 7.0+ 修复——但仍建议不用 MAC 地址做设备标识。

问题 4: 连接失败的 133 错误码

GATT_ERROR (0x85=133) — 最无用的错误码, 需要深度排查

现象：onConnectionStateChange(gatt, status=133, newState=DISCONNECTED)——连接刚建立几百毫秒就断连，错误码 133。
133 = GATT_ERROR——这是 Android 的万能错误码，包罗万象：外设 Supervision Timeout、ATT 层协议错误、链路层加密失败、MTU 协商异常——全部返回 133。
排查路径：① 检查外设日志：连接建立后外设是否收到 LL_TERMINATE_IND（断连原因码）——这是唯一可靠的方向 ② 关闭加密（如果开启了）——验证是否是配对失败导致的 ③ 检查外设的 Supervision Timeout 设置——是否 > 2s（Android 默认需要这个时长）④ 最后手段：空中抓包（见 Ch6）——查看断连前的最后几个包。

3. 广告与连接兼容性问题

设备不能被手机发现，或者发现后连不上——这是 BLE 产品最常见的用户投诉。问题通常出在广告参数或时序的微妙不兼容。

广告间隔与发现的概率性

不是广告发了就一定能被扫到——这是个概率问题

数学事实：外设发 ADV_IND 的时刻和手机打开扫描窗口的时刻需要重叠在同一信道上。Adv Interval=1s + Scan Window=50ms + Scan Interval=200ms → 手机每 200ms 开启 50ms 的接收窗口 → 在这 50ms 内，3 个广播信道中至少有一个正好收到 ADV_IND → 概率 = 50ms/1s = 5% → 期望 20 个广播周期才遇上（20s）。
Apple 的规则加剧了问题：iOS 后台扫描对广告间隔有隐藏的 30ms 因子偏好——Adv Interval 是 30ms 的倍数时被发现的概率显著提高（约 3×）。这是未公开行为。
解决：配网模式用 20-30ms 快速广告；日常运行用 100ms~300ms。使用 Extended Advertising (BLE 5.0) 可以同时在多个信道上发——大幅提高效率。

Scan Response 超时

手机发 SCAN_REQ → 外设没在 150μs 内回复 → 手机丢弃

现象：手机能收到 ADV_IND（广播包），但 App 拿到的 scanRecord 中缺少设备名称（名称放在 SCAN_RSP 中但没收到）。
原因：外设 BLE 芯片在收到 SCAN_REQ 后150μs 内必须开始发送 SCAN_RSP（T_IFS）。如果外设的固件响应太慢（如用低优先级任务处理 SCAN_REQ）→ 超过 150μs → 手机已进入下一个扫描窗口 → SCAN_RSP 被丢弃。
解决：① 关键信息（Service UUID）放在 ADV_IND 而非 SCAN_RSP（SCAN_RSP 不可靠）② 外设 BLE 芯片的 SCAN_REQ 处理必须在硬件中断中完成——不能用任务调度。

地址类型不匹配

Public vs Random Static vs RPA — 用错地址类型 = 连不上

现象：手机能扫描到设备（ADV_IND 能看到），但 connectGatt() 返回失败。
原因 1：外设广播使用 Random Static Address（TxAdd=1），但手机 connectGatt() 的 addressType 参数传的是 PUBLIC——蓝牙栈在 CONNECT_IND 中使用错误的地址类型位 → 外设不认识这个包。
原因 2：外设广播使用 RPA (Resolvable Private Address)（TxAdd=1, 定时更换），但手机没有 Bonding（无 IRK）→ 无法解析 RPA → 手机根本不知道这是之前连接过的设备。
解决：① 广播使用 Random Static Address（每次上电相同，手机可识别）② App 从 scanResult 中读取 addressType 并原样传给 connectGatt()——不要硬编码 PUBLIC。

4. GATT 层的坑 — Service Discovery 与数据交互

GATT Discovery 缓慢、Handle 变化、CCCD 使能失败——这些问题的本质是对 GATT 服务端实现细节的假设过于乐观。

问题: Service Changed 导致 Handle 失效

App 缓存的 Handle 突然全部错位 → 读写全失败

现象：App 首次连接 Discovery 成功，缓存了 Handle。重连后直接使用缓存 Handle 读写 → 返回 Attribute Not Found (0x0A) → App 以为设备故障。
原因：外设的 GATT 数据库在两次连接之间发生了变化——固件升级后新增了 Service/Characteristic → Handle 重新分配 → App 缓存的旧 Handle 全部失效。外设应该发送 Service Changed Indication（Handle 0x0001 的值从 0x0000→0xFFFF 发生变化）通知手机——但如果外设没发送，App 完全不知道 Handle 变了。
解决：① App 端：重连时先订阅 Service Changed Indication（CCCD of Handle 0x0001），再使用缓存 Handle。如果收到 Indication → 清除缓存 → 重新 Discovery。② 外设端：GATT 数据库变更后必须发送 Service Changed Indication（在下一个连接中）。

问题: CCCD 使能后 Notify 不生效

Write CCCD=0x0001 返回 SUCCESS — 但 Notify 永远不来

现象：App 写 CCCD (Handle 0x000D, Value=0x0001 启用 Notify) → Write Response status=SUCCESS。App 等待 onCharacteristicChanged → 永远不来。
常见原因：① 外设固件中只保存了 CCCD 值但未真正使能 Notify——GATT 层写入了 CCCD，但应用层代码没有读取这个标志来启动数据上报 ② CCCD 的权限设置错误——外设在 GATT 数据库中声明 CCCD 有 Write Permission，但实际代码中未处理 Write 事件 ③ Android 手机未调用 setCharacteristicNotification → 即使 CCCD 写入成功，Android 栈在收到 Handle Value Notification 时会检查本地是否订阅了该 Char——未订阅 → 静默丢弃不通知 App。
解决：① App 必须调用 setCharacteristicNotification(char, true)（Android）——这个调用不是可选的 ② 外设固件验证 CCCD 写入后确实触发了数据上报逻辑——用 LED 闪烁或日志确认。

5. 配对与 Bonding — 最频繁的"怎么连不上"来源

配对是 BLE 所有功能中兼容性问题最集中的环节——不同芯片、不同手机 OS 版本、不同配对方法的组合爆炸式增长。

问题 1: 配对方法不匹配

外设只支持 Just Works, 手机要求 Passkey → 配对失败

现象：手机弹窗"输入 PIN 码"——但外设没有键盘也没有显示器（Just Works 设备）。手机收到外设的 IO Capabilities=NoInputNoOutput → 知道对方不能输 PIN → 理应自动降级为 Just Works。但某些 Android 版本不这样做——坚持弹出 PIN 输入框 → 用户随便输 → 配对失败。
解决：外设通过 AuthReq 字段显式声明不需要 MITM 保护（AuthReq MITM bit=0）。部分 BLE 芯片 SDK 默认 AuthReq MITM=1——必须手动改为 0。否则手机可能强制要求 Numeric Comparison 或 Passkey。

问题 2: Bonding 丢失 — 重连要求重新配对

昨天还正常, 今天突然要求重新配对 — 无限循环

现象：昨天配对了设备，今天打开 App 连上 → 手机弹出"要与此设备配对吗？"→ 配对 → 明天再来 → 又弹。
常见原因：① 外设的 Bonding 存储在 RAM 中（掉电丢失）→ 每次上电 Bonding 信息消失 → 手机发 LL_ENC_REQ → 外设找不到 LTK → 加密失败 → 手机判定"需要重新配对"。必须存储在 Flash 中。② 外设使用 RPA 地址（随机地址每次上电更换）→ 手机通过 IRK 可以解析 → 但外设上电后如果忘记了自己上次用的 RPA 对应的 IRK → 手机加密挑战失败。
解决：确保 Bonding 信息（LTK/IRK/CSRK + 对端 BD_ADDR 和地址类型）持久化存储在 Flash中。上电后从 Flash 恢复。

问题 3: LE Legacy vs LE Secure

手机要求 Secure Connections, 外设芯片只支持 Legacy → 配对失败

现象：Android 8.0+ 手机在配对请求中设置了 Secure Connections 标志位——期望使用 ECDH 密钥交换。但外设芯片是 BLE 4.0/4.1 的廉价芯片——只支持 LE Legacy Pairing（AES-CMAC-128, 无 ECDH）。手机收到外设的 Pairing Response 后发现 Secure Connections 位为 0 → 直接发送 Pairing Failed (Reason=0x05 — Pairing Not Supported)。
解决：① 外设升级芯片——使用支持 LE Secure Connections 的芯片（BLE 4.2+）② 如果无法升级：外设在 Pairing Response 中明确声明不支持 Secure Connections——手机会根据策略降级（Android 通常会降级到 LE Legacy；iOS 在某些 iOS 版本中拒绝降级——iOS 13+ 默认要求 Secure Connections）。

6. 空中抓包 — 终极诊断手段

当代码排查无法定位问题时——空中抓包是唯一能告诉你"空口上到底发生了什么"的方法。没有抓包，上面的所有问题都无法真正确认。

方案 1: nRF Sniffer (推荐)

nRF52840 Dongle + Wireshark — ~100 元, 最实用

Nordic nRF52840 USB Dongle（~100 元）+ nRF Sniffer for Bluetooth LE 固件 + Wireshark。支持 BLE 4.x/5.x 全部模式。可以同时抓三个广播信道（三路并行监听）。
使用方法：① Dongle 烧录 Sniffer 固件 ② Wireshark 安装 nRF Sniffer 插件 ③ 在 Wireshark 中选择目标设备的 Access Address → 自动跟踪跳频 → 抓取连接后的所有空中包。
能看到什么：ADV_IND 内容、CONNECT_IND 参数、LL_TERMINATE_IND (断连原因码)、ATT 层的每一条 GATT 操作、加密协商过程、连接参数更新的协商过程。

方案 2: Ellisys / Frontline (专业)

硬件抓包器 — 数万元, 芯片原厂和认证实验室用

Ellisys Bluetooth Explorer / Frontline BPA 600 是专业级的蓝牙协议分析仪——可以同时抓 BLE + BR/EDR + WiFi + Zigbee。自带协议分析软件——自动解析每层协议、标记异常、生成时序图。
优势：① 宽频带——同时监听所有 40 个 BLE 信道（不需要跟踪跳频）② 精确时戳——可以分析 μs 级的时序问题 ③ 自动错误标记（CRC 错误、T_IFS 超时、协议违反）。
成本：4-8 万元——个人开发者用 nRF Sniffer 足够。

方案 3: Android HCI Log

手机自带 — 免费, 但只有 HCI 层

Android 开发者选项中启用"Enable Bluetooth HCI snoop log" → 系统记录所有 HCI 层通信到 /sdcard/btsnoop_hci.log。用 Wireshark 打开。
能看到什么：所有 HCI Command/Event + ACL/SCO Data。包括 LE Set Scan Enable, LE Create Connection, LE Connection Complete (含 status 错误码), Att 层交互。
看不到什么：HCI 层以下——链路层的 LL Control PDU (LL_CONNECTION_PARAM_REQ, LL_TERMINATE_IND 的原因码, LL 层的重传)。广告包也看不到（HCI 只上报 LE Advertising Report 事件——已经过滤和聚合过的）。
价值：快速检查 GATT 层的交互和连接状态变化——是最便捷的初步诊断工具。
