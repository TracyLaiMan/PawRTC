GCC 架构：双控制器协同

GCC 的核心思想：延迟是拥塞的早期信号（缓冲区开始堆积但还未丢包），丢包是拥塞的确认信号（缓冲区已满）。两者独立运行，最终取最保守的码率。

延迟控制器 (Delay-Based)

Kalman 滤波 → 过载检测 → AIMD 调整 · 约 200ms 响应

输入：接收端测量的包间延迟梯度（每 5ms 一组包的到达时间差 vs 发送时间差）。
核心算法：Kalman 滤波器估计网络排队延迟的变化趋势——过滤掉传输延迟的随机噪声（抖动），提取出真正的拥塞信号（队列增长）。
输出：三个状态——Normal（队列在减小→可以增加码率）、Overuse（队列在增长→必须降低码率）、Underuse（队列稳定→保持码率）。用 AIMD (Additive Increase / Multiplicative Decrease) 调整目标码率。
意义：在丢包发生之前就检测到拥塞——比单纯基于丢包的算法早 500ms~2s 响应。

丢包控制器 (Loss-Based)

RTCP RR 丢包率 → 查表调整 · 约 1-3s 响应

输入：RTCP Receiver Report 中的丢包率（fraction lost / cumulative lost）。
核心算法：根据丢包率查表决定码率调整——丢包率 < 2% → 增加码率；2%~10% → 保持；> 10% → 大幅降低。
输出：丢包率触发的最大码率上限。丢包控制器不能增加码率超过延迟控制器给出的值——它只设上限。
意义：延迟控制器可能在非拥塞场景（如 WiFi 信道切换导致的延迟尖峰）误判——丢包控制器作为保守的后备，确保即使延迟信号有噪声，也不会过度压制码率。

A

最终目标码率 = min(延迟控制器输出, 丢包控制器上限)

两个控制器独立运行，各算各的——取 min 作为最终目标码率。编码器再根据目标码率调整分辨率/帧率/QP。

为什么取 min？延迟控制器激进（AIMD 加性增——每帧微增码率探索可用带宽），丢包控制器保守（丢包 > 10%→乘性降 0.85×）。正常情况下延迟控制器主导；当网络真正拥塞到丢包时→丢包控制器接管→强制大幅降码率→延迟控制器在此基础上重新开始 AIMD 增长。
动态平衡：延迟控制器倾向于轻微过载（不断探索带宽上限—偶尔触发 Overuse→微降→再增加→再触发...反复循环）。丢包控制器作为"安全阀"——当延迟控制器的探索触发了真正的丢包（> 2%）→丢包控制器设上限→延迟控制器在上限之内重新 AIMD。

延迟控制器 — Kalman 滤波 + 过载检测 + AIMD

这是 GCC 最核心也最复杂的部分——从一个看似简单的"包到达时间差"中滤除随机噪声、提取真正的拥塞信号。

Step 1: 包组 (Packet Group)

每 5ms 一组——组内包几乎同时发送，组间才有信息量

发送端以突发方式发送 RTP 包（一个视频帧可能封装为多个 RTP 包，在~1ms 内连续发出）。GCC 将连续发送且间隔 ≤ 5ms的包归为一个 Burst Group。组内的包间延迟差没有信息量——它们到达时间的差异几乎全是排队延迟抖动（而非真正的网络变化）。
组间延迟才有意义：第 N 组的最后一个包与第 N-1 组的最后一个包之间的到达时间差 vs 发送时间差——这个差值反映了在这两组之间网络中队列的变化。

Step 2: Kalman 滤波器

从噪声中估计真实的排队延迟变化趋势

状态方程：θ[n] = θ[n-1] + u[n]（θ = 单向排队延迟梯度, u = 过程噪声——网络本身的变化）。
观测方程：d[n] = θ[n] + C·m[n] + v[n]（d = 测量到的包间延迟差, C·m = 传输速率变化导致的基线偏移, v = 测量噪声——抖动）。
Kalman 滤波器根据 过程噪声方差（网络变化多快）和 测量噪声方差（抖动多大）动态权衡：测量噪声大时（抖动大）→更信任预测（θ 平滑——不会因偶然抖动误判Overuse）；过程噪声大时（网络本身波动大）→更信任测量（θ 响应快——快速跟上带宽变化）。

Step 3: 过载检测 + AIMD

θ 与阈值的比较 → 三个状态

Kalman 滤波器输出当前估计的 θ (排队延迟梯度)，单位：ms/s（每秒钟排队时间变化的毫秒数）。将其与动态阈值 γ 比较：
θ > γ（梯度上限，如 10ms/s）→ Overuse：队列在快速堆积→乘性降码率（A = A × 0.85）。
θ < -γ（梯度下限）→ Underuse：队列在快速排空→加性增码率（A = A + 恒定增量——每帧约 5-30kbps）。
-γ ≤ θ ≤ γ → Normal：队列稳定→保持码率。
自适应阈值 γ：静止时 γ 大（容忍更多噪声——避免误判）；检测到 Overuse 后 γ 变小（敏感——快速响应验证是否真的拥堵）；确认拥堵后 γ 恢复——避免持续压制。

// GCC 延迟控制器的核心逻辑（伪代码） // 每收到一个 RTCP Transport-CC 反馈包时执行 void DelayBasedController::OnFeedback(feedback) { for (each packet_group in feedback) { // 1. 计算包间延迟梯度 double send_delta = group.send_time - prev_group.send_time; double arrive_delta = group.arrive_time - prev_group.arrive_time; double delay_gradient = arrive_delta - send_delta; // 2. Kalman 滤波: 估计真实的排队延迟变化趋势 θ kalman_filter_.Predict(); kalman_filter_.Update(delay_gradient); double theta = kalman_filter_.estimated_queue_delay_trend(); // 3. 过载检测 — 与自适应阈值比较 if (theta > overuse_threshold_) { state_ = OVERUSE; OveruseDetected(); // → 乘性降码率: A = A * 0.85 } else if (theta < -overuse_threshold_) { state_ = UNDERUSE; UnderuseDetected(); // → 保持, 准备在 Normal 后加性增 } else { state_ = NORMAL; NormalDetected(); // → 加性增码率: A = A + kBitrateIncrement } } }

丢包控制器 — 保守的安全阀

丢包控制器逻辑比延迟控制器简单得多——本质上是一个查表状态机。它的价值在于可靠——丢包率是确定性的拥塞信号，不会像延迟那样被抖动误导。

丢包率 → 码率乘数

RTCP RR fraction_lost → 查表 → 乘性调整

GCC 根据最近一个 RTCP 报告周期（通常 100ms~1s）内的丢包比例决定调整乘数：
丢包率 < 0.5% → 乘数 = 1.08（增加 8%——探索带宽）
0.5%~2% → 乘数 = 1.05（小幅增加）
2%~10% → 乘数 = 1.0（保持——不增不减，谨慎等待）
> 10% → 乘数 = 0.85（乘性降——大幅降低码率）
注意：丢包控制器的调整对象是最大允许码率的上限，而不是直接设置当前码率。最终生效码率 = min(delay_controller_rate, loss_based_max_rate)。

与延迟控制器的互补

延迟信号模糊时 → 丢包信号直接接管

场景 A: WiFi 信号波动——延迟控制器可能因 AP 的信道竞争产生假阳性 Overuse（延迟尖峰→Kalman 估计 θ 增大→误判→降码率）。但丢包率为 0（WiFi 重传在链路层完成——RTP 层看不到丢包）→ 丢包控制器不设上限→延迟控制器的误降很快被 AIMD 恢复。
场景 B: 真实拥塞——路由器缓冲区满→开始丢包。延迟控制器可能因为缓冲区已满（排队延迟不再增长——θ 可能回落到 Normal 范围——"已经堵死了所以不再恶化"）→ 错误地停止降码率。丢包控制器检测到丢包率 > 10%→强制设上限 = 当前码率 × 0.85——接管控制。延迟控制器在此基础上重新 AIMD。

为什么丢包控制器的阈值是 2% 而不是 0%？
网络中的随机丢包（WiFi 干扰、信号弱）即使在没有拥塞时也可能有 0.1%~1% 的丢包。如果阈值是 0% → 任何微小的随机丢包都触发降码率 → GCC 在 WiFi 环境下永远无法达到合理码率。2% 是一个经验阈值——超过 2% 的丢包几乎一定是拥塞而非随机噪声。在这个阈值以下，延迟控制器继续主导。

反馈机制 — Transport-CC vs REMB

GCC 需要接收端把每个包的到达时间和丢包率反馈给发送端——两种 RTCP 扩展协议承载这些信息。

Transport-CC (推荐)

transport-wide 序列号 + 到达时间戳 · 每包级精度

发送端在每个 RTP 包中嵌入一个全局唯一的 transport-wide 序列号（Transport-CC 扩展头，1-2 字节开销）。接收端记录每个包的到达时间（精确到 250μs/100μs）。接收端周期性发送 RTCP Transport-CC Feedback 包——包含每个 transport-cc 序列号的到达时间 delta（以 base time 为参考的 250μs 增量编码）。
精度：每个包的到达时间独立记录——GCC 可以精确计算任意两个包之间的到达时间差。这是包组 (Burst Group) 划分和延迟梯度计算的数据基础。
WebRTC 默认启用——比 REMB 更精确、更现代。

REMB (旧方案, 仍支持)

接收端估算可用带宽 · 直接告诉发送端

REMB (Receiver Estimated Maximum Bitrate) 让接收端自己估算可用带宽——基于接收到的码率和包到达时间差。接收端运行自己的 GCC 算法（或简化版）→ 计算出一个"建议码率"→ 通过 RTCP REMB 消息发给发送端。
缺点：① 接收端的计算在 JavaScript 中执行——延迟大、精度低 ② 接收端没有发送端的发送时间戳——只能用"接收间隔"估算——无法区分"发送端故意不发"和"网络延迟" ③ 无法做包组级别的精确分析。
现状：Chrome 已默认使用 Transport-CC；REMB 作为兼容旧版 WebRTC 实现的 fallback 保留。

Transport-CC Feedback 包的内容（每个反馈周期 ~100ms 发送一次）
// RTCP Transport-CC Feedback Message 结构 Base Sequence Number: 12345 // 这个反馈包覆盖的起始 transport-cc 序列号 Packet Status Count: 20 // 这个反馈包覆盖 20 个包 Reference Time: 9876543210 // 基准时间 (64-bit NTP 或 24-bit 250μs tick) // 对每个包: Packet Chunk: [0b10=Received, Delta=0x03] // 收到, 到达时间 = RefTime + 3×250μs Packet Chunk: [0b10=Received, Delta=0x05] // 收到, 到达时间 = RefTime + 5×250μs Packet Chunk: [0b11=NotReceived] // 丢失! // 发送端据此计算: 包12345和12346之间的到达间隔 = (5-3)×250μs = 500μs // 对比发送间隔: 如果发送间隔是 400μs → 延迟梯度 = 500-400 = +100μs → 队列增长!

动态行为 — GCC 在不同网络条件下的表现

GCC 不是静态算法——它在不同的网络条件下表现出完全不同的行为特征。理解这些动态是调优 GCC 的关键。

理想网络 — 有线、低延迟、无丢包

GCC 表现: 快速探测可用带宽上限，稳定后微调

① GCC 从低码率启动（如 300Kbps）→ AIMD 加性增（每帧 +5-30Kbps）→ 在 ~2-5 秒内达到带宽上限 ② 接近上限时延迟控制器检测到微弱的 Overuse 信号→乘性降（×0.85）→短暂回落后重新 AIMD ③ 最终在带宽上限附近 ±10% 区间内稳定振荡。
特征：丢包控制器几乎不介入（丢包率=0%）。延迟控制器的 θ 在 NORMAL 和微弱 OVERUSE 之间交替——这是 GCC 在健康运行时的正常行为。

WiFi 环境 — 延迟抖动大、偶发丢包

GCC 表现: Kalman 滤波抵抗抖动、丢包控制器保守

① WiFi 的 MAC 层重传和信道竞争导致间歇性延迟尖峰（延迟突然从 5ms 跳到 50ms，持续 20-50ms 后恢复）→ 延迟控制器可能在尖峰时刻误判 Overuse→降码率 ② Kalman 滤波器在抖动大的环境中自动提高对测量噪声的容忍度（增加测量噪声协方差 R）→ θ 的估计更平滑——减少误判 ③ WiFi 丢包（0.1-1%）< 2% 阈值→丢包控制器不介入。
特征：GCC 码率在 WiFi 下波动更大（±20%），但不会持续压制。Kalman 的自适应方差是关键——如果方差参数固定，GCC 在 WiFi 下会频繁错误降码率。

4G/5G 移动网络 — 带宽快速波动

GCC 表现: 延迟控制器主导、带宽陡降时丢包接管

① 4G 的带宽可能在几秒内从 5Mbps 跌到 500Kbps（切换基站、进入信号盲区）→ 丢包率瞬间飙升到 20%+→丢包控制器立刻介入：最大码率上限 = 当前码率 × 0.85 → 延迟控制器在此基础上重新 AIMD ② 带宽回升时→延迟控制器的 AIMD 加性增 速度太慢（需要几秒才能从 500Kbps 爬回 5Mbps）→ GCC 在恢复阶段码率偏低→ 画质暂时下降 ③ BWE (Bandwidth Estimation) 的 early ramp-up 机制：检测到丢包率连续多个周期 < 0.5%→加速 AIMD（增量从 5Kbps/帧 提升到 20Kbps/帧）。

调优实战 — GCC 参数与 IPC 场景适配

WebRTC 的默认 GCC 参数为视频会议优化——IPC 实时监控场景有完全不同的需求（低延迟 > 画质、稳定 > 峰值吞吐）。

IPC 场景的 GCC 调优方向

降低延迟容限 · 减少 AIMD 惯性 · 加速恢复

1. 减少 AIMD 加性增量（从默认 ~30Kbps/帧 降到 ~5-10Kbps/帧）：GCC 的 AIMD 在正常模式下不断加性增码率→轻微 Overuse→微降→再加→...这个循环在 IPC 场景中被感知为周期性的码率波动（画质时好时坏）。减小增量→波动幅度减小→码率更稳定。
2. 降低 Overuse 检测阈值（γ 从 10ms/s 降到 5-8ms/s）：IPC 端到端延迟目标 < 200ms——需要比视频会议更早检测到拥塞。降低阈值→更快响应→延迟更低但码率可能更保守。
3. 丢包控制器阈值收紧：丢包率 > 5%（vs 默认 10%）→触发降码率。IPC 场景丢包 5% 已经影响 AI 检测准确率——需要更早介入。
4. 禁用或限制 probing：GCC 在空闲时会主动探测更高码率（probe）→ 短暂发送高于当前估计的码率→ 在 IPC 固定码率场景中毫无意义——只会引起短暂的延迟尖峰。

通过 Field Trial 参数调整 GCC

WebRTC 的 GCC 参数通过命令行或代码注入

Chrome/WebRTC 的 GCC 参数不直接暴露为 API——通过 --force-fieldtrials 或 C++ 代码中的 webrtc::field_trial 配置：
WebRTC-Bwe-ProbingConfiguration/probe_bitrate:1000000,min_probe_bitrate:500000/——限制探测码率范围。
WebRTC-Bwe-EstimatedSendRateUpperBound/upper_bound:2000000/——硬限制最大发送码率（2Mbps）。
WebRTC-Bwe-LossBasedBweV2/enabled:true,min_loss_rate:0.05/——启用 V2 丢包控制器、降低丢包触发阈值到 5%。
LiveKit SDK (LinkricAI 使用)：通过 RtpEncodingParameters 的 maxBitrate 和 scaleResolutionDownBy 间接控制 GCC 的行为空间——把上限设得接近期望码率→GCC 的探测范围缩小→更稳定。

⚠ GCC 调优的黄金法则 — 不要在延迟控制器和丢包控制器之间"打架"
最常见的错误：为了"提高画质"把丢包控制器的阈值从 10% 调高到 20%——希望"允许更多丢包来换取更高码率"。结果：延迟控制器在探测带宽时触发了真正的拥塞→路由器缓冲区满+丢包→但丢包控制器因为阈值太高不介入→延迟控制器以为自己还可以继续探测→继续加性增→延迟爆炸→端到端延迟从 50ms 飙升到 2s——视频会议完全不可用。
原则：丢包控制器的阈值应该与端到端延迟目标对应——延迟目标越严格、丢包阈值越低。IPC 场景的丢包 10% 阈值是合理的（延迟目标 200ms）。当延迟明显恶化时→降低阈值而非升高。
