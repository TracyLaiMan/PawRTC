V4L2 子系统深度架构

V4L2 不仅仅是 ioctl 接口——它是一套完整的内核视频框架。深入理解其核心数据结构、ioctl 分发机制、videobuf2 状态机和设备注册流程，才能真正定位和解决视频采集问题。

👤

用户空间 (User Space)

应用通过 open("/dev/videoX") 获得 fd，然后使用 ioctl(fd, VIDIOC_*, &arg) 控制设备，mmap() 映射内核 DMA 缓冲到用户空间，poll() 等待数据就绪。
常用库/框架：GStreamer (v4l2src/v4l2sink)、FFmpeg libavdevice、OpenCV VideoCapture、本项目 ImpSource。
关键点：用户空间不直接操作硬件，所有请求必须经由 V4L2 Core 中转分发。性能关键路径是 mmap + vb2 零拷贝机制。

⚙️

V4L2 Core (核心框架层)

内核视频子系统中枢，管理所有注册的 video_device、v4l2_subdev、vb2_queue。
ioctl 分发机制：用户空间 ioctl → v4l2_ioctl() 系统调用 → video_ioctl2() 查表分发 → 驱动提供的 v4l2_ioctl_ops 回调。每个 ioctl 有独立的 permission check（如 VIDIOC_S_FMT 需要 带写权限的fd）。
核心结构：struct v4l2_device（设备实例根）、struct video_device（/dev/videoX 对应节点）、struct v4l2_subdev（子设备抽象）、struct v4l2_fh（文件句柄，跟踪优先级和事件）。

🔌

子设备层 (Sub-device)

每个独立的硬件模块注册为 v4l2_subdev，拥有独立的 pad 集合和格式控制。
Pad 协商：Sink Pad 接收上游格式，Source Pad 输出给下游。每个 pad 有独立的 v4l2_mbus_framefmt（媒体总线格式，如 SBGGR10、UYVY8）。
路由 (Routing)：多输入设备（如 TV 解码器）通过 VIDIOC_SUBDEV_S_ROUTING 切换输入源。
与 V4L2 Device 的关系：video_device 是面向应用的"门面"，它内部组合一个或多个 subdev 构成完整管线。应用直接操作 /dev/videoX 即可，只有高级场景才需手动操作 /dev/v4l-subdevX。

🔧

硬件驱动层 (HW Driver)

平台特定驱动，实现 V4L2 Core 定义的回调接口。
Sensor 驱动：I2C 驱动(sc231hai)，控制曝光/增益/白平衡，输出 Bayer RAW 到 MIPI CSI。
ISP 驱动：Ingenic T31 ISP，实现 demosaic、降噪、色彩校正、gamma。注册多个 subdev 对应不同处理阶段。
VIDMA/CODEC 驱动：DMA 引擎驱动管理缓冲区搬运，M2M 编码器驱动管理编码会话。
关键回调：v4l2_ioctl_ops（ioctl 处理）、vb2_ops（缓冲区管理）、vb2_mem_ops（内存分配/映射）。

内核核心数据结构
struct video_device
// V4L2 设备的核心注册对象
struct video_device {
  struct device dev;          // 嵌入 device，用于 sysfs
  struct v4l2_device *v4l2_dev;
  struct v4l2_ioctl_ops *ioctl_ops;
  enum v4l2_buf_type vfl_type; // CAPTURE/OUTPUT/M2M
  int minor;                   // /dev/videoX 的次设备号
  struct vb2_queue *queue;    // 关联的缓冲区队列
  struct v4l2_fh *fh;          // 当前打开的文件句柄
  // 注册后自动创建 /dev/videoX 设备节点
};
struct vb2_queue
// videobuf2 缓冲区队列管理器
struct vb2_queue {
  enum v4l2_buf_type type;
  enum v4l2_memory memory;   // MMAP/USERPTR/DMABUF
  struct vb2_ops *ops;       // 驱动回调
  struct vb2_mem_ops *mem_ops;
  u32 min_buffers_needed;
  struct list_head queued_list;  // 入队等待队列
  struct list_head done_list;    // 已完成队列
  unsigned int streaming:1;     // 是否在 streaming
};

V4L2 API 全景

V4L2 以 ioctl 为核心接口，上百个命令码覆盖设备查询、格式协商、缓冲管理、参数控制。按功能分为六大类。

能力查询

VIDIOC_QUERYCAP — 探测设备身份

查询设备能力位掩码：V4L2_CAP_VIDEO_CAPTURE(视频采集)、V4L2_CAP_STREAMING(流式IO)、V4L2_CAP_READWRITE(读写IO)。

返回 v4l2_capability 结构：driver 名、card 名、bus_info、version、capabilities。

struct v4l2_capability cap; ioctl(fd, VIDIOC_QUERYCAP, &cap); // 检查是否支持视频采集 + 流式IO if (!(cap.capabilities & V4L2_CAP_VIDEO_CAPTURE)) return -1; if (!(cap.capabilities & V4L2_CAP_STREAMING)) return -1; // 需要 mmap/DMABUF

格式协商

ENUM_FMT / G_FMT / S_FMT / TRY_FMT

VIDIOC_ENUM_FMT 枚举设备支持的像素格式(fourcc)。

VIDIOC_S_FMT 设置采集格式（宽高、pixelformat、bytesperline、sizeimage）。TRY_FMT 不实际修改，仅探测是否可行。

关键结构：v4l2_format → v4l2_pix_format（单平面）或 v4l2_pix_format_mplane（多平面如 NV12M）。

struct v4l2_format fmt; fmt.type = V4L2_BUF_TYPE_VIDEO_CAPTURE; fmt.fmt.pix.width = 1920; fmt.fmt.pix.height = 1080; fmt.fmt.pix.pixelformat = V4L2_PIX_FMT_NV12; fmt.fmt.pix.field = V4L2_FIELD_NONE; ioctl(fd, VIDIOC_S_FMT, &fmt); // 驱动可能调整参数，读回确认 ioctl(fd, VIDIOC_G_FMT, &fmt);

帧率控制

VIDIOC_G_PARM / S_PARM — timeperframe

通过 v4l2_streamparm 设置帧间隔：timeperframe.numerator / timeperframe.denominator。

例如 15fps → numerator=1, denominator=15。Sensor 驱动据此配置曝光时序。

注意：并非所有 sensor 都支持任意帧率，实际值可能被驱动 round 到最近合法值。

struct v4l2_streamparm parm; parm.type = V4L2_BUF_TYPE_VIDEO_CAPTURE; parm.parm.capture.timeperframe.numerator = 1; parm.parm.capture.timeperframe.denominator = 15; ioctl(fd, VIDIOC_S_PARM, &parm);

控制 API

QUERYCTRL / G_CTRL / S_CTRL / QUERY_EXT_CTRL

查询可用控制项：VIDIOC_QUERYCTRL 或扩展版 VIDIOC_QUERY_EXT_CTRL。

读写控制值：VIDIOC_G_CTRL / VIDIOC_S_CTRL，传入 v4l2_control。

典型控制：V4L2_CID_BRIGHTNESS、V4L2_CID_CONTRAST、V4L2_CID_HFLIP、V4L2_CID_AUTO_WHITE_BALANCE。

ISP 3A(自动曝光/白平衡/对焦)也通过控制接口暴露。

struct v4l2_control ctrl; ctrl.id = V4L2_CID_BRIGHTNESS; ctrl.value = 128; ioctl(fd, VIDIOC_S_CTRL, &ctrl); // 读取自动白平衡状态 ctrl.id = V4L2_CID_AUTO_WHITE_BALANCE; ioctl(fd, VIDIOC_G_CTRL, &ctrl); // ctrl.value == 1 表示自动WB开启

缓冲管理

REQBUFS / QBUF / DQBUF / STREAMON/OFF

VIDIOC_REQBUFS：向 vb2 请求 N 个缓冲区，指定内存模式(V4L2_MEMORY_MMAP 等)。

VIDIOC_QBUF：将空闲缓冲入队交给驱动；VIDIOC_DQBUF：取出填充完数据的缓冲。

VIDIOC_STREAMON：启动 DMA 采集；VIDIOC_STREAMOFF：停止并回收所有缓冲。

核心循环：QBUF → 驱动填充 → DQBUF → 用户处理 → QBUF 重入队。

struct v4l2_requestbuffers req; req.count = 4; req.type = V4L2_BUF_TYPE_VIDEO_CAPTURE; req.memory = V4L2_MEMORY_MMAP; ioctl(fd, VIDIOC_REQBUFS, &req); // mmap each buffer, then QBUF all, then STREAMON for (int i = 0; i < req.count; i++) { buf.index = i; ioctl(fd, VIDIOC_QBUF, &buf); } enum v4l2_buf_type type = V4L2_BUF_TYPE_VIDEO_CAPTURE; ioctl(fd, VIDIOC_STREAMON, &type);

标准与输入

ENUMINPUT / G_INPUT / S_INPUT / ENUMSTD

多输入设备（如 TV 卡）通过 VIDIOC_S_INPUT 切换输入源。

模拟视频设备需设置视频标准：VIDIOC_S_STD (NTSC/PAL/SECAM)。

IPC 场景通常只有单输入(Sensor MIPI)，此类 API 用得不多但需了解。

int input = 0; ioctl(fd, VIDIOC_S_INPUT, &input); // 查询当前输入 ioctl(fd, VIDIOC_G_INPUT, &input); // 设置视频标准 (仅模拟输入) v4l2_std_id std = V4L2_STD_NTSC; ioctl(fd, VIDIOC_S_STD, &std);

videobuf2 框架：零拷贝数据通路的核心

videobuf2 (vb2) 是 V4L2 数据传输的引擎。理解 vb2_queue 生命周期、vb2_ops 回调机制、三种内存模式的内部原理和缓冲策略调优，是掌握 V4L2 高性能采集的关键。

vb2_ops 回调 — 驱动与框架的交互接口

queue_setup()
在 VIDIOC_REQBUFS 和 VIDIOC_CREATE_BUFS 时调用。
驱动在此修正缓冲数量和每个 plane 的大小。
*num_buffers：请求的缓冲数，驱动可增大
*num_planes：plane 数量（NV12=1, NV12M=2）
sizes[]：每个 plane 的字节大小
buf_prepare() / buf_finish()
buf_prepare()：缓冲入队前调用，驱动可做 DMA 缓存一致性准备（dma_sync_single_for_device）。
buf_finish()：缓冲出队后调用，做缓存无效化（dma_sync_single_for_cpu）确保 CPU 读到最新数据。
buf_queue()
每次 VIDIOC_QBUF 时调用。驱动在此将 vb2_buffer 加入硬件队列（通常挂到自定义链表），等待 DMA 引擎取出填充。
vb2_buffer.index：缓冲索引
vb2_buffer.vb2_queue：所属队列
start_streaming() / stop_streaming()
start_streaming()：STREAMON 时调用，驱动启动 DMA 引擎。若缓冲不足 min_buffers_needed 则返回 -ENOBUFS，框架暂不启动。
stop_streaming()：STREAMOFF 时调用，驱动停止 DMA 并回收所有已入队缓冲（调用 vb2_buffer_done(vb, VB2_BUF_STATE_ERROR)）。
// 典型驱动 vb2_ops 实现 static const struct vb2_ops my_vb2_ops = { .queue_setup = my_queue_setup, .buf_prepare = my_buf_prepare, .buf_finish = my_buf_finish, .buf_queue = my_buf_queue, .start_streaming = my_start_streaming, .stop_streaming = my_stop_streaming, .buf_cleanup = my_buf_cleanup, }; // DMA 完成中断中调用 vb2_buffer_done 提交填充好的缓冲 static irqreturn_t my_dma_isr(int irq, void *dev_id) { struct vb2_buffer *vb = dev->active_buf; vb->planes[0].bytesused = dev->frame_size; vb->timestamp = ktime_get_ns(); // 帧时间戳 vb2_buffer_done(vb, VB2_BUF_STATE_DONE); // → done_list // 启动下一个缓冲的 DMA my_start_next_dma(dev); }
MMAP 模式

内核通过 vb2_mem_ops->alloc() 分配 DMA 相容内存（通常来自 CMA 或 dma_alloc_coherent），用户通过 mmap() 将内核物理页映射到用户虚拟地址空间。

内存分配者：内核 (vb2)

零拷贝：是 — 用户直接读写 DMA 缓冲区

限制：应用无法控制内存物理位置和对齐

// MMAP 标准用法 struct v4l2_requestbuffers req = { .count = 4, .type = V4L2_BUF_TYPE_VIDEO_CAPTURE, .memory = V4L2_MEMORY_MMAP, }; ioctl(fd, VIDIOC_REQBUFS, &req); // 查询每个缓冲的偏移和长度，执行 mmap for (int i = 0; i < req.count; i++) { struct v4l2_buffer buf = { .type = req.type, .index = i, .memory = req.memory }; ioctl(fd, VIDIOC_QUERYBUF, &buf); void *p = mmap(NULL, buf.length, PROT_READ | PROT_WRITE, MAP_SHARED, fd, buf.m.offset); }

最优选择：绝大多数采集场景（本项目使用）

USERPTR 模式

用户自行分配内存（malloc / posix_memalign / 自定义分配器），将虚拟地址传给内核。驱动通过 vb2_mem_ops->get_userptr() 做 dma_map_single 建立映射。

内存分配者：用户空间

零拷贝：是 — DMA 直接写入用户缓冲区

限制：内存必须物理连续且 DMA 对齐；高频 QBUF/DQBUF 反复做 map/unmap

// USERPTR 标准用法 void *bufs[4]; for (int i = 0; i < 4; i++) { posix_memalign(&bufs[i], 4096, W * H * 3 / 2); // NV12 } // 入队时传入用户空间地址 struct v4l2_buffer buf = { .type = V4L2_BUF_TYPE_VIDEO_CAPTURE, .memory = V4L2_MEMORY_USERPTR, }; for (int i = 0; i < 4; i++) { buf.index = i; buf.m.userptr = (unsigned long)bufs[i]; buf.length = W * H * 3 / 2; ioctl(fd, VIDIOC_QBUF, &buf); }

适用：需要自定义内存分配器（如 ION）的场景

DMABUF 模式

基于 dma_buf 文件描述符实现跨子系统零拷贝共享。发送端通过 dma_buf_export() 导出 fd，接收端通过 dma_buf_get(fd) 引用同一块物理内存。

内存分配者：导出方 (DRM/GPU/另一个 V4L2 设备)

零拷贝：是 — DMA-BUF 允许不同设备共享同一物理缓冲

限制：需要导出方支持；需要同步 fence 机制防止读写冲突

// DMABUF 典型用法 (采集 → 编码直通) // 1. 从 cap 设备获取 dmabuf fd struct v4l2_buffer cap_buf = { ... }; ioctl(cap_fd, VIDIOC_DQBUF, &cap_buf); int dma_fd = cap_buf.m.fd; // dmabuf fd // 2. 将同一 fd 传给编码设备 struct v4l2_buffer enc_buf = { .type = V4L2_BUF_TYPE_VIDEO_OUTPUT, .memory = V4L2_MEMORY_DMABUF, }; enc_buf.m.fd = dma_fd; ioctl(enc_fd, VIDIOC_QBUF, &enc_buf); // 采集缓冲直接送入编码器，零拷贝

适用：硬件直连管线（采集→编码→显示）

多平面格式 (Multi-plane) 深度解析

V4L2 区分单平面 (single-plane) 和多平面 (multi-plane) API，这是新手最容易混淆的概念之一：

单平面 API：V4L2_BUF_TYPE_VIDEO_CAPTURE + v4l2_pix_format
使用 v4l2_buffer.m.offset 和 v4l2_buffer.length。所有 plane 数据存放在一块连续内存中（NV12: 先 Y 后 UV），只有一个 offset/length 对。

多平面 API：V4L2_BUF_TYPE_VIDEO_CAPTURE_MPLANE + v4l2_pix_format_mplane
使用 v4l2_buffer.m.planes 指向 v4l2_plane 数组。每个 plane 有独立的 offset、length、bytesused、data_offset。
NV12M 的 plane[0] = Y（offset=0, length=W×H），plane[1] = UV（offset=0, length=W×H/2, 独立内存块）。

为什么需要多平面？
① ISP 硬件可能将 Y 和 UV 输出到不同的物理内存区域（non-contiguous），单平面 API 无法表达。
② 编码器 M2M 设备：OUTPUT 端输入 YUV，CAPTURE 端输出 H.264 码流，必须用多平面描述。
③ DMA 引擎分通道搬运 Y 和 UV，分离内存效率更高。

// 多平面缓冲操作 (NV12M) struct v4l2_buffer buf = {}; buf.type = V4L2_BUF_TYPE_VIDEO_CAPTURE_MPLANE; buf.memory = V4L2_MEMORY_MMAP; buf.length = 2; // NV12M = 2 planes struct v4l2_plane planes[2]; buf.m.planes = planes; ioctl(fd, VIDIOC_DQBUF, &buf); // planes[0].bytesused = Y 平面实际字节数 // planes[1].bytesused = UV 平面实际字节数 // planes[0].data_offset = Y 平面内数据偏移（通常为 0）

vb2_mem_ops — 三种内存操作实现

vb2_dma_contig_memops

物理连续 DMA 内存，通过 CMA 或 dma_alloc_coherent() 分配。mmap 零拷贝路径最短。

边界对齐：缓冲区首地址和 size 均 page 对齐，硬件 DMA 可直接搬运。

限制：大分辨率缓冲受 CMA 池大小约束（T31 默认 32MB，1080p NV12 × 4≈12MB，6 缓冲即紧张）。

vb2_vmalloc_memops

虚拟连续物理不连续内存（vmalloc），通过 sgtable 做 scatter-gather DMA 映射。

灵活：不依赖 CMA，碎片化内存也能分配。

开销：每次 QBUF/DQBUF 需要映射/取消映射 sg 表，CPU 开销高于 contig。

适用：驱动调试、不要求极致性能的采集场景。

vb2_dma_sg_memops

物理分散页通过 scatter-gather 表映射，面向 USERPTR 场景优化。

优势：用户空间自定义分配器（ION、dma_heap）可直接传入，缓冲无需内核重新分配。

开销：每次 QBUF 做 dma_map_sg()，DQBUF 做 dma_unmap_sg()，高频操作 CPU 耗损明显。

适用：跨子系统零拷贝（采集→GPU→编码）且用户空间管理内存生命周期的场景。

// vb2_queue_init 中选择 mem_ops struct vb2_queue q = { .io_modes = VB2_MMAP | VB2_DMABUF, // 方案一：连续DMA（T31 ISP采集首选） .mem_ops = &vb2_dma_contig_memops, // 方案二：vmalloc（调试/非实时场景） // .mem_ops = &vb2_vmalloc_memops, // 方案三：scatter-gather（USERPTR/ION场景） // .mem_ops = &vb2_dma_sg_memops, }; vb2_queue_init(&q);

Poll/Wait 机制 & Streaming 错误恢复

vb2_poll() 数据就绪通知

应用调用 poll(fd, POLLIN) → 内核调用 v4l2_poll() → vb2_poll()。

若 done_list 非空，立即返回 POLLIN；否则将 vb2_queue->done_wq 挂入 poll_table 并阻塞。

DMA 完成中断中 vb2_buffer_done() 将缓冲移入 done_list 并 wake_up() 唤醒等待线程。

wait_prepare / wait_finish

驱动可选回调。框架在 poll 阻塞前调用 wait_prepare（释放 v4l2_fh->lock），阻塞结束后调用 wait_finish（重新获取锁）。

为什么需要？若 poll 持有锁且 DMA 中断需同一锁，死锁。释放锁允许中断处理继续执行。

未提供时框架默认调用 v4l2_fh_is_single_file() 安全释放。

start_streaming() 失败恢复

若 start_streaming() 返回错误，框架自动将所有 queued_list 缓冲以 VB2_BUF_STATE_QUEUED 归还到 DEQUEUED 状态。

应用无需手动 REQBUFS(0) 回收，可直接修正参数后重新 QBUF → STREAMON。

常见失败原因：min_buffers_needed 未满足、DMA 引擎初始化失败、Sensor 未就绪。

stop_streaming() 强制回收

STREAMOFF 时框架调用驱动 stop_streaming()，驱动必须在此回调中对每个仍处于 ACTIVE 状态的缓冲调用 vb2_buffer_done(vb, VB2_BUF_STATE_ERROR)。

框架随后将 done_list 中所有缓冲重置为 DEQUEUED，清空所有队列。

注意：若驱动遗漏回收，缓冲永远卡在内核 → 内存泄漏 + 无法再次 STREAMON。

// 用户空间 poll + DQBUF 标准模式 struct pollfd pfd = { .fd = vfd, .events = POLLIN }; while (running) { int ret = poll(&pfd, 1, 1000); // 1s 超时 if (ret < 0) { // EINTR → retry; 真错误 → 退出 } if (pfd.revents & POLLIN) { struct v4l2_buffer buf; buf.type = V4L2_BUF_TYPE_VIDEO_CAPTURE; buf.memory = V4L2_MEMORY_MMAP; ioctl(vfd, VIDIOC_DQBUF, &buf); if (buf.flags & V4L2_BUF_FLAG_ERROR) // 帧数据可能不完整，跳过或重入队 process(frame_addrs[buf.index], buf.bytesused); ioctl(vfd, VIDIOC_QBUF, &buf); // 重入队 } } // 驱动侧 stop_streaming 必须回收所有活跃缓冲 static void my_stop_streaming(struct vb2_queue *q) { // 停止 DMA 引擎 dmaengine_terminate_all(my_chan); // 回收所有仍在硬件队列中的缓冲 while (!list_empty(&dev->active_list)) { struct my_buffer *b = list_first_entry( &dev->active_list, struct my_buffer, list); list_del(&b->list); vb2_buffer_done(&b->vb, VB2_BUF_STATE_ERROR); } }

缓冲策略与延迟优化

低延迟方案 (3 buffers)
延迟 = 3 × 帧间隔
15fps 时 ≈ 200ms
优点：内存占用最小
风险：用户处理超时会导致掉帧
平衡方案 (4-5 buffers)
延迟 = 4~5 × 帧间隔
15fps 时 ≈ 267~333ms
优点：容忍轻微处理抖动
本项目默认选用此方案
高吞吐方案 (6+ buffers)
延迟 = 6+ × 帧间隔
15fps 时 > 400ms
优点：最大容忍处理延迟
代价：内存占用大（1920×1080 NV12 × 6≈18MB）
// 延迟计算公式 帧延迟 = (缓冲数 - 1) × 1000 / fps // ms 内存占用 = W × H × 1.5 × 缓冲数 // NV12 bytes // 例: 1920×1080 NV12 × 4 = 12,441,600 ≈ 11.9 MB // REQBUFS 返回实际分配数 req.count = 6; ioctl(fd, VIDIOC_REQBUFS, &req); // req.count 可能被改为 4 (驱动限制或内存不足) if (req.count < 3) { // 缓冲数不足，无法可靠 streaming }
掉帧诊断
当 VIDIOC_DQBUF 返回的 v4l2_buffer.sequence 不连续时，说明发生了掉帧。常见原因：用户处理耗时超过帧间隔，所有缓冲都在 DEQUEUED 状态，DMA 无可用缓冲写入。解决方案：增加缓冲数量或优化处理路径。

像素格式 & 色彩空间

视频采集输出的原始数据格式直接影响后续编码和显示质量。深入理解 YUV 色彩模型、色度子采样、色彩空间转换和量化范围，是解决画面偏色、暗部断层、编码效率等问题的根本。
点击格式卡片展开详细内存布局与使用指南 ↓

YUYV / UYVY
V4L2_PIX_FMT_YUYV / UYVY

4:2:2 交织格式，每像素 16bit。YUYV = Y₀U₀Y₁V₀，两像素共享 UV。USB 摄像头最常见输出格式。带宽比 NV12 大（UV 不做 4:2:0 子采样），不适合 H.264 编码器直接输入（需格式转换）。

交织字节排列详图

YUYV 每 4 字节构成一个宏块：Y₀ U₀ Y₁ V₀。每两个水平相邻像素共享同一组色度样本。在内存中，亮度与色度紧密交织在一条字节流中，无法像 NV12 那样独立遍历 Y 或 UV。

UYVY 与 YUYV 仅字节序不同：U₀ Y₀ V₀ Y₁。某些硬件（如老式 TV 解码器）输出 UYVY，需确认字节序再做格式转换。

为什么不能直接送 H.264 编码器？

大多数嵌入式 H.264 编码器（包括 T31 的硬件编码器）要求输入 NV12 格式。YUYV 是 4:2:2，编码器需要 4:2:0。转换需要做：① UV 分离（deinterleave）；② 色度垂直 2:1 下采样（丢弃隔行色度行）。该转换开销在 CPU 上约 2-5ms/帧（1080p），在实时场景中不可忽视，应尽量避免。

// YUYV → NV12 转换核心逻辑 (仅示意) for (int y = 0; y < height; y += 2) { for (int x = 0; x < width; x += 2) { // 拷贝 4 个 Y 样本 nv12_y[y*w+x] = yuyv[0]; nv12_y[y*w+x+1] = yuyv[2]; nv12_y[(y+1)*w+x] = yuyv_next_row[0]; nv12_y[(y+1)*w+x+1] = yuyv_next_row[2]; // 取一组 UV（4:2:2 → 4:2:0，丢弃隔行色度行） nv12_uv[(y/2)*w+x] = yuyv[1]; // U nv12_uv[(y/2)*w+x+1] = yuyv[3]; // V } }
NV12 / NV12M
V4L2_PIX_FMT_NV12 / NV12M

4:2:0 半平面格式，IPC 领域最常用。NV12 单平面：Y+UV 连续内存；NV12M 多平面：Y 与 UV 分离。H.264 编码器标准输入格式。Ingenic T31 ISP 输出 NV12M。

NV12 vs NV12M — 单平面与多平面内存模型

NV12 (单平面)：Y 和 UV 存储在一块连续物理内存中。v4l2_buffer.m.offset 指向 Y 起始位置，UV 紧跟其后偏移 W×H 字节。只有一组 offset/length 对。

NV12M (多平面)：Y 和 UV 分配在两块独立物理内存中。需使用 V4L2_BUF_TYPE_VIDEO_CAPTURE_MPLANE + v4l2_plane 数组。planes[0] 描述 Y 平面，planes[1] 描述 UV 平面。T31 ISP 使用此模式，因为 DMA 引擎将 Y 和 UV 分通道输出到不同物理区域。

// NV12M 多平面映射示意 // plane[0]: Y 平面 mmap(NULL, planes[0].length, PROT_READ, MAP_SHARED, fd, planes[0].m.mem_offset); // 独立 offset // plane[1]: UV 平面 mmap(NULL, planes[1].length, PROT_READ, MAP_SHARED, fd, planes[1].m.mem_offset); // 不同 offset

bytesperline & stride 对齐

V4L2 中 v4l2_pix_format.bytesperline 表示每行字节数（stride），不一定等于 width。T31 ISP 输出的 NV12M 的 Y 平面 bytesperline 通常为 ALIGN(width, 64)，即宽度向上对齐到 64 的倍数。UV 平面 bytesperline 与 Y 相同。

实战陷阱：如果用户空间假设 stride == width，而实际有 padding，会导致图像倾斜/撕裂。正确做法是按 bytesperline 逐行拷贝，跳过每行尾部的 padding 字节。

// 正确读取 NV12 Y 平面 uint8_t *y_data = mmap_ptr; for (int r = 0; r < height; r++) { memcpy(dst + r * width, y_data + r * bytesperline, // 跳过 padding width); }

10bit Packing 格式

V4L2 区分 packed 和 unpacked 10bit RAW：
Packed (SBGGR10P_1X10)：5 字节存储 4 个像素，byte[0..3] 各存 8bit 低 8 位，byte[4] 高 8bit 存 4 个像素的 bit[8:9]。内存紧凑但解析复杂。
Unpacked (SBGGR10)：每像素 2 字节（16bit），低 10bit 有效，高 6bit 填 0。内存冗余但硬件对齐度高，T31 ISP 默认使用此格式。

MJPEG
V4L2_PIX_FMT_MJPEG

硬件 JPEG 编码输出，USB 摄像头常见。CPU 解码开销大，不适合实时流。需解码为 YUV 才能送入 H.264 编码器。JPEG 内部使用 BT.601 色彩空间。

JPEG 码流结构速查

MJPEG 每帧是一完整的 JPEG 图片，包含标记段：FFD8(SOI) → FFE0(APP0/JFIF) → FFDB(DQT 量化表) → FFC0(SOF0 帧头,含宽高) → FFC4(DHT 霍夫曼表) → FFDA(SOS 扫描开始) → 熵编码数据 → FFD9(EOI)。

关键点：SOF0 标记中有 Y/Cb/Cr 采样因子，MJPEG 通常为 2:1:1（即 4:2:0），色彩空间隐含 BT.601 full range。编码器从 MJPEG 解码后若未正确标记色彩空间属性，会导致后续 H.264 编码色偏。

为什么 MJPEG 不适合实时流？

① 压缩效率低：JPEG 是帧内编码，I-frame 尺寸约为 H.264 的 5-10 倍，同等画质下带宽占用大。
② 解码开销：H.264 硬件编码器不接受 JPEG 输入，需 CPU/software 解码为 NV12 再送编码器，1080p@15fps 解码约 10-15ms/帧。
③ 延迟跳动：JPEG 编码时间是帧内容的函数（高频细节多→编码慢），无法保证恒定帧间延迟。

硬件编码器输出，已是压缩码流。V4L2 编码设备 (M2M) 的 CAPTURE 端输出此格式。IMP_Encoder 封装了此流程，输出 Annex-B 码流。

M2M 编码器双队列模型

V4L2 Memory-to-Memory 编码器使用两个独立队列：
OUTPUT 队列：V4L2_BUF_TYPE_VIDEO_OUTPUT_MPLANE，接收 NV12M 输入帧。应用 QBUF 一帧 YUV → 编码器取出处理。
CAPTURE 队列：V4L2_BUF_TYPE_VIDEO_CAPTURE_MPLANE，输出 H.264/HEVC 码流。编码完成 → 应用 DQBUF 取出码流。

// M2M 编码标准流程 // 1. OUTPUT 队列配置输入格式 fmt.type = V4L2_BUF_TYPE_VIDEO_OUTPUT_MPLANE; fmt.fmt.pix_mp.pixelformat = V4L2_PIX_FMT_NV12M; ioctl(enc_fd, VIDIOC_S_FMT, &fmt); // 2. CAPTURE 队列配置输出格式 fmt.type = V4L2_BUF_TYPE_VIDEO_CAPTURE_MPLANE; fmt.fmt.pix_mp.pixelformat = V4L2_PIX_FMT_H264; ioctl(enc_fd, VIDIOC_S_FMT, &fmt); // 3. 分别 REQBUFS + QBUF 两个队列，然后 STREAMON
码流输出时序

编码器的输出不保证帧顺序，尤其在 B 帧场景下。典型时序：
① 应用请求关键帧（V4L2_CID_MPEG_VIDEO_FORCE_KEY_FRAME）
② 首次 DQBUF 返回 SPS/PPS + IDR 帧（Annex-B 格式：00 00 00 01 67...）
③ 后续 DQBUF 返回 P 帧码流
④ 应用需自行处理码流分割和封装（MP4/FLV/直播协议）

IMP 对照：IMP_Encoder_PollingStream() 对应 CAPTURE DQBUF 等待，IMP_Encoder_GetStream() 对应 DQBUF 后读取 scatter pack 数据并拼接为 Annex-B。IMP 已处理 SPS/PPS 提取，应用直接获得完整 NAL 单元。

YUV 色彩模型 — 为什么视频不用 RGB？

Y (Luminance)：亮度分量，人眼最敏感。单独的 Y 就是灰度图。R=0.299, G=0.587, B=0.114 的加权平均。

U/V (Chrominance)：色度分量，人眼敏感度低。U = B−Y (蓝色差), V = R−Y (红色差)。因为人眼对色彩细节不敏感，UV 可以大幅子采样而不被察觉。

核心优势：① 人眼对亮度敏感、色度迟钝，UV 子采样可节省 50% 带宽；② Y 分量天然兼容黑白显示；③ 与模拟电视兼容（Y=同步+亮度, UV=色度副载波）。

// BT.601 SD 标准转换矩阵（full range） Y = 0.299R + 0.587G + 0.114B Cb = -0.169R - 0.331G + 0.500B + 128 Cr = 0.500R - 0.419G - 0.081B + 128 // BT.709 HD 标准转换矩阵 Y = 0.2126R + 0.7152G + 0.0722B Cb = -0.115R - 0.386G + 0.500B + 128 Cr = 0.500R - 0.454G - 0.046B + 128 // 逆向：YUV → RGB（BT.601 limited range） R = Y + 1.402 × (Cr - 128) G = Y - 0.344 × (Cb - 128) - 0.714 × (Cr - 128) B = Y + 1.772 × (Cb - 128)

色彩空间深潜 — BT.601 vs BT.709 与量化范围

BT.601 (SD)：标清标准，PAL/NTSC 时代。V4L2 中 V4L2_COLORSPACE_SMPTE170M。转换矩阵侧重 SD 分辨率（≤576 行）的色彩表现。JPEG 默认色彩空间也是 BT.601。

BT.709 (HD)：高清标准，1080p/4K 视频。V4L2 中 V4L2_COLORSPACE_REC709。转换矩阵在红色和蓝色通道的权重比例与 BT.601 不同。

错配症状：如果 ISP 输出标记了 BT.709 但编码器按 BT.601 解码，画面会偏绿偏暗。这是 IPC 画面偏色的最常见原因之一。

Limited Range (16-235)：广播视频标准。Y 范围 [16,235]，UV 范围 [16,240]。留出 headroom 给同步信号和超限值。V4L2 默认。

Full Range (0-255)：计算机图形标准。Y/UV 均使用 [0,255] 全范围。适合屏幕共享、游戏。

范围错配后果：Full range 数据按 Limited 显示 → 画面发雾（对比度降低）。Limited 数据按 Full 显示 → 暗部挤压、亮部过曝。
