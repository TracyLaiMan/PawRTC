1. V4L2 子系统架构

四层分层结构 · 三组设备节点 · 内核注册流程 · video_device · v4l2_subdev · 核心源码位置

四层架构

从应用到硬件的完整调用链

Layer 1 — 用户空间: open/ioctl/mmap/poll/close 操作 /dev/video0。上层框架: GStreamer(v4l2src), FFmpeg(libavdevice), OpenCV, 本项目 ImpSource。
Layer 2 — V4L2 Core: video_ioctl2() 是 ioctl 分发中枢——查 v4l2_ioctls[] 表找对应回调。管理 v4l2_device(设备根) + video_device(设备节点) + v4l2_subdev(子设备) + vb2_queue(缓冲区)。
Layer 3 — 子设备层: Sensor/ISP/DMA 引擎各注册为独立 v4l2_subdev。Pad 格式协商。V4L2 Async 协调跨总线 probe 顺序。
Layer 4 — 硬件驱动: 平台驱动实现 v4l2_ioctl_ops + vb2_ops。操作硬件寄存器、配置 DMA、处理中断。

三组设备节点 & 关键源码

每个/dev 节点对应一种内核对象

/dev/videoX — 主设备。CAPTURE(采集)/OUTPUT(输出)/M2M(编解码)。video_register_device() 创建。
/dev/v4l-subdevX — 子设备。Sensor/ISP 的寄存器级控制。v4l2_device_register_subdev() 创建。
/dev/mediaX — 拓扑。entity/pad/link 查询修改。media_device_register() 创建。

核心源码: drivers/media/v4l2-core/v4l2-ioctl.c (ioctl 分发), v4l2-dev.c (video_device 管理), drivers/media/common/videobuf2/ (vb2 缓冲区引擎), drivers/media/v4l2-core/v4l2-subdev.c (子设备框架), drivers/media/mc/ (Media Controller)。

/* ===== 标准 V4L2 采集循环 — 可编译运行的完整代码 ===== */ #include <fcntl.h> #include <sys/ioctl.h> #include <sys/mman.h> #include <linux/videodev2.h> int main() { int fd = open("/dev/video0", O_RDWR | O_NONBLOCK); if (fd < 0) { perror("open"); return -1; } /* ── ① 查询设备能力 ── */ struct v4l2_capability cap; if (ioctl(fd, VIDIOC_QUERYCAP, &cap) < 0) return -1; if (!(cap.capabilities & V4L2_CAP_VIDEO_CAPTURE)) { fprintf(stderr, "Not a capture device\n"); return -1; } if (!(cap.capabilities & V4L2_CAP_STREAMING)) { fprintf(stderr, "Streaming not supported\n"); return -1; } /* ── ② 设置格式: 1920x1080 NV12, 30fps ── */ struct v4l2_format fmt = {0}; fmt.type = V4L2_BUF_TYPE_VIDEO_CAPTURE; fmt.fmt.pix.width = 1920; fmt.fmt.pix.height = 1080; fmt.fmt.pix.pixelformat = V4L2_PIX_FMT_NV12; fmt.fmt.pix.field = V4L2_FIELD_NONE; if (ioctl(fd, VIDIOC_S_FMT, &fmt) < 0) { perror("S_FMT"); return -1; } ioctl(fd, VIDIOC_G_FMT, &fmt); // ★ 读回确认 (驱动可能调整!) printf("Actual: %dx%d, sizeimage=%u\n", fmt.fmt.pix.width, fmt.fmt.pix.height, fmt.fmt.pix.sizeimage); struct v4l2_streamparm parm = {.type = V4L2_BUF_TYPE_VIDEO_CAPTURE}; parm.parm.capture.timeperframe.numerator = 1; parm.parm.capture.timeperframe.denominator = 30; ioctl(fd, VIDIOC_S_PARM, &parm); /* ── ③ 申请 4 个 MMAP 缓冲区 ── */ struct v4l2_requestbuffers req = {0}; req.count = 4; req.type = V4L2_BUF_TYPE_VIDEO_CAPTURE; req.memory = V4L2_MEMORY_MMAP; if (ioctl(fd, VIDIOC_REQBUFS, &req) < 0) { perror("REQBUFS"); return -1; } void *buffers[4]; size_t buf_sizes[4]; for (int i = 0; i < req.count; i++) { struct v4l2_buffer buf = {.type = req.type, .index = i, .memory = req.memory}; ioctl(fd, VIDIOC_QUERYBUF, &buf); buf_sizes[i] = buf.length; buffers[i] = mmap(NULL, buf.length, PROT_READ | PROT_WRITE, MAP_SHARED, fd, buf.m.offset); if (buffers[i] == MAP_FAILED) { perror("mmap"); return -1; } ioctl(fd, VIDIOC_QBUF, &buf); // 全部入队 } /* ── ④ 启动采集 ── */ enum v4l2_buf_type type = V4L2_BUF_TYPE_VIDEO_CAPTURE; if (ioctl(fd, VIDIOC_STREAMON, &type) < 0) { perror("STREAMON"); return -1; } /* ── ⑤ 采集循环: 取 100 帧 ── */ for (int frame = 0; frame < 100; frame++) { struct pollfd pfd = {.fd = fd, .events = POLLIN}; if (poll(&pfd, 1, 2000) <= 0) { fprintf(stderr, "poll timeout\n"); break; } struct v4l2_buffer buf = {.type = req.type, .memory = req.memory}; if (ioctl(fd, VIDIOC_DQBUF, &buf) < 0) { perror("DQBUF"); break; } // ★ 这里处理数据: buffers[buf.index] 指向 DMA 填充好的帧 // buf.bytesused = 实际数据长度, buf.timestamp = 帧时间戳 (ns) // buf.sequence = 帧序号, buf.field = V4L2_FIELD_NONE (逐行) process_frame(buffers[buf.index], buf.bytesused, buf.timestamp); if (ioctl(fd, VIDIOC_QBUF, &buf) < 0) { perror("QBUF"); break; } } /* ── ⑥ 停止 + 清理 ── */ ioctl(fd, VIDIOC_STREAMOFF, &type); for (int i = 0; i < req.count; i++) munmap(buffers[i], buf_sizes[i]); close(fd); return 0; }
ioctl用途关键结构体 / 字段
VIDIOC_QUERYCAP查询设备能力v4l2_capability.capabilities (V4L2_CAP_VIDEO_CAPTURE|STREAMING)
VIDIOC_ENUM_FMT枚举支持的像素格式v4l2_fmtdesc.pixelformat (四字符码 fourcc)
VIDIOC_S_FMT / G_FMT设置/获取采集格式v4l2_pix_format: width/height/pixelformat/bytesperline/sizeimage/field
VIDIOC_TRY_FMT试探格式 — 不实际修改硬件同 S_FMT，但驱动只返回修正值不应用
VIDIOC_S_PARM / G_PARM设置/获取帧率v4l2_streamparm.parm.capture.timeperframe (numerator/denominator)
VIDIOC_REQBUFS申请 N 个缓冲区v4l2_requestbuffers: count/type/memory (MMAP/USERPTR/DMABUF)
VIDIOC_QUERYBUF查询缓冲物理偏移+长度v4l2_buffer: index/m.offset/length (mmap 必需)
VIDIOC_QBUF / DQBUF入队/出队缓冲区v4l2_buffer: index/bytesused/timestamp/sequence/flags
VIDIOC_STREAMON/OFF启动/停止 DMA 采集参数: v4l2_buf_type (V4L2_BUF_TYPE_VIDEO_CAPTURE)
VIDIOC_G_CTRL / S_CTRL读/写控制项v4l2_control: id/value (亮度/V4L2_CID_BRIGHTNESS 等)

3. videobuf2 — 深度剖析

vb2 是整个 V4L2 数据路径的核心引擎 — 理解它的架构、状态机、内存模型和回调体系，才能真正掌握 V4L2

vb2_queue — 队列管理器

调度中枢 · 锁保护 · 流控制

核心字段: type (CAPTURE/OUTPUT), io_modes (VB2_MMAP|VB2_DMABUF), mem_ops (内存分配器指针), ops (驱动回调), buf_struct_size (每缓冲私有数据大小), min_buffers_needed (启动流的最小缓冲数)。
三个链表: queued_list (已入队待DMA), done_list (DMA完成待取), allocated_bufs (所有已分配缓冲)。
锁: lock — 驱动提供的 spinlock/mutex，保护队列操作。VB2 框架在调用 vb2_ops 回调之前会释放锁，避免回调中死锁。

vb2_buffer — 缓冲区对象

多 plane · 元数据 · 状态

关键字段: index (缓冲编号 0~N-1), state (状态机), planes[] (多平面数据), timestamp (帧时间戳), sequence (帧序号), queued_count (入队次数)。
vb2_plane: 每个平面包含 mem_priv (内存私有数据), length (总长度), bytesused (实际填充字节), m.offset (MMAP偏移), dbuf (DMABUF文件描述符)。
私有扩展: 通过 vb2_buffer 尾部附加 (vb2_queue.buf_struct_size 控制) — 驱动用 container_of 获取私有数据。

vb2_ops — 驱动回调

框架调驱动 · 6 个必须实现

queue_setup: REQBUFS/CREATE_BUFS时—修正 *num_buffers,*num_planes,sizes[]。
buf_prepare: QBUF前—DMA缓存一致性准备(dma_sync_single_for_device)。
buf_finish: DQBUF后—CPU缓存同步(dma_sync_single_for_cpu)。
buf_queue: QBUF时—加入硬件队列链表。
start_streaming: ★ STREAMON—启动DMA。返回-EBUSY若缓冲不足。
stop_streaming: ★ STREAMOFF—停止DMA，用vb2_buffer_done(VB2_BUF_STATE_ERROR)回收所有已入队缓冲。

3.1 缓冲区完整状态机 — 六态流转

① DEQUEUED
初始状态
等待 QBUF 入队
② QUEUED
QBUF 后
在 queued_list 中
等待 DMA 取用
③ ACTIVE
驱动从队列取出
DMA 正在填充
(驱动内部状态)
④ DONE
vb2_buffer_done()
→ 进入 done_list
等待 DQBUF 取出
⑤ ERROR
DMA 出错时
vb2_buffer_done(ERROR)
→ DQBUF 时返回错误
⑥ 回收
STREAMOFF 时
所有 queued+active
→ done(vb, ERROR)

3.2 四种 vb2_mem_ops 实现 — 内存模型深度对比

vb2_dma_contig ★ 嵌入式标配

DMA 连续物理内存 · CMA 或 coherent pool

alloc(): 调用 dma_alloc_coherent() 分配连续物理内存并返回一致的 DMA 地址。底层来自 CMA (Contiguous Memory Allocator) 或内核 coherent pool。
put(): dma_free_coherent() 释放。
mmap(): dma_mmap_coherent() 将 DMA 缓冲区映射到用户虚拟地址。
缓存: coherent 内存 = 非缓存/写透 → CPU 和 DMA 总看到一致数据，不需手动 cache 操作。
限制: 需要连续物理页 → 大缓冲可能失败 → 增大 CMA 区或降级到 dma-sg。

vb2_dma_sg — 分散/聚集

sg_table · 非连续物理页 · 大缓冲

alloc(): 通过 alloc_pages() 分配多个物理页 → 构建 sg_table (scatter-gather 表) → dma_map_sg() 建立 DMA 映射。
mmap(): 逐页 vm_insert_page() 映射 → 用户空间看到虚拟连续但物理分散的缓冲区。
缓存: buf_prepare 中 dma_sync_sg_for_device，buf_finish 中 dma_sync_sg_for_cpu。
优势: 不依赖大块连续内存 → 大画面 (4K) 也能分配。USB 摄像头驱动常用。

vb2_vmalloc — 虚拟连续

vmalloc 分配 · 软件 DMA 映射 · 罕见

alloc(): vmalloc_user() 分配虚拟连续内存。不保证物理连续。
mmap(): remap_vmalloc_range() 映射到用户空间。
缓存: 需要显式 cache 操作 — 性能开销大。
适用: 仅在 CPU 做软件处理的场景 (如软件编解码)，DMA 设备不可用 vmalloc 内存。
现状: 几乎被 dma-sg 替代，仅遗留驱动使用。

vb2_dma_buf — 跨设备共享

DMABUF · 零拷贝跨设备 · DRM→V4L2

设计: 不分配内存 — 接收外部 dmabuf fd → dma_buf_attach() → dma_buf_map_attachment()。缓冲区由其他设备 (GPU/DRM/VPU) 分配。
典型流程: DRM 渲染帧到 GPU 内存 → 导出 DMABUF fd → V4L2 编码器通过 DMABUF 直接访问 → 零拷贝编码。
接口: VIDIOC_EXPBUF 导出 vb2 缓冲为 DMABUF fd；VIDIOC_QBUF 时传 DMABUF fd 作为输入。

3.3 vb2_ops 每个回调的深入解读

/* ===== vb2_ops — 完整定义 + 每个回调的详细语义 ===== */ struct vb2_ops { /* ── ① queue_setup — REQBUFS / CREATE_BUFS 时调用 ── */ /* 驱动在此校验和调整: *num_buffers(最小缓冲数), *num_planes(YUV=1,NV12M=2), sizes[] (每plane字节)。 * 如果 sizes[] == 0 → 框架会在后续 buf_prepare 中由驱动填充。 * alloc_devs[]: 每个 plane 可指定不同设备 (用于跨设备 DMABUF)。 * 返回值: 0=成功, 负值=失败。 */ int (*queue_setup)(struct vb2_queue *q, unsigned int *num_buffers, unsigned int *num_planes, unsigned int sizes[], struct device *alloc_devs[]); /* ── ② buf_init — 每个缓冲区分配后调用一次 ── */ /* 驱动初始化缓冲私有数据 (vb2_buffer 尾部的自定义结构体)。 * 可在此时做一次性的硬件相关设置。 */ int (*buf_init)(struct vb2_buffer *vb); /* ── ③ buf_prepare — 每次 QBUF 前调用 ── */ /* ★ 关键: 驱动在此做 DMA 缓存一致性准备 (dma_sync_single_for_device)。 * 对于 dma_contig memops: 通常不需要 (coherent 内存自动一致)。 * 对于 dma_sg memops: 必须做 dma_sync_sg_for_device。 * 同时校验: vb2_plane.bytesused 是否合法 (≤ length), 缓冲区参数是否正确。 */ int (*buf_prepare)(struct vb2_buffer *vb); /* ── ④ buf_finish — 每次 DQBUF 后调用 ── */ /* 驱动在此同步 CPU 缓存 (dma_sync_single_for_cpu)，确保 CPU 看到最新数据。 * 对于 coherent 内存: 不需要。 * 返回值反作用: 若返回错误，DQBUF 会向用户返回错误。 */ int (*buf_finish)(struct vb2_buffer *vb); /* ── ⑤ buf_cleanup — 缓冲区释放前调用 ── */ /* buf_init 的反操作 — 释放硬件相关资源。 */ void (*buf_cleanup)(struct vb2_buffer *vb); /* ── ⑥ buf_queue — QBUF 时调用 ── */ /* ★ 驱动在此将 vb2_buffer 加入硬件队列 (自定义链表)。 * 若 DMA 空闲，可在此立即启动 DMA。否则排队等待。 * 调用时机: QBUF ioctl → vb2_core_qbuf() → __enqueue_in_driver() → buf_queue()。 * 已经在 vb2 队列锁保护外 — 驱动可安全获取自己的锁。 */ void (*buf_queue)(struct vb2_buffer *vb); /* ── ⑦ start_streaming — STREAMON 时调用 ── */ /* ★★★ 最关键的驱动回调。驱动在此: * 1. 启动硬件 DMA 引擎 (配置寄存器、使能中断) * 2. 从硬件队列取第一个缓冲区 → 启动 DMA * count 参数: 当前已入队的缓冲数。若 < min_buffers_needed，返回 -ENOBUFS。 * 若启动失败: 必须调用 vb2_buffer_done(vb, VB2_BUF_STATE_QUEUED) 返回所有缓冲。 * 不能在 start_streaming 中睡眠 (非原子上下文)。 */ int (*start_streaming)(struct vb2_queue *q, unsigned int count); /* ── ⑧ stop_streaming — STREAMOFF 时调用 ── */ /* ★ 驱动在此: 1.停止 DMA (禁用中断) 2. 回收所有已入队缓冲: * for (each vb in queued_list) vb2_buffer_done(vb, VB2_BUF_STATE_ERROR); * 3. 等待 DMA 完全停止 (若硬件需等待完成) */ void (*stop_streaming)(struct vb2_queue *q); };
/* ===== 驱动 DMA 完成中断 — 提交填充好的缓冲区 ===== */ static irqreturn_t my_dma_frame_done_isr(int irq, void *dev_id) { struct my_dev *dev = dev_id; unsigned long flags; spin_lock_irqsave(&dev->slock, flags); if (dev->active_buf) { struct vb2_buffer *vb = dev->active_buf; /* 填充 vb2 元数据 */ vb->planes[0].bytesused = dev->frame_size; // 实际字节数 vb->timestamp = ktime_get_ns(); // 帧时间戳 (CLOCK_MONOTONIC ns) vb->sequence = dev->frame_seq++; // 帧序号 (自增) /* ★ 关键: vb2_buffer_done 将缓冲从 ACTIVE → DONE → done_list */ vb2_buffer_done(vb, VB2_BUF_STATE_DONE); dev->active_buf = NULL; } /* 从硬件队列取下一个缓冲 → 启动 DMA */ if (!list_empty(&dev->hw_queue)) { struct my_vb2_buf *mbuf = list_first_entry(&dev->hw_queue, ...); list_del(&mbuf->list); dev->active_buf = &mbuf->vb; my_start_dma(dev, mbuf->vb.planes[0].m.offset, mbuf->vb.planes[0].length); } spin_unlock_irqrestore(&dev->slock, flags); return IRQ_HANDLED; } /* ===== buf_queue — QBUF 时将缓冲加入硬件队列 ===== */ static void my_buf_queue(struct vb2_buffer *vb) { struct my_dev *dev = vb2_get_drv_priv(vb->vb2_queue); struct my_vb2_buf *mbuf = container_of(vb, struct my_vb2_buf, vb); unsigned long flags; spin_lock_irqsave(&dev->slock, flags); list_add_tail(&mbuf->list, &dev->hw_queue); // 加入硬件队列 /* 若 DMA 空闲且当前无 active buffer → 立即启动 */ if (!dev->active_buf && !dev->streaming_stopped) { list_del(&mbuf->list); dev->active_buf = vb; my_start_dma(dev, vb->planes[0].m.offset, vb->planes[0].length); } spin_unlock_irqrestore(&dev->slock, flags); }

4. Media Controller — 管线拓扑管理

三要素: entity · pad · link

图论模型描述视频管线

media_entity: 管线中的一个功能节点。每个 entity 有类型 (VIDEO_DEVICE/SUBDEV_SENSOR/SUBDEV_ISP...)。
media_pad: entity 的输入/输出端口。每个 pad 有 flags (MEDIA_PAD_FL_SINK/SOURCE) 和索引。
media_link: 两个 pad 之间的有向连接。flags 含 MEDIA_LNK_FL_ENABLED (是否激活)、MEDIA_LNK_FL_IMMUTABLE (硬件固定不可改)、MEDIA_LNK_FL_DYNAMIC (用户可改)。

V4L2 Async 框架

解决 I2C Sensor 与 Platform CSI 的时序问题

问题: Sensor 在 I2C bus 上 probe, CSI 控制器在 Platform bus 上 probe — 无法保证谁先就绪。
解决方案: v4l2_async_notifier 机制。① 每个 subdev probe 时调用 v4l2_async_register_subdev() → 全局 notifier 链表建立 fwnode → subdev 映射 ② CSI probe 时用 v4l2_async_nf_parse_fwnode_endpoints() 解析 DT remote-endpoint → 创建待匹配列表 ③ v4l2_async_nf_register() 注册 notifier → 检查待匹配 subdev 是否已就绪 → 回调 .bound() → 全部就绪 → .complete() 注册 video_device。

NV12 内存布局 — 理解这个格式是调试花屏/颜色错误的关键
单 plane (NV12): [Y0 Y1 ... Y_{W×H-1}] [U0 V0 U1 V1 ... U_{W/2×H/2-1} V_{W/2×H/2-1}]
双 plane (NV12M): plane[0] = Y 独立缓冲区, plane[1] = UV 独立缓冲区 (可能在不同物理地址)
sizeimage = W×H + W/2×H/2×2 = W×H×1.5 (1920×1080 NV12 = ~3.1MB/帧)
为什么是 NV12? Y 独立=亮度可直读分析 · UV 交错=硬件 H.264/H.265 编码器标准输入 · 12bpp vs YUYV 16bpp=省 25% 带宽
