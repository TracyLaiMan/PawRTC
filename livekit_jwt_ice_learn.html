Secret Key 深度理解

JWT 使用 HMAC-SHA256 签名，Secret Key 是签发和验证双方共享的秘密。
它从不直接传输，只用于本地计算签名——这就是"身份证明"的根基。

核心定义
Secret Key 是什么
Secret Key 是只有签发方和验证方共享的秘密字符串，用于 HMAC-SHA256 签名计算：

Signature = HMAC-SHA256(base64url(header) + "." + base64url(payload), Secret Key)

JWT 的 Header 和 Payload 是明文 Base64 编码，任何人都能解码。Secret Key 保证：
• 无法篡改 payload 后生成有效签名
• 验证方可以确认"这段 payload 确实是你签发的"
安全属性
关键安全特性
1. 对称密钥：HS256 是对称算法，签发和验证用同一个 Secret。所以只有信任的组件才能持有它。

2. 绝不传输：Secret 只在本地的 JWT 签发逻辑中使用，不作为请求参数发送。

3. 泄露 = 灾难：任何人拿到 Secret 就能伪造任意身份的 JWT（iss、sub、room 随便填）。

4. 板端特殊性：板端二进制中硬编码了 API Secret，固件提取 = Secret 泄露。这是嵌入式设备的固有风险。

Base64 编码：二进制与文本的翻译器

Base64 不是加密，不是压缩，而是格式转换——把二进制数据转成纯 ASCII 文本。
因为 JWT、HTTP Header、JSON 等通道只接受文本，遇到不可见字节会截断或乱码。

Base64 编码
编码原理：3 字节 → 4 字符

每 3 个原始字节（24 bits）拆成 4 组，每组 6 bits，查表映射为 A-Z a-z 0-9 + / 共 64 个字符。
末尾不足 3 字节用 = 补齐。

代价：体积膨胀 33% — 每 3 字节变成 4 字符。

两种变体

标准 Base64+ /保留 =邮件附件、PEM 证书
Base64url- _去掉 =JWT、URL 参数

Base64url 是 JWT 的选择：+ / = 在 URL 和 JSON 中有特殊含义，会导致解析错误。

Base64 是加密？
不是，任何人都能解码，没有密钥

格式转换器

误解
Base64 能压缩？
不能，反而膨胀 33%
误解
Base64 防篡改？
不能，解码→改数据→重新编码即可，防篡改靠签名
在 LiveKit/WebRTC 中到处都是

• JWT 三段：Header、Payload、Signature 各自 Base64url 编码后用 . 连接
• X.509 证书 (PEM)：-----BEGIN CERTIFICATE----- 包裹的就是 Base64(DER 编码的证书)
• ICE ice-pwd：原始随机字节 → Base64 传输 → 解码回二进制做 HMAC
• SRTP 密钥导出：PRF 输出二进制 → 按 offset 切割（不需要 Base64，内部使用）

ICE 认证机制

STUN 短期凭证 + TURN 长期凭证，双重保护 ICE 连接的建立过程。

STUN 短期凭证
Binding Request 认证
凭证来源：SDP 中 a=ice-ufrag 和 a=ice-pwd

USERNAME 格式：remote_ufrag:local_ufrag

MESSAGE-INTEGRITY：
HMAC-SHA1(ice-pwd, 整个STUN消息) → 20 字节

FINGERPRINT：
CRC32(整个消息) → 4 字节，检测 NAT/中间盒篡改

特点：对称密钥认证——双方持有相同的 ice-pwd，彼此验证
TURN 长期凭证
Allocate 请求认证
凭证来源：LiveKit Cloud 在 JoinResponse 中动态下发

认证流程：HTTP Digest 式挑战-响应

1. 首次请求无认证 → 401 + REALM + NONCE
2. 计算 md5 = MD5(user:realm:pass)
3. MESSAGE-INTEGRITY = HMAC-SHA1(md5, message)

安全属性：
• NONCE 防重放攻击
• 密码不明文传输
• 凭证每会话不同
// STUN Binding Request 认证流程 板端 → 服务器: Binding Request USERNAME: "KXfxYlfO:4aHMxe4V" MESSAGE-INTEGRITY: HMAC-SHA1(ice-pwd, msg) FINGERPRINT: CRC32(msg) 服务器验证: 1. 用本地 ice-pwd 重新计算 HMAC-SHA1 2. 与 MESSAGE-INTEGRITY 比对 → 不匹配则丢弃 3. CRC32 检测中间盒篡改 4. 验证通过 → 回复 Binding Response

ice-pwd 生命周期

从随机生成到安全传递到最终过期——ice-pwd 是 ICE 认证的信任根。

生成
随机生成 ice-ufrag + ice-pwd
板端和服务器各自生成自己的 credentials：
ice-ufrag: 4-16 字节随机字符串（用户名片段）
ice-pwd: 22-32 字节随机字符串（密码）

来源：/dev/urandom → CTR-DRBG，与 DTLS 临时密钥同源。
代码：IceAgent.c — iceAgentInit()
交换
通过 TLS 保护的信令通道传递
SDP offer/answer 中包含双方的 ice-ufrag 和 ice-pwd：
a=ice-ufrag:4aHMxe4V
a=ice-pwd:5qUzIr4WgQ7T5xRcsCyTZOEF

关键安全依赖：这些字段通过 WSS (TLS 1.2) 传输。如果 TLS 被攻破（如 VERIFY_OPTIONAL），攻击者可以篡改 ice-pwd → 替换为自己的 → 劫持 ICE 连接。
使用
每次 STUN 消息都用 HMAC-SHA1 认证
Connectivity Check 阶段，每一个 Binding Request/Response 都携带：
MESSAGE-INTEGRITY = HMAC-SHA1(ice-pwd, 整个STUN包)

确保只有持有 ice-pwd 的一方才能生成有效的 STUN 消息。
对称认证：双方都知道对方的 ice-pwd（从 SDP 获取），所以可以互相验证。
过期
ICE 会话结束，凭证失效
ice-pwd 绑定于一个 ICE 会话。会话结束后（连接关闭或 ICE restart），旧凭证作废。

如果需要重新连接，双方生成新的 ice-ufrag + ice-pwd → 新的 SDP 交换。
ICE restart：代码中 IceAgent.c — iceAgentRestart()，重新生成所有凭证。

项目中的应用

JWT 通过 URL query 传递，ICE 凭证通过 SDP 传递——两者都依赖 TLS 信令通道的安全性。

JWT 认证 — 身份证明
板端 JWT 的签发与传递
后端用 API Secret 签发 JWT，板端通过 WSS URL query string 传递：
wss://host/rtc?access_token=eyJ...&protocol=17

SFU 验证 JWT 签名 → 确认设备身份和权限 → 允许/拒绝加入房间。

⚠️ 当前风险：JWT 在 URL 中传输，可能被服务器日志/CDN 缓存记录。
缓解措施：短 TTL (3600s) + 迁移到 header 传递。

JWT vs ICE 认证对比

两者都用 HMAC，但角色完全不同：JWT 证明"我是谁"，ICE 证明"这条消息没被篡改"。

总结：JWT 和 ICE 认证是互补关系——JWT 在信令层一次性验证整个会话的身份，ICE 在传输层逐包验证每条消息的完整性。两者都依赖 HMAC，但 JWT 的密钥 (API Secret) 是长期静态的，ICE 的密钥 (ice-pwd) 是短期临时的。这正是 ECDHE 哲学的延伸：长期密钥用于身份，短期密钥用于数据。
