3. 关键补齐项详解 — 从"会用"到"资深"

按优先级排序 — P0 (面试高频+日常工作急需) → P1 (进阶必备) → P2 (进阶加分)

🔴 P0 — 高优先级 (面试必问 + 日常核心)

WebRTC 深度架构

优先级 P0 — 直播/视频通话核心

缺失内容:
① PeerConnection 内核 — SDP Offer/Answer 协商、ICE Candidate 收集、DTLS-SRTP 密钥交换
② 拥塞控制 — GCC (Google Congestion Control) 算法: 基于延迟的估计器 + 基于丢包的估计器 → 目标码率
③ Simulcast — 同时编码 3 路不同分辨率 (720p/360p/180p) → SFU 选择性转发
④ SVC (可伸缩编码) — 时域/空域/质量分层
⑤ NetEQ — 音频抖动缓冲与丢包补偿 (PLC)
⑥ 带宽估计 — REMB/TWCC 接收端/发送端带宽估计
面试频率: 极高 — 几乎是音视频岗位的"必答题"

音频 3A 算法原理

优先级 P0 — IPC 产品核心竞争力

缺失内容:
① AEC (回声消除) — LMS/NLMS 自适应滤波 · 双讲检测 · 非线性处理 · 延迟估计
② ANS (降噪) — 谱减法 · Wiener 滤波 · 深度学习降噪 (RNNoise)
③ AGC (自动增益) — 模拟增益 vs 数字增益 · 压缩/扩展曲线
④ 音频链路延迟 — ALSA 采集→3A→编码→网络 的端到端延迟分析
⑤ 多路混音 — 多路 PCM 合成、溢出处理
面试频率: 高 — 有音频产品经验的公司必问

编码器深度调优

优先级 P0 — 决定画质/码率/延迟的三角平衡

缺失内容:
① x264/x265 调参 — preset (ultrafast→placebo) · tune (zerolatency/film/grain...) · rc-lookahead · scenecut · keyint · bframes
② 码率控制算法深度 — CBR/VBR/CRF/CQP 的内部实现: QP 映射 · VBV 缓冲模型 · 前瞻分析
③ ROI 编码 — 人脸/运动区域低 QP · 背景高 QP → 感知画质不变, 码率降低 20-40%
④ 低延迟编码 — zerolatency 模式 · 无 B 帧 · 单帧编码延迟
面试频率: 高 — 码率控制和 preset/tune 是标配问题

RTP/RTCP 协议深度

优先级 P0 — 所有流媒体协议的地基

缺失内容:
① RTP 包头 — SSRC/CSRC · PT · Sequence Number · Timestamp (与采样率的关系)
② RTCP SR/RR — 发送端/接收端报告 · NTP→RTP 时间戳映射 · jitter 计算
③ RTCP 反馈 — NACK (重传请求) · PLI (关键帧请求) · REMB/TWCC (带宽估计) · TMMBR
④ JitterBuffer — 自适应去抖 · 丢包补偿 · 帧重排序
面试频率: 高 — RTP 是音视频传输的"共同语言"

🔄 中断处理 — 驱动开发的基石

request_irq → top half → bottom half

嵌入式音视频驱动大量依赖中断: Sensor Vsync(帧同步)、音频 DMA 完成(缓冲切换)、编码器完成(码流就绪)。必须掌握:
① request_irq() vs request_threaded_irq() — 后者自动创建中断线程, handler 在硬中断上下文, thread_fn 在线程上下文
② tasklet (轻量, 不可睡眠) vs workqueue (重量, 可睡眠) — 视频 DMA 完成通常用 tasklet, 编码器回调用 workqueue
③ 中断上下文限制: 不能睡眠、不能调 schedule()、不能用 mutex、GFP_ATOMIC 内存

🔧 CCF + regulator + pinctrl — SoC 三大基础框架

没有这三个, Sensor 根本起不来

摄像头 Sensor 的典型启动依赖链:
① regulator_enable(vana/vdig/vddl) — 按 datasheet 时序逐个上电
② clk_prepare_enable(mclk) — 给 Sensor 提供主时钟 (通常 24MHz)
③ gpiod_set_value(reset, 1) — 拉高复位引脚 → Sensor 开始工作
④ pinctrl — 确保 I2C/MIPI data lane 引脚复用正确
这四步中任何一步配置错误 → Sensor probe 失败 → /dev/video0 不存在

📺 DRM/KMS + ASoC — 显示与音频的"另一半"

V4L2 管采集, 它们管输出

完整的音视频链路 = V4L2(采集) + DRM(显示) + ASoC(音频):
① DRM: Camera→ISP→DMA→GPU纹理→KMS 原子提交→显示。PRIME dmabuf 实现零拷贝。
② ASoC: CPU DAI(音频接口)→codec DAI(编解码芯片)→DAPM(动态电源)。I2S/PCM 时序配置。
③ 面试价值: 能讲清楚 V4L2→DRM 零拷贝预览链路 = 资深水平的标志。

#问题考察点当前能否回答
Q1WebRTC 通话中如果出现卡顿，你会从哪些维度排查？端到端链路思维: 采集→编码→发送→网络→接收→解码→渲染 · 区分是丢包、延迟还是编码瓶颈部分 (有弱网优化知识, 但缺 WebRTC 内核链路)
Q2H.264 编码器中 preset 和 tune 分别控制什么？zerolatency 模式为什么能降低延迟？preset=编码速度/质量平衡 · tune=场景优化 · zerolatency=禁B帧+单帧编码+无前瞻不够 — 缺编码器参数深度理解
Q3音频回声是怎么产生的？AEC 的基本原理是什么？声学回声路径→自适应滤波器→NLMS算法→双讲检测几乎无法回答 — 音频 3A 是空白
Q4GCC (Google Congestion Control) 是如何工作的？基于延迟估计 (Trendline) + 基于丢包估计 → 目标码率 · REMB vs TWCC基本无法回答 — 缺拥塞控制深度
Q5RTP Timestamp 和 NTP Timestamp 的关系？RTCP SR 包的作用？RTP Timestamp=采样率相关 · NTP=绝对时间 · SR=建立两者映射 → 音视频同步的基础部分 — 有同步概念但缺 RTP 深度
Q6如何设计一个支持 1000 路并发观看的直播系统？边缘节点 · CDN 分发 · ABR · 转码 · 录制 · 故障切换部分 — 有 LiveKit 经验但缺大规模架构
Q7DMA-BUF 是什么？如何在 Camera→GPU→Encoder 之间实现零拷贝？dma_buf 框架 · DRM PRIME · V4L2 DMABUF · 跨设备 fd 传递可以回答 (vb2 DMABUF + DMA 驱动深度)
Q8如何测量和优化端到端视频延迟？各阶段插桩 · 帧时间戳追踪 · 延迟预算分配 · 瓶颈定位部分 — 有 e2e 延迟文档但缺实战优化经验
