Simulcast vs SVC — 两种多层方案的对比

两者都解决"不同客户端需要不同质量"的问题，但实现机制完全不同——Simulcast 在编码端、SVC 在码流内部。

Simulcast — 多独立码流

编码器同时输出 N 个独立的 H.264/VP8 码流 · 每个有独立的 SSRC

原理：发送端运行多个独立的编码器实例——如 Encoder#1=1080p@3Mbps, Encoder#2=720p@1.5Mbps, Encoder#3=360p@400Kbps。每个编码器独立输出 RTP 包，各自有独立的 SSRC（同步源标识符）、独立的序列号空间、独立的 RTP 时间戳。三个码流通过同一个 PeerConnection发送——SFU 根据每个接收端的下行带宽选择转发其中一路。
优势：① 编码器简单（标准 H.264/VP8——无需 SVC 扩展）② 独立码流互不依赖——丢了一层不影响其他层 ③ H.264 硬件编码器广泛支持。
代价：① CPU/功耗 ↑ —— N 个编码器 = N 倍编码开销（但硬件编码器可以并行——实际开销 < N 倍）② 上行带宽 = 三层之和——即使没人看最高层，也要发。

SVC — 单码流内分层

H.264 SVC / VP9 SVC / AV1 SVC · 基础层 + 增强层嵌在一个 SSRC 内

原理：编码器输出一个包含多层信息的码流——基础层（Base Layer, 低分辨率, 可独立解码）+ 增强层（Enhancement Layer, 依赖基础层, 提供额外分辨率或质量）。RTP 包中通过 Payload Descriptor 中的 TID/SID/LID（Temporal/Spatial/Quality ID）标识每包属于哪一层。
优势：① 编码效率更高（增强层利用了基础层的信息——同质量下总码率更低）② 上行带宽 = 最高层带宽——SFU 丢弃增强层包即可降级（不增加上行开销）③ 单 SSRC——简化 SFU 的信令管理。
代价：① H.264 SVC 硬件编码器支持极少（几乎都是软件）② VP9 SVC 软件编码器计算量大 ③ 增强层依赖基础层——基础层丢包→增强层全部不可解码。

实际选择：为什么 WebRTC 主流用 Simulcast？
H.264 硬件编码器的普及 + VP8 的 Simulcast 支持成熟 → Simulcast 是最实用的选择。SVC 在理论上更优秀（总码率更低），但 H.264 SVC 硬件编码器几乎不存在（除了少数 Intel/Qualcomm 旗舰芯片），而软件 SVC 的 CPU 开销在嵌入式设备上不可接受。VP9 SVC 在 Chrome 中有良好支持——但 IPC 相机几乎不用 VP9。

RTP 层原理 — SSRC · RID · MID 三要素

Simulcast 在 RTP 层的核心设计：如何标识"这些 SSRC 属于同一个视频源的不同层"——这让 SFU 知道"这几个码流可以互相替代"。

SSRC — 每层独立的同步源

Simulcast 3 层 → 3 个 SSRC · 各自独立序列号空间

每个 Simulcast 层是一个完全独立的 RTP 流——有独立的 SSRC(32-bit 随机数)、独立的 RTP 序列号(16-bit, 0→65535 回绕)、独立的时间戳和独立的 RTCP 报告。从 RTP 协议角度看——三个 SSRC 和三个不同的视频源完全一样。
SSRC 的分配：发送端为每层随机生成一个 SSRC——在 SDP 的 a=ssrc: 属性中声明。SFU 根据 SDP 中的 a=simulcast: 属性知道"这三个 SSRC 是同一源的 Simulcast 层"。

RID — RTP 流标识符

RTP 扩展头 · 1-3 字节 ASCII · 唯一标识层名称

SSRC 可以在通话中变化（SSRC 冲突、重新协商）——SFU 不能仅依赖 SSRC 识别层。RID (RTP Stream Identifier) 是稳定的层标识——嵌入在每个 RTP 包的扩展头中（urn:ietf:params:rtp-hdrext:sdes:rtp-stream-id）。
发送端在 RTP 扩展头中写入当前包所属的 RID（如 "h"=high, "m"=medium, "l"=low）。SFU 据此精确知道"这个 RTP 包属于哪一层"——即使 SSRC 变了，RID 不变——SFU 的转发逻辑不受影响。

MID — 媒体源标识

RTP 扩展头 · 关联 SSRC 到 SDP m= 行

MID (Media Identification) 是更粗粒度的标识——标识这个 RTP 流属于 SDP 中的哪个 m= 行（video=0, audio=1）。Simulcast 的所有层同属一个 MID（都是 video）——但 SFU 在转发时需要用 MID 区分"这是 video 还是 audio"。
RID + MID 的组合让 SFU 可以精确路由：MID（选媒体类型→video）→ RID（选层→high/medium/low）→ SSRC（选具体流）。
SDP 中的声明顺序：RID 在 a=rid: 中声明（send方向），MID 在 a=mid: 中声明。

// RTP 包的 Simulcast 标识 — 两个扩展头字段 // RTP Header (12B) + RID Extension (4-6B) + MID Extension (4B) + Payload RTP Header: SSRC = 0x12345678 // 当前层的 SSRC (可能变化) Seq = 10501 // 该层独立的序列号 RTP Extension (RID): ID = 10 // 在 RTP 扩展头映射中的 ID (SDP 中 a=extmap:10 声明) Data = "h" // RID = "h" → 这是 High 层 RTP Extension (MID): ID = 3 // SDP 中 a=extmap:3 声明 Data = "0" // MID = "0" → 这是第一个 m= 行 (video) // SFU 收到此 RTP 包 → 解析 RID="h" → 知道这是 High 层的包 // → 根据接收端 Client B 的带宽 → Client B 只能收 Low 层 // → SFU 丢弃此包 (不转发给 Client B)

SDP 协商 — Simulcast 的信令开销

Simulcast 需要 SDP 中声明各层的 RID、各层之间的依赖关系、以及 SSRC 到 RID 的映射——SDP 行数从几行膨胀到几十行。

// SDP Offer — 发送端声明 Simulcast 3 层 (high / medium / low) // 1. 声明 RID: 三个层的标识符和方向 a=rid:h send a=rid:m send a=rid:l send // 2. 声明 Simulcast 层组: 这些 RID 是互斥的 (同一视频源的不同质量) a=simulcast:send h;m;l // 含义: 发送端将发送 h/m/l 三个 Simulcast 层, 接收端可以请求其中任意一个或多个 // 3. 每个 RID 的编码参数约束 (可选 — 告知 SFU 每层的特性) a=rid:h pt=96;max-width=1920;max-height=1080;max-fps=30 a=rid:m pt=96;max-width=640;max-height=360;max-fps=30 a=rid:l pt=96;max-width=320;max-height=180;max-fps=15 // 4. 每个 RID 对应的 SSRC (发送端随机分配) a=ssrc:1111111111 cname:video a=ssrc:1111111111 msid:stream1 track1 a=ssrc:1111111111 rid:h // SSRC 1111111111 → RID "h" (High) a=ssrc:2222222222 cname:video a=ssrc:2222222222 msid:stream1 track1 a=ssrc:2222222222 rid:m // SSRC 2222222222 → RID "m" (Medium) a=ssrc:3333333333 cname:video a=ssrc:3333333333 msid:stream1 track1 a=ssrc:3333333333 rid:l // SSRC 3333333333 → RID "l" (Low) // 5. RTP 扩展头映射 (告知接收端 RID 和 MID 在哪个扩展 ID 中) a=extmap:3 urn:ietf:params:rtp-hdrext:sdes:mid a=extmap:10 urn:ietf:params:rtp-hdrext:sdes:rtp-stream-id // SDP Answer — 接收端声明希望接收的层 a=rid:h recv a=rid:m recv a=rid:l recv a=simulcast:recv h;m;l

SFU 转发逻辑 — 如何选择转发哪一层

SFU 的核心价值在 Simulcast 场景中体现：不收发包 payload、不解码、不重编——仅根据网络状态决定"这个 RTP 包转发给哪个客户端"。

SFU 转发决策的三层逻辑

带宽检测 → 层选择 → 包路由

Step 1: 带宽检测：SFU 监控每个下行链路的拥塞状态（通过 RTCP RR 的丢包率 + REMB/Transport-CC 的带宽估计）。Client A 下行畅通 (5Mbps)→可接收 High 层。Client B 下行拥塞 (500Kbps)→只能接收 Low 层。
Step 2: 层选择：SFU 根据带宽估计 + 各层的实际码率决定转发哪层。切换策略：带宽足够→升层（Low→Medium→High）；带宽不足→降层（High→Medium→Low）。必须加滞后 (Hysteresis)——不能带宽刚好过阈值就切→来回震荡。如: 升层阈值 = 目标码率 × 1.2, 降层阈值 = 目标码率 × 0.8。
Step 3: 包路由：收到 RTP 包→解析 RID→查路由表→转发给订阅了该层的客户端。不转发的客户端→直接丢弃该包。

切换时机与用户体验

切换太快=画质震荡·切换太慢=卡顿或带宽浪费

升层策略：当前层码率稳定 < 目标层码率 × 0.7（即当前层明显"喂不饱"带宽）→ 等待 3-5 秒确认（避免临时波动触发切换）→ 升层。
降层策略：当前层丢包率 > 5% 或 Transport-CC 带宽估计急剧下降 → 立即降层（不等确认——拥塞时延迟优先）。通常降层比升层快得多——宁可短暂低画质，不可持续卡顿。
关键帧同步：SFU 切换转发层时——新的目标层可能正在传输 P 帧（而非 I 帧）→ 接收端解码器收到无 I 帧基础的 P 帧→花屏。SFU 需要等待目标层的下一个 I 帧（或主动请求 PLI→触发发送端编一个 I 帧）→ 从该 I 帧开始转发——切换点无缝。但等待 I 帧引入 1-3 秒 GOP 的额外切换延迟。

编码器配置 — 三层参数设计

Simulcast 各层之间的码率比、分辨率比、帧率选择直接影响总编码开销和用户体验——需要精心设计而非随意分配。

典型 3 层配置 (1080p IPC)

三层码率比 ≈ 8:3:1 · 满足绝大多数客户端

High (h)：1080p@30fps, 2.0~3.0 Mbps
Medium (m)：640×360@30fps, 0.6~1.0 Mbps。分辨率 = High 的 1/3 (每维)——画质可接受。
Low (l)：320×180@15fps, 0.15~0.3 Mbps。保证在极弱网络 (3G/弱 WiFi) 下至少能看到画面。
码率比 8:3:1 是经验最优——如果差异太小，层间切换无意义（两层都差不多）；差异太大，Low 层画质太差。

2 层配置 (720p IPC, 资源受限)

仅 High + Low · 码率比 ≈ 5:1

High (h)：720p@30fps, 1.0~1.5 Mbps
Low (l)：320×180@15fps, 0.2~0.3 Mbps
2 层配置适合编码能力有限的嵌入式设备——只运行 2 个硬件编码器。大多数场景只有 2 类客户端：WiFi 高质量 + 4G 低质量。第三层 Medium 的使用率极低——可以被 High 或 Low 覆盖。

编码开销的优化

硬件编码器共享参考帧 · Low 层降帧率

Low 层降帧率：15fps vs 30fps——编码时间减半、码率减半——对弱网客户端完全够用。降帧率比降分辨率更有效降低编码开销。
共享 motion vector（部分硬件编码器支持）：High 层和 Medium 层对同一帧做运动估计时，Medium 层可以复用 High 层的 MV 结果——减少计算量。
编码器优先级：High 层 > Medium 层 > Low 层。CPU 不足时——先降 Low 层帧率（15→10fps），再降 Medium 层（30→20fps），保证 High 层质量。

⚠ 各层之间的 GOP 对齐 — 关键帧必须同时出现
如果 High 层的 I 帧在 seq=50，Low 层的 I 帧在 seq=52——当 SFU 从 High 切换到 Low 时（seq=51 时刻）→ Low 层已经过了 I 帧→必须等到 seq=52 才能转发→切换延迟增加 2 秒（如果 GOP=2s）。
解决：所有层的编码器共享同一个 GOP 节奏——当任一层的 I 帧到来时，其他层强制编 I 帧(通过 force_key_frame())。这样 SFU 在任意时刻切换→目标层在下一个 GOP 周期 (≤2s) 内必然出现 I 帧。

LinkricAI 通过 LiveKit SDK 发布 Simulcast 视频——配置集中在 RtpEncodingParameters 中。

LiveKit 配置 Simulcast (发送端)

RtpEncodingParameters 数组 — 每层一个 entry
// LiveKit VideoTrack 发布 Simulcast 3 层 auto track = LocalVideoTrack::create(); std::vector<RtpEncodingParameters> layers = { // Low: 320×180, 15fps, 200Kbps { .rid = "l", .maxBitrate = 200000, .scaleResolutionDownBy = 6.0, .maxFramerate = 15, .active = true }, // Medium: 640×360, 30fps, 800Kbps { .rid = "m", .maxBitrate = 800000, .scaleResolutionDownBy = 3.0, .maxFramerate = 30, .active = true }, // High: 1080p, 30fps, 2.5Mbps (原始分辨率) { .rid = "h", .maxBitrate = 2500000, .scaleResolutionDownBy = 1.0, .maxFramerate = 30, .active = true }, }; track->publish(layers);

IPC 场景的 Simulcast 最佳实践

2 层足够 · 硬件编码器 · H.264 only

推荐 2 层 (不推荐 3 层)：IPC 相机硬件编码器通常只支持 2 路并发（主码流 + 子码流）。3 层需要 3 个硬件编码器——多数 SoC 不支持（Rockchip 的 MPP 可支持 2 路）。
主码流 (High)：1080p@30fps, 1.5~3Mbps——WiFi/LAN 客户端使用。
子码流 (Low)：640×360@15fps, 200~400Kbps——4G/弱网/AI 检测使用。
码率配置建议：Low 层码率 = 真正常用弱网场景的下行带宽（200Kbps 足够在 4G 下稳定传输）。不要设太低的 Low 层（64Kbps 画质不可用——AI 检测无法识别人脸）。
与 Cloud Recording 的互动：LiveKit Egress 录制时——默认录制 High 层（保证录像质量）。Composite 录制时——SFU 将所有客户端的视频流合并为一个布局——每路视频的 Simulcast 层由 SFU 自动选择。
