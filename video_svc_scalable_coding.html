一次编码, 多层输出

编一次 → 多方按需取

SVC 编码器只编一次，生成含多层的单一码流。SFU/路由器可选择性丢弃高层 NAL 单元，低带宽接收方只收基层 → 无需转码。这是 WebRTC 多方会议的核心竞争力。

Simulcast vs SVC

多流独立编码 vs 单流层级编码

Simulcast: 编码器同时编 3 路独立流 (720p/360p/180p) → 发布端 CPU 高 · 上行带宽大。SVC: 一个流含 3 层 → 发布端 CPU 低 · 上行带宽小。但硬件解码兼容性 Simulcast 更好。

WebRTC 中的应用

VP9 SVC + AV1 SVC

Chrome 支持 VP9 SVC (L1T3~L3T3)。SFU 根据订阅方带宽选择性转发层 → 720p 给桌面端, 180p 给移动端 — 同一发布流服务所有订阅方。2024 年 AV1 SVC 逐步成熟。

1. SVC 三轴可伸缩性 — 核心概念

Temporal (帧率) · Spatial (分辨率) · Quality/SNR (画质) — 三者可任意组合

时域可伸缩 (Temporal)

帧率分层: 30fps → 15fps → 7.5fps

原理: 通过层级 B 帧 预测结构实现。基层 (T0) = I/P 帧, 增强层 (T1/T2) = B 帧, 参考 T0。
实现: 每个 NAL 单元带 temporal_id (TID)。丢弃 TID ≥ N 的所有 NAL → 帧率降为 1/2^N。
开销: 几乎零额外开销 — H.264/VP8 本身已支持时域分层。

空域可伸缩 (Spatial)

分辨率分层: 1080p → 720p → 360p

原理: 基层编码低分辨率, 增强层编码差值 (本层原始 - 基层上采样)。
层间预测: 增强层复用基层的运动矢量+模式 (Inter-Layer Motion Prediction) + 残差预测。
开销: ~10-20% 额外码率 (vs 单独编码各分辨率)。典型二倍下采样 (dyadic)。

质量可伸缩 (SNR/Quality)

相同分辨率下的画质分层

CGS (粗粒度): 不同层用不同 QP — 只能在 IDR 切换层。
MGS (中粒度): 可在任意帧切换 — 通过关键帧 (K-picture) 限制漂移。最多 15 个 MGS 质量级。
原理: 增强层 = 基层量化残差的精细编码 → 迭代还原被粗量化的高频细节。

2. H.264 SVC (Annex G) — 经典标准详解

ITU-T + ISO/IEC 联合标准 · NAL 扩展头 · dependency_id · quality_id · temporal_id · 层间预测三模式

NAL 单元扩展头部 (3 字节)

比 AVC 基础 NAL 头多 3 字节的层标识

dependency_id (D, 3-bit): 空域/CGS 依赖层。D=0 为基层 (兼容 AVC), D 递增。
quality_id (Q, 4-bit): MGS 质量细化层。每 D 层可有 0~15 个 Q 子层。
temporal_id (T, 3-bit): 时域层。T=0 为基层帧率, T 递增。
priority_id (P, 6-bit): 优先级 — 网络节点据此决定丢包顺序。
Access Unit: 同一时刻的所有 D/Q/T NAL 聚合 → 解码器从中提取目标层。

层间预测 — 三种模式

增强层复用基层信息 → 大幅提升编码效率

① Inter-Layer Intra Prediction: 增强层 MB 从基层上采样重建块 预测 (空间域)。
② Inter-Layer Motion Prediction: 增强层继承基层运动矢量 + 分区模式 → 无需重新做运动估计。
③ Inter-Layer Residual Prediction: 增强层残差 = 实际残差 - 上采样的基层残差 → 进一步压缩。
单循环解码: SVC 设计目标 — 解码器只需在目标层做一次运动补偿, 保持复杂度可控。

SVC Profile对应 AVC Profile适用场景
Scalable BaselineBaseline视频通话、移动、监控
Scalable HighHigh广播、流媒体、存储

3. VP9 SVC — WebRTC 的 SVC 主力

Google 的设计哲学: 简化 SVC · 单码流多层级 · Chrome 原生支持 · SFU 选择性转发

LxTy 层级命名

L=空域层数, T=时域层数

L1T3: 1 空域 × 3 时域 (仅帧率可伸缩)
L2T3: 2 空域 × 3 时域 (2种分辨率+帧率)
L3T3: ★ 最常用 — 3 空域 (720p/360p/180p) × 3 时域 (30/15/7.5fps) → 9 种组合
L3T3_KEY: keyframe 为同步点 — 切换更快
Chrome 通过 scalabilityMode 字段 (如 "L3T3_KEY") 配置。

层间参考 — On / Off / OnKeyPic

控制高层是否依赖低层做预测

Off: 层间独立 — 每层独立编码, 类似 Simulcast 但共享码流。
On: 层间预测 — 高层复用低层的运动矢量+残差 → 额外节省 10-20% 码率。
OnKeyPic: 仅 keyframe 做层间预测 → 兼顾效率与随机访问。
Chroma: VP9 4:2:0 — 层间色度预测是 SVC 效率的关键之一 (vs Simulcast 重复编码色度)。

WebRTC 集成

Chrome 标志 + SFU 配合

启用: --force-fieldtrials=WebRTC-SupportVP9SVC/EnabledByFlag_3SL3TL/
RTCRtpEncodingParameters.scalabilityMode: "L3T3_KEY"
SFU 职责: 检查每个 RTP 包的 RTP 头扩展 (frame-marking / scalability structure) → 根据订阅方带宽丢弃高层包 → 只转发低层。
关键优势: 发布端只编一次 → CPU+带宽节省 ~30% vs Simulcast 3 路。

⚠️ VP9 SVC 的限制
硬件解码: 即使设备支持 VP9 硬解, 不一定支持 VP9 SVC 码流。Chrome 可能退回到软解 → 高分辨率 (1080p) 时 CPU 过载。这是 Simulcast + H.264 硬解的最大优势 — H.264 硬解几乎通用。
Firefox 不支持 SVC — 只能用 Simulcast。
Safari 15+ 支持 VP9 但不支持 SVC。

4. SHVC (H.265) 与 AV1 SVC

SHVC = H.265 Annex H — 比 H.264 SVC 更高效 · AV1 SVC — 下一代免版税编码的可伸缩扩展

SHVC (Scalable HEVC)

H.265 Annex H — 2014 标准化

核心改进: ① SHVC 的基层可以用不同的编码标准 — 如 H.264 基层 + H.265 增强层 (混合编码) ② 利用 HEVC 本身的 CTU 四叉树 + 更大块 → 更灵活的分区 ③ 支持 Bit-depth Scalability (8-bit 基层 + 10-bit 增强层) ④ 支持 Color Gamut Scalability (BT.709 基层 + BT.2020 增强层)。
现状: SHVC 未被广泛部署 — H.265 专利费问题使大多数 WebRTC 实现选择了 VP9/AV1 路线。

AV1 SVC — 免版税的未来

AOMedia — 2022+ 标准化

AV1 原生支持: 时域 + 空域可伸缩 — 不需要像 H.264 那样的 Annex G 扩展。
关键设计: ① spatial_layer_id 和 temporal_layer_id 内置在 OBU (Open Bitstream Unit) 头中 ② 支持参考帧缩放 (Reference Frame Scaling) — 不同分辨率的参考帧可互相引用, 无需显式上采样 ③ Loop Restoration + CDEF 滤波 对多层编码质量尤其关键。
WebRTC 进展: Chrome M113+ 支持 AV1 SVC (via scalabilityMode: "L3T3_KEY")。编码计算量大但效率优于 VP9 SVC ~20%。

5. Simulcast vs SVC — 选型决策

两者都是解决"多方不同带宽"的方案 — 但实现路径完全不同 · 一张表看清所有差异

维度SimulcastSVC
工作原理编码器同时编 3 路独立流 (如 720p/360p/180p)编码器只编 1 路含多层 — SFU 丢弃高层 NAL
发布端 CPU高 — 3× 编码复杂度中 — 1× + 层间预测开销 (~10-20%)
发布端上行带宽高 — 3 路独立码率之和低 — 1 路含所有层的码率
SFU 复杂度低 — 直接选一路转发中 — 需解析 RTP 扩展 + 选择性丢弃 NAL
硬件解码兼容性极好 — H.264 硬解几乎通用差 — 多数硬解码器不支持 SVC 码流
浏览器支持所有浏览器 (H.264/VP8)Chrome only (VP9/AV1 SVC)
每流编码器数量3 个独立 encoder 实例1 个 encoder 实例
订阅方切换延迟可即时切换 (每流独立)需要等 keyframe 同步
编码标准H.264 / VP8 / H.265 / AV1H.264 SVC / VP9 / AV1 / SHVC
适用场景兼容性优先 · 有硬件编码 · 参与者少带宽受限发布端 · 大规模多方会议 · CPU 有裕量
实战决策: 什么时候用 SVC, 什么时候用 Simulcast?
用 Simulcast: ① 移动端发布 (依赖 H.264 硬编, 不支持 SVC) ② 需要 Safari/Firefox 兼容 ③ 发布端上行带宽充裕 (WiFi)。
用 SVC: ① 桌面端发布 (CPU 充裕) ② 发布端上行受限 (4G/弱网) ③ 大型多方会议 (大量订阅方 → SVC 的发布端节省 × N 订阅方) ④ 追求带宽效率极致优化。

浏览器端 WebRTC API · FFmpeg 命令行 · x264 SVC 参数

/* ===== WebRTC — VP9 SVC 配置 (JavaScript) ===== */ const pc = new RTCPeerConnection(); // VP9 SVC — 3 空域 × 3 时域 const transceiver = pc.addTransceiver('video', { sendEncodings: [ { rid: 'h', scalabilityMode: 'L3T3_KEY', maxBitrate: 2500000 }, // 720p { rid: 'm', scalabilityMode: 'L3T3_KEY', maxBitrate: 800000 }, // 360p { rid: 'l', scalabilityMode: 'L3T3_KEY', maxBitrate: 200000 }, // 180p ] }); // AV1 SVC (Chrome M113+) const tc2 = pc.addTransceiver('video', { sendEncodings: [ { scalabilityMode: 'L3T3_KEY', maxBitrate: 2500000 }, ] }); /* ===== FFmpeg — libx264 SVC 编码 ===== */ # 时域可伸缩 (3 temporal layers) ffmpeg -i input.mp4 -c:v libx264 -x264-params "temporal-layers=3" output.mp4 /* ===== x264 命令行 — 空域 SVC (需 OpenCL) ===== */ # H.264 SVC 空域可伸缩 (支持的硬件/编译选项较少) x264 --bitrate 5000 --svc "2,640x360,1280x720" -o output.264 input.y4m

🔧 面试要点: 能说清楚 SVC vs Simulcast 在 WebRTC 中的权衡
SVC 的"甜区": 发布端上行受限 + 订阅方数 > 3 → SVC 的总上行节省 × N 订阅方 ≥ 发布端 CPU 开销。
Simulcast 的"甜区": 有硬件编码 (H.264 HW Enc) + 需兼容所有浏览器/设备 → Simulcast 是安全选择。
混合部署: 许多 SFU 支持 Simulcast 和 SVC 混合 — 桌面端用 VP9 SVC, 移动端用 H.264 Simulcast。
