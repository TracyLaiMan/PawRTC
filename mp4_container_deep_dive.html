MP4 容器格式深度解析

MP4 不是编码器，而是容器——定义数据如何组织存储，不定义视频/音频如何压缩。
从 Box 二进制结构到 Sample Table 寻帧算法，从普通 MP4 到 fMP4 流式封装——逐层拆解每一个字节。
基于 ISO 14496-12/14/15 标准，结合 T31/SSD20x 平台实战。

↓

容器 ≠ 编码器

理解三层模型的分离是学习 MP4 的第一步。容器管「怎么存」，编码管「怎么压」，像素管「怎么画」。

容器层 MP4/MKV/MOV

定义数据组织结构

Box 树、轨道布局、索引表、时间映射。
同一个 H.264 码流可以放进 MP4、MKV 或 AVI，容器不同但编码相同。

编码层 H.264/H.265/AAC

定义压缩算法

帧内/帧间预测、DCT变换、CABAC熵编码。
同一个 MP4 容器可以装 H.264 也可以装 H.265，编码不同但容器相同。

像素层 YUV420/RGB24

定义原始图像数据

色彩空间、位深、分辨率。
编码器输入 YUV，输出压缩码流；解码器输入码流，输出 YUV——两端都是像素层。

为什么这个区分至关重要？
很多音频画质问题（花屏、不同步、绿屏）表面看是「MP4 坏了」，实际可能是编码层（参考帧丢失）或容器层（stbl 索引错乱）的问题。诊断时要分层定位。
↓

Box/Atom 二进制架构

MP4 的所有数据存储在 Box（也叫 Atom）中。Box 递归嵌套形成树，每个 Box 的头部格式统一。
理解 Box 头部的三种变体是读懂 MP4 二进制的基础。

基本 Box 8 字节

最常用

size(4B) + type(4B)
size 含头部本身。值=0 表示到文件尾，值=1 表示用 largesize。

扩展 Box 16 字节

文件 >4GB

size=1(4B) + type(4B) + largesize(8B)
FAT32 单文件 4GB 限制时可触发此场景。

FullBox 12 字节

带版本/标志

size + type + version(1B) + flags(3B)
version 影响 uint32/uint64 字段宽度。mvhd、stts 等都用 FullBox。

moov — 播放的灵魂

moov 是 MP4 最关键的 Box，包含所有播放所需的元数据：全局头、轨道定义、索引表、编辑列表。
没有 moov，mdat 中的数据无法被解析。这是 Web 播放要求 faststart 的原因。

FastStart：为什么 moov 必须在 mdat 前面？
实时写入时，moov 依赖完整的 stbl 数据，只能在最后写入 → moov 在文件末尾 → Web 播放器必须下载整个文件才能播放。
解决方案：qt-faststart 或 FFmpeg -movflags +faststart 将 moov 前移。本质是重写 stco/co64 的偏移值。
T31 录像回放必须做 faststart，否则 Web 端无法秒开。
↓

trak — 视频 vs 音频轨道

每个 trak 描述一路媒体流。hdlr 区分轨道类型，stbl 定位每一帧数据。视频轨和音频轨的关键差异体现在时间基准和编码配置上。

视频轨道 vide

hdlr.handler_type = vide

tkhd: track_id=1, width=1920, height=1080
mdhd: timescale=24000 (24fps千倍)
vmhd: 视频媒体头，图形模式
stsd: avc1 + avcC ★ (H.264配置)
stts: delta=1000 @30000ts → 30fps
stss: 关键帧间隔=GOP=15~30
stsc: 每chunk 1帧(交错排列)
ctts: B帧时DTS/PTS偏移(IPPP=空)

音频轨道 soun

hdlr.handler_type = soun

tkhd: track_id=2, volume=0x0100
mdhd: timescale=48000 (采样率)
smhd: 音频媒体头，平衡=0
stsd: mp4a + esds ★ (AAC配置)
stts: delta=1024 @48000ts → AAC帧
stss: 不存在(音频帧独立)
stsc: 每chunk 5帧(减少索引)
ctts: 不存在(无B帧概念)

timescale 的关键作用
duration / timescale = 秒数。视频轨和音频轨各自有 mdhd.timescale，而 moov 级别的 mvhd.timescale 统管全局。
T31 IMP 默认 mvhd.timescale=90000，视频 mdhd.timescale=24000，音频 mdhd.timescale=48000——三层时钟各司其职。

stbl — 六表联合寻帧

stbl 是 MP4 的索引引擎，由 6~8 个叶子 Box 联合工作，将「时间点」映射到「文件偏移」。
这是理解 MP4 的核心——播放、seek、转码都依赖这组索引表。

Box全称功能必需
stsdSample Description编解码器配置参数是
sttsTime-to-Sample时长→帧号映射（变帧率）是
stssSync Sample关键帧编号列表否*
stscSample-to-Chunk帧号→Chunk号分段映射是
stszSample Size每帧字节数是
stcoChunk Offset 32bit每个Chunk文件偏移是**
co64Chunk Offset 64bit同stco，偏移>4GB是**
cttsComposition Time OffsetDTS→PTS偏移(B帧)否***

* 全I帧可省略 ** stco与co64二选一 *** 含B帧时必需；T31 IPPP结构可省略

mdat 与 NAL 格式转换

mdat 占文件 99%+ 的空间，但内部无任何结构——解析依赖 moov 中的 stbl 索引。
H.264 在 MP4 中使用 AVCC 格式（4字节长度前缀），WebRTC/RTP 需要 Annex B 格式（起始码）。这是 LiveKit 推流的必经转换。

踩坑：AVCC vs Annex B
MP4 内部：4字节长度 + NAL数据（AVCC格式），SPS/PPS 只存在于 stsd 的 avcC 中
WebRTC/RTP：00 00 00 01 起始码 + NAL数据（Annex B格式），IDR帧前必须插入 SPS+PPS
转换步骤：① 替换4字节长度为起始码 ② 从 avcC 展开 SPS/PPS 插入 IDR帧前 ③ 处理 STAP-A 分片
这个坑在之前 LiveKit 推流开发中已经踩过。

fMP4 — Fragmented MP4

Fragmented MP4 将大 mdat 切分为多个 fragment，每个 fragment 自包含索引，支持边产生边传输。
这是 MSE、DASH、HLS 和 LiveKit 推流的基础格式。

普通 MP4 痛点

无法实时流式传输

moov 必须在所有帧编码完成后才能写入（依赖完整 stbl 数据），无法边编码边传输。Web 播放要求 moov 在 mdat 前（faststart），但实时场景根本不可能。

fMP4 解决方案

编码器每产生一组帧就立即封装

init segment = ftyp + moov（仅含编解码参数，不含帧索引）
media segment = moof + mdat（每帧或每GOP一个fragment）
trun 一次性替代 stts+stsz+stco+stsc+ctts 五张表！

IMP采集 → H.264编码 → fMP4封装 → LiveKit SFU → 浏览器播放

avcC / esds — 编解码配置

stsd 中的 avcC 和 esds 是解码器启动的必需参数。播放器在解码第一帧之前，必须从这里读取 SPS/PPS（视频）或 AudioSpecificConfig（音频）。

avcC — H.264 配置 ★

解码器启动必需

configurationVersion = 1
AVCProfileIndication: 66=Baseline, 77=Main, 100=High
AVCLevelIndication: 31=L3.1, 40=L4.0
lengthSizeMinusOne: 通常=3 → NAL长度字段4字节(AVCC格式)
numOfSequenceParameterSets: SPS数量
sequenceParameterSetNALU[]: [2B长度][SPS数据]
numOfPictureParameterSets: PPS数量
pictureParameterSetNALU[]: [2B长度][PPS数据]

T31 IMP 输出
Profile=High(100), Level=4.0(40), lengthSizeMinusOne=3 → 转 Annex B 时替换4字节长度→00 00 00 01

esds — AAC 配置 ★

音频解码器启动必需

内嵌 ES_Descriptor → DecoderConfigDescriptor → AudioSpecificConfig

audioObjectType: 2=AAC-LC
samplingFreqIndex: 3=48000Hz, 4=44100Hz
channelConfiguration: 1=mono, 2=stereo

AAC 每帧固定 1024 采样 → stts delta = 1024 @48000ts
Opus 则每帧 960 采样 @48000ts

ftyp — 文件类型与版本兼容

ftyp 是 MP4 文件的第一个 Box（强制要求），告诉解析器这个文件遵循哪个版本的规范、兼容哪些播放器。
很多「文件损坏」其实只是 ftyp 写错了。

Brand含义适用场景
isomISO Base Media File Format基础兼容标识，所有 MP4 都应列出
mp41MP4 v1 (ISO 14496-1)早期 QuickTime 兼容
mp42MP4 v2 (ISO 14496-14)含 AVC/H.264 的标准 MP4
avc1H.264 Main/High Profile主流视频文件
iso2ISO BMFF 2nd edition支持 64bit 时间戳
M4V iTunes VideoApple 视频，含 DRM 元数据
M4A iTunes Audio纯音频 MP4
qt QuickTime MovieMOV 文件标识
兼容链的判断规则
播放器检查 major_brand 是否支持，然后逐一检查 compatible_brands[]。
只要其中任意一个品牌被播放器识别，就可以播放。典型写法：avc1isomiso2mp41 — 意味着兼容 AVC 解码器、ISO 基础格式、ISO2 扩展和 MP4v1。
T31 IMP 录像输出 isomiso2avc1mp41，浏览器可正常识别。

stts / stsc 深度拆解 — 游程编码与反查

stts 和 stsc 都使用游程编码 (Run-Length Encoding) 压缩索引表。理解编码结构和反查算法是掌握寻帧的关键。

stts 游程编码 ★

Time-to-Sample 表

每个 entry：{sample_count, sample_delta}
含义：连续 count 个帧，每帧 delta 个 ticks

恒定帧率：1 条 entry
{900, 1000} → 900帧，每帧1000ticks

变帧率：多条 entry
{15, 1000} {1, 1333} {14, 1000}
→ 前15帧每帧1000t，第16帧1333t(3:2 pulldown)

反查算法 — 累加法
给定目标 ticks T，找不到精确帧号需要累加：
ticks_acc = 0; sample = 0;
for each (cnt, delta):
if (ticks_acc + cnt*delta > T):
sample += (T - ticks_acc) / delta;
break;
ticks_acc += cnt*delta; sample += cnt;

stsc 游程编码 ★★

Sample-to-Chunk 映射表

每个 entry：{first_chunk, samples_per_chunk, sample_desc_idx}
含义：从 first_chunk 开始，每个 chunk 有 samples_per_chunk 个帧

最简单：每 chunk 1帧(vbr交错)
{1, 1, 1} → 所有 chunk 各1帧

音频典型：每 chunk 5帧
{1, 5, 1} → 所有 chunk 各5帧

复杂场景：I/P帧混合写入
{1, 1, 1} {16, 2, 1} {31, 1, 1}

反查算法 — 逆推法（最复杂的部分）
正查 chunk→sample 很简单，但帧号→chunk号需要逆推：
given target_sample:
for i = entries.length-1 downto 0:
if target_sample >= entry[i].first_sample:
chunk = entry[i].first_chunk
+ (target - entry[i].first_sample) / spc
offset_in_chunk = (...) % spc
break;

edts / elst — 编辑列表与 A/V 同步

Edit List 定义媒体时间线与展示时间线之间的映射关系。
它是 A/V 不同步问题的常见根因，但很多播放器直接忽略了它。

三种 Edit 类型

elst entry 字段

media_rate = 0 → Empty Edit（空编辑）
插入静音/黑帧延迟，常用于音频提前对齐

media_rate = 1 → Normal Edit（正常编辑）
从 media_time 开始播放 segment_duration 时长

media_rate ≠ 0,1 → Dwell Edit（驻留编辑）
冻结某一帧，慢放效果

典型场景：iOS 录像

音频轨道常见 elst

iOS 录制的 MOV/MP4 通常在音频轨插入 Empty Edit：
version=1, flags=0
{segment_duration=480, media_time=-1, rate=0} → 空：延迟480ticks
{segment_duration=xxx, media_time=0, rate=1} → 正常播放

原因：视频首帧是 I 帧需要额外解码时间，音频提前480ticks 开始会导致唇形不同步。
不处理 elst → 音频提前 → 唇形不同步

播放器兼容性
很多嵌入式播放器（包括 IMP VDEC）直接忽略 edts，导致 iOS 录像回放时 A/V 不同步。
解决方案：在 MP4 封装阶段处理 elst，将延迟体现到 ctts 或帧时间戳中，而非依赖播放器解析。

FastStart — stco 偏移重定位算法

将 moov 从文件末尾移到 ftyp 之后，核心操作是修正 stco/co64 中所有 chunk 偏移值。
偏移量 = moov_size，方向 +1（向后移动）。如果 moov 本来在前面则不需要处理。

// FastStart 核心算法：修正 stco 偏移 void fix_stco_offsets(uint8_t *moov, uint64_t moov_size) { BoxIter it = find_box(moov, "stco"); // 递归搜索 stco if (!it.found) it = find_box(moov, "co64"); // 64bit 版本 uint32_t count = read_u32(it.data + 4); // entry_count for (uint32_t i = 0; i < count; i++) { if (it.type == "stco") { uint32_t off = read_u32(it.data + 8 + i*4); write_u32(it.data + 8 + i*4, off + moov_size); } else { // co64 uint64_t off = read_u64(it.data + 8 + i*8); write_u64(it.data + 8 + i*8, off + moov_size); } } // 同理修正 mfra/tfra 等随机访问相关的偏移 }
T31 实战注意事项
① 录像过程中 moov 还没写入，无法做 faststart → 录像结束后异步执行
② moov_size 在读取时已知，修正只是 +moov_size，无需遍历 mdat
③ exFAT 下文件可以 >4GB，但 stco 偏移必须升级为 co64（8字节/条目），否则溢出
④ FFmpeg 的 -movflags +faststart 会在写完临时文件后原地重写，嵌入式场景需要注意 Flash 寿命

Box 解析实战 — C 语言实现

理解了 Box 结构后，用 C 代码实际解析是嵌入式开发的基础技能。以下是递归式 Box 解析器的核心框架。

// MP4 Box 头部结构 typedef struct { uint32_t size; // Box 总大小 (含头部) uint32_t type; // 四字符类型标识 如 'moov' uint64_t largesize; // 仅当 size==1 时使用 } BoxHeader; // 递归解析 Box void parse_box(FILE *fp, uint64_t end, int depth) { while (ftell(fp) < end) { uint32_t sz = read_u32(fp); // 读取 size uint32_t tp = read_u32(fp); // 读取 type uint64_t box_end = ftell(fp) - 8 + sz; if (sz == 1) { // 扩展大小 box_end = ftell(fp) - 8 + read_u64(fp); } else if (sz == 0) { // 到文件尾 box_end = file_size(fp); } char name[5] = {tp>>24, tp>>16, tp>>8, tp, 0}; print_indent(depth); printf("[%s] size=%lu", name, box_end - (ftell(fp)-8)); // 容器 Box — 递归进入 if (is_container(tp)) { // moov/trak/mdia/minf/stbl/... printf(" →"); parse_box(fp, box_end, depth + 1); } else { // 叶子 Box — 解析具体字段 parse_leaf(fp, tp, box_end, depth); } fseek(fp, box_end, SEEK_SET); // 跳到下一个 Box } }
