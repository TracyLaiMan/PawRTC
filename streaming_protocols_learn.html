RTSP — 实时流协议

Real Time Streaming Protocol，1996 年由 RealNetworks/Netscape/Columbia 大学联合提出，RFC 2326。安防监控行业的事实标准

协议本质

信令 + 媒体分离 — TCP 控制 + UDP/RTP 传输

RTSP 是一个会话控制协议，类似于"网络遥控器"——负责控制媒体的播放/暂停/停止/seek，但不传输媒体数据本身。媒体数据通过独立的 RTP/RTCP 通道传输（通常是 UDP）。默认端口 554。支持 TCP 和 UDP 两种传输模式：UDP 低延迟但可能被防火墙拦截；TCP 穿透性好但延迟略高。

核心方法

OPTIONS · DESCRIBE · SETUP · PLAY · TEARDOWN

OPTIONS：查询服务器支持的方法。DESCRIBE：获取媒体描述（SDP）。SETUP：协商传输参数（RTP/RTCP 端口、传输模式）。PLAY：开始/恢复播放（可指定 Range 实现 Seek）。PAUSE：暂停。TEARDOWN：结束会话释放资源。GET_PARAMETER / SET_PARAMETER：扩展方法，厂商常用于 PTZ 控制等。

优势

安防行业无处不在

• 兼容性极强——几乎所有 IPC/NVR/VMS 都支持
• 会话控制完善——PLAY/PAUSE/SEEK/PTZ
• 延迟可控——UDP 模式下 200-500ms
• 标准化 SDP——媒体参数协商清晰
• ONVIF 标配——安防互操作性基础

劣势

非为互联网设计

• NAT 穿透困难——UDP/RTP 多端口映射复杂
• 无原生自适应码率——需扩展实现
• 文本协议——解析效率低于二进制协议
• 无内置加密——需 RTSPS (TLS 隧道)
• 防火墙不友好——多端口动态分配

典型场景

IPC · NVR · VMS

• IPC 相机直连预览（局域网）
• NVR 取流录像
• VMS 平台接入（通过 ONVIF）
• 第三方 AI 盒子取流分析
• 不适合：公网低延迟直播、浏览器播放（需插件）

RTSP 交互流程（典型 IPC 取流）

RTMP — 实时消息传递协议

Real Time Messaging Protocol，Adobe 2002 年推出。直播行业的推流标配，至今仍是 OBS/FFmpeg 的默认输出格式

协议本质

TCP 长连接 + 二进制分块 + 多路复用

RTMP 基于 TCP（端口 1935），建立持久连接后将音视频数据拆分为 Chunk 传输。单个连接可多路复用多个逻辑流（视频/音频/数据/控制）。变体：RTMPS（TLS 加密）、RTMPE（Adobe 私有加密）、RTMPT（HTTP 隧道穿透防火墙）、RTMFP（UDP 变体，Flash P2P）。

握手与连接

三次握手 + 简单握手 + 复杂握手

TCP 三次握手后，RTMP 有自己的应用层握手：C0/C1/C2（客户端）和 S0/S1/S2（服务端）共 6 个包交换协议版本、时间戳、随机数。Flash Player 时代还有复杂握手（加密验证）。握手完成后进入Connect → CreateStream → Publish/Play 流程。推流端 Publish，拉流端 Play。

优势

直播推流的不二之选

• 延迟低——TCP 模式下 1-3 秒
• 生态成熟——OBS/FFmpeg/nginx-rtmp 全支持
• 推流稳定——TCP 可靠传输保证不丢帧
• 多路复用——单连接传输音视频
• CDN 支持——各 CDN 均有 RTMP 接入点

劣势

Flash 已死，浏览器不支持

• 浏览器端已死——Flash EOL 2020，H5 无法直接播放
• TCP 队头阻塞——丢包重传导致延迟抖动
• 无自适应码率——需扩展或转 HLS/DASH
• 专利风险——Adobe 曾收许可费
• 不适合弱网——TCP 在 4G/WiFi 高丢包场景表现差

典型推拉流架构

RTMP 推 → 服务端转码 → HLS/FLV 拉

现代直播架构：OBS/FFmpeg → RTMP 推流到 nginx-rtmp/SRS → 服务端实时转封装为 HLS/HTTP-FLV/WebRTC → 播放端通过这些协议拉流。RTMP 的角色已从"端到端协议"演变为"推流接入协议"——第一公里用 RTMP，最后一公里用 HLS/WebRTC。

⚠ RTMP 在 IPC 场景的定位
安防 IPC 几乎不使用 RTMP——RTSP 是行业标准。RTMP 更适合直播推流场景（主播推流到 CDN）。IPC 偶尔用 RTMP 推流到 YouTube Live/阿里云直播等云平台做公网直播，但不用于本地监控。如果 IPC 需要公网低延迟推流，WebRTC 是比 RTMP 更好的选择（延迟 0.1s vs 1-3s，且浏览器原生支持）。

HLS — HTTP Live Streaming

Apple 2009 年推出，基于 HTTP 的自适应码率流媒体协议。浏览器/移动端的通用播放标准

HLS

核心原理：切片 + 索引

m3u8 播放列表 → .ts 分片文件 → 自适应码率切换

HLS 将整个视频流切分为小的 .ts（MPEG-TS）文件片段（通常 2-10 秒），通过 HTTP Server 分发。客户端周期性下载 .m3u8 播放列表（索引文件），按序下载并播放 ts 分片。自适应码率：服务端提供多档码率的 m3u8（如 360p/720p/1080p），客户端根据网络带宽动态切换码率——带宽下降时自动切到低码率，恢复后切回高码率。这是 HLS 最核心的竞争力。

优势

互联网最兼容的流媒体协议

• 通用性第一——所有浏览器/手机/机顶盒原生支持
• CDN 友好——标准 HTTP，走 CDN 缓存零成本
• 防火墙友好——HTTPS 443 端口，无穿透问题
• 自适应码率——ABR 是多客户端场景的刚需
• DRM 支持——FairPlay/AES-128 加密

劣势

延迟是硬伤

• 高延迟——典型 10-30 秒（3 个分片 × 6 秒/片）
• 低延迟 HLS (LL-HLS)——分片 2 秒 + 部分分片，延迟 ~3-5s
• 存储碎片——大量 .ts 小文件（HLS fMP4 改善）
• 不适合双向通信——纯拉流，无对讲/信令能力
• 不适合实时监控——10s+ 延迟在安防场景不可接受

m3u8 播放列表结构
# master.m3u8 — 多码率索引 #EXTM3U #EXT-X-STREAM-INF:BANDWIDTH=800000,RESOLUTION=640x360 360p.m3u8 #EXT-X-STREAM-INF:BANDWIDTH=2800000,RESOLUTION=1280x720 720p.m3u8 #EXT-X-STREAM-INF:BANDWIDTH=5000000,RESOLUTION=1920x1080 1080p.m3u8 # 720p.m3u8 — 单码率的播放列表 #EXTM3U #EXT-X-VERSION:3 #EXT-X-TARGETDURATION:6 #EXTINF:6.000, 720p_00001.ts #EXTINF:6.000, 720p_00002.ts #EXTINF:5.500, 720p_00003.ts #EXT-X-ENDLIST

MPEG-DASH — 国际标准自适应流媒体

Dynamic Adaptive Streaming over HTTP，ISO/IEC 23009-1。与 HLS 类似但更开放、更灵活

与 HLS 的本质区别

编解码无关 · 容器无关 · DRM 无关

DASH 是纯标准（ISO），不绑定任何厂商。HLS 绑定 Apple 生态（FairPlay DRM、ts 容器）。DASH 支持任意编解码（H.264/H.265/AV1/VVC）、任意容器（fMP4/WebM/MPEG-TS）、任意 DRM（Widevine/PlayReady/Marlin）。MPD 文件（Media Presentation Description）是 XML 格式的清单，替代 m3u8。

技术特点

更精细的自适应 · 更低的延迟

• Period 分段——不同时段可用不同码率组合（如广告段独立）
• AdaptationSet——同内容不同编码/码率的集合
• Representation——具体的码率/分辨率版本
• SegmentTemplate——用 URL 模板替代逐个列举分片
• 低延迟 DASH (LL-DASH)——Chunked CMAF + HTTP/1.1 chunked transfer → ~2-3s 延迟

HLS vs DASH 选型

市场份额 vs 标准纯粹性

• 移动端→ HLS（iOS/Android 原生优先）
• 桌面/机顶盒→ DASH（更开放）
• 混合方案→ 服务端同时输出 HLS+DASH
• IPC 场景→ 两个都不适合（延迟太高）
• 趋势→ CMAF 统一容器，HLS 和 DASH 指向同一套 fMP4 文件

CMAF — 统一 HLS 和 DASH 的桥梁
Common Media Application Format (CMAF) 定义了一套HLS 和 DASH 都能直接引用的 fMP4 分片格式。编码器输出一套 CMAF 分片 → 同时生成 .m3u8 和 .mpd 索引 → HLS 播放器和 DASH 播放器都能播放。一套存储、两份索引、零额外成本支持两个协议。Apple 从 HLS 也开始支持 fMP4（替代 ts），推动 CMAF 成为实际统一标准。

WebRTC — 超低延迟实时通信

Google 2011 年开源，W3C/IETF 标准。浏览器原生支持，亚秒级延迟——IPC 实时监控的未来

协议栈全景

ICE · DTLS · SRTP · SCTP — 四层安全实时传输

WebRTC 不是一个协议，而是一套协议集合：ICE（NAT 穿透，STUN+TURN）→ DTLS（密钥交换+身份认证）→ SRTP（加密媒体传输）+ SCTP（可靠数据通道）。信令层未标准化（可用 WebSocket/MQTT/SIP）。P2P 模式下媒体数据不经过服务器直接在对等端之间传输；SFU 模式下经过选择性转发单元中转。

延迟优势分析

为什么 WebRTC 能做到 <100ms？

① UDP 传输——无 TCP 队头阻塞，丢包用 NACK/PLC/FEC 恢复而非重传等待
② 无分片缓冲——编码帧立即 RTP 分包发送，不等下一帧
③ Jitter Buffer 自适应——初始缓冲极小（15-30ms），根据网络逐步扩展
④ 硬件编解码——H.264 硬编 2-5ms，VP8/VP9 软编也很快
⑤ P2P/SFU 直连——不经过 CDN 缓存层。端到端延迟典型值 30-150ms（LAN）、50-300ms（WAN）。

IPC 场景优势

实时监控 · 双向对讲 · 浏览器播放

• 延迟碾压——100ms vs RTSP 500ms vs HLS 10s
• 浏览器原生——无需插件，H5 video 直接播放
• 双向对讲——DataChannel 或音频上行天然支持
• 自适应码率——Simulcast/SVC 多客户端适配
• 安全内置——DTLS+SRTP 端到端加密

IPC 场景挑战

信令 · NAT · SFU · 多路预览

• 信令服务依赖——需要额外的信令服务器建立连接
• NAT 穿透不确定性——对称 NAT 需 TURN 中继
• SFU 成本——多客户端时需 SFU（如 LiveKit/Janus）
• 多路预览——同时预览 16 路需要 16 个 PeerConnection
• 录像存储——WebRTC 管传不管存，需额外实现

LiveKit 方案

开源 SFU，简化 WebRTC 部署

LiveKit 封装了 WebRTC 的复杂性：信令（WebSocket+JWT）、SFU（media server）、录制（Egress 转 MP4）、多房间管理。IPC 集成路径：嵌入式 SDK（LinkricAI）→ LiveKit Cloud/Self-Host → Web/App SDK 播放。这是当前 IPC 实时监控的最佳 WebRTC 实践。

IPC 流媒体协议选型指南

没有万能协议——不同场景的最优选型策略

局域网 IPC 直连预览

→ RTSP (UDP) — 不二之选

局域网延迟可控（<500ms），NAT 不是问题，ONVIF 兼容。实现简单——嵌入式 live555 或 GStreamer rtsp-server 几小时就能跑起来。VLC/FFplay 直接播放 rtsp:// URL。

NVR 多路取流录像

→ RTSP (TCP) — 可靠性优先

NVR 同时取 16-64 路 IPC 的流。TCP 模式比 UDP 更可靠（不丢包），略微增加的延迟（+100ms）不影响录像。海康/大华/ONVIF 全部基于 RTSP。

App/浏览器远程实时预览

→ WebRTC — 体验最优

公网延迟敏感。App 和浏览器原生支持 WebRTC（无需插件）。LiveKit SDK 封装后集成成本可接受。配合 TURN 服务器保证 99% 穿透成功率。延迟 <300ms 可流畅对讲。

公网直播/分享

→ RTMP 推流 + HLS 拉流

IPC 作为 RTMP 推流端（FFmpeg 软推或硬件编码直接输出 RTMP），推送到阿里云/腾讯云直播 CDN。App 端通过 HLS/FLV 拉流。支持千万级并发。延迟 3-5s 在直播场景可接受。

AI 盒子第三方取流

→ RTSP — 兼容性第一

AI 分析盒子需要从各品牌 IPC 取流做检测。RTSP 是唯一所有品牌都支持的通用协议。即使 IPC 内部用私有协议，也一定会暴露 RTSP 接口给第三方。

录像回放 Seek

→ RTSP (Range/Scale) 或私有协议

RTSP PLAY 支持 Range 头指定时间段（如 npt=20240601T120000Z-20240601T121000Z），支持 Scale 实现快进/慢放。但实际工程中很多厂商用私有 HTTP 协议回放录像（更灵活的时间轴拖动、事件标记）。

★

IPC 产品推荐协议策略：双协议栈

RTSP（存量兼容）+ WebRTC（增量体验）— 一个都不能少

主协议 RTSP：支持 ONVIF 认证，兼容所有 NVR/VMS/AI 盒子，覆盖存量市场和行业集成需求。UDP 模式保证预览延迟 <500ms。
增强协议 WebRTC：通过 LiveKit SDK 提供超低延迟（<200ms）体验，支持浏览器无插件播放、双向对讲、Simulcast 多客户端。覆盖移动 App 和 Web 端的实时预览需求。
辅助协议 HLS（可选）：用于公网直播分享场景——IPC → RTMP 推流 → CDN 转 HLS → 多人同时观看。延迟敏感场景不用 HLS。
不推荐：单一 RTSP（浏览器无法播放）、单一 WebRTC（存量 NVR 无法接入）、单一 RTMP（延迟 vs 穿透性不如 WebRTC）。
