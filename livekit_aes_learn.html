AES 对称加密：CTR vs GCM
对称加密 = 加密解密用同一个密钥。项目里用两种 AES 模式：
AES-128-CTR 加密 SRTP 视频帧，AES-128-GCM 加密 TLS 信令。
AES-128-CTR (SRTP)
1
明文 "Hello" → AES-CTR 加密 → 密文
2
额外计算 HMAC-SHA1 → auth tag (10B)
3
篡改密文一个 bit → 解密得乱码但不报错
4
auth tag 不匹配 → 丢弃 (但乱码已解密)
⚠️ 加密和认证分离 — 先解密再验证，可能泄露明文结构
AES-128-GCM (TLS)
1
明文 "Hello" → AES-GCM 加密+认证 → 密文 + tag
2
GHASH 一次性计算认证（含 AAD）
3
篡改密文一个 bit → tag 验证失败
4
直接拒绝，不尝试解密
✅ 加密认证一体化 — 先验证再解密，篡改绝不泄露任何信息

项目应用：SRTP 用的是 AES-CTR
为什么 SRTP 选了"旧"的 CTR 而不是"新"的 GCM？
这是历史原因，也是 WebRTC 标准演进的结果。
Layer 6 — SRTP 加密
AES-128-CTR + HMAC-SHA1-80
每个 RTP 包独立加密：从包头获取 SSRC + SeqNum，与 ROC 拼成 12 字节 IV，再和 master_salt 异或得到计数器初始值。

加密流程：IV = (SSRC∥ROC∥Seq∥0x000000) ⊕ salt → AES(IV)=keystream → keystream ⊕ payload = 密文

认证流程：对整个 RTP 包（含加密后 payload）做 HMAC-SHA1，取前 10 字节作 auth tag 附加在包尾。

开销：每包 +10 字节 auth tag。以 MTU 1000B、15fps 计算 ≈ 9KB/min 额外开销。

为什么不用 GCM？SRTP 标准制定于 2004 年（RFC 3711），当时 GCM 尚未成熟。2017 年的 RFC 7714 才定义了 SRTP_AES128_GCM，但 WebRTC 主流实现仍以 CTR 为主。
// SrtpSession.c — 加密 + 认证 srtp_protect(ctx, rtp_buf, &len); // 1. IV = (SSRC || ROC || SeqNum || 0x000000) XOR master_salt // 2. AES-128-CTR: 对每个 16B 块，AES(IV+counter) → keystream // 3. keystream XOR payload → 密文 // 4. HMAC-SHA1(整个RTP包) → 取前10字节作auth tag // 5. 附加 auth tag 到包尾 srtp_unprotect(ctx, rtp_buf, &len); // 1. 计算 ROC (Roll-Over Counter) // 2. 检查重放窗口 (128 packets) // 3. 验证 auth tag (HMAC-SHA1) // 4. 验证通过 → 计算 IV → 解密 payload // 5. 验证失败 → 丢弃
Layer 2 — TLS 信令
AES-128-GCM
TLS 1.2 信令通道使用 GCM 模式。CTR 加密 + GHASH 认证一次遍历完成。

为什么 TLS 用 GCM？TLS 1.2 (2008) 设计时 GCM 已经成熟可选（RFC 5288），比 CTR+HMAC 更高效。浏览器和 CDN 都优先选择 GCM。

GCM 的 AAD 优势：额外认证数据（如 TLS 序号）参与 GHASH 计算但不加密，GCM 天然支持。CTR+HMAC 需要手动把 AAD 拼进 HMAC 输入。
// Tls_mbedtls.c — 密码套件配置 // 优先选择带 GCM 的套件 static const int ciphersuites[] = { MBEDTLS_TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256, // ← CTR 加密 + GCM 认证，取代旧的 CBC+HMAC // 没有 AES-128-CTR + HMAC 的套件 0 };
