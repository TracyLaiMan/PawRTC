核心设计思想

接口隔离 · 依赖注入 · 事件驱动

Interface-based Design: 几乎所有模块都通过抽象接口 (纯虚类) 交互。工厂模式 (Factory + shared_ptr) 创建实例。Executor 线程模型隔离并发。Observer 模式实现解耦通知。

ACL → ADSL → CapabilityAgent

消息处理的核心链路

云端消息经 ACL (通信层) 接收 → ADSL (指令序列化器) 解析 JSON 为 ACSDirective → 按 namespace+name 路由 到对应的 CapabilityAgent → 触发硬件操作。

2. 核心设计模式 — 面试一定会问

Interface-based Design · 工厂模式 · Observer · CRTP · Executor · RequiresShutdown · 依赖注入

① Interface-based Design (IBD)

SDK 最核心的设计范式

每个模块定义两套接口: XxxInterface.h (纯虚类 — 公共契约) 和具体实现类。
例: MessageRouterInterface 定义 sendMessage()/setObserver() → MessageRouter 是实现。
好处: ① 编译期解耦 — consumer 只依赖头文件 ② Mock 测试 — 可以注入假实现 ③ A/B 替换 — 换 MQTT 到 WebSocket 只需换 TransportFactory。
面试要点: 这是 依赖倒置原则 (DIP) 的 C++ 实现 — 高层模块不依赖低层实现，都依赖抽象接口。

② 工厂模式 — Factory + shared_ptr

static createXxx() 静态工厂函数

模式: static std::shared_ptr<T> create(...) 作为类的静态工厂。
原因: ① enable_shared_from_this 需要 shared_ptr 管理生命周期 ② 构造函数可以失败 (返回 nullptr) — C++ 构造函数不能返回错误 ③ 依赖注入 — 工厂参数就是依赖。
面试要点: 为什么不用 new? → shared_ptr + enable_shared_from_this 是 SDK 的对象生命周期管理基石。裸 new 会破坏引用计数一致性。

③ Observer 模式 — 事件通知

ObserverInterface + setObserver()

一推一拉: MessageRouter 推送消息给 MessageRouterObserver。 ChannelObserverInterface 通知焦点变化给各模块。
实现: std::shared_ptr<XxxObserverInterface> 作为回调目标 (避免悬空指针)。
面试要点: Observer 是 发布-订阅 在单进程内的实现 — 让模块间松耦合通信。

④ Executor 线程模型

每个模块独立执行器 — 序列化异步回调

问题: 网络回调在多线程环境下来自任意线程 → 数据竞争。
方案: 每个模块持有一个 Executor (单线程事件循环)。所有外部回调 通过 executor.submit([task]) 序列化到同一条线程 — 消除锁需求。
面试要点: 这是 Actor 模型 的轻量实现 — 每个模块 = 一个单线程 Actor。

/* ===== CapabilityAgent — 多态基类的设计 ===== */ namespace acsCommon::acs { class CapabilityAgent : public acsCommon::sdkInterfaces::DirectiveHandlerInterface { public: virtual ~CapabilityAgent() = default; /* DirectiveHandlerInterface — 模板方法模式 (Template Method) */ void preHandleDirective(...) override final; // final: 禁止子类覆盖 bool handleDirective(...) override final; // 统一管理 messageId→Directive 映射 void cancelDirective(...) override final; protected: /* 虚函数 — 子类覆盖实现具体业务逻辑 */ virtual void preHandleDirective(std::shared_ptr<DirectiveInfo>) = 0; virtual void handleDirective(std::shared_ptr<DirectiveInfo>) = 0; virtual void cancelDirective(std::shared_ptr<DirectiveInfo>) = 0; }; } // namespace
面试杀手锏: 能说清楚 Template Method 模式在 CapabilityAgent 中的应用
CapabilityAgent 在 public 接口 中实现了 preHandleDirective/handleDirective/cancelDirective (带 final 关键字)，这些方法负责公共逻辑 (维护 messageId→Directive 映射表、线程安全)。然后在 protected 区域声明了同名的纯虚函数，供子类覆盖实现业务逻辑。这就是 Template Method 模式 — 基类定义算法骨架，子类填充具体步骤。

CapabilityAgent 注册流程

启动时逐个注册到 DirectiveRouter

① DefaultClient 创建所有 CapabilityAgent 实例 (工厂模式)
② 每个 CA 内部定义它的 DirectiveRoutingRule — namespace+name 组合 (如 {Bluetooth, ScanDevices})
③ CA 调用 DirectiveSequencer::addDirectiveHandler(this) — 将自己注册到路由表
④ 当 ACSDirective 到达 → DirectiveRouter 精确匹配 routingTable → 调用对应 CA
关键: 新增设备能力 = 新增一个 CapabilityAgent 子类 + 注册。

Event 响应构建

CapabilityAgent → Event → ACL → Cloud

CA 处理完 Directive 后需要异步响应事件给云端:
① EventBuilder 构造 JSON Event (namespace + name + payload)
② 封装为 MessageRequest
③ 调用 MessageRouter::sendMessage(request)
④ MQTT/WebSocket 发送到云端
异步: Event 不在 Directive 处理线程中发送 — 通过 Executor 序列化排队。

面试必问: 蓝牙模块如何使用事件总线 (EventBus) 解耦
问题: 底层蓝牙栈 (BlueZ) 在不同的线程中产生事件 (设备发现/连接/数据到达) — 如何安全地通知上层?
答案: BluetoothEventBus — 底层蓝牙栈通过 EventBus 发布事件 → Bluetooth (实现 BluetoothEventListenerInterface) 订阅感兴趣的事件 → EventBus 通过 Executor 将回调序列化到 Bluetooth 的线程 — 消除数据竞争。
优点: ① 底层栈不知道上层是谁 (完全解耦) ② 一个事件可以有多个订阅者 ③ 线程安全由 EventBus + Executor 保证。

5. 线程模型与生命周期 — 最容易被问倒的部分

Executor 单线程 Actor · RequiresShutdown 安全关闭 · enable_shared_from_this · 避免死锁的三条铁律

Executor — 每个模块一个 Actor

acsCommon::utils::threading::Executor

每个功能模块持有一个独立 Executor (内部是一个消息队列 + 单工作线程)。所有异步回调通过 executor.submit([] { ... }) 提交。
效果: 模块内部无锁编程 — 所有操作序列化到同一线程。
面试要点: 对比"全局互斥锁"方案 — Executor 避免了锁竞争、死锁、优先级反转。

RequiresShutdown — 安全关闭协议

替代析构函数的异步清理

问题: C++ 析构函数不应该是虚函数？不 — 真正的麻烦是对象被 shared_ptr 引用时无法确定何时析构。
方案: RequiresShutdown 基类定义 shutdown() → doShutdown() 两步:
① shutdown() — 通知 Executor 停止接受新任务 ② doShutdown() — 子类清理资源 (关闭连接、释放缓冲区)。
关键: shutdown 必须在所有 shared_ptr 释放 之前 调用。

enable_shared_from_this

从 this 安全获取 shared_ptr

问题: 回调中需要将自身作为参数传递 — 但此时只有一个 raw this 指针。
方案: 继承 enable_shared_from_this<T> → shared_from_this() 返回自身的 shared_ptr。
原理: 基类内部存储一个 weak_ptr，在构造后由 make_shared 或 shared_ptr 初始化。
面试要点: 为什么不用 this? → 如果回调时对象已被释放 → 悬空指针。shared_from_this() 保证回调期间对象存活。

/* ===== 典型生命周期模式 — Bluetooth 为例 ===== */ // 1. 创建 (工厂 + 依赖注入) auto bt = Bluetooth::createBluetoothCapabilityAgent(deviceManager, eventBus); // 2. 注册到路由表 auto directiveSequencer = DirectiveSequencer::create(); directiveSequencer->addDirectiveHandler(bt); // 3. 运行期: 外部回调通过 Executor 序列化 // eventBus → Bluetooth::onBluetoothEventFired() [Bluetooth 线程] // directiveRouter → Bluetooth::preHandleDirective() → handleDirective() [Executor 线程] // 4. 关闭 (必须按这个顺序!) directiveSequencer->shutdown(); // ① 先停止分发 bt->shutdown(); // ② 再关闭 CA bt.reset(); // ③ 最后释放引用
⚠️ 面试陷阱: 为什么 shutdown() 必须在 shared_ptr 释放之前? (高频题)
如果先释放 shared_ptr → 对象可能在任意时刻被析构 (当最后一个 shared_ptr 离开作用域) → 但此时 Executor 线程可能仍在运行 → 访问已释放的成员变量 → use-after-free crash。
正确顺序: shutdown() → 停止 Executor → join 线程 → 待线程完全退出后 → reset shared_ptr → 安全析构。

6. 模拟面试问答 — 高频问题与参考答案

涵盖架构理解 · 设计模式 · C++ 特性 · 线程模型 · 故障排查 · 可扩展性

#问题参考答案要点
Q1SDK 的整体架构是怎样的？三层架构: ACL(通信层)→ADSL(指令路由层)→CapabilityAgents(功能层)。基于 Amazon AVS SDK 架构，采用 C++14，Apache 2.0 协议。核心模式: Interface-based Design + 工厂 + Observer + Executor。
Q2一条云端指令如何到达硬件？MessageRouter(MQTT)接收→MessageInterpreter解析JSON→DirectiveSequencer入队→DirectiveRouter按namespace+name路由→CapabilityAgent.preHandle/handleDirective→调用硬件驱动→EventBuilder构造响应→回传云端。
Q3新增一个设备能力需要改哪些地方？① 新建 CapabilityAgent 子类 (继承 DirectiveHandlerInterface) ② 定义 namespace+name 路由规则 ③ 注册到 DirectiveRouter ④ 实现 handleDirective 业务逻辑 ⑤ 在 DefaultClient 中创建实例。不需要修改 ACL/ADSL。
Q4SDK 如何实现线程安全？Executor 单线程 Actor 模型: 每个模块内部所有操作序列化到同一线程 → 无锁。跨模块通信通过 shared_ptr + 接口调用，由 Executor 保证回调在自己线程执行。EventBus + Executor 解耦生产者-消费者。
Q5为什么用 shared_ptr 而不是 raw pointer？① 对象生命周期管理 — 异步回调中保证对象存活 (shared_from_this) ② 多模块共享同一实例 — 引用计数自动回收 ③ 配合 enable_shared_from_this 在回调中安全传递自身引用。
Q6RequiresShutdown 是干什么的？解决异步模块的安全关闭问题。shutdown() 先停止 Executor (不接受新任务)→join 工作线程→doShutdown() 子类清理资源。必须在 shared_ptr 释放之前调用 — 防止线程仍运行时对象被析构。
Q7CapabilityAgent 中 final 关键字的作用？Template Method 模式: public 接口层 (preHandlDirective/handleDirective/cancelDirective) 用 final 禁止子类覆盖 → 保证 messageId→Directive 映射的公共逻辑不变。子类覆盖 protected 的同名纯虚函数 → 实现业务逻辑。
Q8蓝牙模块如何解耦底层协议栈？依赖倒置: BluetoothDeviceManagerInterface 抽象蓝牙操作 → BlueZ (Linux) / ESP Host (FreeRTOS) 各自实现 → CapabilityAgent 只依赖接口。新增蓝牙栈只需实现接口。EventBus 进一步解耦事件通知。
