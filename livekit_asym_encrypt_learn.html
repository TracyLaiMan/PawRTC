概念解析：公钥/私钥对
对称加密一把钥匙管两头，非对称加密把钥匙分成两半 —— 各管一头。
两者不是替代关系，而是互补搭配。
对称加密
AES — 同一把密钥加密解密
  • 密钥数量：1 把，加解密共享
  • 分发方式：必须通过安全的渠道传递密钥
  • 速度：极快（微秒级，0.5μs/KB）
  • 典型算法：AES-128, ChaCha20
  • 适用场景：海量数据加密（视频帧、文件）
  • 本项目：SRTP 视频加密、TLS 数据加密
非对称加密
RSA/EC — 公钥加密，私钥解密
  • 密钥数量：2 把，公钥公开、私钥保密
  • 分发方式：公钥可随意分发，私钥从不离开本地
  • 速度：极慢（毫秒级，500μs/KB）
  • 典型算法：RSA-2048, ECDSA P-256, ECDHE
  • 适用场景：密钥协商、身份签名、证书验证
  • 本项目：DTLS 握手、证书签名、ECDHE 密钥交换
非对称加密解决的核心问题：如何在不安全的信道上安全地交换密钥。
对称加密本身是安全的，但"怎么把密钥给对方"是个鸡生蛋的问题 —— 非对称加密就是那个蛋。

保密性
公钥加密 → 私钥解密
场景：别人要给你发秘密消息

步骤：
1. 你的 公钥 公开发布（任何人都能拿到）
2. 发送方用你的公钥加密消息：C = Enc(PubKey, Msg)
3. 密文在不安全信道传输
4. 只有你的 私钥 能解密：Msg = Dec(PrivKey, C)

项目实例：RSA 加密 premaster secret（TLS RSA 密钥交换，已弃用）
注意：项目的 ECDHE 不直接加密数据，而是用数学协商共享密钥

不可抵赖
私钥签名 → 公钥验证
场景：你需要证明"这段话确实是我说的"

步骤：
1. 你的 私钥 从不离开本地
2. 你用私钥对消息做签名：Sig = Sign(PrivKey, Msg)
3. 任何人用你的 公钥 验证：Verify(PubKey, Msg, Sig)
4. 验证通过 = 确认是你签的 + 消息未被篡改

项目实例：ECDSA P-256 签名 DTLS 证书
关键：签名不是加密——签名保证来源，加密保证内容

前向保密
ECDHE — 双方协商共享密钥
场景：双方要协商出同一个对称密钥，但不传输密钥本身

步骤（ECDHE）：
1. 双方各生成临时 密钥对
2. 互换公钥（通过 DTLS 握手消息）
3. 各自计算：shared = DH(自己的私钥, 对方的公钥)
4. 得到相同的 premaster_secret → 导出对称密钥

项目实例：DLS-ECDHE-P256 握手导出 SRTP 密钥
前向保密：临时密钥对每次握手重新生成，会话结束后销毁。即使以后私钥泄露，过去的会话也无法解密。

为什么 WebRTC 不用非对称加密视频？
非对称加密比对称加密慢 1000 倍以上。
切换视图，直观感受速率差距。

结论：非对称加密太慢，不能用来加密视频数据。

实际分工：非对称加密只用于握手和签名（毫秒级，一次性的），数据加密全部用对称加密（微秒级，持续性的）。

以 1080p 30fps 视频为例：带宽约 2Mbps = 250KB/s。
• AES-128-CTR：0.5μs/KB → 每秒加密耗时 0.125ms ✅
• RSA-2048：500μs/KB → 每秒加密耗时 125ms ❌（丢帧）

非对称层负责握手和身份认证，对称层负责数据加密和完整性。
两层通过密钥协商衔接——非对称产出对称密钥。
非对称层 — 握手/签名
毫秒级 · 一次性
🔄

✍️
ECDSA P-256 — 私钥签名 DTLS 证书 → 对端验证握手完整性
🔗
SDP 指纹 — SHA-256 哈希证书公钥 → 绑定身份到信令

对称层 — 数据加密
微秒级 · 持续性
🎬
AES-128-CTR — 加密 SRTP 视频帧（每个 RTP 包独立加密）
🛡️
HMAC-SHA1 — 认证 RTP 包完整性（auth tag 10B）
📡
AES-128-GCM — 加密 TLS 信令数据（一体化加密+认证）
衔接点：非对称层的 ECDHE 协商出 master_secret，通过 TLS PRF 导出对称层所需的密钥材料（SRTP key/salt、TLS key/IV）。每个方向的密钥独立导出，一个密钥只加密一个方向的数据。

代码中的分工 — mbedTLS API
两条路径清晰分离：握手阶段走非对称，数据阶段走对称。
非对称路径 — 握手阶段（毫秒级）
ECDHE 密钥协商 + ECDSA 签名
DTLS 握手期间，板端使用 ECDHE 临时密钥对与对端协商共享密钥， 使用 ECDSA 私钥签名自己的证书以证明身份。整个过程只在连接建立时执行一次。
// Dtls_mbedtls.c — 握手期间调用 // ECDHE: 生成临时密钥对并发送公钥参数 mbedtls_ecdh_make_params(&ecdh, &olen, buf, sizeof(buf), mbedtls_ctr_drbg_random, &ctr_drbg); // ECDHE: 收到对端公钥后计算共享密钥 mbedtls_ecdh_compute_shared(&ecdh, &olen, buf, sizeof(buf), mbedtls_ctr_drbg_random, &ctr_drbg); // ECDSA: 用私钥签名 ServerKeyExchange mbedtls_ecdsa_write_signature(&ecdsa, MBEDTLS_MD_SHA256, hash, hash_len, sig, &sig_len, mbedtls_ctr_drbg_random, &ctr_drbg); // ECDSA: 验证对端证书签名 mbedtls_ecdsa_read_signature(&ecdsa, hash, hash_len, sig, sig_len);
对称路径 — 数据阶段（微秒级）
AES 加密 + HMAC 认证
握手完成后，导出的对称密钥进入高速路径。每个 RTP 包用 AES-128-CTR 加密、HMAC-SHA1 认证。 TLS 数据则用 AES-128-GCM 一体化加密认证。
// SrtpSession.c — 每个 RTP 包的处理 // AES-128-CTR: 加密视频帧 payload mbedtls_aes_crypt_ctr(&aes_ctx, payload_len, nc_off, nonce, stream_block, payload, payload); // HMAC-SHA1: 计算 auth tag (取前10字节) mbedtls_md_hmac(&md_sha1_info, auth_key, auth_key_len, rtp_buf, rtp_len, tag); // Tls_mbedtls.c — TLS 数据加密 // AES-128-GCM: 自动完成加密 + GHASH 认证 // (由 mbedTLS 内部在 write/writev 中调用)
板端性能风险：
RSA-2048 握手在 MIPS 架构上延迟约 2-5 秒，会导致 DTLS 握手超时。
项目选择 ECDSA P-256 替代 RSA：签名约 300ms，验证约 600ms，完全满足 WebRTC 握手时序要求。

选型依据：mbedTLS 配置中关闭 MBEDTLS_RSA_C 和 MBEDTLS_PKCS1_V15，只保留 ECP + ECDSA + ECDHE，减少代码体积约 40KB，握手延迟降低 10 倍。
