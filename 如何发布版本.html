如何发布版本

本文档描述 JOY1 嵌入式 Linux 固件的版本发布流程。

1. 发布步骤

Step 1: 准备分支

命名规则

<product_name>_release_<phase>-<number>

示例

阶段 分支名示例
DVT joy1_release_DVT-1, joy1_release_DVT-2
MP joy1_release_MP
OTA（量产后） joy1_release_ota1, joy1_release_ota1-1, joy1_release_ota2

操作

两个仓库都需要创建同名分支并推送到 GitLab：

  • T41 工程（VisionT）
  • ESP 工程（esp_hosted_fg）
# T41 工程
git checkout -b joy1_release_ota3-2
git push origin joy1_release_ota3-2

# ESP 工程
git checkout -b joy1_release_ota3-2
git push origin joy1_release_ota3-2

Step 2: 创建发布 commit

在 T41 工程 操作。

commit message 格式

RELEASE_<PRODUCT_UPCASE>_<VERSION>

示例：RELEASE_JOY1_0.0.6.24

重要：CI 会自动构建 两个版本（VERSION 和 VERSION+1） 用于 OTA 升级测试。 例如提交 RELEASE_JOY1_0.0.6.24，会生成 0.0.6.24 和 0.0.6.25 两个固件。

两种方式（任选其一）

方式 A：空 commit

git commit --allow-empty -m "RELEASE_JOY1_0.0.6.24"

方式 B：修改版本号后提交

# 修改 config/config_joy1.release 中的 PRODUCT_VERSION
sed -i 's/PRODUCT_VERSION=.*/PRODUCT_VERSION=0.0.6.24/' config/config_joy1.release
git add config/config_joy1.release
git commit -m "RELEASE_JOY1_0.0.6.24"

Step 3: 推送触发 CI

git push origin joy1_release_ota3-2

推送后 GitLab CI 自动执行构建。等待 Pipeline 完成即可。

2. 检查清单

发布前

  • [ ] T41 仓库（VisionT）分支已推送到远程
  • [ ] ESP 仓库（esp_hosted_fg）同名分支已推送到远程
  • [ ] 版本号格式正确（如 0.0.6.24）

发布后

  • [ ] Pipeline 执行成功
  • [ ] 固件已部署到共享服务器

3. 常见问题

双版本机制

CI 自动构建两个版本：VERSION 和 VERSION+1。

用途：方便 OTA 升级测试。例如： - 设备先烧录 0.0.6.24 - 通过 OTA 升级到 0.0.6.25 - 验证升级流程是否正常

版本号含义

格式：主版本.次版本.修订号.构建号

示例：0.0.6.24 - 0 - 主版本 - 0 - 次版本 - 6 - 修订号 - 24 - 构建号

ESP 分支不存在

CI 会报错：Branch 'xxx' does not exist in ESP repository

解决：在 ESP 仓库创建同名分支并推送。

CI 失败

查看 GitLab Pipeline 日志定位原因。常见问题： - ESP 分支不存在 - 编译错误 - 网络问题

生成的产物

文件 说明
JOY1_*.ota.bin OTA 升级包
JOY1_ESP_*_ota.bin ESP OTA 升级包
JOY1_ESP_*_factory.bin ESP 工厂固件

存储位置：共享服务器 /share/share/JOY1/firmware/
