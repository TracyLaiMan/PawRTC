LiveKit Cloud 适配实践

mipsel 板端连接 LiveKit Cloud 的四重问题与修复

WebSocket ALPN/H2 nanopb ICE/TCP
JY1C / Ingenic T41 · KVS-WebRTC + LWS + mbedTLS

适配背景

mipsel 板端（JY1C/Ingenic T41）通过 KVS-WebRTC + LWS + mbedTLS 连接 LiveKit Cloud，实现 H.264 视频推流到浏览器。适配过程中遇到四重问题，以下逐项记录根因和修复。

架构

板端 → Cloudflare CDN → LiveKit Cloud SFU → 浏览器订阅观看。
信令通道 WSS，媒体通道 ICE/DTLS/SRTP。

1 WebSocket 升级失败（HTTP 200 而非 101）

现象

[signaling] ESTABLISHED_CLIENT_HTTP: status=200 (expected 101 for WS!)
CONNECTION_ERROR: HS: ws upgrade response not 101

板端 curl 和 openssl s_client 均正常返回 101，但 LWS 连接返回 HTTP 200。

根因（三重叠加）

  1. LWS_WITH_HTTP2 重复定义冲突 — CMakeLists 同时传了 -DLWS_WITH_HTTP2=1 和 -DLWS_WITH_HTTP2=0，前者优先生效。CDN（Cloudflare）协商 H2 后，WS upgrade 头丢失（H2 不支持 Upgrade 机制）。
  2. 交叉编译 ALPN 功能检测失败 — LWS 的 CHECK_C_SOURCE_COMPILES 在 x86 主机无法链接 mipsel 的 mbedTLS，LWS_HAVE_mbedtls_ssl_conf_alpn_protocols 未定义。运行时 ALPN 不生效 → CDN 默认协商 H2 → WS upgrade 失败。
  3. mbedtls_ssl_get_alpn_protocol 同样缺失 — 即使 conf_alpn 生效，get_alpn 缺失导致协商结果无法验证。

修复

CMake

文件: kvs-webrtc/CMake/Dependencies/libwebsockets-CMakeLists.txt

1. 删除重复的 -DLWS_WITH_HTTP2=1，只保留 =0

2. 通过 CMAKE_C_FLAGS 注入 ALPN/SNI 宏，绕过 cmake try_compile：

set(LWS_EXTRA_CFLAGS "${CMAKE_C_FLAGS}")
string(APPEND LWS_EXTRA_CFLAGS " -DLWS_HAVE_mbedtls_ssl_conf_alpn_protocols=1")
string(APPEND LWS_EXTRA_CFLAGS " -DLWS_HAVE_mbedtls_ssl_get_alpn_protocol=1")
string(APPEND LWS_EXTRA_CFLAGS " -DLWS_HAVE_mbedtls_ssl_conf_sni=1")

3. CMAKE_ARGS 中用 ${LWS_EXTRA_CFLAGS} 替代 ${CMAKE_C_FLAGS}

信令代码

文件: src/LiveKitSignaling.cpp

ctxInfo.alpn = "http/1.1";   // 强制 ALPN 协商 HTTP/1.1
ccInfo.alpn = "http/1.1";
ccInfo.method = NULL;          // NULL = WS upgrade，非 NULL = HTTP method

验证

LWS: 4.3.5, MbedTLS-3.6.3 NET CLI H1 WS ConMon IPV6-on
WS 101 升级成功 → JoinResponse
原则

LWS 交叉编译时 try_compile 无法执行，feature detection 全部失败。必须手动强制定义 mbedTLS 功能宏。未来升级 LWS 或 mbedTLS 版本时需确认这些宏仍适用。

2 浏览器订阅后设备断开（nanopb SDP 缓冲区溢出）

现象

设备推流正常，浏览器打开 meet.livekit.io 订阅后几秒，设备收到 LEAVE reason=6 断开。

[signaling] decodeSignalResponse FAILED for 4385 bytes
[ICE] CheckCandidatePairConnection: agent=0x75e51cc8 pairs=0 waiting=0 inProgress=0 succeeded=0 failed=0
IceAgent failed with 0x5a00000d
=== LEAVE === can_reconnect=0 reason=6

根因

  1. 浏览器订阅时，LiveKit SFU 向设备发送 subscriber offer（包含远端 ICE 候选），该消息 4385 字节，超过 nanopb SessionDescription.sdp 的 max_size:4096 限制。
  2. protobuf 解码失败 → 设备收不到远端候选 → subscriber ICE agent 有 0 个 candidate pair → ICE 超时失败 → 服务器发送 LEAVE。
为什么能出流几秒？

Publisher ICE agent 的 host→host STUN 对确实成功了，DTLS/SRTP 正常初始化。但这个连接依赖 NAT 映射，没有 TURN relay 备份。浏览器订阅时 SFU 发起新的 subscriber transport，这条消息解码失败才是断开的直接原因。

修复

Protobuf Options

文件: protos/livekit_rtc.options

# 4096 → 8192
livekit.SessionDescription.sdp      max_size:8192

重新生成 nanopb 文件：

cd vendor/linkric/livekit-client/protos
python3 -m grpc_tools.protoc \
  --plugin=protoc-gen-nanopb=$HOME/.local/bin/protoc-gen-nanopb \
  --nanopb_out=./nanopb \
  --proto_path=. \
  --nanopb_opt="-I." \
  livekit_rtc.proto

3 URL 缺少默认路径 /rtc

现象

lk_demo wss://pawrtc-qbbhiyec.livekit.cloud <token> --mp4 /data/test.mp4
→ ESTABLISHED_CLIENT_HTTP: status=200 (expected 101 for WS!)

根因

parseWsUrl() 解析无路径的 URL 时默认 path=/，而 LiveKit Cloud WS 端点是 /rtc。

修复

文件: src/LiveKitSignaling.cpp

// 之前
path = (slashPos != std::string::npos) ? rest.substr(slashPos) : "/";
// 之后
path = (slashPos != std::string::npos) ? rest.substr(slashPos) : "/rtc";

用法：

# 以下两种等价
lk_demo wss://pawrtc-qbbhiyec.livekit.cloud <jwt-token> --mp4 /data/test.mp4
lk_demo wss://pawrtc-qbbhiyec.livekit.cloud/rtc <jwt-token> --mp4 /data/test.mp4

4 KVS ICE 不支持 TCP 候选（已知限制）

现象

[LkTransport] addRemoteCandidate failed: 0x5a000005 for a=candidate:... tcptype passive

根因

  • 0x5a000005 = STATUS_ICE_CANDIDATE_STRING_IS_TCP
  • IceAgent.c:438 显式拒绝所有 TCP 候选
  • LiveKit Cloud 通过 CDN 提供的候选多数是 TCP（tcptype passive），全部被丢弃
  • 目前靠 UDP host 候选 + TURN UDP relay 工作

当前状态

TURN UDP 可正常分配 relay 候选，publisher 和 subscriber 都能通过 TURN relay 连接。TURN TLS（turns:...?transport=tcp）因板端缺 CA 证书而失败，但不影响功能。

经验教训

1
CDN + WS 的坑
Cloudflare 等 CDN 默认协商 H2，H2 不支持 Upgrade 机制。必须通过 ALPN 强制 http/1.1 并禁用 H2。
2
交叉编译 try_compile 不可靠
LWS/mbedTLS 的 feature detection 在交叉编译时全部失败，需要手动注入宏定义。
3
嵌入式 protobuf 缓冲区
nanopb 的固定大小缓冲区是双刃剑——省内存但要预留足够空间。LiveKit Cloud 的 SDP 消息较大（>4KB），max_size 至少 8192。
4
先看官方文档和协议
遇到问题先查 LiveKit 官方 client 实现和协议文档，不要盲目调试。
