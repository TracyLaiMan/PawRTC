2. 三线程模型 — PeerConnection 的并发基石

PeerConnectionFactory 创建并持有三条线程 · 所有 PeerConnection 实例共享 · 跨线程通过 BlockingCall + Proxy 安全通信

Signaling Thread

信令线程 — API 入口

运行内容: ① 所有 W3C API 调用 (createOffer/setLocalDescription/addTrack...) ② PeerConnectionObserver 回调 (onIceCandidate/ontrack/oniceconnectionstatechange) ③ SDP 解析与生成 ④ 信令状态机
约束: 应用代码必须快速返回 — 不能阻塞此线程。任何耗时操作 (编解码/I/O) 必须委托到 Worker/Network。

Worker Thread

工作线程 — 媒体处理

运行内容: ① 音频: 采集→3A 处理(AEC/NS/AGC)→Opus 编码→RTP 打包 ② 视频: 采集→预处理→VP8/VP9/H264/AV1 编码→RTP 打包 ③ 接收侧: RTP 解包→解码→渲染 ④ 带宽估计 (GCC)
Call 对象 及其全部 AudioSendStream/VideoSendStream 在此线程运行。
关键: 媒体处理是 CPU 密集型 — Worker 线程优先级应高于 Signaling。

Network Thread

网络线程 — I/O 与传输

运行内容: ① UDP/TCP socket 读写 ② ICE 候选收集 (STUN/TURN 查询) ③ ICE 连通性检查 (STUN binding request/response) ④ DTLS 握手 ⑤ SRTP 加密/解密 ⑥ SCTP 数据传输 (DataChannel)
特点: 拥有独立的 socket server · 处理所有网络层协议 · 从 Worker 收到的编码数据在此加密后发出。

/* ═══ 线程创建 — PeerConnectionFactory 初始化 ═══ */ LibWebRTC::Initialize() └── PeerConnectionFactory::Create() ├── rtc::Thread::CreateWithSocketServer() // Network Thread (带 socket server) ├── rtc::Thread::Create() // Worker Thread ├── rtc::Thread::Create() // Signaling Thread ├── 命名: network_thread / worker_thread / signaling_thread └── 所有 PC 实例共享这三条线程 /* ═══ 跨线程调用机制 ═══ */ // 1. 同步跨线程调用 (Signaling → Worker) int result = worker_thread()->BlockingCall([&] { RTC_DCHECK_RUN_ON(worker_thread()); // ★ 运行在 Worker Thread 上 return call()->CreateVideoSendStream(config); }); // 调用线程阻塞直到返回 // 2. 异步跨线程调用 (Network → Signaling) signaling_thread()->PostTask([this] { // ★ 运行在 Signaling Thread 上 observer_()->OnIceCandidate(candidate); }); // 3. Proxy 模式 — 自动跨线程代理 // PeerConnectionProxy / RtpSenderProxy 等通过宏自动生成 // PROXY_PRIMARY_THREAD_DESTROYOR(PeerConnection) // PROXY_METHOD0(void, PeerConnection, Close)
线程安全保证 — 三个机制
① RTC_DCHECK_RUN_ON(thread): 编译期+运行期检查 — 确保代码在正确线程执行。生产构建不检查, 调试构建 crash 提示线程错误。
② BlockingCall / PostTask: 显式跨线程调度 — BlockingCall 同步等待返回, PostTask 异步不等待。
③ Proxy 模式: 自动生成线程代理 — 宏展开为基础 BlockingCall/PostTask 调用, 避免手动线程调度错误。

SetLocalDescription — 触发级联

★★★★★ 最关键的 API 调用

setLocalDescription() 是整个连接建立过程的起点:
① SDP 写入本地描述
② ★ ICE 候选收集开始 — 查询 STUN/TURN 服务器
③ JsepTransportController 创建传输对象
④ DTLS 证书生成 (首次)
⑤ iceGatheringState → "gathering"
⑥ onicecandidate 逐次触发 (Trickle ICE)

SetRemoteDescription — 激活连接

★★★★★ 触发 ICE 连通性检查

setRemoteDescription() 是连接激活的触发器:
① SDP 解析 → 创建远程媒体描述
② ★ ontrack 事件触发 — 远程媒体流到达
③ ICE 候选对形成 → 连通性检查开始
④ iceConnectionState → "checking"
⑤ DTLS 角色确定 (controlling vs controlled)
⑥ canTrickleIceCandidates 赋值

4. ICE — 穿越 NAT 的连通性魔法

候选收集 (host/srflx/relay) → 候选对排序 (host>srflx>relay, UDP>TCP) → STUN 连通性检查 → Nomination

/* ═══ ICE 完整流程 ═══ */ Phase 1: 候选收集 (ICE Gathering) setLocalDescription() ──► JsepTransportController ──► PortAllocator ├── 每网络接口创建 AllocationSequence ├── STUN 查询 → 获取 srflx 候选 (服务器反射地址) ├── TURN Allocate → 获取 relay 候选 (中继地址, 如果配置了 TURN) ├── host 候选 = 本地 IP (192.168.x.x / 10.x.x.x) └── onIceCandidate(candidate) — 逐次触发 (Trickle ICE) └── iceGatheringState: new → gathering → complete Phase 2: 候选交换 应用层 Signaling (WebSocket / REST / 自定义) 交换 SDP + ICE 候选 Phase 3: 连通性检查 setRemoteDescription() ──► P2PTransportChannel ──► 形成候选对列表 ├── 排序: host→host > host→srflx > srflx→srflx > relay (UDP > TCP) ├── STUN Binding Request — 对每个候选对发送连通性检查 ├── 角色确定: ice-ufrag 字典序大的一方 = controlling (主动提名) ├── Nomination: controlling 端标记选中的候选对 └── iceConnectionState: new → checking → connected → completed

5. DTLS 握手 — 安全的最后一道门

ICE connected → DTLS ClientHello → 证书验证 (fingerprint) → SRTP 密钥派生 (RFC 5705)

DTLS 握手 — 简化流程

在 ICE 选中的 5-tuple 上运行

① ClientHello (Flight 1+3): 包含加密套件、随机数、use_srtp 扩展 (RFC 5764)
② ServerHello + Certificate + ServerHelloDone (Flight 2+5): 服务端证书、密钥交换参数
③ Client Certificate + ChangeCipherSpec + Finished (Flight 4+6): 客户端认证完成
④ Fingerprint 验证: 收到的证书指纹 vs SDP 中的 a=fingerprint:sha-256 → 不匹配 = MITM 攻击 → 连接失败

SRTP 密钥派生 (DTLS-SRTP)

DTLS 完成后的最关键步骤

① DTLS 握手完成后, 双方都有共享的 master secret
② 调用 RFC 5705 key exporter: SSL_export_keying_material(ssl, label="EXTRACTOR-dtls_srtp", ...)
③ 派生 SRTP master keys (send + recv 方向各自独立)
④ DtlsSrtpTransport 配置 SrtpSession → 此后所有 RTP/RTCP 包通过 SRTP 加密
⑤ connectionState → "connected"

/* ═══ DTLS 在 WebRTC 中的特殊之处 ═══ */ a=setup:actpass — 双方都可以是 DTLS Client 或 Server 角色由 ICE 确定: controlling = DTLS Client (默认) 注意: 这里有 DTLS Client ≠ WebRTC 的 offerer/answerer 分离 DTLS 1.3 (field trial: WebRTC-ForceDtls13) — 更快握手 (1-RTT) 优势: 减少一次往返 → 连接建立更快 SCTP over DTLS (RFC 8261) — DataChannel 在 DTLS 之上运行 SCTP DTLS 提供加密 → SCTP 提供可靠/不可靠多流传输 → DataChannel API
