分辨率与帧率 — 码率的"容器"

分辨率和帧率决定了编码前的数据量上限——选对了事半功倍，选错了后面所有调参都是白费。

分辨率选择

像素数量直接决定编码的计算量

1080p (1920×1080)：主流 IPC 标配。人脸识别需要 ≥150px 面部宽度→1080p 可看清 5-8m 内人脸。码率建议 1.5~4Mbps (H.264) 或 1~2.5Mbps (H.265)。
4MP (2560×1440)：细节提升 78%（相比 1080p），但编码计算量 ↑78%。码率 3~6Mbps。车牌识别推荐。
720p (1280×960)：计算量是 1080p 的 60%。码率 0.8~2Mbps。适合子码流或低成本产品。
原则：分辨率 = 场景需求决定（监控范围、识别距离）——不是越高越好。高分辨率 + 低码率 → 编码块效应严重 → 反而不如"合适分辨率 + 充足码率"。

帧率选择

每多 1fps = 编码计算量 +3.3% + 码率 +1~2%

30fps：流畅视频的黄金标准。运动物体（行人/车辆）轨迹平滑。IPC 主码流标配。
25fps：PAL 制标准（欧洲/中国）。几乎感知不到与 30fps 的差异——但编码负载降 17%。
15fps：静态监控场景（仓库、停车场夜间）——帧间变化极小→15fps 足够。码率约为 30fps 的 70%（而非 50%——因为 I 帧占比增加）。
8-10fps：子码流/AI 检测流。编码负载低——适合多路同时编码。

分辨率 × 帧率 × 码率关系表

H.264 编码器在典型场景 (中等复杂度) 的建议值

1080p@30fps → 2.0~4.0 Mbps · 编码负载 100% (基准)
1080p@15fps → 1.4~2.8 Mbps · 编码负载 ~55%
720p@30fps → 1.2~2.5 Mbps · 编码负载 ~60%
4MP@30fps → 4.0~8.0 Mbps · 编码负载 ~178%
4MP@15fps → 2.8~5.6 Mbps · 编码负载 ~100%
640×360@15fps (子码流) → 0.2~0.4 Mbps · 编码负载 ~15%

码率控制 — CBR vs VBR vs CVBR

码率控制决定了编码器如何分配比特——在相同的目标码率下，不同的码控策略产出的画质差异巨大。

CBR (恒定码率)

每帧目标大小固定 — 画质波动 · 码率严格可控

原理：编码器维持恒定的输出码率——通过调整 QP（量化参数）实现：场景复杂→QP 增大→每个宏块用更少的 bit→画质下降；场景简单→QP 减小→画质提升。
优势：码率严格可控——适合带宽受限场景（4G 上行、多路同时传输）。网络带宽可精确预估。
劣势：画质波动大——静态画面（黑夜、无人）画质极好（浪费码率），动态画面（人多、车多）画质骤降（块效应）。
IPC 适用：子码流（网络预览——码率预算是硬约束）。

VBR (可变码率)

恒定 QP — 码率波动 · 画质一致

原理：编码器维持恒定的画质水平（固定 QP 或 CRF 值）——场景复杂→自动分配更多 bit→码率升高；场景简单→自动分配更少 bit→码率降低。
优势：画质一致——动态场景不糊、静态场景不浪费。
劣势：码率不可控——动态场景（风吹树叶、雨雪）码率可能飙到平时的 3-5×→网络拥塞→丢包。不适合实时流。
IPC 适用：本地 SD 卡录像（带宽不受限——保证录像质量）。

CVBR (约束 VBR) ★推荐

VBR 画质 + CBR 上限 — IPC 最佳选择

原理：VBR 模式下自动调整 QP，但设置码率上限和下限。静态场景→码率可以降到 min_bitrate（省带宽/省存储）；动态场景→码率可以升高但不允许超过 max_bitrate（保护网络）。
参数示例：target=2Mbps, min=1Mbps, max=3Mbps——静态场景实际 ~1.2Mbps（省 40%），动态场景 ~2.8Mbps（画质保持）。
IPC 适用：主码流（兼顾画质和带宽——最佳实践）。

码率选择的经验公式 — H.264 IPC 场景
主码流：码率(Mbps) = 像素数/1000000 × 帧率/30 × 运动系数。运动系数：静态场景(室内、仓库)=1.0；一般室外=1.5；交通路口/人流密集=2.0。
例：1080p@30fps 室外一般 → 1920×1080/1000000=2.07M × 1.5 = 3.1Mbps 建议。
子码流：码率 = 主码流码率 / 10（如主码流 3Mbps→子码流 300Kbps）。
H.265/HEVC 比 H.264 节省 ~40% 码率（同等画质下）。但编码计算量高 1.5-2×——IPC SoC 的硬件编码器需支持。

GOP 结构与帧类型 — I/P/B 的选择

GOP 结构是IPC 编码参数中影响面最广的一个——改了 GOP 长度，延迟、码率、seek 速度、丢包容忍度全部受影响。

GOP 长度 (I 帧间隔)

每 N 帧插入一个 I 帧 — N=GOP 长度

GOP=30 (1 秒 @30fps)：标准配置。I 帧间隔 1 秒——seek 延迟 ≤1s，丢包恢复 ≤1s。I 帧开销占总码率 10-20%。
GOP=60 (2 秒)：更经济——I 帧占比低→同等画质下总码率低 5-10%。但丢包恢复需要 2s（要等到下一个 I 帧才能恢复花屏）。
GOP=15 (0.5 秒)：低延迟——WebRTC/IPC 实时预览推荐。I 帧频繁→seek 快（0.5s）、丢包恢复快。但总码率高 10-15%。
GOP=1 (全 I 帧)：零延迟——无帧间依赖。seek 不受限、丢包只影响单帧。但码率暴增 5-10×——只在极低延迟场景（远程操控、无人机 FPV）使用。

B 帧 — IPC 场景不推荐

B 帧需要等待未来帧——引入 1-3 帧编码延迟

B 帧的问题：① 编码延迟——B 帧要等后面的 P/I 帧先编码完才能参考→引入 1-3 帧延迟（33-99ms @30fps）② 解码延迟——接收端也要等参考帧到齐才能解码 ③ 计算量——B 帧的双向运动估计比 P 帧的单项复杂 1.5-2×。
适用场景：非实时场景（本地录像——延迟不敏感，B 帧的压缩增益 15-25% 有价值）。
不适用：实时预览、双向对讲——延迟 > 50ms 不可接受。IPC 实时流推荐 I-P-P-P...无 B 帧 的 GOP 结构——最低延迟。

场景推荐 GOPB 帧理由
实时预览 (WebRTC)15~30 (0.5~1s)禁用低延迟+快速恢复+无编码延迟
本地 SD 卡录像30~60 (1~2s)可选 (2 B-frames)存储优先省码率·画质优先
云存储上传30~60 (1~2s)禁用网络传输可靠性优先——B帧丢引用影响大
AI 检测流 (子码流)15~30禁用IDR 帧携带 SPS/PPS——AI 检测从任意 IDR 取第一帧即可启动

双码流配置 — 主码流 + 子码流

IPC 相机几乎都支持至少两路同时编码——主码流（高质量录像+实时预览）和子码流（AI 检测+弱网预览+多路预览）。两路编码器的参数配置思路完全不同。

主码流 (Main Stream) — 质量优先

1080p@30fps, 2~4Mbps, GOP=30, CVBR

目标：最高画质——用于 SD 卡录像和 WiFi/LAN 实时预览。
参数建议：分辨率=1080p/4MP、帧率=30fps、码控=CVBR (target=3Mbps/min=1.5Mbps/max=4.5Mbps)、GOP=30 (1s)、Profile=High、编码器优先级=最高。
关键约束：SD 卡写入时码率不能超过卡的最大持续写入速度（Class 10 = 10MB/s——远超编码码率，不是瓶颈。瓶颈是 SD 卡的随机写延迟——频繁 seek 导致写入速度波动）。主码流用 CVBR 为 SD 卡写入提供了码率可预测性。

子码流 (Sub Stream) — 效率优先

640×360@15fps, 200~400Kbps, GOP=15, CBR

目标：最低带宽占用——用于 4G 远程预览、AI 检测（人脸/车牌/运动）、NVR 多路预览。
参数建议：分辨率=640×360 (或 D1 720×576)、帧率=15fps（AI 检测——15fps 足够；降低编码负载）、码控=CBR (固定 300Kbps——4G 上行带宽硬约束)、GOP=15 (0.5s——AI 检测模型需要快速的 IDR 恢复)、Profile=Main或 Baseline（兼容旧解码器）。
关键约束：子码流的编码负载 → 编码器优先级=低。CPU/编码器资源不足时先降子码流帧率（15→10fps），保护主码流。

参数主码流 (高质量)子码流 (高效率)差异原因
分辨率1920×1080640×360 (或 704×576)主: 高清录像 + 大屏预览 · 子: 小窗预览 + AI 最低要求

平台实战 — Rockchip MPP + FFmpeg

IPC 产品最常用的两种编码器平台——Rockchip MPP（硬件编码，嵌入式 SoC）和 FFmpeg/libx264（软件编码，服务器/PC）。

Rockchip MPP (硬件 H.264/H.265)

RV1106/RV1109/RK3588 — MPP API 参数配置
// Rockchip MPP 编码器初始化 — 核心参数 MppEncCfg cfg; mpp_enc_cfg_init(&cfg); // 基础参数 mpp_enc_cfg_set_s32(cfg, "prep:width", 1920); mpp_enc_cfg_set_s32(cfg, "prep:height", 1080); mpp_enc_cfg_set_s32(cfg, "rc:mode", MPP_ENC_RC_MODE_VBR); mpp_enc_cfg_set_s32(cfg, "rc:bps_target", 3000000); // 3Mbps mpp_enc_cfg_set_s32(cfg, "rc:bps_max", 4500000); // 上限 4.5Mbps mpp_enc_cfg_set_s32(cfg, "rc:bps_min", 1500000); // 下限 1.5Mbps // GOP 和帧类型 mpp_enc_cfg_set_s32(cfg, "rc:gop", 30); // GOP=30 (1s@30fps) mpp_enc_cfg_set_s32(cfg, "split:mode", 0); // 0=NoSplit (单slice) mpp_enc_cfg_set_s32(cfg, "profile:level", 42); // High Profile, Level 4.2 // 低延迟优化 (实时预览) mpp_enc_cfg_set_s32(cfg, "rc:quality", MPP_ENC_RC_QUALITY_BEST); mpp_enc_cfg_set_s32(cfg, "prep:rotation", 0); // 不旋转 (除非 Sensor 方向特殊)

FFmpeg / libx264 (软件编码)

x264 preset + tune — 配置思路与 MPP 一致
# FFmpeg 命令行 — IPC 编码参数映射 ffmpeg -f v4l2 -input_format nv12 -video_size 1920x1080 \ -framerate 30 -i /dev/video0 \ # H.264 编码器参数 -c:v libx264 \ -preset ultrafast \ # 编码速度最快→延迟最低 (vs medium/slow) -tune zerolatency \ # 零延迟调优: 无B帧+单线程+关闭lookahead -profile:v high \ -level:v 4.2 \ # 码率控制 -b:v 3M \ # 目标码率 3Mbps -maxrate 4.5M \ # VBV 上限 -bufsize 6M \ # VBV 缓冲大小 (2× 目标码率) # GOP 结构 -g 30 \ # GOP=30 (1s@30fps) -keyint_min 30 \ # 最小 I 帧间隔 = GOP (禁止过早插I帧) -sc_threshold 0 \ # 关闭场景切换检测 (固定 GOP) -bf 0 \ # B 帧=0 (IPC 实时流禁B帧!) -refs 1 \ # 参考帧=1 (降低延迟) # 输出 -f rtsp rtsp://10.31.3.144:554/stream1
⚠ 硬件编码 vs 软件编码的选择
IPC 产品强烈推荐硬件编码 (Rockchip MPP / 海思 MPP / Ingenic)——原因：① 编码延迟 2-5ms vs 软件 5-15ms ② CPU 负载几乎为零（CPU 只做配置和控制，数据面全在硬件+DMA）③ 功耗低（专用硬件 > 通用 CPU 的能效比 10-100×）④ 编码质量持平（现代硬件编码器的画质已接近 x264 medium preset）。
只有以下场景用软件编码：① PC/服务器端编码（无硬件编码器可用）② 需要 x264 的高级参数调优（硬件编码器参数有限）③ 非实时场景——延迟不敏感。

场景自适应编码 — 不是一组参数走天下

日夜切换、晴雨变化、场景复杂度——一组固定的编码参数不可能在所有条件下都最优。场景自适应是画质竞争力的关键。

日夜切换

白天参数 vs 夜晚参数 — 完全不同的优化方向

白天模式（彩色，照度充足）：画面细节丰富（树叶/人脸纹理）→编码复杂度高→需要较大的目标码率（如 3Mbps）。I 帧 QP 可以较小（高质量参考帧）。
夜晚模式（IR 黑白，低照度）：画面噪声大（暗光 Sensor 的读出噪声↑）→编码器的运动估计失效——噪声被误判为"运动"→很多宏块走 Inter 预测但实际上应该走 Intra → P 帧变大→码率异常。
解决：夜晚模式：① 提高 I 帧 QP（噪声多——高 QP 有助于平滑噪声→减少编码波动）② 适度增加去噪强度（ISP 的 2D/3D NR——编码前降噪——减少无效运动矢量）③ 提高目标码率 20-30%（补偿噪声带来的编码效率下降）。

运动检测 → 动态 GOP

静态场景可以拉长 GOP · 运动场景必须缩短 GOP

静态场景（无人仓库、夜间停车场）：画面几乎不变→P 帧几乎全是 Skip Block（直接复制参考帧，零 bit）→P 帧极小（5-15KB）→可以安全拉长 GOP 到 60-120（2-4s）→减少 I 帧占比→总码率降低 15-25%。
动态场景（人流、交通）：画面快速变化→P 帧包含大量运动矢量→→帧间依赖复杂→缩短 GOP 到 15——快速切入 IDR 允许解码器从丢包中快速恢复。
运动检测的实现：① 编码器输出每帧的 intra/inter 宏块比例 ② 连续 N 帧 Inter 比例 < 30%（画面静止）→拉长 GOP ③ Inter 比例 > 80%（运动剧烈）→缩短 GOP。

带宽自适应

网络恶化 → 动态降参数 · 网络恢复 → 恢复参数

GCC/拥塞检测触发：① 丢包率 > 5% → 快速降目标码率（×0.85）→ 同时降帧率（30→20fps——降低码率的 2 倍杠杆）② 丢包率 > 2% 持续 > 3s → 切换子码流分辨率（640×360）作为主预览流 ③ 丢包率 < 0.5% 持续 > 5s → 恢复码率（×1.05 逐步回升）。
不推荐的方案：网络恶化时只降码率不降帧率→码率降低→每帧 bit 骤减→QP 狂升→画质崩塌（严重块效应）。降帧率 + 适度降码率 → 保持单帧画质。
